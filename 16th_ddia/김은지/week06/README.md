## 쓰기 스큐와 팬텀

다른 트랜잭션들이 동시에 같은 객체에 쓰려고 할 때 발생할 수 있는 두가지 경쟁 조건

- 더티 쓰기: 커밋 안된 값을 덮어 씀
- 갱신 손실: 커밋된 값을 읽었지만 동시 수정으로 덮어씀

스냅숏 격리의 경우 읽기 스큐를 방지할 수 있다 그러나 쓰기 스큐는 방지할 수 없다.

### 쓰기 스큐

커밋 시점에 읽은 전제가 깨지게 되는 경우

```
규칙: 당직 의사가 최소 1명은 있어야 한다

현재 DB: A = 당직중, B = 당직중

트랜잭션 1 (A가 퇴근하려고)    트랜잭션 2 (B가 퇴근하려고)
  
  당직 인원 조회 → 2명           당직 인원 조회 → 2명
  "2명이니까 내가 빠져도 되겠다"  "2명이니까 내가 빠져도 되겠다"
  
  A = 퇴근으로 변경               B = 퇴근으로 변경
  
  커밋                           커밋

최종 DB: A = 퇴근, B = 퇴근  ← 당직 의사 0명
```

- 더티 쓰기와 갱신 손실은 다른 트랜잭션이 하나의 동일한 객체를 갱신하는 경우 생기고 쓰기 스큐는 다른 트랜잭션이 다른 객체를 갱신하면서 생기는 경쟁 조건에서 생긴다.

#### 쓰기 스큐 막는 법

- 스냅숏 격리에서의 갱신 손실 자동 감지는 쓰기 스큐를 막지 못한다 → 직렬성 격리가 필요
- 어떤 경우는 전제에 대한 제약 조건을 걸 수도 있다. (트리거나 구체화된 뷰로 구현)
- 트랜잭션이 의존하는 로우를 명시적으로 잠금으로서 쓰기 스큐를 막을 수도 있다.
    
    ```sql
    BEGIN TRANSACTION;
    
    SELECT * FROM doctors
    	WHERE on_call = true
    	AND shift_id = 1234 FOR UPDATE; // 로우 잠금
    ```
    

### 팬텀

```
1. SELECT 질의가 어떤 검색 조건에 부합하는 로우를 검색한다.
2. 첫 번째 질의의 결과에 따라 애플리케이션 코드를 어떻게 진행할지 결정한다.
3. 애플리케이션이 계속 처리하기로 결정했다면 데이터베이스에 쓰고 트랜잭션을 커밋한다. 이 쓰기의 효과로 2단계를 결정한 전제 조건이 바뀐다. 즉 쓰기를 커밋한 후 1단계의 SELECT 질의를 재실행하면 결과가 바뀌게 된다.
```

- 의사 호출 대기의 경우에는 3단계에서 변경될 로우가 1단계에서 반환된 로우 중 하나이기 때문에 1단계 로우를 잠금으로써 쓰기 스큐를 회피할 수 있다.

**그러나 1번에서 로우가 존재하지 않는지 확인하고 쓰기 작업이 같은 조건에 부합하는 로우를 추가하는 경우 1단계의 질의가 아무 로우도 반환하지 않기 때문에 로우 잠금을 할 수가 없다.**

→ 이처럼 어떤 트랜잭션에서 실행한 쓰기가 다른 트랜잭션의 검색 질의 결과를 바꾸는 효과를 팬텀이라고 한다.

= 즉 팬텀이 발생하기 때문에 쓰기 스큐가 생긴다.

#### 충돌 구체화

팬텀이 생기는 이유가 로우 잠금을 할 수 있는 객체가 1단계 질의에서 나오지 않기때문이라면 잠금 객체를 임의로 추가해서 막을 수 있을 수 있다.

- 회의실 예약 같은 경우 회의실과 시간 범위의 모든 조합에 대해 로우를 미리 만들어두고 예약을 원하는 로우를 잠금으로써 쓰기 스큐를 막을 수 있다.
- 팬텀을 데이터베이스에 존재하는 모든 구체적인 로우 집합에 대한 잠금 충돌로 변환
- 방법을 알아내기 어렵고 오류가 발생하기 쉬우며, 동시성 제어 메커니즘이 애플리케이션 단으로 나오는 것도 옳지 않아서 **직렬성 격리 수준**이 더 선호된다

# 직렬성

가장 각렬한 격리 수준으로 여러 트랜잭션이 병렬로 실행되더라도 최종 결과는 동시성 없이 한 번에 하나씩 직렬로 실행될 때와 같도록 보장한다. 데이터베이스가 발생할 수 있는 모든 경쟁 조건을 막아준다.

완회된 격리 수준보다 직렬성 격리가 모두 좋다면 왜 그것을 사용하지 않을까?

1. 트랜잭션을 순차적으로 실행하기
2. 2단계 잠금(PL)
3. 직렬성 스냅숏 격리 같은 낙관적 동시성 제어

3가지 방법으로 직렬성을 제공한다.

## 순차적인 직렬 실행

동시성을 완전히 제거해서 동시성 문제를 피하자. 한번에 트랜잭션 하나씩만 직렬로 단일 스레드에서 실행하므로써 트랜잭션 사이의 충돌을 감지하고 방지하는 문제를 회피한다.

오랜 기간 동안 높은 성능을 위해 다중 스레드 동시성이 필수적인 것으로 여겨졌으나 최근에서야 단일 스레드에서 트랜잭션 실행하는 게 실현 가능하다고 결론 내려짐

- 이유1. 램 가격이 저렴해져서 많은 사례에서 활성화된 데이터 셋 전체를 메모리에서 유지할 수 있을 정도가 됨. 트랜잭션이 접근해야하는 모든 데이터가 메모리에 있다면 데이터를 디스크에서 읽어 오기를 기다릴 때보다 트랜잭션이 빠르게 실행됨
- 이유2. OLTP 트랜잭션이 보통 짧고 읽기와 쓰기의 개수가 적다는 것을 깨달았기 때문. 오래 실행되는 분석 질의는 읽기 전용이므로 직렬 실행 루프 밖에서 일관된 스냅슛을 사용하고 OLTP는 단일 스레드에서 실행함
- 주로 Redis, 볼트DB/H-스토어 등에서 제공
- 잠금을 코디네이션하는 오버헤드를 피함으로써 성능이 동시성 지원하는 시스템보다 나을 수도 있음. 이들의 처리량은 CPU 코어 하나의 처리량으로 제한됨

### 트랜잭션을 스토어드 프로시저 안에 캡슐화

#### 초창기

```sql
트랜잭션 시작
  → 사용자에게 화면 보여줌
  → 사용자 입력 기다림 (몇 초? 몇 분?)
  → 입력 받아서 DB 수정
트랜잭션 커밋
```

#### 문제

사용자가 입력하는 동안 트랜잭션이 잠금을 붙잡고 대기하게 되면 데이터베이스 대부분은 유휴 상태지만 잠재적으로 매우 많은 동시 실행 트랜잭션을 지원해야 하므로 효율적이지 않다고 봄

→ 대부분의 OLTP 애플리케이션은 트랜잭션 내에서 대화식으로 사용자 응답을 대기하는 것을 회피함으로써 **트랜잭션을 짧게 유지한다**

= 새로운 HTTP 요청은 새로운 트랜잭션을 시작한다.

#### 상호작용식 트랜잭션

```
앱 서버                    DB 서버
  질의1 전송 ────────────→
             ←──────────── 결과1 반환
  결과1 보고 질의2 결정
  질의2 전송 ────────────→
             ←──────────── 결과2 반환
  질의3 전송 ────────────→
             ←──────────── 결과3 반환
  커밋 ──────────────────→
```

사람이 제외됐음에도 한 번에 구문 하나씩 애플리케이션 코드와 데이터베이스 서버 사이에서 질의 결과를 주고 받는다. **네트워크 통신**에 많은 시간을 소비한다.

동시성을 허용하지 않게 되면 다음 질의를 발행하기까지 대기하는데 대부분의 시간을 쓰데 되므로 비효율적이다

→ 여러 트랜잭션을 동시에 실행할 필요가 있다.

**단일 스레드에서 트랜잭션을 순차적으로 처리하는 시스템들은 상호작용하는 다중 구문 트랜잭션을 허용하지 않는다.**

#### 해결책: 스토어드 프로시저

```
앱 서버                    DB 서버
  프로시저 전송 ──────────→
  (로직 전체를 한방에)       내부에서 알아서 다 처리
             ←──────────── 최종 결과만 반환
```

로직 전체를 DB안에 미리 저장해두고 이거 실행하라고 한 번만 호출하는 것 (네트워크 왕복이 1번으로 줄어든다)

→ 트랜잭션에 필요한 데이터는 모두 메모리에 있고 스토어드 프로시저는 네트워크나 디스크 I/O 대기 없이 매우 빠르게 실행됨

### 스토어드 프로시저(Stored Procedure)

```sql
DELIMITER
CREATE PROCEDURE myProc()
BEGIN
     SELECT * FROM employees where first_name = 'georgi';
     SELECT * FROM departments where dept_name = 'Development';
END
DELIMITER;

CALL myProc(); // 프로시저 호출
```

- 데이터베이스에서 실행되는 코드는 관리가 어려운 문제
- 데이터베이스는 애플리케이션 서버보다 성능에 더 민감하므로(애플리케이션 서버에서 데이터베이스 인스턴스 하나를 공유하므로) 잘못 작성된 스토어드 프로시저가 더 악영향을 줄 수도 있음

### 파티셔닝

단일 CPU 코어의 속도로 트랜잭션 처리량이 제한되므로 단일 스레드가 병목이면 **데이터를 쪼개서 각자 따로 처리하게 한다** (ex. 볼트DB)

```
파티션A (코어1)    파티션B (코어2)    파티션C (코어3)
트랜잭션1          트랜잭션2          트랜잭션3
트랜잭션4          트랜잭션5          트랜잭션6
```

트랜잭션이 단일 파티션 내에서만 데이터를 읽고 쓰도록 데이터셋을 파티셔닝할수 있다면 각 파티션은 다른 파티션과 독립적으로 실행되는 자신만의 트랜잭션 처리 스레드를 가질 수 있다 → 각 CPU 코어에 각자의 파티션을 할당해서 트랜잭션 처리량을 CPU 코어 개수에 맞춰 선형적으로 확장할 수 있음

**조건: 트랜잭션이 하나의 파티션 안에서만 끝나야 한다**

- 여러 파티션에 접근해야하는 트랜잭션이 있다면 데이터베이스가 해당 트랜잭션이 접근하는 모든 파티션에 걸침함 → 오버헤드 발생해서 단일 파티션 트랜잭션보다 매우 느림

### 직렬 실행 요약

1. 트랜잭션의 범위는 작고 빨라야 함
2. 활성화된 데이터셋이 메모리에 전부 적재될 수 있는 경우에만 직렬 실행 이용함(단일 스레드 트랜잭션에서 디스크에 접근해야 되면 시스템이 많이 느려짐)
3. 쓰기 처리량이 단일 CPU 코어에서 처리할 수 있도록 낮아야 함
    1. 그렇지 않으면 여러 파티션에 걸친 트랜잭션이 없도록 파티셔닝 해야함
4. 여러 파티션에 걸친 트랜잭션은 성능 제한이 있다

## 2단계 잠금(PL)

- 트랜잭션 A가 하나를 읽고 B가 그 객체에 쓰기를 원한다면 B는 진행하기 전에 A가 커밋되거나 어보트될 때까지 기다려야 함(B가 A 몰래 객체 변경하면 안됨)
- 트랜잭션 A가 객체에 썼고 트랜잭션 B가 그 객체를 읽기 원한다면 B는 진행하기 전 A가 커밋되거나 어보트될 때까지 기다려야함(객체의 과거 버전 읽으면 안됨)

**스냅숏 격리는 읽는 쪽은 쓰는 쪽을 막지 않고 쓰는 쪽도 읽는 쪽을 막지 않는다는 원칙이 있는데 이것이 차이다**

→ 스냅숏 격리의 경우 무조건 과거 버전을 읽음. 쓰기↔쓰기만 막음

### 구현

데이터베이스의 각 객체에 잠금을 사용해 구현한다

**공유 모드(shared mode)나 독점 모드(exclusive mode)**

- 객체를 읽으려면 **공유 모드** 잠금 획득 (여러 트랜잭션이 동시에 공유 모드 잠금을 획득하는 것은 허용되지만 한 트랜잭션이 독점 모드 잠금을 얻고 있다면 대기해야함)
- 객체에 쓰기 위해서는 독점 모드 잠금을 획득해야 함 (동시에 잠금 얻을 수 없음 공유, 독점 잠금이 있다면 대기해야함)
- 객체를 읽다가 쓰기를 실행할 때는 공유 잠금을 독점 잠금으로 업그레이드 해야함
- 트랜잭션이 잠금을 획득한 이후에는 트랜잭션이 종료 될 때까지 잠금을 갖고 있어야 함

**교착 상태**가 발생할 수도 있음(서로가 서로를 대기하는 것)

→ 데이터베이스는 트랜잭션 사이의 교착 상태를 자동으로 감지하고 트랜잭션 중 하나를 어보트시켜 다른 트랜잭션이 진행할 수 있게 함

### 약점

완화된 격리 수준 보다 트랜잭션 처리량과 질의 응답 시간이 나쁨

- 잠금을 획득하고 해제하는 오버헤드 때문
- 동시성이 줄어들기 때문
    - 설계에 따라 경쟁 조건을 유발하는 일을 하려고 하면 한 트랜잭션은 다른 트랜잭션이 완료될 때까지 기다려야 함
- 교착 상태가 잦으면 어보트가 자주 발생하므로 문제이기도 함
    - 커밋 후 읽기 격리 수준에서도 교착 상태가 생길 수 있지만 더 자주 발생
    - 커밋 후 읽기 격리는 트랜잭션들이 쓰기 잠금만을 보유하지만 2PL은 읽기, 쓰기 잠금이 모두 존재하므로 교착 상태가 더 자주 발생한다.

### 서술 잠금(predicate lock)

팬텀(한 트랜잭션이 다른 트랜잭션의 검색 질의 결과를 바꿔버리는 문제)를 해결하기 위해 서술 잠금이 필요하다

2PL의 읽기/쓰기 잠금은 기존 로우에 락을 걸어 보호하는 방식인데 팬텀의 경우 특정 로우가 아니라 검색 조건에 해당하는 범위가 바뀌는 것이라 공유/독점 잠금으로는 못막음

- 특정 객체가 아니라 검색 조건에 부합하는 모든 객체에 속한다.

```sql
SELECT * FROM bookings
	WHERE room_id = 123 AND
		end_time > '2018-01-01 12:00' AND
		start_time < '2018-01-01 13:00'
```

- 트랜잭션이 SELECT 처럼 어떤 조건에 부합하는 객체를 읽으려면(읽기) 조건에 대한 공유 모드 서술 잠금을 획득해야 함. 다른 트랜잭션이 그 조건에 부합하는 어떤 객체에 대해 독점 잠금을 갖고 있으면 잠금 해제를 기다려야 함
- 트랜잭션이 어떤 객체를 삽입, 갱신, 삭제하려면 (쓰기) 기존 값이나 새로운 값 중 기존의 서술 잠금에 부합하는게 있는지 확인 → 부합하는 서술 잡금을 트랜잭션이 잡고 있으면 진행하기전 그 트랜잭션이 커밋되거나 어보트될 때까지 기다려야 함
- **기존 행 잠금과 다른 점은 조건에 해당하는 아직 없는 행(새로운 행)을 삽입/수정하려는 쓰기도 막힌다라는 것**

### 색인 범위 잠금(index-range locking, next-key locking)

트랜잭션들이 획득한 잠금이 많게 되면 조건에 부합하는 잠금을 확인하는데 시간이 걸려서 서술 잠금은 잘 동작하지 않음 → 따라서 색인 범위 잠금을 구현함

더 많은 객체가 부합하도록 서술 조건을 간략화시키는 것

ex. 정오와 오후 1시 사이에 123번 방을 예약하는 것에 대한 서술 잠금은 모든 시간 범위에 123번 방을 예약하는 것에 대한 잠금 or 정오와 오후 1시 사이에 모든 방을 잠그는 것으로 근사 

팬텀과 쓰기 스큐를 보호할 수 있음(더 넓은 범위를 잠구니까) 큰 범위를 잠구긴 하지만 오버헤드가 더 낮다

- 혹시 범위 잠금을 잡을 수 있는 적절한 색인이 없으면 테이블 전체에 공유 잠금을 잡음

## 직렬성 스냅숏 격리(SSI)

성능도 좋고 다양한 경쟁 조건에도 잘 막아지는 격리

→ 완전한 직렬성을 제공하고 스냅숏 격리에 비해 약간의 성능 손해만 존재

2단계 잠금은 비관적 동시성 제어 메커니즘(잘못될 가능성이 있다면 상황이 안전해질 때까지 기다리는게 낫다는 원칙) = 상호 배제와 비슷

직렬성 스냅숏 격리는 **낙관적 동시성 제어 기법** 으로 위험한 상황이 발생할 가능성이 있을 때 트랜잭션을 막는 대신 괜찮아질 것이라고 낙관적으로 생각하고 진행한다는 뜻

→ 트랜잭션이 커밋되길 원할 때 격리가 위반됐는지 확인하고 그렇다면 어보트하고 재시도, 아니라면 커밋하도록 결정

**경쟁이 심한 경우 어보트시켜야 할 트랜잭션의 비율이 높아지므로 성능이 떨어지지만 트랜잭션 사이에 경쟁이 너무 심하지 않으면 성능이 좋다**

스냅숏 격리를 기반으로 하므로 모든 읽기는 데이터베이스의 일관된 스냅슛을 보게 하고 거기에 쓰기 작업 사이의 직렬성 충돌을 감지하고 어보트시킬 트랜잭션을 결정하는 알고리즘을 추가한다

### 쓰기 스큐를 감지하는 방법

전제가 깨졌는지 확인하는 방법

#### 1. 오래된 MVCC 읽기 감지하기

![image.png](attachment:8e5e23dc-06d9-4a53-86bf-3a42a8040060:image.png)

스냅숏 격리는 보통 다중 버전 동시성 제어로 구현한다. MVCC 데이터베이스의 일관된 스냅숏에서 읽으면 스냅숏 생성 시점에 다른 트랜잭션이 썼지만 커밋되지 않은 데이터는 무시한다

→ 내가 스냅숏으로 읽은 데이터가 이미 다른 트랜잭션에 의해 바뀌었는지를 판단해야함 (= 트랜잭션이 MVCC 가시성 규칙에 따라 다른 트랜잭션의 쓰기를 무시하는 경우를 추적해야 함)

- MVCC의 경우 읽기 트랜잭션 시작 시점에 데이터베이스느 그 시점에 진행 중인 모든 다른 트랜잭션 목록을 만든다. 이 목록에서 이전 트랜잭션중 내가 읽고 있는 이 데이터를 써서 커밋된 게 있는지만 확인하면 됨

**내가 무시했던 트랜잭션 중 커밋된게 있는지 확인**

- 커밋 됐으면 어보트시킴, 커밋 안됐으면 그냥 트랜잭션 진행

**읽기 트랜잭션이 내 읽기가 오래됐는지 판단하는 것**

#### 2. 과거의 읽기에 영향을 미치는 쓰기 감지하기

![image.png](attachment:e58b5ee6-73f1-473a-913a-4d48fd7c3422:image.png)

트랜잭션이 데이터베이스에 쓸 때 영향받는 데이터를 최근에 읽은 트랜잭션이 있는지 색인에서 확인함 (키 범위에 쓰기 잠금을 획득하는 것과 비슷하지만 읽는 쪽에서 커밋될 때까지 차단하진 않음 = 지뢰선 처럼 동작)

- 색인에 누가 읽었는지를 저장해두고 그를 통해 찾는다.

→ 읽기 트랜잭션이 읽은 데이터가 최신이 아니라고 알려주기만 함

읽기 트랜잭션도 쓰기 트랜잭션에게 알려줌, 둘중 하나가 먼저 커밋된후 충돌이 나면 뒤의 트랜잭션이 어보트됨

**쓰기 트랜잭션이 다른 읽기에 영향을 미쳤는지 확인**

### 성능

트레이드 오프: 트랜잭션의 읽기 쓰기를 추적하는 세밀함 정도

- 데이터베이스가 각 트랜잭션의 동작을 매우 상세하게 추적하면 어보트돼야 하는 트랜잭션을 정확히 판단할 수 있지만 기록 오버헤드가 심해짐
- 덜 상세하게 추적하면 빠르지만 진짜 필요한 것보다 지나치게 많은 트랜잭션이 어보트될 수 있음

어떤 경우에는 다른 트랜잭션에서 덮어쓴 정보를 트랜잭션이 읽어도 괜찮음

- **2PL에 비해 트랜잭션이 다른 트랜잭션들이 잡고 있는 잠금을 기다리느라 차단될 필요가 없다**
- 순차 실행에 비해 단일 CPU 코어의 처리량에 제한되지 않는다. 여러 장비에 걸쳐서 파티셔닝 되어 있어도 직렬성 격리를 보장하면서 읽고 쓸수 있음
