# 03. 저장소와 검색

## B트리

가장 널리 사용되는 색인 구조. 고정 크기 블록이나 페이지를 단위로 데이터를 읽고 씀

정렬된 키-값 쌍 유지 (페이지에서 하위 페이지는 키가 이어지는 범위를, 참조 사이의 키는 해당 범위 경계가 어딘지를 나타냄)

갱신 시: 키를 포함한 리프 페이지 검색 후 페이지 값을 바꾼 다음 디스크에 다시 기록

추가 시: 새로운 키 포함하는 범위 페이지 찾아 추가/ 페이지를 둘로 나누고 상위 페이지가 새로운 키 범위의 하위 부분을 알 수 있게 갱신

- 로그 구조화 색인은 같은 위치의 파일은 변경하지 않는 대신 파일을 계속 추가하고 컴팩션하는 과정에서 위치가 바뀔 수 있음
- B트리는 페이지를 덮어쓰지만 페이지 위치를 변경하지 않음

### 쓰기 전 로그 (WAL, 재실행 로그)

B트리 변경 사항을 트리에 적용하기 전 별도의 추가 전용 파일에 기록(디스크에 덮어쓰는 과정이 복잡하고 훼손될 가능성이 있어서)

#### 래치

다중 스레드가 동시에 B트리에 접근하면 일관성 깨질 수 있어서 동시성 제어 해야함

- 메모리 내의 공용 구조체에 대한 물리적 일관성을 위해 사용
- 특정 스레드가 메모리 페이지 수정하는 동안 다른 스레드가 메모리 읽거나 수정하지 못하도록
- 페이지 래치
    - 수정할 리프 노드 찾으면 해당 페이지에 대해 X-Latch(배타적 래치) 획득
- 로그 버퍼 래치
    - 로그 버퍼에 기록할 때 로그 버퍼의 Tail을 점유하기 위해 획득
    - 순서가 뒤섞이지 않도록

### B트리 최적화

- **쓰기 시 복사**: WAL 대신 변경된 페이지를 다른 위치에 기록하고 상위 페이지가 새 위치를 가리키도록 갱신
- **키 축약**: 페이지에 키 전체 대신 축약된 형태를 저장해 트리 깊이 감소
- **연속 배치**: 리프 페이지를 디스크 상에 연속된 순서로 배치
    - 트리 커질수록 순서 유지 어려움
- **포인터 추가**: 각 리프 페이지가 형제 페이지를 참조해 상위 페이지 이동 없이 키 스캔 가능
- **프랙탈 트리**: 디스크 탐색을 줄이기 위해 로그 구조화 개념을 일부 도입한 B트리 변형

## B트리 vs LSM 트리 비교

| 항목 | B트리 | LSM 트리 |
| --- | --- | --- |
| 읽기 성능 | 빠름 | 상대적으로 느림 (각 컴팩션 단계에 있는 여러 세그먼트 확인 필요) |
| 쓰기 성능 | 상대적으로 느림 | 빠름 (순차 쓰기) |
| 쓰기 횟수 | 2회 (쓰기 전 로그, 트리 페이지) | 컴팩션과 병합으로 여러 번 |
| 키 중복 | 한 위치에만 존재 | 여러 세그먼트에 복사본 존재 가능 |
| 트랜잭션 | 트리에 직접 잠금 포함, 강력한 트랜잭션 보장 | 상대적으로 복잡 |
| 압축률 | 파편화로 인해 낮음 | 높음 |
| 구현 성숙도 | 높음 | 상대적으로 낮음 |

SSD는 블록 덮어쓰기 횟수가 제한되므로 쓰기 증폭 중요

→ LSM 트리는 순차 쓰기와 높은 압축률 덕분에 SSD 환경에서 유리

### LSM 트리 장점

**LSM 트리는 보통 B 트리보다 쓰기 처리량을 높게 유지할 수 있다**

- 여러 페이지를 덮어 쓰는 것이 아니라 순차적으로 컴팩션된 SS테이블 파일을 쓰기 때문
    - 자기 하드드라이브는 순차 쓰기가 임의 쓰기보다 훨씬 빠름
- 압축률이 좋아 디스크에 더 적은 파일 생성
    - B트리는 파편화로 인해 사용하지 않는 디스크 공간이 남음
    - LSM 트리는 주기적으로 SS테이블을 다시 기록해 파편화를 제거하므로 저장소 오버헤드가 낮음 (특히 레벨 컴팩션 사용 시)
- 대부분의 SSD 펌웨어는 임의 쓰기를 순차 쓰기로 전환하기 위해 내부적으로 로그 구조화 알고리즘을 사용함
    - 낮은 쓰기 증폭과 파편화 감소는 SSD에서 훨씬 유리
    - 데이터를 더 밀집해 표현하면 I/O 대역폭 내에서 더 많은 읽기와 쓰기 요청이 가능

### LSM 트리 단점

- 컴팩션 과정이 읽기/쓰기 성능에 영향을 줄 수 있음 (= 쓰기 중단/쓰기 압력)
    - 저장소 엔진은 동시 접근 영향이 없게 컴팩션을 수행하려고 함 → 디스크 자원의 한계 → 컴팩션 연산이 끝날 때 까지 요청 대기해야하는 상황이 존재
- 디스크의 쓰기 대역폭은 유한한데 쓰면 쓸수록 사용가능한 대역폭이 줄어듦
- 컴팩션 설정이 미흡하면 유입 쓰기 속도를 따라가지 못해 세그먼트 수가 계속 증가

## 색인 종류

### 보조 색인

관계형 데이터베이스에서 다양한 컬럼에 보조 색인을 생성 가능

- 키가 고유하지 않음

| 종류 | 설명 | 특징 |
| --- | --- | --- |
| 클러스터드 색인 | 색인 안에 실제 로우 데이터를 직접 저장 | 읽기 성능 높지만 추가적인 저장소 필요, 쓰기 과정에서 오버헤드 → 트랜잭션 보장 강화 |
| 비클러스터드 색인 | 색인 안에 데이터의 주소만 저장(힙 파일 참조) | 키를 변경하지 않고 값만 갱신할때 효과적 |
| 커버링 색인 | 일부 컬럼을 색인 안에 저장하는 중간 방식 | 읽기 성능 높지만 추가적인 저장소 필요, 쓰기 과정에서 오버헤드 → 트랜잭션 보장 강화 |

### 다중 컬럼 색인

- **결합 색인**: 여러 필드를 단순 결합해 하나의 키로 만드는 방식
- **다차원 색인**: 지리 공간 데이터처럼 여러 차원을 동시에 질의할 때 사용
    - 표준 B트리/LSM 트리로는 위도와 경도를 동시에 범위 질의할 수 없음
    - 공간 채움 곡선(이차원 위치 → 단일 숫자 변환) 활용 후 B트리 색인 이용
    - R트리 같은 전문 공간 색인 사용

### 퍼지 색인

오타 처리 등 유사한 키를 검색할 때 사용

- 루씬은 특정 편집 거리 내의 단어를 검색 가능
- SS테이블과 같은 구조 이용
- 인메모리 색인으로 유한 상태 **오토마톤**(트라이와 유사) 사용
    - 접두사와 접미사를 공유해 색인에서 이 형태로 저장

## 인메모리 데이터베이스

전체 데이터셋을 디스크가 아닌 메모리에 보관하는 방식

**지속성 확보 방법**

- 특수 하드웨어 사용
- 디스크에 변경사항 로그 기록
- 디스크에 주기적 스냅숏 기록
- 다른 장비에 인메모리 상태 복제

**장점**

- 디스크 기반 색인으로 구현하기 어려운 데이터 모델 지원
- 디스크 중심 아키텍처의 오버헤드 없음

**안티 캐싱**

메모리가 부족할 때 최근에 사용하지 않은 데이터를 디스크로 내보내고 필요 시 다시 적재 → OS의 페이지 단위보다 개별 레코드 단위로 처리해 더 효율적으로 메모리를 관리

## 트랜잭션 처리와 분석 (OLTP vs OLAP)

### 비교

| 항목 | OLTP | OLAP |
| --- | --- | --- |
| 설명 | 사용자의 입력을 기반으로 레코드 삽입 및 갱신 | 많은 수의 레코드를 스캔해 레코드당 일부 칼럼만 읽어 집계 통계 계산 |
| 주요 읽기 패턴 | 질의당 적은 수의 레코드,
키 기준으로 가져옴 | 많은 레코드에 대한 집계 |
| 주요 쓰기 패턴 | 임의 접근, 사용자 입력을 낮은 지연 시간으로 기록 | 대규모 불러오기 또는 이벤트 스트림 |
| 주요 사용처 | 웹 애플리케이션을 통한 최종 사용자/소비자 | 의사결정 지원을 위한 내부 분석가 |
| 데이터 표현 | 데이터의 최신 상태 | 시간이 지나며 일어난 이벤트 이력 |
| 데이터셋 크기 | GB~TB | TB~PB |
| 병목 | 디스크 탐색 | 디스크 대역폭 |

### 데이터 웨어하우스

OLTP 시스템에 영향을 주지 않고 분석 질의를 수행하기 위한 별도의 읽기 전용 데이터베이스

- **ETL 과정**: OLTP에서 추출 → 분석 친화적 스키마로 변환 → 웨어하우스에 적재

**스키마 설계**

- **별 모양 스키마**: 중심에 사실 테이블, 주변에 차원 테이블 배치
    - 사실 테이블
        - 특정 시각에 발생한 이벤트 → 페이지 뷰나 사용자 클릭에 해당
        - 로우에 개별 이벤트 담음 → 분석 유연성 극대화
        - 칼럼은 다른  테이블을 참조하는 외래키
- **눈꽃송이 모양 스키마**: 차원 테이블을 하위 차원으로 더 세분화한 변형된 템플릿

## 칼럼 지향 저장소

분석 질의는 보통 수백 개의 컬럼 중 4~5개만 접근

→ 칼럼 지향 저장소는 각 컬럼별로 같은 속성끼리 몰아서 하나의 파일로 저장

- 같은 값이 반복되는 컬럼을 효율적으로 압축
    - **비트맵 부호화**
        - 데이터를 비트 벡터로 변환 → 벡터 자체를 런 렝스 부호화로 압축
        - 가능한 각 값에 대한 비트맵 존재
        - 해당 비트맵을 보고 0, 1의 개수를 세서 압축(0 다음엔 1, 1다음엔 0이 어차피 나오고 전체 비트맵 길이는 고정되어 있으니까 마지막 값은 보통 기록안함
- 디스크로부터 메모리로 데이터 가져오는 대역폭 관련 문제
    - 메인 메모리에서 CPU 캐시로 가는 대역폭 효율적으로 사용/ 분기 예측 실패와 버블을 피하려고 함
    - 필요한 컬럼만 디스크에서 읽어 I/O 감소
    - 벡터화 처리로 CPU 연산 효율 향상

**순서 정렬**

- 자주 사용하는 질의에 맞게 1차 정렬 키 선택 → 특정 로우만 스캔할 수 있어서 빠름
- 보조 정렬 키 사용 → 그룹화/필터링 하는 질의에 도움
- 정렬된 컬럼은 같은 값이 연속으로 반복되어 압축률이 높음
- 여러 장비에 복제 시 서로 다른 방식으로 정렬해(서로 다른 키를 가지는) 저장하면 질의에 적합한 버전을 선택할 수 있음

**쓰기**: 칼럼 지향 저장소는 읽기에 최적화되어 쓰기 어려움

→ LSM 트리를 활용해 인메모리 저장소로 이동해 정렬된 구조에 추가하고 디스크에 쓸 준비

→ 충분한 쓰기를 모으면 디스크 칼럼 파일에 병합하고 대량으로 새로운 파일에 기록

### 구체화 뷰

SUM, COUNT, AVG 등 자주 사용하는 집계 결과를 캐시해두는 방식

- 원본 데이터 변경 시 갱신 비용이 비싸므로 OLTP보다 데이터 웨어하우스에 적합
- 데이터 큐브(OLAP 큐브): 일반화된 구체화 뷰의 특별 사례, 다차원 집계를 미리 계산해 저장

## OLTP 저장소 엔진 관점

- 로그 구조화 관점: 파일에 추가와 오래된 파일의 삭제만 허용하고 한 번 쓰여진 파일 갱신 X(LSM 트리)
- 제자리 갱신 관점: 덮어쓰기 되는 고정 크기 페이지의 셋으로 디스크를 다룸 (B트리)


# 04. 부호화와 발전

## 데이터 부호화 형식

부호화(직렬화, 마샬링): 인메모리 표현→바이트열

복호화(파싱, 역직렬화, 언마샬링): 바이트열→인메모리 표현

### 상위/하위 호환성

- 상위 호환성: 예전 코드는 새로운 코드가 기록한 데이터를 읽을 수 있어야 함
- 하위 호환성: 새로운 코드는 예전의 코드가 기록한 데이터를 읽을 수 있어야 함

**언어별 형식:** 프로그래밍 언어에 부호화가 묶여 있어 다른 언어의 데이터를 읽기 어려움

데이터를 빠르고 쉽게 부호화하기 위해 상위,하위 호환성 문제 의식 X

→ 주로 사용 X

### 이진 변형(JSON, XML, CSV의 이진화)

- 텍스트 형식보다 파싱이 빠르고 공간 덜 차지
- 필드 이름을 매번 포함해야하므로 비효율적
- 대규모 데이터 플로에서는 필드 이름 생략하고 스키마 사용하는 방식으로 넘어감

## 이진 부호화

- JSON/XML 데이터 모델에서의 타입 셋을 확장하지만 모델은 변경하지 않음 → 부호화된 데이터 안에 객체의 모든 필드 이름 포함 되어야 함 → 공간 절약이 크게 드러나지 않음
- **MessagePack**을 통해 부호화함

#### Apache Thrift(아파치 스리프트)

optional, required 표시자 존재

- 바이너리 프로토콜
    - 메세지팩 부호화와 달리 필드 이름은 없지만 숫자와 같은 필드 태그를 포함함
- 컴팩트 프로토콜
    - 바이너리 프로토콜과 같지만 필드 타입과 태그 숫자를 단일 바이트로 줄이고 가변 길이 정수를 사용해서 부호화함.

#### Protocol Buffers(프로토콜 버퍼)

- 스리프트의 컴팩트 프로토콜처럼 데이터를 줄여 부호화함

### 스키마 발전

**필드에 데이터 추가**

- 상위 호환성 유지 : 예전 코드에서 새로운 코드로 새로운 태그 번호를 가진 필드가 포함된 데이터를 읽으려는 경우 해당 필드를 무시할 수 있음
- 하위 호환성 유지: 각 필드에 고유한 태그 번호가 있는 동안에는 같은 의미를 가지므로 새로운 코드가 예전 데이터를 항상 읽을 수 있음
    - 새로운 필드 추가시 optional or 기본값설정

### 아브로

- 필드나 데이터타입을 식별하기 위한 정보가 없음
- 데이터 파싱하려면 스키마를 이용해 각 필드의 데이터타입 미리 파악 = 정확히 같은 스키마를 사용하는 경우에만 복호화 올바르게 가능

<aside>

#### 스키마 발전

- 쓰기 스키마와 읽기 스키마가 동일하지 않아도 되고, 호환 가능하면 됨
- 복호화할때 쓰기 스키마와 읽기 스키마를 함께 보고 다음 쓰기 스키마에서 읽기 스키마로 데이터 변환
    - 스키마 해석으로 필드 일치 시킴
- 상위/하위 호환성을 유지하기 위해 **기본값이 있는 필드만 추가 삭제**
- 기본값으로 널 허용을 위해 유니온 타입 사용
    
    `union { null, long, string } field;`
    
</aside>

- 태그번호가 없어서 → 동적 생성 스키마에 적용 쉬움
    - 스리프트나 프로토콜 버퍼 사용시 필드 태그 수동으로 할당해야함

## 데이터플로 모드

하나의 프로세스에서 다른 프로세스로 데이터를 전달하는 방식

### 데이터베이스를 통해 전달

하나의 프로세스가 부호화한 데이터를 기록하고 다른 프로세스가 그 데이터를 복호화해서 읽는 방식

하위 호환성 & 상위 호환성이 모두 필요하다

→ **데이터 라운드 트립**: 예전 버전의 코드가 새로운 필드가 없는 예전 레코드를 읽고 갱신한 후 갱신한 값 기록 시 새로운 필드에 대해 유지하는 것이 옳은 방식 (데이터가 유실될 가능성이 있어서)

---

### 서비스 호출을 통해 전달

하나의 프로세스가 네트워크를 통해 다른 프로세스로 요청을 전송하고 빠른 응답을 기대하는 방식

서비스를 배포와 변경에 독립적으로 만들어 애플리케이션 변경과 유지보수를 더 쉽게 할 수 있게 만들어야 함 

→ 서버 클라이언트 사이의 데이터 부호화는 여러 버전 간 호환이 가능해야함

데이터를 이때 부호화하는 방식으로 REST 와 RPC가 있다.

#### REST(Representational State Transer)

네트워크 상에서 API 요청을 하기 위한 HTTP 원칙 기반의 설계 철학 (외부 클라이언트와 통신시 사용)

- JSON 형식의 부호화
- 매개변수 추가나 응답 객체의 새로운 필드 추가와 같은 변경시 호환성을 유지하는 변경으로 간주(모르는 필드는 무시한다)

#### RPC(Remote Procedure Call)

로컬 함수를 부르는 것처럼 보이게 하는 원격 네트워크 요청 (MSA 환경에서 서버끼리 통신시 사용)

- 이진 부호화 형식

→ 네트워크의 경우 문제가 일상적이기 때문에 로컬 함수 호출처럼 취급하면 timeout, 멱등성 이런부분을 고려할 수 없고, 같은 작업 실행시에도 시간이 다르게 걸릴 수 있고 통신할때 부호화를 해줘야함

- RPC 프레임워크는 원격 요청이 로컬 함수 호출과 다르다는 사실을 보여주려고 함
    - 피네글/Rest.li : 비동기 작업을 캡슐화 하기 위해 Future(퓨처) 사용
        
        ex) `Future<Integer> result;`
        
        이렇게 선언하면 나의 로컬 함수 호출이 아닌 메모리 밖의 내용이므로 결과가 올 때까지 비워놓고 있어라 라는 뜻
        
        → 이 객체에 타임아웃 설정, 재시도 설정 등을 추가할 수 있음
        

RPC 클라이언트와 서버를 독립적으로 변경하고 배포해야함 

→ 모든 서버 갱신 후, 모든 클라이언트 갱신하면되므로 하위 호환성만 필요하다

---

### 비동기 메시지를 통해 전달

#### 메시지 브로커

메시지를 직접 네트워크 연결로 전송하지 않고 임시로 메시지를 저장하는 중간 단계

- 생산자: 메시지 보내는 서버
- 소비자: 메시지 받아서 처리하는 서버
- 메시지 브로커: 중간에서 메시지 보관하고 전달해주는 서버
    - 큐/토픽이라는 저장 창고 있음

메시지를 넣은건 구 버전인데, 서버 업데이트 돼서 읽는 서버가 신버전 일 수 있음 → 그러면 모르는 필드가 들어와도 보존될 수 있도록 주의해야함

#### 분산 액터 프레임워크

> 액터 모델: 자기만의 상태를 가지고 있고 다른 액터와 **메시지**를 통해서만 대화함
> 

메시지 브로커 + 액터 프로그래밍 모델

→ 상하위 호환성에 주의해야함

---

#### 순회식 업그레이드

동시에 모든 노드에 새로운 버전의 서비스를 배포하는 방식이 아닌 일부 노드에만 서서히 배포하는 방식

→ 하위 호환성과 상위 호환성을 제공하는 방식으로 부호화해야함
