# 데이터베이스 저장소 엔진

데이터베이스가 데이터를 저장하고 요청 시 다시 찾는 방법에 대한 정리

#### 저장소 엔진

- 로그 구조 계열
- 페이지 지향 계열

---

## 로그 구조 저장소 엔진

### 기본 개념

- **로그**: 추가 전용(append-only) 데이터 파일
- 읽기 시 전체 스캔이 필요 → O(n) 비용 발생

### 색인

데이터베이스에서 특정 키의 값을 효율적으로 찾기 위한 부가적인 데이터 구조

- 질의 성능에만 영향을 줌
- 쓰기 시 매번 갱신이 필요하므로 쓰기 오버헤드 발생

### 해시 색인

키를 데이터 파일의 **바이트 오프셋에 매핑**하는 키-값 해시 맵 방식

- 조회 시 해시 맵으로 오프셋을 찾아 해당 위치를 구하고 디스크에서 해당 값을 읽음
- 모든 키가 메모리에 유지 → 고성능 읽기, 쓰기 보장
- 각 키의 값이 자주 갱신되는 상황에 적합(ex. 비트캐스크)
- 값은 한 번의 디스크 탐색으로 읽어올 수 있어 메모리보다 더 많은 공간 사용 가능

**제한 사항**

- 메모리에 모든 키를 저장해야 하므로 키가 너무 많으면 문제
- 범위 질의에 비효율적

### 세그먼트와 컴팩션

파일에 계속 추가만 하면 디스크 공간이 부족해지므로 다음 방식으로 해결

- 특정 크기에 도달시 세그먼트 파일을 닫고 새로운 세그먼트 파일에 이후 쓰기 수행
- 컴팩션: 로그에 중복된 키를 제거하고 각 키의 최신 값만 유지
- 컴팩션 이후 여러 세그먼트를 병합해 새로운 파일로 생성
    - 세그먼트 이후에는 변경이 안됨
- 세그먼트 병합과 컴팩션은 백그라운드에서 수행되며 완료 후 이전 파일은 삭제
    - 이전 세그먼트 파일을 사용해 읽기, 쓰기 요청 처리 정상적으로 가능

### 실제 구현 시 고려사항

| 항목 | 내용 |
| --- | --- |
| 파일 형식 | CSV보다 바이너리 형식이 적합 |
| 레코드 삭제 | 툼스톤(tombstone) 레코드를 추가하고 병합 시 삭제 처리
→ 데이터 파일에 특수한 삭제 레코드를 추가해야 키와 관련된 값 삭제 가능 |
| 고장 복구 | 세그먼트 해시 맵의 스냅숏을 디스크에 저장해 빠른 복구 |
| 부분 쓰기 | 체크섬으로 손상된 로그를 탐지 및 무시 |
| 동시성 | 쓰기는 단일 스레드, 읽기는 다중 스레드로 처리 |

### 추가 전용 로그가 좋은 설계인 이유

- 순차 쓰기이므로 무작위 쓰기보다 빠름
- 불변 상태인 세그먼트 덕분에 동시성 및 고장 복구가 단순해짐
- 세그먼트 병합으로 데이터 조각화 방지

## SS테이블과 LSM 트리

### SS테이블 (Sorted String Table)

세그먼트 파일 내 키-값 쌍을 키 기준으로 정렬해 저장하는 방식.

**해시 색인 대비 장점**

- 효율적인 병합: 세그먼트 파일 합칠 때 각 파일의 첫 번째 키를 비교하여 가장 낮은 키를 복사
    - 정렬된 파일을 병합 정렬 방식으로 빠르게 합칠 수 있음
- 색인 감소: 모든 키가 아닌 일부 키의 위치만 메모리에 유지해도 됨
    - 찾으려는 키가 어떤 키들 사이인지 확인 후 그 사이만 스캔
- 압축률 향상: 인접한 유사 키들을 함께 압축해 디스크 공간 절약
    - 읽기 요청시 디스크에서 압축된 블록 전체를 한 번에 읽어와 I/O 대역폭 사용 줄임

### 쓰기 방식

1. 쓰기 요청 → 인메모리 균형 트리(**멤테이블**, ex. 레드 블랙 트리, AVL 트리)에 삽입
2. 멤테이블이 임곗값을 초과하면 SS테이블 파일로 디스크에 기록
    - 기록하는 동안 쓰기는 새로운 멤테이블 인스턴스에 기록
3. 읽기 시 멤테이블에서 키 찾음 → 디스크 최신 세그먼트에서 찾음 → 순차 탐색
4. 백그라운드에서 병합과 컴팩션 과정 수행

### LSM 트리 (로그 구조화 병합 트리)

SS테이블을 기반으로 정렬된 파일을 지속적으로 병합하는 키-값 저장소 엔진 라이브러리

- 레벨DB, 록스DB 등이 이 방식을 사용
- 루씬(엘라스틱서치나 솔라의 전문 검색 색인 엔진)도 유사한 구조 사용

#### 성능 최적화

- **블룸 필터**: 존재하지 않는 키에 대한 불필요한 디스크 읽기를 방지하는 메모리 효율적 자료구조
    - 집합 내용을 근사한 메모리 효율적 데이터 구조

#### 컴팩션 전략

| 전략 | 설명 | 사용 사례 | 특징 |
| --- | --- | --- | --- |
| 크기 계층 컴팩션 | 작고 새로운 SS테이블을 크고 오래된 SS테이블에 병합 | HBase, 카산드라 | 장점: 쓰기 최적화,
단점: 읽기 성능 저하 |
| 레벨 컴팩션 | 키 범위를 작은 SS테이블로 나누고 레벨별로 점진적 컴팩션 | 레벨DB, 록스DB, 카산드라 | 장점: 디스크 공간 효율 좋음
단점: 쓰기 부하 발생 |

## B트리

가장 널리 사용되는 색인 구조. 고정 크기 블록이나 페이지를 단위로 데이터를 읽고 씀

- 트리는 루트 페이지에서 시작해 하위 페이지 참조를 통해 원하는 키에 도달
- 리프 페이지에서 개별 키와 값을 저장
- 분기 계수: 한 페이지에서 하위 페이지를 참조하는 수

**갱신 방식**: 리프 페이지를 찾아 값을 변경한 후 디스크에 덮어씀 (페이지 위치 불변)

**페이지 분할**

- 새로운 키 포함하는 범위의 페이지 찾아 키와 값 추가
- 새 키 삽입 시 공간이 부족하면 페이지를 둘로 나누고 상위 페이지 갱신

### 쓰기 전 로그 (WAL, 재실행 로그)

B트리 변경 사항을 트리에 적용하기 전 별도의 추가 전용 파일에 기록

- 고장 이후 일관성 있는 상태로 B트리 복구하는 데 사용
- **래치**: 트리 데이터 구조 보호를 위한 동시성 제어

### B트리 최적화

- **쓰기 시 복사**: 변경된 페이지를 다른 위치에 기록하고 상위 페이지가 새 위치를 가리키도록 갱신
- **키 축약**: 페이지에 키 전체 대신 축약된 형태를 저장해 트리 깊이 감소
- **연속 배치**: 리프 페이지를 디스크 상에 연속된 순서로 배치
    - 트리 커질수록 순서 유지 어려움
- **포인터 추가**: 각 리프 페이지가 형제 페이지를 참조해 상위 페이지 이동 없이 키 스캔 가능
- **프랙탈 트리**: 디스크 탐색을 줄이기 위해 로그 구조화 개념을 일부 도입한 B트리 변형

## B트리 vs LSM 트리 비교

| 항목 | B트리 | LSM 트리 |
| --- | --- | --- |
| 읽기 성능 | 빠름 | 상대적으로 느림 (각 컴팩션 단계에 있는 여러 세그먼트 확인 필요) |
| 쓰기 성능 | 상대적으로 느림 | 빠름 (순차 쓰기) |
| 쓰기 횟수 | 2회 (쓰기 전 로그, 트리 페이지) | 컴팩션과 병합으로 여러 번 |
| 키 중복 | 한 위치에만 존재 | 여러 세그먼트에 복사본 존재 가능 |
| 트랜잭션 | 트리에 직접 잠금 포함, 강력한 트랜잭션 보장 | 상대적으로 복잡 |
| 압축률 | 파편화로 인해 낮음 | 높음 |
| 구현 성숙도 | 높음 | 상대적으로 낮음 |

SSD는 블록 덮어쓰기 횟수가 제한되므로 쓰기 증폭 중요

→ LSM 트리는 순차 쓰기와 높은 압축률 덕분에 SSD 환경에서 유리

### LSM 트리 장점

**LSM 트리는 보통 B 트리보다 쓰기 처리량을 높게 유지할 수 있다**

- 여러 페이지를 덮어 쓰는 것이 아니라 순차적으로 컴팩션된 SS테이블 파일을 쓰기 때문
    - 자기 하드드라이브는 순차 쓰기가 임의 쓰기보다 훨씬 빠름
- 압축률이 좋아 디스크에 더 적은 파일 생성
    - B트리는 파편화로 인해 사용하지 않는 디스크 공간이 남음
    - LSM 트리는 주기적으로 SS테이블을 다시 기록해 파편화를 제거하므로 저장소 오버헤드가 낮음 (특히 레벨 컴팩션 사용 시)
- 대부분의 SSD 펌웨어는 임의 쓰기를 순차 쓰기로 전환하기 위해 내부적으로 로그 구조화 알고리즘을 사용함
    - 낮은 쓰기 증폭과 파편화 감소는 SSD에서 훨씬 유리
    - 데이터를 더 밀집해 표현하면 I/O 대역폭 내에서 더 많은 읽기와 쓰기 요청이 가능

### LSM 트리 단점

- 컴팩션 과정이 읽기/쓰기 성능에 영향을 줄 수 있음
    - 저장소 엔진은 동시 접근 영향이 없게 컴팩션을 수행하려고 함 → 디스크 자원의 한계 → 컴팩션 연산이 끝날 때 까지 요청 대기해야하는 상황이 존재
- 컴팩션 설정이 미흡하면 유입 쓰기 속도를 따라가지 못해 세그먼트 수가 계속 증가

## 색인 종류

### 보조 색인

관계형 데이터베이스에서 다양한 컬럼에 보조 색인을 생성 가능

| 종류 | 설명 | 특징 |
| --- | --- | --- |
| 클러스터드 색인 | 색인 안에 실제 로우 데이터를 직접 저장 | 읽기 성능 높지만 추가적인 저장소 필요, 쓰기 과정에서 오버헤드 → 트랜잭션 보장 강화 |
| 비클러스터드 색인 | 색인 안에 데이터의 주소만 저장(힙 파일 참조) | 키를 변경하지 않고 값만 갱신할때 효과적 |
| 커버링 색인 | 일부 컬럼을 색인 안에 저장하는 중간 방식 | 읽기 성능 높지만 추가적인 저장소 필요, 쓰기 과정에서 오버헤드 → 트랜잭션 보장 강화 |

### 다중 컬럼 색인

- **결합 색인**: 여러 필드를 단순 결합해 하나의 키로 만드는 방식
- **다차원 색인**: 지리 공간 데이터처럼 여러 차원을 동시에 질의할 때 사용
    - 표준 B트리/LSM 트리로는 위도와 경도를 동시에 범위 질의할 수 없음
    - 공간 채움 곡선(이차원 위치 → 단일 숫자 변환) 활용 후 B트리 색인 이용
    - R트리 같은 전문 공간 색인 사용

### 퍼지 색인

오타 처리 등 유사한 키를 검색할 때 사용

- 루씬은 특정 편집 거리 내의 단어를 검색 가능
- SS테이블과 같은 구조 이용
- 인메모리 색인으로 유한 상태 **오토마톤**(트라이와 유사) 사용
    - 접두사와 접미사를 공유해 색인에서 이 형태로 저장

## 인메모리 데이터베이스

램 가격 하락으로 전체 데이터셋을 디스크가 아닌 메모리에 보관하는 방식

**지속성 확보 방법**

- 특수 하드웨어 사용
- 디스크에 변경사항 로그 기록
- 디스크에 주기적 스냅숏 기록

**장점**

- 디스크 기반 색인으로 구현하기 어려운 데이터 모델 지원
- 디스크 중심 아키텍처의 오버헤드 없음

**안티 캐싱**

메모리가 부족할 때 최근에 사용하지 않은 데이터를 디스크로 내보내고 필요 시 다시 적재 → OS의 페이지 단위보다 개별 레코드 단위로 처리해 더 효율적으로 메모리를 관리

## 트랜잭션 처리와 분석 (OLTP vs OLAP)

### 비교

| 항목 | OLTP | OLAP |
| --- | --- | --- |
| 설명 | 사용자의 입력을 기반으로 레코드 삽입 및 갱신 | 많은 수의 레코드를 스캔해 레코드당 일부 칼럼만 읽어 집계 통계 계산 |
| 주요 읽기 패턴 | 질의당 적은 수의 레코드,
키 기준으로 가져옴 | 많은 레코드에 대한 집계 |
| 주요 쓰기 패턴 | 임의 접근, 사용자 입력을 낮은 지연 시간으로 기록 | 대규모 불러오기 또는 이벤트 스트림 |
| 주요 사용처 | 웹 애플리케이션을 통한 최종 사용자/소비자 | 의사결정 지원을 위한 내부 분석가 |
| 데이터 표현 | 데이터의 최신 상태 | 시간이 지나며 일어난 이벤트 이력 |
| 데이터셋 크기 | GB~TB | TB~PB |
| 병목 | 디스크 탐색 | 디스크 대역폭 |

### 데이터 웨어하우스

OLTP 시스템에 영향을 주지 않고 분석 질의를 수행하기 위한 별도의 읽기 전용 데이터베이스

- **ETL**: OLTP에서 추출 → 분석 친화적 스키마로 변환 → 웨어하우스에 적재

**스키마 설계**

- **별 모양 스키마**: 중심에 사실 테이블, 주변에 차원 테이블 배치
    - 사실 테이블
        - 특정 시각에 발생한 이벤트 → 페이지 뷰나 사용자 클릭에 해당
        - 로우에 개별 이벤트 담음 → 분석 유연성 극대화
        - 칼럼은 다른  테이블을 참조하는 외래키
- **눈꽃송이 모양 스키마**: 차원 테이블을 하위 차원으로 더 세분화한 변형된 템플릿

## 칼럼 지향 저장소

분석 질의는 보통 수백 개의 컬럼 중 4~5개만 접근

→ 칼럼 지향 저장소는 각 컬럼별로 같은 속성끼리 몰아서 하나의 파일로 저장

**장점**

- 같은 값이 반복되는 컬럼을 효율적으로 압축 (비트맵 부호화)
- 디스크로부터 메모리로 데이터 가져오는 대역폭 관련 병목 문제 해결
    - 필요한 컬럼만 디스크에서 읽어 I/O 감소
    - 벡터화 처리로 CPU 연산 효율 향상

**순서 정렬**

- 자주 사용하는 질의에 맞게 1차 정렬 키 선택 → 특정 로우만 스캔할 수 있어서 빠름
- 보조 정렬 키 사용 → 그룹화/필터링 하는 질의에 도움
- 정렬된 컬럼은 같은 값이 연속으로 반복되어 압축률이 높음
- 여러 장비에 복제 시 서로 다른 방식으로 정렬해(서로 다른 키를 가지는) 저장하면 질의에 적합한 버전을 선택할 수 있음

**쓰기**: 칼럼 지향 저장소는 읽기에 최적화되어 쓰기 어려움

→ LSM 트리를 활용해 인메모리 저장소로 이동해 정렬된 구조에 추가하고 디스크에 쓸 준비

→ 충분한 쓰기를 모으면 디스크 칼럼 파일에 병합하고 대량으로 새로운 파일에 기록

### 구체화 뷰

SUM, COUNT, AVG 등 자주 사용하는 집계 결과를 캐시해두는 방식

- 원본 데이터 변경 시 갱신 비용이 비싸므로 OLTP보다 데이터 웨어하우스에 적합
- 데이터 큐브(OLAP 큐브): 일반화된 구체화 뷰의 특별 사례, 다차원 집계를 미리 계산해 저장

## OLTP 저장소 엔진 관점

- 로그 구조화 관점: 파일에 추가와 오래된 파일의 삭제만 허용하고 한 번 쓰여진 파일 갱신 X(LSM 트리)
- 제자리 갱신 관점: 덮어쓰기 되는 고정 크기 페이지의 셋으로 디스크를 다룸 (B트리)
