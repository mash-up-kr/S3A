# 9장. 일관성과 합의

분산 시스템에서는 많은 것이 잘못될 수 있다. 가장 단순한 대응은 전체를 실패시키고 오류를 반환하는 것이지만, 이를 받아들일 수 없으면 내결함성이 필요하다. 일부 구성 요소에 결함이 있어도 서비스 전체는 올바르게 동작하게 만드는 것이다.

좋은 방법은 매번 직접 처리하는 대신 **유용한 보장을 제공하는 추상화**를 만들고 애플리케이션이 그 위에 기대게 하는 것이다. 가장 중요한 추상화가 **합의(consensus)** — 모든 노드가 한 가지에 동의하게 만드는 것이다.

합의가 필요한 대표 상황이 리더 선출이다. 단일 리더 복제에서 리더가 죽으면 남은 노드들이 합의로 새 리더를 뽑아야 한다. 이것이 어긋나 두 노드가 동시에 자신을 리더로 믿는 상황이 스플릿 브레인(split brain)이며, 보통 데이터 손실로 이어진다. 올바른 합의 구현은 이를 막는다.

이 장의 흐름: 가장 강한 일관성 모델인 **선형성** → 분산 시스템의 **이벤트 순서화** → 분산 트랜잭션의 **원자적 커밋**.

## 일관성 보장

복제 DB에서는 노드마다 쓰기 도착 시점이 다르다. 그래서 두 노드를 동시에 보면 서로 다른 데이터가 보일 수 있다. 복제 방식과 무관하게 발생한다.

대부분의 복제 DB가 최소한 제공하는 보장은 **최종적 일관성**이다. 쓰기를 멈추고 충분히 기다리면 결국 모든 읽기가 같은 값을 반환한다는 뜻이다. 불일치는 일시적이며 스스로 해소된다.

그러나 이는 약한 보장이다. *언제* 수렴하는지에 대해서는 아무것도 보장하지 않는다. 단일 스레드 프로그램의 변수 동작과 크게 달라 애플리케이션 개발자에게 까다롭다. 이 장에서는 더 강한 일관성 모델을 다룬다. 강한 보장은 성능이나 내결함성에서 손해를 볼 수 있으나, 올바르게 사용하기 쉽다는 장점이 있다.

## 선형성

복제본이 여러 개라도 **하나뿐인 것처럼** 보이게 만들면 시스템이 단순해진다. 이것이 선형성의 기본 아이디어다. 데이터 복사본이 하나만 있고 모든 연산이 원자적인 것처럼 보이게 한다. 따라서 읽은 값은 항상 가장 최근 값이며, 뒤처진 캐시나 복제본 값이 아니다. **선형성은 곧 최신성 보장이다.**

### 시스템에 선형성을 부여하는 것은 무엇인가

연산 표기

- `read(x) ⇒ v` : x를 읽어 값 v를 반환
- `write(x, v) ⇒ r` : x를 v로 설정 요청, 응답 r 반환
- `cas(x, v_old, v_new) ⇒ r` : 원자적 compare-and-set. 현재 값이 `v_old`면 `v_new`로 설정, 아니면 그대로 두고 오류 반환

네트워크 지연 때문에 클라이언트는 요청이 정확히 언제 처리됐는지 모른다. "요청을 보낸 시각 ~ 응답을 받은 시각" 사이라는 것만 안다.

x가 0인 상태에서 c가 1로 바꾸는 쓰기를 하는 동안 a, b가 반복해 읽는 상황의 규칙

- 쓰기 시작 전 완료된 읽기 → 0
- 쓰기 완료 후 시작한 읽기 → 1
- 쓰기와 겹치는 읽기 → 0 또는 1 (그 시점에 쓰기 효과가 났는지 알 수 없음)

겹치는 읽기가 둘 다 가능하다는 점 때문에, 자칫 쓰기 진행 중에 값이 0↔1을 오가는 것을 보게 될 수 있다. 이는 "복사본 하나"를 모방하는 시스템이 기대하는 동작이 아니다. 따라서 제약을 추가한다. **한 번 새 값이 읽히면, 이후 모든 읽기는 쓰기가 끝나지 않았더라도 새 값을 반환해야 한다.**

결국 선형성의 요구사항은 연산을 시간순으로 배열했을 때 결코 뒤로 가지 않는 것이다. 요청·응답 시점을 기록해 유효한 순차 순서로 배열되는지 확인하면 선형성 여부를 테스트할 수 있다.

### 선형성 대 직렬성

이름이 비슷해 혼동되지만 다른 개념이다.

|  | 선형성 | 직렬성 |
| --- | --- | --- |
| 대상 | 레지스터 하나의 읽기/쓰기 | 여러 객체를 다루는 트랜잭션들의 격리 |
| 핵심 | 최신성 보장 | 트랜잭션들이 어떤 직렬 순서로 실행된 것처럼 동작 |
| 한계 | 연산을 트랜잭션으로 묶지 않아 쓰기 스큐는 막지 못함 | 실제 실행 순서와 달라도 무방 |

둘 다 제공하면 **엄격한 직렬성(strict serializability)**이라 한다. 2단계 잠금이나 실제 직렬 실행 기반 구현은 보통 선형적이나, 직렬성 스냅숏 격리(SSI)는 선형적이지 않다.

### 선형성에 기대기

선형성이 시스템의 올바른 동작에 중요한 요구사항이 되는 영역이 있다.

- **잠금과 리더 선출** : 노드들이 시작 시 잠금 획득을 시도하고 성공한 노드가 리더가 된다. 이 잠금은 선형적이어야 하고 모든 노드가 잠금 소유자에 동의해야 한다. ZooKeeper, etcd가 합의 알고리즘으로 구현한다. Oracle RAC는 디스크 페이지마다 잠금을 쓰며, 트랜잭션 중요 경로에 있어 전용 클러스터 네트워크를 사용한다.
- **유일성 제약** : 동일 사용자명·파일을 동시에 생성할 때 한 명만 성공시키려면 선형성이 필요하다. 사용자명에 잠금을 거는 것, 또는 원자적 CAS와 유사하다.
- **채널 간 타이밍 의존성** : 웹서버가 이미지를 파일 저장소에 올리고 동시에 메시지 큐로 리사이즈 모듈에 작업을 지시할 때, 저장소가 선형적이지 않으면 큐가 저장소 내부 복제보다 빨라 리사이즈 모듈이 아직 도착하지 않은 이미지를 읽는 경쟁 조건이 생긴다. 통신 채널이 둘이기 때문에 발생한다.

### 선형성 시스템 구현하기

복사본을 진짜 하나만 쓰면 선형적이지만 내결함성이 없다(노드 장애 시 데이터 손실). 그래서 복제를 쓰는데, 방식마다 선형성 여부가 다르다.

| 복제 방식 | 선형성 |
| --- | --- |
| 단일 리더 | 리더·동기 팔로워 읽기는 가능. 단 스냅숏 격리 설계나 동시성 버그면 깨짐. "누가 리더인지" 확실히 안다는 전제 필요 |
| 합의 알고리즘 | 가능. 스플릿 브레인·복제 지연 방지 장치가 포함되어 안전하게 선형 저장소 구현 |
| 다중 리더 | 불가. 여러 노드가 동시에 쓰고 비동기로 복제 → 충돌 발생 |
| 리더 없는 복제 | 대체로 불가. 시계 기반 LWW는 시계 스큐로 비선형, 느슨한 정족수도 깨뜨림 |

**선형성과 정족수** — 리더 없는 복제는 "엄격한 정족수면 선형적"이라 주장되기도 하나, 정족수 설정과 정의에 따라 항상 참은 아니다. 엄격한 정족수 읽기·쓰기는 선형적으로 보이지만 네트워크 지연 변동이 크면 경쟁 조건이 생긴다. 성능을 희생하면 다이나모 스타일 정족수도 선형적으로 만들 수 있다 — 읽는 쪽은 결과 반환 전에 동기 읽기 복구를 수행하고, 쓰는 쪽은 쓰기 전에 정족수에서 최신 상태를 읽는다. 리악은 성능 때문에 동기 읽기 복구를 하지 않고, 카산드라는 읽기 복구를 기다리나 LWW라 같은 키에 동시 쓰기가 들어오면 선형성을 잃는다. 또한 이 방식으로는 읽기·쓰기만 가능하고 CAS는 불가하다(CAS는 합의가 필요).

### 선형성의 비용

데이터센터 두 곳 사이 네트워크가 끊긴 상황. 각 DC 내부망과 클라이언트 접근은 정상이나 DC끼리는 통신 불가다.

- 다중 리더 : 각 DC가 계속 동작하고 쓰기는 큐에 쌓였다가 복구 시 전달된다. 가용하지만 비선형이다.
- 단일 리더 : 리더가 한쪽 DC에 있어야 하고 모든 쓰기·선형 읽기는 그쪽으로 가야 한다. 팔로워 DC 클라이언트는 끊긴 동안 쓰기·선형 읽기가 불가하다. 팔로워에서 읽을 수는 있으나 뒤처진 데이터라 비선형이다.

#### CAP 정리

이는 단일/다중 리더만의 문제가 아니다. 선형성을 요구하는 순간, 분단이 생기면 일부 복제 서버는 복구를 기다리거나 오류를 반환해야 한다. 반대로 선형성을 요구하지 않으면 각 복제 서버가 독립적으로 요청을 처리해 가용성을 유지한다. 따라서 **선형성이 필요 없는 애플리케이션이 네트워크 문제에 더 강인하다.**

CAP는 "분단 시 일관성과 가용성 중 하나를 선택해야 한다"로 이해하는 것이 좋다. 분단을 덜 겪으려면 더 신뢰성 있는 네트워크가 필요하지만, 어느 시점에는 선택이 불가피하다. CAP는 공유 저장소 클러스터에 쏠려 있던 시선을 대규모 웹서비스에 적합한 비공유 시스템 쪽으로 넓힌 의의가 있다.

#### 선형성과 네트워크 지연

현실에서 진짜 선형적인 시스템은 드물다. 멀티코어 CPU의 RAM조차 선형적이지 않다. 한 코어가 메모리에 쓴 값을 다른 코어가 곧바로 읽어도 그 값이 보장되지 않는다. 코어마다 캐시·저장 버퍼가 있고 메인 메모리 반영이 비동기이기 때문이다. 여기서 선형성을 제거한 이유는 내결함성이 아니라 **성능**이다. 따라서 멀티코어 메모리 모델을 CAP로 정당화하는 것은 타당하지 않다.

선형성을 포기한 분산 DB들도 마찬가지로 대부분 내결함성이 아니라 성능 때문이다. 더 빠른 선형 알고리즘이 따로 있는 것은 아니지만, 완화된 일관성 모델은 훨씬 빠를 수 있다.

## 순서화 보장

### 순서화와 인과성

순서화는 **인과성**을 보존하는 데 도움을 준다. 인과적 의존성의 예

- 질문과 답변 : 관찰자는 질문을 본 후 답변을 봐야 한다.
- 어떤 쓰기가 다른 쓰기를 추월하면 일부 복제 서버 관점에서 존재하지 않는 로우를 갱신한 것처럼 보인다. 로우는 갱신 전에 생성돼야 한다.
- 동시 쓰기 감지의 "이전 발생(happens-before)" 관계.
- 스냅숏 격리의 일관된 스냅숏 : 스냅숏에 답변이 있으면 그 질문도 포함돼야 한다.
- 쓰기 스큐 : 호출 대기에서 빠지는 동작은 현재 누가 대기 중인지 관찰한 것에 의존한다.

인과성이 부과한 순서를 지키는 시스템을 **인과적으로 일관적**이라 한다.

### 인과적 순서가 전체 순서는 아니다

선형성 시스템에서는 모든 연산의 **전체 순서**를 정할 수 있다. 복사본이 하나뿐인 것처럼 동작하고 모든 연산이 원자적이면 어떤 두 연산이든 선후를 말할 수 있다.

인과성은 다르다. 인과 관계가 있는 두 이벤트는 순서가 있지만 동시에 발생한 둘은 비교할 수 없다. 즉 인과성은 전체 순서가 아니라 **부분 순서**를 정의한다.

### 선형성은 인과적 일관성보다 강하다

선형성은 인과성을 내포한다. 그러나 선형성을 만드는 성능 손해 없이도 인과적 일관성은 만족시킬 수 있다. 인과적 일관성은 네트워크 지연으로 느려지지 않고 장애가 발생해도 가용한, 그 조건에서 가장 강한 모델이다. 선형성이 필요해 보이는 시스템 중 상당수는 실제로는 인과적 일관성만 필요하며, 이는 더 효율적으로 구현된다.

### 인과적 의존성 담기

인과성을 유지하려면 어떤 연산이 다른 연산보다 먼저인지 알아야 한다(부분 순서). 복제 서버가 연산을 처리할 때 인과적으로 앞선 연산이 모두 처리됐음을 보장할 수 있어야 한다. 리더 없는 저장소가 갱신 손실 방지를 위해 같은 키 동시 쓰기를 검출하듯, 인과적 의존성은 단일 키를 넘어 DB 전체에 걸친 의존성을 추적한다. 이를 위해 버전 벡터를 일반화한다.

### 일련번호 순서화

인과성은 중요하지만 모든 인과 의존성을 실제로 추적하는 것은 비실용적이다. 대신 **일련번호·타임스탬프**로 순서를 부여한다. 논리적 시계는 연산마다 증가하는 카운터로 일련번호를 만들어 전체 순서를 제공한다. 이를 인과성에 일관적으로 생성할 수도 있다. 전체 순서는 인과 정보를 모두 담되 필요 이상으로 순서를 부과한다.

단일 리더 DB에서는 복제 로그가 인과 일관적인 전체 순서를 정의한다. 리더가 연산마다 카운터를 올려 단조 증가 일련번호를 붙이면 된다.

#### 비인과적 일련번호 생성기

단일 리더가 없을 때의 일련번호 생성 방법

- 한 노드는 홀수, 다른 노드는 짝수만 생성
- 일련번호 비트 일부에 노드 ID 포함
- 각 연산에 일 기준 시계 타임스탬프 부여
- 일련번호 블록을 미리 할당

모두 고유하고 근사 증가하지만 **인과성에 비일관적**이다. 노드별 처리 속도가 달라 선후를 알 수 없고, 물리 시계는 시계 스큐를 타며, 블록 할당은 인과적으로 나중 연산이 더 작은 번호를 받는 역전이 생긴다.

#### 램포트 타임스탬프

인과 일관적인 전체 순서를 만드는 간단한 방법이다. 각 노드는 고유 ID를 갖고 처리한 연산 수를 카운터로 유지하며, 램포트 타임스탬프는 **(카운터, 노드ID)** 쌍이다. 카운터가 겹쳐도 노드ID로 유일해진다. 물리 시계와 무관하나 전체 순서를 제공한다. 비교는 카운터가 큰 쪽이 크고, 같으면 노드ID가 큰 쪽이 크다.

핵심은 모든 노드·클라이언트가 지금까지 본 카운터 최댓값을 추적해 모든 요청에 포함시키고, 자기 카운터보다 큰 값을 받으면 즉시 그 값으로 올리는 것이다. 이로써 인과성이 보존된다.

버전 벡터와 비슷하나 목적이 다르다. 버전 벡터는 두 연산이 동시적인지 인과 의존인지 구별할 수 있으나, 램포트 타임스탬프는 항상 전체 순서를 강제해 그 구별이 불가하다. 대신 크기가 더 작다.

#### 타임스탬프 순서화로는 충분하지 않다

램포트 타임스탬프는 인과 일관 전체 순서를 정의하지만 현실 문제에는 충분하지 않다. 사용자명 유일성 보장이 그 예다. "타임스탬프가 낮은 쪽이 성공, 높은 쪽이 실패"는 사후에 승자를 가리는 데는 효과적이다.

그러나 노드가 사용자명 요청을 받고 즉시 성공/실패를 결정해야 할 때는 부족하다. 다른 노드가 동시에 더 낮은 타임스탬프로 같은 사용자명을 처리 중이 아님을 확신하려면 모든 노드의 상태를 확인해야 하기 때문이다. 핵심은 **전체 순서가 모든 연산을 모은 후에야 드러난다**는 점이다. 유일성 제약을 구현하려면 순서가 있는 것만으로는 부족하고, 그 순서가 *언제 확정되는지*도 알아야 한다.

### 전체 순서 브로드캐스트

모든 노드에서 연산의 전체 순서를 동일하게 합의하기는 까다롭다. 단일 리더는 리더 한 대의 CPU 코어에서 연산을 차례로 배열해 전체 순서를 정하지만, 처리량이 리더 한 대의 한계를 넘을 때의 확장과 리더 장애 복구가 문제로 남는다. 이 문제를 **전체 순서 브로드캐스트(원자적 브로드캐스트)**라 한다.

보통 노드 간 메시지 교환 프로토콜로 기술되며 두 안전성 속성을 항상 만족해야 한다.

1. **신뢰성 있는 전달** : 메시지는 손실되지 않으며, 한 노드에 전달되면 모든 노드에 전달된다.
2. **전체 순서가 정해진 전달** : 모든 노드에 같은 순서로 전달된다.

노드·네트워크 결함이 있어도 두 속성이 보장돼야 한다. ZooKeeper, etcd가 실제로 구현한다.

#### 전체 순서 브로드캐스트 사용하기

- **DB 복제** : 모든 메시지가 쓰기를 나타내고 모든 복제 서버가 같은 순서로 처리하면 일관된 상태를 유지한다(상태 기계 복제).
- **직렬성 트랜잭션** : 메시지가 결정적 트랜잭션(스토어드 프로시저)이고 모두 같은 순서로 처리하면 파티션·복제본이 일관성을 유지한다.
- **로그 생성** : 메시지 전달이 곧 로그에 append하는 것과 같다. 순서가 전달 시점에 고정된다.
- **펜싱 토큰 잠금 서비스** : 잠금 요청을 로그에 append하면 등장 순서대로 단조 증가 번호가 붙어 펜싱 토큰 역할을 한다.

#### 전체 순서 브로드캐스트를 사용해 선형성 저장소 구현하기

전체 순서 브로드캐스트는 비동기다. 고정된 순서로 신뢰성 있게 전달되지만 언제 전달될지는 보장하지 않는다. 반면 선형성은 최신성 보장이다. 결이 다르지만 서로 변환된다.

사용 가능한 사용자명마다 CAS 레지스터가 있는 선형 저장소를 가정한다. 점유할 사용자명을 메시지로 로그에 append하고, 로그를 읽으며 자기 메시지가 돌아오기를 기다린다. 해당 사용자명의 첫 번째 메시지가 자신의 것이면 성공(커밋), 다른 사용자의 것이면 어보트한다. 이것은 선형성 쓰기만 보장한다. 읽기까지 선형적으로 하려면, 로그에 메시지를 넣고 돌아왔을 때 읽거나, 최신 로그 위치를 선형적으로 얻어 그 지점까지 전달된 뒤 읽거나, 동기 갱신으로 최신이 보장된 복제 서버에서 읽는다.

#### 선형성 저장소를 사용해 전체 순서 브로드캐스트 구현하기

원자적 `increment-and-get`(또는 CAS)이 지원되는 선형 레지스터를 가정한다. 보낼 메시지마다 이 연산으로 번호를 받아 붙이고, 메시지를 모든 노드에 보내면 수신자는 번호 순서대로 전달한다. 램포트 타임스탬프와 달리 이 번호들은 **틈 없는 순열**이라, 메시지 4까지 전달하고 6을 받으면 5를 기다려야 함을 안다. 이 선형 정수는 장애가 없으면 노드 하나의 변수로 쉽게 만들 수 있으나, 네트워크 단절과 장애 후 값 복구를 제대로 처리하려면 결국 합의 알고리즘에 도달한다.

정리하면 **선형성 CAS 레지스터, 전체 순서 브로드캐스트, 합의는 모두 서로 동등함이 증명된다.**

## 한눈에 정리

- **선형성** : 최신성 보장, 복사본 하나처럼 동작. 강력하지만 느리고 네트워크 분단에 약함.
- **CAP** : 분단 시 일관성과 가용성 중 택일. 선형성이 필요 없는 앱이 더 강인.
- **인과적 일관성** : 분단에도 가용한, 그 조건에서 가장 강한 모델. 선형성이 필요해 보여도 실제로는 이것으로 충분한 경우가 많음.
- **램포트 타임스탬프** : 인과 일관 전체 순서를 저비용으로 제공. 단, 순서가 언제 확정되는지는 알려주지 못함.
- **전체 순서 브로드캐스트** : 그 약점을 해결. 선형성 CAS = 전체 순서 브로드캐스트 = 합의, 셋은 동등.
- **합의 / Raft** : 스플릿 브레인 없이 리더 선출·원자적 커밋을 안전하게 수행. 단, 과반수 생존 필요.
