# 7장 트랜잭션
7장 전반: ACID, 격리 수준(Read Committed, 스냅숏 격리·MVCC), 경쟁 조건 이상 현상(더티 읽기·더티 쓰기, 읽기 스큐, 갱신 손실), 갱신 손실 방지(원자적 쓰기, 명시적 잠금, Compare-and-Set)

## 트랜잭션

데이터베이스에 접속하는 애플리케이션에서 프로그래밍 모델을 단순화하려는 목적으로 만듦 → 애플리케이션에서 잠재적인 오류 시나리오와 동시성 문제를 무시할 수 있음 

데이터베이스에서 이러한 일을 맡아줌(안전성 보장)

데이터베이스에서 사용하는 트랜잭션 알고리즘

- 주로 동시성 제어 부분을 다루고 발생할 수 이쓴 다양한 종류의 경쟁 조건과 데이터베이스에서 커밋후 읽기, 스냅숏 격리, 직렬성과 같은 격리 수준을 어떻게 구현하는지 설명
- 트랜잭션은 성능과 고가용성/신뢰성 사이에서 트레이드오프가 존재한다

## ACID

데이터베이스에서 트랜잭션이 제공하는 안정성 보장

### 원자성(Atomicity)

동시성과 관련 없음 → 여러 프로세스가 동시에 같은 데이터에 접근하려고 할때 생기는 일은 격리성에서 다룸

클라이언트가 쓰기 작업 몇개 실행하다가 일부만 처리된 후 결함이 생겼을때 어떤 일이 생기는지 설명

- 여러 쓰기 작업이 하나의 원자적 트랜잭션으로 묶여 있을때 결함 때문에 커밋될 수 없으면 어보트되고 지금까지 실행한 쓰기를 무시하거나 취소한다
    - 어보트되면 이 트랜잭션이 어떤 것도 변경하지 않았음을 알 수 있음

### 일관성(Consistency)

ACID에서는 데이터베이스가 좋은 상태에 있어야 한다는 것의 애플리케이션에 특화된 개념

= 항상 진실이어야하는 데이터에 관한 불변식이 있다는 것

데이터베이스는 불변식을 위반하는 잘못된 데이터를 쓰지 못하도록 막을 수는 없다(즉 불변식 개념에서 일관성 아이디어가 나온 것은 맞으나 트랜잭션을 올바르게 정의하는 것은 애플리케이션의 책임이다) (외래 키 제약 조건, 유일성 제약 조건 등이 가능하긴 하나 애플리케이션에서 데이터가 유효한지 아닌지를 정의하고 데이터베이스는 데이터를 저장하기만 할뿐)

**원자성, 격리성, 지속성은 데이터베이스의 속성이지만 일관성은 애플리케이션의 속성이다**

**→ 실제 ACID개념에 속하지않음**

### 격리성(Isolation)

여러 클라이언트에서 동일한 데이터베이스 레코드에 접근하면 동시성 문제가 생긴다

동시에 실행되는 트랜잭션은 서로 격리된다는 것을 의미 

**직렬성(serializable)**(각 트랜잭션이 전체 데이터베이스에서 실행되는 유일한 트랜잭션인 것처럼 동작할 수 있다는 의미)이 이상적인 목표

- 여러 트랜잭션 동시에 실행되더라도 트랜잭션이 커밋됐을 때 결과가 순차적으로 실행했을 때와 동일하도록 보장
- 직렬성 격리는 성능 손해가 존재하므로 보통은 직렬성보다는 격리 수준이 낮은 스냅숏 격리를 구현

### 지속성(Durability)

트랜잭션이 성공적으로 커밋됐다면 데이터베이스가 죽어도 트랜잭션에 기록한 모든 데이터는 손실되지 않는다

- 단일 노드 데이터베이스: 비휘발성 저장소에 기록 & 복구할 수 있게 쓰기 전 로그 등 존재
- 복제 기능 데이터베이스: 데이터가 성공적으로 다른 노드에 복사됨

지속성을 보장하기 위해서는 트랜잭션이 성공적으로 커밋됐다고 보고하기 전에 쓰기나 복제가 완료될 때까지 기다려야함

**데이터베이스가 직접 지속성을 보장함**

|  | 테이프/아카이브 | 디스크/SSD |
| --- | --- | --- |
| 장점 | 장기 보관, 물리적으로 안전 | 빠른 접근, 즉각 복구 가능 |
| 단점 | 복구 느림, 접근 불편 | 화재/침수 등 물리적 사고에 취약 |
- 현대 DB는 아카이브와 디스크 저장 방식 두개를 함께 사용

```
SSD/디스크        → 빠른 읽기/쓰기, 즉각 복구
+ WAL 로그        → 장애 직후 복구
+ 원격 복제       → 데이터센터 단위 장애 대비
+ 아카이브 백업   → 장기 보관, 최후의 수단
```

## 단일 객체 연산과 다중 객체 연산

ACID에서 원자성과 격리서은 클라이언트가 한 트랜잭션 내에서 여러 번의 쓰기를 하면 데이터베이스가 어떻게 해야하는지 서술함

- 원자성: 오류가 발생하면 트랜잭션은 어보트되어야하고 그전까지 쓰인 내용 전부 폐기. 데이터베이스는 전부 반영되거나 아무것도 반영되지 않는 것을 보장함으로써 부분 실패를 걱정할 필요가 없게 함
- 격리성: 동시에 실행되는 트랜잭션은 서로를 방해하면 안됨. 한 트랜잭션이 여러 번 쓰면 다른 트랜잭션은 그 내용을 전부 볼 수 있든지 아무것도 못보든지 하나여야 함

### 다중 객체 쓰기

- 예시 1 — 격리성이 필요한 경우
    
    상황
    한 트랜잭션에서 새로운 메일을 쓰고, 안 읽은 메일함에도 쓰기를 추가해야 하는데
    그 전에 다른 트랜잭션에서 메일함을 조회 → 둘 사이에 일관성이 깨진 중간 상태를 읽어버림
    
    문제 — 더티 읽기 (Dirty Read)
    트랜잭션이 다른 트랜잭션에서 썼으나 아직 커밋되지 않은 데이터를 읽는 경우
    
    격리성의 역할
    내용을 모두 보거나 모두 못 보게 하고, 일관성이 깨진 중간 지점을 보지 않게 함
    
- 예시 2 — 원자성이 필요한 경우
    
    상황
    새로운 메일을 쓰고 안 읽은 메일함에 쓰기를 하기 전에 오류가 발생하는 경우
    
    원자성의 역할
    트랜잭션에서 이전에 발생한 쓰기를 전부 취소(롤백)
    

#### 동일한 트랜잭션에 속하는지 알아내는 법

어떤 읽기 연산과 쓰기 연산이 동일한 트랜잭션에 속하는지 알아내기 위해 

- 관계형 데이터베이스: 클라이언트와 데이터베이스 서버 사이의 TCP 연결을 기반으로 함

TCP 연결 하나 = 트랜잭션 하나

```
클라이언트 ──TCP 연결──▶ DB 서버

BEGIN TRANSACTION;       ← "트랜잭션 시작"
INSERT INTO 메일 ...     ← 같은 연결에서 온 쿼리
UPDATE 안읽은메일함 ...  ← 같은 연결에서 온 쿼리 → 같은 트랜잭션!
COMMIT;                  ← "트랜잭션 끝"
```

→ 클라이언트가 COMMIT 요청을 보냈지만 서버가 커밋 여부 확인 전 연결이 끊기면 클라이언트는 트랜잭션이 커밋됐는지 아닌지 알 수 없다 → 트랜잭션 관리자는 특정 TCP 연결에 엮이지 않은 고유한 트랜잭션 식별자를 사용해서 연산을 묶을 수 있다

- Q. 고유한 트랜잭션 식별자로 연산을 묶는다 뭔소리지
    
    TCP 연결 자체에 트랜잭션을 묶지 말고 **별도의 트랜잭션 ID(XID)** 를 붙이자
    
    - 어떤 트랜잭션 아이디 트랜잭션 커밋 잘 됐니? 물었을때 됐다 아니다 확인 쉬움
- 비관계형 데이터베이스는 동시에 쓰는 연산이 있어도 하나의 트랜잭션을 보장하지 못함
    
    ```
    // 예시: MongoDB 여러 문서 동시 쓰기
    db.bulkWrite([
      { insertOne: { 메일 } },
      { updateOne: { 안읽은메일함 } }
    ])
    ```
    
    - Q. 비관계형 데이터베이스에서 어떻게 트랜잭션 보장
        
        1. 애플리케이션 레벨에서 직접 처리
        
        - 실패하면 개발자가 직접 롤백 로직 짜야함 → 코드가 복잡해짐
        
        2. 단일 문서로 모델링
        
        - 메일 + 안읽은메일수를 하나의 문서에 넣어버림 → 다중 객체 문제 자체를 없애버림
        
        3. 최근엔 트랜잭션 지원 추가
        
        - MongoDB 4.0+, DynamoDB Transactions 등 → 비관계형 DB들이 트랜잭션을 지원하기 시작함
        - 관계형 DB처럼 BEGIN ~ COMMIT 지원

### 단일 객체 쓰기

객체 하나 쓰는 중 중간에 연결이 끊기게 되면 절반만 저장한 상태가 되는 문제

- **원자성:** 저장 중 실패 → 이미 쓴 것도 전부 롤백→ 절반만 저장된 상태 없음
    - 장애 복구용 로그로 구현
- **격리성:** 20KB 저장 중일 때 다른 트랜잭션이 읽으면 → 이전 값(저장 전) 또는 새 값(저장 후)만 보임 → 중간에 10KB만 저장된 상태는 안 보임
    - 객체에 잠금 사용해 동시에 한 스레드만 객체에 접근하도록

### read-modify-write 문제와 해결책

- 문제: 값을 바꾸려면 보통 3단계를 거침
1. **read** — 현재 값 읽기 (조회수 = 10)
2. **modify** — 값 수정 (10 + 1 = 11)
3. **write** — 다시 쓰기 (조회수 = 11)

동시에 두 트랜잭션이 같은 값을 읽고 수정하면?

```
트랜잭션 A: 읽기(10) → 수정(11) → 쓰기(11)
트랜잭션 B: 읽기(10) → 수정(11) → 쓰기(11)
결과: 11  ← 12가 되어야 함
```

---

#### 해결책 1 — 원자적 증가 연산

DB가 read-modify-write를 **한방에** 처리

```sql
UPDATE 게시글 SET 조회수 = 조회수 + 1 WHERE id = 1
```

- 중간에 다른 트랜잭션이 끼어들 틈 없음
- 애플리케이션이 아니라 **DB가 직접 처리**

#### 해결책 2 — Compare-and-Set (CAS)

"내가 읽었을 때 값이랑 지금도 같으면 써라"

```sql
UPDATE 게시글
SET 조회수 = 11
WHERE id = 1 AND 조회수 = 10
```

- 누군가 중간에 바꿨으면 조건 불일치 → **실패 후 재시도**
- 내가 읽은 사이에 다른 트랜잭션이 건드렸는지 감지

### 다중 객체 트랜잭션의 필요성

분산 데이터베이스에서 분산 트랜잭션의 구현

1. 어떤 테이블의 로우가 다른 테이블의 로우를 참조할 때 참조가 유효한 상태로 유지되도록 보장
    - 문제
        
        ```
        트랜잭션 A: 유저 삭제 중...
                    유저 테이블에서 철수 삭제
                                                ← 이 타이밍에
                                                트랜잭션 B: 주문 조회
                                                user_id: 1 참조했는데
                                                유저가 없음 
                    주문 테이블에서도 삭제
        ```
        
    - 해결
        
        ```
        BEGIN;
          유저 테이블에서 철수 삭제
          주문 테이블에서 관련 주문 삭제
        COMMIT;  ← 여기서 한번에 반영
        ```
        
2. 문서 데이터 모델에서 비정규화된 정보를 갱신할 때 동기화가 깨지는 것을 방지하기 위해 다중 객체 트랜잭션 필요
    - 문제
        
        ```
        유저 문서 name: "영희" 로 수정
                                          ← 여기서 오류 발생 💀
        주문 문서 user_name: 아직 "철수"
        ```
        
    - 해결
        
        ```
        BEGIN;
          유저 문서 name: "영희" 수정
          주문 문서 user_name: "영희" 수정
        COMMIT;  ← 둘 다 한번에 반영
        ```
        
3. 보조 색인이 있는 데이터베이스에서 값을 변경할 때 색인 갱신도 보장
    - 해결
    
    ```
    BEGIN;
      유저 테이블 city = "부산" 수정
      보조 색인 "서울"→1 삭제, "부산"→1 추가
    COMMIT;  ← 둘 다 한번에 반영
    ```
    

**원자성이 없으면 오류 처리가 복잡해지고 격리성이 없으면 동시성 문제가 발생할 수 있다**

트랜잭션의 핵심 기능은 오류가 생기면 어보트되고 안전하게 재시도 할 수 있다는 것

→ ACID 데이터베이스는 이 철학을 바탕으로 함

= 데이터베이스가 원자성, 격리성, 지속성 보장을 위반할 위험이 있으면 트랜잭션을 폐기함

**그러나 모든 시스템이 이 철학을 따르지는 않는다**

→ 리더 없는 복제 데이터스토어는 best effort 원칙을 기반으로 훨씬 더 많은 일을 한다

### Best Effort 철학

리더 없는 복제 DB(Cassandra, DynamoDB 등)는 다름

```
→ 할 수 있는 만큼 최대한 해
→ 잘못되면? 그냥 거기까지
→ 롤백? 없음
```

즉 **"완벽하진 않아도 최대한 많이 처리하자"** 는 철학

어보트된 트랜잭션을 재시도하는 것은 간단하고 효과적이지만 완벽하지 않음

- 오류가 과부하때문에 발생한 것이라면 재시도는 문제를 악화시킴 → 재시도 횟수를 제한하든지 exponential backoff 사용
- 일시적 오류(교착 상태, 격리성 위반, 일시적 네트워크 단절 등)만 재시도할 가치 있고 영구적인 오류는 재시도해도 소용이 없음
- 트랜잭션이 외부에 주는 side effect가 있으면(ex 이메일을 보낸다면 트랜잭션이 재시도될 때마다 이메일 다시 전송됨) 2단계 커밋이 도움됨
    - 2단계 커밋(2PC): 여러 참여자가 관련된 트랜잭션에서 모두 커밋되거나 모두 롤백 되도록 보장
        - **코디네이터** — 전체 과정을 조율하는 역할
        - **참여자** — DB, 외부 시스템 등 실제로 작업하는 것들
        
        ```
        1단계 — Prepare
        코디네이터: "다들 커밋할 수 있어?"
        참여자들:   각자 "응 가능" / "아니 못해" 응답
        
        2단계 — Commit or Rollback
        모두 OK  → 코디네이터: "다같이 커밋해!" → 전부 실행
        하나라도 NO → 코디네이터: "다같이 롤백해!" → 전부 취소
        ```
        

## 완화된 격리 수준

동시성 문제는 추론하기 어려움

데이터베이스는 오랫동안 트랜잭션 격리를 제공함으로써 애플리케이션 개발자들에게 동시성 문제를 감추려고 함

**직렬성 격리(Serializable Isolation)**: 여러 트랜잭션들이 직렬적으로 실행되는 것(동시성 없이 한 번에 트랜잭션 하나만 실행)과 동일한 결과가 나오도록 보장한다는 것

- 성능 비용이 있기 때문에 완화된 격리 수준을 사용하는 시스템들이 흔하다

### 커밋 후 읽기(Read Committed)

기본적인 수준의 트랜잭션 격리

- 데이터베이스에서 읽을 때 커밋된 데이터만 보게 함(더티 읽기 없음)
    - 트랜잭션이 쓴 내용을 커밋된 후에 다른 트랜잭션에게 보이게 함
- 데이터베이스에 쓸 때 커밋된 데이터만 덮어쓰게 함(더티 쓰기 없음)
    - 먼저 쓴 트랜잭션이 커밋되거나 어보트될 때까지 두 번째 쓰기를 지연시키는 방법 이용

구현 방법

- 더티 쓰기 방지
    - 로우 수준 잠금을 사용
    - 특정 객체를 변경하고 싶으면 해당 객체에 대한 잠금을 획득하고 트랜잭션이 커밋되거나 어보트될 때까지 잠금을 보유하고 있어야 함
- 더티 읽기 방지
    - 로우 수준 잠금을 사용
        - 객체를 읽기 원하는 트랜잭션이 잠시 잠금을 획득한 후 읽기가 끝난 후 바로 해제
        - 그러나 읽기만 하는 트랜잭션이 쓰기 트랜잭션 하나 완료될때까지 기다려야 해서 운용성이 나쁨
    - 잠금 대신 과거값과 새 값 두개를 모두 기억해두고 커밋 전에는 과거의 값을 읽게 함

### 스냅숏 격리(Snapshot Isolation)

커밋 후 읽기에서도 동시성 문제가 생길 수 있다 → **비반복 읽기, 읽기 스큐**

- 커밋된 데이터만 읽었는데도 같은 트랜잭션 안에서 읽을 때마다 값이 달라지는 현상

```
앨리스 트랜잭션:
  계좌 1 조회 → 500원 

                        ← 이 타이밍에
                        이체 트랜잭션 커밋:
                        계좌 1: 500 → 400원
                        계좌 2: 500 → 600원

  계좌 2 조회 → 600원
  
총합: 500 + 600 = 1100원  ← 실제론 1000원
```

- 데이터베이스의 일관된 스냅숏으로부터 읽음(데이터베이스에 커밋된 상태였던 모든 데이터를 본다)
    
    → 데이터가 나중에 다른 트랜잭션에 의해 바뀌더라도 특정한 시점의 과거 데이터를 봄
    

구현 방법

- 쓰기 잠금 사용
    - 쓰기를 실행하는 트랜잭션은 같은 객체에 쓰는 다른 트랜잭션의 진행을 차단할 수 있음
- 읽을 때에는 잠금을 하지 않음
    - **읽는 쪽에서 쓰는 쪽을 절대 차단하지 않고 쓰는 쪽에서 읽는 쪽을 절대 차단하지 않음**
    - **다중 버전 동시성 제어(MVCC)**
        - 데이터베이스는 객체마다 커밋된 버전을 여러 개 유지(진행 중인 여러 트랜잭션에서 서로 다른 시점의 데이터베이스 상태를 봐야할 수 있어서)
        - 객체를 수정할 때 덮어쓰지 않고 새 버전 추가
        
        ```
        계좌 1의 버전 목록:
        버전 1: 잔액 500원, 트랜잭션 1이 만듦, 트랜잭션 5가 삭제
        버전 2: 잔액 400원, 트랜잭션 5가 만듦, 아직 유효
        ```
        
        - 어떤 버전을 읽을지 판단하는 규칙: 지금 진행중인 트랜잭션 목록을 기록해둠
        
        ```
        트랜잭션 12 시작 시점:
        진행 중인 트랜잭션: [13, 14, 15...]
        → 이것들이 만든 버전은 안 보임
        → 이것들보다 먼저 커밋된 버전만 보임
        ```
        
        - 고유한 트랜잭션 ID(txid)를 할당 받아 데이터를 쓸 때마다 쓰기를 실행한 트랜잭션 ID가 함께 붙음 → 항상 증가하는 숫자
        - 계속 값이 증가하면 40억개 만들어진 후에 오버플로가 발생하므로 postgreSQL에서 vaccum이 오버플로가 발생하지 않도록 cleanup 작업을 함
            - 오래된 버전 삭제(아무도 참조안하는 버전) - 디스크 공간 확보
            - txid Freeze(오버플로 방지) - 40억에 가까워진 오래된 로우면 “FrozenXID”라고 표시함 (과거이므로 비교할 필요 없다고 기록) - 데이터는 살아있음
        - 가시성 규칙 → 어떤 버전을 읽을지 결정하는 규칙
            - 트랜잭션을 만들 때 진행 중인 모든 트랜잭션 목록 기록함
            - 어보트된 트랜잭션 무시
            - 트랜잭션 ID가 더 큰 트랜잭션 무시
            - 외의 데이터는 모두 읽기 가능

#### MVCC에서 색인

- 색인이 객체의 모든 버전을 가리키게 하고 색인 질의가 현재 트랜잭션에서 볼 수 없는 버전을 걸러냄
    
    ```
    색인
    계좌 1 → [버전1(500원, txid:1), 버전2(400원, txid:13)]
    ```
    
- copy-on-write B트리
    - 트리의 페이지가 갱신될 때 색인을 덮어쓰는 대신 변경된 페이지에 대한 새로운 색인 복사본 생성
    
    ```
    수정 발생
    → 데이터 페이지 복사 (새 버전 데이터)
    → 그 데이터를 가리키는 색인 페이지도 복사
    → 기존 색인 페이지는 건드리지 않음
    ```
    
    - 쓰기에 영향 받지 않는 페이지 복사 안하고 변함 없음
    - 수정된 리프부터 루트까지 경로 전체를 복사함, 나머지는 기존걸 공유함
    
    ```
    기존 트리:          새 트리:
    
    루트                루트'
    /    \              /    \
    부모A  부모B        부모A'  부모B  ← 부모B는 공유!
    /   \              /   \
    자식1 자식2        자식1' 자식2   ← 자식2도 공유!
    (500원)            (400원)
    ```
    
    - 가비지 컬랙션 실행하는 프로세스 필요
    - Oracle에서는 직렬성(Serializable), PostgreSQL과 MySQL에서는 반복 읽기(Repeatable Read)라고 부름
    

### 갱신 손실(lost update)

read-modify-write 문제

쓰기 트랜잭션 사이에서 발생할 수 있는 충돌

#### 1. 원자적 쓰기 연산

데이터베이스에서 원자적 갱신 연산을 제공함 → 동시성 안전

- 커서 안정성: 객체를 읽을 때 **객체에 독점적인 잠금을 획득**해서 구현여 갱신이 적용될 때까지 다른 트랜잭션에서 그 객체를 읽지 못하게 함
- **단일 스레드**에서 실행되도록 함

#### 2. 명시적인 잠금

애플리케이션에서 갱신할 객체를 명시적으로 잠금

→ 첫 번째 read-modify-write 주기가 완료될 때 까지 다른 트랜잭션이 기다리도록 강제됨

```
BEGIN TRANSACTION;

SELECT * FROM figures
	WHERE name = 'robot' AND game_id = 222
	FOR UPDATE;  ← 이 로우를 잠금!

-- 이동이 유효한지 확인한 후
-- 이전의 SELECT에서 반환된 것의 위치를 갱신함
UPDATE figures SET position = 'c4' WHERE id = 1234;

COMMIT;
```

#### 3. 갱신 손실 자동 감지

- read-modify-write 주기 병렬 실행을 허용하고 트랜잭션 관리자가 갱신 손실을 발견하면 트랜잭션을 어보트시키고 read-modify-write 주기를 재시도하도록 하는 방법
- 스냅숏 격리와 결합해 효율적으로 데이터베이스가 수행할 수 있음
- 자동으로 갱신 손실 감지되므로 오류 덜 발생
- PostgreSQL의 Repeatable Read, Oracle의 Serializable, SQL의 Snapshot Isolation (MySQL의 Repeatable Read는 지원X)

#### 4. Compare-and-set

값을 마지막으로 읽은 후로 변경되지 않았을 때만 갱신 허용함으로써 갱신 손실 회피함

- 이전의 읽은 값과 일치하지 않으면 갱신 반영되지 않고 read-modify-write 주기 재시도해야함
- 데이터베이스가 WHERE절이 오래된 스냅숏으로부터 읽는 것을 허용한다면 갱신 손실을 막지 못할 수도 있음
    - 스냅숏 격리에서는 WHERE절(트랜잭션 시작시) 최신 값이 아니라 스냅숏 기준으로 읽기 때문에 안 바뀐 것으로 착각할 수 있음
    

### 충돌 해소와 복제

잠금과 compare-and-set 연산은 데이터의 최신 복사본이 하나만 있다고 가정함

→ 다중 리더 또는 리더 없는 복제를 사용하는 데이터베이스는 일반적으로 여러 쓰기가 동시에 실행되고 비동기식으로 복제되는 것을 허용하므로 최신 복사본이 하나만 있다고 보장 안됨

이 경우 **쓰기가 동시에 실행될 때 한 값에 대해 여러 개의 충돌된 버전(형제)을 생성하는 것을 허용하고 사후에 충돌 해소하여 병합**하는 방식을 이용 = 낙관적 동시성 제어

원자적 쓰기 연산은 복제 상황에서도 동작함 (ex. 조회수 + 1) 

- 순서만 보장되면 결과가 같으므로

최종 쓰기 승리(LWW) 충돌 해소 방법은 단순하지만 개인 손실이 발생함 (과거  데이터가 그냥 버려짐)
