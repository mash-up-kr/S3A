# 분산 시스템의 골칫거리

비관주의를 최대한 끌어올려 어떤 것이든 잘못될 가능성이 있다면 잘못된다고 가정한다.

엔지니어로서의 임무는 모든 게 잘못되더라도 제 역할을 해내는(즉 사용자가 기대하는 보장을 만족시키는) 시스템을 구축시키는 것이 목표이다.

하드웨어의 경우 컴퓨터에 내부 결함이 발생하면 잘못된 결과를 반환하기 보다 완전히 동작하지 않게하도록 설계한다.

## 분산 시스템

### 부분 장애(partial failure)

시스템의 어떤 부분은 잘 동작하지만 다른 부분은 예측할 수 없는 방식으로 고장나기도 한다

- **비결정적**: 어떨 때는 동작하지만 어떨 때는 예측할 수 없는 방식으로 실패함
- 비결정성과 부분 장애 가능성이 분산 시스템을 다루기 어렵게 한다.

## 대규모 컴퓨팅 시스템 구축 방법

### 슈퍼컴퓨팅

수천 개의 CPU를 가진 슈퍼컴퓨터가 계산 비용이 매우 높은 과학 계산 작업에 쓰이는 것

- 계산 상태를 지속성 있는 저장소에 체크포인트로 저장한다(로그처럼)
- 노드 하나에 장애가 발생했을 때 전체 클러스터 작업부하를 중단하고 복구된 이후 마지막 체크포인트부터 계산을 재시작한다 → **단일 노드 컴퓨터**에 가까움
- 부분 장애를 전체 장애로 확대하는 방법으로 처리함 
→ **그러나 인터넷 관련 어플리케이션에서는 서비스를 이용할 수 없게 하는 것은 허용되지 않는다**
- 노드들은 신뢰성이 높으며 노드 사이에는 공유 메모리와 원격 직접 메모리 접근을 사용해 통신함

### 클라우드 컴퓨팅

- 상용 장비를 사용해 구축함
- 낮은 비용으로 높은 성능을 제공하지만 각 노드의 고장날 확률이 높아 **고장을 당연한 것으로 보고 이를 극복하도록 설계함**
    - 항상 고장난 상태라고 가정하는 것이 합리적이다
- 시스템에서 장애가 난 노드가 감지되더라도 사용자에게 서비스를 계속 제공하면서 한 번에 노드 하나씩 재시작하는 순회식 업그레이드를 할 수 있다
    - 가상 장비 하나의 성능이 좋지 않으면 그것을 죽이고 새 가상 장비를 요청함

**분산시스템이 동작하게 만들려면 부분 장애 가능성을 받아들이고 소프트웨어에 내결함성 메커니즘을 넣어 신뢰성 있는 시스템을 구축해야 한다**

= 신뢰성 없는 구성 요소를 사용해 신뢰성 있는 시스템을 구축해야 한다

## 신뢰성 없는 네트워크

다루고자 하는 분산 시스템은 비공유 시스템(=네트워크로 연결된 다수의 장비)로 클라우드 컴퓨팅에 가깝다

각 장비들이 자신만의 메모리와 디스크를 갖고 있고 다른 장비의 메모리나 디스크에 접근할 수 없다

### 비동기 패킷 네트워크

인터넷과 데이터센터 내부 네트워크 대부분

- 노드는 다른 노드로 메시지(패킷)를 보낼 수 있지만 네트워크는 메시지가 언제 도착할지 혹은 메시지가 도착하기는 할 것인지를 보장하지 않음
    - 요청 보내고 응답을 기다릴 때까지 여러 문제가 발생할 수도 있음
    
    ```
    1. 요청이 손실될 수 있다
    2. 요청이 큐에서 대기하다 나중에 전송될 수 있다
    3. 원격 노드에 장애가 생겼을 수 있다
    4. 원격 노드가 이시적으로 응답하기를 멈췃지만 나중에 다시 응답하기 시작할 수 있다
    5. 원격 노드가 요청을 처리했지만 응답이 네트워크에서 손실됐을 수 있다
    6. 원격 노드가 요청을 처리했지만 응답이 지연되다가 나중에 전송될 수 있다
    ```
    
    - 전송 측은 패킷이 전송됐는지 아닌지 확인할 수 없고, 수신 측에서 보낸 응답 메시지도 손실되거나 지연될 수도 있다 ⇒ **아직 응답을 받지 못했다는 것**만 알 수 있음
    - **장애를 감지하는 방법 ⇒ 타임아웃**: 얼마 간의 시간이 지나면 응답 대기를 멈추고 응답이 도착하지 않는다고 가정한다 (타임아웃이 발생했을 때 원격 노드가 실제 응답을 받았는지 아닌지는 알 수 없다)

### 네트워크 결함

실제 네트워크 장비를 중복 추가하는 것은 결함을 줄여주지 않는다

네트워크 결함이 드물더라도 일어날 수 있기 때문에 이를 처리할 수 있어야 한다

- 네트워크 결함을 무조건 견디도록 할 필요는 없고 네트워크에 문제가 있을 때 사용자에게 오류 메시지를 보여주는 것도 타당하다. 시스템이 복구할 수 있도록 보장만 하면 된다.
    - 그러기 위해서 고의로 네트워크 문제를 유발하고 시스템의 반응을 테스트한다.

### 결함 감지

시스템은 결함이 있는 노드를 자동으로 감지할 수 있어야 한다.

- 로드 밸런서는 죽은 노드로 요청을 그만 보내야 함
- 단일 리더 복제를 사용하는 분산 데이터베이스에서 리더에 장애가 발생하면 팔로워 중 하나가 리더로 승격돼야 한다.

#### 결함 감지 방법

- 만약 목적지 포트에 수신 대기하는 프로세스가 없다면(프로세스가 죽음) OS가 RST/FIN 패킷 응답을 보내서 TCP 연결을 닫거나 거부하는 방식으로 결함을 감지하기도 한다. **그러나 노드가 요청을 처리하다가 죽었다면 결함을 감지할 방법이 없다.**
- 노드가 죽었지만 OS가 실행중이라면 스크립트로 다른 노드에게 프로세스가 죽었음을 알려서 다른 노드가 빠르게 역할을 넘겨 받을 수 있게 할 수 있다
- 데이터센터 내 네트워크 스위치의 관리 인터페이스에 접근할 수 있으면 질의를 보내 하드웨어 수준의 장애를 감지할 수 있다.
- 접속하려는 IP 주소에 도달할 수 없다고 라우터가 확신하면 ICMP Destination Unreachable 패킷으로 응답할 수 있다.

### 타임아웃과 기약 없는 지연

타임아웃이 너무 길면 노드가 죽었다고 선언될 때까지 기다리는 시간이 길어지고, 짧으면 노드가 잠깐 느려졌을 뿐인데도 죽었다고 잘못 선언해서 같은 일을 두 번 해야할 수도 있다. → 부하를 줄 수 있음

- 만약 패킷의 최대 지연 시간이 보장된 네트워크를 사용한다면 전송 시간, 요청 최대 시간 등을 이용해 2d+r(d는 전송 최대 시간, r은 요청 처리 최대 시간) 을 타임아웃으로 설정할 수 있다.
- 그러나 비동기 네트워크는 기약 없는 지연이 있어서 어떤 최대 시간 내에 요청을 처리한다고 보장할 수 없다

### 네트워크 혼잡과 큐 대기

컴퓨터 네트워크에서 패킷 지연의 변동성은 큐 대기 때문인 경우가 많다.

- 네트워크 스위치는 패킷을 큐에 넣고 한 번에 하나씩 목적지로 넘긴다. 네트워크 링크가 붐비면 패킷은 슬롯을 얻을 수 있을 때까지 잠시 기다려야 할 수도 있다.
- 패킷이 목적지에 도착했을 때 모든 CPU 코어가 바쁜 상태라면 요청이 애플리케이션에서 처리할 준비가 될때까지 OS가 큐에 넣어두고 대기할 수도 있다.
- 가상 환경에서 실행되는 OS는 다른 장비가 CPU 코어를 사용하는 동안 멈출 수도 있다. 가상 장비 모니터가 들어오는 데이터를 큐에 넣어 대기한다.
- TCP는 흐름 제어를 수행하여 네트워크 링크나 수신 노드에 과부하를 가하지 않도록 송신율을 스스로 제한하여 부가적은 큐 대기를 수행한다.
- TCP에서 타임아웃 안에 확인 응답을 받지 않으면 패킷이 손실되었다고 간주하고 손실된 패킷을 자동으로 재전송 한다.

다음과 모든 요인들이 네트워크 지연의 변동성에 영향을 준다. 

장애 감지 지연과 너무 이른 타임아웃의 위험성 사이에서 적절한 트레이드 오프를 결정해야 한다

→ 고정된 타임아웃을 설정하는 대신 시스템이 지속적으로 응답 시간과 변동성(지터)을 측정하고 관찰된 응답 시간 분포에 따라 타임아웃을 자동으로 조절하게 하는 **파이 증가 장애 감지기**를 사용하는 것도 가능하다

### 동기 네트워크 대 비동기 네트워크

하드웨어 수준에서 네트워크를 신뢰성 있게 만들어서 소프트웨어에서 걱정할 필요 없게 하는 방법은 없을까

#### 회선 교환 네트워크

전화 네트워크에서 통화를 하기 위해서 회선(circuit)을 먼저 만든다 → 이 회선은 그 통화에 대해 고정되고 보장된 양의 대역폭이 할당된다.

동기식 네트워크로, 데이터가 여러 라우터를 거치더라고 큐 대기 문제를 겪지 않는다. 큐 대기가 없으므로 네트워크 종단 지연 시간의 최대치가 고정돼 있다 → **제한 있는 지연(bounded delay)**

#### 패킷 교환 네트워크

TCP 연결의 패킷은 가용한 네트워크 대역폭을 기회주의적으로 사용한다. 가변 크기의 데이터 블록을 보내면 짧은 시간 안에 전송하려고 할 것이다. TCP 연결이 유휴 상태에 있는 동안은 어떤 대역폭도 사용하지 않는다. 

데이터센터 네트워크와 인터넷은 패킷 교환을 사용한다. 순간적으로 몰리는 트래픽에 최적화되어 있다.

- 만약 회선을 통해 파일을 전송한다면 대역폭 할당을 추정해야 한다.
    - 추정치가 너무 낮으면 네트워크 용량을 쓰지 않고 남겨 둔 상태로 전송이 불필요하게 느려진다
    - 추정치가 너무 높으면 회선은 구성될 수 없다(회선 교환 네트워크는 대역폭 할당을 보장할 수 없으면 회선 생성을 허용하지 않기 때문이다)

TCP는 가용한 네트워크 용량에 맞춰 데이터 전송률을 동적으로 조절할 수 있다

### 신뢰성 없는 시계

지속시간과 시점 관해서 시계에 의존한다.

- 분산 시스템에서는 통신이 즉각적이지 않으므로 시간을 다루기 까다롭다. 네트워크에 있는 개별 장비는 자신의 시계를 갖고 있는데 각 장비는 자신만의 시간 개념이 있으며 다른 장비보다 시간이 빠르거나 느리거나 할 수 있다.
- 이 시간을 도기화 시켜주는 메커니즘이 **네트워크 시간 프로토콜(Network Time Protocol)**으로 서버 그룹에서 보고한 시간에 따라 컴퓨터 시계를 조정할 수 있게 한다.

### 단조 시계 대 일 기준 시계

#### 일 기준 시계

어떤 달력에 따라 현재 날짜와 시간을 반환한다. (리눅스의 `clock_gettime(CLOCK_REALTIME)`, 자바의 `System.currentTimeMillis()` 가 epoch 이래로 흐른 초 수를 반환한다)

일 기준 시계는 NTP로 동기화된다. 한 장비의 타임스탬프는 다른 장비의 타임스탬프와 동일한 의미를 가질 수 있다.

- 그러나 로컬 시계가 NTP 서버보다 너무 앞서면 강제로 리셋되어 과거 시점으로 거꾸로 뛰는 것처럼 보일 수 있다.

#### 단조 시계

타임아웃이나 서비스 응답 시간 같은 지속 시간을 재는 데 적합하다. 

**시간은 항상 앞으로 흐른다는 사실에서 나옴**

시계의 절대적인 값은 의미가 없다. 두 대의 다른 컴퓨터에서 나온 단조 시계 값을 비교하는 것은 의미가 없다. 

- 멀티 소켓 서버의 경우 각각의 CPU가 독립된 시계를 가지고 있고 100% 동기화 되지 않는데 스케줄링되더라도 시계가 단조적으로 보이게 하려고하므로 무조건적으로 단조시계가 뒤로가지않는건 아니다
- OS 스케줄러는 스레드를 한 CPU에서 다른 CPU로 자유롭게 옮길 수 있다. 스케줄링 없이 한 CPU에 고정돼 있다면 문제가 없으나 중간에 다른 CPU로 옮겨지면 CPU 1의 타이머가 CPU 0보다 살짝 뒤처져 있었기 때문에, 실제로는 시간이 흘렀는데도 측정값은 거꾸로 나올 수 있다
    
    ```
    시각 측정 ①  →  CPU 0에서 실행
                  CPU 0 타이머: 1,000,000,000 ns
    
      [작업 수행 중... 스케줄러가 스레드를 CPU 1로 이동]
    
    시각 측정 ②  →  CPU 1에서 실행
                  CPU 1 타이머:   999,999,847 ns   ← 어긋나 있음!
    
    경과시간 = ② - ① = -153 ns   ← 음수! 시간이 뒤로 감
    ```
    

NTP는 컴퓨터 로컬 시계가 서버보다 느리거나 빠르다는 걸 발견하면 단조 시계가 진행하는 진도수를 조정할 수 있다.

### 시계 동기화와 정확도

단조 시계는 동기화가 필요 없지만 일 기준 시계는 NTP 서버나 다른 외부 시간 출처에 맞춰 설정돼야 유용하다

그러나 시계가 정확한 시간을 알려주게 하는 방법은 신뢰성이 높지 못하다.

- 컴퓨터의 수정 시계는 drift 현상(온도에 따라 변하는 현상)에 의해 아주 정확하지 못하다
- 컴퓨터 시계가 NTP와 너무 많은 차이가 나면 시계가 강제로 리셋
- 방화벽으로 막히면 잘못된 설정이 알려지지 않을 수 있음
- 패킷 지연의 변화가 큰 혼잡한 네트워크에서는 정확도에 한계가 있어서 오차가 발생한다
- 어떤 NTP 서버들은 이상이 있거나 설정이 잘못돼서 오류가 있을 수 있다
- 윤초를 고려하지 않은 시스템에서 윤초가 발생하면 문제가 생길 수 있다. 대부분은 윤초 조정을 하루에 걸쳐서 서서히 수행함으로써 smearing한다
- CPU 코어가 가상 장비 사이에서 공유될 때 각 VM은 다른 VM이 실행되는 동안 수십 밀리초동안 멈추는 일이 생긴다.

시계 정확도가 중요한 경우 매우 높일 수도 있다.

GPS 수신기, 정밀 시간 프로토콜(Precision Time Protocol, PTP)과 세심한 배포 및 모니터링을 사용해서 해결할 수 있다.

### 동기화된 시계에 의존하기

장비의 수정 시계에 결함이 있거나 NTP 클라이언트가 잘못 설정됐다면 시계는 드리프트가 생겨 점점 실제 시간으로부터 멀어져 가지만 대부분이 잘 동작하는 것처럼 보인다 → 미묘한 데이터 손실이 발생할 수 있다.

동기화된 시계가 필요한 소프트웨어를 사용한다면 **모든 장비 사이의 시계 차이를 조심스럽게 모니터링 해야한다**

너무 차이나는 노드는 죽은 것으로 선언되고 클러스터에서 제거해야함

### 이벤트 순서화용 타임스탬프

여러 노드에 걸친 이벤트 순서를 타임스탬프로 정하면 위험하다.

인과적으로 나중에 발생한 쓰기라도 시계가 뒤처지는 노드에서 실행됐으면 더 이른 타임스탬프를 받아서 시계 차이 3ms 미만인데도 순서가 뒤집힐 수 있다

최종 쓰기 승리(LWW)를 통해서 충돌을 해소할 수 있다.

- 최근 값을 유지하고 다른 것들을 버림으로써 충돌을 해소하고 싶은 유혹이 들더라도 최근의 정의는 로컬 일 기준 시계에 의존하며 그 시계는 틀릴 수도 있다는 것을 아는게 중요하다

**논리적 시계**는 증가하는 카운터를 기반으로 하며 이벤트 순서화의 안전한 대안이다.

- 물리적 시계: 일기준 시계, 단조 시계

### 시계 읽기는 신뢰 구간이 있다

시계 읽기를 어떤 시점으로 생각하는 것은 옳지 않다. 어떤 신뢰 구간에 속하는 시간의 범위로 읽는 것이 나을 것이다.

- 불확실성 경계는 시간 출처를 기반으로 계산할 수 있다.
- 서버와 마지막으로 동기화한 시간 이후로 예상되는 시계 드리프트에 NTP 서버의 불확실성을 더하고 그 서버와 통신할 때 걸리는 네트워크 왕복 시간을 더한 값을 기반으로 불확실성을 계산할 수 있다
- 스패너에 있는 구글 트루타임의 경우 로컬 시계의 신뢰 구간을 명시적으로 보고한다
    - 현재 시간을 요청하면 가능한 타임 스탬프 범위 중 가장 이른 것과 가장 늦은 것을 가리키는 두 개의 값 [earlist, latest]를 받는다

### 전역 스냅숏용 동기화된 시계

#### 스냅숏 격리

스냅숏 격리 구현의 경우 단조 증가하는 트랜잭션 ID가 필요하다

- 단일 노드 데이터베이스에는 단순한 카운터가 트랜잭션 ID를 생성하는 데 충분하지만 여러 데이터센터에 있는 여러 장비에 분산돼 있을 경우 코디네이션이 필요하므로 전역 단조 증가 트랜잭션 ID를 생성하기 어렵다.
- 트랜잭션 ID는 인과성을 반영해야 하므로 작고 빠른 트랜잭션이 많은 경우 분산 시스템에서 트랜잭션 ID 생성은 방어할 수 없는 병목이 된다.
    - 트랜잭션 A가 쓴 값을 트랜잭션 B가 읽는다면 B가 A보다 높은 트랜잭션 ID를 가져야 함
- 동기화된 일 기준 시계의 타임스탬프를 트랜잭션 ID로 쓸수 없다 → 시계 정확도에 관한 불확실성때문
- **스패너**는 데이터센터에 걸쳐서 스냅숏 격리를 구현한다
    - 스패터는 트루타입 API가 보고한 시계 신뢰 구간을 사용하며 가장 이른 타임스탬프과 가장 늦은 타임스탬프를 포함하는 두 개의 신뢰 구간이 있고 두 신뢰구간이 겹치지 않는다면 먼저 실행된 트랜잭션을 확정할 수 있다
        - A = [A_earlist, A_latest], B = [B_earlist, B_latest] → A_earlist < A_latest < B_earlist < B_latest
    - 트랜잭션 타임스탬프가 인과성을 반영하는 것을 보장하기 위해 읽기 쓰기 트랜잭션을 커밋하기 전에 의도적으로 **신뢰 구간의 길이만큼 기다린다.**
    - 충분히 뒷 트랜잭션이 나중에 실행되는 게 보장되므로 신뢰 구간이 겹치지 않는다
    - 대기 시간을 짧게 유지하기 위해 스패너는 시계 불확실성을 가능하면 작게 유지해야 한다.
