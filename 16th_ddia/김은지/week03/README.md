## 리더 기반 복제

1. 클라이언트는 쓰기 요청시 **리더**에게 먼저 요청을 보냄
    - 무조건 리더에게 쓰기 요청을 먼저 보냄(읽기는 리더/팔로워 둘다 가능)
2. 리더가 로컬 저장소에 데이터 기록
3. 데이터 변경을 복제 로그나 변경 스트림으로 팔로워에게 전송
4. 팔로워가 로그를 받은 이후 쓰기 적용(로컬 복사본 갱신)

### 동기 vs 비동기

- 대부분 복제는 빠르게 일어나지만 장애 복구 중/시스템 최대 가용량 부근/노드 간 네트워크 문제가 있으면 복제에 지연이 생길 수 있음
    
    #### 동기
    
    - 장점: 팔로워가 리더와 일관성 있게 최신 데이터 복사본을 가지는 것을 보장
    - 단점: 동기 팔로워가 응답하지 않으면 쓰기 처리 안됨
- **반동기식**을 이용함 : 팔로워 1개는 동기, 나머지는 비동기로 처리하여 동기식 팔로워가 사용할 수 없게 되거나 느려지면 비동기식 팔로워 중 하나가 동기식 팔로워가 됨
    - 장애 발생시
        - 동기 팔로워가 죽으면, 리더가 기다리다가 비동기 팔로워 한 개를 동기 팔로워로 승격 → 계속 쓰기 작업이 원활하게 돌아감 → 죽었던 기존의 팔로워가 다시 살아나면 로그의 일련번호 확인 후 리더로부터 로그를 전달받아 재동기화함

- 보통 리더 기반 복제는 완전히 비동기식으로 구성
    - 팔로워에게 모두 쓰기가 전달되지 못하고 유실되면 지속성을 보장하지 못함 - 그래도 리더가 쓰기 처리 계속 가능
    - 많은 팔로워가 있거나 지리적으로 분산된 경우 사용

- 체인복제: 동기식 복제의 변형
    - 노드들을 사슬처럼 연결해서 데이터를 복제하는 방식
    - LinkedList와 유사한 구조
    - Head(쓰기담당)/Middle(데이터 전달받아 복제)/Tail(읽기담당&쓰기작업완료)
    - 기존의 리더 기반 복제와 유사하나 차이점
        - 메인 리더 노드가 모든 팔로워 노드에게 싸줘야하는 것과 달리 다음 노드에게만 전달하면 되므로 네트워크 부하가 일정하다
        - 강한 일관성을 가지고 있다(Tail만 읽으면 되니까)
        - 쓰기 지연 시간이 큰 단점이기 때문에 수천 대의 노드를 복제하기보다 적은 복제본을 유지하면서 **강한 일관성과 높은 처리량**이 필요한 시스템에서 사용
    - Tail에만 몰리는 읽기 부하를 해결하기 위해 모든 노드에서 읽을 수 있고 데이터가 최신인지 확인하는 **CRAQ** 기법 등장
        - 체인 복제처럼 Head에서 Tail까지 데이터를 전달하는 것은 같지만 각 노드에 새로운 쓰기가 일어나면 상태를 Dirty로 표시
        - Tail까지 데이터 도착하면 Tail은 상태를 Clean으로 바꾸고 사슬의 역방향(Tail→Head)로 Ack 신호를 보냄
        - 신호를 받은 노드는 순차적으로 상태를 Dirty→Clean으로 변경
        - 아무 노드에서나 읽을 수 있다(Clean 상태이면)
            - 만약 Dirty 상태면 Tail에게 최신 버전 번호가 뭔지 묻고 Tail이 알려준 버전의 데이터 반환
- 복제 일관성과 합의 사이에는 강한 연관성이 있다 << 이게 무슨말이냐
    - 여러 서버가 똑같은 데이터를 가지기 위해서는, 무엇이 최신인지 다같이 동의하는 과정이 필요하다
    - 합의 알고리즘: 모든 노드가 똑같은 순서의 로그를 가지도록 하는 것

1. 새로운 팔로워 설정
    - 리더의 스냅숏을 새로운 팔로워에 가져온 후, 새로운 팔로워 노드에 복사하여 스냅숏 이후 발생한 모든 데이터 변경을 요청함
2. 장애 복구
    - 팔로워 장애: 수신한 데이터 변경 로그를 로컬 디스크에 보관 후 처리한 마지막 트랜잭션 알아낸 후에 리더에게 데이터 변경 요청
    - 리더 장애
        - 팔로워 중 하나를 리더로 승격/클라이언트는 새로운 리더로 쓰기 전송 하기 위한 재설정/다른 팔로워는 새로운 리더로부터 데이터 변경 소비해야함
    

## 복제 로그 구현

### 구문 기반 복제

- 리더에게 온 쓰기 요청(구문)을 실행 후 구문 그대로 팔로워들에게 보냄
- `NOW()` , `RAND()` 같은 비결정적 함수가 포함되어있으면 데이터가 달라짐
- SQL 문장 자체
- ex) `UPDATE users SET level = level + 1 WHERE id = 10;`

### 쓰기 전 로그 배송

- 데이터베이스가 파일에 데이터를 쓰기 전에 기록하는 로그 파일을 그대로 복제
- 바이너리 로그 - 디스크의 어디 위치에 어떤 데이터를 썼는지
- ex) `파일: ibdata1, 위치: 0x5F2A, 데이터: 01011010...`

### 논리적(로우 기반) 로그 복제

- 구문 기반 방식과 물리적 로그(WAL)의 중간 단계
- 각 행이 어떻게 변했는지, 어떻게 트랜잭션이 커밋됐는지 등을 표시
- 행 단위 변경 이벤트
- ex) `[ID: 7] 행 추가, 이름: '은지', 가입일: '2026-04-27'`

### 트리거 기반 복제

- 애플리케이션 계층이나 DB의 트리거 기능을 사용
- 데이터 베이스 시스템(리더)의 데이터 변경시 자동 실행
- 사용자 정의 데이터
- ex) `데이터가 변하면 이 코드를 실행하라` 고 트리거 걸어둠
- 트리거 발생시 팔로워들에게 코드 실행됨

## 복제 지연 문제

- 리더 기반 복제에서는 읽기 요청이 팔로워에서도 가능하다
- 동기적으로 하게 되면 노드가 많아질수록 다운될 가능성이 커져서 불안정
- 비동기적으로 하게 되면 모든 쓰기가 팔로워에 반영되지 못해 서로 다른 결과를 얻을 수도 있음
- **최종적 일관성**: 이때 데이터베이스에 쓰기를 멈추고 잠시 기다리면 팔로워가 리더와 결국 일치하게 되는 효과

### 쓰기 후 읽기 일관성(자신의 쓰기 읽기 일관성)

- 문제: 비동기식 복제에서는 사용자가 쓰기 수행 시 리더에게 쓰이지만 아직 복제 서버에 반영되지 않을 수 있음
- **쓰기 후 읽기 일관성**을 위해 사용자가 수정한 내용을 읽을 때에는 리더에게서, 나머지는 팔로워에게서 읽는다.
    - 이때 무엇이 수정됐는지 알기 위해서
        - 1) 사용자 소유의 내용은 리더에게서 나머지는 팔로워에게서 읽기
        - 2) 마지막 갱신 시각을 찾아서 마지막 갱신 후 1분 동안은 리더에게서 모든 읽기 수행/ 팔로워에서 복제 지연을 모니터링해 리더보다 1분 이상 늦은 팔로워에 대한 질의 금지
        - 3) 가장 최근 쓰기의 타임스탬프 기억 후 읽기를 위한 복제 서버가 최소한 해당 타임스탬프까지 갱신을 반영하게 하도록 대기
    - 문제: 디바이스 간 쓰기 후 읽기 일관성이 제공되려면?
        - 3)의 방식의 경우 다른 디바이스의 타임스탬프를 알 수 없어서 어려움
        - 복제 서버가 여러 데이터센터 간에 분산되어있으면 동일한 데이터센터로 라우팅된다는 보장이 없으므로, **사용자 디바이스의 요청을 동일한 데이터센터로 라우팅해야함**

### 단조 읽기

- 문제: 사용자가 각기 다른 복제 서버에서 여러 읽기를 수행하게 되면 시간이 거꾸로 흐르는 현상을 발견할 수도 있음
- **단조 읽기**를 통해 이런 이상 현상이 발생하지 않도록 보장할 수 있음
- 데이터를 읽을 때 이전 값을 볼 수 있음(여러번 걸쳐 읽어도 시간이 되돌아가지 않음)
    - **각 사용자의 읽기가 항상 동일한 복제 서버에서 수행되게끔 함**
    - 임의로 서버를 선택하는게 아닌 사용자 ID 해시 기반으로 복제 서버 선택
    - 문제: 복제 서버 고장나면 사용자 질의를 다른 복제 서버로 재라우팅 해야함

### 일관된 순서로 읽기

- 문제: 일부 파티션이 복제 지연으로 인해 느리게 복제되면 관찰자는 두 답변 사이의 인과성이 다르게 보여 인과성이 위반될 수 있다
- **일관된 순서로 읽기**: 일련의 쓰기가 특정 순서로 발생되면 읽기를 읽는 모든 사용자가 같은 순서로 읽어야함
    - 인과성이 있는 쓰기가 동일한 파티션에 기록되게끔 하는 방법

## 다중 리더 복제

- 성능: 모든 쓰기를 로컬 데이터센터에 처리한다음 비동기로 다른 데이터센터에 복제하므로 리더가 있는 데이터센터로 이동해야 하는 단일 리더 설정과 달리 지연 시간을 줄일 수 있음
- 데이터센터 중단 내성: 다른 데이터센터와 독립적으로 동작하고 중단되면 복구되었을 때 다른 데이터센터 복제를 통해 따라잡음
- 네트워크 문제 내성: 데이터센터 간에는 공개 인터넷으로 처리 → 안정성 떨어지는데 비동기 복제를 사용하는 다중 리더 설정은 네트워크 중단이 생겨도 쓰기 처리 진행되므로 더 안정적
- 단점: 동일한 데이터를 다른 두개의 데이터 센터에서 변경할 수 있어서 쓰기 충돌을 해소해야함

### 쓰기 충돌

- 동기식 충돌 감지
    - 첫 번째 쓰기가 완료될 때 까지 두 번째 쓰기를 차단해서 기다림
    - 두 번째 쓰기 트랜잭션 중단 시켜 쓰기를 재시도하게 함
- 그러면 다중 리더 복제의 주요 장점인 복제 서버가 독립적으로 쓸 수 있는 장점을 잃음

### 충돌 회피

- 다중 리더 복제에서는 충돌 처리가 어렵기 때문에 충돌을 회피함
- 특정 사용자의 요청을 동일한 데이터센터로 라우팅해서 데이터센터 내 리더를 사용해 읽기 쓰기를 하게끔 보장함

**데이터는 수렴의 방식으로 충돌을 해소해야함**

→ 모든 변경이 복제되어 모든 복제 서버에 동일한 최종 값이 전달되어야함

#### 사용자 정의 충돌 해소 로직

- 쓰기 수행 중: 충돌 감지시 충돌 핸들러 호출
- 읽기 수행 중: 충돌 감지시 모든 충돌 쓰기 저장 → 애플리케이션에서 충돌 내용을 보여주거나 자동으로 충돌 해소

## 리더 없는 복제

- 모든 복제 서버가 클라이언트롭터 쓰기 직접 받을 수 있게 허용
- **다이나모 스타일**: 클라이언트가 여러 복제 서버에 직접 쓰기 전송 or 코데네이터 노드가 클라이언트를 대신해 수행(특정 순서가 있는 것은 아님)
1. 모든 복제 서버에 쓰기 요청을 보냄
2. 중단된 노드 빼고 다 받아서 설정해놓은 기준 이상의 서버에서 ok 응답을 받으면 쓰기 성공으로 간주
3. 해당 노드에서 데이터 읽기 문제를 해결하기 위해 데이터를 읽을 때 읽기 요청을 병렬로 전송하여 여러 노드에서 여러 응답을 받음
4. 이때 받은 값의 버전 숫자를 활용해 최신 내용을 판단

사용 불가능한 노드의 누락된 쓰기를 어떻게 따라잡아야하나?

- **읽기 복구**: 여러 노드에서 병렬로 읽기를 수행하여 특정 서버가 오래된 값을 갖고 있음을 알게되고 새로운 값 기록
- **안티 엔트로피 처리**: 복제 서버 간 데이터 차이를 백그라운드에서 지속적으로 찾아 누락된 데이터를 하나의 복제 서버에서 다른 서버로 복사

### 정족수

**n개의 복제 서버가 있을 때 모든 쓰기가 w개의 노드에서 성공해야 쓰기가 확정되고, 이후 모든 읽기를 최소 r개의 노드에 질의해서 쓰기가 누락된 서버를 파악할 수 있다고 할때 w + r > n이면 읽을 때 최신 값을 얻을 것으로 기대할 수 있다**

→ 이때의 r과 w를 따르는 읽기와 쓰기를 정족수 읽기와 쓰기라고 부른다

```
정족수 조건이 w + r > n일때

- w < n 이면 노드 하나를 사용할 수 없어도 여전히 쓰기 처리를 할 수 있다

- n = 3, w = 2, r = 2이면 사용 불가능한 노드 1개를 용인
```

#### 이전 발생 관계 파악

- 동시 작업: 작업이 다른 작업보다 먼저 발생하지 않으면(다른 작업을 기반으로 한 작업이 아니어서 **다른 작업에 대해 알지못하는**)
- **이전 발생 관계 파악**
    - 서버가 모든 키에 대한 버전 번호를 유지하고 키를 기록할 때마다 버전 번호 증가시킴
    - 특전 버전 번호를 가진 쓰기를 받을 때 이보다 낮은 값은 덮어쓰지만 높은 버전 번호는 유지해야함
- 자동으로 데이터 값을 삭제하지는 않음 → 그러나 동시에 정리해야함
- 보통 **합집합**으로 형제끼리 병합함
    - 그러나 여기서 제거의 경우에는 형제의 합집합에 제거된 상품이 다시 나타날 수 있음 → **툼스톤** 도입

### 버전 벡터

데이터에 버전 번호 매겨서 충돌 여부 감지 후 묻거나 수동으로 병합함

→ 다중 복제본에서는??

버전 벡터- `V = [A: 0, B: 0, C: 0]`  같은 형태

1. 각 서버에 데이터 쓰기가 발생하면 해당하는 번호의 최신 버전을 저장함
2.  A에 쓰기가 1개 발생했으면
    
    `V = [A: 1, B: 0, C: 0]`
    
3. 데이터를 맞추기 위해 각 서로의 벡터를 확인함
    - `V = [A: 2, B: 1, C: 0] vs V = [A: 1, B: 1, C: 0]` → 이면 최신버전이 왼쪽
    - `V = [A: 1, B: 0, C: 0] vs V = [A: 0, B: 1, C: 0]` → 이면 충돌 발생
