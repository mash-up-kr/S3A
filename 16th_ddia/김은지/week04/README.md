## 파티셔닝

대용량 데이터셋을 여러 노드(디스크)에 분산시켜 질의 부하를 분산시키는 것

- 트랜잭션 작업부하용(빠른처리속도, OLTP) & 분석용(대용량 데이터 질의, OLAP) 모두에서 사용

## 핫스팟 회피 방법

### **해시 파티셔닝**

레코드를 할당할 노드를 무작위로 선택

- 문제: 모든 노드에서 병렬적으로 질의 실행해야함

### **키 범위 파티셔닝**

키-값 데이터 모델 사용시 각 파티션에 연속된 범위의 키를 할당 → B트리 구조 같은

- 문제: 특정한 접근 패턴이 핫스팟을 유발
    - ex) 타임스탬프가 키면 파티션은 시간 범위에 대응되므로 하루치의 데이터를 파티션 하나가 담당하니까 하루치를 쓰거나 읽으면 쓰기/읽기 연산이 모두 동일한 파티션으로 전달되어 해당 파티션만 과부하
    - 해결: 센서 이름을 키의 첫 번째 요소로 사용 → 동시에 동작하는 센서가 많으면 쓰기 부하가 파티션 사이에 퍼짐

### **키의 해시값 기준 파티셔닝**

키 범위 대신 해시값 범위 할당 → 키를 파티션 사이에 균일하게 분산시킬 수 있음

- 문제: 인접했던 키들이 모든 파티션에 흩어져 정렬 순서가 유지되지 않음 → 범위 질의 효율적으로 실행하던 능력 잃어버림

### **카산드라**

여러 칼럼을 포함하는 복합 기본키 지정하여 키의 첫 부분에만 해싱 적용해 파티션 결정에 사용하고 남은 칼럼은 정렬하는 연쇄된 인덱스로 사용

- (유저아이디, 타임스탬프) 면 정렬된 상태를 쉽게 가져올 수 있다

### 그래도 핫스팟 해결이 안되면?

ex) 유명인 아이디에 뭔가 막대한 양의 쓰기 작업이 발생하면, 동일한 ID의 해시값은 동일하므로 해싱은 도움되지 않음

→ 각 키의 시작이나 끝에 임의의 숫자를 붙여 파티션을 분산한다

그러나 추가적인 읽기 작업, 쓰기 작업이 발생한다 (= 따라서 요청이 몰리는 소수의 키에만 적용해야함)

## 보조 인덱스 파티셔닝

### 로컬 인덱스(문서 기반)

각 파티션이 자신의 데이터에 대한 독자적인 보조 인덱스를 가짐

- 기본키를 차량ID로만 설정해두면 색상같은 컬럼은 기본키가 아니어서 빠른 검색이 안되므로 보조 색인을 추가하여 각 노드가 자기 파티션 안에서 새로운 색인을 만든다.
- 쓰기 작업의 경우 해당 파티션에만 쓰면 되므로 빠르나 읽기의 경우 모든 파티션에 물어봐야 함 → **scatter/gather** 발생, 노드 늘어날수록 느려짐

### 글로벌 인덱스(용어 기반)

보조 인덱스 자체도 별도로 파티셔닝함 → 전역 색인

기본 파티션들은 유지하고 보조 색인으로 둔 값을 모아놓은 노드를 새로 추가하여 보조 색인 전용 파티션을 생성한다

→ 이때 색인을 용어 자체를 쓸 수도 있고 용어의 해시값을 사용할 수도 있음

- 읽기가 효율적이다. 모든 파티션에 scatter/gather 사용할 필요 없이 원하는 용어를 포함하는 파티션으로 요청을 보내면됨
- 쓰기의 경우 느리고 복잡 → 데이터 노드와 인덱스 노드 모두 업데이트 해야되므로 비동기로 처리함(쓰기 실행후 바로 색인 읽으면 반영 안될 수 있음)

## 파티션 리밸런싱

클러스터에서 한 노드가 담당하던 부하를 다른 노드로 옮기는 과정을 재균형화라고 함

```
최소 요구사항
- 재균형화 후 부하가 노드들 사이에서 균등하게 분배되어야함
- 재균형화 도중에도 읽기/쓰기 요청을 받아들여야함
- 재균형화가 실행되고 네트워크와 디스크 I/O를 최소화해야함
```

### mod N 연산

hask(key) mod 노드 수

- 노드 개수 N이 바뀌면 대부분의 키가 이동해야 하므로 재균형화 비용이 너무 커져서 사용하면 안됨

### 파티션 개수 고정

파티션을 노드의 수보다 많이 만들고 각 노드에 여러 파티션 할당

- 파티션이 노드 사이에서 통째로 이동하기만 함(파티션이 어떤 노드에 할당되는가)
- 네트워크를 통한 데이터 전송이므로 시간이 조금 걸림
    - 전송 진행 중에는 기존의 파티션을 사용함
- 파티션의 개수가 딱 적절할때 성능이 좋음(너무 작으면 오버헤드가 크고 너무 크면 재균형화 실행할때와 노드 장애 복구시에 비용 큼)

### 동적 파티셔닝

키 범위 파티셔닝을 사용하는 DB에서는 파티션을 동적으로 만듦

파티션 크기가 설정된 값보다 크면 파티션을 두 개로 쪼개 각각에 원래 파티션의 절반 정도의 데이터가 포함되게 함 → 설정된 값보다 작아지면 인접한 파티션과 합침

**B트리 최상위 레벨 작업과 유사**

- 파티션은 노드 하나에 할당되고 노드는 여러 파티션 담당 가능
    - 분할된 파티션 중 하나가 다른 노드로 이동할 수 있음
- 빈 데이터베이스의 경우 파티션 경계에 대한 정보가 없어서 파티션이 하나임
    - 그래서 HBase와 몽고DB에서는 빈 데이터베이스에 초기 파티션 집합을 설정 → **사전 분할**
- 파티션의 개수가 데이터셋 크기에 비례한다 = 노드 대수와 독립적이다

### 노드 비례 파티셔닝

파티션 개수가 노드 대수에 비례하게 함 = 노드당 할당되는 파티션 개수 고정

- 노드 추가되면 파티션 개수가 늘어나므로 개별 파티션 크기가 줄어듦
    - 고정된 개수의 파티션을 **무작위로** 선택해서 분할하고 분할된 파티션 절반은 그대로, 다른 절반은 새 노드에 할당
    - **해시 기반 파티셔닝** 사용해서 파티션 경계 무작위 선택

## 요청 라우팅

클라이언트에서 요청을 보낼 때 어떤 노드로 접속해야하는지 어떻게 아는가?

→ 어떤 key를 읽고 쓰기 위해서 어떤 IP주소와 포트번호로 접속해야 하는가?

### 라운드로빈

아무 노드에 접속한 후 해당 노드에 파티션이 있으면 처리하고 없으면 요청을 올바른 노드로 계속 전달함

### 라우팅계층

모든 요청을 라우팅 계층(파티션 인지 로드밸런서)으로 먼저 보냄 → 각 요청을 처리할 노드를 알아내고 그 노드로 요처을 전달

### 클라이언트 인지

클라이언트가 파티셔닝 방법과 파티션이 어떤 노드에 할당됐는지를 알고 있음 = 중개자 없이 올바른 노드로 접속

### 상태 전달 방법

**주키퍼(ZooKeeper)**: 어떤 파티션이 어떤 노드에 있는지 중앙에서 추적하고 변경되면 노드/라우터들에게 알려주는 코디네이션 서비스

- 노드는 주키퍼에 자신을 등록하고, 주키퍼는 파티션과 노드 사이의 정보를 관리

**가십 프로토콜**: 클러스터 상태를 노드끼리 직접 전파

- 외부 서비스에 의존하지 않고 랜덤하게 몇개의 이웃 노드에게 자신이 아는 클러스터 상태를 전송하는 방식

| 요청 라우팅 방식 | 상태 전달 방법 | 사용 시스템 |
| --- | --- | --- |
| 라운드로빈 | 가십 프로토콜 | Riak |
| 라우팅 계층 | ZooKeeper | HBase, Kafka, SolrCloud |
| 클라이언트 인지 | 가십 프로토콜 | Cassandra |
