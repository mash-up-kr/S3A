# 08. 분산 시스템의 골칫거리
## 프로세스 중단

### 분산 시스템에서 시계를 위험하게 사용하는 예시

파티션마다 리더가 하나씩 있는 데이터베이스에서 리더만 쓰기를 받아들이도록 허용됐을 때 **노드가 여전히 리더인지 그리고 안전하게 쓰기를 받아들일 수 있는지 알기 위해 리더가 다른 노드들로부터 임차권을 얻는다.**

- 특정 시점에 오직 하나의 리더만 임차권을 얻을 수 있다.
- 어떤 노드가 임차권을 획득하면 → 임차권이 만료될 때까지 자신이 리더일 것이라고 알 수 있다.
    - 노드가 계속 리더이려면 임차권이 만료되기 전에 주기적으로 갱신해야 함

```java
if (lease.expiryTimeMillis - System.currentTimeMillis() < 10000) {
		lease = lease.renew(); // 임차권 갱신
}
// 임차권을 발급한 노드의 만료 시간 - 리더 노드의 현재 시간
if (lease.isValid()){
		process(request);
}
```

### 문제

1. 동기화된 시계에 의존한다.
    - 임차권의 만료시각(`expiryTimeMillis`)은 임차권을 발급한 노드의 시계 기준으로 설정되고 이걸 확인하는 건(`currentTimeMillis`) 리더 노드 자신의 시계
    
    ```
    임차권 발급자: "만료 시각 = 10:00:05"
    리더 노드 시계: 현재 10:00:07 → "만료됨" → 요청 거절  ← 실제로는 아직 유효한데
    리더 노드 시계: 현재 10:00:03 → "유효함" → 요청 처리  ← 실제로는 이미 만료됐는데
    ```
    
2. 시간을 확인하는시점과 요청이 처리되는 시점 사이의 시간 차이 존재
    
    ```
    [리더 노드]
    1. lease 확인 → "남은 시간 15초, 유효함" ✓
    2. ← 여기서 GC 멈춤(stop-the-world) 30초 발생 →
    3. 요청 처리 시작
       → 그런데 이미 15초 전에 임차권이 만료됨!
    ```
    
    - 임차권이 만료된 동안 다른 노드가 새 임차권을 얻고 새 리더가 되지만, GC에서 깨어난 이전 리더는 자신이 리더라고 착각하고 계속 쓰기를 함
    - 스레드가 오래 멈출 수 있는 일은 자주 발생함
        - GC, VM의 라이브 마이그레이션, OS 스케줄러가 프로세스를 재우는 경우, 디스크 I/O 블로킹
        - 실행 중인 스레드를 어떤 시점에 선점하고 얼마간의 시간이 흐른후 재개할 수 있음 → 스레드 입장에서는 자신이 멈췄다는 걸 모라서 다시 재개될 때 임차권이 만료됐는지 아닌지 알 수가 없다
    - 단일 장비에서 다중 스레드 코드를 스레드 안전하게 만드는 것과 비슷하다
        - 뮤택스, 세마포어, 원자적 카운터, 잠금 없는 자료구조, 블로킹 큐 등을 통해서 해결 가능
        - 그러나 분산 시스템은 공유 메모리가 없고 신뢰성 없는 네트워크를 통해 메시지를 보낼 수만 있기 때문에 변형 어렵다

## 응답 시간 보장

- 근본 해결책은 GC 같은 중단 원인을 제거해야함
    - 소프트웨어가 응답해야 하는 데드라인을 만족시키지 못하면 전체 시스템의 장애를 유발하는 **엄격한 실시간 시스템**에서는 자원이 중단되지 않도록 해야함
- RTOS(실시간 운영체제)로 CPU 시간을 보장할 수 있지만 비용이 커서 현실적이지 않다.

## 가비지 컬렉션의 영향을 제한하기

GC 중단을 노드가 잠시 동안 **계획적으로 중단**되는 것으로 간주하고 노드가 가비지 컬렉션을 하는 동안 클라이언트로부터의 요청을 다른 노드들이 처리하게 하는 방법이 있다. 런타임이 애플리케이션에게 노드가 곧 GC 중단이 필요하다는 경고를 할 수 있다면 애플리케이션은 그 노드로 새 요청 보내기를 멈추고 그 노드가 처리되지 않은 요청을 완료한 후 아무 요청도 처리하지 않는 동안 GC를 실행하기를 기다릴 수 있다.

수명이 짧은 객체만 가비지 컬렉터를 사용하고 수명이 긴 객체의 전체 GC가 필요할 만큼 객체가 쌓이기 전에 주기적으로 프로세스를 재시작함

## 지식, 진실, 그리고 거짓말

분산 시스템에서 우리는 동작에 관해 정한 가정을 명시하고 이런 가정을 만족시키는 방식으로 실제 시스템을 설계한다. 

신뢰성 없는 시스템 모델에서 잘 동작하는 소프트웨어를 만드는 게 가능할지라도 간단하게는 어렵다 → 우리가 할 수 있는 가정과 제공할 수 있는 보장의 종류에 대해 이야기하자

## 진실은 다수결로 결정된다

### 비대칭적인 결함이 있는 네트워크

- 노드가 자신에게 오는 메시지는 모두 받을 수 있지만 보내는 메시지는 유실되거나 지연됨
    - 다른 노드가 보낸 요청을 받을 수 있더라도 다른 노드는 그 응답을 받을 수 없어서 다른 노드들은 그 노드를 죽었다고 선언함
    - 연결이 끊긴 노드가 자신이 보낸 메시지가 다른 노드들로부터 응답을 받지 못하는 것을 깨닫고 네트워크에 결함이 있음을 알아 챔
    - 긴 stop-the-world GC 중단을 경험하는 노드는 다른 노드들로부터 죽었다고 선언되고 나중에 갑자기 실행하게 됨

**노드는 상황에 대한 자신의 판단을 반드시 믿을 수 있는 것은 아니다**

#### 정족수

분산 시스템은 한 노드에만 의존할 수 없다. 대신 여러 분산 알고리즘은 노드들 사이의 투표인 **정족수(quorum)** 에 의존한다.

- 정족수를 이룬 노드들이 다른 노드를 죽었다고 선언하면 실제 살아있더라도 죽은 노드처럼 물러나야함
- 보통 노드의 과반수 이상을 정족수로 삼음

## 리더와 잠금

분산 시스템에서 잠금을 구현하려면 주의해야한다. 어떤 노드가 선택된 자(파티션의 리더, 잠금 획득)라고 믿을지라도 노드의 정족수도 반드시 동의한다는 것은 아니다.

→ 어떤 노드가 이전에 리더였더라도 시간이 흐른 사이에 다른 노드들이 그 노드가 죽었다고 선언하면 그 노드는 강등되고 이미 다른 리더가 선출됐을 수도 있다.

→ 이 경우 기존의 노드가 쓰기를 하려고 해서 클라이언트들의 쓰기가 충돌되고 파일이 오염된다.

## 펜싱 토큰

리소스에 대한 접근을 보호하기 위해 잠금이나 임차권을 쓸때 자신이 살아있는 노드라고 믿는 노드가 나머지 시스템을 방해할 수 없도록 보장해야 한다.

잠금 서버가 잠금이나 임차권을 승인할 때마다 펜싱 토큰을 반환한다고 가정한다

**펜싱토큰**: 잠금이 승인될 때마다 증가하는 숫자

클라이언트가 쓰기 요청을 서비스로 보낼 때마다 현재 펜싱 토큰을 포함하도록 요구한다 → 더 큰 토큰번호를 가진 쓰기를 이미 처리했으면 그보다 작은 토큰을 처리하지 않도록 함

- Zookeeper 사용 시 트랜잭션 ID zxid나 노드 버전 cversion을 펜싱 토큰으로 사용할 수 있다 → 단조 증가를 보장함

GC에서 깨어난 이전 리더가 오래된 토큰으로 쓰기를 시도하면 저장소가 토큰 번호가 작다는 이유로 거부한다 → 그래서 임차권의 2번 문제(확인-처리 사이 시간 차)를 해결함

## 비잔틴 결함

펜싱 토큰은 부주의에 의한 오류에 빠진 노드를 감지하고 차단할 수 있다(GC 중단 같은 노드)

그러나 **노드가 고의로 시스템의 보장을 무너뜨리려 한다면** 가짜 펜싱 토큰을 포함한 메시지를 보내기만 하면 되므로 이를 방지하지 못한다.

**비잔틴 결함**: 어떤 노드가 실제로는 받지 않은 특정 메시지를 받았다고 거짓말 하는 경우

→ **비잔틴 장군 문제**: 이렇게 신뢰할 수 없는 환경에서 합의에 도달하는 문제

- n명의 장군이 동의해야 하는데 그들의 노력은 그들 가운데 배신자들이 있다는 사실에 방해를 받음. 누가 배신자인지 알려지지 않았으며 누군가가 가짜나 허위 메시지를 보내 다른 장군들을 속이거나 혼란스럽게 하려고 함

일부 노드가 오작동하고 프로토콜을 준수하지 않거나 악의적인 공격자가 네트워크를 방해하더라도 시스템이 계속 올바르게 동작한다면 **비잔틴 내결함성을 지닌다**

대부분의 비잔틴 내결함성 알고리즘은 **노드의 2/3 이상**의 압도적 다수가 올바르게 동작하기를 요구한다

## 약한 형태의 거짓말

노드들이 일반적으로 정직하다고 가정하지만 약한 형태의 거짓말(하드웨어 문제, 소프트웨어 버그)로부터 보호해주는 메커니즘을 소프트웨어에 추가하는 게 가치가 있을 수도 있다.

- 네트워크 패킷이 오염되면 TCP, UDP에 내장된 체크섬으로 검출하거나 애플리케이션 수준 프로토콜에서 체크섬과 같은 단순한 수단을 사용
- 공개적으로 접근 가능한 애플리케이션에서 사용자 입력값에 대해 정상성 점검을 실시
- NTP 클라이언트는 여러 서버 주소를 설정하여 모든 서버에 접속해 그들의 오차를 추정한 후 서버 중 다수가 어떤 시간 범위에 동의하는지 확인하고 잘못 설정된 NTP 서버를 이상치로 검출해 동기화 대상에서 제거

## 시스템 모델과 현실

시스템에서 발생할 것으로 예상되는 결함의 종류를 어떻게든 정형화해야 알고리즘이 하드웨어와 소프트웨어 설정의 세부 사항에 의존하지 않을 수 있다

### 시스템 모델

알고리즘이 가정하는 것을 기술한 추상화

- 동기식 모델
    - 네트워크 지연, 프로세스 중단, 시계 오차에 모두 제한이 있다고 가정 → 현실적인 모델이 아님(기약 없는 지연과 중단이 발생해서)
- 부분 동기식 모델
    - 대부분의 시간에는 동기식 시스템처럼 동작하지만 때때로 네트워크 지연, 프로세스 중단, 시계 드리프트의 한계치를 초과한다는 뜻
- 비동기식 모델
    - 타이밍에 대한 어떠한 가정도 할 수 없음

### 노드 장애

- 죽으면 중단하는(crash-stop) 결함
    - 노드에 장애가 나는 방식은 죽는 것 뿐 → 결코 복구할 수 없음
- 죽으면 복구하는(crash-recovery) 결함
    - 노드가 어느 순간에 죽을 수 있지만 얼마 시간이 흐른 후에 다시 응답할 것이라고 가정
- 비잔틴(임의적인 결함)
    - 다른 노드를 속이는 일이 발생할 수 있음

**현실적인 모델링에서는 부분 동기식 모델이 가장 유용하다**

## 알고리즘의 정확성

펜싱 토큰 사용시 속성은 **유일성**, **단조 일련번호**, **가용성**을 만족해야 한다

- 유일성: 펜싱 토큰 요청이 같은 값을 반환하지 않음
- 단조 일련번호: 요청 x가 토큰 t_x를, 요청 y가 토큰 t_y를 반환했고 y가 시작하기 전에 x가 완료됐다면 t_x < t_y를 만족함
- 가용성: 펜싱 토큰을 요청하지 않고 죽지 않은 노드는 결국에는 응답을 받음

## 안정성과 활동성

- 안정성: 나쁜 일은 일어나지 않는다라는 속성
    - 안전성 속성이 위반되면 그 속성이 깨진 특정 시점을 가리킬 수 있다
    - 안전성 속성이 위반된 후에는 그 위반을 취소할 수 없다
    - 예: "두 노드가 동시에 리더가 되면 안 된다”
- 활동성: 좋은 일은 결국 일어난다라는 속성
    - 어떤 시점을 정하지 못할 수 있지만 항상 미래에 그 속성을 만족시킬 수 있다는 희망이 있다
    - 예: "요청은 언젠가 반드시 처리된다”

**분산 알고리즘은 시스템 모델의 모든 상황에서 안전성 속성이 항상 만족되기를 요구한다**

→ 즉 모든 노드가 죽거나 네트워크 전체에 장애가 생기더라도 알고리즘은 잘못된 결과를 반환하지 않는다고 보장해야 한다

그러나 활동성 속성에 대해서는 경고를 하는게 허용된다

## 시스템 모델을 현실 세계에 대응시키기

안전성 및 활동성 속성과 시스템 모델은 분산 시스템의 정확성을 따져보는데 유용하다

죽으면 복구하는 모델(crash-recovery)에서 알고리즘은 일반적으로 안정된 저장소에 있는 데이터는 노드가 죽어도 남아 있다고 가정한다

- 디스크에 있는 데이터가 오염되거나 데이터가 지워지게 되면?
- 하드 디스크를 인식하는 데 실패하면?

**정족수 알고리즘**은 노드가 저장했다고 선언한 데이터를 기억하고 있다는 것에 의존한다

- 노드가 저장한 데이터를 잊어버리거나 손실될수 있는 가능성이 있다면 알고리즘의 정확성도 깨뜨린다.

**알고리즘을 이론적으로 설명할 때는 어떤 일이 일어나지 않는다고 가정할 수 있다**

- 비잔틴 알고리즘에서는 일어날 수 있는 일과 일어날 수 없는 일에 대한 가정도 해야한다.

→ 실제 구현에는 여전히 불가능하다고 가정했던 일이 발생하는 경우를 처리하는 코드를 포함시켜야 할 수 있다.

**추상 시스템 모델은 현실 시스템의 복잡함에서 우리가 추론할 수 있는 관리 가능한 결함의 집합을 뽑아내서 문제를 이해하고 체계적으로 해결하려고 노력할 수 있게 하는데 도움이 된다**

- 알고리즘이 올바르다고 증명됐어도 현실 시스템에서의 구현도 언제나 올바르게 동작한다는 뜻은 아니다

## 정리

### 분산 시스템에서 나타날 수 있는 광범위한 문제

- 네트워크 패킷을 보내려 할 때 패킷이 손실되거나 임의대로 지연될 수 있다
- 노드의 시계가 다른 노드와 맞지 않을 수 있다
- 프로세스는 실행 도중 어느 시점에서 상당기간 동안 멈출 수 있고 다른 노드에 의해 죽었다고 선언될 수 있고 되살아났을 때 멈췄다는 사실을 모를 수도 있다

**부분 실패가 생길 수 있다는 사실은 분산 시스템의 뚜렷한 특성**

→ 일부가 고장 나더라도 전체 시스템은 동작할 수 있도록 부분 실패에 대한 내성을 내장하려고 함

→ 결함을 견디기 위해서 그것을 감지해야 함

**타임아웃**을 원격 노드를 쓸 수 있는지 결정하기 위해 사용하지만 네트워크 장애와 노드 장애를 구별할 수 없고 변동이 큰 네트워크 지연 때문에 때때로 노드가 죽은 것으로 잘못 의심 받을 수 있음

결함이 발견됐을 때도 시스템이 이를 견딜 수 있게 만들어야 함

= 단일노드가 안전하게 중대한 결정을 할 수 없으므로 다른 노드의 도움을 요청하고 동의할 정족수를 이루려고 시도하는 프로토콜이 필요하다.
