## 코드 변경과 호환성

- 대규모 애플리케이션에서 코드 변경은 대개 즉시 반영할 수 없다. 순회식 업그레이드(rolling upgrade)를 주로 활용한다.
- 클라이언트 측 애플리케이션은 사용자에게 전적으로 좌우된다. 어떤 사용자는 한동안 업데이트를 설치하지 않을 수도 있다.
- 즉, 예전 버전의 코드와 새로운 버전의 코드, 이전 데이터 타입과 새로운 데이터 타입이 시스템에 동시에 공존할 수 있다.
- 시스템이 원활하게 실행되려면 양방향으로 호환성을 유지해야 한다.

### 상위 호환성과 하위 호환성
- 하위 호환성(backward compatibility): 새로운 코드가 예전 코드로 기록한 데이터를 읽을 수 있다.
- 상위 호환성(forward compatibility): 예전 코드가 새로운 코드로 기록한 데이터를 읽을 수 있다.
- 하위 호환성은 비교적 쉽다. 새 코드를 작성할 때 이전 형식을 알고 있으니까.
- 상위 호환성은 더 까다롭다. 예전 코드가 미래에 추가될 필드를 무시할 수 있어야 한다.

## 데이터 부호화 형식

- 데이터를 파일에 쓰거나 네트워크를 통해 전송하려면, 일련의 바이트열 형태로 부호화해야 한다.
- 인메모리 표현 → 바이트열: 부호화(encoding), 직렬화(serialization), 마샬링(marshalling)
- 바이트열 → 인메모리 표현: 복호화(decoding), 파싱(parsing), 역직렬화(deserialization), 언마샬링(unmarshalling)

### 언어 내장 부호화의 문제점
- 언어 내장 부호화(Java의 Serializable, Python의 pickle 등)는 특정 프로그래밍 언어와 묶여 있어 다른 언어에서 데이터를 읽기가 매우 어렵다.
- 동일한 객체 유형의 데이터를 복원하려면 복호화 과정이 임의의 클래스를 인스턴스화할 수 있어야 한다.

> [!WARNING]
> 공격자가 임의의 바이트열을 복호화할 수 있는 애플리케이션을 얻을 수 있으면, 임의의 클래스를 인스턴스화할 수 있다. 복호화 시 클래스의 생성자나 매직 메서드가 자동 실행되기 때문에, 공격자가 악의적인 클래스를 포함한 바이트열을 주입하면 원격 코드 실행(RCE)이 가능해진다. Java의 ObjectInputStream 취약점이 대표적인 사례.

## JSON, XML의 한계

- JSON과 XML은 유니코드 문자열을 지원하지만 이진 문자열(binary string)은 지원하지 않는다.
- 이진 데이터를 Base64로 텍스트 부호화하여 이 제한을 피할 수 있지만, 데이터 크기가 33% 증가한다.
- 필수는 아니지만 XML과 JSON 모두 스키마를 지원한다.
- XML과 JSON을 정의하는 스키마 언어는 상당히 강력하지만, 구현하기가 난해하다.

> [!TIP]
> **최소 공통 분모(lowest common denominator) 부호화 형식**: 서로 다른 시스템(언어, 플랫폼)이 데이터를 주고받을 때, 모든 쪽이 이해할 수 있는 가장 기본적인 형식을 뜻한다. JSON이 대표적. 누구나 파싱할 수 있지만 정수/부동소수점 구분 없음, 이진 데이터 미지원 등 표현력이 제한된다. "모두가 쓸 수 있는 대신 기능이 빈약한 형식"이라는 트레이드오프.

## 이진 부호화

- JSON은 XML보다 덜 장황하지만, 이진 형식과 비교하면 둘 다 훨씬 많은 공간을 사용한다.
- 이런 관찰이 JSON/XML용 다양한 이진 부호화의 개발로 이어졌다.

### Thrift와 Protocol Buffers
- 둘 다 인터페이스 정의 언어(IDL)로 스키마를 기술해야 한다.
- Thrift는 두 가지 이진 부호화 형식이 있다.
	- Binary Protocol: 단순하고 직관적
	- Compact Protocol: 필드 타입과 태그 숫자를 단일 바이트로 줄이고, 가변 길이 정수를 사용해서 부호화한다.
- 가변 길이 정수 예시: 숫자 1337을 8바이트 전부 사용하는 대신 2바이트로 부호화할 수 있다.

## 스키마 발전(Schema Evolution)

- 스키마는 시간이 지남에 따라 필연적으로 변한다.
- 각 필드는 태그 숫자(tag number)로 식별하고, 데이터 타입을 주석으로 단다.
- 필드 값을 설정하지 않은 경우 부호화 레코드에서 단순히 생략한다.

### 필드 추가 시 호환성
- 새로운 태그 번호를 부여하는 방식으로 스키마에 새 필드를 추가할 수 있다.
- 상위 호환성: 예전 코드는 인식하지 못하는 태그 번호의 필드를 무시하면 된다.
- 하위 호환성: 필드에 고유한 태그 번호가 있는 동안에는 태그 번호가 계속 같은 의미를 가지므로, 새로운 코드가 예전 데이터를 항상 읽을 수 있다.
- 단, 새로운 필드를 required로 추가하면 예전 코드가 해당 필드를 기록하지 않기 때문에 하위 호환성이 깨진다.
- 그러므로 초기 배포 후 추가되는 모든 필드는 optional이거나 기본값을 가져야 한다.

### 필드 삭제
- 추가의 반대. optional 필드만 삭제할 수 있고, 같은 태그 번호는 절대 다시 사용할 수 없다.

## Avro

- Thrift, Protocol Buffers와 대적할 만한 이진 부호화 형식.
- 부호화할 데이터 구조를 지정하기 위해 스키마를 사용한다.
- 2개의 스키마 언어가 있다.
	- Avro IDL: 사람이 편집할 수 있는 형식
	- JSON 기반 언어: 기계가 더 쉽게 읽을 수 있는 형식

### Avro의 부호화 방식
- 이진 데이터를 파싱하려면 스키마에 나타난 순서대로 필드를 살펴보고, 각 필드의 데이터 타입을 미리 파악해야 한다.
- 데이터를 읽는 코드가 기록한 코드와 정확히 같은 스키마를 사용해야만 이진 데이터를 올바르게 복호화할 수 있다.

### 쓰기 스키마와 읽기 스키마
- Avro의 핵심 아이디어: 쓰기 스키마와 읽기 스키마가 동일하지 않아도 되며, 단지 호환 가능하면 된다.
- 필드 순서가 달라도 문제 없다. 스키마 해석에서 이름으로 필드를 일치시키기 때문이다.
- 읽기 스키마에 없고 쓰기 스키마에만 있는 필드를 만나면 무시한다.
- 읽기 스키마에서 기대하는 필드가 쓰기 스키마에 없으면, 읽기 스키마에 선언된 기본값으로 채운다.

### Avro의 호환성 규칙
- 기본값이 있는 필드만 추가하거나 삭제할 수 있다.
- 기본값이 없는 필드를 추가하면 → 새로운 읽기가 예전 쓰기의 데이터를 읽을 수 없음 → 하위 호환성 깨짐
- 기본값이 없는 필드를 삭제하면 → 예전 읽기가 새로운 쓰기의 데이터를 읽을 수 없음 → 상위 호환성 깨짐

### 읽기는 쓰기 스키마를 어떻게 알 수 있을까?
- 대용량 파일(많은 레코드): 파일 시작 부분에 쓰기 스키마를 한 번만 포함한다.
- 개별 레코드가 있는 데이터베이스: 각 레코드의 시작 부분에 버전 번호를 포함하고, DB에 스키마 버전 목록을 유지한다. 읽기는 레코드에서 버전 번호를 추출한 뒤 해당 쓰기 스키마를 가져온다.
- 네트워크 연결: 두 프로세스가 양방향 연결을 설정할 때 스키마 버전을 합의한다.

## 스키마 기반 이진 부호화의 장점

- 부호화된 데이터에서 필드 이름을 생략할 수 있어, JSON 변형보다 크기가 훨씬 작다.
- 스키마 데이터베이스를 유지하면, 스키마 변경 적용 전에 상위/하위 호환성을 확인할 수 있다.
- 정적 타입 프로그래밍 언어 사용자에게 스키마로부터 코드를 생성하는 기능이 유용하다.

## 데이터플로

- 상위/하위 호환성은 발전성에서 중요하다.
- 발전성: 한 번에 모든 것을 변경할 필요 없이, 시스템의 다양한 부분을 독립적으로 업그레이드해 변경사항을 쉽게 반영하는 능력.

### 데이터베이스를 통한 데이터플로
- 데이터베이스에 저장하는 일은 미래의 자신에게 메시지를 보내는 것과 같다. 하위 호환성은 분명히 필요하다.
- 새로운 버전의 코드로 기록된 값을 예전 버전의 코드가 읽고 갱신한 뒤 다시 기록할 수 있다.
- 이때 바람직한 동작은 예전 코드가 해석할 수 없더라도 새로운 필드를 그대로 유지하는 것이다.
- 하지만 애플리케이션에서 DB 값을 모델 객체로 복호화한 뒤 다시 재부호화하면, 변환 과정에서 알지 못하는 필드가 유실될 수 있다.

### Data Outlives Code
- 애플리케이션은 새 버전 배포 시 몇 분 내에 완전히 대체할 수 있지만, 데이터베이스 내용은 그렇지 않다.
- 5년 전 데이터는 명시적으로 다시 기록하지 않는 한 원래 부호화 상태로 그대로 남아 있다.
- 데이터를 새로운 스키마로 마이그레이션하는 작업은 가능하지만, 대용량 데이터셋에서는 값비싼 작업이라 대부분의 DB에서는 피한다.
- 대부분의 관계형 DB는 기존 데이터를 다시 기록하지 않고, null 기본값을 갖는 새 칼럼을 추가하는 간단한 스키마 변경을 허용한다.
- 스키마 발전은 저장소가 여러 버전의 스키마로 부호화된 레코드를 포함하더라도, 전체 DB가 단일 스키마로 부호화된 것처럼 보이게 한다.

### 보관 저장소
- 백업이나 데이터 웨어하우스 적재를 위해 DB 스냅샷을 수시로 만든다.
- 소스 DB에 다양한 시점의 스키마 버전이 섞여 있더라도, 데이터 덤프는 최신 스키마를 사용해 일관되게 부호화하는 편이 낫다.
- 데이터 덤프는 한 번 기록하고 이후 변하지 않으므로, Avro 객체 컨테이너 파일 같은 형식이 적합하다.

### 서비스를 통한 데이터플로 (REST, RPC)
- 서버가 API를 공개하고 클라이언트가 요청을 보내는 구조.
- REST: HTTP 원칙 기반. 주로 JSON 사용.
- SOAP: XML 기반. WSDL로 스키마 기술. 복잡하고 무겁다.
- RPC: 원격 네트워크 요청을 로컬 함수 호출처럼 사용할 수 있게 해준다.
	- 하지만 네트워크 요청은 로컬 호출과 근본적으로 다르다. 타임아웃, 재시도, 지연시간 등.
	- gRPC, Thrift, Avro RPC 등.
- 서버를 클라이언트보다 먼저 갱신하는 게 일반적이므로, 요청에는 하위 호환성, 응답에는 상위 호환성이 필요하다.

### 메시지 전달을 통한 데이터플로
- REST/RPC(직접 통신)와 DB(저장 후 조회) 사이의 비동기 방식.
- 메시지 브로커를 중간에 두고 통신한다. (RabbitMQ, Kafka 등)
- 장점:
	- 수신자가 죽어있어도 버퍼 역할을 해서 메시지 유실 방지
	- 송신자가 수신자의 IP/포트를 알 필요 없다
	- 하나의 메시지를 여러 수신자에게 전송 가능
	- 송신자와 수신자를 논리적으로 분리
- 통신이 단방향이라는 점에서 RPC와 다르다.
- 분산 액터 프레임워크(Akka, Orleans, Erlang OTP)는 액터 모델로 동시성과 메시지 전달을 통합한다.
