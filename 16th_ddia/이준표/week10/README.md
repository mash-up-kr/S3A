## 합의(Consensus)

- 9장 전반에서 선형성(linearizability)과 인과성(causality), 전체 순서 브로드캐스트(total order broadcast)를 다뤘다. 후반의 주제는 하나다. 여러 노드를 어떻게 하나의 결정에 동의시킬 것인가.
- 말로 하면 "노드 여러 개가 뭔가에 대해 합의한다"가 전부인데, 이게 분산 시스템에서 가장 어렵고 오해가 많은 문제다.
- 합의가 필요한 대표 상황
	- 리더 선출 : 단일 리더 복제에서 모든 노드가 "누가 리더인지"에 동의해야 한다. 동의 못 하면 스플릿 브레인(split brain).
	- 원자적 커밋(atomic commit) : 여러 노드/파티션에 걸친 트랜잭션을 전부 커밋 or 전부 어보트로 맞추는 것.
- 이 둘은 결국 같은 문제로 수렴한다. 먼저 원자적 커밋부터 본다.

## 원자적 커밋과 2단계 커밋(2PC)

### 단일 노드에서는 쉽다
- 원자성(atomicity) : 트랜잭션은 전부 커밋되거나 전부 어보트되거나. 반쯤 걸친 상태를 안 남긴다. (5장 ACID의 A)
- 단일 노드에서 커밋 여부는 커밋 레코드가 디스크에 기록된 그 한 순간에 결정된다. 스토리지 엔진이 담당하는 원자적 쓰기 하나면 끝난다.
- 그런데 노드가 여러 개면 이 "한 순간"이 사라진다.
	- EX. 파티셔닝된 DB의 다중 파티션 쓰기, 용어 파티셔닝된 보조 색인 갱신.
	- 어떤 노드는 커밋에 성공하고 어떤 노드는 제약 위반으로 어보트할 수 있다.
	- 커밋 메시지 일부가 네트워크에서 손실될 수 있다.
	- 커밋 도중 일부 노드가 죽을 수 있다.
- 핵심 원칙 : 한 번 커밋한 트랜잭션은 취소할 수 없다. 커밋된 값은 이미 다른 트랜잭션에게 보였고, 그 위에서 또 다른 결정이 내려졌을 수 있다. → 그래서 "커밋해도 되는지"를 커밋 이전에 확정지어야 한다.

### 2PC의 동작
- 다중 노드 원자적 커밋을 달성하는 고전 알고리즘. 코디네이터(coordinator, 트랜잭션 관리자)라는 컴포넌트가 새로 등장한다.
- 1단계 (prepare) : 코디네이터가 모든 참여자(participant)에게 "커밋 준비됐냐"를 묻는다.
	- 참여자는 커밋에 필요한 모든 것(제약 검증, 디스크 쓰기)을 실제로 해두고 "예"라고 답한다.
	- "예"라고 답한 순간, 참여자는 어보트할 권리를 포기한다. 실제 커밋은 아직 안 했지만 반드시 커밋할 수 있다고 약속한 것.
- 2단계 (commit/abort) : 코디네이터가 답을 취합한다.
	- 전원 "예" → 커밋 결정을 내리고, 그 결정을 자기 디스크(트랜잭션 로그)에 먼저 기록한다. 이 지점이 커밋 포인트(commit point).
	- 하나라도 "아니오"거나 무응답 → 어보트 결정.
	- 결정을 참여자에게 통보. 참여자는 무조건 따라야 하고, 실패하면 될 때까지 재시도한다.
- 원자성을 떠받치는 건 돌아올 수 없는 두 개의 약속이다. ① 참여자가 "예" 하면 반드시 커밋 가능, ② 코디네이터가 결정하면 번복 불가.

### 코디네이터가 죽으면 — 2PC의 한계
- 최악은 코디네이터가 커밋 결정을 내린 직후, 일부에게만 통보하고 죽는 경우다.
- "예"라고 투표했지만 통보를 못 받은 참여자는 의심 상태(in-doubt)에 빠진다.
	- 커밋인지 어보트인지 스스로 알 수 없다. 옆 참여자에게 물어도 걔도 모른다.
	- 함부로 커밋/어보트하면 원자성이 깨진다. → 할 수 있는 게 코디네이터가 살아 돌아오길 기다리는 것뿐이다.
	- 그동안 그 트랜잭션이 쥔 잠금을 못 놓으므로, 다른 트랜잭션이 통째로 블록된다.
- 그래서 코디네이터의 트랜잭션 로그는 반드시 디스크에 내구성 있게 남겨야 한다. 복구하면 로그를 읽어 의심 트랜잭션들의 운명을 결정한다.
- 이 성질 때문에 2PC를 블로킹 원자적 커밋 프로토콜이라 부른다. (블로킹 안 하는 3PC도 있지만, "네트워크 지연에 상한이 있다"는 비현실적 가정이 필요해 실무에선 거의 안 쓴다.)

### 실전 분산 트랜잭션 — XA
- 두 종류를 구분해야 한다.
	- DB 내부 분산 트랜잭션 : 같은 소프트웨어의 노드끼리(EX. VoltDB, MySQL Cluster의 NDB). 내부 최적화가 가능해 꽤 잘 돈다.
	- 이종(heterogeneous) 분산 트랜잭션 : 서로 다른 기술을 하나로 묶는 것(EX. 벤더가 다른 두 DB, DB + 메시지 브로커). 훨씬 어렵다.
- XA(eXtended Architecture) : 이종 분산 트랜잭션 표준. 2PC를 구현하는 C API고, Java에서는 JTA로 노출된다. 상당수 DB/메시지 브로커가 지원한다.
- XA의 현실적 골칫거리
	- 코디네이터가 애플리케이션 프로세스 안의 라이브러리인 경우가 많다. 이 프로세스가 죽으면 의심 트랜잭션이 미아가 된다. 로그가 그 서버 로컬 디스크에만 있어 다른 데서 복구도 못 한다.
	- 의심 트랜잭션의 잠금 때문에 애플리케이션 전체가 멈추기도 한다.
	- 복구 불가 시 결국 DBA가 손으로 커밋/어보트를 결정해야 한다. 압박 속의 위험한 수작업.
	- 많은 XA 구현이 "휴리스틱 결정(heuristic decision)"이라는 탈출구를 둔다. 참여자가 코디네이터 없이 일방적으로 판단하는 것 → 사실상 원자성을 깨는 비상 수단이다.

> [!tip] 그래서 분산 트랜잭션은 왜 평이 나쁜가
> XA류 이종 분산 트랜잭션은 운영 비용이 커서 많은 클라우드 서비스가 아예 지원을 뺀다. 대신 로그 기반 파생 데이터·멱등성·CDC로 "결국 일관성"을 맞추는 쪽을 택한다(11·12장). 그렇다고 원자적 커밋 문제가 사라지는 건 아니다. 이걸 더 일반적이고 내결함성 있게 푸는 것이 아래의 합의(consensus)다.

## 내결함성을 갖춘 합의

### 합의가 만족해야 하는 것
- 하나 이상의 노드가 값을 제안(propose)하고, 알고리즘이 그중 하나를 결정(decide)한다. 형식적으로 4가지를 요구한다.
	- 동의(uniform agreement) : 두 노드가 서로 다른 값을 결정하지 않는다.
	- 무결성(integrity) : 한 번 결정하면 번복하지 않는다.
	- 유효성(validity) : 결정된 값은 누군가 실제로 제안한 값이어야 한다. (아무 값이나 정하는 트리비얼 해 배제)
	- 종료(termination) : 죽지 않은 노드는 결국 뭔가를 결정한다.
- 앞의 셋은 안전성(safety), 종료는 활동성(liveness)이다.
- 종료는 과반(정족수) 노드가 살아있다는 가정을 요구한다. 과반이 죽으면 진행은 못 한다. 단, 그래도 안전성(동의·무결성)은 안 깨진다.
- 합의 알고리즘은 비잔틴 결함이 없다고(노드가 거짓말 안 한다고) 가정한다. 비잔틴 내결함성 합의는 훨씬 복잡하고 정족수도 더 든다.

> [!tip] FLP 불가능성
> Fischer·Lynch·Paterson이 1985년에 증명한 결과(FLP). 완전 비동기 시스템에서 노드 하나만 죽어도 항상 종료하는 합의 알고리즘은 없다. 그런데 현실의 Paxos·Raft는 잘 돈다. 이유는 FLP가 "시계도 타임아웃도 못 쓴다"는 극단적 모델을 가정하기 때문. 실제로는 타임아웃으로 죽은 노드를 짐작할 수 있어서, 불가능성을 우회한다. → 이론적 하한과 실무가 갈리는 대표 사례.

### 합의 ≈ 전체 순서 브로드캐스트
- 유명한 내결함성 합의 알고리즘 : Viewstamped Replication(1988), Paxos(Lamport), Raft(2014), Zab(ZooKeeper).
- 이들은 값 하나가 아니라 값의 순서(sequence)에 합의한다. 이게 곧 9장 전반의 전체 순서 브로드캐스트다.
	- 전체 순서 브로드캐스트 = 메시지를 모든 노드에 정확히 한 번, 같은 순서로 전달.
	- 매 라운드의 합의가 "다음 메시지가 뭐냐"를 정하는 것 → 합의를 반복하면 전체 순서 브로드캐스트가 된다.
- 이것이 상태 기계 복제(state machine replication)의 토대. 같은 순서의 같은 연산을 각 노드가 적용하면 상태가 수렴한다.

### 단일 리더 복제와의 순환
- "단일 리더 복제도 리더가 정한 순서로 쓰기를 적용하니 전체 순서 브로드캐스트 아닌가?" → 맞다.
- 리더를 사람이 수동 지정하고 고정하면 합의가 필요 없다. 대신 리더가 죽으면 사람이 개입해야 하고 자동 복구가 안 된다(활동성 없음).
- 자동 페일오버를 하려면 리더 선출에 합의가 필요하다. 그런데 리더를 뽑는 것 자체가 합의 문제라 순환에 빠진 것처럼 보인다. 이걸 어떻게 깨는가.

### 에포크 번호와 정족수
- 합의 프로토콜은 약한 리더를 쓰되, 두 번의 투표로 순환을 깬다.
- 에포크(epoch) 번호 (Paxos의 ballot, Raft의 term, Zab의 epoch, VSR의 view) :
	- 리더 선출 때마다 단조 증가하는 번호를 붙인다.
	- 두 리더가 충돌하면(스플릿 브레인 의심) 에포크가 높은 쪽이 이긴다. 낮은 쪽은 물러난다.
- 두 라운드의 투표
	1. 리더를 선출하는 투표.
	2. 리더가 제안을 확정하기 전에 받는 정족수(quorum) 투표.
- 이때 두 정족수는 반드시 겹쳐야 한다(overlap). 어떤 제안이 통과했다면 그 투표자 중 최소 하나는 최신 리더 선출에도 참여했다는 게 보장된다. → 낡은 에포크의 리더가 몰래 커밋하는 걸 막는다.

> [!tip] 2PC vs 합의
> 둘 다 "여러 노드를 하나의 결정에 맞추는" 프로토콜이지만 결정적으로 다르다.
>
> |        | 2PC                        | 합의(Paxos/Raft/…)          |
> | ------ | -------------------------- | ------------------------- |
> | 코디네이터/리더 | 고정, 선출 안 함              | 선출됨, 에포크로 교체 가능        |
> | 필요한 표  | 모든 참여자의 만장일치 "예"       | 과반(정족수)만                |
> | 노드 장애  | 코디네이터 죽으면 블록(의심 상태) | 과반만 살면 계속 진행           |
> | 성격     | 블로킹                       | 내결함성                    |
>
> 한 줄 요약 : 만장일치를 과반으로 낮추고, 코디네이터를 선출·교체 가능하게 만든 것이 합의다.

### 합의의 한계
- 강력하지만 공짜가 아니다.
	- 제안을 투표로 확정하는 과정이 곧 동기식 복제라, 성능(지연·처리량) 대가가 있다.
	- 정족수(보통 과반)가 살아있어야 한다. 네트워크 파티션 시 소수 측은 멈춘다.
	- 참여 노드 집합이 고정이라 가정하는 경우가 많다. 동적 멤버십 변경은 어려운 확장 기능.
	- 타이밍에 예민하다. 네트워크 지연이 크게 튀면 리더가 죽은 줄 알고 재선출이 잦아져 오히려 성능이 무너진다. 불안정한 네트워크에서 특히.

## 합의로 풀리는 문제들 & 코디네이션 서비스

### 전부 같은 문제였다
- 다음은 전부 (전체 순서 브로드캐스트 → 합의)로 환원된다. 잘 만든 합의 하나 위에 이 모두를 올릴 수 있다.
	- 선형성 있는 compare-and-set 레지스터 : 값이 예상과 같을 때만 원자적 갱신.
	- 원자적 커밋 : 다중 파티션 트랜잭션의 커밋/어보트 결정.
	- 잠금과 임차권(locks & leases) : 경합 시 누가 잠금을 쥐는지 결정론적으로 결정.
	- 유일성 제약 : 사용자명/이메일 중복 방지처럼 "먼저 온 하나만 승인".
	- 멤버십 서비스 : 어떤 노드가 지금 살아있는 멤버인지 결정.

### ZooKeeper / etcd
- 애플리케이션 개발자가 합의 알고리즘을 직접 짜는 건 위험하고 어렵다. → 검증된 코디네이션 서비스에 아웃소싱하는 게 정석이다.
- ZooKeeper(Zab), etcd(Raft)는 합의를 내부에 구현하고 그 위에 프리미티브를 노출한다. 소량(메모리에 다 올라가는) 데이터를 완전 복제하며, 그 데이터에 내결함성 있는 전체 순서 브로드캐스트를 건다. (대용량 저장용이 아니다.)
- 이들이 제공하는 것
	- 선형성 있는 원자적 연산 : 원자적 CAS. 여러 노드가 동시에 같은 잠금을 만들려 해도 정확히 하나만 성공한다 → 분산 잠금.
	- 연산의 전체 순서화 + 펜싱 토큰 : 모든 연산에 단조 증가하는 트랜잭션 ID(zxid)를 부여. 이게 8장의 펜싱 토큰(fencing token) 역할을 해, GC 멈춤 등으로 뒤처진 잠금 보유자의 낡은 접근을 거부한다.
	- 장애 감지 : 클라이언트가 세션을 유지(하트비트). 세션이 끊기면 그 세션이 만든 임시 노드(ephemeral node)가 자동 삭제된다 → 노드 다운 감지.
	- 변경 알림(watch) : 다른 클라이언트의 변경을 폴링 없이 구독. 노드 참여/이탈을 즉시 안다.
- 대표 활용
	- 리더 선출 / 파티션 배정 : 후보들이 잠금 획득을 다투고 성공한 하나가 리더가 된다. 리더가 죽으면(세션 만료) 임시 노드가 사라져 재선출이 트리거된다.
	- 서비스 디스커버리 / 멤버십 : 어떤 인스턴스가 어느 IP로 떠 있는지 등록·조회.
- ZooKeeper에 올리는 건 천천히 변하는 데이터다. 누가 리더인가, 파티션이 어느 노드에 붙었나 정도. 초당 수천 번 바뀌는 런타임 상태를 넣는 곳이 아니다. 그런 건 실제 DB에 두고, 조율만 ZooKeeper에 맡긴다.

## 정리
- 선형성 있는 저장소, 유일성 보장, 원자적 커밋, 전체 순서 브로드캐스트, 잠금 — 전부 합의라는 하나의 문제로 수렴한다.
- 합의는 정족수 투표 + 에포크 번호로 스플릿 브레인 없이 리더를 뽑고, 결정을 번복 없이 확정한다. 대가는 과반 생존과 안정적 네트워크.
- 2PC는 만장일치·블로킹이라 코디네이터 장애에 약하고, 합의는 과반·선출로 그 약점을 넘는다.
- 이걸 직접 짜지 말고 ZooKeeper/etcd에 위임하는 게 현실적이다.
- 반대로, 모든 시스템이 합의를 필요로 하진 않는다. 조율을 피하고 인과성·결국 일관성으로 충분한 경우가 훨씬 많다(11·12장). 합의는 꼭 필요할 때 아껴 쓰는 강력한 도구다.
