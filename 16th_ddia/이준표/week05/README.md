## 트랜잭션
- 데이터 시스템에서는 많은 것이 잘못될 수 있다.
	- DB 소프트웨어/하드웨어가 쓰기 도중 죽거나, 애플리케이션이 도중에 죽거나
	- 네트워크 단절, 동시 클라이언트 간 덮어쓰기, 부분 갱신 상태 읽기, 경쟁 조건(race condition)
- **트랜잭션**은 여러 읽기/쓰기를 하나의 논리적 단위로 묶는 방법.
	- 전체가 성공(commit)하거나, 전체가 실패(abort/rollback)한다.
	- 실패하면 안전하게 재시도할 수 있다. → 애플리케이션 입장에선 부분 실패를 신경 안 써도 된다.
- 트랜잭션은 자연 법칙이 아니다. 1975년 IBM System R에서 처음 도입된 이후 40년간 거의 같은 형태로 유지되어 왔다.
- 모든 애플리케이션에 트랜잭션이 필요한 건 아니다. 성능/가용성을 위해 트랜잭션 보장을 약화시키거나 포기하는 경우도 있다.

## ACID
- 트랜잭션이 제공하는 안전성 보장을 흔히 **ACID**(Atomicity, Consistency, Isolation, Durability)로 표현.
	- 1983년 Theo Härder와 Andreas Reuter가 도입한 용어.
- 그러나 실무에서 ACID 구현은 DB마다 다르다. 특히 격리(Isolation)의 의미가 모호하다.
	- "ACID 호환"이라고 광고해도 실제 보장 수준은 천차만별. ACID는 거의 마케팅 용어가 되었다.
- ACID에 못 미치는 시스템을 **BASE**(Basically Available, Soft state, Eventual consistency)라고 부르기도 하는데, ACID보다 더 모호하다. (사실상 "ACID가 아님" 정도의 의미)

### Atomicity (원자성)
- 여러 쓰기 도중 오류가 나면, 트랜잭션은 어보트되고 지금까지 한 쓰기는 **모두 폐기**된다.
	- 즉 all-or-nothing 보장.
- 동시성과는 관련이 없다. 동시 접근 문제는 격리(I)가 다룬다.
- 책에서는 차라리 **abortability**라는 단어가 더 적절했을 거라고 말한다.

### Consistency (일관성)
- 데이터에 대한 **불변식(invariant)** 이 항상 참이어야 한다는 애플리케이션 수준의 개념.
	- EX. 회계 시스템에서 모든 계좌의 credit과 debit 합이 0이어야 한다.
- 이 일관성은 DB가 보장하는 게 아니라 **애플리케이션 책임**.
	- DB는 유일성 제약, 외래 키 제약 정도만 강제할 수 있다.
- 따라서 사실 ACID에서 C는 좀 어울리지 않는 글자다.
	- 책 각주: Joe Hellerstein은 "C는 그냥 약어를 만들기 위해 끼워 넣은 것"이라고 평했다.

### Isolation (격리성)
- 동시에 실행되는 트랜잭션끼리 서로 간섭하지 않게 하는 것.
- 교과서적 정의는 **직렬성(serializability)**. 동시에 실행되도 마치 한 번에 하나씩 직렬로 실행된 것과 같은 결과를 보장.
- 그러나 직렬성 격리는 성능 비용이 커서 실무에서 거의 안 쓴다.
	- Oracle 11g는 직렬성 격리를 구현조차 안 하고, "serializable"이란 이름으로 사실 스냅숏 격리를 제공한다.

### Durability (지속성)
- 트랜잭션이 commit 되면, 그 데이터는 하드웨어 결함이나 DB 충돌이 발생해도 손실되지 않는다.
- 단일 노드에서는 보통 디스크/SSD에 쓰고 WAL로 보강.
- 복제된 DB에서는 일정 수의 노드에 복제 완료가 되어야 한다.
- 완벽한 지속성은 존재하지 않는다. 모든 디스크와 백업이 동시에 파괴되면 DB가 할 수 있는 일은 없다.

> [!tip] 복제 vs 디스크
> 디스크에 쓰면 장비가 죽었을 때 데이터에 접근이 안 되지만 잃지는 않는다. 복제하면 다른 노드가 살아있어 가용성은 유지되지만, 비동기 복제 중 리더가 죽으면 최근 쓰기는 잃을 수 있다. SSD의 fsync도 항상 보장된 동작은 아니다(전원 차단 시 보장 위반 사례 존재). 결국 디스크 + 원격 복제 + 백업을 함께 써야하고, 모든 "보장"은 의심하면서 받아야 한다.

## 단일 객체 vs 다중 객체 연산

### 단일 객체 쓰기
- 한 객체에 대한 쓰기도 원자성과 격리성이 필요하다.
	- EX. 20KB JSON 문서를 쓰는 중 네트워크가 끊기면 10KB짜리 파편이 DB에 남을 수 있다.
- 거의 모든 저장소 엔진은 **단일 객체 수준**에서는 원자성과 격리성을 제공한다.
	- 원자성: 충돌 복구용 로그
	- 격리성: 객체 단위 잠금
- 일부 DB는 더 복잡한 단일 객체 연산도 제공한다.
	- 증가(increment) 연산: read-modify-write 사이클을 없애줌.
	- **compare-and-set(CAS)**: 값이 동시 변경되지 않은 경우에만 쓰기 허용.
- 이런 단일 객체 연산은 "경량 트랜잭션" 또는 "ACID"로 마케팅되곤 하는데, 일반적인 의미의 트랜잭션은 아니다.

### 다중 객체 트랜잭션이 필요한 경우
- 다중 객체 트랜잭션은 파티션을 가로질러 구현하기 어려워서, 분산 데이터스토어에서는 종종 포기된다.
- 그러나 여러 객체에 걸친 쓰기 조율이 필요한 경우가 있다.
	- 관계형 모델에서 외래 키 참조 무결성 유지
	- 문서 모델에서 비정규화된 데이터의 동기화
	- 보조 색인 갱신 (보조 색인은 트랜잭션 관점에서 별도 객체)

## 오류 처리와 어보트
- 트랜잭션의 핵심 기능: 문제가 생기면 어보트하고 **재시도**할 수 있다.
- 다만 재시도는 **DB가 알아서 해주지 않는다.** 보통 애플리케이션 코드가 명시적으로 retry 루프를 작성해야 한다.
	- Rails의 ActiveRecord, Django 같은 인기 ORM 프레임워크들도 어보트된 트랜잭션을 자동 재시도하지 않는다.
	- 에러가 그냥 예외로 스택을 타고 올라가버려서 사용자 입력은 버려지고 에러 메시지만 뜬다.
	- 책에서는 "어보트의 본래 목적이 안전한 재시도인데 이걸 안 살리는건 안타까운 일(a shame)"이라고 평한다.
- 그러나 무지성 재시도가 정답은 아니다.
	- 트랜잭션이 사실 성공했는데 네트워크가 ack를 잃으면, 재시도하면 두 번 실행됨 (애플리케이션 수준 중복 제거 필요).
	- 오류 원인이 과부하면 재시도가 상황을 더 악화시킨다. → 지수 백오프, 재시도 횟수 제한.
	- 일시적 오류(데드락, 격리 위반, 네트워크 끊김, 페일오버)만 재시도 가치가 있다. 영구적 오류(제약 위반)는 재시도해도 의미 없다.
	- 트랜잭션이 DB 외부에 부수 효과(이메일 발송 등)를 일으키면 재시도 시 중복 부수 효과 발생.
	- 클라이언트 자체가 재시도 중 죽으면 데이터 손실.

## 약한 격리 수준
- 직렬성 격리는 성능 비용이 커서, 대신 더 약한 격리 수준이 흔히 쓰인다.
- 약한 격리는 일부 동시성 문제만 막고 다른 문제는 허용한다. 이해하기 어렵고 미묘한 버그를 유발한다.
- 약한 격리에 의한 동시성 버그는 실제로 큰 금전적 손실, 감사 조사, 고객 데이터 손상까지 일으킨 사례가 있다.
- "ACID DB를 쓰면 된다"는 흔한 조언은 오해다. 인기 있는 관계형 DB들도 대부분 약한 격리를 사용한다.

### Read Committed (커밋 후 읽기)
가장 기본적인 격리 수준. 두 가지를 보장한다.
1. **더티 읽기 없음** : 커밋된 데이터만 읽을 수 있다.
2. **더티 쓰기 없음** : 커밋된 데이터만 덮어쓸 수 있다.

#### 더티 읽기(Dirty Read) 방지
- 한 트랜잭션이 쓴 데이터를 commit 전에 다른 트랜잭션이 읽으면 더티 읽기.
- 막아야 하는 이유:
	- 여러 객체를 갱신하는 트랜잭션의 부분만 보이면 사용자가 혼란.
	- 어보트되면 그 데이터는 사실 존재한 적이 없는데, 다른 트랜잭션이 봤다면 결과를 추론하기 매우 어렵다.

#### 더티 쓰기(Dirty Write) 방지
- 아직 커밋되지 않은 값을 다른 트랜잭션이 덮어쓰면 더티 쓰기.
- 보통 첫 쓰기 트랜잭션이 끝날 때까지 두 번째 쓰기를 **지연**시키는 방식으로 막는다.
- 막지 않으면 여러 객체에 걸친 갱신이 뒤섞일 수 있다.
	- EX. 자동차 판매 사이트에서 listings 테이블 갱신은 Bob이 이기고, invoices 테이블 갱신은 Alice가 이기는 상황.

#### 구현 방식
- Read Committed는 Oracle 11g, PostgreSQL, SQL Server 2012, MemSQL 등 많은 DB의 **기본 격리 수준**.
- 더티 쓰기 방지: 객체 단위 **행 락(row-level lock)**. 트랜잭션이 끝날 때까지 락을 유지한다.
- 더티 읽기 방지: 락을 쓰면 긴 쓰기 하나가 많은 읽기를 막아 응답 시간이 망가진다. → 그래서 대부분 DB는 **이전 커밋 값과 새 값을 둘 다 유지**해서, 쓰기 중인 동안 읽는 트랜잭션에는 옛 값을 보여준다.

### 스냅숏 격리(Snapshot Isolation) / MVCC
- Read Committed로도 못 막는 문제가 있다.

> [!tip] 읽기 스큐(read skew) 예시
> Alice가 두 계좌(각 $500)에 총 $1,000를 가지고 있다. 다른 트랜잭션이 $100을 옮기는 동안 Alice가 잔액을 조회하면, 한 계좌는 $500(입금 전), 다른 계좌는 $400(출금 후)으로 보일 수 있다. Alice 입장에서는 $100이 사라진 것처럼 보인다.
> 이 비정상을 **non-repeatable read** 또는 **읽기 스큐(read skew)** 라고 한다. Read Committed에서는 이를 허용한다.

- 잠깐의 이상 현상은 새로고침하면 사라지지만, 다음 상황에서는 치명적이다.
	- **백업** : 백업 중간에 일관성이 깨진 스냅숏이 만들어지면 복구 시 영구 불일치.
	- **분석 질의/무결성 검사** : 큰 범위 스캔이 시점별로 다른 데이터를 보면 결과가 엉망.

**스냅숏 격리**는 이 문제의 가장 일반적인 해법이다.
- 각 트랜잭션이 DB의 **일관된 스냅숏**, 즉 특정 시점에 커밋된 데이터만 읽는다.
- 이후 다른 트랜잭션이 데이터를 바꿔도 영향 없음.
- 또 다른 이름 : **다중버전 동시성 제어(MVCC)**

#### 구현 방식 (MVCC)
- 지원 제품 : PostgreSQL, InnoDB(MySQL), Oracle, SQL Server 등.
- Read Committed처럼 쓰기에는 락을 쓰지만, **읽기에는 락을 쓰지 않는다.**
	- 핵심 원칙 : *읽기는 쓰기를 막지 않고, 쓰기는 읽기를 막지 않는다.*
- 한 객체에 대해 여러 커밋 버전을 동시에 유지해야 한다. (Read Committed는 2개만 유지하면 됐지만, 스냅숏 격리는 그 이상 필요)
- PostgreSQL 구현 :
	- 트랜잭션마다 단조 증가하는 트랜잭션 ID 할당.
	- 각 행에 `created by`(생성 tx id)와 `deleted by`(삭제 tx id) 필드.
	- 업데이트는 내부적으로 delete + create로 변환.
	- 더 이상 어떤 트랜잭션도 접근 못 하면 GC가 행을 실제로 제거.

#### 가시성 규칙
1. 트랜잭션 시작 시점에 **진행 중**인 다른 트랜잭션 목록을 기록한다. 그 트랜잭션들의 쓰기는 (나중에 커밋되어도) 무시한다.
2. 어보트된 트랜잭션의 쓰기는 무시한다.
3. 현재 트랜잭션보다 **늦은 ID**의 트랜잭션 쓰기는 (이미 커밋됐어도) 무시한다.
4. 그 외 모든 쓰기는 보인다.

#### Repeatable Read와 이름 혼란
- 스냅숏 격리를 Oracle은 `serializable`이라 부르고, PostgreSQL/MySQL은 `repeatable read`라 부른다.
- SQL 표준은 1975년 System R 기반이라 스냅숏 격리 개념이 없다. 그래서 표준의 `repeatable read`와 매핑이 흐릿하다.
- IBM DB2의 `repeatable read`는 사실 직렬성을 의미한다.
- **결론** : repeatable read가 뭘 보장하는지 아무도 모른다.

### 갱신 손실(Lost Update) 방지
- 동시 쓰기-쓰기 충돌 중 가장 잘 알려진 문제.
- 패턴: read-modify-write 사이클 (값 읽고 → 수정 → 다시 쓰기).
	- EX. 카운터 증가, JSON 일부 갱신, 위키 페이지 동시 편집

#### 원자적 쓰기 연산
- 가능하면 read-modify-write를 DB의 **원자적 연산**으로 대체.
	- EX. `UPDATE counters SET value = value + 1 WHERE key = 'foo'`
- MongoDB는 JSON 일부 갱신용 원자 연산을 제공하고, Redis는 우선순위 큐 같은 자료구조용 원자 연산을 제공.
- 구현은 보통 객체에 **배타적 락**을 거는 방식. (커서 안정성, cursor stability)
- ORM 프레임워크는 의도치 않게 read-modify-write 코드를 쓰기 쉬워서 주의 필요.

#### 명시적 잠금
- 원자 연산으로 표현하기 어려운 경우, 애플리케이션이 명시적으로 잠근다.
	- EX. SQL의 `SELECT ... FOR UPDATE`
- 잠금을 빼먹기 쉬워서 경쟁 조건이 생기기 쉽다.

#### 갱신 손실 자동 감지
- 락 대신, 트랜잭션을 병렬로 실행하다가 갱신 손실이 감지되면 어보트 후 재시도.
- 스냅숏 격리와 잘 결합된다.
- PostgreSQL의 repeatable read, Oracle의 serializable, SQL Server의 snapshot isolation은 갱신 손실을 **자동 감지**한다.
	- 그러나 MySQL/InnoDB의 repeatable read 는 **감지하지 않는다**.
- 장점 : 애플리케이션 코드가 락이나 원자 연산을 신경 안 써도 됨. 락을 빼먹어서 생기는 버그를 자동으로 막을 수 있다.

> [!tip] 낙관적 동시성 제어(OCC)와의 관계
> "병렬로 실행 → 끝에서 충돌 감지 시 abort/retry" 는 **낙관적 동시성 제어(Optimistic Concurrency Control)** 의 핵심 패턴이다. 책은 7장 전반에서는 이 용어를 안 쓰다가, 7장 후반의 SSI 설명에서 "Pessimistic vs. optimistic concurrency control"이라는 섹션으로 정식 분류한다.
>
> 흔히 말하는 "낙관적 락"과는 구현 위치가 다르다.
>
> |        | 애플리케이션 레벨 낙관적 락           | 갱신 손실 자동 감지              |
> | ------ | -------------------------- | ----------------------- |
> | 누가 감지  | 애플리케이션/ORM (EX. JPA `@Version`) | DB가 자동으로                |
> | 방식     | `version` 컬럼 + `UPDATE ... WHERE version = ?` (사실상 CAS) | 스냅숏 격리의 tx ID로 read-modify-write 충돌 감지 |
> | 입도     | 보통 단일 행                    | 트랜잭션 전체의 read-modify-write 사이클 |
> | 코드 작업  | 버전 필드/조건을 개발자가 직접 작성       | 트랜잭션만 열면 자동             |
>
> 즉 둘은 같은 OCC 패밀리이지만 같은 개념은 아니다. 7장 전반의 **CAS는 애플리케이션 레벨 OCC의 가장 단순한 형태**, **갱신 손실 자동 감지는 DB 레벨 OCC** 라고 정리하면 깔끔하다.

#### Compare-and-Set (CAS)
- 트랜잭션을 제공하지 않는 DB에서 흔히 쓰는 단일 객체 원자 연산.
- 마지막으로 읽은 이후 값이 변하지 않은 경우에만 쓰기를 허용한다.
- 일치하지 않으면 update가 효과 없고, read-modify-write를 재시도해야 한다.

```sql
UPDATE wiki_pages SET content = 'new content'
WHERE id = 1234 AND content = 'old content';
```

- 주의: WHERE 절이 **옛 스냅숏**에서 읽도록 허용된 DB라면, 다른 동시 쓰기가 발생했어도 조건이 참이 되어 갱신 손실을 못 막을 수 있다. 사용 전에 DB의 CAS 동작이 안전한지 확인해야 한다.

#### 충돌 해소와 복제
- 다중 리더/리더 없는 복제 DB는 동시에 여러 노드에서 쓰기가 일어나고 비동기로 복제되므로, "단일 최신 사본"이라는 전제가 깨진다. → 락이나 CAS가 통하지 않는다.
- 대신 동시 쓰기로 인한 **여러 버전(siblings)** 을 만들고, 애플리케이션 코드나 특수 자료구조가 사후에 병합한다.
- 원자 연산이 **교환 법칙(commutative)** 을 만족하면 복제 환경에서도 잘 동작한다.
	- EX. 카운터 증가, 집합에 원소 추가
	- Riak 2.0의 CRDT 데이터 타입이 이 아이디어를 활용해서 복제 간 갱신 손실을 방지한다.
- **LWW(Last Write Wins)** 충돌 해소는 갱신 손실에 취약하다. 그런데 안타깝게도 많은 복제 DB의 기본 설정이다.
