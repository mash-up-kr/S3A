## 분산 시스템의 골칫거리
- 단일 컴퓨터에서 돌아가는 프로그램은 보통 결정론적으로 동작한다. 작동하거나, 작동 안 하거나.
- 컴퓨터는 의도적으로 그렇게 설계된다. 내부 결함이 발생하면 잘못된 결과를 내놓느니 차라리 완전히 죽는 쪽을 택한다.
- 분산 시스템은 다르다. 네트워크로 연결된 여러 컴퓨터에서는 그 이상화된 모델이 깨진다.
- 일부 노드는 멀쩡한데 다른 일부는 알 수 없는 방식으로 망가져 있는 상태를 부분 장애(partial failure)라고 한다.
- 부분 장애는 비결정적이다. 같은 요청을 보내도 어떤 때는 성공, 어떤 때는 알 수 없는 이유로 실패한다. 심지어 성공인지 실패인지 알 수 없는 경우도 있다.
- 이 비결정성이 분산 시스템을 다루기 어렵게 만드는 핵심.

## 결함과 부분 장애

### 클라우드 컴퓨팅 vs 슈퍼컴퓨팅
- 대규모 컴퓨팅을 짓는 두 가지 철학.
- 슈퍼컴퓨팅(HPC): 수천 개 CPU로 과학 계산. 노드 하나가 실패하면 전체 클러스터를 멈추고 마지막 체크포인트에서 재시작. 부분 장애를 전체 장애로 격상시켜 처리. 단일 노드 컴퓨터에 가깝다.
- 클라우드 컴퓨팅: 다중 테넌트 데이터센터, 상용 하드웨어, IP 네트워크, 탄력적 자원 할당.
- 인터넷 서비스는 슈퍼컴퓨터와는 매우 다른 요구를 갖는다.
	- 온라인 서비스라 멈출 수 없다. 슈퍼컴처럼 "수리 후 재개"는 받아들일 수 없다.
	- 상용 하드웨어라 장애율이 높다.
	- 노드가 많아질수록 어느 한 부품이 망가져 있을 확률은 거의 1이 된다.
	- 장애를 견디면서 동작해야 롤링 업그레이드, 노드 교체 같은 운영이 가능.
	- 지리적으로 분산되면 인터넷을 거치므로 더더욱 느리고 신뢰성 떨어진다.
- 결론: 분산 시스템을 만들려면 부분 장애의 가능성을 받아들이고, 소프트웨어에 내결함성 메커니즘을 넣어야 한다. 신뢰성 없는 부품으로 신뢰성 있는 시스템을 만드는 것.

### 신뢰성 없는 부품으로 신뢰성 있는 시스템 만들기
- 직관과 달리, 신뢰성 없는 기반 위에 더 신뢰성 있는 시스템을 만드는 건 컴퓨팅의 오랜 아이디어.
- EX. 오류 정정 코드는 가끔 비트가 뒤집히는 채널에서 정확한 데이터 전송을 가능하게 한다.
- EX. IP는 패킷을 잃거나/중복하거나/뒤집기도 하지만, TCP가 그 위에 더 신뢰성 있는 전송 계층을 제공한다.
- 단 한계는 있다. TCP가 패킷 손실은 가릴 수 있지만 네트워크 지연 자체를 마법처럼 없앨 수는 없다.

## 신뢰성 없는 네트워크
- 우리가 다루는 분산 시스템은 비공유(shared-nothing) 시스템이다. 각 머신은 자기 메모리/디스크만 갖고 있고, 다른 머신의 메모리/디스크에 접근하려면 네트워크 요청을 보내는 수밖에 없다.
- 인터넷과 대부분의 데이터센터 내부망은 비동기 패킷 네트워크. 한 노드가 다른 노드에 패킷을 보내도 도착 시점이나 도착 여부 자체를 보장하지 않는다.
- 요청을 보내고 응답이 없을 때 일어났을 수 있는 일들
	1. 요청이 손실됨 (누가 케이블을 뽑았을 수도)
	2. 요청이 큐에 갇혀 늦게 도착
	3. 원격 노드가 죽음
	4. 원격 노드가 일시적으로 응답 안 함 (긴 GC pause 등)
	5. 원격 노드가 처리는 했는데 응답이 손실됨
	6. 원격 노드가 처리도 했고 응답도 보냈는데 지연됨
- 송신자는 어느 경우인지 구분할 수 없다. 보통은 타임아웃으로 처리. 그러나 타임아웃이 발생해도 원격 노드가 요청을 받았는지 여부는 알 수 없다.

### 네트워크 결함의 현실
- 잘 관리된 데이터센터도 네트워크 결함은 흔하다. 한 중간 규모 데이터센터 연구에서 월 12건의 네트워크 결함이 보고됨. 절반은 단일 머신, 나머지 절반은 랙 전체가 끊김.
- 이중화 장비를 두어도 결함이 크게 줄지 않는다. 사람의 실수(설정 오류 등)가 큰 비중이라.
- AWS EC2 같은 퍼블릭 클라우드는 일시적 네트워크 장애가 잦은 것으로 유명.
- 한 방향으로는 잘 가는데 반대 방향은 모든 패킷을 떨궈버리는 NIC 같은 황당한 결함 사례도 있다.

> [!tip] 네트워크 파티션(network partition / netsplit)
> 네트워크 결함으로 시스템의 일부가 나머지와 단절된 상태. 6장의 파티셔닝(샤딩)과 헷갈리지 않게 책에서는 일반 용어 "네트워크 결함"을 더 자주 쓴다.

### 결함 감지
- 많은 시스템이 결함 노드를 자동으로 감지해야 한다. (LB의 헬스 체크, 리더 페일오버 등)
- 명시적 피드백을 받을 수 있는 경우
	- 포트는 열려있는데 프로세스가 죽었으면 OS가 RST/FIN으로 TCP 연결 거부
	- 프로세스가 종료되면 다른 노드에 알리는 스크립트를 둘 수 있다 (HBase가 그렇게 함)
	- 데이터센터 스위치 관리 인터페이스 직접 조회
	- 라우터가 ICMP Destination Unreachable로 알려주는 경우
- 그러나 빠른 피드백을 보장할 수 없다. TCP가 ack를 줬어도 애플리케이션이 처리하기 전에 죽었을 수 있다. 결국 확신을 가지려면 애플리케이션 레벨의 응답이 필요하다.

### 타임아웃과 기약 없는 지연
- 타임아웃이 결함 감지의 거의 유일한 수단이라면, 얼마로 잡아야 할까?
- 너무 길면: 사용자 대기/에러 메시지가 길어진다.
- 너무 짧으면: 일시적 지연을 결함으로 오인할 위험. 멀쩡한 노드를 죽었다고 단정하면 그 부하가 다른 노드로 가서 시스템을 망가뜨릴 수 있다. 극단적으로는 연쇄 장애.
- 만약 네트워크가 최대 지연 d 를 보장하고, 노드가 r 시간 안에 응답을 보장하면 2d + r 이 합리적 타임아웃이 된다.
- 그런데 현실의 비동기 네트워크는 기약 없는 지연(unbounded delay)을 갖는다. 상한이 없다. 그래서 "정답"인 타임아웃 값은 없고, 실험적으로 결정할 수밖에 없다.

### 네트워크 혼잡과 큐잉
- 네트워크 지연 변동의 가장 큰 원인은 큐잉(queueing).
	- 여러 노드가 같은 목적지로 동시에 패킷을 보내면 스위치 큐에 쌓인다. 큐가 가득 차면 드롭.
	- 목적지 머신의 CPU가 바쁘면 OS가 요청을 큐잉.
	- VM 환경에서는 VM 자체가 잠시 멈춰있는 동안 패킷이 hypervisor에 버퍼링된다.
	- TCP의 흐름 제어가 발신 측에서도 큐잉을 추가.
	- 패킷이 timeout 내에 ack 안 되면 재전송. 애플리케이션은 패킷 손실은 안 보지만 그로 인한 지연은 본다.
- 시스템이 최대 용량 근처면 큐가 빠르게 쌓이고 지연이 크게 튄다. 여유 있는 시스템은 큐를 잘 비운다.
- 멀티 테넌트 환경에서는 옆 테넌트의 부하(noisy neighbor)가 내 지연에 영향을 준다.
- 그래서 타임아웃은 보통 실험적으로 결정. 더 좋은 방법은 응답 시간 분포를 측정해서 자동으로 조정하는 것 (Akka, Cassandra가 쓰는 Phi Accrual failure detector 등).

### 동기식 vs 비동기식 네트워크
- 전화망(고정 회선, 비-VoIP)은 매우 신뢰성 있다. 음성이 끊기거나 통화가 떨어지는 일이 드물다.
- 전화 통화를 걸 때 회선(circuit)을 만든다. 두 끝 사이에 고정된 대역폭이 통화 내내 예약된다. EX. ISDN은 초당 4000프레임, 통화당 매 프레임의 16비트를 차지. → 큐잉 없음 → 종단간 지연이 고정 (bounded delay).
- 데이터센터/인터넷은 이렇게 못 할까? 못한다. Ethernet과 IP는 패킷 교환(packet-switched) 프로토콜이라 큐잉과 unbounded delay를 갖는다.
- 패킷 교환을 쓰는 이유: 버스트 트래픽에 최적화. 웹페이지 요청이나 파일 전송은 대역폭이 가변적이라 회선 예약이 낭비. TCP가 가용 대역폭에 동적으로 적응하는 게 더 효율적.
- ATM, InfiniBand 같은 하이브리드도 있지만 멀티 테넌트 데이터센터/퍼블릭 클라우드에서는 활성화되지 않는다.
- 결론: 네트워크 지연 변동은 자연 법칙이 아니라, 비용-효용의 trade-off다. 정적 자원 분할은 지연을 보장하는 대신 활용도가 낮고 비싸다. 동적 분할은 활용도가 높고 싸지만 지연이 변동한다.

## 신뢰성 없는 시계
- 분산 시스템에서 시간은 까다롭다. 네트워크 지연 때문에 메시지가 언제 도착할지 모르고, 각 머신마다 자기 시계가 있어 서로 약간씩 어긋난다.
- 컴퓨터의 시계는 보통 quartz 진동자. 완벽히 정확하지 않다.
- 동기화 방법으로 NTP(Network Time Protocol)를 흔히 쓴다. NTP 서버는 GPS 같은 더 정확한 시간 소스로부터 시간을 얻는다.

### 단조 시계 vs 일중 시계
- 현대 컴퓨터에는 두 종류의 시계가 있다.
- **일중 시계(time-of-day clock)**: 달력상의 현재 날짜/시간을 반환. EX. Linux `clock_gettime(CLOCK_REALTIME)`, Java `System.currentTimeMillis()`.
	- NTP와 동기화되므로 다른 머신과 비교 가능하지만, 로컬 시계가 NTP 서버보다 너무 앞서가면 강제로 되돌려져서 시간이 뒤로 점프하는 일이 발생.
	- 경과 시간 측정에는 부적합.
- **단조 시계(monotonic clock)**: 지속 시간(타임아웃, 응답 시간 등) 측정용. EX. Linux `clock_gettime(CLOCK_MONOTONIC)`, Java `System.nanoTime()`.
	- 항상 앞으로만 간다.
	- 절대값은 의미 없다 (부팅 후 ns 같은 임의의 값). 두 머신 간 비교도 의미 없다.
	- NTP가 로컬 quartz가 빠르다/느리다는 걸 감지하면 단조 시계의 진행 속도를 조정(slewing)할 수 있지만, 점프시키지는 않는다.

### 시계 동기화와 정확도
- 단조 시계는 동기화 필요 없지만, 일중 시계는 NTP 등 외부 소스가 있어야 의미가 있다. 그런데 그 동기화가 생각보다 신뢰성 없다.
	- quartz 시계가 drift 한다. Google은 자기 서버를 200ppm 정도로 가정. 30초에 한 번 동기화하면 6ms drift, 하루 한 번이면 17초 drift.
	- 로컬 시계가 NTP 서버와 너무 차이나면 동기화를 거부하거나 강제 리셋. 그러면 시간이 점프.
	- NTP 서버에서 방화벽으로 차단되면 한참 동안 눈치 못 챔.
	- NTP 동기화 정확도는 네트워크 지연에 종속. 인터넷 통한 동기화의 최저 오차는 35ms 수준, 가끔 1초 가까이 튄다.
	- 일부 NTP 서버는 시간이 몇 시간씩 틀린 경우도 있다. NTP 클라이언트가 outlier를 무시하긴 하지만, 모르는 서버에 시스템 정확성을 거는 건 좀 께름칙하다.
	- 윤초(leap second). 59초 또는 61초짜리 분이 생기면, 윤초를 가정하지 않은 시스템이 망가진다. 윤초로 큰 시스템이 죽은 사례가 여럿. 해결책으로 NTP 서버가 하루에 걸쳐 윤초 보정을 분산(smearing)하는 방법.
	- VM에서는 하드웨어 시계가 가상화되어, VM이 잠시 멈춘 동안 시계가 갑자기 앞으로 점프하는 것처럼 보일 수 있다.
	- 모바일/임베디드는 사용자가 일부러 시계를 막 바꿔놓을 수 있다.
- 정확도가 정말 중요하면 가능하긴 하다. MiFID II 규제는 고빈도 트레이딩 펀드에 UTC 기준 100µs 이내 동기화를 요구. GPS + PTP(Precision Time Protocol) + 세심한 배포로 달성.

### 동기화된 시계에 의존하기
- 시계는 단순해 보이지만 함정이 많다. 하루가 정확히 86400초가 아니고, 일중 시계는 뒤로 갈 수 있고, 노드 간 시간이 꽤 다를 수 있다.
- 잘못된 시계는 잘 눈에 띄지 않는다는 게 문제. CPU가 고장 나면 즉시 알 수 있지만, quartz drift나 NTP 오설정은 조용히 데이터를 망가뜨린다.
- 따라서 동기화된 시계에 의존하는 소프트웨어를 쓴다면, 노드 간 시계 오프셋을 모니터링하고 너무 어긋난 노드는 클러스터에서 빼야 한다.

### 이벤트 순서를 위한 타임스탬프
- 여러 노드 간 이벤트 순서를 시계로 정하는 건 위험.
- 다중 리더 복제에서 시계가 3ms 정도밖에 안 어긋난 잘 동기화된 환경에서도, write x = 1 의 타임스탬프가 42.004이고 write x = 2(인과적으로 나중)의 타임스탬프가 42.003인 상황이 가능. → LWW가 x = 2를 버리고 x = 1 을 살리는 데이터 손실.
- LWW의 문제
	- 시계가 느린 노드가 시계 빠른 노드의 값을 덮어쓸 수 없다. 그 시간 격차만큼의 쓰기가 조용히 버려진다.
	- 순차적 쓰기와 진짜 동시 쓰기를 구분 못 함. 인과성 추적을 위해 version vector 같은 추가 메커니즘 필요.
	- 같은 ms에 두 노드가 같은 타임스탬프를 만들 수 있다. tiebreaker(랜덤값)가 필요하지만 그것도 인과성을 깨뜨릴 수 있다.
- NTP 동기화 정확도 자체가 네트워크 RTT에 종속되므로, 이런 순서 오류를 NTP만으로 막을 수는 없다.
- 대안은 논리 시계(logical clock). 발진자가 아니라 카운터 기반. 시각이 아닌 상대적 순서(어느 이벤트가 어느 이벤트보다 먼저인지)만 측정. 9장에서 다시 다룸.

### 시계 읽기는 신뢰 구간이다
- 마이크로초 단위로 시계를 읽을 수 있어도, 그 값이 실제로 그 정확도인지는 별개. NTP로 매분 동기화해도 drift만 수 ms 차이가 날 수 있다.
- 시계 값을 점이 아닌 신뢰 구간(confidence interval)으로 봐야 한다. EX. "95% 확률로 현재 시각이 10.3 ~ 10.5초 사이".
- 그러나 대부분의 시스템은 이 불확실성을 노출하지 않는다. `clock_gettime()`은 그 값의 오차를 알려주지 않는다.
- 예외는 Google의 Spanner에서 쓰는 TrueTime API. 현재 시각을 묻으면 [earliest, latest] 두 값을 돌려준다. 구간의 너비는 마지막 정확한 시간 소스 동기화 이후 quartz drift에 비례.

### 전역 스냅숏을 위한 동기화 시계
- 7장 스냅숏 격리는 단조 증가하는 트랜잭션 ID가 필요. 단일 노드에서는 카운터로 충분.
- 여러 데이터센터에 걸친 분산 DB에서 모든 파티션을 가로지르는 단조 트랜잭션 ID 생성은 어렵다. 조율(coordination)이 필요하기 때문에 작은 트랜잭션이 많으면 병목.
- Spanner의 접근: TrueTime의 신뢰 구간을 트랜잭션 ID로 활용.
	- 두 트랜잭션의 신뢰 구간이 겹치지 않으면(A_latest < B_earliest) B가 A 이후라고 단정 가능.
	- 겹치면 순서를 단정할 수 없다.
- Spanner는 read-write 트랜잭션을 commit 하기 전에 신뢰 구간 길이만큼 일부러 기다린다. 그래야 이후 트랜잭션의 구간이 안 겹친다.
- 대기 시간을 짧게 유지하려면 시계 불확실성을 최소화해야 한다. 그래서 Google은 각 데이터센터에 GPS 수신기 또는 원자 시계를 둬서 약 7ms 이내 동기화.
- 이런 시계 동기화 기반 분산 트랜잭션은 활발한 연구 영역이지만, Google 외부의 메인스트림 DB에는 아직 거의 안 들어와 있다.
