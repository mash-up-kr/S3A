## 파티셔닝(Partitioning)
- 데이터셋이 너무 커서 단일 장비로 감당하기 어려울 때, 데이터를 여러 조각(파티션, 샤드)으로 쪼개 여러 노드에 분산시키는 방식.
- 다른 이름들: MongoDB/Elasticsearch/SolrCloud에서는 **shard**, HBase에서는 **region**, BigTable에서는 **tablet**, Cassandra와 Riak에서는 **vnode**, Couchbase에서는 **vBucket**.
- 파티셔닝의 주된 이유는 **확장성(scalability)** 이다.
	- 대용량 데이터셋을 여러 디스크에 분산하고, 질의 부하를 여러 프로세서에 분산할 수 있다.
- 각 파티션은 그 자체로 작은 DB처럼 동작하며, 한 번에 한 파티션만 다루는 질의는 노드 수에 비례해 처리량이 증가한다.

## 파티셔닝과 복제
- 파티셔닝은 데이터를 쪼개는 것, 복제는 같은 데이터를 여러 노드에 사본으로 두는 것.
- 한 노드는 여러 파티션을 저장할 수 있고, 각 파티션은 다른 노드에 복제본(리더 + 팔로워)을 갖는다.
- 한 노드가 죽어도 복제본을 가진 다른 노드가 그 파티션을 서비스할 수 있다.
- 파티셔닝 방식의 선택은 복제 방식과 대체로 독립적이라, 이 장에서는 복제는 무시하고 파티셔닝만 다룬다.

## 키-값 데이터 파티셔닝

> [!tip] 핵심 목표
> 데이터와 질의 부하를 노드들에 **고르게(evenly)** 분산하는 것

- 파티셔닝이 불공평해서 일부 파티션에 데이터/질의가 몰리는 상황을 **쏠림(skew)** 이라고 한다.
- 부하가 매우 높은 파티션을 **핫스팟(hot spot)** 이라고 부른다.
- 가장 단순한 방법은 레코드를 노드에 무작위로 할당하는 것.
	- 부하는 잘 분산되지만, 특정 키를 읽으려면 모든 노드를 뒤져야 한다. → 비효율
- 따라서 키를 기반으로 결정론적으로 노드를 정하는 방식이 필요하다.

### 키 범위 기준 파티셔닝
- 각 파티션에 연속된 **키 범위**(min ~ max)를 할당한다.
	- EX. 백과사전을 A~B권, C~D권, ... 으로 나누는 것.
- 범위 사이의 경계가 균등할 필요는 없다. 데이터 분포에 맞춰 경계를 조정한다.
- 경계는 관리자가 수동으로 정하거나 DB가 자동으로 정한다. (HBase, RethinkDB, BigTable이 사용)
- 장점: 키가 정렬되어 있으므로 범위 질의(range query)가 쉽다.
	- EX. 센서 데이터를 타임스탬프 키로 저장하면 "2026년 4월의 모든 데이터" 같은 질의가 효율적이다.
- 단점: 특정 접근 패턴이 핫스팟을 유발할 수 있다.
	- 타임스탬프를 키로 쓰면 오늘 들어오는 모든 쓰기가 같은 파티션으로 몰린다.
	- 해결책 : 키 앞에 다른 항목(예: 센서 이름)을 prefix로 붙여 분산.

### 키의 해시 기준 파티셔닝
- 키 분포가 쏠리는 문제를 피하기 위해 키에 해시 함수를 적용해 그 해시값으로 파티션을 정한다.
- 해시 함수는 일정한 분포로 출력하기만 하면 되므로 암호학적으로 강할 필요는 없다.
	- Cassandra/MongoDB는 MD5, Voldemort는 Fowler-Noll-Vo를 사용.
- 언어에 내장된 해시 함수는 파티셔닝에 부적합할 수 있다. 같은 키에도 다른 프로세스에서는 다른 해시값을 반환할 수 있기 때문.
	- EX. Java의 `Object.hashCode()`, Ruby의 `Object#hash`
- 장점 : 키가 균등하게 흩어져 핫스팟을 피하기 좋다.
- 단점 : 정렬 순서를 잃기 때문에 효율적인 범위 질의가 불가능하다. → 모든 파티션에 질의를 보내야 함

> [!tip] Cassandra의 절충안 - 컴파운드 기본 키
> 컴파운드 기본 키의 **첫 번째 칼럼만** 해싱해서 파티션을 결정하고, 나머지 칼럼은 SSTable 내부 정렬 순서로 사용한다. 따라서 첫 칼럼이 동일한 레코드들에 한해서는 효율적인 범위 스캔이 가능하다. (EX. `(user_id, timestamp)` → 특정 user의 시간 범위 질의 가능)

### 쏠린 작업부하와 핫스팟 완화
- 해시 파티셔닝도 만능이 아니다. 모든 읽기/쓰기가 단 하나의 키로 향하면 (EX. 유명 연예인의 SNS 계정 ID) 그 키 하나가 한 파티션에 핫스팟을 만든다.
- 현재 대부분의 시스템은 이런 극단적 쏠림은 자동으로 보정하지 못한다. → 애플리케이션의 책임.
	- 가장 단순한 기법: 키 끝에 임의의 2자리 숫자를 붙여 100개 키로 분산.
	- 단점: 읽을 때 100개 변형 키를 다 조회해서 결과를 합쳐야 하므로, 어떤 키가 분산이 필요한지 추적해야한다.

## 파티셔닝과 보조 색인
- 키-값 데이터 모델만이라면 위 방법으로 충분하지만, 보조 색인이 들어오면 복잡해진다.
- 보조 색인은 보통 레코드를 유일하게 식별하지 않고, 특정 값이 발생한 위치를 찾는 용도다.
	- EX. `color = red`인 모든 자동차 찾기.
- 두 가지 접근법이 있다 : **문서 기반(Document-based) / 용어 기반(Term-based) 파티셔닝**

### 문서 파티셔닝 보조 색인 (지역 색인)
- 각 파티션이 자기 파티션 안의 문서들에 대해서만 보조 색인을 유지한다.
- 다른 파티션에 어떤 데이터가 있는지 신경 쓰지 않는다. → **지역 색인(local index)**

```
파티션 0:               파티션 1:
docs: 1, 2, 3, 4        docs: 5, 6, 7, 8
secondary index:        secondary index:
  color:red → [1, 3]      color:red → [6]
  color:blue → [2, 4]     color:blue → [5, 7, 8]
```

- 쓰기: 한 문서가 들어가는 파티션 한 곳만 갱신하면 된다. → 단순.
- 읽기: `color=red`를 찾으려면 모든 파티션에 질의를 보내고 결과를 합쳐야 한다.
	- 이를 **scatter/gather** 방식이라고 한다.
- 보조 색인 질의는 비싸지만, 많은 DB에서 채택. (MongoDB, Riak, Cassandra, Elasticsearch, SolrCloud, VoltDB)
- 보조 색인 질의의 지연시간 증폭(tail latency amplification) 문제가 있다. → 가장 느린 파티션을 기다려야 함.

### 용어 파티셔닝 보조 색인 (전역 색인)
- 모든 파티션의 데이터를 아우르는 **전역 색인(global index)** 을 만든 뒤, 그 색인 자체를 파티셔닝한다.
- 색인을 어떤 키로 파티셔닝할지: 용어(term) 자체로.
	- EX. `color:red`로 시작하는 색인 항목은 파티션 0에, `color:blue`는 파티션 1에.

```
파티션 0의 색인:                파티션 1의 색인:
  color:red → [1, 3, 6]         color:blue → [2, 4, 5, 7, 8]
```

- 읽기: 클라이언트는 원하는 용어를 가진 파티션 하나에만 질의하면 된다. → 효율적.
- 쓰기: 한 문서를 쓸 때 여러 파티션의 색인을 갱신해야 한다. (각 보조 색인 항목이 다른 파티션에 있을 수 있음)
	- 결과적으로 분산 트랜잭션이 필요하거나, 비동기 갱신을 받아들여야 한다.

|       | 문서 파티셔닝(지역 색인)         | 용어 파티셔닝(전역 색인)   |
| ----- | --------------------- | ----------------- |
| 쓰기    | 한 파티션만 갱신 (빠름)        | 여러 파티션 갱신 (느림)    |
| 읽기    | scatter/gather (느림)   | 단일 파티션 (빠름)       |
| 일관성   | 강한 일관성 유지 쉬움          | 비동기 갱신 → stale 가능 |
| 사용 제품 | MongoDB, Riak, Cassandra, ES | DynamoDB, Riak Search, Oracle |

## 파티션 재균형화(Rebalancing)
- 시간이 지나면 다음과 같은 변화가 생긴다.
	- 질의 처리량이 늘어 더 많은 CPU가 필요.
	- 데이터셋이 커져 더 많은 디스크/RAM이 필요.
	- 장비가 고장 나서 다른 장비가 그 자리를 대신해야 한다.
- 위 변화에 따라 파티션과 노드의 매핑을 재조정하는 과정을 **재균형화(rebalancing)** 라고 한다.
- 재균형화의 최소 요건
	- 부하(스토리지, 읽기/쓰기)가 노드 사이에 공평하게 분배된다.
	- 재균형화 동안에도 DB는 읽기/쓰기를 받아야 한다.
	- 노드 간 데이터 이동은 최소화 (네트워크/디스크 I/O 부담 감소)

### 안티 패턴 - hash mod N
- 가장 단순해 보이는 방법은 `hash(key) % N`으로 파티션을 정하는 것이다.
- 문제 : 노드 수 N이 변하면 거의 모든 키의 파티션이 바뀐다.
	- N=10일 때 hash=123456 → 파티션 6, N=11일 때 → 파티션 3
	- 재배치 비용이 너무 커서 실무에서 쓸 수 없다.

### 파티션 개수 고정
- 노드 수보다 훨씬 많은 파티션을 만들고, 각 노드에 여러 파티션을 할당한다.
	- EX. 10개 노드의 클러스터에 1000개의 파티션을 만든다. (노드당 100개)
- 새 노드가 추가되면 기존 노드들에서 파티션 몇 개씩을 가져와 채운다.
- 파티션 개수와 키→파티션 매핑은 변하지 않는다. 노드와 파티션의 매핑만 변한다.
- 단점은 파티션 개수를 처음에 잘 정해야한다는 점. 너무 많으면 오버헤드, 너무 적으면 확장 한계.
- Riak, Cassandra(1.2 부터), Elasticsearch, Couchbase, Voldemort가 사용.

### 동적 파티셔닝
- 키 범위 파티셔닝에서 주로 사용. 처음에는 파티션 수가 적게 시작.
- 파티션이 일정 임계치(EX. HBase 10GB)를 넘으면 두 개로 분할, 너무 작아지면 인접 파티션과 병합.
- 데이터 양에 비례해 파티션 수가 자동 조정되는 게 장점.
- 빈 DB에서는 파티션이 1개라 한 노드만 사용된다는 단점이 있다. → HBase/MongoDB는 미리 파티션을 분할하는 **사전 분할(pre-splitting)** 옵션 제공.
- HBase, RethinkDB, MongoDB가 사용.

### 그 외의 재균형화 전략
- Cassandra 1.2 이전에는 Karger의 일관성 해싱 기반의 의사 무작위 파티션 경계를 사용했다.
	- 노드당 작은 파티션 여러 개가 아니라, 노드당 큰 파티션 하나가 큰 해시 범위를 담당하는 방식.
- 부하 분산이 잘 안 되고, 노드 추가 시 기존 노드가 자기 범위를 둘로 쪼개 절반을 새 노드에 줘야 했다. 이 작업이 백그라운드로 처리하기 어려웠다.
- 이런 이유로 Cassandra의 파티셔닝 전략은 위에서 본 **고정 개수 파티션 방식으로 대체**되었다.
- 더 좋은 해시 함수를 쓰면 이런 문제를 피할 수 있지만, 실무에서 가장 널리 쓰이는 방식은 둘 중 하나다.
	- 해시 + 고정 개수 파티션
	- 키 범위에 의한 동적 파티셔닝 (범위 질의가 필요할 때)

> [!tip] 일관성 해싱 (Consistent Hashing)
> Karger 등이 1997년 제안. 무작위로 선택된 파티션 경계를 사용한다. 여기서 consistent 는 복제 일관성/ACID 일관성과 무관하니 헷갈리지 말 것. 책에서는 DB에서는 잘 동작하지 않는다며, 이 용어를 피하고 그냥 "해시 파티셔닝"으로 부르길 권한다.

### 자동 vs 수동 재균형화
- 완전 자동 재균형화 : 운영자 개입 없이 시스템이 알아서 처리.
- 완전 수동 재균형화 : 관리자가 명시적으로 명령해야 동작.
- 자동화는 편리하지만 위험할 수 있다.
	- 노드가 일시적으로 느려져 응답이 늦으면 자동 시스템이 "죽었다"고 판단해 재균형화를 시작.
	- → 더 많은 부하 발생 → 다른 노드도 느려짐 → 연쇄 장애.
- 사람이 루프 안에 있는 것이 안전하다. 더 느리지만 운영 사고를 막을 수 있다.

## 요청 라우팅
- 클라이언트가 어떤 노드로 요청을 보내야 할까? 파티션이 노드들 사이에 재균형화되면 매핑이 변한다.
- 이는 **서비스 디스커버리(service discovery)** 문제의 일종이다.

### 라우팅 결정의 세 가지 방식

```
1) 클라이언트 → 아무 노드 (라운드 로빈)
   노드가 자기에게 없으면 올바른 노드로 전달

2) 클라이언트 → 라우팅 계층(프록시)
   프록시가 어디로 보낼지 결정

3) 클라이언트가 직접 매핑 정보를 알고 있음
   → 올바른 노드로 직접 요청
```

- 모든 방식의 핵심 문제는 **결정자(노드/프록시/클라이언트)가 파티션-노드 매핑의 변화를 어떻게 알 수 있는가** 이다.
- 합의(consensus)에 도달하는 것이 어려운 분산 시스템 문제.

### 코디네이션 서비스 (ZooKeeper)
- 많은 분산 시스템이 별도 코디네이션 서비스인 **ZooKeeper**에 의존해서 클러스터 메타데이터를 추적한다.
- 각 노드는 ZK에 등록하고, ZK는 파티션-노드 매핑을 관리한다.
- 라우팅 계층이나 클라이언트는 ZK를 구독해서 매핑 변경을 통보받는다.
- 사용 제품 : HBase, SolrCloud, Kafka, LinkedIn Espresso 등.
- MongoDB는 ZooKeeper 대신 자체 config 서버와 mongos 데몬을 라우팅 계층으로 둔다.
- Cassandra, Riak는 다른 접근법. **가십 프로토콜(gossip protocol)** 로 노드 간에 클러스터 상태 변경을 전파한다.
	- 외부 의존성이 없는 대신 더 복잡하다.

> [!tip] LinkedIn Espresso의 라우팅
> ZooKeeper 위에 별도의 라우팅 계층(Helix)을 두고, 클라이언트는 이 라우팅 계층에 질의한다. 라우팅 계층은 ZK에서 변경을 구독하므로 매핑이 항상 최신.

### 병렬 질의 실행 (MPP)
- 지금까지의 라우팅은 단순한 단일 키 읽기/쓰기 질의에 대한 것이었다.
- 분석용(OLAP) 데이터베이스, 즉 **MPP(Massively Parallel Processing)** 시스템은 훨씬 복잡한 질의를 다룬다.
	- 여러 조인, 필터링, 그룹화, 집계 연산 포함.
- MPP 질의 옵티마이저는 복잡한 질의를 여러 단계와 파티션으로 분해한 뒤, 병렬로 실행해서 결과를 합친다.
- 데이터 웨어하우스의 빠른 병렬 실행은 11장에서 더 자세히 다룬다.
