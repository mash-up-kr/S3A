## 쓰기 스큐(Write Skew) 와 팬텀
- 더티 쓰기, 갱신 손실 외에도 동시 쓰기 사이에 미묘한 경쟁 조건이 더 있다.
- 쓰기 스큐는 두 트랜잭션이 같은 데이터를 읽고, 각자 그 결과에 따라 서로 다른 객체를 업데이트하는 패턴에서 발생한다.
- 책의 대표 예시 — 병원 당직 의사
	- 규칙: 적어도 한 명의 당직 의사가 있어야 한다.
	- Alice와 Bob 모두 아파서 동시에 휴가 신청 버튼을 누른다.
	- 각자 트랜잭션이 "현재 당직 의사 ≥ 2"를 확인 → 자기 행만 off-call로 변경 → commit.
	- 결과: 당직 의사 0명. 불변식 위반.

### 쓰기 스큐의 특성
- 더티 쓰기도 갱신 손실도 아니다. 두 트랜잭션이 다른 객체(Alice 행 / Bob 행)를 갱신했기 때문.
- 책은 쓰기 스큐를 갱신 손실의 일반화(generalization of lost update)라고 설명한다.
	- 같은 객체에 쓰면 갱신 손실, 다른 객체에 쓰면 쓰기 스큐.
- 막을 방법이 제한적이다.
	- 단일 객체 원자 연산은 객체가 여러 개라 안 통한다.
	- 스냅숏 격리의 갱신 손실 자동 감지도 쓰기 스큐는 못 잡는다. (PostgreSQL repeatable read, Oracle serializable, SQL Server snapshot 모두 마찬가지)
	- 다중 객체에 걸친 제약은 일반 DB가 지원 안 한다. 트리거나 구체화 뷰로 우회 가능하지만 hacky.
	- 차선책: `SELECT ... FOR UPDATE` 로 의존하는 행을 명시적으로 잠그기.

### 다른 예시들
- 회의실 예약: "겹치는 예약 없음을 확인 → INSERT". 두 트랜잭션이 동시에 통과 가능.
- 멀티플레이어 게임: 두 figure가 같은 위치로 이동되는 것을 락만으로는 못 막는다.
- 유저네임 선점: "이름 사용 가능 확인 → 계정 생성". 다행히 유일성 제약으로 해결 가능.
- 더블 스펜딩 방지: 잠정적 지출을 INSERT → 합계 ≥ 0 검사. 두 지출이 동시에 들어가면 합계가 음수가 될 수 있다.

### 팬텀(Phantom)
- 위 예시들의 공통 패턴
	1. SELECT 로 어떤 조건을 만족하는 행을 검색한다.
	2. 결과에 따라 분기 결정.
	3. 결정에 따라 INSERT/UPDATE/DELETE 후 commit.
	4. 만약 1번 SELECT를 다시 한다면 결과가 달라져 있을 것이다. 다른 트랜잭션의 쓰기가 검색 조건 매칭 집합을 바꿨기 때문.
- 한 트랜잭션의 쓰기가 다른 트랜잭션의 검색 결과를 바꾸는 효과를 **팬텀(phantom)** 이라 부른다.
- 스냅숏 격리는 read-only 질의에서는 팬텀을 막지만, read-write 트랜잭션의 쓰기 스큐에서는 막지 못한다.
- 의사 당직 예시는 1번에서 읽은 행이 3번에서 쓰는 행과 같아서 `SELECT FOR UPDATE`로 막을 수 있다. 그러나 나머지 4개 예시는 "조건에 매칭되는 행이 없음을 확인 후 INSERT"라 잠글 행 자체가 없다.

### 충돌 구체화(Materializing Conflicts)
- 잠글 행이 없으면, 인위적으로 잠금용 객체를 DB에 만든다.
	- EX. 회의실 예약 → `(room, 15분 단위 시간슬롯)` 행을 6개월치 미리 생성. 예약 트랜잭션은 해당 슬롯 행에 `SELECT FOR UPDATE`.
- 단점은 동시성 제어가 애플리케이션 데이터 모델로 새어 나오는 것이 추하고, 어떤 충돌을 구체화할지 알아내기 어렵다.
- 책은 최후의 수단으로만 쓰고, 가능하면 직렬성 격리를 쓰라고 권한다.

## 직렬성(Serializability)
- 직렬성 격리는 가장 강한 격리 수준. 트랜잭션들이 병렬로 실행되더라도 결과가 하나씩 순차로 실행된 것과 같다고 보장한다.
- 모든 경쟁 조건을 막아준다.
- 그런데 왜 다들 안 쓰나? → 성능. 직렬성 격리는 셋 중 하나로 구현된다.
	1. 실제로 직렬 실행 (Actual serial execution)
	2. 2단계 잠금 (2PL)
	3. 직렬성 스냅숏 격리 (SSI) — 낙관적 동시성 제어 기법

## 실제 직렬 실행(Actual Serial Execution)
- 동시성 자체를 없애기. 단일 스레드로 트랜잭션을 한 번에 하나씩 처리.
- 2007년경부터 실용성이 다시 검토된 아이디어. 두 가지 변화가 계기였다.
	- RAM이 충분히 싸져서 활성 데이터셋을 전부 메모리에 둘 수 있게 됨.
	- OLTP 트랜잭션이 대부분 짧고 작은 수의 읽기/쓰기만 한다는 점이 인식됨. 긴 분석 질의는 어차피 read-only이므로 스냅숏 격리로 분리 가능.
- 사용 제품: VoltDB/H-Store, Redis, Datomic
- 처리량이 단일 CPU 코어로 제한된다는 게 가장 큰 한계.

### 저장 프로시저로 트랜잭션 감싸기
- 대화식(interactive) 트랜잭션은 네트워크 왕복마다 시간이 들어 단일 스레드 직렬 실행에 부적합.
- 그래서 직렬 실행 시스템은 저장 프로시저(stored procedure)로 트랜잭션 전체를 미리 DB에 보낸다.
- 데이터가 전부 메모리에 있고, 프로시저가 네트워크/디스크 I/O 없이 끝나면 매우 빠르게 실행 가능.
- 단점들
	- 각 DB마다 자체 프로시저 언어 (Oracle PL/SQL, T-SQL, PL/pgSQL …) → 못생기고 라이브러리 빈약
	- DB 안에서 도는 코드라 디버깅/버전 관리/배포가 까다롭다.
	- DB는 보통 여러 앱 서버가 공유하기 때문에, 잘못 짠 프로시저 하나가 큰 사고를 부른다.
- 현대적 구현은 일반 언어를 쓴다.
	- VoltDB : Java/Groovy
	- Datomic : Java/Clojure
	- Redis : Lua

> [!tip] VoltDB의 결정론적 프로시저
> VoltDB는 복제도 저장 프로시저로 한다. 즉 쓰기를 복제하는 게 아니라, 같은 프로시저를 각 복제본에서 다시 실행한다. 그래서 프로시저는 결정론적이어야 한다 (현재 시각이 필요하면 특수 결정론적 API로 받아야 함).

### 파티셔닝으로 확장
- 단일 코어 한계를 넘으려면 6장의 파티셔닝과 결합해야 한다. VoltDB가 지원.
- 각 파티션이 자기 트랜잭션 스레드를 가지므로, 단일 파티션 트랜잭션이면 처리량이 코어 수에 선형 증가.
- 그러나 다중 파티션 트랜잭션은 모든 관련 파티션이 lock-step으로 진행해야 해서 매우 느리다.
	- VoltDB는 다중 파티션 쓰기가 초당 약 1,000건 정도로 보고된다. 노드 추가로 늘리기도 어렵다.

### 직렬 실행의 한계
- 사용 조건이 빡빡하다.
	- 모든 트랜잭션이 작고 빠르게 끝나야 한다. (한 개라도 느리면 전체가 멈춤)
	- 활성 데이터셋이 메모리에 들어가야 한다.
	- 쓰기 처리량이 단일 코어로 처리 가능하거나, 파티션 단위로 깔끔히 분리 가능해야 한다.

## 2단계 잠금 (Two-Phase Locking, 2PL)
- 약 30년간 직렬성을 구현하는 거의 유일한 알고리즘이었다.
- 사용 격리 수준: MySQL(InnoDB)/SQL Server의 serializable, DB2의 repeatable read.

> [!tip] 2PL ≠ 2PC
> 이름이 비슷한 2-Phase Commit (2PC)와 완전히 다른 개념. 2PC는 9장에서 다룬다.

### 동작 방식
- Read Committed의 락보다 훨씬 빡센 락 정책.
	- A가 객체를 읽는 중이면, B는 그 객체에 쓰지 못한다. A가 끝날 때까지 대기.
	- A가 객체에 쓰는 중이면, B는 그 객체를 읽지도 쓰지도 못한다.
- 즉 읽기와 쓰기가 서로를 막는다. (스냅숏 격리의 "readers/writers don't block each other" 와 정반대)
- 락 모드 두 가지
	- shared mode : 여러 트랜잭션이 동시 보유 가능. 읽기용.
	- exclusive mode : 한 트랜잭션만 보유. 쓰기용. 다른 어떤 락도 함께 못 잡는다.
- read → write 로 진행할 경우 shared 락을 exclusive 락으로 업그레이드 가능.
- "2 phase" 인 이유: 트랜잭션 진행 중에는 락을 획득만, 트랜잭션 끝(commit/abort)에 모든 락을 해제.

### 데드락과 성능
- 락이 많이 잡혀 있어서 A↔B 가 서로의 락을 기다리는 데드락이 자주 발생.
	- DB가 자동 감지해서 한 트랜잭션을 어보트. 어보트된 쪽은 애플리케이션이 재시도해야 한다.
- 1970년대부터 모두가 안 쓰던 진짜 이유는 성능.
	- 락 획득/해제 오버헤드
	- 동시성이 줄어드니 처리량 저하
	- 트랜잭션 길이 제한이 없어, 한 트랜잭션이 길게 락을 잡으면 줄줄이 멈춤
	- 꼬리 지연시간(tail latency)이 매우 불안정. 책 표현으로 "단 한 개의 느린 트랜잭션이 시스템 전체를 멈출 수 있다"
	- 데드락이 Read Committed의 락 기반 격리보다도 훨씬 더 자주 발생 → 어보트된 트랜잭션은 처음부터 다시 → 빈번하면 낭비가 크다

### 술어 잠금(Predicate Locks)
- 팬텀까지 막으려면 검색 조건에 락을 걸어야 한다. 특정 행이 아니라 "조건에 매칭되는 모든 (미래에 생길 행 포함) 객체"를 잠근다.
	- EX. `room_id = 123 AND end_time > 12:00 AND start_time < 13:00` 조건에 락.
- 동작
	- A가 위 조건으로 SELECT 하면 그 조건에 shared predicate lock을 잡는다. B가 해당 조건에 매칭되는 객체에 exclusive 락을 보유 중이면 A는 대기.
	- A가 INSERT/UPDATE/DELETE 하려면, 옛/새 값이 기존 술어 잠금과 매칭되는지 검사. 매칭되면 락 보유자가 끝날 때까지 대기.
- 핵심은 술어 잠금이 아직 존재하지 않는 객체(미래에 INSERT될 행)에도 적용된다는 점이다. → 팬텀 방지.

### 인덱스 범위 잠금(Index-Range Locks)
- 술어 잠금은 활성 락이 많으면 매칭 검사가 느려진다. 그래서 실제로는 술어 잠금의 근사인 인덱스 범위 잠금(next-key locking)을 쓴다.
- 술어를 더 큰 범위로 단순화해도 안전하다 (실제 검색을 매칭하는 쓰기는 더 큰 범위에도 반드시 매칭됨).
	- EX. "room 123 의 12-13시" 술어 → "room 123 의 모든 시간" 또는 "모든 방의 12-13시" 로 근사 가능.
- 즉 인덱스 범위 잠금은 의도한 술어보다 더 넓은 범위를 잠그게 된다. 정합성은 유지되지만 그만큼 무관한 트랜잭션도 같이 막힌다.
- 인덱스 항목(EX. `room_id` 인덱스의 키 123)에 shared 락을 붙인다. 다른 트랜잭션이 같은 방을 쓰려면 같은 인덱스 항목을 갱신해야 하므로 락에 걸린다.
- 적절한 인덱스가 없으면 테이블 전체에 shared 락을 거는 안전한 fallback이 있다. 정합성은 유지되지만 그 테이블에 쓰려는 모든 트랜잭션이 멈춘다.

> [!tip] "그럼 쓰기 스큐는 그냥 `SELECT FOR UPDATE` 광역 잠금으로 막으면 되는 거 아닌가?"
> 막을 수는 있지만 대가가 비싸다. 위에서 본 인덱스 범위 잠금의 광범위 락, 2PL 성능 섹션의 데드락 폭증과 꼬리 지연시간 불안정이 같이 따라온다. 한 줄로 말하면 "쓰기 스큐를 잠금으로 막으려면 시스템 전반의 동시성을 깎아야 한다." 이게 SSI 같은 낙관적 방식이 나온 동기다.

## 직렬성 스냅숏 격리 (SSI)
- 2008년에 처음 제안된 비교적 새로운 알고리즘. Michael Cahill의 박사 논문 주제.
- 현재 사용 제품: PostgreSQL 9.1+ 의 serializable 격리, FoundationDB (유사 알고리즘).
- 직렬성을 제공하면서 스냅숏 격리 대비 작은 성능 패널티.

### 비관적 vs 낙관적 동시성 제어
- 2PL은 비관적. "뭔가 잘못될 수 있으면 안전해질 때까지 일단 막는다." 멀티스레드의 mutex와 비슷.
- 직렬 실행은 비관적의 극단. 사실상 전체 DB에 대해 트랜잭션 동안 exclusive 락을 잡는 것과 같다.
- SSI는 낙관적. "일단 진행하고, commit 시점에 격리가 위반됐는지 검사. 위반됐으면 어보트."
- 낙관적 동시성 제어의 특성
	- 경쟁(contention)이 높으면 어보트 비율이 높아져 성능이 나빠진다.
	- 여유 있고 충돌이 적으면 비관적 방식보다 성능이 좋다.
	- 교환 가능한 원자 연산(EX. 카운터 증가)을 쓰면 충돌을 줄일 수 있다.
- 이름대로 스냅숏 격리 위에 직렬성 충돌 감지 알고리즘을 얹은 것이다.

### 낡은 전제 위에 결정을 내리는 문제
- 쓰기 스큐의 본질을 다시 보면
	1. 트랜잭션이 DB에서 데이터를 읽는다.
	2. 그 결과를 보고 결정(쓰기) 한다.
	3. 그러나 commit 시점에는 원래 읽기의 결과가 더 이상 유효하지 않을 수 있다.
- 즉 트랜잭션은 트랜잭션 시작 시 참이었던 전제(premise) 위에 결정을 내린다. commit 무렵엔 그 전제가 깨졌을 수 있다.
- DB는 애플리케이션이 그 결과를 어떻게 쓸지 모르므로, 안전을 위해 전제가 변했으면 쓰기를 무효로 봐야 한다.
- 전제가 변했는지 감지하는 두 경우
	1. 오래된 MVCC 읽기 감지 (쓰기가 읽기 이전에 미커밋 상태였음)
	2. 과거 읽기에 영향을 미치는 쓰기 감지 (쓰기가 읽기 이후에 발생)

### 오래된 MVCC 읽기 감지
- MVCC 스냅숏에서 읽을 때, 아직 커밋되지 않은 다른 트랜잭션의 쓰기는 무시된다.
- 그런데 그 트랜잭션이 나중에 commit 됐다면, 내가 읽은 값은 사실 낡은 값이 되고 결정의 전제가 깨진다.
- SSI는 트랜잭션이 MVCC 가시성 규칙으로 무시한 쓰기를 추적한다. commit 시점에, 무시했던 쓰기 중 이미 커밋된 게 있다면 어보트.
- 왜 읽기 시점에 즉시 어보트 안 하나? read-only 트랜잭션은 어보트할 필요가 없다. 트랜잭션이 나중에 쓰기를 할지는 그 시점에 알 수 없으므로 commit까지 미룬다.

### 과거 읽기에 영향을 미치는 쓰기 감지
- 한 트랜잭션이 데이터를 읽은 이후, 다른 트랜잭션이 그 데이터를 수정하는 경우.
- 2PL의 인덱스 범위 잠금과 비슷한 메커니즘. 차이는 SSI의 락이 다른 트랜잭션을 막지 않는다는 점.
- 인덱스 항목(EX. `shift_id = 1234`)에 "이 데이터를 읽은 트랜잭션 목록"을 기록.
- 쓰기 시점에 그 항목을 보고, 최근 그 데이터를 읽은 트랜잭션들에게 "당신 읽기가 낡아질 수도 있음"을 통지 (tripwire 역할).
- 둘 다 commit 시도 시, 나중에 commit 하는 쪽이 어보트.

### SSI 성능
- 추적 단위(granularity)에 트레이드오프가 있다. 세밀하면 정확하지만 오버헤드 큼. 거칠면 불필요한 어보트가 늘어남.
- 2PL과의 비교
	- 한 트랜잭션이 다른 트랜잭션의 락을 기다릴 필요 없음 → 응답 시간이 더 예측 가능.
	- 읽기 전용 질의는 락 없이 일관된 스냅숏에서 실행 가능. 읽기 중심 워크로드에 유리.
- 직렬 실행과의 비교
	- 단일 CPU 코어 한계 없음. FoundationDB는 충돌 감지를 여러 머신에 분산해 매우 높은 처리량을 낸다. 파티션을 가로지르는 트랜잭션도 직렬성 유지 가능.
- 어보트 비율이 SSI 성능을 좌우한다 → read-write 트랜잭션은 짧게 유지하는 것이 좋다. 단, 긴 read-only 트랜잭션은 비교적 괜찮다 (구현에 따라).

### 2PL과의 비교

|           | 2PL (비관적)            | SSI (낙관적)                |
| --------- | ------------------- | ------------------------ |
| 락 동작      | 인덱스 범위 락 → 의도보다 넓게 잡힘 | 락은 있으나 차단하지 않는 tripwire  |
| 데드락       | 빈번                  | 없음 (commit 시 어보트로 처리)    |
| 읽기-쓰기     | 서로 차단               | 서로 차단하지 않음 (스냅숏 격리 그대로)  |
| 꼬리 지연시간   | 불안정, 큐 형성           | 비교적 예측 가능                |
| 약점        | 위 모든 것              | 경쟁(contention) 높으면 어보트 비율 폭증 |
| 적합한 워크로드  | 충돌 많고 안정성 우선        | 충돌 적고 여유 용량 있을 때         |
