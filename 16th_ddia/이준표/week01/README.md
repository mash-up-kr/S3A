## DB의 가장 기본적인 Operation 정의
1. 데이터 저장
2. 데이터 제공

## 인덱스(색인)
- 부가적인 메타데이터를 유지하여 데이터의 위치를 찾는데 도움을 주는 것
- primary data에서 파생된 추가적인 구조
- 데이터 자체의 내용에 영향을 미치지 않고 쿼리 성능에만 영향을 준다.
- 추가적인 구조를 가짐으로써 데이터 쓰기에 오버헤드가 발생한다.
	- 데이터를 쓸 때마다 색인도 갱신해야한다.
- 저장소 시스템에서 가장 중요한 트레이드 오프 중에 하나가 바로 인덱스(색인)이다.
	- 애플리케이션의 잦은 질의 패턴을 활용해서 색인을 설계하자.

## Bitcask
- 쓰기
	- append-only 로그 파일 끝에 key-value 쌍을 append
	- 메모리에 모든 key와 offset 쌍으로 해시맵 업데이트
- 읽기
	- 해시맵으로 key에 해당하는 데이터의 offset을 찾은 뒤, disk에서 해당 offset으로 data를 seek한다.
- 동영상 URL : 재생 횟수 같이 한 key의 value가 자주 업데이트 되는 경우에 적합하다.
	- 업데이트가 잦지만, key를 캐시할 공간이 많이 필요하지 않기 때문에 해시맵을 메모리에 적재할 수 있다.
- 중복 값이 존재하기 때문에 나중에 compaction 하는 로직이 있다.


## apppend-only log 설계는 낭비인가?
- append 와 segment merge 는 순차적 쓰기 작업이기 때문에 Random Disk write보다 훨씬 빠르다.
	- 특히 HDD에서 그렇다.
- Concurrency 와 Recovery 측면에서 유리하다.
	- 이전 값과 새로운 값을 전부 함께 남기기 때문이다.
- segment merge로 데이터 파일 조각화 문제를 피할 수 있다.

## 해시 테이블 색인의 단점

> [!quote] 원문
> It requires a lot of random access I/O, it is expensive to grow when it becomes full, and hash collisions require fiddly logic.

> [!tip] "it becomes full"의 주어
> 여기서 "it"은 물리적 디스크가 아니라 **디스크상의 해시맵(해시 테이블)**을 가리킨다. 해시 테이블의 load factor가 임계치를 넘어 rehashing이 필요한 상황을 의미한다.

- 키가 너무 많을 경우 메모리에 적재가 어려울 수 있다.
- 디스크상에 hash map을 유지할 경우 Random disk I/O가 많이 필요해 성능이 나쁘다.
	- 디스크상의 해시맵의 load factor가 임계치를 넘으면 2배 크기로 확장(rehashing)해야 하는데, 모든 항목을 읽고 → 새 해시 계산 → 다시 쓰는 과정에서 수백만 번의 랜덤 I/O가 발생한다.
	- 해시 충돌 해소 시 체이닝이나 open addressing 모두 다음 슬롯을 탐색할 때마다 디스크 seek이 필요하다. (메모리에서는 포인터 따라가기가 ns 단위지만, HDD seek은 ~10ms)
- 범위 질의(range query)가 비효율적이다. 해시맵은 정렬되어 있지 않으므로 `key1~key100` 같은 범위 스캔을 할 수 없다.

## SS테이블과 LSM 트리
- **SS Table** = Sorted String Table. 키-값 쌍을 키로 정렬해서 디스크에 저장하는 파일 포맷.
- **LSM-Tree** = Log-Structured Merge-Tree. 멤테이블 + SSTable + Compaction + Sparse Index + Bloom Filter를 조합한 전체 저장 엔진 아키텍처.
	- "Log-Structured": Bitcask처럼 append-only로 쓴다는 뜻
	- "Merge": 정렬된 세그먼트들을 병합한다는 뜻
	- LevelDB, RocksDB, Cassandra, HBase 등이 LSM-Tree 기반이다.

### 멤테이블(Memtable)을 통한 쓰기 흐름
1. 쓰기 요청이 오면 멤테이블(메모리의 Red-Black Tree / Skip List)에 삽입한다. 트리 구조이므로 삽입 시 자동 정렬된다.
2. 멤테이블 크기가 임계치를 넘으면, 정렬된 채로 디스크에 flush하여 SSTable 세그먼트를 생성한다.
3. 디스크의 SSTable 세그먼트들은 백그라운드에서 compaction으로 병합된다.

> [!tip] Bitcask와의 차이
> Bitcask는 디스크에 들어온 순서대로(append-only) 썼기 때문에 정렬이 안 됐다. 멤테이블을 중간에 끼워 넣어 **디스크에 쓰기 전에 메모리에서 정렬을 해결**하는 것이 핵심 아이디어다.

- 멤테이블은 실제 key-value 데이터를 직접 저장하며, 트리 구조 자체가 O(log n) 탐색을 제공하므로 별도 인덱스가 필요 없다.
- Sparse Index는 디스크에 flush된 SSTable 세그먼트를 탐색할 때만 필요하다.

### 병합(Compaction)
- 각 세그먼트가 이미 키로 정렬되어 있으므로 merge sort의 merge 단계와 동일하게 동작한다.
- 같은 키가 여러 세그먼트에 존재하면 가장 최신 세그먼트의 값만 남기고 나머지는 버린다.
- 양쪽 다 정렬되어 있으므로 포인터를 앞으로만 이동하면 되어 순차 I/O만으로 O(n)에 병합이 끝난다.

### Sparse Index
- 해시 인덱스는 모든 키의 오프셋을 메모리에 저장해야 했지만, SSTable은 키가 정렬되어 있으므로 일부 키의 오프셋만 알면 된다.
- 예: sparse index에 `apple(offset 0)`, `cherry(offset 120)`만 있어도 `banana`는 그 사이 구간만 스캔하면 찾을 수 있다.
- 해시 테이블 색인의 두 가지 약점(메모리에 모든 키 필요, 범위 질의 불가)을 모두 해결한다.

### 읽기 시 여러 세그먼트 탐색
- compaction 전에는 여러 세그먼트에 같은 키가 존재할 수 있다.
- 최신 세그먼트부터 순서대로 sparse index로 탐색하고, 처음 찾은 값을 반환한다.
- 단점: 가장 오래된 세그먼트에만 있거나 존재하지 않는 키를 조회하면 모든 세그먼트를 뒤져야 한다.

### 블룸 필터(Bloom Filter)로 읽기 성능 최적화
- 위 단점을 해결하기 위해 각 세그먼트마다 블룸 필터를 둔다.
- "이 키는 이 세그먼트에 **확실히 없다**"를 O(1)로 판별하여 불필요한 디스크 I/O를 건너뛴다.
- "있다"는 답은 확률적(false positive 가능)이지만, "없다"는 100% 확실하다.
	- false negative가 없다.
- LevelDB, RocksDB, Cassandra 등 LSM-Tree 기반 저장소는 거의 다 블룸 필터를 기본 탑재한다.


## B트리
- 가장 일반적인 인덱스 유형
- 설계 철학이 LSM 인덱스와는 다르다.
	- 수 MB의 가변 크기를 가진 세그먼트가 아니라 4KB 정도의 고정 크기 블록, 페이지 단위로 데이터를 저장한다.
- 각 페이지는 주소나 위치를 이용해 식별할 수 있고, 이 방식으로 하나의 페이지가 다른 페이지를 참조할 수 있다.(포인터와 비슷하지만 메모리 대신 디스크에 있다.)
- B 트리의 한 페이지에서 하위 페이지를 참조하는 수를 **분기 계수(branching facotor)**라고 부른다.
- n개의 키를 가진 B트리의 깊이는 항상 O(log n)
- 대부분의 데이터 베이스는 B 트리의 깊이가 3이나 4단계 정도면 충분하다.

### 쓰기 동작
- 새로운 데이터를 디스크 상의 페이지에 덮어쓴다.
- 페이지의 위치는 변경하지 않는다.
- 보통 redo log(WAL - write-ahead log)로 복구를 보장하는 B트리를 사용한다. 
- LSM 접근 방식에 비해서 동시성 제어가 복잡하다.
	- 데이터를 덮어쓰다 보니 페이지 분할 시 여러 페이지를 동시에 수정해야할 경우가 있다.
	- 구조 변경 중 다른 스레드가 접근하면 수정 중의(일관성이 깨진 상태) 상태를 볼 수있다. 
	- 트리의 모든 페이지마다 latch를 통해서 이를 제어한다.
	- 반면, LSM은 메모리의 멤테이블에서만 동시성 제어를 하면 된다.
		- 덮어쓰기가 아니라 append-only이기 때문에 구조 변경이 없어 언제든지 읽을 수 있다.

### 최적화
- copy-on-write-scheme 방식
	- 일부 DB는 redo log 대신 사용한다. (EX. LMDB)
	- 기존 페이지를 덮어쓰지 않고, 새 복사본으로 만들고 부모 역시 새로운 복사본의 포인터를 가진 복사본으로 바꾸는 것
	- 루트까지 경로 전체를 새로 만든다.(WAL 생성 및 유지에 대한 트레이드 오프)
- 축약
	- 키를 경계 판단에 필요한 최소한의 정보만 유지하도록 줄이는 것
		- EX. algorithm -> alg , application -> app
	- 축약을 통해서 한 페이지에 더 많은 키를 넣을 수 있고 공간이 절약된 만큼 트리의 높이(깊이)가 낮아진다.
- 리프 페이지 연속 배치
	- 키 범위가 가까운 페이지들을 메모리에 연속 배치할 이유는 없다.
	- 하지만 정렬된 순서의 range query를 할 때는 리프 페이지를 디스크에 연속적으로 배치 해야 디스크 찾기를 최소화할 수 있다.
	- 따라서 대부분의 DB는 리프 페이지를 연속적으로 배치하려고 노력하는데, 트리가 커지면 순서를 유지하기 어려워진다.
	- 반면 LSM 트리는 디스크에서 연속된 값을 유지하기가 상대적으로 더 쉽다.
	- SSTable flush/compaction 시
		- 정렬된 데이터를 처음부터 끝까지 순차적으로 새 파일에 쓴다
		- 디스크에 자연스럽게 연속 배치됨
		- 단, 세그먼트가 여러 개면 세그먼트 간 이동 시 랜덤 I/O가 발생할 수 있다.
		- Leveled Compaction으로 완화: 같은 레벨의 세그먼트들이 키 범위가 겹치지 않으므로, 키 범위 메타데이터로 불필요한 세그먼트를 통째로 스킵할 수 있다.
		- 쓰기에 유리하나, 읽기는 BTree가 더 유리하다.
- 리프 노드에 형제 노드에 대한 포인터 유지
	- 부모로 이동하지 않아도 순차 탐색 가능
- 프랙탈 트리
	- B트리의 변형으로 LSM 개념을 일부 차용했다.


## LSM 트리의 장단점

### 장점
- 쓰기 처리량이 높다. Append-only 순차 쓰기이기 때문에 B-Tree의 랜덤 쓰기보다 훨씬 빠르다.
- 쓰기 증폭(write amplification)이 낮다. B-Tree는 한 번의 쓰기로 여러 페이지를 갱신 + WAL까지 써야 하지만, LSM은 상대적으로 적다.
- 압축률이 좋다. 주기적으로 compaction을 통해 조각화 없이 재작성되므로 저장 공간 효율이 높다.
- 동시성 제어가 단순하다. 디스크의 SSTable이 불변이므로 락 없이 읽기 가능.

### 단점
- Compaction이 성능에 영향을 준다. 백그라운드 compaction이 디스크 대역폭을 잡아먹어 읽기/쓰기 지연이 튈 수 있다.
- 쓰기 처리량이 매우 높으면 compaction이 따라가지 못해 세그먼트가 쌓이고 읽기가 느려진다.
- 동일한 키에 대한 트랜잭션 격리가 B-Tree보다 어렵다. 같은 키가 여러 세그먼트에 존재할 수 있어서.

## 세컨더리 인덱스 vs 기본키 인덱스(클러스터드 인덱스)
- 클러스터드 인덱스(Clustered Index): 인덱스 안에 실제 row 데이터를 직접 저장한다. 인덱스 = 데이터.
	- 예: MySQL InnoDB의 기본키 인덱스
- 세컨더리 인덱스(Secondary Index): 인덱스는 key만 갖고 있고, 실제 데이터는 별도 위치(힙 파일 또는 기본키)를 참조한다.
	- 키가 유일하지 않다는 점이 기본 인덱스와의 중요한 차이점이다.
- 논클러스터드 인덱스(Nonclustered Index): 인덱스 안에 데이터 위치(포인터)만 저장.
- 커버링 인덱스(Covering Index): 인덱스 안에 일부 컬럼까지 같이 저장해서, 인덱스만 읽어도 쿼리가 완료되게 하는 방식. 두 방식의 절충.

|        | 클러스터드      | 세컨더리     | 커버링        |
| ------ | ---------- | -------- | ---------- |
| 데이터 저장 | 인덱스에 직접    | 포인터만     | 일부 컬럼 포함   |
| 읽기 속도  | 빠름 (1번 탐색) | 2번 탐색 필요 | 빠름         |
| 쓰기 비용  | 보통         | 낮음       | 높음 (중복 저장) |

## 힙 파일(Heap File) 접근 방식
- 세컨더리 인덱스에서 실제 데이터를 저장하는 방식 중 하나.
- 특정 순서 없이 데이터를 저장하는 파일. (append-only일 수도, 삭제된 공간을 덮어쓸 수도)
- 여러 세컨더리 인덱스가 있어도 데이터 중복 없이 힙 파일 위치만 가리키면 된다.
- 새 값이 기존 값보다 크지 않으면 제자리 업데이트 가능. 크면 새 위치로 이동하고 모든 인덱스 업데이트 필요.
	- 한 번의 추가 디스크 홉(hop)이 성능 패널티가 될 수 있어, 클러스터드 인덱스가 선호되는 경우가 있다.

## 다중 칼럼 색인 vs 다차원 색인

### 다중 칼럼 색인 (결합 색인, Concatenated Index)
- 여러 칼럼을 하나의 키로 결합한 인덱스. (예: `(last_name, first_name)`)
- 키 순서에 의존적이다. `last_name`으로 검색은 되지만 `first_name`만으로는 인덱스를 쓸 수 없다.
- 1차원적인 접근.

### 다차원 색인
- 여러 칼럼을 동시에 질의할 때 사용. 공간 데이터(위도/경도) 등에 유용.
- R-Tree가 대표적. 지리 공간 인덱스에서 많이 쓰인다.
- 예: "위도 X~Y, 경도 A~B 범위의 식당 찾기" 같은 질의에 최적.

## 레벤슈타인 오토마톤 (Levenshtein Automaton)
- 오타, 철자 오류가 있어도 유사한 단어를 찾기 위한 자료구조.
- 편집 거리(삽입/삭제/치환 횟수) 내에 있는 모든 단어를 허용하는 유한 상태 기계(FSA).
- Lucene(전문 검색 엔진)에서 사용. 문서 검색 시 근접 유사어 매칭에 활용된다.
- 풀 텍스트 검색에서 "fuzzy query"(유사 질의)를 효율적으로 처리하기 위한 핵심 기술.

## 인메모리 데이터베이스
- 전체 데이터셋을 메모리에 상주시킨다.
- 예: Redis, Memcached, VoltDB, MemSQL
- 디스크 I/O 제거보다는 "메모리에 맞는 형태로 변환할 필요가 없다"는 점이 주된 성능 이점.
- 내구성은 로그 기록, 스냅샷, 복제 등으로 확보.

### 안티 캐싱(Anti-Caching)
- 데이터셋이 메모리보다 크면 안 되는가? → 안 됨. 안티 캐싱으로 해결 가능.
- 최근에 사용하지 않은 데이터를 메모리에서 디스크로 내리고, 필요할 때 다시 메모리로 올린다.
- OS 가상 메모리의 페이지 스왑과 비슷하지만, 레코드 단위라 더 효율적이다.
- VoltDB가 사용.

## OLTP vs OLAP

| | OLTP (Online Transaction Processing) | OLAP (Online Analytical Processing) |
|---|---|---|
| 읽기 패턴 | 소량 레코드 조회 (키 기반) | 대량 레코드 집계 |
| 쓰기 패턴 | 사용자 입력 (짧은 트랜잭션) | 벌크 import (ETL) 또는 이벤트 스트림 |
| 주 사용자 | 웹/앱 최종 사용자 | 분석가, BI |
| 데이터 양 | GB~TB | TB~PB |
| 대표 제품 | MySQL, PostgreSQL, Oracle | Redshift, BigQuery, Snowflake, Hive |

## 데이터 웨어하우징과 ETL
- 데이터 웨어하우스: OLAP 전용 분리된 DB. OLTP 시스템의 부하를 주지 않고 분석 가능.
- ETL (Extract, Transform, Load)
	1. Extract: 각 OLTP DB에서 데이터 추출 (주기적 덤프 또는 실시간 스트림)
	2. Transform: 분석에 맞게 정제/변환
	3. Load: 데이터 웨어하우스에 적재
- 최근에는 ELT(먼저 Load하고 나중에 Transform) 방식도 많이 쓴다.

## 분석용 스키마: 별 모양 & 눈꽃송이 모양

### 별 모양 스키마(Star Schema)
- 가운데에 사실 테이블(fact table) 하나, 주변에 차원 테이블(dimension table) 여러 개가 별처럼 배치됨.
- 사실 테이블이 차원 테이블을 외래 키로 참조.

```
         [dim_date]
             |
[dim_store]—[fact_sales]—[dim_product]
             |
         [dim_customer]
```

### 눈꽃송이 모양 스키마(Snowflake Schema)
- 별 모양의 변형. 차원 테이블이 다시 하위 차원으로 정규화되어 눈꽃처럼 뻗어나감.
- 예: `dim_product` → `dim_brand`, `dim_category`로 분리
- 더 정규화되어 있지만 복잡도 증가. 분석가들은 보통 별 모양을 선호.

## 사실 테이블과 차원 테이블

### 사실 테이블(Fact Table)
- 개별 이벤트를 한 행으로 기록 (예: 각 판매 트랜잭션)
- 대량의 행 (수십억 건 가능)
- 숫자 측정값(가격, 수량 등)과 외래 키들로 구성
- "누가, 언제, 어디서, 무엇을, 얼마에" 중 what/how much를 담당

### 차원 테이블(Dimension Table)
- 이벤트의 속성(누가, 언제, 어디서, 무엇을, 왜)을 담음
- 상대적으로 적은 행 수
- 예: `dim_product`(상품 정보), `dim_date`(날짜 속성), `dim_store`(매장 정보)

## 칼럼 지향 저장소(Column-Oriented Storage)
- 기존 행 지향은 한 행의 모든 칼럼을 묶어서 저장. → OLTP에 적합.
- 칼럼 지향은 각 칼럼별로 따로 저장. → OLAP에 적합.

```
행 지향 (Row-oriented):
[row1: a,b,c,d] [row2: a,b,c,d] [row3: a,b,c,d]

칼럼 지향 (Column-oriented):
[col_a: row1, row2, row3]
[col_b: row1, row2, row3]
[col_c: row1, row2, row3]
```

- 분석 쿼리는 보통 수백 칼럼 중 몇 개만 읽으므로, 필요한 칼럼 파일만 읽으면 I/O가 크게 감소.
- 각 칼럼은 같은 타입의 값들이 모여 있어 압축 효율이 높다.
- 대표 제품: Parquet, ORC, Redshift, BigQuery, ClickHouse

## 칼럼 압축 메커니즘

### 비트맵 인코딩(Bitmap Encoding)
- 칼럼의 고유 값 수(cardinality)가 행 수보다 훨씬 작을 때 효과적.
- 각 고유 값에 대해 비트맵(0/1 배열)을 만든다.

```
칼럼: [red, blue, red, green, blue]

비트맵:
red:   1, 0, 1, 0, 0
blue:  0, 1, 0, 0, 1
green: 0, 0, 0, 1, 0
```

- 질의: `WHERE color IN ('red', 'blue')` → red 비트맵 OR blue 비트맵 → 매우 빠름

### 런 렝스 인코딩(Run-Length Encoding)
- 같은 값이 연속되면 `(값, 반복 횟수)`로 압축.
- 정렬된 칼럼에서 특히 효과적.

## 메모리 대역폭과 벡터화 처리
- OLAP에서 병목은 디스크가 아니라 메모리 → CPU 캐시 대역폭인 경우가 많다.
- 칼럼 데이터는 연속된 바이트 배열이라 CPU 캐시에 잘 적재된다.
- 벡터화 처리(Vectorized Processing): SIMD(Single Instruction Multiple Data) 명령으로 여러 값을 한 번에 처리.
- CPU의 분기 예측, 파이프라이닝, 명령어 병렬성을 최대한 활용할 수 있는 구조.

## 칼럼 저장소의 순서 정렬

- 칼럼별로 저장하더라도 행 단위의 정렬 순서는 일관되게 유지해야 한다. (n번째 값 = n번째 행)
- 관리자가 자주 사용하는 질의를 기준으로 정렬 순서를 선택.
- 예: 날짜로 정렬하면 "최근 1개월" 같은 질의가 빠름.
- 두 번째 정렬 칼럼을 추가하면 첫 번째 칼럼 내에서 서브 정렬됨.
- 장점: 정렬된 칼럼은 런 렝스 인코딩으로 압축이 매우 효과적이다.

### C-Store / Vertica의 다중 정렬
- 같은 데이터를 여러 다른 정렬 순서로 중복 저장한다.
- 복제본이 어차피 필요하니, 복제본마다 다른 정렬 순서로 저장하면 질의 패턴에 맞는 최적 순서를 고를 수 있다.

## 칼럼 지향 저장소 쓰기
- 칼럼 저장 + 정렬 + 압축이 되어 있으면 제자리 업데이트가 매우 어렵다. 압축된 칼럼 중간에 행 삽입 시 모든 칼럼 파일을 다시 써야 할 수 있음.
- 해결: LSM-Tree 방식 차용.
	1. 쓰기는 메모리의 정렬된 구조(멤테이블)에 먼저 쌓음
	2. 충분히 쌓이면 디스크의 칼럼 파일과 병합
- Vertica가 이 방식을 사용.

## 데이터 큐브와 구체화 뷰

### 구체화 뷰(Materialized View)
- 자주 사용하는 질의 결과를 미리 계산해서 저장해 둔 것.
- 일반 뷰(가상 뷰)는 질의 시마다 계산하지만, 구체화 뷰는 디스크에 물리적으로 저장된다.
- 원본 데이터가 변경되면 자동 갱신 (비용 발생).
- 쓰기는 느려지지만 읽기는 매우 빨라진다. → OLAP 워크로드에 적합.

### 데이터 큐브(Data Cube, OLAP Cube)
- 구체화 뷰의 특수한 형태.
- 여러 차원으로 그룹화된 집계 값을 미리 계산한 격자.
- 예: `(날짜, 상품, 매장)`별 매출 합계를 2D/3D 테이블로 저장.

```
         product_A   product_B   product_C
date1      100         50          20
date2      120         60          25
date3      110         55          30
```

- 특정 차원으로 집계하는 질의가 즉시 응답된다. (미리 합산되어 있음)
- 단점: 유연성 감소. 큐브에 없는 차원 조합은 여전히 원본을 봐야 함.
