# 트랜잭션

- 어떤 문제가 있어서 등장했나?
    - 데이터베이스를 사용할 때 애플리케이션이 연산 중 죽을 수 있음
    - 데이터베이스 소프트웨어나 하드웨어는 실패 가능성이 있음
    
    → 시스템의 신뢰성을 보장할 수 있는 방식이 필요해서 트랜잭션이라는 개념이 등장함
    
- 애플리케이션에서 다수의 읽기와 쓰기를 하나의 논리적 단위로 묶는 방법
- 개념적으로 한 트랜잭션 내의 모든 읽기/쓰기는 한 연산으로 실행됨
- 트랜잭션은 전체가 성공(커밋) 하거나 실패(롤백) 할 수 잇음
    - 실패 시에는 애플리케이션에서 안전하게 재시도 가능
- 트랜잭션은 애플리케이션에서 오류 처리를 쉽게 할  수 있도록 도와줌
    - 왜냐면 전체가 성공하거나 전체가 실패하니까
    - 그래서 애플리케이션은 잠재적인 오류 시나리오와 동시성 문제를 무시할 수 있음 (안전성 보장)
- 트랜잭션이 필요한 경우? ← 이거 알려면 트랜잭션이 제공하는 안전성 보장의 종류를 먼저 알아야함. 알아보고 문서 마지막에 정리

## 애매모호한 트랜잭션의 개념

### ACID 의 의미

- Atomic - 원자성
- Consistency - 일관성
- Isolation - 격리성
- Durability - 지속성
- ACID 를 지키지 않는 시스템은 BASE 라고 함
    - Basically Availbale - 가용성 제공
    - Soft state - 유연한 상태
    - Eventual consistecy - 최종적 일관성

#### 원자성

- ACID 의 맥락에서 원자성은 동시성과 관련이 없음
    - 여러 프로세스가 동시에 같은 데이터에 접근하려고 할 때! 의 문제는 ‘격리성’ 에서 다룸
- ACID 에서의 원자성은 트랜잭션이 커밋되거나 롤백되는 것을 의미함
    - 사실 ACID 에서의 원자성은 롤백(어보트) 능력 이라고 표현하는 게 더 잘 어울림

#### 일관성

- ACID 에서는 데이터에 관한 불변식이라는 의미로 사용됨
    - 시스템이 정상이라면 항상 지켜져야 하는 데이터 규칙이라는 뜻
- 원자성, 격리성, 지속성과 달리, 일관성은 애플리케이션의 책임임
    - 왜냐면 다른 건 다 데이터베이스의 속성인데, 얘만 애플리케이션 속성이기 때문
    - 즉, 데이터베이스만으로는 일관성을 지킬 수 없음 (e.g. 규칙에 어긋나는 데이터를 쓰는 걸 막을 수 없음)
        - 특정 종류의 불변식은 확인 가능하긴 함
        - **Q. 어떤 게 있음?**
            
            ```lua
            * NOT NULL: 값이 반드시 존재해야 함
            * UNIQUE: 중복 값 금지
            * PRIMARY KEY: 식별자 유일성
            * FOREIGN KEY: 참조 무결성
            * CHECK: 값 범위나 조건 검사
            * EXCLUDE constraint: PostgreSQL의 겹침 방지 제약
            * 트리거: 복잡한 조건 검사 후 예외 발생 가능
            ```
            

#### 격리성

- 동시성 문제를 여기서 다룸
- ACID 에서의 격리성은 동시에 실행되는 트랜잭션은 서로 격리된다는 보장임
    - 직렬성이라고도 부르는데, 이건 각 트랜잭션이 전체 데이터베이스에서 실행되는 유일한 트랜잭션인 것처럼 동작할 수 있다는 뜻임
- 데이터베이스는 여러 트랜잭션이 동시에 실행되어도, 트랜잭션이 커밋되었을 때의 결과가 트랜잭션이 순차적으로 실행됏을 때의 결과랑 동일하도록 보장함
- 문제 ) 근데 이건 성능 손해가 있음. 그래서 잘 안 씀

- 해결 ) 스냅숏 격리를 사용함 (아래에 서술)

#### 지속성

- ACID 에서의 지속성은 트랜잭션이 커밋되었다면 하드웨어 결함이나 데이터베이스의 문제가 발생해도 트랜잭션에서 기록한 데이터가 손실되지 않는다는 보장임
- 단일 노드 데이터베이스에서는 복구가 가능하도록 WAL 같은 걸 사용하는 거고, 노드가 다수일 경우에는 성공적으로 다른 노드에 데이터가 복제된 걸 의미함
    - 지속성을 보장하려면 이 동작이 완료된 후에 커밋되었다고 보고해야함
- 물론 완벽한 지속성이라는 건 존재하지 않음. 그래서 WAL, 복제, 백업 등을 같이 사용함

### 단일 객체 연산과 다중 객체 연산

- 여러 객체를 한 번에 변경할 수 있다고 할 때, 연관된 모든 필드가 한 번에 처리되어야 클라이언트는 정확한 정보를 얻을 수 있음
- 이를 보장하기 위해 다중 객체 트랜잭션이 필요함
- 다중 객체 트랜잭션은 어떤 읽기 연산과 쓰기 연산이 동일한 트랜잭션에 속하는지 알 수 있어야 함
    - **Q. 이거 이유가 그래야 같이 처리해야할 연산을 같이 처리할 수 있기 때문이라고 생각햇는데, 내가 생각한 이유 맞음?**
        - ㅇㅇ 맞음
    - 관계형 DB 에서는 TCP 연결을 기반으로 함
        - 특정 연결 내에서 BEGIN TRANSACTION 문과 COMMIT 문 사이의 모든 것은 같은 트랜잭션에 속한다고 여겨짐
            - **Q. 이거 뭔가 문제 있을 거 같은데? 아닌가? 먼가 요청이 섞일 수 있을 거 같다는 찜찜함이 남네. 채널 다중화나 커넥션 공유 같은 문제로. 아니면 BEGIN TRANSACTION 문만 오고, COMMIT 문이 오지 않는다거나**
                - 맞아용
                - 채널 다중화 / 커넥션 공유는 애플리케이션 문제로 발생할 수 있고요
                - BEGIN 만 오는 문제도 간으합니당
                - 그래서 커넥션 풀/세션 관리를 잘 해야하고, 열린 트랜잭션을 방치하지 않게 잘 해야합니당
    - 비관계형 데이터베이스는 다중 객체 트랜잭션 지원 안하는 경우가 많음
        - **Q. 왜? 그리고 지원 안 해도 되는 이유가 머임? 데이터 특성과 관련해서**
            - 성능이랑 분산 구조 때문에 지원 안하는 경우가 많음
            - 비관계형 DB 는 수평 확장, 낮은 지연 시간, 높은 가용성 등의 특성을 중시하는데, 다중 객체 트랜잭션은 복잡한 프로토콜이 필요하고 지연 시간 + 장애 처리 비용이 필요하기 때문
            - 지원 안해도 되는 경우는, 애초에 데이터 모델을 한 번에 함께 갱신해야 하는 데이터를 하나의 문서나 객체에 넣도록 설계했을 때임
                - 이러면 어차피 그 문서 안에서 전부 갱신할 수 있으니까 다중 객체 트랜잭션이 필요없음

#### 단일 객체 쓰기

- 단일 객체 쓰기 시 발생할 수 있는 몇 가지 문제 상황이 있음
    - 요청에 필요한 일부 데이터만 보낸 경우에, 데이터베이스는 파싱이 불가능한 이 데이터를 저장할 것인가?
    - 데이터베이스가 쓰기 도중 죽는다면?
    - 다른 요청이 레코드를 변경하는 동안, 또 다른 요청이 이를 읽는다면?
- 데이터베이스는 이 문제를 해결하기 위해 원자성과 격리성을 제공하는 것을 목적으로 함
    - 쓰기 도중 죽는다면? → 장애 복구 로그
    - 동시에 접근한다면? → 객체 잠금 사용
- 어떤 데이터베이스는 증가 연산과 같은 더 복잡한 원자적 연산을 제공하기도 함
    - 여기서 원자적은 다중 스레드 프로그래밍 관점에서 원자적이라는 걸로, 동시성 문제 안 생기는 걸 의미함
    - 이러한 연산을 사용하면 애플리케이션이 직접 ‘읽고 계산하고 다시 쓰는’ 과정을 하지 않아도 됨

#### 다중 객체 트랜잭션의 필요성

- 분산 데이터스토어 환경에서 다중 객체 트랜잭션을 도입하는 건 어려움
    - 여러 파티션에 걸쳐서 트랜잭션을 처리하게 구현하기 어렵고, 가용성과 성능을 저하시키기 때문
    - 근데 어려운 거지 못하는 건 아님
- 의문) 진짜 다중 객체 트랜잭션이 필요한가??
    - 단일 객체 연산만 사용해서 분산 데이터스토어 환경 설계할 수는 없는가?
    - 단일 객체 연산으로 가능한 사례가 있으나, 대다수의 경우에 여러 개의 다른 객체에 실행되는 쓰기 작업은 조율되어야 함
        - 조율되어야 하는 사례
            - 그래프 사용하는 경우, 연관된 노드에도 한 번에 최신 정보가 반영되어야 함
            - 문서 데이터 모델에서도 필요함
            - 보조 색인의 경우에도 필요함

#### 오류와 어보트(롤백) 처리

- 원칙적으로는 중간 상태를 막고, 커밋 처리하거나 롤백해야하나, 모든 시스템이 이 철학을 따르지는 않음
    - 이런 시스템들은 이미 한 일을 취소하지 않고, 재시도 하지도 않음
    - 그래서 애플리케이션 단에서 이를 처리해야함
    - 근데 무작정 재시도 하면 안됨. 아래와 같은 경우가 존재하기 때문
        - 트랜잭션은 성공했으나, 클라이언트에 응답이 전송되지 않아 클라이언트 입장에서는 트랜잭션이 실패한 것처럼 보일 수 있음
            - → 이건 멱등성 키 같은 걸로 별도로 보장해야함
        - 5xx 같은 에러일 경우에는 서버 문제라, 재시도 자체가 문제를 악화할 수 있기 때문
            - 그래서 재시도 해도 되는 케이스와, 재시도 하면 안되는 케이스를 분리해야함

## 완화된 격리 수준

- 동시성 문제는 애플리케이션 수준에서 재현하거나 발견하기 어려움
    - 그래서 데이터베이스에서 트랜잭션 격리를 제공해 이를 해결하고자 했음
    - 개발자는 온전히 비즈니스 로직에만 집중할 수 있게 해주려는 거!
- 직렬성 격리는 트랜잭션들이 직렬적으로 실행되는 것과 동일한 결과가 나오도록 보장한다는 거임
    - 문제) 근데 이거 성능 비용이 비쌈
    - 해결) 그래서 완화된 격리 수준을 사용하기도 함

### 커밋 후 읽기

- 가장 기본적인 수준의 트랜잭션 격리
- 두 가지를 보장함
    1. 데이터베이스에서 읽을 때, 커밋된 데이터만 보게 됨(더티 읽기가 없음)
    2. 데이터베이스에서 쓸 때, 커밋된 데이터만 덮어쓰게 됨(더티 쓰기가 없음)

#### 더티 읽기 방지

- 다른 트랜잭션에서 커밋되거나 어보트되지 않은 데이터를 읽는 경우를 더티 읽기라고 함
- 이를 방지해야하는 이유
    1. 클라이언트의 경험 저하 때문
        1. 일관된 정보를 제공하지 않는 경우, 클라이언트가 매우 화가 날 수 있음!
    2. 읽을 당시의 데이터가 미래에 어보트 될 수 있음
        1. 있었는데요, 없었습니다.

#### 더티 쓰기 방지

- 서로 다른 트랜잭션이 동시에 동일한 객체를 갱신하려고 하는 경우, 아직 커밋되지 않은 다른 트랜잭션의 쓰기 위에 덮어쓰는 상황
- 보통 먼저 들어온 트랜잭션이 커밋/롤백 되기 전까지 두 번째 트랜잭션을 지연시키는 방법을 사용함

#### 커밋 후 읽기 구현

- 커밋 후 읽기는 많은 데이터베이스의 기본 설정임
- 더티 쓰기 방지
    - 로우 수준 잠금 사용
    - 트랜잭션이 로우를 변경해야할 경우, 먼저 락을 획득해야 접근할 수 있음
- 더티 읽기 방지
    - 이것도 로우 수준 잠금을 사용할 수 있음
    - 왜냐면, 쓰기 처리 중인 경우에는 락이 걸려있으니까, 읽기 요청이 락을 획득하는 경우는 쓰기 요청이 완료된 시점일 것이기 때문임
    - 문제) 근데 이건 시간 지연이 너무 커
    - 해결) 그래서 버전을 도입하고, 트랜잭션이 커밋되기 전에는 과거 버전의 데이터를 보여줌(MVCC)

### 스냅숏 격리와 반복 읽기

- 문제) 커밋 후 읽기 격리를 사용해도 동시성 문제가 발생할 수 있음
    - Lost Update
        - 애플리케이션은 읽은 시점의 데이터를 기준으로 변경을 하고 쓰기를 요청함
    - 비반복 읽기
        - 같은 트랜잭션 안에서 같은 데이터를 다시 읽었을 때, 중간에 다른 트랜잭션이 커밋한 변경 때문에 이전과 다른 값이 조회될 수 있음
    - 읽기 스큐
        - 하나의 트랜잭션 안에서 여러 데이터를 읽는 동안, 중간에 다른 트랜잭션의 커밋이 끼어들어 서로 다른 시점의 데이터가 섞여 보일 수 있음
- 해결) 스냅숏 격리
    - 트랜잭션 시작 시점의 일관된 데이터 버전을 끝까지 읽게 해주는 격리 수준
    - 백업/분석처럼 실행하는 데 오래 걸리며, 읽기만 실행하는 질의에 쓰기 좋음
    - InnoDB 엔진 사용하는 데이터베이스에서 지원됨

#### 스냅숏 격리 구현

- 커밋 후 읽기 구현과 마찬가지로 락을 사용하나, 읽기와 쓰기가 서로를 차단하지 않음
    - 위의 커밋 후 읽기 구현의 문제점을 해결함
- 데이터베이스는 객체마다 커밋된 버전 여러 개를 유지하고, 트랜잭션은 들어온 시점에 커밋이 완료된 데이터만 읽게 됨
- 이걸 MVCC 라고 함
- Postgresql 에서는 TXID 라는 걸 사용함
    - 이게 저번주에 설명했던 내용과 관련된 거임(Notion 샤딩)
    - TXID
        - 전통적인 트랜잭션 ID는 32비트라 약 42억 개의 공간을 가짐
        - 이미 처리된 트랜잭션은 frozen 처리
            - 그래야 0 으로 돌아간 시점에, 기존에 남은 TXID 가 먼 미래 시점의 트랜잭션처럼 보이지 않게 할 수 있음
            - 이건 vacuum 이 함

#### 일관된 스냅숏을 보는 가시성 규칙

- 트랜잭션은 데이터베이스에서 객체를 읽을 때 트랜잭션ID 를 사용해 볼 수 있는 것과 볼 수 없는 것을 결정함
    1. 트랜잭션을 시작할 때 진행중인 모든 트랜잭션의 목록을 만듦. 해당 트랜잭션이 쓴 모든 데이터는 무시됨
    2. 어보트된 트랜잭션이 쓴 데이터는 무시됨
    3. 트랜잭션 ID 가 더 큰 트랜잭션이 쓴 데이터는, 해당 트랜잭션의 커밋 여부와 관계 없이 무시됨
    4. 그 밖의 모든 데이터는 애플리케이션이 볼 수 있음
- 데이터베이스는 갱신될 때마다 값을 교체하지 않고, 새 버전을 생성함으로써 작은 오버헤드만 두고 일관된 스냅숏을 제공할 수 있음

#### 색인과 스냅숏 격리

- 다중 버전 데이터베이스에서의 색인은, 색인이 객체의 모든 버전을 가리키게 하고, 색인 질의가 현재 트랜잭션에서 볼 수 없는 버전을 걸러내게 함으로써 구현할 수 있음
    - GC 가 버전 삭제할 때 대응하는 색인도 삭제하는 방식으로 동작함
- 현실에서는 구현 세부에 따라 다중 버전 동시성 제어의 성능이 결정됨
    - e.g. Postgresql 은 동일한 객체의 다른 버전들이 같은 페이지에 저장될 수 있다면 인덱스 갱신을 회피하는 최적화를 함
        - **Q. 이게 무슨 말임?**
            - 우선 여기서 말하는 페이지는 heap page 임
            - postgresql 은 로우를 업데이트 하면 새 버전의 로우를 만듦
                - 근데 이 경우에, 인덱스가 적용되지 않은 컬럼을 수정햇다면 같은 heap page 에 새 버전을 넣는다는 거임
                    - 이걸 HOT update 라고 함
                    - 인덱스가 변경되지 않았기 때문에 인덱스는 같은 객체를 가리킬 수 있으니 인덱스 갱신을 하지 않는다는 거임
            - 핵심은 갱신되는 로우가 인덱스 영향을 주지 않는다면 인덱스를 갱신하지 않는다는 것
    - 카우치DB, 데이토믹, LMDB 에서는 추가 전용 B트리(B트리 + append only + copy on write) 를 사용함
        - 트리의 페이지가 갱신될 때 새로운 복사본을 생성하고, 트리의 루트까지 존재하는 부모 페이지들을 복사한 다음, 그것들이 자식 페이지의 새로운 버전을 가리키도록 갱신함
        - 쓰기에 영향을 받지 않는 페이지들은 복사되지 않고 변함 없는 상태로 남음
        - **Q. 먼말임? 그림으로 설명가능?**
            
            ```lua
            기존 B-tree:
                    Root1
                   /     \
                 A        B
            
            B 페이지에 쓰기가 발생하면 새 B를 만듦
            그리고 루트까지 존재하는 B 의 부모를 복사하고, 그것들이 자식 페이지인 B 의 새로운 버전을 가리키게 함
                    Root1
                   /     \
                 A        B   ← 기존 스냅숏이 계속 사용
            
                    Root2
                   /     \
                 A        B'  ← 새 쓰기 결과
            
            이 때 A 는 변경되지 않았으므로 공유함
            ```
            
        - 이 경우, 쓰기를 실행하는 모든 트랜잭션은 새로운 B 트리 루트를 생성하고, 특정 루트는 그것이 생성된 시점에 해당하는 데이터베잇의 일관된 스냅숏이 됨
            - **Q. 이럼 저장 공간이 더 필요할 거 같은데?**
                - ㅇㅇ 맞음
                - 그래서 전체를 복사하는 게 아니라, 변경된 경로만 복사하는 방식을 사용함
            - 나중에 실행되는 쓰기는 새로운 트리 루트만 생성 가능하고, 존재하는 B 트리를 변경할 수 없으므로 트랜잭션 ID 로 객체 걸러낼 필요가 없음
                - **Q. 먼말임?**
                    - MVCC 라면 같은 row 의 여러 버전이 있어서, 현재 실행 중인 TXID 를 보고 내가 볼 수 있는 버전인지 판단해야함
                    - 반면 copy-on-write B-tree 에서는 특정 시점의 root 가 이미 그 시점의 일관된 데이터 구조를 가리킴
                        - 트랜잭션이 특정 root 를 잡고 있으면 해당 root 아래의 데이터는 전부 그 시점의 스냅숏이라는 것
                    - 그래서 각 row 마다 TXID 를 비교할 필요가 없어진다는 거임
            - 그러나 이 방법에도 컴팩션과 GC 를 실행하는 백그라운드 프로세스가 필요함

### 갱신 손실 방지

- 애플리케이션이 데이터베이스에서 값을 읽은 시점과, 값을 업데이트 하는 시점이 달라 발생하는 문제를 lost update (갱신 손실) 라고 함
- 두 트랜잭션이 작업을 동시에 하는 경우, 두 번째 쓰기 작업이 첫 번째 변경을 포함하지 않으므로 변경 중 하나가 손실될 수 있음
- 아래에 이를 해결하는 방법들에 대해 기술

#### 원자적 쓰기 연산

- 읽기와 쓰기를 논리적으로 하나의 작업처럼 다루기 때문에, 다른 트랜잭션의 변경을 고려하지 않아도 됨
- 그러나 모든 쓰기가 쉽게 원자적 연산으로 표현되지는 않음
    - e.g. 텍스트 편집
        - 이거 해결하는 게 준혁이 형이 설명한 구글 docs 의 방식
- 원자적 연산은 보통 객체를 읽을 때 객체에 대한 독점적인 락을 획득해서 구현함
    - 갱신이 적용될 때까지 다른 트랜잭션에서 해당 객체에 쓸 수 없게 하는 것
        - **Q. 이게 커밋 후 읽기랑 머가 다름?**
            - 커밋  후 읽기는 격리 수준이고, 원자적 연산은 갱신 방식임
- 아니면 단일 스레드에서 실행되도록 해서 동시에 두 트랜잭션이 끼어들 틈이 없게 하는 방법도 있음
    - **Q. 이게 레디스 동작하는 방식일 텐데 맞음? 레디스는 이거 사용해서 락 안 거니까 더 빠르게 동작한다고 알고 있는데**
        - ㅇㅇ 맞음
- 객체 관계형 매핑 프레임워크를 사용하면 원자적 연산을 사용하지 못할 수 있음
    - **Q. 왜?**
        - ORM 을 사용하는 경우 조회 - 갱신 - 수정 처럼 동작할 수 있기 때문
        - 이거 해결하려면 JPQL bulk update, SQL update 등을 써야함
            - **Q. 이게 왜 해결책이야**
                - 이거 사용하면 애플리케이션 메모리에서  read-modify-write 를 하는 게 아니라, DB 에 직접 SQL 날리기 때문
                - 연산 자체가 DB 내부에서 수행되게 하기 때문에 문제를 해결하는 것

#### 명시적인 잠금

- 원자적 쓰기 연산을 제공하지 않을 때, 애플리케이션 단에서 갱신할 객체에 락을 걸 수 있음
- 데이터베이스 질의로는 합리적으로 구현할 수 없는 로직이 포함되는 경우에 사용

#### 갱신 손실 자동 감지

- 위의 두 방법은  read-modify-write 주기가 순차적으로 실행되도록 강제하는 방법임
- 근데 트랜잭션의 병렬 실행을 허용하고, lost update 가 감지될 때만 트랜잭션 롤백 후 read-modify-write 주기가 수행되게 할 수도 있음
- 이 방식은 데이터베이스가 lost update 감지를 스냅숏 격리와 결합해 효율적으로 수행할 수 있음
    - **Q. 감지는 어떤식으로 하나? 백그라운드 프로세스가 있나? 아니면 트랜잭션 관리자가 무언가를 보고 처리하나?**
        - 백그라운드 프로세스는 아니고, 트랜잭션 관리자가 커밋 시점에 검사함
        - 일반적으로 다음을 검사
            
            ```lua
            * 내가 읽은 row가 이후 다른 트랜잭션에 의해 변경됐는가?
            * 내가 쓰려는 row가 내 스냅숏 이후 변경됐는가?
            * write-write 충돌이 있는가?
            ```
            
        - 충돌이 있는 경우 커밋을 거부하고 트랜잭션을 어보트 시킴

#### Compare-and-set

- 트랜잭션을 제공하지 않는 데이터베이스 중에는 원자적 compare-and-set 연산을 제공하는 것도 있음
- 이건 애플리케이션이 마지막으로 데이터를 읽은 시점 이후에 데이터가 변경되지 않았을 때만 데이터 갱신을 허용함
    - 그래서 갱신 손실을 회피할 수 있음
- 갱신을 반영하고자 하는 시점의 값이 이전에 읽은 값과 일치하지 않으면 갱신은 반영되지 않고 read-modify-write 주기를재시도해야함
- 그러나 데이터베이스가 WHERE 절이 오래된 스냅숏으로부터 읽는 걸 허용하면 갱신손실이 발생할 수도 있음
    - **Q. 갱신 반영하고자 하는 시점의 값이 아니라, 이전의 값을 읽을 수 있기 때문인가?**
        - ㅇㅇ 맞음

#### 충돌 해소와 복제

- 복제가 적용된 데이터베이스에서의 갱신 손실을 막는 건 다른 차원의 문제임
- 왜냐면 여러 노드들에서 동시에 데이터가 변경될 수 있기 때문임
- 명시적 잠금 방식과 compare-and-set 은 모두 최신 복사본이 하나만 읽고, 여기서 값을 읽는다고 가정했음
- 그러나 복제가 적용된 데이터베이스에는 최신 복사본이 하나만 있다는 보장을 할 수 없음
    - **Q. 최신 복사본이 여러 개 있는 게 왜 문제가 되는 거임? 최신이 아닌 데에서 읽는 게 문제가 된다면 이해를 하겠는데**
        - 서로 다른 노드가 각자 최신이라고 생각할 수 있기 때문
        - 멀티 리더 복제 방식에서, 서로 다른 두 노드가 동시에 값을 수정한다면 자기 값이 최신이라고 생각할 수 있음
        - 단일 최신 복사본이 있는 경우에는 그게 전역적으로 최신이 되니까 이런 문제가 안 생김
    - 그래서 명시적 잠금 방식과 compare-and-set 으로 해결이 불가능함
- 이걸 해결하려면, 이전에 배웠던 sibling(쓰기가 동시에 진행될 때 한 값에 대해 여러 개의 충돌된 버전) 을 생성하면 됨
    - 그리고 이후에 애플리케이션이나 데이터 구조를 사용해 충돌을 해소하고 버전을 병합하면 됨
        - **Q. 여기서 데이터 구조를 어떻게 사용한다는 거임?**
            - 충돌을 애플리케이션이 해결하는 게 아니라, 병합 가능한 자료구조를 쓰는 걸 말하는 거임
            - 이걸 CRDT 라고 함. 대표적으로 grow-only set, counter 등이 있음
- 원자적 연산 방식은 복제 상황에서도 잘 동작함
    - 특히 교환 법칙(다른 복제본에 다른 순서로 연산을 적용해도 같은 결과가 나오는 경우) 이 성립하는 연산이라면 더 잘 동작함
        - **Q. 왜 교환 법칙 얘기가 나옴? 마스터가 여러 개일 경우에, 각 트랜잭션의 순서가 전역적이지 않아서?**
            - ㅇㅇ 맞음. 멀티 리더나 리더리스 복제에서는 연산 적용 순서가 노드마다 다를 수 있기 때문
- 반면 최종 쓰기 승리(Last Write Wins) 에서는 갱신 손실이 발생하기 쉬움
    - **Q. 왜냐면 마지막에 쓴 걸 기록하기 때문임?**
        - ㅇㅇㅇ

### 쓰기 스큐와 팬텀

- 서로 다른 두 트랜잭션이 각각 다른 객체를 갱신할 때 발생할 수 있는 문제
- 공유 자원을 애플리케이션에서 읽고, 이 값에 따라 트랜잭션을 수행하는 경우에 발생할 수 있음

#### 쓰기 스큐를 특징짓기

- 이 문제를 해결하는 선택지는 제한되어있음
    - 여러 객체가 관련되므로 원자적 단일 객체 연산은 도움이 되지 않음
    - 갱신 손실 자동 감지도 도움이 되지 않음
        - **Q. 왜? 각각 다른 걸 바꾸기 때문에 충돌이 나지 않는다고 판정하나? 근데 공유 자원 자체가 충돌이 나는 걸로 판정이 되지 않나?**
            - 각자 다른 객체를 갱신하기 때문에 감지할 수 없는 거 맞음
            - 공유 자원 자체는 조건의 결과이기 때문에 일반적인 lost update 감지만으로는 부족함
    - 제약 조건을 트리거나 구체화 뷰를 사용해 구현할 수 있음
        - **Q. 구체화 뷰가 머임?**
            - 질의 결과를 미리 계산해서 실제 테이블처럼 둔 거
        - **Q. 트리거는 레코드 수정 후 자동으로 수행되는 로직이니까, 이거 사용해서 조건 검사하게 하나? 그리고 애플리케이션에 알리고 롤백 트랜잭션을 발행하게 하나?**
            - 트리거 내부에서 조건 검사하는 건 맞음
            - 근데 애플리케이션에 롤백 트랜잭션 발행하라 하는 건 아니고, 에러를 내림.
                - 그럼 에러 보고 애플리케이션에서 재시도하는 구조
    - 트랜잭션이 의존하는 로우를 명시적으로 잠그는 것이 차선책임

#### 쓰기 스큐를 유발하는 팬텀

- 쓰기 스큐가 발생하는 상황은 특정 패턴이 있음
    1. SELECT 질의가 어떤 검색 조건에 부합하는 로우를 검색함으로써, 요구사항을 만족하는지 확인함
    2. 첫 번째 질의의결과에 따라 애플리케이션 코드를 결정
    3. 애플리케이션이 트랜잭션을 커밋함. 이로 인해 2단계를 결정한 전제 조건이 변경되게 됨. 즉, 1단계의 SELECT 질의를 수행하면 다른 결과를 얻게 됨
- 이렇게 트랜잭션에서 실행한 쓰기가 다른 트랜잭션의 검색 질의 결과를 바꾸는 효과를 팬텀이라고 함
- SELECT 가 이뤄지는 로우에 락을 걸어 이 문제를 회피할 수 있으나, 항상 검색 조건에 부합하는 로우가 존재하지 않는 게 문제임
    - 예를 들어, 조건을 충족하는 경우에만 로우를 생성하는 경우, 로우에 락을 걸어 문제를 회피할 수 없음 ㅠㅠ
        - **Q. 그럼 어케 해결해야하나요?**
            - 이걸 아래에 설명하겠습니다~
- 스냅숏 격리는 읽기 전용 질의에서는 팬텀 회피가 가능하나, 읽기 쓰기 트랜잭션에서는 팬텀 유발이 가능함
    - **Q. 이거 가능한 이유가, 두 트랜잭션이 공유 자원에 접근할 때 같은 버전의 데이터를 보고 있기 때문인가? 그럼 해결하려면 커밋 시점에 한 번 더 조건 읽기를 수행해야겟네? 근데 이 읽기 시점에 첫 번째 트랜잭션이 반영이 되지 않앗다면 또 같은 문제가 발생할 거 같은데, 그럼 어떻게 해결해야해?**
        - 마자용 두 트랜잭션이 같은 스냅숏 보고 판단해서 문제가 생김니당
        - 단순히 커밋 시점에 다시 읽는 걸로는 부족해요. 이거 해결 방법도 아래에서 설명합니당

#### 충돌 구체화

- 앞서 팬텀 해결 방법이 로우에 잠금을 거는 것이라 했는데, 로우가 없는 경우가 문제가 됐음
- 그럼 데이베이스에 잠금 객체를 추가하는 건 어떨까?
    - 그니까, 명시적인 로우를 생성해두고 여기에 락을 거는 방식을 사용하는 거임
- 이 방법을 충돌 구체화라고함
    - 팬텀을 데이터베이스에 존재하는 구체적인로우 집합에 대한 잠금 충돌으로 변환하기 때문
    - 추상적인 문제에 실체를 부여해서 해결하는 느낌임
- 근데 이 방법은 오류가 발생하기 쉽고, 동시성 제어 로직이 애플리케이션에 데이터 모델로 보이게 되어 깔끔한 해결 방법은 아님
    - **Q. 오류 발생은 왜 쉬움?**
        - 실제 데이터가 아니라 동시성 제어를 위해 인위적으로 만든 데이터에 락을 걸기 때문
        - 오류가 생기기 쉬운 이유
            
            ```lua
            * 어떤 조건을 어떤 잠금 row로 매핑할지 실수할 수 있음
            * 모든 코드 경로에서 같은 잠금 규칙을 지켜야 함
            * 새 기능이 추가되면 잠금 규칙을 빠뜨리기 쉬움
            * 잠금 범위를 너무 넓게 잡으면 성능이 나빠짐
            * 너무 좁게 잡으면 팬텀을 못 막음
            ```
            

## 직렬성

- 앞서 살펴본 내용에서는 몇 가지 문제가 있음
    - 격리 수준은 난해하고 데이터베이스마다 구현에 일관성이 없음
    - 애플리케이션 코드만으로는 특정 격리 수준에서 로직이 정상 동작할지 확신할 수 없음
    - 경쟁 조건을 감지하는 데 도움이 되는 좋은 도구가 없음
        - **Q. 이거 아직도 없음?**
            - 넹 ㅠ 없다고 합니다
            - 보조 도구는 있다고 하네요. (DB 락 대기/데드락 로그, transaction retry 로그 등)
- 그래서 이 문제들을 해결하기 위해 직렬성 격리를 사용함
- 직렬성 격리는 가장 강력한 격리 수준으로, 여러 트랜잭션이 병렬 실행되어도 최종 결과는 동시성 없이 한 번에 하나씩 직렬로 실행될 때와 같도록 보장함
    - 이 격리 수준에서는 데이터베이스가 발생 가능한 모든 경쟁 조건을 막아줌
- 그럼 왜 직렬성 격리를 사용 안하는가?
    - 구현 방식과 관련이 있음
        1. 말 그대로 트랜잭션을 순차적으로 실행하기
        2. 2단계 잠금
        3. 직렬성 스냅숏 격리 같은 낙관적 동시성 제어
    - 이제부터 각 구현 방식을 보고 왜 문제가 있는지 살펴보겟삼~

### 실제적인 직렬 실행

- 동시성 문제를 피하려면 동시성을 없애면 되잖음! 에서 출발한 아이디어
- 싱글 스레드에서 트랜잭션을 하나씩 실행시키는 구조
- 비교적 최근에 싱글 스레드에서 트랜잭션 돌려도 되겟다는 결론을 내릴 수 있었음
    - 램이 저렴해졌고
    - OLTP(Online Transaction Processing, 온라인 트랜잭션 처리) **** 트랜잭션이 짧고, 실행하는 읽기/쓰기 개수가 적다는 것을 알았기 때문
        - **Q. 왜 짧음?**
            - 보통 적은 개수의 row 를 읽고 쓰고 끝내기 때문!
        - 분석 질의는 읽기 전용이기 때문에 직렬 실행 루프 밖에서 돌리면 됩니당
- 락과 관련된 오버헤드가 없기 때문에 동시성 지원하는 시스템보다 빠를 때도 있음(e.g. 레디스)
    - 근데 싱글 스레드에서 돌아가기 때문에 처리량이 CPU 코어 하나의 처리량으로 제한됨
    - 그래서 트랜잭션이 전통적인 형태와 다르게 구조화되어야 함
        - **Q. 어떻게?**
            - 트랜잭션이 아주 짧고, DB 내부에서 한 번에 실행될 수 있어야 함
            - 이런 식으로 동작하면 아주 베스트
            
            ```lua
            클라이언트가 필요한 입력을 모두 준비
            → DB에 프로시저/명령 하나로 전달
            → DB가 짧게 실행
            → 즉시 커밋
            ```
            

#### 트랜잭션을 스토어드 프로시저 안에 캡슐화하기

- 데이터베이스의 초창기에는 DB 터미널에서 사용자가 직접 DB 에게 입력을 주는 방식이 많았고, 이때 사용자 입력 대기 시간이 트랜잭션 안에 포함될 수 있었음
- 근데 이러면 DB 스레드가 대기하는 시간이 너무 길어진다는 문제가 있음
    - e.g. 상품 선택 → 배송지 입력 → 결제수단 입력 → 주문 확정 을 하나의 트랜잭션으로 묶는다면, 각 단계에서 받아야 할 정보가 여러 개인데, 그 때마다 DB 가 대기하게 됨
- 그래서 클라이언트 인풋을 배제하는 형태로 트랜잭션을 구성하게 됨
    - 비즈니스 로직 전체를 트랜잭션으로 묶는 게 아니라, 클라이언트 인풋을 받지 않고 동작할 수 있는 범위로 트랜잭션을 묶었다는 것
    - 그래서 각 HTTP 요청은 하나의 트랜잭션을 커밋하는 요청임
- 근데 클라이언트 인풋이 배제되어도, 애플리케이션에서 질의 - DB 응답 이 반복되는 구조는 여전히 남음
    - 이런 상호작용식 트랜잭션은 애플리케이션과 데이터베이스 사이의 네트워크 통신에 많은 시간을 소비함
    - 근데 만일 데이터베이스에서 동시성을 허용하지 않는다? 그럼 처리량이 너무 저조할 것임
- 그래서 이런 상호작용 방식에서는 처리량을 높일 필요가 있음
- 그럼 어떻게 하나요?
    - 우선 상호작용하는 다중 구문 트랜잭션을 허용하지 않음
        - 왜냐면 위에서 보았던 문제를 답습할 거니까
    - 그리고 스토어드 프로시저를 사용함
        - 여기서 잠깐! 스토어드 프로시저란~
            - 5장에서 살펴보았던 트리거 친구로, 트리거는 특정 테이블 변경 이벤트에 의해 자동 실행되는 DB 내부 코드, 스토어드 프로시저는 애플리케이션이나 SQL에서 명시적으로 호출하는 DB 내부 코드임니다
        - 애플리케이션은 트랜잭션 코드 전체를 스토어드 프로시저 형태로 데이터베이스에 ‘미리’ 제출함
            - 그러면 작업 수행에 필요한 모든 데이터와, 이후에 수행되어야 하는 절차를 하나의 트랜잭션에 태워 보낸다고 가정함
            - 그럼 디스크 I/O 대기나 네트워크 왕복 시간을 줄일 수 있음

#### 스토어드 프로시저의 장단점

- 스토어드 프로시저는 다양한 단점이 있었음
    - 데이터베이스마다 다른 스토어드 프로시저용 언어가 존재햇음
        - 해결) 자바, 루비 등의 범용 프로그래밍 언어 사용하도록 변경함
    - 데이터베이스에서 실행되는 코드는 관리하기 어려움
        - **Q. 루아 스크립트를 서버 재배포 없이 변경할 수 있을까? 예를 들어 S3 에 저장하고 주기적으로 캐싱한다면?**
            - 가능은 해용
            - 그래도 운영 안정성 측면에서는 엄격히 관리해야함
    - 데이터베이스는 애플리케이션 서버보다 성능에 더 민감함
        - 자원과 시간을 많이 소비하는 스토어드 프로시저가 있다면 서버보다 더 치명적인 문제로 작용함
    - 볼트DB 의 경우 복제에도 스토어드 프로시저를 사용하기 때문에 스토어드 프로시저가 결정적이어야함
        - 다른 노드에서 같은 스크립트가 실행돼도 같은 결과가 나와야 함
        - 그래서 날짜나 시간을 사용해야 할 때 결정적 API 를 사용함
- **Q. 레디스가 루아 스크립트 사용하는 이유도 다중 구현 트랜잭션을 사용 가능하게 하려고 했기 때문이구나. 그래서 이 스크립트 사용할 때에는 KV 를 스크립트 넘길 때 한 번에 넘기는 거고**
    - 상당히 맞다고 합니다!

#### 파티셔닝

- 모든 트랜잭션을 순차로 실행하면 동시성 문제는 해결하나, 처리량이 단일 CPU 의 처리 성능으로 제한됨
- 읽기 전용 트랜잭션은 스냅숏 격리를 사용해 다른 곳에서 실행될 수 있지만, 쓰기의 경우는 CPU 가 처리해야함
    - 이거 가정이 갑자기 복제 DB 로 변한 거 같은데..?;;
- 여러 CPU 코어, 여러 노드로 확장하기 위해서는 데이터를 파티셔닝(샤딩) 할 수도 있음
    - 각 트랜잭션이 단일 파티션 내에서만 데이터를 읽고 쓰도록 데이터셋을 샤딩한다면 각 파티션은 다른 파티션과 독립적인 트랜잭션 처리 스레드를 가질 수 있음
    - 즉, CPU 코어에 각 파티션을 할당해서 트랜잭션의 처리량을 확장할 수 있음
- 문제) 여러 파티션에 접근해야하는 트랜잭션이 있다면, 접근해야하는 모든 파티션에 락을 걸고 이를 코디네이션 해야함
    - 왜냐면 직렬성 보장해야하니까
    - **Q. 그럼 여기서 아까 말했던 스냅숏 격리 사용하면서 충돌 날 때만 어보트 시키는 거 사용을 못하나?**
        - 가능은 해용. 이게 뒤에 말할 분산 SSI 나 낙관적 분산 트랜잭션에 가까운 방식입니당
        - 그래도 여전히 어려운 점은 남아요
            
            ```lua
            * 여러 파티션의 읽기/쓰기 의존성을 추적해야 함
            * 커밋 시 전역적으로 충돌 여부를 판단해야 함
            * 여러 노드 간 원자적 커밋이 필요함
            * 네트워크 지연 때문에 어보트 비용이 큼
            * 경쟁이 심하면 재시도가 많아짐
            ```
            
    - 이 경우에는 어엄청 느려짐. 그리고 이건 장비 추가로 처리량을 늘릴 수 없음
        - **Q. 왜?**
            - 장비를 추가하면 파티션 수는 늘지만, cross-partition 트랜잭션은 더 많은 노드와 통신해야 할 수 있기 때문!
            - 차라리 업그레이드가 낫습니당
    - **Q. 그럼 이 문제 어떻게 해결해?**
        - 가장 현실적인 해결책은 데이터 모델링입니다
            - 기술보다는 데이터 접근 패턴 자체를 바꾸는 게 현실적이에요
        - 이렇게 설계하면 돼요
            - 트랜잭션이 한 파티션 안에서 끝나도록 샤드 키 설계
            - cross-partition 작업을 비동기 saga로 분리
            - 보조 색인 사용을 제한하거나 비동기 갱신
            - 충돌 가능성이 높은 데이터를 단일 파티션으로 모음
            - 필요할 때만 분산 트랜잭션 사용
- 이 방법은 데이터셋의 특성에 따라 구현 가능성이 갈림
    - 6장에서 보았듯, 단순한 KV 는 파티셔닝이 금방 되지만, 보조 색인이 있는 경우에는 파티셔닝이 복잡해지기 때문

### 2단계 잠금(2PL)

- 30년 전통 맛집임
- 2단계 잠금은, 쓰기 트랜잭션은 다른 쓰기 트랜잭션뿐만 아니라, 읽기 트랜잭션도 진행하지 못하게 막고, 그 역도 성립시킴으로 강력한 보장을 제공함
    - 스냅숏 격리인 읽는, 쓰기는 서로를 막지 않는다는 원칙과 상반된 차이임
    - 쓰기를 실행하는 트랜잭션이 없는 객체는 여러 트랜잭션에서 동시에 읽을 수 있으나, 누군가 어떤 객체에 쓰려고 접근한다면 독점적인 접근이 필요함
        - 읽기 진행 중에 쓰려고 한다면 읽기가 끝나야 하고, 쓰는 도중에 읽으려 한다면 쓰기가 끝날 때까지 기다려야함
    - **Q. 스냅숏 격리는 이렇게 처리하고 버전 컨트롤을 하는데, 그럼 얘는 더 강력한 보장을 제공하는 대신 버전 처리를 안하겟네?**
        - 전통적인 2PL 에서는 맞아요
        - 근데 실무에서는 혼합해서 사용한다고 해요
    - **Q. 그리고 읽기, 쓰기 둘 다 락을 잡고 들어가겠네? 락은 공유하고**
        - 맞아용 아래에 설명합니당
- 직렬성을 제공하여 갱신 손실, 쓰기 스큐 등을 포함한 모든 경쟁 조건으로부터 보호해줌

### 2단계 잠금 구현

- 락은 공유 모드(shared mode) 랑 독점 모드(exclusive mode) 를 사용함
    - 읽을 때는 공유 모드로 락을 걺. 이 경우에는 동시에 여러 트랜잭션이 공유 락을 획득하고 읽기를 진행할 수 있음
        - 만일 독점 모드로 락이 걸려있다면 해당 락이 풀릴 때까지 대기해야함
    - 쓸 때는 독점 모드로 락을 획득해야함. 이 경우에는 한 번에 하나의 트랜잭션만 락을 획득할 수 있음
        - 종류와 관계없이 락이 잡혀있으면 해당 락이 풀릴 때까지 대기애햐암
        - 이 경우에는 공유 락도 잡을 수 없음
        - **Q. 그러면 쓰기는 이론 상 무한 대기가 가능한 거 아님? 읽기 트랜잭션이 공유 락 잡았을 때 해당 락이 풀리기 전에 다른 트랜잭션이 공유 락 잡고 … 이런 식으로 가면 무한대기 발생할 거 같은디**
            - 이론적으로는 가능성 있습니다
            - 그래서 실제 DB 의 락 매니저는 대기 큐, 공정성 정책, writer 우선 정책 등을 사용해용
    - 트랜잭션이 객체를 읽다가 쓰기를 실행할 때에는 공유 락을 독점 락으로 업그레이드해야함
        - 업그레이드는 독점 잠금을 획득할 때와 동일하게 수행됨
            - **Q. 그럼 읽기 풀고 쓰기 잡나?**
                - 풀고 잡는 건 아님. 공유 락을 소유한 상태에서 독점 락으로의 업그레이드를 요청함
                - 업그레이드가 성공하려면 같은 객체에 대해 다른 트랜잭션의 공유 락이나 독점 락이 없어야 함
                - 즉, 결과적으로 해당 객체를 나만 잠글 수 있는 상태가 되어야 한다는 것
    - 트랜잭션이 잠금을 획득한 후에는 종료나 어보트 될 때까지 락을 소유해야함
        - 그래서 2단계 라고 이름을 붙임
        - 첫 번째 단계는 잠금을 획득, 두 번째 단계는 잠금을 해제
- 락을 매우 많이 사용하므로, 오버헤드가 큼
- 그리고 교착 상태가 발생할 수 있음
    - **Q. 시나리오 하나만**
        
        ```lua
        T1: row A exclusive lock 획득
        T2: row B exclusive lock 획득
        T1: row B lock 요청 → T2가 잡고 있어서 대기
        T2: row A lock 요청 → T1이 잡고 있어서 대기
        ```
        
    - 이 경우에는 DB 가 자동으로 감지하여 교착 상태를 발생시킨 트랜잭션 중 하나를 어보트 시킴
    - 어보트 된 트랜잭션은 애플리케이션에서 재시도 해야함
    - **Q. 어보트 될 트랜잭션은 어떻게 정해? 코카콜라?**
        - DB 마다 정책이 다름
        - 일반적으로 비용이 적은 쪽을 희생시킴
            - 수행 시간이 짧거나
            - 수정한 row 가 적거나
            - 롤백 비용이 작거나 (보통 얘로 결정함)
            - 나중에 시작했거나

#### 2단계 잠금의 성능

- 2단계 잠금은 앞서 보았듯이 락을 많이 걸고, 이로 인해 교착 상태 등의 문제가 발생하기 때문에 성능 오버헤드가 큼
    - 거기다 동시성이 제한되기 때문에 성능이 어마어마어마마 하게 나빠짐
- 트랜잭션 자체의 실행 시간을 줄인다 해도 동시성이 제한되고, 앞선 트랜잭션이 종료될 때까지 기다려야 할 수 있기 때문에 느림
- **Q. 자체적인 해결 방식은 없나?**
    - 본질적인 해결책은 없으나 완화책은 있음
    
    ```lua
    * 락 범위 줄이기
    * 트랜잭션 짧게 만들기
    * 인덱스로 잠금 범위 줄이기
    * 데드락 감지 최적화
    * lock escalation 조절
    * hot row 분산
    * 읽기 전용 트랜잭션은 MVCC 활용
    * 적절한 격리 수준 선택
    ```
    

#### 서술 잠금

- 직렬성 격리를 사용하는 데이터베이스는 팬텀 문제도 막야야함
- 이를 막는 방법이 서술 잠금 방식임
    - 서술 잠금은 공유/독점 잠금과 비슷하지만, 특정 객체에 락을 거는 게 아니라 검색 조건에 부합하는 모든 객체에 락을 걺
        - 이거 책에서는 이렇게 설명햇는데, 찾아보니까 조건 자체에 락을 거는 거에 더 가까운 거 같음
        
        > 서술 잠금은 특정 객체가 아니라 검색 조건, 즉 predicate 자체에 잠금을 거는 개념에 가깝다. 따라서 현재 존재하는 로우뿐 아니라, 앞으로 그 조건을 만족하게 될 삽입/갱신도 충돌로 감지할 수 있다.
        > 
    - 그리고 읽기/쓰기 시 공유/독점 잠금 방식을 사용하는 거임
    - 즉, 차이는 락을 거는 대상이 단일 객체인지, 조건에 부합하는 모든 객체인지 임
- 서술 잠금은 미래에 추가될 수 있는 객체에도 적용할 수 있음
    - 즉, 몀확한 객체가 없어서 발생하는 팬텀도 방지할 수 있음
    - **Q. 어떻게 가능함?**
        - 조건에 락을 걸기 때문에 가능
        - 특저 트랜잭션의 조건에 락을 건 상태에서, 다른 트랜잭션이 락이 걸린 조건을 만족하는 row 를 INSERT 하려고 하면, 현재 존재하지 않는 row 라도 조건과 충돌하므로 대기 혹은 실패하게 함

#### 색인 범위 잠금

- 서술 잠금은 문제가 있음
    - 이미 획득된 락이 많으면 조건에 부합하는 잠금을 확인하는데 시간이 오래 걸림
        - 서술 잠금 방식에서는 조건 자체에 락을 걺
        - 그래서 새로운 락이 들어오면 기존 조건과 충돌하는지 확인해야함
        - 이 과정에서 조건이 복잡하면 더 많은 시간을 소요하게 되는 문제가 있다는 거임
- 그래서 서술 잠금을 근사한 방식을 사용함. 이게 색인 범위 잠금임
- 색인 범인 잠금은, 서술 잠금의 검색 조건 자체를 간략화하는 방식임
    - 예를 들어, 특정 시간에 특정 장소를 예약해야한다면, 모든 시간대에 특정 장소를 예약하는 거에 락을 걸면 됨
    - 즉, 조건 자체를 간략화해서 락을 거는 범위를 넓히는 방식임. 이러면 원래 조건에 부합하는 범위가 무조건 포함되니까 괜찮음
- 만일 범위를 특정할 수 없다면 테이블 전체에 공유 락을 걺
    - 이러면 성능은 저하되는데 안전하긴 함

### 직렬성 스냅숏 격리

- 2단계 잠금은 직렬성 격리를 제공하나 성능이 좋지 않고, 완화된 격리 수준은 성능은 좋으나 보장이 완벽하지 않음
- 이 둘의 장점만을 가져가는 방식이 직렬성 스냅숏 격리(Serializable Snapshot Isolation, SSI) 라는 방식임
    - 완전한 직렬성을 제공하며, 스냅숏 격리에 비해 약간의 성능 손해만 존재함
    - 단일 노드 DB 와 분산 DB 모두에서 사용됨

#### 비관적 동시성 제어 대 낙관적 동시성 제어

- 2단계 잠금은 비관적 동시성 제어 매커니즘임
    - 다중 스레드 프로그래밍에서의 상호 배제와 비슷함
    - 트랜잭션의 속도를 높여 락을 유지하는 시간을 줄여 성능을 보완함
- 직렬성 스냅숏 격리는 낙관적 동시성 제어 매커니즘임
    - 커밋을 원하는 시점에 직렬성 격리를 위반했는지 확인하고, 위반하지 않았다면 커밋, 위반했다면 어보트 시킴
    - 용량이 충분하고 트랜잭션 간 경쟁이 심하지 않으면 낙관적 동시성 제어를 사용해도 좋음
        - 반대로 말하면 위 조건을 충족하지 않는다면 비관적 동시성 제어가 더 좋을 수 있음
    - 경쟁은 원자적 연산을 사용해 줄일 수 있음
- SSI 는 스냅숏 격리를 사용하므로, 모든 읽기는 일관된 스냅숏을 보게 됨

#### 뒤처진 전제에 기반한 결정

- 스냅숏 격리는 MVCC 방식을 사용해 구현할 수 있음
- 이 방식은 트랜잭션이 수행될 시점의 스냅숏을 전제로 동작함
- 근데 트랜잭션을 수행하는 도중에 전제 자체가 변경되었다면? 그럼 해당 트랜잭션의 결과가 최선이라고 할 수 있을까?
    - 아님!
- 그래서 트랜잭션은 전제에 변경이 있다면 트랜잭션에서 실행하는 쓰기가 유효하지 않을 수 있다고 가정해야함
- 직렬성 격리를 사용하기 위해서는 트랜잭션이 뒤처진 전제를 기반으로 동작하는 상홍을 감지하고 해당 트랜잭션을 어보트 시킬 수 있어야 함
    - **Q. 근데 이건 무조건 적용하는 게 아니라, 문제 성격에 따라서 달라질 거 같은디?**
        - 마자용. 굳이 모든 stale read 를 처리하려 하지 않아도 괜찮아요
- 데이터베이스가 전제가 변경됨을 감지하려면 두 가지를 고려해야함
    1. 오래된(stale) MVCC 객체 버전을 읽었는가?
        1. **Q. 근데 애초에 이럴 일이 있나?**
            1. 처음부터 오래된 버전을 읽는 게 아니라, 버전을 읽고 난 이후에 새로운 커밋이 발생해 이전 버전 취급 당하는 거임 (억울하겟다..ㅜㅜ)
    2. 전제에 영향을 미치는 쓰기가 있었는가?
        1. **Q. 이거 전제를 확인한 이후에 새로운 커밋이 발생햇는가? 로 확인하는 거 맞음?**
            1. 대체로 맞음
            2. 더 정확하게는 ‘내가 읽은 데이터나 읽은 범위에 영향을 주는 다른 트랜잭션의 쓰기가 있었는가?’ 를 보는 거임

#### 오래된 MVCC 읽기 감지하기

- 스냅숏 격리(MVCC) 를 사용하면, 트랜잭션이 스냅숏에서 읽는 시점에 커밋되지 않은 데이터는 무시함
- 전제가 변경되었는지 확인하려면, 트랜잭션이 커밋하려고 하는 시점에, 앞서 무시한 데이터 중 커밋된 것이 있는지 확인해야함
    - 그리고 커밋된 게 있다면 커밋하고자 했던 트랜잭션을 어보트해야함

#### 과거의 읽기에 영향을 미치는 쓰기 감지하기

- 후에 실행되는 트랜잭션이 먼저 실행된 트랜잭션(아직 커밋되지 않은) 에게 자기가 읽은 값을 알려주는 방식
    - SSI 에서는 트랜잭션이 읽은 특정 row 나 index range 에 대해 “누가 읽었는지” 를 추적함
    - 이후 다른 트랜잭션이 위의 특정 row 나 range 에 쓰기를 하면 DB 는 읽기-쓰기 의존성이 생겼다고 기록함
    - 커밋 시점에 의존성 그래프가 직렬성을 깨뜨릴 위험이 있으면 트랜잭션을 어보트시키는 거임

#### 직렬성 스냅숏 격리의 성능

- 2단게 잠금과 비교해 직렬성 스냅숏 격리는 다른 트랜잭션이 잡은 락을 고려하지 않아도 됨
    - 왜냐면 읽기와 쓰기는 서로를 막지 않기 때문임
    - 읽기 질의의 경우, 어떤 락도 없이 일관된 스냅숏 위에서 실행될 수 있기 때문에 성능적 이점이 큼
- 순차 실행과 비교했을 때에도, 단일 CPU 코어의 처리량에 제한되지 않는다는 장점이 있음
    - 왜냐면 스냅숏 격리 사용하니까~
    - 데이터가 파티셔닝 돼있더라도 트랜잭션은 직렬성 격리를 보장하면서 여러 파티션으로부터 읽고 쓸 수 있음
        - Q. 이거 가능한 이유가 MVCC 기반으로 동작하기 때문인가? 근데 앞에서 전제 변경을 체크한다햇잖아. 복제 데이터베이스에서는 복제 속도가 제각각이기 때문에 이건 복제 완료 시간을 정형화할 수 없을 거 같은데 맞음?
            - 크게는 맞음
            - 근데 추가로 이런 게 필요한 거임
        
        ```lua
        MVCC 기반 일관된 스냅숏
        + 읽기/쓰기 의존성 추적
        + 여러 파티션에 걸친 커밋 조율
        + 충돌 시 abort/retry
        ```
        
        - 복제 시간 문제의식도 맞음!
