# 2장. 데이터 모델과 질의 언어

## 관계형 모델과 문서 모델

- 관계형 모델의 데이터
    - 데이터는 관계(SQL 에서의 테이블)로 구성됨
    - 각 관계는 순서 없는 튜플(SQL 에서의 로우) 모음임
- 관계형 모델의 목표는 정리된 인터페이스 뒤로 구현 세부 사항을 숨기는 것
    - **Q. 먼말임?**
        - 풀어서 얘기하면, SQL 과 같은 질의 언어(인터페이스) 를 사용하기만 하면, 실제로 데이터가 어떻게 저장되고 처리되는지(구현 세부 사항) 를 몰라도 된다는 것!
- 대부분의 서비스는 관계형 데이터베이스에 기반함

### NoSQL

- RDB 보다 대규모 데이터셋이나 높은 쓰기 처리량 달성이 용이함
- RDB 에서 지원하지 않는 특수 질의가 가능함
    - **Q. 예를 들어?**
        - 문서 내부의 중첩 필드나 배열을 바로 질의할 수 있음
        - 단순 일치가 아니라 형태소, 유사어, relevance score 기반 검색 가능
- 스키마 제약이 없어, RDB 에 비해 데이터를 더 동적으로 표현 가능함

### 객체 관계형 불일치

- 객체지향 프로그래밍 언어에서는, 데이터를 관계형 데이터 테이블에 저장하려면 애플리케이션 코드와 데이터 베이스 모델 객체 사이에 전환 계층이 필요함
    - 객체 자체를 바로 저장할 수 없고, 테이블 구조를 만들어서 매핑을 해야한다는 뜻!
- 이런 모델 사이의 분리를 ‘임피던스 불일치’ 라고 부름
- ActiveRecord 나 Hibernate 는 이 전환 계층에 필요한 boilerplate code 의 양을 줄이긴 하지만, 모델 간의 차이를 완벽히 숨기진 못함
- NoSQL 방식을 사용하면, 객체를 Json 으로 변환해서 바로 저장할 수 있음
    - 그리고 하나의 테이블에 여러 속성을 담을 수 있음. 이는 특히 사용자마다 특성이 다를 때 용이함
- RDB 를 사용하는 경우에는 각 테이블에 질의를 수행하거나 조인을 해서 데이터를 가져와야 함
- 반면 Json 표현에서는 관련된 정보가 한 곳에 있어 질의 하나로 충분함

### 문서 모델에서의 스키마 유연성

- 스키마가 없다는 것은 임의의 키와 값을 문서에 추가할 수 있고, 읽을 때 클라이언트는 문서에 포함된 필드의 존재 여부를 보장하지 않는다는 의미
- 문서 데이터베이스 == 스키마리스? NO!
    - 얘는 사실 ‘읽기 스키마’ 입니당
- 쓰기 스키마(schema-on-write)
    - RDB 의 전통적인 접근 방식
    - 스키마는 명시적이고, 데이터는 쓰여진 모든 데이터가 스키마를 준수하고 있음을 보장
    - 정적(컴파일 타임) 타입 확인과 유사
- 읽기 스키마(schema-on-read)
    - 데이터 구조는 암묵적이고, 읽을 때만 해석
    - 프록르램이 언어에서 동적(런타임) 타입 확인과 유사
    - 컬렉션 안의 항목이 모두 동일한 구조가 아닐 때 유리함
- 이 둘의 차이는 데이터 타입 변경 시에 두드러짐
    - 예를 들어, 기존 필드를 분할해야 하는 경우, 정적 타입의 데이터 베이스 스키마에서는 마이그레이션이 진행되어야 하나, 동적 타입의 데이터 베이스 스키마에서는 애플리케이션 코드만 수정하면 됨

### 질의를 위한 데이터 지역성

- 애플리케이션이 전체 문서에 자주 접근해야 한다면 ‘저장소 지역성’ 을 활요하면 성능 이점이 있음
    - 만일 데이터가 다중 테이블로 나뉘어 저장되어 잇다면 조회에 더 많은 시간을 소요해야함
- 한 번에 문서의 많은 부분을 필요로 하는 경우에만 적용됨
- DB 에 문서를 넣는 경우에는, 문서의 작은 부분에만 접근해도 문서 전체를 적재해야할 필요가 있고, 문서 갱신에도 전체 문서를 재작성해야함
- 때문에 문서를 작게 유지하고, 문서의 크기가 증가하는 쓰기를 피하는 걸 권장함
- 구글의 스패너 데이터베이스는 부모 테이블 내에 테이블 로우를 교차 배치되게끔 선언하는 스키마를 허용해 관계형 데이터 모델에서 지역성 특성을 동일하게 제공함 (오라클은 다중 테이블 색인 클러스터 테이블 기능으로 동일한 특성을 제공함)
    - **Q. 구체적으로 어떤 걸까?**
        - 일반 관계형 DB (예: MySQL)
        
        `[Users 테이블]          [Orders 테이블]
        user_id=1              order_id=101, user_id=1
        user_id=2              order_id=102, user_id=1
        user_id=3              order_id=103, user_id=2`
        
        → 두 테이블이 **디스크의 다른 공간**에 따로 저장됨
        → `JOIN`하려면 **두 곳을 각각 읽어야** 함 (I/O 비용 발생)
        
        - 스패너 (인터리빙 적용 시)
        
        `user_id=1
          └─ order_id=101   ← 디스크에 user=1 바로 옆에 저장
          └─ order_id=102   ← 디스크에 user=1 바로 옆에 저장
        user_id=2
          └─ order_id=103   ← 디스크에 user=2 바로 옆에 저장`
        
        → 부모(User) 행 옆에 자식(Order) 행이 **물리적으로 붙어서** 저장됨
        → `JOIN` 시 **한 번의 디스크 읽기**로 해결 가능
        
- 빅데이터 모델의 칼럼 패밀리 개념이 지역성 관리와 유사한 목적이 있음
    - **Q. 칼럼 패밀리?**
        - 관련된 칼럼들을 묶어서 함께 저장하는 단위
        - 칼럼 패밀리 기반 DB(HBase, Cassandra 등) 는 자주 접근하는 칼럼들을 하나의 패밀리로 묶어 디스크의 같은 위치에 저장함

## 그래프형 데이터 모델

- 데이터에서 다대다 관계가 매우 일반적인 경우에 사용
    - e.g. 소셜 그래프, 웹 그래프, 도로나 철로 네트워크
- 정점(vertex, 노드, 엔티티) 과 간선(edge, 관계, 호) 으로 구성됨
- 서로 다른 유형의 객체를 같은 노드로 표현하면 단일 데이터 저장소에 완전히 다른 유형의 객체를 일관성 있게 저장할 수 있음

### 속성 그래프 모델

- 각 정점은 이렇게 구성됨
    - 고유 식별자
    - 유출 간선 집합
    - 유입 간선 집합
    - 속성 컬렉션(키-값 쌍)
- 각 간선은 이렇게 구성됨
    - 고유 식별자
    - 꼬리 정점(간선이 시작하는 정점)
    - 머리 정점(간선이 끝나는 정점)
    - 두 정점 간 관계 유형을 설명하는 레이블
    - 속성 컬렉션
- 그래프 모델은 관계형 데이터 베이스에서 표현 가능함
    - 정점은 다른 정점과 간선으로 연결됨
    - 정점이 주어지면 정점의 유입과 유출 간선을 효율적으로 찾을 수 있고 그래프를 순회할 수 있음
        - 일련의 정점을 따라 뒤 방향으로 순회함
        - tail_vertex 와 head_vertex 칼럼에 대해 색인을 생성하는 이유가 이거 때문임
- 그래프 모델을 사용하면 전통적인 관계형 스키마에서 표현하기 어려운 걸 표현할 수 있음
    - e.g. 서로 다른 계층 구조를 표현해야 할 때 (프랑스 : 주 - 도, 미국 : 군 - 주)
- 그래프는 데이터 구조 변경을 수용하도록 확장하는 것이 쉬움

### 사이버 질의 언어

- 속성 그래프를 위한 선언형 질의 언어
- 질의를 실행하는 방법은 여러 가지가 있음. 근데 굳이 신경 안써도 질의 최적기가 가장 효율적이라고 예측한 전략을 자동으로 선택해줌
    - **Q. 어떻게 가능한 건가?**
        1. 질의 해석기가 질의를 해석함
        2. 가능한 실행 계획을 만듦
        3. 통계 정보를 바탕으로 비용을 추정함
        4. 제일 괜찮은 걸로 진행시킴
    - 근데 이건 사이버 질의 언어만의 특성이 아니라, 선언형 질의 언어 전반의 특징임
        - SQL 에서도 이거 똑같이 진행함

### 트리플 저장소와 스파클

- 속성 그래프 모델과 거의 동등함
- 모든 정보를 주어, 서술어, 목적어처럼 세 부분 구문 형식으로 저장함
    - 주어
        - 그래프의 정점과 동등
    - 목적어
        - 둘 중 하나임
        1. 문자열이나 숫자같은 원시 데이터 타입의 ‘값’
            1. 이 경우, 서술어는 정점의 ‘키’ 가 됨
        2. 그래프의 다른 정점
            1. 이 경우, 서술어는 그래프의 간선이 되고, 주어는 꼬리 정점, 목적어는 머리 정점이 됨

# 3장. 저장소와 검색

## 데이터베이스를 강력하게 만드는 데이터 구조

### 해시 색인

- append only 로그를 사용하면 디스크 공간이 부족해지는 문제가 생김
- 이를 해결하기 위해서는 로그 파일을 특정 크기의 세그먼트로 나눔
    - 특정 크기에 도달하면 세그먼트 파일을 닫고, 새로운 세그먼트 파일에 이후 쓰기를 수행
- 그리고 각 세그먼트에 대해서 ‘컴팩션’ 을 진행할 수 있음
    - 로그에서 중복된 키를 버리고 각 키의 최신 갱신 값만을 유지하는 것을 ‘컴팩션’ 이라고 함
- 컴팩션을 진행 후에는 세그먼트 크기가 더 작아짐
- 그래서 컴팩션을 수행할 때 동시에 여러 ‘세그먼트 병합’ 이 가능함
    - 세그먼트는 쓰여진 이후에 수정할 수 없기 때문에 병합된 세그먼트는 새로운 파일로 만듦
- 세그먼트 병합과 컴팩션 과정이 진행되는 중에는, 이전 세그먼트 파일을 사용해 읽기, 쓰기 요청을 처리함
- 작업 후에는 새로 만들어진 세그먼트를 사용하도록 전환하고, outdated 된 세그먼트 파일을 제거함
- 추가 전용 로그를 사용하는 것의 장점
    - append 와 세그먼트 병합은 순차 쓰기이기 때문에 무작위 쓰기보다 훨씬 빠름 (순차 쓰기 : O(1), 무작위 쓰기 : O(n))
    - 동시성과 고장 복구가 간단함
        - e.g. 값을 덮어쓰는 경우에 DB 가 죽는 문제를 고려할 필요가 없음. 왜냐면 이전 값 부분과, 새로운 값을 포함한 파일을 나누어 남기기 때문
    - 오래된 세그먼트를 병합함으로써 조각화 되는 데이터 파일 문제를 피할 수 있음
- 해시 테이블 색인 제한 사항
    - 테이블을 메모리에 저장해야하므로, 키가 많으면 문제가 됨
        - 디스크에 저장하는 것도 가능하지만 성능이 별로라 비추
    - 범위 질의에 효율적이지 않음
    - 이걸 해결한 게 SS테이블과 LSM 트리임

### SS테이블과 LSM 트리

- 세그먼트 파일에 있는 키-값을 ‘키로 정렬’ 한 것을 ‘SS테이블’ 혹은 정렬된 문자열 테이블이라고 함
- 각 키는 병합된 세그먼트 파일 내에 한 번만 나타내야함
    - 근데 이건 컴팩션 과정에서 보장됨
- SS테이블은 해시 색인을 가진 로그 세그먼트에 비해 장점이 있음
    - SS테이블은 병합 후에도 정렬이 유지되기 때문에, 새 인덱스를 만들 때도 훨씬 효율적임
        - **Q. 먼말임?**
            - 자~ 설명 들어갑니다
            - 예를 들어, 이렇게 데이터를 추가했다고 해볼게요
                1. cat : mew 추가
                2. dog : wolf 추가
                3. aa : bb 추가
                4. cat : meww 추가
            - 그리고 크기가 2인 세그먼트로 분할을 했어요
                1. 세그먼트1 : cat:mew, dog:wolf
                2. 세그먼트2 : aa:bb, cat:meww
            - 이제 이걸 병합해볼텐데! 여기서 차이가 발생합니다!
                - SS테이블
                    - 이미 세그먼트 내부 키가 정렬되어 있어서, 머지소트의 merge 처럼 동작하면서 정렬이 가능함
                - 해시 색인
                    - 내부가 정렬되어 있지 않기 때문에 머지소트의 merge 방식 불가능
                    - 그래서 각 세그먼트의 전체 데이터를 다 읽어서 해시맵에 최신 값만 남기고, 다시 세그먼트로 만들어야함
            - 즉, SSTable은 정렬되어 있어서 비교하면서 합칠 수 있고, 해시 색인은 일단 다 읽고 재구성해야 한다는 차이가 있음!
    - SS테이블은 정렬이 되어있기 때문에, 색인이 없는 값의 위치도 유추가 가능함!

### SS테이블 생성과 유지

- 레드 블랙 트리나 AVL 트리와 같은 트리 구조를 사용하면, 임의 순서로 키를 삽입하고 정렬된 순서로 키를 읽을 수 있음
- 저장소 엔진을 이렇게 만들 수 있음
    - 쓰기가 들어오면 인메모리 균형 트리에 데이터 구조 추가
        - 인메모리 트리를 memtable 이라고 함
    - 멤테이블 크기가 임계값을 넘으면 SS테이블 파일로 디스크에 기록함
    - 읽기 요청이 들어오면 먼저 멤테이블에서 키를 찾고, 없으면 디스크 상의 가장 최신 세그먼트부터 찾음
    - 세그먼트 파일의 병합과 컴팩션 과정을 수행함 (백그라운드 수행)
- 근데 문제가 있음!
    - 데이터베이스에 장애가 발생한 경우, 디스크에 기록되지 않고 멤테이블에 있는 가장 최신 쓰기가 유실됨
- 이를 해결하기 위해서는 쓰기를 즉시 추가할 수 있게 분리된 로그를 디스크에 유지해야함
    - 이 로그는 어차피 멤테이블 복원에만 필요한 거라서 굳이 정렬할 필요 없음
    - SS테이블로 기록된 후에는 이 로그 버려도 됨

### SS테이블에서 LSM 트리 만들기

- Log-Structured Merge-Tree
- 위에서 설명한 걸 구조화한 게 LSM 트리임
- 정렬된 파일 병합과 컴팩션 원리를 기반으로 하는 저장소 엔진을 LSM 저장소 엔진이라고 함

### 성능 최적화

- LSM 트리 알고리즘에서는 ‘없는 값’ 을 찾는 경우에 느릴 수 있음
    - 왜냐면 멤테이블 확인햇을 때 없을 테니까, 디스크에 있는 최신 세그먼트부터 뒤지기 시작할 건데, 여기에도 전부 없다는 걸 확인해야 없네! 라고 할테니까
- 그래서 이거 해결하기 위해서 블룸 필터(Bloom Filter) 사용함
    - 키가 데이터베이스에 존재하지 않을 가능성을 알려줌
    - **Q. 어떻게?**
        - 블룸 필터는 어떤 키들을 넣어둿는지 비트 배열로 요약해둔 자료구조임
        - 키를 추가할 때, 여러 해시 함수를 돌려서 나온 위치의 비트를 1로 설정해둠
        - 만일 어떤 키가 존재하는지 확인하려고 그 키에 해시 함수를 돌려서 나온 위치의 값을 확인햇는데 다 0이다! → 그럼 SS테이블에 키가 없다는 것을 알 수 있음
        - 물론 반대로 해시 함수 돌려서 나온 위치의 값이 모두 1일 때도 SS테이블에 키가 있다는 것을 보장할 수는 없음
        - 그래서 없는 건 확실히 말할 수 있고, 혹시 있을지도..? 하는 것만 검색할 수 있게 해줌
- SS테이블을 압축하고 병합하는 순서와 시기를 결정하는 전략
    - 크기 계층 컴팩션
        - **Q. 머임?**
            - 수박게임임
            - 비슷한 크기의 SS테이블을 모아서 더 큰 SS테이블로 병합하는 방식
            - 컴팩션 횟수를 줄일 수 있음 → 쓰기 성능에 유리
                - **Q. 왜요? 어차피 컴팩션은 백그라운드 작업 아님?**
                    - 백그라운드 작업은 조상님이 돌려주는 게 아님
                    - 얘도 자원을 사용하기 때문에 성능에 영향을 줌
            - 읽기 성능은 나빠질 수 있음
                - 왜냐면 같은 키 범위가 여러 SS테이블에 존재할 수 있어서, 특정 키를 찾을 때 테이블을 여러 개 확인해야할 수 있기 때문
    - 레벨 컴팩션
        - **Q. 머임?**
            - SS테이블을 여러 레벨로 나눠서 관리하는 방식
            - 각 레벨에는 허용 가능한 크기나 파일 수가 정해져 있어서, 이걸 넘으면 다음 레벨로 병합되는 구조임
            - 그래서 최신 데이터는 상위 레벨에 있고, 시간이 지날 수록 아래 레벨로 내려감. 그리고 아래 레벨로 갈 수록 더 큰 용량을 가지게 됨
            - 같은 레벨 내부에서는 키 범위가 겹치지 않도록 관리함
                - e.g. L2 에 있는 SS테이블이 아래처럼 되어있는 것
                    - 파일 A: aa ~ cat
                    - 파일 B: dog ~ moon
                    - 파일 C: zoo ~ zzz
            - 읽기 성능은 유리함
                - 왜냐면 동일 레벨에서는 키 범위가 겹치지 않게 관리하기 때문에, 키가 어디 있을지 예상할 수 있기 때문
                - 그리고 같은 이유 덕분에 범위 조회에 유리함
            - 근데 쓰기 성능은 불리함
                - 컴팩션이 자주 일어나기 때문
                - 그리고 각 레벨에는 허용 가능한 크기나 파일 수가 정해져 있어서, 이걸 넘으면 다음 레벨로 병합되는 구조이기 때문에, L0→L1 병합, L1→L2 병합, … 이렇게 같은 키가여러 번 재기록 될 수 잇음
