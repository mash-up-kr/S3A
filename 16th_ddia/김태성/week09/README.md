# 일관성과 합의

## 선형성

- 실제로 복사본은 여러 개일 수 있지만, 외부에서 시스템을 봤을 때 시스템에 데이터 복사본이 하나만 있고, 그 데이터를 대상으로 수행하는 모든 연산이 어느 한 순간에 원자적으로 적용된 것처럼 보이게 만드는 것
    - 클라이언트가 쓰기를 성공적으로 완료하자마자 해당 데이터베이스를 읽는 모든 클라이언트는 방금 쓰여진 값을 볼 수 있어야 함
    - 최신성 보장이라고 이해하면 됨

### 시스템에 선형성을 부여하는 것은 무엇인가?

- 클라이언트 관점에서는 요청 - 응답이라고 생각하지만, 이 사이에는 DB 에서 논리적 읽기나 쓰기가 확정되는 “특정 순간” 이 있음
    - 연산의 요청 - 응답 시간을 막대로 표현하면, 그 내부에 연산이 실행된 시점이 수직선으로 표시되어 있음
- 이 “특정 순간” 을 기준으로 수행되는 읽기, 쓰기, CAS(Compare And Set) 등이 확정된다고 생각하면 됨

#### 선형성 vs 직렬성

- 둘 다 순차적인 순서로 배열되는 것을 의미함
- 다만 직렬성은 트랜잭션 수준, 선형성은 레지스터(개별 객체) 수준에 적용된다는 차이가 있음
- 또한 직렬성은 실제 시간 순서를 반드시 보존하지 않아도 되지만, 선형성은 실제 시간 순서를 보존해야 함!
    - 직렬성은 여러 트랜잭션이 동시에 실행되더라도 그 결과가 어떤 순서로 하나씩 실행된 것과 같기만 하면 됨
    - 선형성은 각 연산이 실행 구간 안의 어느 한 순간에 원자적으로 수행된 것처럼 보여야 하며, 이미 완료된 연산은 그 이후 시작된 연산보다 앞선 순서로 배치되어야 함

### 선형성에 기대기

- 쓰기가 이후 발생하는 모든 읽기에 즉시 반영되어야 하는 경우가 있음

#### 잠금과 리더 선출

- 단일 리더 복제 환경에서는 리더가 하나만 선출되어야 함
- 이 때 락을 사용할 수 있음
    - 락을 획득한 노드가 리더가 되는 방식
    - 모든 노드는 어느 노드가 락을 소유하는지에 대해 동의해야함
- 분산 락과 리더 선출을 구현하기 위해 아파치 주키퍼나 etcd 같은 코디네이션 서비스를 사용함
    - 합의 알고리즘을 사용해 선형성 연산을 내결함성이 있는 방식으로 구현해줌
    - 알고리즘은 후반에 설명

#### 제약 조건과 유일성 보장

- 제약 조건이나 유일성을 강제하기 위해서는 선형성이 필요함
- 특정 제약 조건(전역 유일성, 중복 생성 방지, 동일 자원에 대한 단일 소유권 등) 은 모든 노드가 동의하는 하나의 최신 값(선형성) 을 요구함
    - 모든 서비스에서 필요한 건 아님. 데이터 민감한 데에서만 사용하면 됨

#### 채널 간 타이밍 의존성

- 선형성의 경쟁 조건은 서로 다른 두 채널을 이용해 같은 사실을 전파할 때에 발견된다
    - 특정 채널의 동작이 먼저 수행된 다음, 다른 채널의 동작이 수행되어야 하는데 선형성이 지켜지지 않으면 경쟁 조건이 발생함!
    
    ```lua
    e.g.
    1. API 서버가 DB 에 사진 정보를 기록한다.
    2. 메시지 큐에 “이 사진 리사이즈해라” 이벤트를 보낸다.
    3. 워커가 메시지를 받고 DB 에서 사진 정보를 읽는다.
    ```
    

### 선형성 시스템 구현하기

- 선형성은 데이터 복사본이 하나만 잇는 것처럼 동작하고, 모든 연산이 원자적이기만 하면 됨
- 따라서 가장 간단하게는 데이터 복사본 하나만 사용하면 됨
    - 근데 이건 장애 발생에 취약해지는 문제가 있음
    - 그래서 복제를 사용함
- 복제 방식 중, 선형적으로 만들 수 있는 것만 추려보자!
    - 단일 리더 복제(선형적이 될 가능성 있음)
        - 리더나 동기적으로 갱신된 노드에서 실행한 읽기는 선형적이 될 가능성이 있음
            - 그러나 이 경우에는 누가 리더인지 확실하다고 가정해야함
            - 왜냐면 리더가 아닌 노드가 자신이 리더라고 생각할 수도 있기 때문(GC 로 장기간 멈춘 경우 등)
        - 모든 단일 리더 복제가 선형적인 것은 아님
            - 설계, 동시성 버그 등등
            - **Q. 구체적으로 어떻길래 선형적인 게 아님?**
                - 복제가 느려서 팔로워가 최신 데이터를 유지하지 못할 때 읽거나,
                - stale leader 문제가 발생하면 선형성이 깨질 수 있음
    - 합의 알고리즘(선형적)
        - 후반부에서 설명
        - 단일 리더 복제와 비슷하나, 스플릿 브레인과 복제본이 뒤쳐지는 문제를 막을 수단이 포함되기 때문에 선형성 저장소를 안전하게 구현 가능함
        - e.g. 주키퍼, etcd
    - 다중 리더 복제(비선형적)
        - 여러 노드에서 동시에 r/w && 비동기 복제하기 때문에 충동 가능성 높음
    - 리더 없는 복제(아마도 비선형적)
        - 정족수를 요구하기 때문에 엄격한 일관성 가능하다고 보기도 하지만, 정족수 설정과 엄격한 일관성의 정의에 따라 주장이 완전한 진실은 아님

#### 선형성과 정족수

- 직관적으로 다이나모 스타일(리더 없는 복제) 에서 엄격한 정족수를 사용한 읽기 쓰기는 선형적인 것처럼 보임
- 그러나 네트워크 지연의 변동이 심하면 경쟁 조건이 생길 수 있음
    - 지연이 길어져 정족수 이하의 노드에만 쓰기가 반영된 시점에 읽기를 수행하는 경우, 선형성을 충족할 수 없음
- 성능이 떨어지는 비용을 지불하고 선형성 획득이 가능하긴 함
    - 읽기 수행하는 클라이언트는 읽기 복구(병렬 조회 후 가장 높은 버전의 값을 신뢰, 이후 이전 버전을 가진 노드 업데이트) 를 동기식으로 수행하고
    - 쓰기 수행하는 클라이언트는 요청 보내기 전에 노드들의 정족수로부터 최신 상태를 읽으면 됨
    - 근데 이 방식으로는 선형성 읽기/쓰기만 구현 가능하고, 선형성 compare-and-set 은 안됨
        - **Q. 왜? compare and set 은 원자적으로 수행되어야 하는데, 이건 읽기 쓰기가 별개의 동작처럼 수행되어서 그런가?**
            - ㅇㅇ. compare-and-set 은 원자적으로 읽기/쓰기가 수행되어야 하는데, 이 방식에서는 읽기 따로 쓰기 따로잖음

### 선형성의 비용

- 데이터센터가 분산되어 있는 경우에 네트워크 분단이 발생한다면, 선형성은 챙길 수 없지만 가용성을 챙길 수 있다
- 반면에 단일 리더 복제를 선택하면 선형성은 챙길 수 있지만, 리더가 데이터센터 중 하나에 잇어야 하며, 모든 쓰기와 선형성 읽기는 리더로 보내져야함
    - 그래서 네트워크 분단이 발생하면 팔로워 데이터센터로 접속한 클라이언트들은 리더로 연결이 안되어 쓰기, 선형성 읽기 모두 불가능함

#### CAP 정리

- 위 문제는 어떤 선형성 데이터베이스라도 구현과 관계없이 발생할 수 있는 문제임
- 신뢰성이 없는 네트워크나, 한 데이터센터 내에서도 발생할 수 있는 문제임
    - **Q. 한 데이터센터 내에서 발생하는 네트워크 분단은 어떤 거임?**
        - 리더-팔로워 네트워크 장애
        - 스위치 장애
        - GC pause or STW(stop the world) 등등
- 트레이드 오프는 아래와 같음
    - 애플리케이션에서 선형성 요구하는 경우, 네트워크 문제로 복제 서버와 연결이 끊기면 장애가 복구될 동안은 요청을 처리할 수 없음(가용성이 없음)
    - 애플리케이션에서 선형성 요구하지 않는 경우, 네트워크 문제로 복제 서버와 연결이 끊기더라도 독립적으로 요청 처리는 가능함. 그러나 선형적이지 않음
- 따라서 선형성이 필요 없는 애플리케이션은 네트워크 문제에 더 강인함
    - 제약이 하나 줄기 때문
- 이걸 정리한 게 CAP 정리임
    - 근데 범위가 매우 좁음
    - 하나의 일관성 모델(선형성) 과 한 종류의 결함(네트워크 분단 or 노드가 살아있으나 연결이 끊긴 상황) 만 고려함
    - 그래서 시스템 설계 시에 실용적 가치는 거의 없음

#### 선형성과 네트워크 지연

- 선형성은 유용하나, 실제로 선형적인 시스템은 드물다
    - 성능을 중시하기 때문에 선형성을 버리는 것
- 다중 코어 CPU 의 RAM 조차 선형적이지 않다!
    - 하나의 CPU 코어에서 실행 중인 스레드가 메모리 주소에 쓴 후, 곧 다른 CPU 코어에서 실행되는 스레드가 같은 주소를 읽어도 첫 번째 스레드가 쓴 값을 읽을 것이라고 보장되지 않음
        - **Q. 왜? 그 사이에 값이 변경된다는 가정이 있는 거임?**
            - 값이 변경되는 건 아니고, 이런 이유 때문임
            
            ```lua
            코어 1에서 실행 중인 스레드가 쓰고, 이어서 코어 2의 스레드가 읽는 경우:
            
            코어 1의 쓰기가 store buffer에 머물러 있을 수 있음
            코어 2는 자기 cache에 있던 오래된 값을 읽을 수 있음
            CPU가 성능을 위해 메모리 연산 순서를 재배치할 수 있음
            컴파일러도 동기화가 없다면 일부 순서를 최적화할 수 있음
            ```
            
        - 메모리 배리어나 펜스를 사용하면 보장 가능함
            - **Q. 이게 뭔데?**
                - cpu 하고 컴파일러한테 제약 거는 장치임
                    - 이 지점을 기준으로 특정 메모리 연산들이 앞뒤로 재배치되지 않게 하라
                    - 그리고 이전 쓰기가 다른 코어에 관찰 가능해지도록 필요한 동기화를 수행하라.
    - 이건 모든 CPU 코어가 각각 메모리 캐시와 저장 버퍼를 갖기 때문인데, 메모리 접근은 기본적으로 캐시로 먼저 가고, 변경은 메인 메모리에 비동기로 기록됨
        - 캐시 접근이 훨씬 빠르기 때문에 이렇게 함
        - 비동기 복제이기 때문에 선형성 손실됨
    - 이렇게 하는 이유는 성능을 중시하기 때문!
    - 여러 분산 데이터베이스에서도 성능을 중시하기 때문에 선형성을 보장하지 않음
    - 선형성을 원하면 읽기와 쓰기 요청의 응답 시간이 적어도 네트워크 지연의 불확실성에 비례해야 하나, 대부분 컴퓨터 네트워크는 지연 변동이 심하기 때문에 지연 시간에 민감한 시스템에서는 선형성을 버림

## 순서화 보장

### 순서화와 인과성

- 순서화는 인과성을 보장하는데 도움을 줌
- 시스템이 인과성에 의해 부과된 순서를 지키면 시스템이 인과적으로 일관적이라고 함
    - e.g. 스냅숏 격리

#### 인과적 순서가 전체 순서는 아니다

- 전체 순서
    - 항상 어떤 것이 더 크거나 작다는 것을 말할 수 있는 것
    - e.g. 자연수 비교
- 부분 순서
    - 항상 어떤 것이 더 크거나 작다는 비교가 가능하지 않은 것
    - e.g. 집합 비교
- 선형성 시스템에서는 연산의 전체 순서를 정할 수 있음
    - 데이터 복사본이 하나만 있는 것처럼 동작하기 때문에, 모든 연산이 원자적이면 어떤 두 연산 중 하나가 먼저 실행됐다고 할 수 있음
    - 따라서 선형성 데이터스토어에는 동시적 연산이 없음
        - 모든 연산은 하나의 타임라인 위에 있고, 해당 타임라인을 따라 전체 순서가 정해짐
        - 다수의 요청이 있을 수 있지만, 데이터스토어는 타임라인을 따라 단일 데이터 복사본에 연산을 실행해 모든 요청이 한 시점에 원자적으로 처리되도록 보장함
- 인과성에서는 연산의 전체 순서를 항상 정할 수는 없음
    - 동시에 두 연산이 실행되는 경우, 어느 연산이 더 먼저 수행된 것인지 비교할 수 없기 때문
    - 이건 git 같은 거임
        - 한 시점에 여러 브랜치가 있을 수 있음

#### 선형성은 인과적 일관성보다 강하다

- 선형성은 인과성을 내포한다
    - 선형적인 모든 시스템은 인과성도 올바르게 유지한다
- 그런데 성능 상의 이유로 선형성을 버리기도 한다
- 대신 절충이 가능하다
    - 선형적으로 만드는 성능 손해를 유발하지 않고 인과적 일관성을 만족시킬 수 있다
        - **Q. 오잉 어케하는 거임?**
            - 선형성에서는 모든 연산이 순서가 있어야 하지만, 인과적 일관성에서는 인과 관계 있는 것만 순서를 지키면 된다는 아이디어에서 출발
            - 원인-결과 관계가 있는 연산의 순서만 지키고, 서로 무관한 동시 연산은 순서를 강제로 정하지 않음
            
            ```lua
            1. 각 쓰기에 버전 정보 또는 dependency metadata를 붙임
            2. 클라이언트가 읽은 버전 정보를 기억함
            3. 클라이언트가 나중에 쓰기를 보낼 때 “나는 이 버전들을 본 상태에서 이 쓰기를 만든다”는 정보를 함께 보냄
            4. 복제본은 후속 쓰기를 적용하기 전에, 그 쓰기의 선행 의존성이 먼저 반영되어 있는지 확인함
            5. 선행 의존성이 없으면 기다림
            ```
            
    - 인과적 일관성은 네트워크 지연의 영향을 받지 않고, 네트워크 장애 발생에도 가용하다
- **Q. 최신 데이터베이스 연구는 어떻게 되고 있나?**
    1. 버전 벡터 / dependency vector 기반 causal consistency
        - 각 객체나 세션에 대해 “어떤 노드의 몇 번째 쓰기까지 봤는지” 를 기록
        - 쓰기 요청에 클라이언트가 읽은 버전 정보를 함께 전달
        - 복제본은 쓰기를 적용하기 전에 그 쓰기가 의존하는 선행 버전들이 먼저 도착했는지 확인
        - 인과성을 정확히 추적할 수 있으나, 메타데이터가 커질 수 있음
    2. 세션 보장(session guarantees) 방식
        - 완전한 전역 causal consistency 보다 약하지만 실용적
        - 대표 보장은 아래와 같음
            - read-your-writes
                - 내가 쓴 값은 내가 다시 읽을 수 있어야 함
            - monotonic reads
                - 한 번 본 값보다 더 오래된 상태로 되돌아가지 않아야 함
            - monotonic writes
                - 같은 클라이언트의 쓰기는 순서대로 반영되어야 함
            - writes-follow-reads
                - 내가 읽은 값을 기반으로 한 쓰기는 그 읽기 이후에 반영되어야 함
        - 요청을 보낼 때 lastSeenVersion 을 같이 보내고, 적어도 lastSeenVersion 이상을 반영한 replica 에서 읽게 하는 방식으로 구현. 쓰기도 마찬가지로 version 을 보내서 해당 version 을 따라잡은 뒤에 반영되도록 구현
        
        ```lua
        1. 클라이언트가 읽기/쓰기 응답에서 version 또는 timestamp를 받음
        2. 클라이언트는 세션 컨텍스트에 최신 version을 저장함
        3. 다음 요청에 이 세션 컨텍스트를 함께 보냄
        4. 서버는 해당 context를 만족하는 replica에서 처리함
        5. 만족하지 못하면 대기, 리더 라우팅, 다른 replica 선택, 재시도 등을 수행함
        ```
        
        - 많은 실무 시스템은 완전한 선형성 대신 이런 세션 보장을 제공함. 왜냐면 사용자가 체감하는 일관성 문제 상당수는 전역 선형성보다 세션 보장으로 해결되기 때문

#### 인과적 의존성 담기

- 인과성을 유지하기 위해서는 어떤 연산이 다른 연산보다 먼저 실행됐는지 알아야 함
- 한 연산이 다른 연산보다 먼저 실행됐다면 모든 복제 서버에서 그 순서로 처리되어야 함
    - 따라서 복제 서버가 연산을 처리할 때 인과적으로 앞서는 모든 연산이 이미 처리됐음을 보장할 수 있어야 함
    - 만일 선행 연산 중 누락된 게 있으면 후속 연산은 선행 연산이 완료될 때까지 기다려야함
- 어떤 연산이 다른 연산보다 먼저 실행됐는지 결정하려면 전체 데이터베이스에 걸친 인과적 의존성을 추적해야함
    - 이를 위해 버전 벡터를 일반화할 수 있음
    - **Q. 버전 벡터?**
        - 분산 시스템에서 각 노드가 어느 버전까지 알고 있는지 기록한 메타데이터
        - e.g. { A: 3, B: 2, …} 이렇게 되어있으면 A 에서 발생한 3번째 업데이트까지 반영, B 에서 발생한 2번째 업데이트까지 반영 … 이런 식임
- 인과적 순서를 결정하기 위해 데이터베이스는 애플리케이션이 데이터의 어떤 버전을 읽었는지 알아야 함
    - 이건 SSI(직렬성 스냅숏 격리 에서 설명한 SSI 충돌 검출 방식과 비슷한데, 트랜잭션이 커밋을 원할 때 데이터베이스가 해당 트랜잭션이 읽은 데이터의 버전이 여전히 최신 버전인지 확인하는 거임
    - 이를 목적으로 데이터베이스는 어떤 데이터를 어떤 트랜잭션이 읽었는지 추적함

### 일련번호 순서화

- 모든 인과적 의존성을 추적하는 것은 실용성이 떨어짐
- 그래서 일련번호나 타임슽탬프를 사용해 이벤트의 순서를 정함
    - 크기가 작고 인과성에 일관적인 전체 순서를 제공한다는 장점이 있음
- 단일 리더 복제 데이터베이스에서는 복제 로그가 인과성에 일관적인 쓰기 연산의 전체 순서를 정의함
    - 이거 바탕으로 팔로워가 쓰기 적용하면 리더에 뒤쳐지더라도 인과성에 일관적일 수 있음

#### 비인과적 일련번호 생성기

- 단일 리더 방식이 아니라 다중 리더 or 리더 없는 데이터베이스를 사용한다면 연산에 사용할 일련번호를 생성하는 방법이 명확해보이지 않음 ㅜ.ㅜ
    - 왜냐면 발행 주체가 여럿인데, 이들을 동기화하기가 까다롭기 때문
- 현실에서는 이렇게 함. 근데 이렇게 생성한 일련번호가 인과성에 일관적이지 않음
    - 각 노드가 독립적인 일련번호 집합을 생성하게 하기
        - e.g. A 는 홀수만, B 는 짝수만 생성하게 하는 식으로
        - 노드마다 초당 연산수가 달라 홀짝이 번갈아 나오지 않을 수 있음
            - 이 경우 인과적으로 어떤 게 먼저인지 알 수 없음
    - 각 연산에 일 기준 시계(물리적 시계) 에서 얻은 타임스탬프 붙이기
        - 해상도가 높다면 실용적
        - 물리적 시계에서 얻은 타임스탬프는 시계 스큐에 종속적임
            - 그래서 인과적으로 나중에 실행된 연산이 더 낮은 타임스탬프를 배정받을 수도 있음
    - 일련번호 블록을 미리 할당하기
        - A 는 1~1000, B 는 1001~2000 이렇게 범위 정해두고 범위 내에서 배정하기
        - B 노드에서 먼저 일련번호 받고, A 노드에서 받으면 일련번호가 인과성에 일관적이지 않게 됨

#### 램포트 타임스탬프

- 인과성에 일관적인 일련번호를 생성하는 간단한 방법이 잇음
- 램포트 타임스탬프를 사용하면 됨
    - (카운터, 노드 ID) 의 쌍으로 표현됨
    - 카운터 값은 처리한 연산의 개수임
    - 특정 노드가 자신의 카운터 값보다 더 큰 카운터를 가진 요청이나 응답을 받으면 자신의 카운터를 해당 값 + 1으로 설정함
    - 카운터 값이 같은 경우, 노드 ID 가 큰 것이 타임스탬프가 크다고 판정
- 즉, 모든 연산에 최대 카운터 값이 따라다니게 하는 방법임
- 버전 벡터랑 혼동될 수 있음
    - 참고로 버전 벡터는 각 노드가 해당 데이터 key 에 대해 몇 번째 버전까지 생성/반영했는지를 기록한 메타데이터임. 예전에 나왓음
    - 버전 벡터를 사용하면 두 연산이 동시적인지, 혹은 인과적으로 의존하는지 알 수 있지만, 램포트 타임스탬프는 항상 저체 순서화를 강제함
    - 램포트 타임스탬프의 전체 순서화로는 두 연산이 동시적인지 혹은 인과적으로 의존하는지 알 수 없음
        - 그냥 앞서서 연산이 있었다~ 를 아는 거지 인과적인지 동시적인지 이런 건 모른다는 거임
    - 버전 벡터는 레코드/객체 버전 관리 쪽에서 적용되고, 램포트 타임스탬프는 분산 이벤트 순서화 쪽 개념임. 둘이 다루는 영역이 다름
    - 그리고 램포트 타임스탬프가 크기가 더 작음

#### 타임스탬프 순서화로는 충분하지 않다

- 전체 순서는 모든 연산을 모아야 드러남
    - 예를 들어, 두 명의 사용자가 같은 아이디로 회원 가입을 하는 상황이라면, 한 명만 성공해야함
    - 근데 두 사용자가 각각 다른 노드에 쓰기를 진행한다면, 쓰는 시점에는 각 요청 중 어떤 것이 성공해야하고 실패해야하는지 알 수 없음
        - 왜냐면 다른 노드에서 동일한 아이디로 쓰는 중인지, 그 노드의 타임스탬프가 어떤 상태인지 모르니까
- 그래서 유일성 제약 조건 같은 건 전체 순서만으로 충분하지 않음
    - 언제 그 순서가 확정되는지도 알  수 있어야 함

### 전체 순서 브로드캐스트

- 방금 본 것처럼, 다중 노드 환경에서는 전체 순서를 확정하는 시점이 중요함
- 이를 가능하게 하는 게 전체 순서 브로드캐스트임
    - 노드 사이에 메시지를 교환하는 프로토콜로 기술되며, 두 가지 안전성 속성을 항상 만족해야함
        1. 신뢰성 있는 전달
            1. 어느 메시지도 손실되면 안되고, 메시지가 한 노드에 전달되면 모든 노드에 전달되어야 함
        2. 전체 순서가 정해진 전달
            1. 메시지는 모든 노드에 같은 순서로 전달되어야 함
- 노드나 네트워크에 결함이 있더라도 신뢰성과 순서화 속성이 항상 만족되도록 보장해야함
    - 네트워크 장애가 발생햇다면, 복구 이후에 메시지가 올바른 순서로 전달되게 해야함

#### 전체 순서 브로드캐스트 사용하기

- 상태 기계 복제
    - 모든 메시지가 데이터베이스에 쓰기를 나타내고, 모든 복제 서버가 같은 쓰기 연산을 같은 순서로 처리하면 복제 서버들은 서로 일관성 있는 상태를 유지한다는 것
    - 11장에서 다시 살펴볼 예정!
- 전체 순서 브로드캐스트는 직렬성 트랜잭션 구현에도 사용할 수 있음
- 전체 순서 브로드캐스트에서는 메시지가 전달되는 시점에 순서가 고정됨
    - 후속 메시지가 전달됏다면 그 순서의 앞에 메시지를 끼워넣지 못함
    - 그래서 타임스탬프 순서화보다 강함
        - **Q. 타임스탬프 순서화에서는 병합 과정에서 순서 조정이 되어서 그런 건가?**
            - 대충 맞는데 좀 더 보완하면 이렇게 됨
            
            > 타임스탬프 순서화는 각 이벤트에 순서값을 붙이지만, 그 순서가 언제 최종 확정되는지 보장하지 못한다. 반면 전체 순서 브로드캐스트는 메시지가 전달되는 순간, 그 앞에 어떤 메시지도 나중에 끼어들 수 없도록 순서를 확정한다.
            > 
- 전체 순서 브로드캐스트로 로그를 만들 수도 있고, 펜싱 토큰의 역할도 하게 할 수 있음

#### 전체 순서 브로드캐스트를 사용해 선형성 저장소 구현하기

- 전체 순서 브로드캐스트는 비동기식임
    - 그래서 메시지가 고정된 순서로 신뢰성 있게 전달되는 것은 보장되지만, 언제 전달될지는 보장되지 않음
- 선형성은 최신성 보장이 됨
    - 읽기가 최근에 쓰여진 값을 보는 것이 보장됨
- 선형성을 보장하기 위해서는 순서를 확정하는 시점이 중요했음(앞서 살펴본 아이디 생성 문제)
    - 이를 전체 순서 브로드캐스트로 해결할 수 있음
    - 사용 가능한 아이디마다 원자적 cas(compare and set) 연산이 구현된 선형성 저장소가 있다고 하면, 아이디를 생성하는 시점에 레지스터 값이 null 이라면 cas 연산을 수행해 아이디를 설정하게 할 수 있음
    - 만일 동시에 여러 사용자가 같은 아이디를 가지려 한다면 cas 연산 중 하나만 성공함
        - 다른 연산은 선형성 때문에 null 이 아닌 값을 보게 되기 때문
- 구현은 전체 순서 브로드캐스트를 append-only 로그로 사용해 할 수 있음
    - 첫 번째 요청이라면 수락하고 이전에 다른 요청이 있었다면 어보트 시킴으로써 구현 가능함
    - 로그는 모든 노드에 같은 순서로 전달되므로, 여러 개의 쓰기가 동시에 실행되면 모든 노드가 어떤 쓰기가 먼저 실행된 것인지 동의함
    - 로그 기반 직렬성 다중 객체 트랜잭션 구현도 이와 비슷하게 할 수 있음
- 근데 이건 선형성 쓰기만 보장하지, 선형성 읽기는 보장하지 않음
    - 비동기 갱신이기 때문에, 로그가 느릴 수 있음
    - 이를 해결하기 위해서 선택할 수 있는 선택지는 아래와 같음
        1. 로그에서 최신 로그 메시지의 위치를 선형적으로 얻고, 해당 위치까지의 모든 항목이 전달되기를 기다린 후에 읽기 수행
        2. 쓰기 실행 시에 동기적으로 갱신 보장하는 복제 서버에서 읽기

#### 선형성 저장소를 사용해 전체 순서 브로드캐스트 구현하기

- 전체 순서 브로드캐스트를 통해 보내고 싶은 모든 메시지에 선형성 정수로 increment-and-get 연산 수행하고, 레지스터에서 얻은 값을 일련번호로 메시지에 붙여서 모든 노드에 전송
- 후에 수신자들은 일련번호 순서대로 메시지를 전달하면 됨
- 예를 들어, 어떤 노드가 메시지 4를 전달하고 메시지 6을 받았다면, 메시지 6을 전달하기 전에 메시지 5를 기다리면 되는 거임
    - 램포트 타임스탬프에서는 이렇게 안 함. 거기서는 그냥 자기가 보내는 다음 메시지를 7로 설정함
    - 이게 전체 순서 브로드캐스트와 타임스탬프 순서화의 차이임
