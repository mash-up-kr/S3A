## 분산 트랜잭션과 합의

- 여러 노드들이 뭔가에 동의하게 만드는 것(비공식적 정의)
- 리더 선출, 원자적 커밋 등에서 사용

### 원자적 커밋과 2단계 커밋(2PC)

- 원자성은 중간 상태를 노출시키는 것을 방지해줌
    - 트랜잭션이 항상 성공하거나 실패한 상태로 남게 해줌
    - 다중 객체 트랜잭션과 보조 색인 유지에 중요함

#### 단일 노드에서 분산 원자적 커밋으로

- 단일 노드에서 실행되는 트랜잭션의 원자성은 저장소 엔진에서 구현됨
- 트랜잭션 커밋 요청 → DB 가 트랜잭션의 쓰기가 지속성 있게 함 → 디스크에 있는 로그에 커밋 레코드 추가
    - **Q. 쓰기가 지속성 있게 한다는 게 머임?**
        - 변경 기록을 WAL 로그에 남겨서 장애 발생해도 복구 가능하게 한다는 거임
        - ‘로그에 커밋 레코드 추가’ 는 트랜잭션이 최종적으로 commit 되었음을 WAL 로그에 표기하는 것
    - DB 가 죽으면 노드가 재시작 될 때 노드로부터 복구
- 단일 노드에서 트랜잭션 커밋은 데이터가 디스크에 지속성 있게 쓰여지는 순서에 결정적으로 의존함
- 트랜잭션이 커밋되거나 어보트되는지를 결정하는 시점은 디스크기 커밋 레코들 쓰기를 마치는 시점임
    - 이 시점 전에는 db 가 죽어서 커밋이 어보트 될 가능성이 잇지만, 이 시점 후에는 db 가 죽더라도 커밋된 상태임
- 근데 트랜잭션에 여러 노드가 관여한다면?
    - 다중 객체 트랜잭션을 사용하거나, 보조색인을 사용하거나
    - 이 경우에는 특정 노드에서만 성공할 수 잇기 때문에 원자성 보장을 위반하기 쉬움
        - **Q. 같은 데이터를 다루는데 왜 특정 노드에서만 성공할 수 있지? 네트워크 장애로 인한 상황이나 특정 노드가 죽은 상황 제외하고**
            - 각 노드가 가진 로컬 상태와 제약 조건이 다를 수 있기 때문
            - 모든 노드에 같은 트랜잭션을 수행하는 게 아니라는 걸 생각해야함
            
            ```lua
            A 노드: 주문 row insert 성공
            B 노드: 결제 row insert 실패
            ```
            
- 트랜잭션에 참여하는 다른 노드들도 커밋될 것이라고 확신할 때에만 커밋이 되어야 함
- 커밋된 트랜잭션을 되돌리는 것을 허옹하면 커밋 후 읽기의 보장이 깨지게 됨
    - 따라서 커밋 이후에 어보트 하는 것은 허용되지 않음
    - 되돌리는 트랜잭션을 발행해서 커밋시켜야함(보상 트랜잭션)

#### 2단계 커밋(2PC) 소개

- 여러 노드에 걸친 원자적 트랜잭션을 달성하는, 즉 모든 노드가 커밋되거나 모든 노드가 어보트되도록 보장하는 알고리즘
    - 코디네이터(coordinator) 라고도 함
- 커밋/어보트 과정이 두 단계로 나뉘어서 2단계 커밋임
- cf) 2단계 잠금(2PL) 과 혼동하면 안됨
    - 2PC 는 분산 데이터베이스에서 원자적 커밋을 제공하는 것
    - 2PL 은 직렬성 격리를 제공하는 것
- 애플리케이션이 커밋할 준비가 되면 코디네이터가 1단계를 시작함
    - 각 노드에 준비 요청을 보내서 커밋할 수 있는지 물어보고, 응답을 추적함
        - 모든 참여자가 ok 라고 응답하면 2단계에서 커밋 요청을 보내고 커밋이 실제로 일어남
        - 참여자 중 누구라도 no 라고 응답하면 2단게에서 모든 노드에게 어보트 요청을 보냄
        - **Q. 참여자는 그럼 ok 라고 응답하고 다른 커밋 요청을 어보트 시키나? 현 상태를 유지해야 하니까**
            - 무조건 abort 는 아니고, row 에 락 잡고 해당 row 에 쓰려는 다른 트랜잭션을 abort 하거나 대기시킴
        - **Q. 1단계에서 준비 요청만 보내고 아직 실제 요청을 보내지 않은 상태아님? 왜 no 라고 응답 받으면 모든 노드에게 어보트 요청을 보냄?**
            - 실제 요청(SQL)은 미리 전달 됨
            - 준비 요청은 이미 전달된 트랜잭션을 확정하기 위한 준비를 하라는 요청임
            - 그래서 no 라고 응답을 받으면 어보트 요청을 보내서 트랜잭션을 롤백시킴

#### 약속에 관한 시스템

- 준비 요청과 커밋 요청은 2단계 경우에서 쉽게 손실될 수 있는데, 왜 2PC 는 다를까?
    1. 애플리케이션은 분산 트랜잭션을 시작하기 원할 때 코디네이터에게 트랜잭션 ID 를 요청함
        1. 이 트랜잭션 ID 는 전역적으로 유일함
    2. 애플리케이션은 각 참여자에서 단일 노드 트랜잭션을 시작하고, 단일 노드 트랜잭션에 트랜잭션 ID 를 붙임
        1. 모든 읽기/쓰기는 단일 노드 트랜잭션 중 하나에서 실행됨
        2. 노드가 죽거나 타임아웃 발생하면 코디네이터나 참여자 중 누군가 어보트할 수 있음
    3. 애플리케이션이 커밋할 준비가 되면 코디네이터가 모든 참여자에게 전역 트랜잭션 ID 로 태깅된 준비 요청을 보냄
        1. 요청 중 실패하거나 타임아웃 발생하면 코디네이터는 모든 참여자에게 해당 트랜잭션 ID 로 어보트 요청 보냄
    4. 참여자가 준비 요청 받으면 트랜잭션 커밋 가능한지 확인함
        1. 트랜잭션 데이터를 디스크에 쓰는 데 용량이 충분한지, 충돌이나 제약 조건 위반 안하는지 등등 체크하고 yes/no 응답 보냄
    5. 코디네이터가 모든 준비 요청에 대해 응답을 받으면 커밋/어보트에 대한 최종 결정을 함
        1. 모든 참여자들이 yes 라고 하면 커밋, 아니면 어보트
        2. 이 결정을 트랜잭션 로그에 남김(커밋 포인트)
    6. 결정이 로그로 작성되면 모든 참여자에게 커밋이나 어보트 요청이 전송됨
        1. 이 요청이 실패하거나 타임아웃 발생하면 코디네이터는 성공할 때까지 재시도해야함
        2. **Q. 근데 결정을 로그로 작성하고 커밋/어보트 요청 보내면, 코디네이터가 죽고 재시작 하는 경우에 해당 요청이 커밋/어보트 되었는지 확인 못하는 거 아님? 순서가 커밋/어보트 하고 로그 작성해야 재시작해도 확신 가능한 거 아님?**
            1. 만약 커밋/어보트 확인 이후에 로그 작성한다면, 로그 작성 전에 죽은 경우 기존에 내린 결정을 알 수 없는 문제가 발생하게 됨
            2. 그래서 결정을 로그로 먼저 작성함. 이러면 재시작 됐을 때 같은 결정을 노드에게 다시 내려 원자성을 보장할 수 있음
- 이 프로토콜에는 돌아갈 수 없는 지점이 두 개 있음
    1. 참여자가 yes 라고 응답하는 시점
    2. 코디네이터가 커밋/어보트 결정하는 시점
- 이러한 약속이 2PC 의 원자성을 보장함

#### 코디네이터 장애

- 코디네이터가 준비 요청을 보내기 전에 장애가 발생하면 참여자가 안전하게 트랜잭션 어보트 가능함
- 근데 참여자가 준비 요청 받고 yes 라고 투표하면 일방적으로 어보트할 수 없음
    - 코디네이터로부터 트랜잭션이 커밋됏는지 어보트됐는지 회신 받을 때까지 기다려야함
    - 이 때 코디네이터가 죽거나 네트워크 장애 발생하면 참여자는 기다려야함
        - 이 상태에 있는 참여자의 트랜잭션을 의심스럽다 또는 불확실하다 고 함
    - 참고로 타임아웃도 도움이 되지 앟음
        - 특정 노드만 타임아웃으로 어보트하고, 다른 참여자만 커밋할 수 있기 때문
        - 각 노드 간의 데이터 정합성 보장이 되지 않을 수 있기 때문에, 다른 신호가 아닌 코디네이터의 응답으로만 결정을 내려야 함
- 이론상으로는 참여자끼리 통신해서 각 참여자의 투표 여부를 알아낸 후 합의가 가능하지만, 이건 2PC 프로토콜의 일부가 아님
- 그래서 코디네이터는 참여자들에게 커밋이나 어보트 요청을 보내기 전에 트랜잭션 로그에 결정을 씀(커밋 포인트 작성)
    - 코디네이터가 복구될 때 트랜잭션 로그를 읽어서 모든 의심스러운 트랜잭션의 상태를 결정하기 때문
    - 코디네이터의 로그에 커밋 레코드가 없는 트랜잭션들은 어보트됨
    
    **Q. 그럼 코디네이터는 재시작하자마자 모든 커밋 포인트를 검사해서 의심스러운 트랜잭션 상태를 결정해야함? 리소스 많이 들 거 같은데? 그리고 모든 참여자의 트랜잭션 상태를 코디네이터가 어떻게 확인함?**
    
    - 모든 커밋 포인트 검사하지 않음. 코디네이터 로그에는 이런 정보가 남음
    
    ```lua
    T1 시작됨
    참여자: A, B, C
    prepare 진행됨
    최종 결정: commit 또는 abort
    T1: decision = COMMIT, completed = false
    T2: decision = ABORT, completed = false
    T3: decision 없음
    T4: completed = true
    ```
    
    - 이 로그를 보고 의심스러운 트랜잭션을 가진 노드에게 커밋/어보트 요청을 다시 보내는 것

#### 3단계 커밋

- 지연에 제한이 있는 네트워크와 응답 시간에 제한이 있는 노드를 가정하기 때문에, 실용적 시스템에서 원자성을 보장하지 못함
- 완벽한 장애 감지기가 필요하나, 불가능하기 때문에, 블로킹 가능성이 있는 2PC 를 계속 사용함

### 현실의 분산 트랜잭션

- 분산 트랜잭션이 안정성 보장을 제공하기는 하나, 운영상 문제 때문에 사용하지 않기도 함
    - MySQL 의 분산 트랜잭션은 단일 노드 트랜잭션보다 10배 느리다고 보고됨
    - 2단계 커밋에 내장된 성능 비용의 많은 부분은 장애 복구를 위해 필요한 부가적인 디스크 강제 쓰기와 부가적인 네트워크 왕복 시간 때문임
        - **Q. 여기서 부가적인 디스크 강제 쓰기는 커밋 포인트가 맞나?**
            - 맞는데 더 잇음
                - 참여자의 준비 로그 기록
                - 커밋 포인트 기록
                - 참여자의 커밋/어보트 결과 기록
                - 코디네이터의 완료 상태 기록
- 분산 트랜잭션은 두 가지 종류임
    - 데이터베이스 내부 분산 트랜잭션
        - 데이터베이스 노드 사이에 내부 트랜잭션을 지원하는 경우가 있음
            - e.g. Mysql, vaultDB
        - 트랜잭션에 참여하는 모든 노드가 동일한 데이터베이스 소프트웨어 실행함
            - 그래서 호환성에 대한 부담이 적고, 아무 프로토콜이나 사용할 수 잇음
            - 특정 기술에 특화된 최적화를 사용할 수도 잇음
    - 이종 분산 트랜잭션
        - 이종 트랜잭션에서 참여자들은 둘 이상의 다른 기술임
            - 두 가지 서로 다른 벤더의 데이터베이스 혹은 메시지 브로커처럼 비데이터베이스 시스템일 수도 있음
        - 서로 다른 시스템에 걸친 분산 트랜잭션은 시스템 내부가 완전히 다르더라도 원자적 커밋을 보장해야 함

#### 정확히 한 번 메시지 처리

- 분산 트랜잭션이 지원되면 메시지 브로커와 데이터베이스가 서로 다른 장비에서 실행되는 두 가지 무관한 기술이더라도 원자적 커밋이 가능함
- 메시지 전달이나 데이터베이스 트랜잭션 중 하나가 실패하면 둘 다 어보트되고 메시지 브로커는 나중에 메시지를 안전하게 다시 전달할 수 잇음
- 메시지와 그 처리 과정의 부수 효과를 원자적으로 커밋함으로써 메시지가 결과적으로 정확히 한 번 처리되도록 보장이 가능함
    - 성공까지 몇 번의 재시도가 발생할 수는 잇음
- 근데 이게 가능하려면 트랜잭션 영향을 받는 모든 시스템이 동일한 원자적 커밋 프로토콜 사용이 가능해야함
    - 예를 들어 메시지 처리의 부수 효과를 처리할 때 2단계 커밋을 지원하지 않으면, 메시지 처리가 실패해 재시도 되는 경우 부수 효과가 두 번 이상 나타날 수 있음
    - 2단계 커밋 지원하면 실패하는 경우 부수 효과가 롤백되어 결과적으로 한 번만 처리됨

#### XA 트랜잭션

- 이종 기술에 걸친 2단계 커밋을 구현하는 표준
    - Postgresql, Mysql 등등 전통적인 관계형 DB 와 메시지 브로커에서 지원됨
- 네트워크 프로토콜이 아니라, 트랜잭션 코디네이터와 연결되는 인터페이스를 제공하는 C API 임
    - **Q. C API 가 머임?**
        - C 언어로 만들어진 API 라는 뜻임
        - 즉, XA 는 HTTP, TCP 기반의 네트워크 프로토콜이라기보다, C 언어 함수 호출 형태로 정의된 표준 인터페이스라는 것임
    - 자바에서는 JTA(자바 트랜잭션 API) 를 사용해 구현되며, 이는 JDBC 를 사용하는 DB 용 드라이버 다수와 JMS(자바 메시지 서비스) API 를 사용하는 메시지 브로커용 드라이버에서 지원됨
- 드라이버가 XA 를 지원한다는 것은 연산이 분산 트랜잭션의 일부가 되어야 하는지 알아내기 위해 XA API 를 호출한다는 것을 뜻함
    - 만약 그렇다면 드라이버는 데이터베이스 서버로 필요한 정보를 보냄
    - 드라이버는 코디네이터가 참여자에게 준비, 커밋, 어보트를 요청할 수 있는 콜백도 제공함
        - **Q. 이게 왜 콜백 제공임?**
            - 콜백이라고 적으면 헷갈릴 수 있음
            - 정확히는 드라이버가 코디네이터에게 호출할 수 있는 표준 제어 인터페이스를 제공한다는 뜻임
            
            ```lua
            prepare 호출 지점
            commit 호출 지점
            rollback 호출 지점
            recover 호출 지점
            ```
            
- 트랜잭션 코디네이터는 XA API 를 구현함
    - 트랜잭션을 시작하는 애플리케이션과 같은 프로세스에 로딩디는 라이브러리임
    - 트랜잭션의 참여자를 추적하고, 참여자들에게 드라이버가 제공한 콜백을 통해주니 요청 보낸 후, 응답을 수집하고 각 트랜잭션에 대한 커밋/어보트 결정을 추적하기 위해 로컬 디스크에 있는 로그를 사용함
        - **Q. 여기서 로컬 디스크의 로그는 커밋 포인트가 맞나?**
            - ㅇㅇ 마즘
- 애플리케이션 프로세스가 죽거나, 애플리케이션이 실행 중인 장비가 죽으면 코디네이터가 사라지고, 준비됐지만 커밋되지 않은 트랜잭션을 가진 참여자들은 의심스러운 상태에 빠짐
    - 커밋 포인트는 애플리케이션 서버의 로컬 디스크에 있으므로 해당 서버가 재시작 되어야 커밋/어보트 결과 판정이 가능함

#### 의심스러운 상태에 있는 동안 잠금을 유지하는 문제

- 의심스러운 상태에 있는 동안 로우에 대한 잠금을 획득하기 때문에, 의심스러운 트랜잭션을 무시할 수 없음
    - 2단계 잠금을 사용하면 읽은 로우에도 공유잠금이 걸려서, 해당 잠금이 풀릴 때까지 쓰기가 불가능함
- 이 잠금이 유지되는 동안 다른 트랜잭션은 해당 로우를 변경하거나, 심지어 읽을 수 없기도 함

#### 코디네이터 장애에서 복구하기

- 이론상으로는 코디네이터가 재시작하면 상태를 깨끗하게 복구하고 의심스러운 트랜잭션을 해소해야 함
- 그러나 현실에서는 고아가 된 의심스러운 트랜잭션이 발생할 수 있음
    - 어떤 이유로 결과를 결정할 수 없는 트랜잭션이 생길 수 잇다는 것
    - e.g. 트랜잭션 로그 손실, 소프트웨어 버그로 인한 오염 등
- 이러한 트랜잭션은 자동으로 해소될 수 없어 락을 점유하고 다른 트랜잭션을 차단하게 됨
- 이를 해결하기 위해서는 관리자가 수동으로 개입해야함
    - 의심스러운 트랜잭션의 참여자를 조사해서 커밋/어보트 결과를 확인하고 동일한 결과를 다른 참여자들에게도 적용해야함
- 너무 오래 대기하는 것은 리스크가 크기 때문에, XA 구현에는 참여자가 코디네이터로부터 확정적 결정을 얻지 않고 의심스러운 트랜잭션을 결정할 수 잇도록 하는 경험적 결정을 구현해 둠
    - **Q. 어떻게 동작함? 다른 참여자 조사하나?**
        
        ```lua
        관리자가 강제로 commit
        관리자가 강제로 rollback
        리소스 매니저가 timeout 정책에 따라 heuristic rollback
        ```
        
        - 근데 코디네이터 기반 의사 결정이 아니라 원자성이 깨질 수 잇음
        - 경험적이라는 말은 2단계 커밋의 원자성 보장을 위반할 수 있다는 뜻임

#### 분산 트랜잭션의 제약

- XA 트랜잭션도 운영상 문제가 잇음
    - 핵심 구현은 트랜잭션 코디네이터 자체가 트랜잭션 결과를 저장할 수 있는 일종의 데이터베이스여야 한다는 점이고, 다른 중요한 데이터베이스와 동일하게 신경 써서 접근해야함
        - 코디네이터가 복제되지 않으면 SPOF(단일 장애 지점) 이 될 수 있음
        - 서버 사이드 애플리케이션은 영속 상태를 DB 에 저장하고, 상태 비저장 모드로 개발되나, 코디네이터가 애플리케이션 서버의 일부가 되면 배포 특성이 바뀌어 로그가 지속적인 시스템의 중대한 부분이 됨
            - 이러면 더 이상 상태 비 저장 모드가 아니게 됨
            - **Q. 그럼 커밋 포인트 저장을 DB 에 하면 해결할 수 있는 문제 아님?**
                - 완전한 해결은 아니나, 애플리케이션은 stateless 하게 만들 수잇음
                - 상태를 저장소로 이동한 것이기 때문에 시스템 전체적으로는 여전히 코디네이터의 state 가 존재함

### 내결함성을 지닌 합의

- 하나 이상의 노드들이 값을 제안할 수 있고 합의 알고리즘이 그 값 중 하나를 결정하는 방식으로 합의 문제가 형식화 됨
- 이 때 합의 알고리즘은 이런 속성을 만족해야 함
    - 균일한 동의
        - 어떤 두 노드도 다르게 결정하지 않는다
    - 무결성
        - 어떤 노드도 두 번 결정하지 않는다
    - 유효성
        - 한 노드가 값 v 를 결정한다면, v 는 어떤 노드에서 제안된 것이다
    - 종료
        - 죽지 않은 모든 노드는 결국 어떤 값을 결정한다
- 합의 시스템 모델은 노드가 죽으면 돌아오지 않는다고 가정함
    - 노드가 복구되기를 기다리는 알고리즘은 종료 속성을 만족할 수 없음
    - 2PC 는 코디네이터가 돌아와 결정을 내린다고 가정하기 때문에 종료 속성을 만족하지 않음
- 종료 속성은 죽거나 연결할 수 없는 노드 개수가 절반 미만이라는 가정에 종속적임
    - 그러나 대부분의 합의 구현은 과반수의 노드에 장애가 나거나 심각한 네트워크 문제가 잇더라도 안정성 속성(동의, 무결성, 유효성) 을 항상 만족함
    - 따라서 대규모 중단이 발생하면 시스템이 요청을 처리할 수 없게 될 수 있지만 유효하지 않은 결정을 내려서 합의 시스템을 오염시키지는 않음

#### 합의 알고리즘과 전체 순서 브로드캐스트

- 내결함성을 지닌 합의 알고리즘 중 대다수는 위에서 설명한 합의 알고리즘의 속성을 직접 사용하지 않음
- 대신 값의 순차열(sequence) 에 대해 결정해서 전체 순서 브로드캐스트 알고리즘을 만듦
    - 전체 순서 브로드캐스트는 모든 노드에게 메시지가 정확히 한 번, 같은 순서로 전달되어야 함
    - 이걸 바꿔 생각하면 합의를 여러 번 하는 거임
        - 각 회마다 특정 노드가 다음에 보낼 메시지를 제안하고 결정하는 것
        - **Q. 근데 전체 순서 브로드캐스트에서는 wal 로그 그대로 공유하는 거 아니엇음?**
            - WAL 로그 자체를 전달하는 게 아니라, 모든 노드가 같은 순서로 로그를 작성하게 메시지를 전달하는 거임
            - 최종적으로는 논리적인 WAL 로그가 다 같아지게 됨
                - 저장 포맷이나 체크포인트, 스냅샷/로그 압축 등이 있을 수 있으므로, 물리적으로 WAL 파일 바이트가 완전히 동일하다는 뜻은 아님

#### 단일 리더 복제와 합의

- 단일 리더 복제에서 하는 것은 쓰기 요청을 리더에게 전달하고, 쓰기를 같은 순서로 팔로워에 적용하는 거엿음
- 이건 넓게 보면 전체 순서 브로드캐스트인데, 왜 합의에 대해 걱정을 하지 않았을까?
    - 리더를 선택하는 방식에 따라 답이 달라짐
        - 수동으로 리더를 선택햇다면 독재자 방식의 합의 알고리즘을 사용한 것임
        - 자동 선출하는 것은 내결함성을 지닌 전체 순서 브로드캐스트에 가까워지는 것이고, 합의에 가까운 방식임
    - 리더를 선출하려면 합의가 필요함. 근데 합의 알고리즘이 사실 전체 순서 브로드캐스트 알고리즘이라면 전체 순서 브로드캐스트는 단일 리더 복제와 같아 리더가 필요함
        - 전체 순서 브로드캐스트를 하려면 리더가 필요한데, 리더가 없는 상황에서 리더 선출을 하는 알고리즘으로 사용한다면 리더 선출을 위해 리더가 필요한 것처럼 보인다는 거임

#### 에포크 번호 붙이기와 정족수

- 합의 프로토콜은 내부적으로 리더를 사용하지만, 리더가 유일하다고 보장하지는 않음
- 합의 프로토콜들은 에포크 번호를 정의하고 각 에포크 내에서는 리더가 유일하다고 보장함
- 현재 리더가 죽었다고 생각될 때마다 리더 선출을 함
    - 이 선출은 에포크 번호를 증가시킴. 따라서 에포크 번호는 전체 순서가 있고 단조 증가함
- 두 가지 다른 에포크에 있는 두 가지 다른 리더 사이에 충돌이 발생하면 에포크 번호가 높은 리더가 이김
    - 그래서 리더의 결정을 허용하기 전에 해당 리더보다 에포크 번호가 높은 다른 리더가 있는지 먼저 확인해야함
- 그래서 두 번의 투표가 있음
    - 리더 선출 투표
    - 리더의 제안에 투표
    - 두 번의 투표를 하는 정족수가 겹쳐야 함
        - 리더 제안에 대한 투표가 성공하면, 투표한 노드 중 최소 하나는 리더 선출에도 참여했어야 함
        - **Q. 왜 리더 선출에 참여하지 않은 노드만으로 구성되면 안됨?**
            - 두 가지 문제가 있음
                1. 이전 에포크에서의 값을 새 리더가 모르고 제안할 수 있음
                    - 합의에서 말하는 값은 DB 에 반영된 값이 아니라, 복제 로그에 기록된 값임
                        - 로그 항목이 충분히 복제되어 커밋되면 DB 에 적용되는 것
                    - 그래서 모든 노드가 값을 알지 못하는 상태에서 리더 교체가 발생할 수 잇음
                    - 근데 합의에서는 정족수 이상의 노드에 반영이 되엇으면 합의가 되엇다고 봄
                        - 즉, 특정 값이 정족수 이상의 노드에 잇으면 합의된 값임
                    - 리더가 선출될 때, 참여 노드는 리더 노드에게 자기가 알고 잇는 최신 로그 정보를 제공함
                        - 그래서 리더 선출에 참여한 노드가 리더 제안에 투표해야함
                    - 만일 리더 선출에 참여하지 않은 노드만으로 리더 제안 투표가 성공햇다면, 최신 값이 아닌 상태에서 제안이 승인될 수 잇음
                2. 이전 리더의 제안이 더 높은 에포크를 무시하고 승인될 수 있음
                    - 리더 선출에 참여하지 않은 노드가 리더 제안에 투표하면, 리더 선출에 참여하지 않았기 때문에 현재 제안한 노드가 최신 에포크의 노드인지 알 수 없음

#### 합의의 제약

- 합의 알고리즘을 모든 곳에 사용하지는 않음. 제약이 따르기 때문
    - 리더의 제안에 노드가 투표하는 것은 일종의 동기식 복제임
    - 합의 시스템은 항상 엄격한 과반수가 동작하기를 요구함
    - 대부분의 합의 알고맂므은 투표에 참여하는 노드 집합이 고정돼있다고 가정함
        - 따라서 클러스터에 노드를 그냥 추가하거나 제거할 수 없음
        - 합의 알고리즘의 동적 멤버십 확장은 클러스터에 있는 노드 집합이 시간이 지남에 따라 바뀌는 것을 허용하기도 함
    - 합의 시스템은 장애 노드를 감지하기 위해 일반적으로 타임아웃에 의존함
        - 네트워크 지연 변동이 심하면 리더 선출에 많은 리소스를 사용하게 될 수 잇음
