## 결함과 부분 장애

- 이상적인 시스템의 동작
    - 시스템 전체는 성공하거나 실패해야 함. 중간 상태라는 것은 없음
    - (단일 컴퓨터의 경우) 하드웨어가 올바르게 동작하면 연산은 항상 같은 결과를 내야함(결정적이어야함)
- 그러나 분산 시스템은 물리 세계의 한계로 이상화된 시스템 모델에서 동작할 수 없음
    - 네트워크 문제, 전력 공급 문제, 그 외 다른 문제 등등..
- 이렇게 분산 시스템에서 특정 부분이 예측할 수 없는 방식으로 고장나는 걸 ‘부분 장애’ 라고 함
    - 부분장애는 비결정적임
        - 성공과 실패를 예측하기 어렵다는 것!
    - 심지어 성공했는지조차 알 수 없을 때도 있음
        - 응답이 오는 시간 또한 비결정적이기 때문

### 클라우드 컴퓨팅과 슈퍼 컴퓨팅

- 대규모 컴퓨팅 시스템 구축 방법
    1. 고성능 컴퓨팅(High Perfomance Computing, HPC)
    2. 클라우드 컴퓨팅
    - 전통적인 기업형 데이터센터는 이 중간에 위치
- 슈퍼컴퓨터의 결함 처리 방식
    - 부분 장애를 전체 장애로 확대하는 방식을 사용함
        - 계산 상태를 지속성 있는 저장소에 체크포인트로 저장
        - 노드 하나에 장애가 발생하면, 전체 클러스터 작업부하를 중단
        - 장애가 발생한 노드가 복구된 후 마지막 체크포인트로부터 계산을 재시작
    - 분산 시스템보다는 단일 노드 컴퓨터에 가까움
        - 부분 장애 발생하면 단일 장비의 커널 패닉과 비슷하게 전체가 죽게 함
- 슈퍼컴퓨터 vs 클라우드 서비스
    - 하드웨어
        - 슈퍼컴퓨터는 특화된 하드웨어 사용함
            - 각 노드는 신뢰성이 높고, 공유 메모리와 원격 직접 메모리 접근을 사용해 통신
                - **Q. 공유메모리? 원격 직접 메모리 접근?**
                    - 공유메모리는 여러 CPU 나 프로세스가 같은 메모리 공간을 함께 접근하는 것
                        - e.g. 하나의 서버 안에 CPU 코어가 여러 개 있으면, 여러 스레드가 같은 RAM 데이터 읽고 쓰는 것
                    - 원격 직접 메모리 접근(RDMA, Remote Direct Memory Access) 는 한 서버가 다른 서버의 메모리에 직접 데이터를 읽거나 쓰는 기술
                        - 이 과정에서 커널이나 CPU 의 개입을 줄여서, 네트워크 카드가 상대 서버의 메모리에 직접 접근하는 것처럼 동작함
                        - **Q. 이거 위험해보이는데**
                            - 맞음. 커널 우회하고 원격 메모리 직접 접근하는 구조라서 보안 상 취약함
                            - 그래서 이런 조건을 걸어둠
                                - 접근 가능한 메모리 영역을 미리 등록함
                                - 해당 메모리 영역에 대한 key 또는 token이 필요함
                                - 접근 권한을 read/write/send 등으로 제한함
                                - RDMA NIC와 드라이버가 권한 검사를 수행함
                                - 데이터센터/HPC 내부처럼 신뢰된 네트워크에서 주로 사용함
    - 클라우드 서비스는 상용 장비 사용함
        - 낮은 비용으로 동일한 성능을 제공하나, 실패율이 높음
    - 통신
        - 슈퍼컴퓨터는 다차원 메시(mesh) 나 토러스(torus) 같은 특화된 네트워크 토폴로지 사용
            - **Q. 이거 뭐임?**
                - 메시는 노드들이 격자처럼 연결된 구조, 토러스는 메시의 양 끝을 다시 연결한 구조
                - 예를 들어 2차원 메시라면 각 노드가 상하좌우 이웃 노드와 연결되고, 토러스까지 적용되면 맨 왼쪽과 맨 오른쪽, 맨 위쪽과 맨 아래쪽을 연결한 도넛처럼 이어진 구조가 됨
            - 통신 패턴이 정해진 작업 부하에서 높은 성능을 보여주기 때문에 사용함
                - 예를 들어, 시뮬레이션에서 각 노드가 주변 격자 영역 담당하고, 이웃 노드와 자주 통신하는 구조에서 효율적임
                    - e.g. 기상 예측이나 물리 시뮬레이션의 경우, 하나의 노드의 계산이 주변 노드에 영향을 미침
                    - 슈퍼컴퓨터가 푸는 문제의 특성이 격자 기반 문제이기 때문애 사용함
        - 데이터센터는 클로스 토폴로지 사용
            - **Q. 이건 뭐임?**
                - 이건 데이터센터에서 많이 쓰는 계층형 스위치 네트워크 구조임
                - leaf-spine 구조로 이해하면 됨
                    - 서버는 leaf 스위치에 연결되고, leaf 는 여러 spine 스위치와 연결됨
                    - 한 서버에서 다른 서버로 갈 때 보통 leaf → spine → leaf 경로를 탐
                - 그래서 데이터센터 내부의 여러 서버 쌍이 동시에 통신할 때 비교적 높은 대역폭을 제공할 수 있음
                    - **Q. 이건 spine 을 거치기 때문인가? spine 이 많은 대역폭을 가지기 때문에?**
                        - 더 정확하게는, 여러 spine 으로 경로를 분산할 수 있기 때문임
            - IP 와 이더넷 기반으로 하며 높은 양단 대역폭을 제공하기 때문
            - **Q. 왜 슈퍼컴퓨터와 데이터센터는 통신 방식이 다른가?**
                - 슈퍼컴퓨터의 경우, 격자 기반의 문제를 풀기 때문에 통신 패턴이 정해진 반면, 데이터센터는 통신 패턴이 정해지지 않아 범용 구조가 적합하기 때문!
- 분산 시스템이 동작하게 하려면 부분 장애 가능성을 받아들이고 소프트웨어에 내결함성 메커니즘을 넣어야 함
    - 즉, 각 구성 요소의 신뢰성이 없다고 가정하고, 이를 사용해 신뢰성 있는 시스템을 구축하는 것이 목표임
        - e.g. 오류 수정 코드(송수신 시 정보가 정확히 전달되었는지 확인하는 용도), TCP(패킷 누락, 순서 변경 등의 이유로 신뢰성 없는 IP 프로토콜 위에서 동작하며, 신뢰성을 높여줌)

## 신뢰성 없는 네트워크

- 인터넷과 데이터센터 네트워크 대부분은 비동기 패킷 네트워크임
    - **Q. 대부분이라면 그렇지 않은 것도 잇음?**
        - 있음. 대표적으로 회선 교환 네트워크, 시간 민감형 네트워크(TSN) 등등
            - **Q. TSN 이 머임**
                - 기본 이더넷은 패킷 지연 시간이 상황에 따라 달라질 수 있음
                - TSN 은 이더넷 위에서 특정 트래피에 대해 지연 시간, jitter, 전송 순서, 신뢰성을 더 잘 보장하려는 표준 기술 묶음임
                - 즉, 실시간 제어에 더 적합한 이더넷을 구성하기 위한 기술임
                - 시간 슬룻 예약, 우선순위 제어, 트래픽 셰이핑 등을 통해 지연을 더 예측 가능하게 함
                    - **Q. 시간 슬룻 예약? 트래픽 셰이핑?**
                        - 시간 슬룻 예약
                            - 특정 시간 구간에는 특정 트래픽만 보내도록 예약함
                            - 예를 들어 1ms를 여러 구간으로 나누고:
                                - 0~0.2ms: 제어 메시지
                                - 0.2~0.5ms: 센서 데이터
                                - 0.5~1ms: 일반 데이터
                        - 트래픽 셰이핑
                            - 트래픽이 한 번에 몰리지 않도록 전송 속도나 패턴을 조절하는 방식
                            - 네트워크에 들어가는 트래픽의 모양을 다듬어서 혼잡과 지연 변동을 줄이는 방식
- 이 과정에서 노드는 다른 노드로 패킷을 보낼 수 잇지만, 네트워크는 메시지 도착 시간이나 도착 여부를 보장하지 않음
    - 유일한 선택지는 수신 측에서 응답을 보내는 것이나, 이마저도 손실되거나 지연될 수 있음
- 그래서 타임아웃을 사용함
    - 일정 시간이 지나면 응답 대기를 멈추고 응답이 도착하지 않는다고 가정하는 것
    - 그러나 이 경우에도 수신 측에서 패킷을 받았는지, 받지 않았는지는 알 수 없음

### 현실의 네트워크 결함

- 아직까지 신뢰성 있는 네트워크를 만들지는 못함
- 네트워크 중단의 주요 원인은 인적 오류임
    - 따라서 네트워크 장비를 중복 추가하는 것은 결함을 줄여주지 못함
- 현실에서는 여러 문제들이 나타남
    - 스위치 소프트웨어 업그레이드 중, 네트워크 토폴로지가 재구성될 수 있고, 이 경우 네트워크 패킷이 1분 이상 지연될 수 있음
    - 상어가 해저 케이블을 손상시킴 (ㄷㄷ)
    - 수신 패킷은 누락하나, 송신 패킷은 잘 보내는 네트워크 인터페이스(NI) 가 있을 수 있음
        - **Q. 어떤 경우가 있음?**
            - 네트워크 카드(NIC) 나 드라이버, 케이블, 스위치 포트 등에 문제가 발생하면 한 방향 통신만 깨질 수 있음
- 이러한 문제들이 항상 나타날 수 있다는 가정 하에, 소프트웨어가 이를 처리할 수 있게 설계해야함
    - 즉, 소프트웨어가 네트워크 문제에 어떻게 반응하는지 알고, 시스템이 그로부터 복구할 수 있도록 보장해야함

### 결함 감지

- 네트워크의 불확실성 때문에 특정 노드가 죽었는지 확인할 확실한 방법은 없음
- 그러나 특정 환경에서는 동작하지 않는다는 명시적인 피드백을 받을 수는 있음
    - 노드가 실행중인 장비에 연결은 가능하나, 목적지 포트에 실제 프로세스가 대기하지 않는 경우(프로세스가 죽거나 하는 이유로) 수신 운영체제가 RST 나 FIN 응답을 보내줌
    - 데이터센터 내 스위치의 관리 인터페이스에 접근할 수 있다면 질의를 보내 하드웨어 수준의 링크 장애를 감지할 수 있음
        - **Q. 스위치가 db 형태로 연결된 장비를 관리하나?**
            - 일반적인 DB 라기보다는, 내부적으로 여러 테이블을 관리함
                - MAC 주소 테이블, ARP 테이블, 라우팅 테이블, NAT 등등..
                    - **Q. 이거 상태를 주기적으로 프로토콜로 확인하고 기록하는 거지?**
                        - 테이블마다 다름
                        - 예를 들어, MAC 주소 테이블은 트래픽을 보면서 자동 학습하고, ARP 는 주기적으로 ARP 요청 보내서 확인함
    - Dest IP 에 도달할 수 없다고 라우터가 확신하는 경우, ICMP Destination Unreachable 패킷으로 응답함
        - **Q. TCP 로 요청 보내는데 응답이 ICMP 로 오나?**
            - ICMP 는 TCP 와 별개로 IP 계층의 오류 보고 및 제어 메시지를 담당함
            - ICMP 는 IP 패킷 안에 담겨서 전달됨
            - e.g. TCP 패킷을 보냈는데, 중간 라우터가 목적지로 못 간다! 라고 판단하면, 라우터가 TCP 응답이 아니라 ICMP 를 주는 거임
                - 즉, 내가 TCP 로 응답했어도 ICMP 로 응답이 돌아올 수 있음
                    - **Q. 그러면 이게 내가 보낸 TCP 에 대한 ICMP 인지는 어떻게 알아? TCP 는 SIP, DIP, SPORT, DPORT, seq, ack 등으로 자기 패킷인 걸 아는데, 이거 IP 계층 패킷이면 이 정보 없는 거 아님?**
                        - ICMP 는 단순히 오류만 보내는 게 아니라, 문제가 된 원래 IP 패킷의 일부를 같이 실어 보냄
                        - 정확히는 원래 IP 헤더와 그 뒤 payload 의 앞부분을 포함해서 보냄
                        - TCP 의 경우 payload 앞부분에 TCP 헤더가 있고, TCP 헤더 앞부분에는 source port 와 destination port 가 있으므로, OS 는 이 정보를 이용해 어떤 TCP 연결과 관련된 ICMP 오류인지 매핑할 수 있음
- TCP 가 패킷이 전달됏다는 확인 응답(ack) 를 했더라도, 애플리케이션이 그걸 처리하기 전에 죽을 수도 있음
    - 즉, 요청 자체의 최종 성공은 애플리케이션에서 받아야함
    - **Q. 근데 요청을 받기만 햇을 때 ack 를 주나? 보통 애플리케이션에서 다 처리하고 ack 에 payload 랑 같이 보내지 않음?**
        - TCP ACK 는 애플리케이션 처리가 끝났다는 게 아니라, TCP 계층에서 바이트를 받았다는 확인임
        - 그래서 TCP ACK 을 받았다고 해서 요청이 성공한 것은 아님. 애플리케이션이 연산을 마치면 응답을 다시 TCP 로 보내줌
            - **Q. 근데 그럼 보냄 - 받음(TCP 계층 확인) - 받음(애플리케이션 응답) 이렇게 오는 거야?**
                - 개념적으로는 맞음
                - HTTP 요청을 TCP 위에서 보내는 경우 이렇게 동작함
                    1. 클라이언트 애플리케이션이 HTTP 요청 바이트를 TCP에 씀
                    2. 클라이언트 TCP가 세그먼트 전송
                    3. 서버 TCP가 세그먼트 수신
                    4. 서버 TCP가 ACK 전송
                    → “바이트는 TCP 계층까지 도착했다”
                    5. 서버 애플리케이션이 소켓에서 요청을 읽음
                    6. 서버 애플리케이션이 비즈니스 로직 처리
                    7. 서버 애플리케이션이 HTTP 응답 바이트를 TCP에 씀
                    8. 클라이언트 애플리케이션이 HTTP 응답을 받음
                    → “애플리케이션 처리가 끝났다”

- 무언가 잘못된 경우, 오류 응답을 받을 수도 잇지만 일반적으로는 아무 응답도 받지 못한다고 가정해야함
    - 이 경우, 재시도 해보고(TCP 는 자동으로 재시도 해줌. 이게 flag 에 R 붙여서 나가나 그럼) 타임아웃이 만료된다면 노드가 죽었다고 선언함
        - **Q. 근데 자동 재시도 한다면 요청이 두 번 처리될 가능성 잇는 거 아닌가?**
            - TCP 재전송이 발생해도, TCP 는 자체적으로 시퀀스 번호를 사용해 중복 세그먼트 식별하고, 수신 측 TCP 스택이 중복 데이터를 애플리케이션에 두 번 넘기지 않도록 처리하기 때문에 괜찮음
            - 여기서 말하는 재전송은 애플리케이션 요청 단위가 아니라 TCP 세그먼트/바이트 스트림 단위의 재전송이라는 거! 기억하기~

### 타임아웃과 기약없는 지연

- 타임아웃은 얼마나 길어야 하는가?
    - 너무 길게 잡는 경우, 클라이언트가 대기하거나 오류 메시지를 오래 봐야함
    - 너무 짧게 잡는 경우, 결함은 빨리 발견하나, 일시적인 오류임에도 불구하고 죽었다고 판단할 수 있음
        - 이 경우, 같은 동작이 두 번 실행될 수도 있음(처음에 요청 받은 노드에서 한 번, 재시도 요청 받은 노드에서 또 한 번)
        - 또한, 다른 노드가 죽은 노드의 역할을 위임하는 과정에서 네트워크 부하가 발생함
            - 이미 시스템이 과부하 상태라면 상황을 더 악화시킬 수 있음
                - 1% 의 장애가 전체 장애로 번질 수 있음
- 패킷의 최대 지연 시간이 보장되지 않기 때문에, 타임아웃은 정형화 할 수 없음
    - 즉, 비동기 네트워크는 기약 없는 지연임

### 네트워크 혼잡과 큐 대기

- 네트워크에서 패킷 지연의 변동성은 큐 대기 때문인 경우가 많음
    - 스위치에 물린 여러 노드가 동시에 같은 목적지에 패킷을 보내려 하는 경우
        - 스위치는 패킷을 큐에 넣고 한 번에 하나씩 목적지 네트워크 링크로 넘겨야함
            - **Q. 스위치는 목적지 별로 큐를 가짐?**
                - 항상 목적지별 큐를 가지는 건 아님. 구현에 따라 달라짐
                    - 입력 포트 큐, 출력 포트 큐, 우선순위 큐 등등..
                - 여기서 말하는 문제는 여러 입력에서 들어온 패킷이 같은 출력 링크로 나가려고 하면, 출력 링크의 처리 속도 때문에 큐 대기가 생긴다는 것!
        - 이 과정에서 네트워크 링크가 붐비기 때문에 네트워크 혼잡으로 인한 지연 발생이 가능함
        - 또한, 큐가 꽉 차면 패킷이 유실되어 재시도 해야함
            - **Q. 이거 항상 그렇지는 않을 거 같은데? RabbitMQ 도 큐 꽉 찼을 때 전략이 다른 것처럼 얘도 기존 거 버리고 항상 수용하거나 하는 전략이 다를 거 같은데**
                - ㅇㅇ 맞아용. 여러 정책이 가능합니다
                    - Tail drop: 큐가 꽉 차면 새로 들어온 패킷을 버림
                    - Random Early Detection: 큐가 완전히 차기 전에 일부 패킷을 미리 버려 혼잡을 알림
                    - Priority drop: 낮은 우선순위 트래픽을 먼저 버림
                    - ECN: 패킷을 버리지 않고 혼잡 표시를 해서 송신자가 속도를 줄이게 함
    - 패킷이 목적지에 도달했으나, 목적지 장비의 CPU 가 바쁜 상태라면 운영체제의 큐에서 대기해야함
        - **Q. Spring 사용하는 경우에는 Netty 사용하잖아. 이 때는 큐에서 대기하는 게 운영체제 큐가 아니라 Netty 의 자체 큐인가? Netty 가 이벤트 루프 돌면서 새 요청 들어왔을 때 accept 하기 전까지 대기하는 큐가 운영체제인가? 가 궁금함. 맞는 거 같기도 하고..? 근데 보통 서버 열 때 listen 으로 대기하잖아. 이건 운영체제 큐임?**
            - 연결 수립 및 요청 처리 과정은 대충 이렇게 됨
                1. NIC 하드웨어 큐
                2. 커널 네트워크 스택 큐
                3. TCP listen backlog / accept queue
                4. 애플리케이션이 accept한 뒤 Netty Channel로 등록
                5. Netty EventLoop task queue
                6. 애플리케이션 로직 처리 큐 또는 Reactor 연산자 내부 대기
            - listen() 은 운영체제 레벨의 대기 큐와 관련이 있음. 연결이 수립되었지만, 애플리케이션이 아직 accept 하지 못한 연결이 accept 큐에서 대기하는 것
            - Netty 는 accept 이후에 이벤트 루프에서 채널 read/write 이벤트를 처리함
            - 정리하자면, accept 전 대기는 OS 영역, accept 후 처리 지연은 Netty 및 애플리케이션 영역임
    - TCP 의 흐름 제어로 지연이 발생할 수 있음
        - TCP 는 혼잡 회피, backpressure(흐름 제어) 를 사용함. 혼잡 회피는 네트워크가 붐비지 않게 송신율을 조절하는 거고, 흐름 제어는 수신자가 감당 가능한 만큼만 보내게 하기 위해 사용하는 것!
            - **Q. 여기서 각각은 어떻게 동작하나?**
                - 혼잡 회피는 네트워크 경로를 보호하는 거임
                    - 송신자가 손실, 지연, ACK 패턴 등을 보고 네트워크 상태를 추정해서 보내는 양을 조절하는 것
                - 흐름 제어는 수신자가 ACK 을 보낼 때 수용할 수 있는 수신 버퍼 크기를 송신자에게 알려줌. 이걸 receive window, rwnd 라고 함
                    - 송신자는 이 값을 보고 수신자가 감당 가능한 양보다 많이 보내지 않는 것
                    - 이건 Reactive Streams ****에서도 나타나는 특성임. 여기서의 rwnd 는 Subscription.request(n) 임
                - 실제 전송하는 양은 혼잡 회피에서 추정한 양과, 흐름 제어에서 받은 양 중에 더 작은 값을 사용해서 결정함
- TCP 가 수행하는 재시도는 애플리케이션에 보이지 않지만, 결과로 생기는 지연은 나타남
- 공개 클라우드나 멀티 테넌트 데이터센터에서는 여러 소비자가 자원을 공유하기 때문에 자원을 많이 사용하는 특정 노드 때문에 네트워크 지연 변동이 클 수 있음
    - **Q. 멀티 테넌트?**
        - 하나의 물리적/논리적 인프라를 여러 사용자나 조직이 공유하는 구조임
        - AWS 같은 클라우드에서 여러 고객의 컨테이너가 같은 물리 서버, 같은 스토리지, 같은 네트워크 장비를 공유하는 걸 말함
        - 그래서 한 사용자의 높은 자원 사용량이 다른 사용자에게 영향을 줄 수 있음(noisy neighbor)
- 따라서, 타임아웃은 실험적으로 선택해야함
    - 장기간 동안 여러 장비에 걸쳐 네트워크 왕복 시간의 분포를 측정하고, 애플리케이션 특성을 고려해 타임아웃을 결정해야함
- 더 좋은 방법은 고정된 타임아웃 대신 지속적으로 응답 시간과 변동성을 고려해 타임아웃을 자동으로 조절하게 하는 방식임

### 동기 네트워크 대 비동기 네트워크

- 앞서 살펴본 문제는, 패킷 지연 시간의 최대치가 정해져있지 않다는 것이 원인이었음
- 그럼 패킷 지연 시간의 최대치를 정하고, 패킷이 유실되지 않게 하면 안되나?
    - 바꿔 말하자면, 하드웨어 수준에서 신뢰성을 보장하면 되는 거 아닌가?
- 이걸 해결하는 게 동기식 네트워크임
    - 전화를 생각하면 쉬움
    - 전화 네트워크는 극단적인 신뢰성을 지님
        - 꾸준히 end-to-end 지연 시간이 낮아야 하고, 목소리의 음성 샘플을 전달할 대역폭이 충분해야함
    - 전화 네트워크는 동기식 네트워크인 회선을 사용함
        - 통화 참여자 사이에 있는 전체 경로를 따라 고정, 보장된 양의 대역폭이 할당되어 통화가 끝날 때까지 유지됨
        - 이렇게 하면 데이터가 여러 라우터를 거쳐도 큐 대기 문제를 겪지 않음
            - 왜냐면 다음 홉(hop) 에 데이터를 받을 공간이 이미 할당됐기 때문임
            - **Q. 할당할 수 없는 경우에는 어떻게 됨? 가령 통화가 몰려서 다음 홉의 큐가 다 찼다거나**
                - 그럼 통화 연결이 안됨!
        - 이걸 제한 있는 지연이라고 함

#### 그냥 네트워크 지연을 예측 가능하게 만들 수는 없을까?

- 전화 네트워크 회선은 TCP 연결과 다름
    - 회선은 만들어져있는 동안 다른 누구도 사용할 수 없는 고정된 양의 대역폭임
    - 반면 TCP 연결의 패킷은 가용한 네트워크 대역폭을 기회주의적으로 사용함
        - TCP 연결이 유휴 상태에 있는 동안에는 어떤 대역폭도 사용하지 않음
            - **Q. 이 말은 왜 나온 거임? 회선은 유휴 상태일 때도 대역폭 사용한다는 건가? 고정된 양을 미리 확보하고 유지하니까 데이터 전송이 실제로 없어도 일단 회선이 열린 동안에는 대역폭 사용한다는 뜻에서 나온 말임?**
                - ㅇㅇ
- 회선 교환 네트워크에서는 회선 구성 시 왕복 시간의 최대치를 보장할 수 있으나, 이더넷과 IP 는 큐 대기의 영향을 받는 패킷 교환 프로토콜이라 네트워크에 기약 없는 지연이 잇음
- 그럼 왜 데이터센터 네트워크와 인터넷은 패킷 교환을 사용할까?
    - 순간적으로 몰리는 트래픽을 처리하기 위해서!
    - 웹페이지 요청, 이메일 전송 등의 인터넷 상에서 이뤄지는동작은 특별한 대역폭 요구사항이 없고, 그 대역폭을 추정할 수도 없음
        - 추청치가 없기 때문에 회선이 부적합함! 그래서 가용한 네트워크 용량에 맞춰 전송률을 동적으로 조절 가능한 TCP 와 같은 패킷 교환 프로토콜이 더 적합함!
- 회선 교환과 패킷 교환을 모두 지원하는 하이브리드 네트워크를 만드려는 시도도 있었음
    - ATM(Asynchronous Transfer Mode, 비동기 전송 모드)
        - 현금 인출기와 무관
        - 링크 계층에서 end-to-end 제어를 구현해 네트워크에서 큐 대기를 할 필요성을 줄임
        - 그러나 링크 혼잡 때문에 여전히 지연의 영향을 받을 수 잇음
            - **Q. 링크 혼잡은 왜 발생하는 거임?**
                - 특정 링크로 들어오려는 트래픽의 총량이 그 링크의 처리 용량보다 큰 경우에 발생함
                - 트래픽 예측이 틀리거나, 여러 경로가 특정 링크로 몰리거나 하는 경우에도 발생함
                - 미리 확보했으니 혼잡이 없다 가 아니라, 진입 제어와 예약을 엄격하게 처리하면 혼잡 가능성을 줄일 수 있다는 뜻에 가까움
        - Qos(Quality of service, 서비스 품질) 과 진입 제어를 사용하면 패킷 네트워크에서 회선 교환을 흉내 내거나 통계적으로 제한 있는 지연을 제공하는 것도 가능함
            - **Q. Qos 랑 진입 제어 사용한다는 건 요청 들어오는 상태에 따라서 처리하는 품질 낮추고 어쨋든 대역폭 확보해준다는 건가? 그래서 회선 교환 흉내내서 가능하다는거?**
                - 대충 비슷함
                - 품질 낮춘다기 보다는 우선순위와 자원 할당 정책이 달라진다고 보는 게 맞음
                    - e.g. 음성이나 영상은 트래픽 지연이 중요하니 우선 처리하고, 대용량 파일 다운로드는 조금 늦어도 되니 우선순위 낮게 주고 하는 방식
- 자원을 정적으로 분할하는 건 공평하나, 비효율적일 수 잇음(모든 걸 다 받아주다보니 각각이 느려짐)
- 근데 동적으로 분할하는 건 공평하진 않으나, 효율적일 수 잇음(상황에 따라 특정 요청은 처리 가능하나, 다른 요청은 처리 안될 수 도 잇음)

## 신뢰성 없는 시계

- 애플리케이션에서 시계와 시간은 중요함
    - 지속 시간(요청 보낸 시점과 응답 받은 시점 사이의 시간 구간) 을 측정하거나, 시점(특정 시간에 발생한 이벤트) 를 기술해야하기 때문!
- 분산 시스템에서는 통신이 비결정적이므로 시간을 다루기 까다로움
    - 메시지가 언제 도착할지 모르기 때문!
- 그리고 개별 장비가 시계를 가지기 때문에 약간의 오차가 존재할 수 있음
    - 이거 해결하기 위해서 네트워크 시간 프로토콜(NTP, Network Time Protocol) 사용해서 서버 그룹에서 보고한 시간에 따라 컴퓨터 시계를 조정할 수 있게 함
    - **Q. 근데 이것도 결국 조회 요청 - 응답 까지 시간이 비결정적이니까 오차는 존재할 수 있는 거 아님?**
        - 맞음. 그래서 분산 환경에서는 NTP 사용하면 각 서버의 시간이 오차 범위 안에서 대략 맞춘다고 보는 게 더 타당함

### 단조 시계 대 일 기준 시계

- 현대 컴퓨터는 최소 두 종류의 시계를 가지고 있음
    - 일 기준 시계
    - 단조 시계
- 둘 다 시간을 측정하지만 다른 목적으로 사용됨

#### 일 기준 시계

- 어떤 달력에 따른 현재 날짜와 시간 같이 특정  시점을 측정하는 데 적합함
    - e.g. 리눅스의 clock_gettime(CLOCK_REALTIME), 자바의 System.currentTimeMillis()
        - 이것들은 에포크(UTC 의 1970년 1월 1일 자정) 기준으로 흐른 시간(초 or 밀리초)을 반환함
        - 윤초는 세지 않음
            - **Q. 윤초?**
                - 지구 자전속도와 원자 시계 기반 시간의 차이 보정을 위해 UTC 에 1초를 추가하거나 빼는 것
- 일 기준 시계는 보통 NTP 로 동기화됨
    - 즉, 이상적으로 한 장비의 타임스탬프는 다른 장비의 타임스탬프와 동일한 의미를 지닌다는 것
    - 근데 로컬 시계가 NTP 보다 앞서 있는 경우, 동기화 시점에 강제로 리셋되어 과거로 돌아가는 것처럼 보일 수 있음
        - **Q. 이건 상황에 따라 트랜잭션에 치명적일 거 같은데 맞음?**
            - 맞아용 ㅜㅜ
            - 특히 일 기준 시계를 트랜잭션 순서, 락 / 세션 만료, 이벤트 정렬에 직접 사용하면 더 치명적임
- 일 기준 시계는 거친 해상도를 가짐
    - 단위가 대강대강이라는 뜻인 듯

#### 단조 시계

- 타임아웃이나 서비스 응답 시간 같은 지속 시간을 재는 데 적합함
    - e.g. 리눅스의 clock_gettime(CLOCK_MONOLOTIC), 자바의 System.nanoTime()
- 한 시점에서 단조 시계의 값을 측정하고, 이후 특정 시점에 단조 시계 값을 다시 측정해서 그 차이로 두 번의 확인 사이에 흐른 시간을 측정할 수 있음
- 단조 시계의 절대적인 값은 의미가 없음
    - 그래서 두 대의 다른 컴퓨터에서 나온 단조 시계 값을 비교하는 것은 의미가 없음
- 여러 개의 CPU 소켓이 잇는 서버는 CPU 마다 독립적인 타이머가 있을 수도 잇음
    - 이 타이머는 다른 CPU 와 반드시 동기화되는 것은 아님
    - 운영체제는 차이를 보정해서 애플리케이션 스레드가 여러 CPU 에 걸쳐 스케줄링되더라도 시계가 단조적으로 보이게 하려고 함
        - **Q. 이거 CPU 마다 독립적인 타이머 가지고 있으니까, 특정 서비스가 두 번 시간 측정을 해서 흐른 시간 구한다고 할 때, 처음에는 A CPU 에서 측정하고, 이후에 B CPU 에서 측정해서 둘의 차이를 측정했을 때의 값이 A CPU 에서 두 번 측정한 거랑 같게 한다는 거 맞음?**
            - ㅇㅇ 맞음
- NTP 는 컴퓨터의 로컬 시계가 NTP 와 차이가 나는 것을 발견하면 단조 시계가 진행하는 진도수를 조정할 수도 있음. 이를 시계를 돌린다고 함
    - 진도수를 0.05%까지 올리거나 내릴 수 있으나, 단조 시계 자체를 앞이나 뒤로 뛰게 할 수는 없음
    - **Q. 이거 필요한 이유가 뭐임?**
        - 로컬 시계가 빠르거나 느린 경우, 시간을 바로 변경하면 위험할 수 있으니 시간이 흐르는 속도를 늦추거나 빠르게 하는 거임
- 단조 시계의 해상도는 보통 상당히 좋음
    - 그래서 마이크로초나 그 이하 단위로 측정 가능함
- 분산 시스템에서 경과 시간을 재는 데 단조 시계를 쓰는 건 괜찮음
    - 다른 노드의 시계 사이에 동기화가 돼야 한다는 가정이 없고, 측정이 약간 부정확해도 민감하지 않기 때문

### 시계 동기화와 정확도

- 단조 시계는 동기화가 필요 없으나, 일 기준 시계는 NTP 서버 혹은 외부 시간 출처에 맞춰야 할 필요가 있음
- 그러나 시계는 여렁 이유로 정확한 값을 나타내지 못함
    - 컴퓨터가 사용하는 수정 시계에는 드리프트 현상(더 빠르거나 느리게 실행되는 현상) 이 생김
    - 컴퓨터 시계가 NTP 와 많은 차이가 나면 동기화가 거부되거나, 로컬 시계가 강제로 리셋됨
        - **Q. 근데 거부는 왜 되는 거임? 항상 맞춰야하는 거 아닌가?**
            - 차이가 너무 크면 NTP 서버가 이상한 건지, 내가 이상한 건지, 설정이 잘못된 건지 확신할 수 없기 때문
            - 그래서 큰 오차는 자동 보정하지 않고 경고하거나, 수동 개입을 요구하거나, 초기 부팅 시에만 큰 변화를 허용함
    - 노드와 NTP 서버가 방화벽으로 막힐 수 있음
    - NTP 동기화 과정에 네트워크 지연이 발생할 수 있음
    - NTP 서버의 설정이 잘못 되어있을 수 있음
    - 윤초가 발생하면 1분의 길이가 59초나 61초가 되는 문제가 발생함
        - 윤초 조정을 하루에 걸쳐 천천히 수행하게 하는 방식으로 문제를 해결함
- 시계 정확도가 중요하다면 이를 매우 높이는 것도 가능함
    - GPS 수신기, 정밀 시간 프로토콜(PTP, Precision Time Protocol), 세심한 배포 및 모니터링으로 달성 가능함
        - **Q. 어떻게 배포해야하고, 무엇을 모니터링 해야하는가?**
            - 배포 측면
                - 데이터센터에 GPS 수신기 또는 원자시계 기반 시간 소스 배치
                - 시간 서버 중복 배치 및 애플리케이션 서버가 가까운 시간 소스와 동기화하도록 구성
                - 등등
            - 모니터링 측면
                - 마지막 동기화 시점
                - 각 노드의 clock offset
                - drift rate
                - 등등

### 동기화된 시계에 의존하기

- 앞에서 네트워크 지연에 대해 얘기한 문제는 시계에서도 고려해야하는 문제임
    - 대부분에 시간에 잘 동작하지만, 잘못될 가능성을 염두에 두고 이에 대비해야함
- 근데 시계가 잘못된다는 걸 눈치채기 힘듦!
    - 장비의 CPU 결함, 네트워크 설정 등의 문제로 장비가 아에 동작하지 않으면 문제 발견 가능성이 높음
    - 반면 장비 수정 시계에 결함이 있거나 NTP 클라이언트가 잘못 설정됐다면 시계는 드리프트가 생겨서 실제 시간으로부터 멀어지나, 잘 동작하는 것처럼 보여서 문제 발견 가능성이 낮음
- 따라서 동기화된 시계가 필요한 소프트웨어를 사용한다면 필수적으로 모든 장비 사이의 시계 차이를 모니터링 해야함
    - 다른 노드와 시계가 너무 차이나면 해당 노드를 죽은 것으로 판단하고 클러스터에서 제거해야함
    - **Q. 근데 이거 원격으로 모니터링하는 경우에는 또 정확한 측정이 안될 거 같은데? 보정하는 방법 없나?**
        - 맞아용 원격 측정도 네트워크 지연 영향이 있어요
        - 그래서 왕복 시간 측정해서 오차 범위 추정하거나, 여러 시간 소스와 비교하거나, 여러 번 측정해서 분포, 추세를 보거나 하는 방식 등을 사용해용

#### 이벤트 순서화용 타임스탬프

- 분산 환경에서, 각 노드가 가진 타임스탬프가 다르기 때문에 어떤 이벤트가 먼저 발생했는지 정확히 알 수 없는 문제가 발생할 수 있다.
- 가령, 두 클라이언트가 분산 데이터베이스에 쓰는 경우, 쓰기가 복제될 때 쓰기가 발생한 노드의 일 기준 시계에 따른 타임스탬프가 붙는다
    - 이 경우, 노드 간의 일 기준 시계 차이에 의해, 인과 상 먼저 발생했음에도 불구하고 나중에 발생한 것처럼 보이는 문제가 발생할 수 있다
- 따라서, 가장 최근의 값을 사용하는 LWW 방식을 채택한 경우에도, 최근이라는 말의 정의가 로컬 일 기준 시계에 의존하기 때문에 틀릴 수 있다는 것을 아는 것이 중요하다
- 잘못된 순서화가 발생하지 않을 정도로 NTP 동기화를 정확하게 하는 것은 불가능하다
    - 왜냐면 시계 드리프트 외에도 네트워크 왕복 시간에 따라 제한되기 때문
    - 올바른 순서화를 위해서는 시계 출처가 네트워크 지연으로 발생하는 오차를 해결할 만큼 정확해야함
- 논리적 시계
    - 수정 대신 증가하는 카운터를 기반으로 하며, 이벤트 순서화의 안전한 대안임
        - 물리 시계에 의존하지 않기 때문에, 메시지 전달로 생기는 인과가 깨지지 않기 때문
    - 이벤트의 상대적인 순서(한 이벤트가 다른 이벤트의 앞이나 뒤에 일어났는가?) 만 측정함
    - 일 기준 시계와 단조 시계는 실제 경과 시간을 측정하기 때문에 물리적 시계라고 함
    - 이런 식으로 동작함
        
        ```lua
        A.clock = 5
        A가 메시지 보낼 때 clock + 1 → 6
        메시지에 6을 붙여서 B에게 전송
        
        B.clock = 2
        B가 메시지를 받음
        B.clock = max(B.clock, 받은 clock) + 1
                = max(2, 6) + 1
                = 7
        ```
        

#### 시계 읽기는 신뢰 구간이 있다

- 시계에서의 세밀한 측정이 정밀성이나 정확성을 의미하진 않음
    - NTP 서버와 동기화를 해도 드리프트가 발생할 수 있고, 네트워크 지연이 발생할 수도 있기 때문
- 따라서 시계 읽기를 어떤 시점으로 생각하는 것보다, 어떤 신뢰 구간에 속하는 시간 범위로 읽는 게 나음
    - 신뢰 구간은 시간 출처를 기반으로 계산함
    - GPS 수신기나 컴퓨터에 직접 부착된 원자 시계는 제조사에서 예상 오류 범위를 제공함
    - 여기에 서버와 마지막으로 동기화한 시간 이후로 예상되는 드리프트를 더하고, 서버와 통신할 때 걸린 네트워크 왕복 시간을 더하면 신뢰 구간을 구할 수 있음
        - **Q. 왜 왕복 시간이지? 클라 → 서버 → 클라 구조에서, 서버는 요청을 받은 시점의 값을 주니까 서버 → 클라 시간만 더해야하는 거 아님?**
            - 그럼 좋겟지만, 서버 → 클라의 편도 지연 시간을 정확히 알 수 없기 때문에 왕복 시간을 사용하는 거임
            - 서버 → 클라의 편도 시간을 구하기 위해서는, 응답을 받은 시간 - 서버가 보낸 시간 을 구해야하는데, 두 시계의 편차가 얼마인지 정확하지 않기 때문에 이를 확정할 수 없는 것
- 시스템의 대부분은 이 불확실성(오차 발생) 을 말해주지 않음
- 그러나 구글 트루타임 API 는 로컬 시계의 신뢰 구간을 명시적으로 보고함
    - 이 때 신뢰 구간의 너비는 로컬 수정 시계가 더 정확한 시계 출처(e.g. NTP) 와 마지막으로 동기화된 이후 얼마나 지났는지에 의존함
        - Q. 결국 드리프트 발생이 얼마나 됐는지가 중요하다는 거임?
            - ㅇㅇ 맞음

#### 전역 스냅숏용 동기화된 시계

- 스냅숏 격리 구현에는 단조 증가하는 트랜잭션 ID(TXID) 가 필요함
- 일 기준 시계의 타임스탬프로 TXID 값을 정하면 어떨까?
    - 어차피 일 기준 시계도 이상적으로는 단조 증가하니까!
- 스패너는 구글 트루타입 API 가 보고한 시계 신뢰 구간을 사용해 이를 구현려고 함
    - 두 개의 신뢰 구간이 있는 경우, 두 구간이 겹치지 않는다면 누가 나중에 실행됐는지 확실히 알 수 있음
    - 구간이 겹칠 때만 순서를 확신할 수 없음
    - 트랜잭션 타임스탬프가 인과성을 반영하는 것을 보장하기 위해, 읽기 쓰기 트랜잭션을 커밋하기 전에 의도적으로 신뢰 구간의 길이만큼 기다림
        
        > 
        > 
        > 
        > 트랜잭션 타임스탬프가 실제 시간 순서와 어긋나지 않도록 하기 위해,
        > 읽기-쓰기 트랜잭션은 커밋 타임스탬프를 정한 뒤,
        > 그 타임스탬프가 실제 시간상 확실히 과거가 될 때까지 기다린 다음
        > 커밋 성공을 외부에 알린다.
        > 
        > 이렇게 하면 클라이언트가 T1 요청 → T1 커밋 응답 → T2 요청 순서로 동작할 때,
        > T2는 실제 시간상 T1의 커밋 타임스탬프 이후에 시작한 트랜잭션임이 보장된다.
        > 따라서 T1 이후에 시작한 T2가 T1보다 더 이른 타임스탬프를 받는 문제를 막을 수 있다.
        > 
        - 이렇게 하면 해당 데이터를 읽을지도 모르는 트랜잭션은 충분히 나중에 실행되는 게 보장되므로 신뢰 구간이 겹치지 않음
            - **Q. 왜 신뢰 구간 길이만큼 기다리는 거임?**
                - 간단 예시로 보면:
                    1. T1이 커밋 timestamp `s = 100`을 받음
                    2. 현재 TrueTime이 `[95, 105]`라면 실제 시간이 아직 100 이전일 수도 있음
