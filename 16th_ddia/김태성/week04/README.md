# 파티셔닝

## 파티셔닝과 복제

- 각 파티션(샤드) 의 복사본을 여러 노드에 저장할 수 있음
- 이렇게 하면 내결함성 보장이 가능함
- 즉, 하나의 노드에 서로 다른 파티션이 저장될 수 있음
- **Q. 그럼 이 경우에 라우팅은 누가 관리하나?**
    - 아래의 ‘요청 라우팅’ 챕터에서 설명

## 키-값 데이터 파티셔닝

- 샤딩의 목적은 부하 분산인데, 질의가 특정 샤드로 쏠린다면 부하 분산이 고르게 되지 않음
    - 이 문제에서 ‘특정 샤드’ 에 해당하는 샤드를 핫스팟이라고 함
- 해결1) 무작위로 데이터 적재
    - 데이터가 고르게 저장됨
    - 그러나 데이터 저장에 규칙이 없어, 데이터를 찾을 때 모든 노드에 질의를 던져야 함
- 해결2) 키-값 데이터 모델 사용
    - 항상 기본키를 통해 레코드에 접근함

### 키 범위 기준 파티셔닝

- 각 파티션에 연속된 범위의 키를 할당해둠
- 이렇게 하면 특정 키가 속한 파티션 파악이 가능하기 때문에 빠른 질의가 가능함
- 각 파티션 내부에서도 키를 정렬된 순서로 저장할 수 있음
    - SS테이블이나 LSM 트리 사용하면 됨
    - 이러면 범위 탐색이나, 여러 레코드를 한 번에 읽기 쉬움
- 질문) 타임스탬프를 키로 사용하는 건 적절한가?
    - 범위 스캔에 유용함 (특정 월에 해당하는 모든 데이터 탐색 등)
    - 문제) 특정 시간대에 특정 노드만 부하를 받게 됨
    - 해결) 타임스탬프 외에 다른 걸 키로 사용해야함 (e.g. 센서 이름 - 타임스탬프 를 키로 사용)

### 키의 해시값 기준 파티셔닝

- 쏠림과 핫스팟 문제를 해결하기 위해 해시함수 사용할 수도 있음
    - **Q. MD5?**
        - 문자열이나 키를 넣으면 고정 길이(128비트) 해시값을 만들어주는 해시 함수
        - MongoDB, Casandra 에서 사용
    - 참고) 자바의 Object.hashCode() 는 안정성이 부족해 보통 사용하지 않음
        - **왜?**
            - 객체의 논리 값(value) 보다 객체 인스턴스의 identity(동일 객체 여부) 에 기반할 수 있기 때문에, 값이 달라질 수 있음
                
                ```lua
                UserKey a = new UserKey(1)
                UserKey b = new UserKey(1)
                
                // 두 객체의 논리 값은 같지만, Object.hashCode() 의 값은 다를 수 있음
                ```
                
            - String, Integer 등은 값 기반으로 hashCode() 가 override 되어, 동일한 값이면 동일한 hashCode 가 나오도록 정의되어 있음
- 각 샤드에 해시값 범위를 할당하는 방식으로 사용함
- 이 방식으로 키를 파티션에 균일하게 분산시킬 수 있음
- 문제) 범위 질의가 효율적이지 못함
    - 인접한 키들이 모든 파티션에 흩어져서 정렬 순서가 유지되지 않기 때문
    - 범위 질의가 모든 샤드에 전송되어야 함
- 이 문제를 카산드라는 복합 기본키로 해결함
    - 키의 첫 부분에만 해싱해서 샤드 결정에 사용
    - 남은 칼럼은 카산드라의 SS테이블에서 데이터 정렬하는 색인으로 사용
    - 이렇게 하면, 키의 첫 부분 사용해서 범위 질의를 할 수는 없지만, 첫 부분이 같은 키들은 범위 질의 가능함
        - 왜냐면 샤드 내부는 SS테이블로 정렬되었으니까

### 쏠린 작업부하와 핫스팟 완화

- 해싱 방법으로 핫스팟 문제를 일부 해결할 수는 있지만, 완벽히 해결은 못함
- 동일한 키를 읽고 쓰는 작업에서는 모든 요청이 동일한 샤드로 쏠리게 됨
- 이는 애플리케이션에서 해결해야함
    - e.g. 요청이 많이 쏠리는 키를 발견하면, 각 키의 시작이나 끝에 임의의 숫자를 붙임(쪼개기)
        - 이러면 키가 달라지므로, 쓰기 부하를 줄일 수 있음
        - 문제) 근데 이 경우에는 읽을 때 문제가 됨
            - 그래서 진짜 필요할 때만 사용하고, 어떤 키를 쪼갰는지 별도로 기록해두어야 함

## 파티셔닝과 보조 색인

- 보조 색인은 샤드에 깔끔하게 대응되지 않음
    - 왜냐면 샤딩의 기준이 되는 키로 샤딩을 하니까, 같은 컬럼 값을 가진 데이터가 흩어질 수 있기 때문
    - e.g. userId 로 샤딩했을 때
        
        ```lua
        userId: 1, name: Kim → 1번 샤드
        userId: 111, name: Kim → 12번 샤드
        userId: 11132, name: Kim → 27번 샤드
        ```
        
        - 이러면 Kim 을 가진 데이터가 흩어지므로 보조 색인이 깔끔하게 대응되지 않음

### 문서 기준 보조 색인 파티셔닝

- 특정 데이터를 가진 레코드가 데이터베이스에 추가되면, 데이터베이스의 샤드가 별도로 기록해두는 방식
- 각 파티션에 별도로 기록을 해두기 때문에, 각 파티션은 완전히 독립적으로 동작함
- 문제) 그래도 같은 값을 가진 데이터가 흩어져 있기 때문에, 모아서 볼 방법이 필요함
- 해결) 스캐터/개더
    - 모든 파티션으로 질의를 보내서 얻은 결과를 모두 모으는 것
    - 비용이 크고, 꼬리 지연 시간 증폭이 발생하기 쉬움
        - **Q. 꼬리 지연 시간이 왜 발생하기 쉽나?**
            - 꼬리 지연 시간은 전체 요청 시간이 가장 느린 샤드에 의해 좌우된다는 것
            - 질의하는 샤드 수가 많아질 수록 느린 응답이 발생할 확률이 크기 때문

### 용어 기준 보조 색인 파티셔닝

- 각 샤드가 별도의 보조 색인(지역 색인)을 갖는 게 아니라, 모든 샤드의 데이터를 담당하는 전역 색인을 만들 수도 있음
- 이 또한 샤딩을 해두어야 부하 분산이 가능함
- 이런 식으로 함 : 보조 색인이 a ~ r 로 시작하면 0번 샤드, s ~ z 로 시작하면 1번 샤드
- 이 경우, 용어에 따라 색인의 샤드가 결정되므로 용어 기준으로 샤딩되엇다고 함
- 용어의 해시값으로 샤딩할 수도 있음. 이러면 부하가 좀 더 고르게 분산됨
- 문서 파티셔닝 색인 방식에 비해 효율적인 읽기가 가능함
    - 어느 샤드에 데이터가 있는지 알고 있기 때문
- 그러나 느리고 복잡하며, 지연 가능성이 있음(비동기로 색인 반영하기 때문)

## 파티션 재균형화

- 재균형화 시에 신경써야 할 것
    - 재균형화 후, 부하가 균등하게 분배되어야 함
    - 재균형화 도중에도 읽기/쓰기 요청 처리가 되어야 함
    - 재균형화를 빨리 처리하고 네트워크와 디스크 I/O 부하를 최소화할 수 있도록, 데이터가 필요 이상으로 옮겨지면 안됨

### 해시값에 모드 N 연산 수행 (비추)

- N 이 변경될 때마다 데이터가 저장될 샤드가 자주 변경됨
- 데이터가 필요 이상으로 옮겨질 가능성이 농후하므로 비추!

### 파티션 개수 고정

- 파티션을 노드 개수보다 많이 만들고, 각 노드에 여러 파티션을 할당하는 방식
- 클러스터에 노드가 추가/제거되는 경우, 파티션을 통째로 다른 노드로 이동시키면 됨
    - 근데 이 경우에는 즉시 반영되지 않고, 대량의 데이터 전송이 필요하므로 시간이 좀 걸림
- 노드 간 성능 차이가 있는 경우, 성능 좋은 노드에 여러 파티션을 할당시킬 수도 있음
- 이 방식은 리악, 엘라스틱서치, 카우치베이스, 볼드모트에서 사용함
- 문제) 확장은 가능하지만, 처음 정한 파티션 개수에 의해 확장 유연성이 제한됨
    - **Q. 왜?**
        - 왜냐면 처음에 정한 파티션 개수에 따라 노드 확장이 한계가 있기 때문
            - 노드가 처음에 정한 파티션 개수의 약수가 아니거나, 더 많은 경우에 균등한 파티션 배분이 안됨
        - 그리고 파티션 개수가 정해져 있어서 데이터셋이 커질 수록 각 파티션 크기가 증가하는데, 이러면 파티션을 다른 노드로 옮길 때 더 많은 비용이 발생함

### 동적 파티셔닝

- 파티션 개수 고정하면 확장이 어렵다는 문제를 해결함
- 파티션 크기가 임계치를 넘으면, 파티션을 두 개로 쪼개 각각에 원래 파티션의 절반 정도의 데이터가 포함되게 하는 방식
    - 반대로 데이터가 줄어들면 합침
    - B트리 방식이랑 유사함
- 문제) 빈 데이터베이스는 파티션 경계에 대한 정보가 없으므로, 처음에는 파티션이 하나임
    - 그래서 파티션이 분할되기 전까지는 하나의 파티션에서 요청 처리함
- 해결) 사전 분할
    - HBase 나 MongoDB 는 빈 데이터베이스에 초기 파티션 집합을 설정할 수 있게 함

### 노드 비례 파티셔닝

- 노드 당 할당되는 파티션 개수를 고정할 수도 있음
- 노드 대수가 변경되지 않으면 개별 파티션의 크기가 데이터셋 크기에 비례해 증가함
- 노드 대수가 늘어나면 파티션 크기가 다시 작아짐
    - 노드가 추가되는 경우, 기존 파티션을 무작위로 선택해 반으로 분할하고, 절반을 새 노드에 할당함
    - 문제) 무작위 선택이라 분할이 분균등할 수 있음
    - 해결) 카산드라 3.0 에는 불균등한 분할을 회피할 수 있는 대안적인 재균형화 알고리즘이 추가됨
        - **Q. 머임?**
            - Token Allocation Algorithm 이라는 방식임
            - 새 노드 추가 시, 기존 토큰 분포와 데이터 분포를 고려해 더 균등한 토큰 배치를 계산함
- 개별 파티션 크기를 안정적으로 유지할 수 있음

## 요청 라우팅

- 서비스를 찾는 방법은 여러가지가 있음
    1. 클라이언트가 무작위 노드에 접속하게 한 후, 찾고자 하는 데이터가 해당 노드에 있으면 데이터 반환, 없다면 데이터를 가지고 있는 노드를 반환
        1. 이 경우에, 각 노드는 서로 어떤 데이터를 가지고 있는지 알고 있어야 함
        2. 대표적으로 가십 프로토콜을 사용함(카산드라, 리악)
            1. 이는 클러스터 상태 변화를 노드 사이에 퍼뜨리는 프로토콜로, 요청을 받은 노드가 요청을 처리할 파티션을 가지고 있는 노드로 요청을 전달해줌
    2. 클라이언트의 모든 요청이 라우팅 계층을 거치게 함
        1. 라우팅 계층에서 어떤 노드가 어떤 데이터를 가지고 있는지 알아야 함
    3. 클라이언트가 파티셔닝 방법과 파티션이 어느 노드에 할당되었는지 알고 요청하게 함
        1. 중개자 없이 바로 노드 접속 가능
- 결국 핵심은 ‘데이터 변경을 클라이언트가 어떻게 알게 할 것인가?’ 임
- 많은 시스템이 전역 변경 저장소(e.g. 주키퍼) 를 두고 사용함
