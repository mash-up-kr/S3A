### 프로세스 중단

- 분산 시스템에서 시계를 위험하게 사용하는 다른 예로는, 리더 선출 문제가 있음
- 파티션마다 리더가 하나씩 있는 데이터베이스에서, 리더를 선출하는 방식으로 임차권을 얻는 방식을 사용함
    - 이는 타임아웃이 있는 잠금과 비슷함
    - 특정 시점에 하나의 노드만 임차권을 얻을 수 있기 때문에, 어떤 노드가 임차권을 획득하면 해당 노드가 리더인 기간을 알 수 있음
    - 노드가 계속해서 리더로 남기 위해서는 임차권이 만료되기 전에 주기적으로 갱신해야함
- 임차권 만료와 관련해, 두 가지 방법이 있음
    1. 동기화된 시계에 의존하기
        1. 임차권 만료 시간을 다른 장비에 설정하고 로컬 시계와 비교하는 경우, 시계 동기화가 깨져 문제가 발생할 가능성이 존재함
            1. **Q. 근데 리더가 변경되면 로드밸런서에 반영이 되어야 할 텐데, 이 반영되는 시간을 고려하나?**
                - 하긴 하는데, 임차권 관련해서는 하지 않아용. 왜냐면 언제 로드밸런서에 반영이 되는지 모르잖아. 그리고 이렇게 하면 부정확해서 안해용~
                - 대신 요청이 잘못된 노드로 갈 수 있음을 전제로 하고, 잘못 간 요청을 올바르게 처리하거나 거절하게 만듭니당
                    - 요청을 받은 노드가 리더일 수도 있고, 리더가 아닐 수도 잇음. 이건 로드밸런서가 모름
                    - 근데 요청을 받은 노드가 리더가 아니라면 다시 로드밸런서에게 알리거나, 아니면 내부적으로 포워딩하는 방식으로 요청을 넘김
    2. 로컬 단조 시계만 사용하기
        1. 일반적으로는 정상 동작하나, 예기지 못한 중단이 발생할 수 있음
            1. GC 로 인한 중단
                1. GC 가 동작하면 실행 중인 모든 스레드가 멈출 수 있음(때로는 몇 분 동안이나 지속됨)
                2. 핫스팟 JVM 의 CMS 처럼 동시적인 GC  도 대부분 작업을 애플리케이션과 병행해서 수행하지만, 일부 단계에서는 여전히 stop-the-world 가 필요함
                    1. **Q. CMS 가 머임?**
                        1. CMS는 이름 그대로 대부분의 마킹/스윕 작업을 애플리케이션 스레드와 동시에 수행하려는 GC 임
                        2. 이렇게 동작함
                            
                            ```lua
                            Initial Mark  ->  Concurrent Mark  ->  Remark  ->  Concurrent Sweep
                            (STW)             (동시 실행)           (STW)      (동시 실행)
                            ```
                            
                            1. Initial Mark
                                1. 시작점에 바로 연결된 객체만 표시하는 과정임
                                2. 이 과정에서 stop-the-world 가 발생하는데, 왜냐면 동시성 문제를 배제해야 하기 때문! 체크 중에 변경되면 정확하지 않으니까~
                            2. Concurrent Mark
                                1. Initial Mark 에서 찾은 객체들을 시작점으로 해서, 참조를 따라가며 살아 있는 객체를 계속 찾는 과정임
                                2. 이건 애플리케이션이랑 동시실행 가능함
                            3. Remark
                                1. Concurrent Mark 단계가 병렬 수행되기 때문에, 참조 관계가 변했을 수 잇음
                                2. 이 변한 참조 관계를 감지하기 위해 Remark 를 수행함
                                3. 여기서도 마찬가지로 동시성 문제 배제하고 정확하게 체크하기 위해 stop-the-world 가 발생함
                            4. Concurrent Sweep
                                1. 죽엇다고 판단한 객체의 메모리를 회수함
                        3. 근데 지금은 deprecated 됨
                3. 할당 패턴 변경이나, GC 설정 튜닝으로 중단 시간을 줄일 수 있지만, 견고한 보장을 하기 위해서는 최악을 가정해야함
                    1. **Q. 어떤 패턴이 있고, 어떻게 튜닝해야함?**
                        1. 할당 관련해서는 불필요한 임시 객체를 줄여야 함
                            1. 요청마다 큰 DTO, 문자열, 컬렉션, 람다 캡처 객체, ByteArray를 과도하게 만들면 young generation GC가 자주 발생함
                        2. 그리고 객체가 오래 살게 하면 안됨. 짧게 사용하는 객체는 young GC 에서 비교적 싸게 처리되는데, 몇 번의 GC 를 거치고 살아남은 객체는 old generation 으로 승격되어서 GC 비용이 커짐
                            1. **Q. young generation 는 머고 old generation 은 머임? 왜 GC 비용이 커지는 거임?**
                                1. young generation
                                    1. 새로 생성된 객체가 들어가는 영역
                                    2. young GC 가 이 영역에서 죽은 객체를 청소함
                                        1. 이 과정에서 살릴 것만 다른 영역으로 복사하고, 기존 공간을 통째로 비우기 때문에 GC가 빠름
                                    3. young GC 에서 여러 번 살아남으면 old generation 으로 이동함
                                2. old generation
                                    1. 여러 번 살아남은 객체가 들어가는 영역
                                    2. 대표적으로 이런 게 잇음
                                        - 애플리케이션 전역 캐시
                                        - Spring Bean
                                        - 오래 유지되는 커넥션 풀
                                    3. old generation 에는 오래 살아남은 객체가 많아, 이게 진짜 살아있는지를 GC 가 판단해야함
                                        1. 참조를 따라가며 확인해야하는데,
                                        2. 객체 그래프가 복잡함(다른 객체를 많이 참조해서)
                                        3. 그리고 메모리 단편화가 발생할 가능성이 큼(일부는 죽고 일부는 살아서) 그래서 compation 비용이 추가로 들음
                                - 정리하자면, Young GC는 보통 “대부분 죽은 객체 중 일부 살아남은 객체만 복사” 하는 작업이라 빠른 편이고, Old Generation GC는 “오래 살아 있는 많은 객체를 추적하고 정리” 해야 해서 더 비쌀 수 있음
                        3. 튜닝 측면에서는 heap 크기 조정, GC 종류 선택, young generation 크기 조정 등을 통해 중단 시간을 줄임
                            1. **Q. 무슨 관련이길래 영향을 주는가??**
                                1. 힙이 너무 작으면 GC 가 빈번하게 일어남, 근데 힙이 크면 GC 는 가끔 일어나는데 정리하는데 오랜 시간을 사용하게 됨
                                2. young generation 이 너무 작으면 새로운 객체가 조금만 생겨도 young GC 가 자주 발생함
                    2. **Q. 일반적으로 서버에서 GC 로 정리되어야 하는 객체는 머가 잇음?**
                        - HTTP 요청/응답 객체
                        - DTO, Command, Query 객체
                        - JSON 파싱 중 생기는 객체
                        - 문자열, StringBuilder, byte 배열
                        - 컬렉션: List, Map, Set
                        - DB 조회 결과 객체
                        - 로그 메시지 생성 객체
                        - WebFlux/Coroutine/CompletableFuture 등의 비동기 처리 중간 객체
            2. cpu 퀀텀 타임 재분배
            3. 동기식으로 디스크에 접근하는 경우, I/O 병목으로 인한 중단
            4. 페이징이 설정된 경우, 페이지 폴트 발생으로 인한 중단
        2. 이 경우, “리더였던” 노드는 자신이 임차권을 소유한다고 생각하지만, 실제로는 임차권이 넘어갔을 수 있음
- 따라서 분산 시스템의 노드는 어느 시점에 실행이 상당한 시간 동안 멈출 수 있다고 가정해야함

#### 응답 시간 보장

- 위에서 살펴본 중단의 원인은 노력하면 제거가 가능함
- 시스템이 데드라인을 엄격하게 지키는 걸 엄격한 실시간 시스템이라고 함
- 실시간 보장을 제공하려면 소프트웨어 스택의 모든 수준에서 지원이 필요함
    - 프로세스가 명시된 간격의 CPU 시간을 할당받을 수 있게 보장되도록 스케줄링해주는 실시간 운영체제(real-time operating system, RTOS) 가 필요함
    - 라이브러리는 최악의 실행 시간을 문서화해야함
    - 동적 메모리 할당은 제한되거나 금지될 수 있음
        - **Q. 왜여?**
            - 할당 시간이 일정하지 않을 수 있고, 실패 가능성이 잇으며, 나중에 회수 비용이 발생하기 때문
    - 많은 테스트와 측정을 통해 보장을 확신할 수 있어야함
- 대부분의 언어와 도구는 실시간 보장을 제공하지 않아,사용 가능한 언어나 라이브러리 범위가 제한됨
    - **Q. 언어 수준에서 실시간 보장은 어떻게 제공하는 거임? 그리고 대표적으로 어떤 언어가 이 보장 제공함?**
        - 이런 방식으로 실시간 보장 제공함
            - GC를 없애거나 제한함
            - 동적 메모리 할당을 제한함
            - 객체 수명과 메모리 사용을 예측 가능하게 만듦
            - 스레드 우선순위와 스케줄링 제어를 제공함
            - 최악 실행 시간 분석이 가능하도록 언어 기능을 제한함
            - 예외, 리플렉션, JIT 같은 예측 어려운 기능을 제한함
        - 대표적인 언어로는 Ada, Spark 와 같은 언어가 있음

#### 가비지 컬렉션의 영향을 제한하기

- 실시간 스케줄링 없이도 GC 로 인한 프로세스 중단 완화 가능함
- 언어 런타임은 객체 할당률과 시간에 따라 남아 있는 여유 메모리 추적이 가능함 → 언제 가비지 컬렉션을 할지와 관련된 유연성을 가지고 있음
    - **Q. 어케 추적함?**
        - 런타임은 heap 을 직접 관리하기 때문에 아래 정보를 가지고 있고, 이 정보로 추적할 수 있음
            - 현재 힙 전체 크기
            - 현재 사용 중인 메모리
            - 남은 메모리
            - Eden 영역이 얼마나 찼는지
            - Old Generation이 얼마나 찼는지
            - 단위 시간당 allocation pointer가 얼마나 빨리 증가하는지
            - GC 후 살아남은 객체 비율
            - Young에서 Old로 승격되는 객체 양
            - 최근 GC pause 시간
            - 최근 GC 빈도
- 최근 아이디어는, GC 가 도는 시점에 들어오는 요청을 다른 노드로 보내는 거임
    - 런타임 수준에서 GC 언제 할지 경고가 가능하니까!
    - 이렇게 하면 응답 시간 상위 백분위가 줄어듦
    - 지연 시간에 민감한 금융 거래 시스템에 사용 가능함
- 이 아이디어 변형한 건, GC 빨리 할 수 있는 수명 짧은 객체만 GC 를 사용하고, 수명이 긴 객체 전체 GC 가 필요해지기 전에 프로세스를 재시작 하는 거임
    - 한 번에 하나씩 재시작 가능
    - 재시작 전에 트래픽을 다른 노드로 옮길 수도 있음

## 지식, 진실, 그리고 거짓말

### 진실은 다수결로 결정된다

- 노드는 특정 상황(내가 리더인가? 나는 살아있는가? 등) 을 판단할 때 자신의 판단을 무조건 믿으면 안됨
- 왜냐면 관찰자 입장에서는 내가 죽어있거나, 혹은 나도 모르게 멈춰있을 수 있기 때문
    - 비대칭적 결함으로 인한 네트워크 장애(받는 건 되는데 보내질 못함 혹은 받지 못하는데 보내기만 함)
    - GC 로 인한 stop-the-world
- 그래서 판단은 정족수(노드 사이의 투표) 에 의존해야 한다
    - 즉, 다른 N 개의 노트로부터 최소 개수의 투표를 받아야 상황을 판단할 수 잇게 해야 한다!
    - 보통 노드의 과반수 이상을 정족수로 삼는다
        - 왜냐면 개별 노드에 장애가 발생해도 시스템은 계속 동작할 수 있기 때문

#### 리더와 잠금

- 분산 시스템에서 ‘시스템 상 오직 하나의 무언가가 존재해야 하는 상황’ 을 구현하려면 주의해야함
    - 스플릿 브레인을 피해야하는 경우
    - 동시성 문제 해결을 위해 특정 객체만 락 잡고 접근하게 하는 경우
- 노드의 과반수가 특정 노드가 죽었다고 선언했음에도 그 노드가 선택된 자인 것처럼 계속 행동한다면 문제가 발생할 수도 있음

#### 펜싱 토큰

- 위에서 문제가 발생하는 이유는, 판단을 노드가 스스로 하기 때문이었음
- 펜싱 토큰은 서버가 락이나 임차권을 승일할 때마다 단조 증가하는 토큰임
- 클라이언트가 쓰기 요청을 보낼 때 해당 토큰을 포함하게 하고, 만일 현재 기록된 펜싱 토큰보다 작은 펜싱 토큰 값으로 요청이 들어오면 거절함

### 비잔틴 결함

- 펜싱 토큰은 ‘정직한 노드’ 를 전제로 성립함
- 만일 정직하지 않은 노드가 가짜 토큰을 포함해 요청을 보낸다면 전제가 깨져 문제가 발생할 수 있음
    - 고의로 그럴 수도 있고 아닐 수도 잇음!
    - 버그나 하드웨어 오류 등으로도 발생할 수 있는 문제이기 때문
- 이와 같이 정직하지 않은 노드가 포함되어 문제가 발생할 수 있는 것을 비잔틴 결함이라고 함
- 일부 노드가 오작동하고 프로토콜을 준수하지 않거나, 악의적인 공격자가 네트워크를 방해하더라도 시스템이 계속 올바르게 동작한다면 해당 시스템은 비잔틴 내결함성을 지닌다고 함
    - 예를 들어, 항공우주 산업 환경에서 메모리나 레지스터에 저장된 데이터는 방사선에 오염되어 예기치 않은 동작이 발생할 수 있음
- 시스템이 비잔틴 내결함성을 지니도록 만드는 프로토콜은 복잡하고, 내결함성을 지닌 임베디드 시스템은 하드웨어 수준의 지원에 의존함
    - 여기서 말하는 네트워크 프로토콜이 아니라, 의사결정 절차를 말하는 거임
    - 정족수를 반영해서 다수결로 의사결정하는 과정 자체 라고 이해하면 됨
- 웹 애플리케이션의 경우, 클라이언트의 행동이 임의적이고 악의적일 것이라고 예상해야함
    - 그래서 입력 확인, 살균(잠재적으로 위험할 수 있는 입력 값을 유효한 값으로 변경하는 처리), 출력 이스케이핑이 중요함
    - e.g. SQL Injection, 크로스 사이트 스크립팅을 막아야함
    - 근데 비잔틴 내결함성 프로토콜으로 이걸 막는 건 아니고, 클라이언트 행동이 허용된 것인지 결정하는 권한을 서버가 가지는 거임
    - 비잔틴 내결함성은 중앙 권한이 없는 p2p 에 더 적합함
        - **Q. 왜 p2p 에 더 적합함? 중앙 권한이라고 하면 중앙 서버 잇는 구조 말하는 거 같은데, 개인 - 개인의 경우에는 서버가 판단을 못하니까 프로토콜 수준에서 막으려고 하는 건가?**
            - ㅇㅇ 맞음. 중앙 판단자가 없어서 그럼
- 비잔틴 내결함성 알고리즘은 충분한 수(2/3) 이상의 정상 노트가 같은 프로토콜을 사용하는 것을 전제로 함
    - 따라서, 오류가 있는 소프트웨어를 여러 노드에 배포하는 것으로는 비잔틴 결함을 막을 수 없음

#### 약한 형태의 거짓말

- 노드가 정직하다고 가정하지만, 약한 형태의 거짓말을 하기도 함
    - e.g. 하드웨어 문제, 소프트웨어 버그, 잘못된 설정 때문에 유효하지 않은 메시지
- 약한 거짓말로부터 보호해주는 메커니즘을 소프트웨어에 추가하는 것을 가치가 있을 수 있음
    - 완전한 비잔틴 내결함성을 지니지는 않지만, 그럼에도 더욱 나은 신뢰성으로 향할 수 있기 때문
- 예시
    - 오염된 패킷은 TCP 와 UDP 에 내장된 체크섬으로 검출되지만, 완벽하지 않음
        - **Q. 어떤 경우일까?**
            - 다른 데이터인데 우연히 같은 체크섬이 나올 수 있음
            - 체크섬 이후에 애플리케이션 내부에서 데이터가 손상될 수 잇음
        - 오염으로부터 보호하려면 애플리케이션 수준 프로토콜에서 체크섬을 사용해야함
            - **Q. 애플리케이션 수준 프로토콜이라는 건 뭐고, 체크섬을 어떻게 사용한다는 거?**
                - 수준 말고 계층이라고 이해하면 됨
                - 애플리케이션 계층 프로토콜은 http 같은 프로토콜을 말하는 거임
                - 그리고 받은 데이터를 대상으로 체크섬을 한 번 더 계산해서 값을 비교한다는 거임
    - 공개적으로 접근 가능한 애플리케이션은 사용자 입력을 신중하게 살균해야함
    - NTP 서버를 여러 대 사용해서 이상한 값 보내는 애 배제할 수 있음

### 시스템 모델과 현실

- 분산 시스템 문제를 해결하기 위해 많은 알고리즘이 설계되고 있음
- 알고리즘은 실행하는 하드웨어와 소프트웨어 설정의 세부 사항에 너무 심하게 의존하지 않도록 작성되어야 함
    - 이렇게 하려면 시스템에서 발생할 것으로 예상되는 결함의 종류를 정형화 해야함
    - 시스템 모델(알고리즘이 가정하는 것을 기술한 추상화) 를 정의해서 정형화 함
- 타이밍 가정에 대해서는 세 가지 시스템 모델이 사용됨
    1. 동기식 모델
        1. 네트워크 지연, 프로세스 중단, 시계 오차에 모두 제한이 있다고 가정함
            1. 네트워크 지연, 중단, 시계 드리프트가 고정된 상한치를 초과하지 않는다는 것!
        2. 현실적인 모델은 아님(현실에서는 기약 없는 지연과 중단이 발생하기 때문)
    2. 부분 동기식 모델
        1. 대부분의 시간에는 동기식 시스템처럼 동작하지만, 때때로 네트워크 지연, 프로세스 중단, 시계 트리프트의 한계치를 초과한다고 가정함
        2. 많은 시스템에서 현실적인 모델
    3. 비동기식 모델
        1. 이 모델에서는 알고리즘이 타이밍에 대한 어떤 가정도 할 수 없음
        2. 심지어 시계가 없을 수도 잇음
        3. 어떤 알고리즘은 비동기식 모델용으로 설계할 수 있지만 매우 제한적임
- 노드 장애 또한 고려해야함
    - 죽으면 중단하는(crash-stop) 결함
        - 알고리즘은 노드 장애가 나는 원인이 노드가 죽었기 때문이라고 가정할 수 있음
        - 노드가 응답을 멈추면 그 노드는 영원히 사용할 수 없고, 복구되지 않는다는 뜻
    - 죽으면 복구하는(crash-recovery) 결함
        - 노드가 죽을 수 있으나, 알려지지 않은 시간 후에는 다시 응답한다고 가정하는 것
        - 메모리에 있는 상태는 손실되지만, 죽어도 데이터가 남아 있는 안정적인 저장소가 있다고 가정함
    - 비잔틴 결함
        - 노드가 다른 노드를 속이거나 기만하는 것을 포함해 전적으로 무슨 일이든 할 수 있다고 가정하는 것
- 현실 시스템을 모델링 할 때는 죽으면 복구하는 결함을 지닌 부분 동기식 모델이 일반적으로 가장 유용한 모델임. 분산 시스템에서 알고리즘은 이 모델에 어떻게 대응할 수 있을까?

#### 알고리즘의 정확성

- 알고리즘이 정확하다는 것은, 알고리즘의 속성과 관련이 잇음
    - 가령 정렬 알고리즘의 경우, 결과 목록에서 두 개의 다른 요소를 선택하면 왼쪽 요소가 오른쪽 요소보다 작다는 속성이 있음
    - 이 속성을 충족할 때 알고리즘이 정확하다고 할 수 있음
- 따라서 분산 시스템의 속성을 사용하면 분산 시스템에서의 알고리즘이 정확하다는 것이 무엇인지 파악할 수 있음
    - 펜싱 토큰의 경우, 알고리즘은 다음 속성을 지님
        - 유일성 : 펜싱 토큰 요청이 같은 값을 반환하지 않음
        - 단조 일련번호 : 두 번째 요청이 시작되기 전에 첫 번째 요청이 완료되었다면, 첫 번째 요청이 가지고 있던 토큰이 두 번째 요청이 가지고 있던 토큰보다 작음
        - 가용성 : 펜싱 토큰을 요청하고 죽지 않은 노드는 결국 응답을 받음
- 알고리즘은 특정 시스템 모델에서 발생할 것으로 가정한 모든 상황에서 그 속성을 항상 만족시키면 해당 시스템 모델에서 정확하다고 함

#### 안전성과 활동성

- 안전성
    - ‘비공식적으로 나쁜 일은 일어나지 않는다’ 라고 정의됨
    - 유일성과 단조 일련번호는 안전성 속성임
    - 안전성 속성이 위반되는 경우, 속성이 깨진 시점을 가리킬 수 있음
        - e.g. 중복된 펜싱 토큰이 발행되어 유일성이 깨졌다면, 중복된 토큰을 반환한 특정 연산을 식별할 수 있음
        - 안전성 속성이 위반된 후에는 해당 위반을 취소할 수 없음. 이미 손상된 상태임
- 활동성
    - ‘좋은 일은 결국 일어난다’ 라고 정의됨
    - 가용성은 활동성 속성임
    - 흔히 정의에 ‘결국에는’ 이라는 말이 포함됨
        - 그래서 최종적 일관성은 활동성 속성임
- 분산 알고리즘은 시스템의 모든 상황에서 안전성 속성이 항상 만족되기를 요구하는 게 일반적임
    - 모든 노드가 죽거나 네트워크 전체에 장애가 생기더라도 알고리즘은 잘못된 결과를 반환하지 않는다고 보장해야함
- 그러나 활동성 속성에 대해서는 경고가 허용됨
    - 노드의 다수가 죽지 않고 네트워크가 중단으로부터 복구됐을 때만 요청이 응답받아야 한다고 말할 수 잇음

#### 시스템 모델을 현실 세계에 대응시키기

- 알고리즘이 올바르다고 증명됐더라도, 그게 반드시 현실 시스템에서의 구현도 언제나 올바르게 동작하다는 뜻은 아님
    - 가정이 깨질 수 있기 때문
        - 죽으면 복구하는 모델에서 알고리즘은 안정된 저장소에 있는 데이터는 노드가 죽어도 남아있다고 가정하지만, 디스크에 잇는 데이터가 오염되거나 하드웨어 오류로 데이터가 지워질 수 있음
- 그래서 실제로 구현할 때에는 불가능하다고 가정했던 일이 발생하는 경우를 처리하는 코드를 포함해야 할 수도 잇음
- 그래도 이론적인 추상 시스템 모델은 현실 시스템의 복잡함에서 우리가 추론할 수 있는 관리 가능한 결함의 집합을 뽑아내고, 이를 이해하고 체계적으로 해결하려고 노력할 수 있게 하는 데 도움을 줌
