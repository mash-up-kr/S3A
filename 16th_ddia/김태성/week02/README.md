# 3장. 저장소와 검색

## B트리

- SS테이블과 마찬가지로 정렬된 키-값 쌍 유지
    - 키-값 검색과 범위 질의에 효율적
- 그러나 설계 철학이 SS테이블과 다름
    - B트리는 세그먼트를 고정 크기 블록(4KB 혹은 그 이상)이나 페이지로 나눔
    - 그리고 한 번에 하나의 페이지에 읽기 또는 쓰기를 진행함
    - 이렇게 하는 이유는 하드웨어와 관련이 잇음
        - **Q. 어떤 관련임?**
            - 디스크/SSD/메모리는 블록 단위로 읽고 씀
            - 그래서 값을 가져올 때 해당 값만 가져오는 게 아니라, 해당 값이 속한 블록을 가져와야함
            - 즉, I/O 단위를 맞춰버려서 한 번의 접근에 최대한의 효율을 뽑으려는 목적으로 설계됨
                - 블록에 여러 키를 저장하여 특정 값이 속한 범위까지 접근하기 위한 탐색 비용을 줄이겠다는 것
    - 페이지는 주소, 위치를 통해 식별이 가능하기 때문에, 한 페이지에서 다른 페이지를 참조할 수 있음
        - 이걸 사용하면 페이지 트리 구성이 가능함
- 한 페이지에서 하위 페이지를 참조하는 수를 분기 계수라고 함
    - 이는 페이지 참조와 범위 경계를 저장할 공간의 양에 의존적임
- 키 값 갱신을 하기 위해서는
    1. 키를 포함하고 있는 리프 페이지 검색
    2. 페이지 값을 바꿈
    3. 페이지를 디스크에 다시 기록
- 새로운 키를 추가하려면
    1. 새로운 키를 포함하는 범위의 페이지를 찾음
    2. 해당 페이지의 키와 값 추가
        - 만일 추가할 공간 없다면 중앙값을 상위 페이지로 올리고, 해당 값을 기준으로 나머지 값을 두 개의 페이지로 분할
- 이렇게 하면 트리가 계속 균형을 유지함
    - N 개 의 키를 가진 트리는 깊이가 항상 O(log N)

### 신뢰할 수 있는 B 트리 만들기

- B 트리의 기본 쓰기 동작은, 페이지 위치를 변경하지 않는다고 가정함
    - **Q. 당연한 거 아님?**
        - 좀 더 풀어서 설명하자면, B트리의 핵심은 제자리 수정임
        - copy-on-write 방식은 쓰기 동작 시, 새로운 페이지에 쓰고 참조를 변경함
        - 근데 B 트리는 제자리에 쓰기 때문에 부모가 가진 자식 페이지의 참조를 유지할 수 있다는 거임
    - 이는 LSM 트리와 대조적임
        - **Q. 왜?**
            - LSM 트리는 쓰기 시 memtable 에 먼저 반영하고, 나중에 SS테이블로 flush 후 컴팩션 하기 때문에 새 세그먼트를 생성하기 때문임
- 데이터 삽입 시, 페이지 공간이 부족하다면 이를 두 개의 페이지로 나누게 됨
    - 이 때 상위 페이지에 분할된 두 페이지의 주소를 갱신해야하는데, 일부 페이지만 기록하고 DB 가 고장날 수 있음
    - 이렇게 트리에서 끊어진 페이지를 고아 페이지라고 함
- DB 가 이를 스스로 복구할 수 있게 하기 위해서는 WAL(write-ahead log, 재실행 로그) 데이터구조를 추가해야함
    - 이건 트리 페이지에 변경된 내용을 적용하기 전에 모든 B트리의 변경을 기록해두는 것
- 같은 자리의 페이지를 갱신하는 작업은 동시성 문제가 생길 수 있기 때문에 latch(래치, 가벼운 lock) 를 사용해 데이터 구조를 보호함
    - **Q. 이건 왜 가볍다하나?**
        - 보호 범위가 작고, 락 잡는 시간도 짧기 때문
        - 그리고 롤백이나 격리 같은 개념도 없음

### B 트리 최적화

- WAL 대신 쓰기 시 복사 방식(copy-on-write scheme) 사용
    - 변경된 페이지는 다른 위치에 기록하고, 트리에 상위 페이지를 새로 만들고, 변경된 위치를 가리키게 하는 것
        - **Q. 왜 이렇게 함? 상위 페이지는 새로 만들지 말고 자식 페이지 참조만 변경하면 되는 거 아님?**
            - 기술적으로는 가능함
            - 근데 원자성 때문에 새로 만드는 거임
                - 장애 발생 시, 자식 페이지는 새로 만들어졌는데, 부모는 수정 중이고, 조상 노드는 옛날 구조일 수 있음
                - 이러면 트리가 중간 상태가 되기 때문에, 이를 방지하기 위해 상위 페이지를 새로 만듦
- 페이지에 키를 축약해 씀
    - **Q. 어떤식일까?**
        - prefix compression 을 대표적으로 사용함
        - 어차피 정렬되니까, 인접한 키가 공통된 prefix 를 갖는다면 그걸 활용하게 하는 방식
            
            ```lua
            * apple
            * (4, ication)  // appl + ication
            * (4, y)        // appl + y
            * (2, t)        // ap + t
            ```
            
- 범위 색인을 할 때는 페이지가 디스크 상의 연속된 공간에 있는 게 유리함
- 트리에 포인터를 추가하여, 리프페이지가 양쪽 형제 페이지를 알 수 있게 함
    - 그러면 범위 색인에 유리해짐. 굳이 상위 페이지 거칠 필요 없음
- 프랙탈 트리(B+ 트리)는 디스크 찾기를 줄이기 위해 로그 구조화 개념을 일부 빌림
    - **Q. 프랙탈 트리란? 왜 로그 구조화 개념을 빌렸다고 하지?**
        - B 트리 + buffer 개념
        - 각 노드에 write buffer 가 존재함
            - 원래는 삽입 요청 들어오면 리프 페이지까지 내려가야 하는데, write buffer 사용해서 저장하다가, 버퍼 다 차면 자식으로 내리는 방식

### B 트리와 LSM 트리 비교

- LSM 트리는 쓰기에서 빠르고, B 트리는 읽기에서 빠름
    - LSM 트리에서 읽기가 느린 이유는, 각 컴팩션 단계에 있는 데이터 구조와 SS테이블을 확인해야 하기 때문
    - 엄밀히 말하면 읽기 속도는 상황에 따라 다른데, 이렇게 정리할 수 있음
        
        
        | **상황** | **더 빠른 구조** |
        | --- | --- |
        | point lookup | B트리 |
        | range scan | 둘 다 괜찮음 |
        | write-heavy | LSM |
- LSM 트리의 장점
    - B 트리는 두 번 기록해야함
        - WAL, 트리 페이지
    - 그리고 페이지 내에서 특정 부분만 변경되어도 전체 페이지를 기록해야함
        - **Q. 왜?**
            - SS테이블을 불변으로 두기 때문
            - 그래서 특정 부분만 변경하려면 다시 정렬 구조 유지해야하고, 압축해야하고, 중간 값 길이가 기존과 달라지면 뒤에 데이터 다 밀리고..
            - 이런 이유 때문에 걍 새로 씀
    - 로그 구조화 색인에서도 SS테이블의 컴팩션과 병합 때문에 데이터 여러 번 다시 써야함
    - 이렇듯 쓰기 한 번이 여러 번의 쓰기를 야기하는걸 쓰기 증폭이라고 함
    - LSM 트리는 B 트리보다 쓰기 처리량을 높게 유지할 수 있음
        - B 트리는 여러 페이지를 덮어쓰지만, LSM 트리는 순차적으로 컴팩션된 SS테이블 파일을 씀
        - **Q. 이거 자세히 이해 ㄱㄱ**
            - LSM 에서는 디스크 쓰기를 모아서 처리하기 때문
                - memtable 에 먼저 모으고, 한 번에 디스크로 flush 함
                - 즉, 임의 쓰기를 줄일 수 있음
    - LSM 트리는 압축률이 더 좋음
        - B 트리 저장소 엔진은 파편화로 인해 사용하지 않는 디스크 공간 일부가 남음
        - 그러나 LSM 은 주기적으로 파편화를 없애기 위해 컴팩션을 진행함
            - 레벨 컴팩션을 사용하는 경우에 특히 더 그럼
    - 참고) 대다수의 SSD 펌웨어는 내장 저장소 칩에서 임의 쓰기를 순차 쓰기로 전환함
        - **Q. how?**
            - FTL (Flash Translation Layer) 을 사용함
            - 내부적으로 매핑 테이블을 사용해서 임의 쓰기를 순차 쓰기로 전환하는 방식
- LSM 트리의 단점
    - 컴팩션이 읽기 쓰기 성능에 영향을 줌
        - 특히 DB 가 커질수록 컴팩션을 위해 더 많은 디스크 대역폭이 필요함
    - 컴팩션이 유입 속도를 따라가지 못하는 경우, 병합되지 않는 세그먼트 수가 계속 증가하여 읽기 시간이 느려짐
        - 이건 엔진이 스스로 유입 속도 조절 안하니까 명시적 모니터링 해야함
            - **Q. 왜 안해?**
                - DB 가 Generic 시스템이기 때문
                - 그리고 정책은 운영자의 권한이기 때문임
    - B 트리는 각 키가 색인에 한 곳에만 존재하나, 로그 구조화 저장소 엔진에서는 다른 세그먼트에 같은 키의 복사본이 존재할 수 있음
        - 때문에 트랜잭션 시맨틱을 제공하는 DB 에는 B 트리가 훨씬 좋음
            - **Q. 트랜잭션 시맨틱?**
                - ACID, Isolation, Consistency 보장하는 것
            - LSM 도 트랜잭션 시맨틱 지원이 가능하나, 구현이 더 복잡함
    - 많은 RDB 에서 트랜잭션 격리는 키 범위의 잠금을 사용해 구현하나, B 트리 색인에서는 트리에 직접 잠금을 포함시킴
        - **Q. 와이!**
            - 정확히 말하면 노드/경로 수준의 잠금을 건다고 보는 게 맞음
            - 예를 들어, 두 스레드가 동시에 같은 리프 페이지에 삽입하면 부모 업데이트가 충돌나거나, 이미 split 되었는데 또 split 시도할 수 있음
            - 그래서 트리 구조 자체가 깨질 수 있기 때문에 락을 이렇게 걸어용

## 기타 색인 구조

### 색인 안에 값 저장하기

- 색인에서의 키는 질의가 검색하는 대상
- 색인에서의 값은
    1. 질문의 실제 로우 [키] → [실제 값]
    2. 다른 곳에 저장된 로우를 가리키는  참조 [키] → [포인터]
        1. 이 경우, 로우가 저장된 곳을 힙 파일이라고 하고, 순서 없이 데이터 저장함
        - 각 색인은 힙 파일에서 위치만 참조하고 실제 데이터는 일정한 곳에 유지함
        - 키를 변경하지 않고 값을 갱신할 때 효율적
- 색인에서 힙 파일로 이동하느 것은 읽기 성능에 불이이익임
    - 그래서 색인 안에 바로 색인된 로우를 저장하는 게 나음
    - 이걸 클러스터드 색인이라고 함 [키] → [행 전체 데이터]
        - **Q. 이게 먼말임? 값을 실제 로우로 저장하는 거랑 머가 다름? 미리 찾아서 넣어둔다는 건가?**
            - 자~ 예시 들어갑니다잉~
            
            ```lua
            User 테이블
            --------------------------------
            id | name | age | city
            --------------------------------
            1  | Kim  | 25  | Seoul
            2  | Lee  | 30  | Busan
            3  | Park | 28  | Incheon
            ```
            
            1. [키] → [실제 값] ==ㅡ [name] → [age, row 위치]
                
                Kim  → (25, row #1)
                Lee  → (30, row #2)
                Park → (28, row #3)
                
                - 참고로 여기서 row 같이 넣어두는 이유는, 값만으로는 쿼리를 끝낼 수 없기 때문
                    - e.g. `SELECT city FROM User WHERE name = 'Kim’` 이러면 인덱스에 city 가 없기 때문에 row 를 타고 city 를 확인해야함
            2. [키] → [포인터] == [name] → [row 위치]
                
                Kim  → (row #1 위치)
                Lee  → (row #2 위치)
                Park → (row #3 위치)
                
            3. [키] → [행 전체 데이터] == [id] → [행 전체 데이터]
                
                1 → (1, Kim, 25, Seoul)
                2 → (2, Lee, 30, Busan)
                3 → (3, Park, 28, Incheon)
                
- 클러스터드 색인이랑 비클러스터드 색인의 절충안을 커버링 색인이나 포괄열이 있는 색인이라고 함
    - 색인 안에 테이블의 칼럼 일부를 저장하는 것
    - 이렇게하면 색인만 사용해 일부 질의에 응답이 가능함 (이 경우 색인이 질의를 커버햇다고 함)

## 트랜잭션 처리나 분석

- OLTP(online transaction processing)
    - 실시간으로 발생하는 짧고 빈번한 트랜잭션을 처리하는 시스템
    - 색인을 사용해 일부 키에 대한 적은 수의 레코드를 찾아 원시 데이터 반환
- OLAP(online analytic processing)
    - 대량의 데이터를 기반으로 분석·집계·통계 쿼리를 수행하는 시스템
    - 데이터 분석에 사용
        - 많은 수의 레코드를 스캔해 레코드당 일부 칼럼만 읽어 집계 통계를 계산
- OLTP vs OLAP
    
    
    | **특성** | **트랜잭션 처리 시스템(OLTP)** | **분석 시스템(OLAP)** |
    | --- | --- | --- |
    | 주요 읽기 패턴 | 질의당 적은 수의 레코드 키 기준으로 가져옴 | 많은 레코드에 대한 집계 |
    | 주요 쓰기 패턴 | 임의 접근, 사용자 입력을 낮은 지연 시간으로 기록 | 대규모 불러오기(bulk import, ETL) 또는 이벤트 스트림 |
    | 주요 사용자 | 웹 애플리케이션을 통한 최종 사용자/소비자 | 의사결정 지원을 위한 내부 분석가 |
    | 데이터 표현 | 데이터의 최신 상태(현재 시점) | 시간이 지남에 따른 이벤트 이력 |
    | 데이터셋 크기 | 기가바이트에서 테라바이트 | 테라바이트에서 페타바이트 |
- 분석을 위해 별도로 구축한 데이터베이스를 데이터 웨어하우스라고 함

### 데이터 웨어하우징

- 데이터 웨어하우스는 분석가들이 OLTP 작업에 영향을 주지 않고 질의할 수 있는 개별 데이터베이스
    - OLTP 시스템에 있는 데이터의 읽기 전용 복사본
- 데이터 웨어하우스로 데이터를 가져오는 과정을 ETL 이라고 함
- 개별 데이터 웨어하우스를 사용하면 분석 접근 패턴에 맞게 최적화가 가능함
    - 위에 설명한 색인 알고리즘은 OLTP 에서는 잘 동작하나, 분석 질의에서는 좋지않음

### OLTP 데이터베이스와 데이터 웨어하우스 차이점

- 표면적으로는 둘 다 SQL 질의 인터페이스 지원해서 비슷해보임
- 그러나 내부는 각각 다른 질의 패턴에 맞게 최적화됨

### 별 모양 스키마와 눈꽃송이 모양 스키마

- 별 모양 스키마(차원 모델링)
    - 테이블 관계가 시각화될 때 사실 테이블이 가운데에 있고, 차원 테이블이 둘러싸고 있는 형태
    - 사실 테이블
        - 별 모양 스키마 중심에 존재
        - 각 로우가 특정 시각에 발생한 이벤트임
        - 사실은 개별 이벤트를 담음
            - 근데 이러면 사실 테이블이 매우 커질 수 있는 문제가 잇음
                - **Q. 해결은?**
                    - partitioning, aggregation 등으로 해결 가능
        - 사실 테이블의 일부 칼럼은 속성, 다른 칼럼은 테이블을 가리키는 외래 키 참조(차원 테이블) 임
    - 차원 테이블
        - 이벤트의 속성인 육하원칙을 나타냄
        - 날짜와 시간도 보통 차원테이블 사용해 표기
- 눈꽃송이 모양 스키마
    - 별 모양 스키마의 변형
    - 차원이 하위차원으로 더 세분화된 형태

## 칼럼 지향 저장소

- 사실 테이블(분석하려는 사건/행위 자체를 저장하는 테이블) 은 보통 칼럼이 100개 이상이지만, 일반적인 데이터 웨어하우스  질의는 한 번에 4~5개 칼럼만 접근함
- 이 질의를 어떻게 효율적으로 실행할 수 있을까?
    - OLTP DB 는 로우 지향 방식으로 데이터를 배치함
        - 한 로우의 모든 값은 서로 인접하게 저장됨
    - 칼럼 지향 저장소는, 모든 값을 하나의 로우에 저장하지 않고, 각 칼럼별로 모든 값을 함께 저장함
        - 이러면 질의에 사용되는 칼럼만 읽고 분석이 가능

### 칼럼 압축

- 데이터를 압축하면 디스크 처리량을 더 줄일 수 있음
- 많은 값이 반복해서 나타날수록 압축하기 좋음
- 비트맵 부호화
    - 일반적으로 칼럼 수(N)보다, 칼럼이 가질 수 있는 값의 수(M)가 더 적음
    - 그럼 칼럼이 가질 수 있는 값의 개수(M)만큼의 비트맵으로 표현이 가능함
        - M 개의 로우 각각은 비트맵 하나를 가지고, 각 비트맵의 길이는 N
        - 해당 로우가 값을 가지면 비트는 1, 아니면 0으로 표현
- 런 렝스 부호화
    - **Q. 머임?**
        - 이렇게 부호화하는 걸 말함
            - AAAAAABBBB → (A,6)(B,4)

### 메모리 대역폭과 벡터화 처리

- 데이터 웨어하우스 질의는 디스크 → 메모리로 데이터 가져오는 대역폭이 병목임
- 이외에도 메인메모리 → CPU 캐시로 가는 대역폭의 효율적 사용, CPU 명령 처리 파이프라인에서 분기 예측 실패, 버블을 피하며 단일 명령 다중 데이터 명령을 사용하게 신경써야함
    - **Q. 버블이 머야**
        - CPU 파이프라인이 비어있는 시간을 버블이라고 함
        - 예를 들어, 분기 예측 실패하면 예상하고 준비한 명령을 버리게 됨. 이 때 파이프라인이 놀게 되고, 이걸 버블이라고 함
- 칼럼 저장소 배치는 CPU 주기를 효율적으로 사용하기에 적합
    - 질의 엔진은 압축된 칼럼 데이터를 CPU 의 L1 캐시에 맞게 덩어리로 나눠 가져오고, 이걸 타이트 루프에서 반복함
    - 타이트 루프는 함수 호출이나 분기 없이 반복만 수행하는 루프임
        - 그래서 빠름
- AND, OR 연산자는 압축된 칼럼 데이터 덩어리를 바로 연산할 수 있게 설계할 수 있는데, 이걸 벡터화 처리라고 함

### 칼럼 저장소의 순서 정렬

- 각 칼럼을 독립적으로 정렬할 수는 없음
    - 왜냐면 이렇게 하면 칼럼의 어떤 항목이 동일한 로우에 속하는지 알 수 없기 때문
    - **Q. 먼말임?**
        - 예시로 봅시다
            
            
            | **row** | **name** | **age** |
            | --- | --- | --- |
            | 1 | Kim | 30 |
            | 2 | Lee | 20 |
            | 3 | Park | 25 |
            - 여기서 각 칼럼을 독립적으로 정렬하면
                - name sorted: `Kim, Lee, Park`
                - age sorted: `20, 25, 30`
            - 이걸 매칭하면 기존 행이 깨지게 됨
- 칼럼별로 저장이 되어도 데이터는 한 번에 전체 로우를 정렬해야함
- 정렬을 하면 칼럼 압축에 도움이 됨
- 여러 장비에 데이터를 복제하는 경우, 각 데이터를 서로 다른 방식으로 정렬해 저장하면 질의 처리 시 질의 패턴에 가장 적합한 버전을 사용할 수 있음

### 칼럼 지향 저장소에 쓰기

- 압축된 칼럼에서는 B 트리와 같은 제자리 갱신 접근 방식은 불가능
    - 정렬된 테이블의 중간 로우에 삽입을 원하는 경우, 모든 칼럼 파일을 재작성해야함
    - 삽입은 모든 칼럼을 일관되게 갱신해야함
    - **Q. 먼소리?**
        - 이것도 위의 예시랑 비슷한데, 특정 칼럼을 기준으로 정렬되어 있으면, 해당 칼럼을 기준으로 삽입할 데이터의 인덱스를 정했을 때, 모든 칼럼에 대해 같은 인덱스로 삽입이 되어야 한다는 거임
        - 즉, row alignment 를 유지해야한다~
- 이건 LSM 트리 방식으로 해결 가능
    - 모든 쓰기는 인메모리 저장소에서 정렬된 구조에 추가
    - 충분한 쓰기 데이터가 모이면 디스크의 칼럼 파일에 병합하고 새로운 파일에 기록
    - 이 경우, 질의는 디스크의 칼럼 데이터와 메모리의 쓰기를 모두 조사해 두 가지를 결합해야함

### 데이터 큐브와 구체화 뷰 ← 더 자세히 알아보기

- 구체화 집계
    - 동일한 집계를 다양한 질의에서 사용한다면, 매번 연산할 필요 없이 캐시해두면 되는 거 아님?
    - 이 캐시를 구체화 뷰로 만들 수 있음
    - 구체화 뷰
        - 디스크에 기록된 질의 결과의 실제 복사본
        - 원본 데이터 변경 시 갱신되어야함
            - 갱신 비용이 비싸기 때문에 OLTP 에서는 잘 안 씀
            - 데이터 웨어하우스는 읽기 비중이 크기 때문에 자주 씀
    - OLAP 큐브(데이터 큐브)
        - 일반화된 구체화 뷰의 특별 사례
        - 다양한 차원으로 그룹화한 집계 테이블
        - 원시 데이터에 질의하는 것과 동일한 유연성을 보장하지는 못함

# 4장. 부호화와 발전

## 이진 부호화

- 데이터를 0, 1 로 변경해서 보내는 방식

### 스리프트와 프로토콜 버퍼

- 둘은 같은 원리를 기반으로 함
    - 필드 이름을 문자열로 쓰는 게 아니라, 미리 정의해둔 스키마와 필드 번호로 데이터를 이진 부호화 함
    - 이렇게 하면 필드 이름을 넣을 필요가 없기 때문에, 용량을 더 줄일 수 있고, 파싱 속도도 빨라짐
- 필드 번호를 사용하는 경우, 새로운 필드가 추가되어도 하위 호환성을 유지할 수 있음
    - 예전 프로그램은 새로운 필드의 필드 번호를 가지고 있지 않기 때문에 알아서 무시함

### 아브로

- 미리 공유된 스키마를 기준으로 값을 해석하는 방식
    - 필드의 이름, 번호, 타입 정보를 제거하고 보내기 때문에 살펴본 이진 부호화 방식 중 공간 절약 효과가 가장 큼
    - 스리프트나 프로토콜 버퍼 방식과 달리 스키마에 태그 번호가 없기 때문에 동적 생성 스키마에 더 친숙함
- 데이터를 쓸 때 사용하는 스키마를 쓰기 스키마 라고 함
    - 실제 데이터는 이걸 기준으로 부호화됨
- 데이터를 읽을 때 사용하는 스키마를 읽기 스키마 라고 함
- 아브로는 쓰기 스키마와 읽기 스키마를 비교해 필드 변경을 해석함
    - 때문에 스키마 발전에 강함

## 데이터플로 모드

- 하나의 프로세스에서 다른 프로세스로 데이터를 전달하는 방법

### 데이터베이스를 통한 데이터플로

- 하위 호환성과 상위 호환성이 필요
    - 하위 호환성은 이전에 기록한 내용을 미래에 복호화하기 위해 필요
    - 상위 호환성은 최신 버전의 서버에서 작성한 데이터를 이전 버전의 서버가 읽을 수 있게 하기 위해 필요
- 최신 버전의 서버에서 작성한 데이터를 이전 버전의 서버에서 읽고 재기록 하는 경우 데이터가 유실될 수 있음
    - 따라서 이전 서버에서 해석하지 못하는 필드도 그대로 유지하게 해야함

## 서비스를 통한 데이터플로: REST와 RPC

### 웹서비스

- 두 가지 방법이 있음. REST, SOAP
    - 철학 측면에서 둘은 정반대임
- REST
    - 프로토콜이 아니라 HTTP 원칙을 토대로 한 설계 철학
    - 간단한 데이터 타입 강조, URL 사용해 리소스 식별
    - 캐시 제어, 인증, 콘텐츠 유형 협상에 HTTP 기능 사용
    - 이 원칙에 따라 설계된 API 를 RESTful 이라고 함
- SOAP
    - 네트워크 API 요청을 위한 XML 기반 프로토콜
        - WSDL(Web Services Description Language) 이라고 부르는 XML 기반 언어를 사용해 기술함
        - 사람이 읽을 수 있게 설계되지 않음
    - HTTP 와 독립적이며 대부분의 HTTP 기능을 사용하지 않음

### 원격 프로시저 호출(RPC) 의 문제

- RPC(Remote Procedure Call)
    - 원격 네트워크 서비스 요청을 같은 프로세스 안에서 특정 프로그래밍 언어의 함수나 메서드를 호출하는 것과 동일하게 사용 가능하게 하는 기술
    - 이러한 추상화를 위치 투명성이라고 함
- RPC 문제
    - 언어에 제한적임
        - 특정 언어의 객체 모델이나 직렬화 방식에 묶이면 다른 언어와 통신하기 어려움
    - 플랫폼에 제한적임
        - 특정 운영체제나 특정 벤더 기술에 의존하면 다른 기종의 환경에서 호환성이 떨어짐
    - 시스템 설계가 복잡함
        - 브로커, 라우팅, 장애 처리, 재시도, 서비스 발견 같은 분산 시스템 문제를 함께 다뤄야 해서 구조가 복잡함
    - 버전 호환성이 약함(초기 RPC)
        - 인터페이스나 데이터 구조가 바뀌면 상위 호환성, 하위 호환성을 유지하기 어려워 기존 클라이언트나 서버가 쉽게 깨질 수 있음
- 로컬 함수 호출은 예측 가능하나, 네트워크 요청은 예측이 어려움
    - 타임아웃, 멱등성, 지연, 부호화 등 고려할 문제가 많음
    - 그래서 네트워크 문제를 함께 고려해야함
- REST 는 네트워크 프로토콜이라는 사실을 숨기려 하지 않는 게 장점임
    - **Q. 왜?**
        - 개발자가 네트워크 프로토콜임을 인지하니까 더 현실적으로 설계할 수 있게 함

### RPC 의 현재 방향

- gRPC
    - 프로토콜 버퍼를 이용한 RPC 구현
    - 하나의 요청과 하나의 응답이 아니라, 시간에 따른 일련의 요청과 응답으로 구성된 스트림을 지원함
- 차세대 RPC 프레임워크는 원격 요청이 로컬 함수 호출과 다르다는 사실을 분명히함
    - 예를 들어, 실패 가능성이 있는 비동기 작업을 캡슐화하기 위해 future, promise 를 사용함
    - 이런 프레임워크 중 일부는 서비스 찾기를 제공함
        - 클라이언트가 특정 서비스를 찾을 수 있는 IP 주소와 포트를 제공하는 것
        - **Q. 이게 왜 필요하고 중요함? DNS 확장 버전 같은데**
            - DNS 진화된 버전 맞음
            - IP/Port 의 동적 변경에 대응하기 위해 필요함
                - Q. 동적 변경은 왜 돼?
                    - 서버 재시작, 오토스케일링 등의 이유로 됨

### 데이터 부호화와 RPC 의 발전

- 서비스를 통한 데이터플로의 발전성은, 모든 서버를 먼저 갱신하고 모든 클라이언트를 갱신해도 문제가 없다 가정할 수 잇음
- 즉, 하위 호환성만 필요하고 상위 호환성은 필요하지 않음
- 이런 가정을 유지하기 위해 RPC 는 보통 내부 서버에서만 사용함

## 메시지 전달 데이터플로

- 메시지 전달 통신은 일반적으로 단방향임
    - 이건 RPC 랑 다른 점임
- 그래서 송신 프로세스는 보통 메시지에 대한 응답을 기대하지 않음
    - 응답 전송을 별도의 채널에서 수행함. 이걸 비동기라고 함
    - 송신 프로세스는 단순히 메시지를 보내고 잊음

### 메시지 브로커

- 프로세스가 메시지를 이름이 지정된 큐나 토픽으로 전송
    - 동일한 토픽에 여러 생산자가 있을 수 있음
- 브로커는 큐나 토픽의 하나 이상의 소비자나 구독자에게 메시지 전달
- 메시지는 일부 메타데이터를 가진 바이트열이므로 모든 부호화 형식 사용이 가능함
    - 그래서 메시지 브로커는 특정 데이터 모델을 강요하지 않음
- 부호화가 상하위 호환성을 모두 가지면 메세지 브로커가 생산자, 소비자를 독립적으로 변경해 임의 순서로 배포할 수 잇는 유연성을 얻게 됨
- 소비자가 다른 토픽으로 메시지를 다시 게시하는 경우에는 필드가 유실되지 않게 유의해야함
    - 앞서 설명한 최신 버전의 서버가 저장한 데이터를 이전 버전의 서버가 읽고 다시 저장하는 경우와 유사

### 분산 액터 프레임워크

- 액터 모델
    - 단일 프로세스 안에서 동시성을 위한 프로그래밍 모델
    - 스레드(경쟁 조건, 락, 교착 상태와 연관된 문제들)를 직접 처리하는 대신, 로직이 액터에 캡슐화됨
    - 각 액터는 하나의 클라이언트나 엔티티
    - 액터는 공유되지 않는 로컬 상태를 가질 수 있고, 비동기 메시지의 송수신으로 다른 액터와 통신함
    - 그러나 메시지 전달을 보장하지는 않음
    - 각 액터 프로세스는 한 번에 하나의 메시지만 처리
        - 그래서 스레드에 대해 걱정 안해도 되고, 프레임워크와 독립적 실행이 가능함
- 분산 액터 프레임워크에서 프로그래밍 모델은 애플리케이션 확장에 사용됨
    - 송신자, 수신자가 같은 노드에 있는지 여부와 관계없이 동일한 메시지 전달 구조를 사용함
        - **Q. 이건 RPC 랑 다른 느낌인 거 같은데 맞음? 결국 송, 수신자 변경과 무관하기 때문에 가능하다는 거 같은데**
            - ㅇㅇ 맞음
- 단일 프로세스 안에서도 메시지 유실을 가정하기 때문에 위치 투명성이 RPC 보다 더 잘 동작함
- 로컬과 원격 통신 간 근본적인 불일치가 적음
    - **Q. RPC 의 문제를  해결한다고 봐도 되나?**
        - 일부 해결함. 주로 철학 관련된 부분
        - 근데 분산 시스템 관련 문제는 여전히 남아있음
- 분산 액터 프레임워크는 메시지 브로커와 액터 프로그래밍 모델을 단일 프레임워크에 통합함
    - 그러나 여전히 상하위 호환성에 유의해야함
