## 복제

- 데이터의 가용성, 읽기 확장, 지리적 지연 감소를 위해 복제를 사용
    - 데이터셋이 단일 장비에 들어가지 않는 경우에는 파티셔닝(샤딩) 사용
    - 실무에서는 복제와 샤딩을 모두 사용

### 문제: 복제된 데이터의 변경

- 데이터가 변경되지 않는다면 복제는 쉬움
- 그러나 실제로는 데이터가 변경되기 때문에 복제가 복잡해짐
- 대표적인 복제 알고리즘은 single-leader, multi-leader, leaderless 임

## 리더와 팔로워

- 복사본을 저장하는 각 노드를 복제 서버라고 함
- 모든 복제 서버에 모든 데이터가 있다는 걸 어떻게 보장하는가?
    - 리더 기반 복제(마스터 슬레이브 복제) 를 사용함
    - 복제 서버 중 하나를 리더로 선출
    - 쓰기는 리더에서만 수행
    - 이후 팔로워(복제 서버) 로 복제 로그나 변경 스트림을 전송하여 업데이트
        - **Q. 내부에서 돌아가는 프로세스가 있는 건가?**
            - 맞음. DB 의 경우 replication thread, background worker, WAL sender, WAL receiver 과 같은 내부 프로세스가 있음
    - 읽기는 모든 노드에서 가능

### 동기식 vs 비동기식 복제

- 동기식
    - 모든 팔로워가 쓰기 로그를 수신한 것을 보장
    - 리더와 팔로워의 일관성을 보장함
    - 팔로워가 쓰기 로그 수신 응답을 주지 않으면 쓰기가 처리될 수 없음
        - 이 경우 모든 쓰기를 block 하고 복제 서버의 복구를 기다려야함
        - 따라서 이 방식은 비현실적임. 그래서 일부는 동기식, 다른 일부는 비동기식으로 업데이트하는 반동기식 방식을 사용
            - **Q. 어떤 경우에 사용할 수 있을까?**
                - 데이터 유실은 줄이고 싶지만, 모든 팔로워를 기다리기는 부담스러운 경우에 사용 가능
                - e.g. 결제, 주문 등의 유실되면 안되는 데이터
- 비동기식
    - 모든 팔로워가 쓰기 로그를 수신한 것을 보장하지 않음
    - 리더가 죽으면 팔로워에 복제되지 않는 쓰기는 유실됨
    - 모든 팔로워가 죽어도 리더는 계속 동작 가능

### 새로운 팔로워 설정

- 새로 추가한 팔로워가 모든 시점에 최신 데이터를 들고 있을 수는 없음
    - 그래서 조회 시점에 따라 결과가 다를 수 있음
- Q. 그럼 어떻게 하나?
    1. 특정 시점의 데이터 스냅샷을 가져옴
    2. 스냅샷을 팔로워 노드에 복제
    3. 팔로워는 리더에 스냅샷 이후의 변경 요청
    4. 응답받은 쓰기 로그로 데이터 업데이트

## 노드 중단 처리

### 팔로워 장애: 따라잡기 복구

- 리더로부터 수신한 데이터 변경 로그를 로컬에 보관
- 장애 후 재부팅 시 장애 발생 전 마지막으로 처리한 트랜잭션 이후의 데이터 변경을 요청
- 이후 업데이트

### 리더 장애: 장애 복구

- 장애 판단은 일반적으로 타임아웃을 사용함
- 일정 시간 동안 응답이 없으면 장애로 판단
    - **Q. 여기서 일정 시간은 어떻게 정하는 게 좋을까?**
        - 평균 네트워크 지연, GC pause 가능성, 디스크 I/O 지연, 데이터센터 간 RTT 등을 고려해야함
        - 실무에서는 고정 타임아웃만 신뢰하는 게 아니라, heartbeat + election timeout (리더가 죽었다고 판단하는 시간) 을 같이 확인함
- 리더 장애 발생 시 장애 복구(failver)
    1. 새로운 리더 선출
        1. **Q. 어떻게? 각자 가진 최신 로그를 보내고, 제일 최신인 애가 리더로 되나? 근데 비동기식 방식에서는 팔로워의 업데이트를 보장하지 않는데, 모든 팔로워가 업데이트 되지 못한 상태에서 리더가 죽으면 데이터 복구는?**
            1. ㅇㅇ. 일반적으로 가장 최신 로그를 가진 팔로워를 리더 후보로 삼음
            2. 비동기식이라서 장애 시 데이터 유실 가능성을 완전히 제거할 수는 없음
    2. 클라이언트의 쓰기가 변경된 리더로 전송되게 재설정
        1. **Q. 이건 서버에서 재설정하는 건가?**
            1. 아님. 서버 재배포로 처리하는 거는 너무 느리고 리스크가 큼
            2. 그래서 DB 드라이버가 클러스터 메타데이터를 보고 새 리더로 재연결하거나, 내부 로드밸런서 갱신하는 방식으로 처리함
        2. **Q. 새로운 리더가 선출된 경우, 기존 리더가 복구되엇을 때 자신이 리더가 아님을 어떻게 알게 하지? (스플릿 브레인 문제)**
            1. 리더 세대 번호를 외부 시스템에 저장해둠. 그러면 기존 리더가 복구되었을 때 더 높은 세대 번호가 존재하는 걸 보고 더 이상 자기가 리더가 아님을 알 수 있음
    3. 팔로워가 새로운 리더로부터 변경 소비하게 변경
- 복구에 대해 고민할 점이 많아 일부 팀은 수동 복구를 선호하기도 함

## 복제 로그 구현

### 구문 기반 복제

- 리더가 쓰기 요청(구문) 을 기록하고 쓰기 실행 후, 팔로워에게 로그를 전송하는 경우
- 이건 복제가 일관되지 않을 가능성이 존재함
    - NOW() 나 RAND() 같은 비결정적 함수 호출
    - 자동증가 칼럼 사용이나, 데이터베이스에 있는 데이터에 의존하는 경우, 노드가 현재 가지고 있는 데이터에 따라 결과가 달라질 수 있음
- 비결정적 함수 호출이 고정 값을 반환하도록 하는 등의 해결 방법은 있으나, 엣지 케이스가 많아 일반적으로 다른 방식을 선호함 → 로우 기반 복제(MySQL) or 트랜잭션 결정적인 구문 복제(VoltDB)
    - **Q. 트랜잭션 결정적인 구문 복제가 머임?**
        - SQL 문장을 그대로 복사하면서 모든 노드에서 결과가 반드시 같아지도록 만드는 방식
        - 트랜잭션 실행 순서를 모든 노드에서 맞추고, 비결정적 값을 미리 고정해서 멱등성이 유지되도록 함
        - **Q. 근데 설명만 보면  비결정적 함수 호출이 고정 값 반환하게 하는 거랑, 트랜잭션 결정적인 구문 복제랑 다른 게 없는 거 같은데 뭐가 다른 거임?**
            - 비슷한데 커버하는 범위가 다름
            - 비결정적 함수 고정은 작은 해결책이고, 트랜잭션 결정적 복제는 시스템 전체를 결정적으로 실행하게 만드는 설계임
            - 비결정적 함수 고정은 SQL 에서 비결정적인 함수 값을 치환함
            
            ```lua
            INSERT INTO logs(created_at) VALUES (NOW());
            
            // NOW 를 치환함 
            INSERT INTO logs(created_at) VALUES ('2026-04-27 17:00:00'); 
            ```
            
            - 트랜잭션 결정적 구문 복제는 더 큰 개념으로, 다루는 범위가 넓음
                - 트랜잭션 순서 결정
                - 동시성 제어 방식 제한
                - 랜덤값/시간값 고정
                - 외부 상태 의존 제거
                - stored procedure 실행 결과 결정성 보장

### 쓰기 전 로그(WAL) 배송

- 모든 쓰기는 로그에 기록됨
    - 로그 구조호 저장소 엔진(SS테이블, LSM트리)은 로그 자체가 저장소임
        - 로그 세그먼트는 작게 유지되고 백그라운드로 GC
    - B 트리
        - 모든 변경은 WAL 에 쓰여지기 때문에 장애 이후에도 일관성 있는 상태로 복원 가능
- 이 두 경우 모두 로그는 append-only 임
    - 동일한 로그로 다른 노드에 복제 서버 구축 가능
    - postgresql, 오라클 등에서 사용되는 방식임
    - 그러나 로그가 저수준의 데이터를 기술하기 때문에 엔진에 영향을 받음
        - **Q. 예를 들어?**
            - 페이지 구조, 인덱스 내부 형식, WAL 레코드 포맷 등의 저수준 변경 내용까지 포함되어 있기 때문에, 엔진 버전이 다르면 WAL 을 해석하지 못할 수 있음
        - 그래서 저장소나 소프트웨어 버전을 리더랑 맞춰야함
        - 복제 프로토콜이 버전 불일치를 허용하지 않는다면 업그레이드 시 중단시간이 필요함
            - **Q. 중단하고 어떻게 하는데?**
                - 쓰기 중단 or 리더 전환 → 팔로워들의 팔로업 수행 → DB 중지 후 버전 업 → DB 재시작 → 복제 재개 → 필요하다면 새 버전의 노드를 리더로 승격

### 논리적(로우 기반) 로그 복제

- 앞서 살펴본 WAL 을 배송하는 방식은 저장소 엔진에 영향을 받는 문제가 있었음
- 그래서 저장소 엔진의 물리적 데이터 표현과 구분된 논리적 로그를 사용함
    - **Q. 형태 예시 ㄱㄱ**
        - 물리적인 페이지 변경이 아니라, 테이블, row, column 수준의 변경을 표현함
        
        ```lua
        // Insert 로그 예시
        {
          "table": "users",
          "op": "INSERT",
          "after": { "id": 11, "name": "Lee", "age": 25 }
        }
        ```
        
- RDB 의 논리적 로그는 로우 단위로 데이터베이스 테이블에 쓰기를 기술한 레코드 열임
    - 삽입된 로우의 로그는 모든 칼럼의 새로운 값 포함
    - 삭제된 로우의 로그는 로우 식별자 포함. 일반적으로 기본키이나, 없다면 삭제 전의 모든 칼럼 값을 로깅
    - 갱신된 로우의 로그는 로우 식별자 + 변경된 모든 칼럼의 새로운 값(모든 칼럼의 새로운 값이기도 함) 포함
- 여러 로우를 수정하는 트랜잭션은, 여러 로그 레코드 생성 후 트랜잭션이 커밋됐음을 레코드에 표기
    - **Q. 왜 여러 row 트랜잭션에서 commit 을 레코드에 기록해? 단일 수정은 commit 표기 안 해?**
        - 팔로워가 커밋을 보고 하나의 확정된 트랜잭션으로 적용하게 하기 위함
        - 단일 수정도 커밋  개념은 있긴 함. 근데 트랜잭션에서 더 중요하게 쓰이는 거임
    - MySQL 의 이진 로그는 이 방식 사용
        - **Q. 이진 로그가 머임? 그리고 레코드에 표기하면 머 어떡함? 내부 스레드가 알아서 읽고 보냄?**
            - 이진 로그는 데이터 변경 이벤트를 기록하는 로그임
            - PostgreSQL WAL 은 물리적 로그에 가까움
                - 어떤 페이지의 어떤 바이트가 바뀌었는지와 같은 저장소 내부 변경을 기록함
            - MySQL binlog 은 논리적 로그에 가까움
                - users 테이블의 id=1 row가 age=20에서 age=21로 변경됨 과 같이 고수준에서 작성됨
- 논리적 로그를 사용하면 하위 호환성 유지 및 리더와 팔로워에서 다른 버전이나 저장소 엔진 사용이 가능함
- 그리고 CDC 에 유용함
    - **Q. 왜 CDC 에 유용하지? 저수준 로그가 아니라서 데이터만 파싱하기 편해서 그런가?**
        - ㅇㅇ 맞음

### 트리거 기반 복제

- 위 방식은 애플리케이션 코드 사용 없이 복제를 처리하는 방식이기 때문에 유연성이 떨어짐
    - e.g. 데이터의 서브셋만 복제, 다른 종류의 데이터베이스로 복제 등등
- 그래서 애플리케이션 단으로 복제를 올릴 필요가 있음
- 오라클의 골든게이트같은 도구가 DB 로그를 읽어 애플리케이션이 데이터를 변경 가능하게 함
    - **Q. 오라클 골든게이트가 먼가용**
        - CDC 도구임니다
        - 이외에도 Debezium 같은 도구가 있어요. Debezium 은 스프링 생태계에 녹아있어서 추천드려요
- 다른 방법으로는 트리거나 스토어드 프로시저가 있음
- 트리거
    - 사용자 정의 애플리케이션 코드를 등록할 수 있게 함
        - **Q. 어디에?**
            - DBMS 내부에 등록함
            - PostgreSQL 은 이런 식으로 등록함
            
            ```lua
            CREATE TRIGGER ...
            AFTER INSERT OR UPDATE OR DELETE
            ON table_name
            FOR EACH ROW
            EXECUTE FUNCTION function_name(); // 얘가 사용자 정의 코드
            ```
            
        - DBMS 에서 데이터가 변경되면(쓰기 트랜잭션) 애플리케이션 코드가 자동 실행됨
        - 데이터 변경을 분리된 테이블에 로깅할 수 있는 기회를 가짐. 그리고 외부 프로세스가 이 테이블에서 데이터 변경을 읽을 수 있음
            - 그러면 외부 프로세스는 필요한 애플리케이션 로직을 적용해 다른 시스템으로 변경을 복제함
            - **Q. 근데 DBMS 에 애플리케이션 코드를 두는데 왜 또 외부 프로세스에 애플리케이션 코드를 둬야함?**
                - 하는 일이 달라서 그럼
                - DB 트리거
                    - 데이터 변경 감지
                    - 변경 내용을 change_log 테이블에 기록
                - 외부 프로세스
                    - change_log 읽기
                    - 필터링
                    - 변환
                    - 재시도
                    - 다른 DB/Kafka/검색엔진으로 전송
                - 왜 DB 안에서 다 안 하냐면, DB 트리거 안에서 외부 API 호출, Kafka 전송, 재시도, 장애 복구 같은 걸 하면 DB 쓰기 성능과 안정성이 저하되기 때문
    - 다른 복제 방식에 비해 오버헤드가 많음
        - **Q. 왜?**
            - DB 쓰기 로직에 추가 작업을 끼워넣기 때문
                - 쓰기 마다 트리거 함수를 실행됨
                - 변경 로그 테이블에 추가 insert 발생
                - 등등 …
- **Q. 스토어드 프로시저?**
    - DB 안에 저장해두고 실행하는 함수/절차
    - **Q. 그럼 얘랑 트리거랑 머가 다름?**
        - 스토어드 프로시저는 명시적으로 호출해야 실행됨
        - 트리거는 이벤트 발생 시 자동으로 실행됨

## 복제 지연 문제

- 읽기 확장 아키텍처에서는 팔로워를 늘림으로써 읽기 부하를 분산할 수 있음
- 그러나 이건 비동기식 복제에서만 가능함
    - 동기식으로 모든 팔로워에 복제하는 건 지연도 길 뿐더러, 각 노드가 단일 장애 지점이 되기 때문 << 정리 맞는지 검토
- 그리고 일시적으로 리더-팔로워 간의 데이터 불일치가 생길 수 있고, 이는 시간이 지나면 해결됨. 이를 최종적 일관성이라고 함
- 여기서 ‘시간이 지나면’ 이라는 말이 모호함. 실제로 수 분이 걸릴 수 있기 때문. 따라서 복제 지연 시 발생할  수 있는 사례와 해결하는 방법을 소개함

### 자신이 쓴 내용 읽기

- 리더-팔로우 + 비동기식 복제 아키텍처에서는 쓰기는 리더, 읽기는 리더 or 팔로워에서 진행하기 때문에, 쓰기가 완료된 시점에 읽기는 불가능할 수 있음
- 이런 상황에서는 쓰기 후 읽기 일관성이 필요함
    - 이는 쓰기를 요청한 사용자가 페이지를 재로딩했을 때 항상 자신이 제출한 모든 갱신이 볼 수 있음을 보장해야한다는 것(쓰기를 요청하지 않은 사람에 대해서는 보장하지 않음)
- 이를 구현하는 방법 몇 가지가 잇음
    - 쓰기를 요청한 사용자가 자신이 수정한 내용을 읽을 때는 리더에서 읽고, 그 외는 팔로워에서 읽기
        - 이를 위해서는 실제로 질의하지 않고 무엇이 수정되었는지 알 수 있는 방법이 필요함
            - **Q. 이거 먼말임? 조회하기 전에 자기가 수정한 건지 알 수 있어야 한다는 거 맞음? 그래야 리더로 조회 요청 보낼지, 팔로워로 보낼지 결정 가능하니까?**
                - ㅇㅇ 맞음용
        - **Q. 단점은?**
            - 어떤 데이터가 사용자가 수정한 데이터인지 알아야 하기 때문에 라우팅 로직이 복잡해짐
            - 리더 읽기 비율이 늘어나서 리더 부하 증가
                - 특히 사용자가 많은 데이터를 수정하면 대부분을 리더로 읽게 됨
            - 여러 디바이스 사용 시 최근에 쓴 데이터 정보를 공유해야함
    - 사용자 데이터의 마지막 갱신 시각을 찾아서, 일정 시간 동안은 리더에서 조회하게 하는 방법
    - 클라이언트가 가장 최근 쓰기의 타임스탬프를 들고 있으면서, 팔로워가 최근 쓰기를 반영했는지 판단하여 조회 시점을 조절하는 방식
- 근데 이건 동일한 사용자가 여러 디바이스로 서비스에 접근할 때 문제가 발생함
    - 예를 들어, 클라이언트가 최근 쓰기 타임스탬프를 들고 있으면 동기화가 안돼서 디바이스 간 일관성이 깨질 수 있음. 그래서 이 메타데이터를 중앙에서 관리해야함
    - 팔로워가 여러 데이터센터에 분산되어 있을 수 가능성이 있음. 리더에서 읽어야 하는 정보가 있다면 사용자가 소유한 모든 디바이스에서의 요청을 동일한 데이터센터로 보내야함
        - **Q. 근데 왜 리더에서 읽어야한느 것만임? 리더가 속한 데이터센터로 보내야한다는 거?**
            - 여러 데이터센터가 있다면 리더가 속한 데이터센터의 팔로워가 최신 데이터를 따라잡았을 가능성이 크기 때문
            - 다른 데이터센터의 팔로워는 복제 지연이 발생할 수 있어서 그럼
            - 근데 자꾸 가정을 막 바꾸니까 이해하기 힘든 거 같네 이 책;; 앞에서는 나중에 설명한다더니

### 단조 읽기

- 비동기식 팔로워에서 읽을 때 시간이 거꾸로 흐르는 현상을 목격할 수 잇음
- 최신 복제 서버에서 먼저 읽고, 그 다음에 예전 복제 서버에서 읽는 경우에 발생 가능함
- 단조 읽기는 이런 현상이 발생하지 않음을 보장함
- **Q. 어떻게?**
    - 사용자의 읽기 요청은 항상 동일한 팔로워에서 수행되도록 하는 방법
        - 사용자의 ID 해시를 기반으로 동작하게 하면 됨
        - 근데 질의를 요청하던 팔로워가 죽으면? → 재라우팅 하는 과정이 필요함

### 일관된 순서로 읽기

- 이건 줄라탄 같은 거임
- 일부 파티션이 다른 것보다 느리게 복제되는 경우, 관찰자는 질문을 보기 전에 대답을 볼 수 있음
- 이 문제는 서로 다른 파티션은 독립적으로 동작해 쓰기의 전역 순서를 보장하지 않기 때문에 발생함
    - **Q. 근데 쓰기 순서가 왜 달라짐? WAL 로그 같은 건 순서가 일관적으로 넘어갈 텐데**
        - 같은 파티션 안에서는 순서 일관적인 거 맞음
        - 근데 지금 여기서는 서로 다른 파티션에 쓰는 걸 가정하고 있음
        - 서로 다른 파티션에는 로그가 따로 있기 때문에 로그 간 전역 순서가 없다는 거임
- 해결책으로는 서로 인과성이 있는 쓰기가 동일한 파티션에 기록되게 하는 방법임
    - **Q. 어떻게?**
        - 예를 들어 댓글이라면, postId 를 파티션 키로 두고 같은 게시글의 질문, 답변, 댓글이 같은 파티션 안에 들어가게 해야함

### 복제 지연을 위한 해결책

- 복제를 비동기식으로 유지하면서, 동기식으로 동작하게 하는 척하는 게 해결 방법임
    - **Q. 정확히 무슨 말인가?**
        - 비동기식 복제의 단점인 복제 지연을 숨긴다는 거임
        - 위에서 살펴본 방법으로 가능함
            - 방금 쓰기를 요청한 사용자는 리더에서 읽게 하거나, 같은 사용자는 같은 팔로워에게 읽게 하거나 등등
- 트랜잭션
    - 애플리케이션이 더 단순해지기 위해 데이터베이스가 강력한 보장을 제공하는 방식
    - 개발자가 애플리케이션 단에서 코드로 문제를 해결하지 않고 데이터베이스를 신뢰할 수 있다면 작업이 간편해짐
    - 단일 노드 트랜잭션은 사용했으나, 복제 데이터베이스로 전환하는 과정에서 트랜잭션을 포기함
        - 왜냐면 성능, 가용성 측면에서 비싸기 때문
            - **Q. 왜여?**
                - 여러 노드가 같은 트랜잭션 결과에 동의해야하기 때문입니다
                - 네트워크 왕복 증가, 느린 노드의 전체 트랜잭션 지연 유발, 일부 노드 장애 시 커밋 불가, lock 유지 시간 길어짐 등의 비용이 추가로 발생해서 잘 안 써용

## 다중 리더 복제

- 단일 리더 복제 방식에서는 리더가 죽으면 쓰기 자체가 블로킹된다는 단점이 있음
- 그래서 리더를 여러 개 두는 방식을 고려함
    - 이 방식에서는 각 리더가 동시에 다른 리더의 팔로워 역할도 함

### 다중 리더 복제의 사용 사례

- 단일 데이터센터 내에 다중 리더 설정을 하는 것은 복잡도에 비해 이점이 크지 않음
    - **Q. 왜?**
        - 네트워크 지연이 낮기 때문. 그래서 굳이 여러 리더를 둬서 충돌 리스크 감수할 이유가 없음
        - **Q. 만일 쓰기 부하를 낮추려는 목적이라면?**
            - 그래도 합리적이지는 않음. 이 경우에는 샤딩을 우선적으로 고려함. 다중 리더 설정하는 건 충돌 해결, 동시성, 중복 등의 문제 해결 비용이 추가로 발생하기 때문
            - 샤딩을 도입하고 각 샤드에 단일 리더를 두는 게 일반적인 해결책임
- 다중 리더 복제가 합리적인 상황은 아래와 같음
    
    #### 다중 데이터센터 운영
    
    - 여러 데이터센터에 팔로워가 있다고 가정
    - 리더가 하나 있는 경우에는 모든 쓰기 요청이 리더가 속한 데이터센터를 거쳐야 함
        - 이는 네트워크 지연 문제가 발생할 수 있는 구조임
    - 그러나 다중 리더 설정에서는 각 데이터센터마다 리더가 있을 수 있음
        - 각 데이터센터 내에서는 리더-팔로워 복제가 이뤄지고, 리더 간에도 변경 사항 복제가 이뤄짐
        - 리더가 지리적으로 가까이 위치할 수 있기 때문에 쓰기 지연 시간을 감소시킬 수 있음
        - 리더가 죽어도 다른 데이터센터의 리더를 사용하면 되기 때문에 안정성에도 강함
            - **Q. 그러면 이 경우에는 쓰기 지연 감수해야겟네**
                - ㅇㅇ 마자용
    - 일부 데이터베이스는 다중 리더 설정을 제공하나, MySQL 의 Tungsten Replicator, PostgreSQL BDR 처럼 외부에서 구현한 도구를 사용하기도 함
    - 이 방식의 문제는, 동일한 데이터를 다른 두 개의 데이터센터에서 동시에 변경할 수 있다는 거임
        - 해결 방법은 아래 쓰기 충돌 챕터에서 기술
    - 동시성 문제나 중복 문제가 발생할 수 있으므로 도입에 신중해야함
        - e.g. 자동 증가 키, 트리거, 무결성 제약
            - **Q. 트리거가 왜?**
                - 왜냐면 트리거는 데이터 변경 시에 자동으로 실행되기 때문에, 원래 쓰기에서 실행되고, 복제 적용 중에 또 실행될 수 있기 때문에 문제가 돼용
    
    #### 오프라인 작업을 하는 클라이언트
    
    - 여러 디바이스에서 오프라인으로 동작하는 서비스가 있다고 가정
    - 이 경우, 각 디바이스에 로컬 데이터베이스가 있다고 생각하고, 복제는 디바이스가 온라인 상태가 되었을 때 다중 리더 복제 방식으로 수행된다고 생각하면 됨(동기화)
    - 이런 종류의 다중 리더 설정을 쉽게 하기 위한 도구로는 카우치DB 가 있음
        - **Q. 카우치DB 란? 얘가 왜 쉬움?**
            - CouchDB 는 문서 지향 NoSQL 임
            - 특정
                - 각 문서에 revision 정보가 있음
                - 오프라인 로컬 DB 사용 가능
                - 나중에 서버와 동기화 가능
                - 충돌 버전을 보관하고 애플리케이션이 해결 가능
            - 각 클라이언트가 리더처럼 동작하기 때문에 위 문제에 잘 맞음
    
    #### 협업 편집
    
    - 동시에 여러 사람이 문서를 편집할 수 있는 애플리케이션이 잇다고 가정
    - 이는 위에서 살펴본 오프라인 작업을 하는 클라이언트 문제와 유사함
        - 변경을 로컬에 저장하고, 비동기로 복제시킴
    - 편집 충돌이 없음을 보장하기 위해서는 편집 전에 락을 획득해야함
        - (협업 방식에 따라 다르긴 하겠지만, 여기서는 편집 전에 락을 획득해야한다고 가정)
        - 다른 사용자가 같은 문서를 편집하려면 이전 사용자의 변경이 커밋되고 락이 풀릴 때까지 기다려야함
        - 이는 리더에서 트랜잭션을 사용하는 단일 리더 복제와 동일
    - 더 빠른 협업을 위해서 변경 단위를 작게 해서 락을 피할 수 있긴 함
        - **Q. 어떻게 가능한 거임? 락 회피?**
            - 문서 전체에 락을 거는 게 아니라, 텍스트 수정 단위로 다루면 됨
            - 각 사용자의 텍스트 변경을 작은 operation 으로 만들고, 이를 병합하는 방식으로 처리
        - 근데 이러면 충돌 회피 등 여러 문제가 발생하긴 함

## 쓰기 충돌 다루기

### 동기 vs 비동기 충돌 감지

- 단일 리더 데이터베이스에서는 쓰기 시점 조율이 가능함
    - 첫 번째 쓰기가 완료될 때까지 두 번째 쓰기를 블로킹 한다거나 등
- 반면 다중 리더 설정에서는 쓰기 시점 조율이 불가능함
    - 각 쓰기는 모두 성공하며, 충돌은 비동기로 감지됨
    - 이 경우 사용자에게 충돌 해결을 요구하는 건 너무 늦는 걸 수 있음
- 근데 이론적으로 충돌 감지는 동기식으로 만들 수 있음
    - 사용자가 쓰기 요청을 하면, 완료 응답을 내리기 전에 모든 복제 서버가 쓰기를 복제하기를 기다림
        - **Q. 이게 동기식이랑 머가 다른 거임?**
            - 거의 같음 ㅇㅇ. 실제로 운영 비용 차이 거의 없음
    - 근데 이러면 다중 리더 복제 방식의 장점이 사라짐
        - 쓰기를 여러 리더에서 동시에 처리할 수 있는 게 장점인데, 이게 불가능해지니까

### 충돌 회피

- 동일한 사용자가 항상 동일한 데이터센터의 리더를 통해 읽기/쓰기를 진행하게 한다면, 해당 사용자 입장에서는 충돌이 발생하지 않음
- 그러나 읽기/쓰기 요청을 담당하던 데이터센터가 바뀌면 충돌 회피가 실패함
- 따라서 여러 리더를 통해 기록할 때에도 충돌을 회피할 수 있는 방법이 필요함

### 일관된 상태 수렴

- 단일 리더 데이터베이스에서는 순차적으로 쓰기가 적용되나, 다중 리더 설정에서는 쓰기 순서가 정해지지 않음
    - 동시에 수정이 될 수 있기 때문에 어느 것이 먼저 커밋되는지에 따라 달라짐
- 즉, 단순히 각 리더가 쓰기를 본 순서대로 적용한다면 데이터베이스는 일관성을 보장할 수 없음
    - 그래서 수렴 방식으로 충돌을 해소해야함
    - 모든 변경이 복제되어 모든 서버에 동일한 값이 전달되게 해야한다는 것
- 어떻게 하나?
    - 각 쓰기에 고유 ID 를 부여하고, 가장 높은 ID 를 고르는 방식
        - 이걸 최종 쓰기 승리라고 함
        - 근데 이 방식은 데이터 유실 가능성 있음
    - 각 서버에 고유 ID 를 부여하고 무조건 높은 ID 를 가진 서버의 데이터를 신뢰하는 방식
        - 이 또한 데이터 유실 가능성 있음
    - 병합 방식
        - 이건 유실 가능성 없음
        - 충돌을 기록하고, 수동으로 충돌을 해소하게 하기

### 사용자 정의 충돌 해소 로직

- 방금 위에서 설명한 수동 충돌 해소 방식임
- 대부분의 다중 리더 복제 도구에서 애플리케이션 코드 사용해 충돌 해소 로직을 작성함
- 이 코드는 읽기/쓰기 중에도 실행될 수 있음
    - 쓰기 수행 중일 때
        - 복제 중 충돌이 감지되면 충돌 핸들러 호출하고, 백그라운드 프로세스에서 해소 로직이 동작
    - 읽기 수행 중일 때
        - 충돌 감지 시, 모든 충돌 쓰기를 저장하고, 다음 조회 시에 여러 버전의 데이터 중에서 하나를 선택하는 방식
        - **Q. 어떤식으로 저장되나?**
            
            ```lua
            {
              "key": "cart:user1",
              "siblings": [
                {
                  "value": ["apple", "banana"],
                  "version": {"A": 2, "B": 0}
                },
                {
                  "value": ["apple", "orange"],
                  "version": {"A": 1, "B": 1}
                }
              ]
            }
            ```
            
        - 이거는 워드 프로그램에서 비정상적 종료되었을 때 버전 보여주고 선택하는 거랑 비슷
- 충돌 해소는 개별 로우나 문서 수준에서 적용됨. 그래서 원자적으로 다른 쓰기를 수행하는 트랜잭션이라면 각 쓰기는 충돌 해소를 위해 여전히 별도로 간주됨
    - **Q. 왜 별도로 간주됨?**
        - 왜냐면 트랜잭션에서 여러 row 가 변경될 수 있기 때문임
        - 이걸 row 단위로 처리되면 트랜잭션 보장이 안될 수 잇어서 위험함

### 자동 충돌 해소

- 동시에 데이터를 수정할 때 발생하는 충돌을 자동으로 해소하기 위한 방법
- 충돌 없는 복제 데이터타입
    - set, map 등의 데이터 구조를 사용해 동시 편집을 허용하고, 합리적인 방법으로 충돌을 해소하는 방법
        - **Q. 여기서 말하는 합리적인 방법이란?**
            - set에 동시에 원소 추가 → 둘 다 남김
            - counter에 동시에 +1 → 합산
            - map에 서로 다른 key 수정 → 둘 다 반영
            - 삭제와 추가가 충돌하면 미리 정한 규칙으로 처리
    - 이중 병합을 사용함
        - **Q. 이중 병합?**
            - 두 버전만 비교해서 합치는 방식
            - 문제는, 두 버전의 공통된 조상이 없다면 원래 있던 값이 삭제된 것인지, 아니면 원래 없던 값인 건지 구분할 수 없다는 거
                
                ```lua
                // 예를 들어 이렇게 변경이 되었으면, left 가 C, D 를 추가한 건지
                // 아니면 right 가 C, D 를 삭제한 건지알 수 없음
                left:  {A, B, C, D}
                right: {A, B}
                ```
                
            - 그래서 삼중 병합을 사용함
- 병합 가능한 영속 데이터 구조
    - git 과 유사한 히스토리 추적 방식
    - 삼중 병합 함수 사용함
        - **Q. 삼중 병합?**
            - 이건 이중 병합에서 조상도 함께 보는 방식
            - 이중 병합에 비해 정확함
- 운영 반환
    - 구글 독스같은 협업 편집 애플리케이션의 충돌 해소 알고리즘
        - 텍스트 문서나 정렬된 항목 목록의 동시 편집을 위해 설계됨
            - **Q. 어떻게 동작하는데?**
                - operation 을 그대로 적용하지 않고, 동시에 발생한 다른 operation 을 고려해서 반환함
                - 예를 들어, 문자열 abc 가 있을 때 아래 두 요청을 한다고 가정
                    - 명령1 : [1] 위치에 A 삽입, 의도 : a 다음 위치에 A 삽입
                    - 명령2 : [2] 위치에 B 삽입, 의도 : b 다음 위치에 B 삽입
                - 이 두 작업이 동시에 발생하면 위치가 충돌함
                - 그래서 의도를 파악해서 명령2 를 [2] 위치가 아니라, [3] 위치에 삽입하게 함

## 다중 리더 복제 토폴로지

- 복제 토폴로지는 쓰기를 한 노드에서 다른 노드로 전달하는 통신 경로를 설명함
    - 쓰기가 전파되는 방향을 설명하는 것
- 전체 연결
    - 가장 일반적인 토폴로지
    - 모든 리더가 각자의 쓰기를 다른 모든 리더에게 전송
    - 근데 이 경우에는 메시지가 도착하는 속도가 달라 일관성이 깨질 수 있음
        - e.g. 리더1 : B 추가 → 리더2: B를 수정 한 경우, 리더3 에 리더2 메시지가 먼저 도착하여, 없는 필드인 B 를 수정하려고 하는 문제가 발생할 수 있음
    - 이 문제를 해결하기 위해서는 버전 벡터라는 기법을 사용할 수 잇음
        - 이건 아래 버전 벡터 챕터에서 자세히 서술
- 원형 토폴로지
    - 각 노드가 하나의 노드로부터 쓰기를 받고, 받은 거 + 자기 쓰기를 다른 한 노드에게로 전송하는 방식
    - 그러나 특정 노드에 장애 발생하면 다른 노드들로 복제 전달이 안된다는 단점이 있음
        - 그래서 하나의 노드에서 여러 노드로 퍼지는 방식이 내결함성이 더 좋긴 함
    - MySQL 이 이거 씀
        - **Q. 내부적으로 스레드가 도나?**
            - ㅇㅇ. DB 마다 명칭은 다른데,  MySQL 복제에서는 팔로워가 리더의 binlog 를 읽는 I/O thread 와 relay log 를 적용하는 SQL thread/worker 를 사용함
- 별 모양 토폴로지
    - 지정된 루트 노드 하나가 다른 모든 노드에게 쓰기를 전달
        - **Q. 그럼 각 노드에서 발생한 쓰기는 어떻게 전파되나?**
            - 각 노드의 쓰기가 루트로 전파되고, 루트가 다른 리더들에게 배포하는 방식임
            - 루트가 중계자 역할을 하는 거라고 생각하면 됨
        - **Q 그리고 루트 노드가 죽는다면? 그럼 이건 다중 리더 방식의 장점을 무마시키게 될 텐데?**
            - 맞음 ㅠㅠ
            - 그래서 루트를 이중화하거나 선출하는 방식 등을 함께 사용해야함

## 리더 없는 복제

- 다이나모 스타일
    - 리더 개념 없이 모든 복제 서버가 클라이언트의 쓰기를 처리할 수 있게 하는 방식
    - 리악, 카산드라, 볼드모트에서 사용됨
- 리더 없는 복제 구현에서는 클라이언트가 여러 복제 서버에 쓰기를 직접 전송함
    - 코디네이터 노드가 이를 대신하기도 함
    - 그러나 코디네이터는 특정 순서로 쓰기를 처리하지 않음
        - **Q. 그럼 일관성 보장을 어케함..? ㅇㅅㅇ**
            - 강한 일관성 보장을 하는 게 아니라 최종적 일관성을 달성하는 걸 목표로 함

### 노드가 다운됐을 때 데이터베이스에 쓰기

- 리더가 없는 경우에는 복제 서버 중 하나의 서버가 죽어도 장애 복구가 필요하지 않음
- 예를 들어, 클라이언트가 복제 서버들에게 병렬로 쓰기 요청을 보낸 경우에는 일정 개수 이상의 서버만 OK 응답을 주면 쓰기가 성공했다고 간주함
- 그러나 이후 클라이언트가 조회를 요청할 때, 복구된 서버(앞서 죽어서 쓰기 요청을 받지 못했던 서버)에서 데이터를 받는다면 outdated 된 데이터일 수밖에 없음
    - 이를 해결하기 위해서는 읽기 요청도 여러 서버에게 병렬로 보내고, 최신 버전의 데이터를 신뢰하게 해야함

### 읽기 복구와 안티 엔트로피

- 죽었던 노드가 복구되었다면, 그 사이에 발생한 쓰기 요청을 어떻게 따라잡아야할까?
- 읽기 복구
    - 클라이언트가 여러 노드에게 병렬로 읽기 요청을 보내면, 어떤 노드가 outdated 된 데이터를 가지고 있는지 확인할 수 있음. 이 때 outdated 된 데이터를 가진 노드를 업데이트 하면 됨
    - 캐시 전략이랑 비슷함
- 안티 엔트로피 처리
    - 백그라운드 프로세스를 두고 노드 간 데이터 차이를 찾아 지속적으로 업데이트하는 방식
    - 특정 순서로 쓰기를 복사하지 않기 때문에 데이터가 업데이트 되기까지 지연이 있을 수 있음
        - **Q. 그럼 어떻게 쓰기 복사함? 리더 기반 복제랑 다르다는데, 리더 기반 복제는 쓰기 로그를 읽어서 반영하는 방식인데, 얘는 그렇게 안함?**
            - ㅇㅇ 좀 다름
            - 리더 기반은 리더의 로그를 팔로워가 순서대로 따라가는 방식
            - 여기서는 특정 리더 로그를 따라가는 게 아니라, 노드끼리 데이터를 비교해서 빠진 값을 채우는 거임

### 읽기와 쓰기를 위한 정족수

- 앞서 리더 없는 경우에는, 쓰기 요청 시 일정 개수 이상의 노드만 OK 하면 쓰기가 처리되었다고 간주한다 했음
    - 여기서 일정 개수의 노드를 어떻게 결정할 수 있을까?
- n : 복제 서버 수, w : 쓰기 성공해야 OK 하는 노드의 수, r : 질의 요청을 보낼 최소 노드 개수
    - 이 경우, w + r > n 이면 읽을 때 최신 값을 얻을 수 있다고 기대함
    - 이런 r 과 w 를 따르는 읽기와 쓰기를 정족수 읽기와 쓰기라고 함
- 다이나모 스타일에서는 n 을 홀수로 하고, w = r = (n + 1) / 2 로 설정함
- 근데 쓰기가 적고 읽기가 많을 때에는 w = n, r = 1 로 설정해도 좋음
    - 이 경우 하나의 노드로만 읽기 요청을 보내기 때문에 읽기가 빨라짐
    - 그러나 노드 하나가 고장나면 모든 데이터베이스 쓰기가 실패함
        - w = n 이기 때문

### 정족수 일관성의 한계

- w, r 로 오래된 값을 읽을 확률을 조정할 수 있지만, 완전히 보장하지는 못함
- w + r > n
    - 이 경우에는 읽기를 하는 노드와 쓰기를 하는 노드가 겹치기 때문에 모든 읽기는 키의 최신 값을 반환할 것으로 생각함
        - **Q. 맞음 이거?**
            - 이론적으로는 가능하나, 항상 보장하는 건 아님
            - 네트워크 지연, 동시 읽기/쓰기, 느슨한 정족수 등으로 최신 값 반환이 되지 않을 수도 있음
    - 근데 이 방식도 엣지 케이스가 존재함
        - 느슨한 정족수를 사용하는 경우 outdated 된 값을 반환할 가능성이 있음
            - 느슨한 정족수는 아래 챕터에서 기술
        - 충돌 발생 시의 해결 전략이 데이터 유실 가능성이 있는 전략일 경우 일관성이 유지되지 않을 가능성이 있음
            - e.g. 최종 쓰기 승리 등
        - 쓰기와 읽기가 동시에 발생하는 경우, 일부는 쓰기 반영이 되지 않은 데이터를 반환할 수 있음
        - w 를 채우지 못한 경우, 이미 쓰기가 완료된 노드에서는 롤백을 진행하지 않음. 그래서 일관성이 유지되지 않음
        - w 에 속한 노드가 죽은 경우, w 의 의미가 깨져 정족수 조건을 충족할 수 없음
- r, w > n/2
    - 보통 선택하는 값
    - n/2 노드 장애까지 허용해도 w + r > n 이 보장되기 때문
- w, r 이 작을 수록 outdated 된 값을 읽을 확률이 높아짐
- 견고한 보장이 필요하다면 트랜잭션이나 합의가 필요함

### 최신성 모니터링

- 운영 관점에서 DB 가 최신 결과를 반영하는지 모니터링하는 것은 중요함
- 복제 지연이 발생한다면 이를 조사할 수 있어야 함
- 리더 기반 복제에서는 복제 지연에 대한 지표를 볼 수 있음
    - 이는 각 노드가 복제 로그의 수를 가지기 때문에 가능함
    - 리더의 로그 개수 - 팔로워의 로그 개수로 지연율 구할 수 있음
- 반면 리더 없는 복제 시스템에서는 쓰기 순서를 고정할 수가 없음
    - 그래서 모니터링이 어려움
    - 최종적 일관성의 ‘최종적’ 이라는 것을 정량화할 수 있어야 함

## 느슨한 정족수와 암시된 핸드오프

- 노드가 n 개 이상인 대규모 클러스터에서, 클라이언트는 정족수 구성에 들어가지 않는 노드에 연결될 수 있음
    - **Q. 이거 먼말임?**
        - 원래 노드 중 일부가 죽은 경우, 클러스터에는 존재하지만 n 에는 속하지 않았던 노드에 임시로 기록한다는 거임
        - 느슨한 정족수는 원래 복제 대상 n개 중 w개에 쓰는 것이 아니라, 접근 가능한 임의의 w개 노드에 쓰기를 받아들이는 방식이라고 생각하면 됨
- 이 경우, 트레이드 오프가 발생함
    - w 나 r 노드 정족수를 만족하지 않는 모든 요청에 오류를 반환해야하나?
    - 일단 쓰기를 받아들이고, 연결 가능한 노드에만 기록할까?
        - 이 경우를 느슨한 정족수라고 함
- 네트워크 장애가 해결되면, 한 노드가 다른 노드를 위해 장애 발생 동안의 모든 쓰기를 해당 홈 노드로 전송함
    - 이 방식을 암시된 핸드오프라고 함
        - **Q. 홈 노드는 머임?**
            - 홈 노드는 원래 n 에 속하는 노드임
            - 장애 중 임시 노드가 받은 쓰기를 홈 노드에게 돌려주는 걸 암시된 핸드오프라고 하는 것
- 느슨한 정족수는 쓰기 가용성을 높이는 것에 유용함
    - w 를 충족하지 못해도 쓰기가 되기 때문
    - 다만, w 에 속하지 못한 노드에 기록될 가능성이 있어서, w + r > n 일 때 최신 값을 읽는다고 보장할 수 없음
        - 왜냐면 최신 값이 n 이외의 노드에 기록될 수 있기 때문
- 따라서 느슨한 정족수는 전통적인 정족수가 아님
    - 데이터가 w 어딘가에는 저장된다는 뜻임
    - 암시된 핸드오프가 완료될 때까지는 r 노드의 읽기가 최신 데이터를 본다는 보장을 할 수 없음

## 동시 쓰기 감지

- 다이나모 스타일(리더 없는 복제) 데이터베이스는 여러 클라이언트가 동시에 같은 키에 쓰는 게 허용됨
- 따라서 엄격한 정족수를 사용하더라도 충돌이 발생함
- 문제는 네트워크 문제로 각 노드가 받는 이벤트 순서가 모두 다를 수 있다는 거임
    - 단순히 받는 순서대로 기록한다면 일관성이 깨짐
    - 최종적 일관성을 달성하기 위해서는 데이터베이스 내부에서 충돌을 다루는 방식을 알아야 함

### 최종 쓰기 승리(LWW, 동시 쓰기 버리기)

- 각 복제본이 가진 예전 값을 버리고 최신 값을 택하는 방식
- 예를 들어 쓰기에 타임스탬프를 붙여두고, 가장 최신의 타임스탬프 값을 가진 쓰기를 신뢰함
- LWW 는 최종적 수렴 달성은 가능하지만 지속성은 희생함
    - 즉, 모든 사용자의 쓰기는 성공으로 처리되지만, 결국 하나의 쓰기만 남음
- 그리고 동시 쓰기가 아니라도 쓰기가 삭제될 수 있음
    - **Q. 어떤 경우일까?**
        - 먼저, 여기서 말하는 삭제는 성공한 쓰기 결과가 최종 상태에서 사라지는 걸 말함
        - 예를 들어 타임스탬프를 사용하는 경우, 더 작은 값을 가진 쓰기가 삭제됨
- 손실 데이터를 허용하지 않는다면 부적절함
- LWW 로 데이터베이스를 안전하게 사용하려면 키를 한 번만 쓰고 불변 값으로 다뤄야함
    - 예를 들어, 카산드라는 키를 UUID 로 구성해둠

### 이전 발생 관계와 동시성

- 두 작업이 동시에 수행됨을 어떻게 알 수 잇나?
- 이전 발생
    - 두 작업 사이에 인과성이 존재하는 경우
    - 예를 들어, A 가 먼저 작성되고, 그 다음에 A 가 B 로 수정되는 경우 두 작업에 인과성이 존재한다고 함
    - 이 경우에는 나중 작업으로 이전 작업을 덮어쓰면 됨
- 동시 작업
    - 두 작업이 서로 연관이 없는 경우
    - 이 경우에는 충돌을 해소해야함

### 이전 발생 관계 파악하기 ??

- 동작 방식
    1. 서버는 모든 키에 대한 버전 번호를 유지하고, 키를 기록할 때마다 번호를 증가시킴
    2. 기록한 값은 새로운 버전 번호를 가지고 기록됨
    3. 클라이언트는 쓰기 전에 키를 읽어야 함
    4. 클라이언트는 키를 기록할 때 이전 읽기의 버전 번호를 포함하고, 이전 읽기에서 받은 모든 값을 합쳐야함
    5. 서버는 특정 버전 번호를 가진 쓰기를 받을 때 해당 버전 이하 모든 값을 덮어씀

### 동시에 쓴 값 병합

- 데이터 유실이 되지 않음을 보장하나, 클라이언트가 추가적인 작업을 수행해야함
    - 여러 작업이 동시에 발생하면 클라리언트가 조율해야하기 때문
    - 이런 동시 값을 형제 값이라고 함
- 두 형제 값을 다룰 때 단순히 두 값을 합치는 것으로는 부족함
    - 왜냐면 값을 삭제하는 게 반영이 되지 않기 때문
    - 예를 들어, A, B, C 라는 데이터 셋이 있는데, 형제1은 A, B, C, D 로 수정하고, 형제2는 A, B 로 수정한 경우, 두 집합을 합치면 형제2에서 삭제한 C 가 살아남
- 따라서 형제 병합 시 데이터가 제거되었음을 나타내기 위해 해당 버전 번호에 표시를 해야함
    - 이를 툼스톤이라고 함
    - **Q. 예시?**
        - 일반적인 병합
        
        ```lua
        사용자1: D 추가 → {A, B, C, D}
        사용자2: C 삭제 → {A, B}
        결과: {A, B, C, D} // C 삭제가 사라짐
        ```
        
        - 그래서 이렇게 툼스톤을 추가함 `C deleted at version 5`
        - 그럼 병합 결과를 {A, B, D} 이렇게 만들 수 있음
- 리악에서는 형제를 자동 병합할 수 있는 CRDT 라는 데이터 구조군을 사용함
    - **Q. CRDT?**
        - Conflict-free Replicated Data Type
        - 여러 노드에서 동시에 수정해도, 정해진 병합 규칙을 따르면 모든 복제본이 같은 값으로 수렴하는 데이터 타입

### 버전 벡터

- 다중 복제본의 동시 쓰기를 받아들일 때에는 각 key마다 버전 정보를 관리해야 하고, 그 버전 정보는 복제본별 카운터의 묶음으로 표현될 수 있음
    - **Q. 왜 키별 번호와 복제본별 버전 번호가 둘 다 필요함?**
        - 키별 번호는 데이터마다 변경 이력이 다르기 때문에 필요함
        - 근데 리더 없는 복제나 다중 리더 복제에서는 여러 서버에서 동시에 같은 키 수정이 가능함
        - 그래서 어느 복제본에서 몇 번째 변경까지 반영했는지 알아야 하므로 아래처럼 관리함
        
        ```lua
        "cart:user1": {
          "version": {
            "A": 3,
            "B": 2,
            "C": 0
          }
        }
        ```
        
    - 각 복제본은 쓰기를 처리할 때 자체 버전 번호를 증가시키고, 각기 다른 복제본의 버전 번호도 계속 추적하면서 덮어쓸 값과 형제로 유지할 값을 나타내야함
    - **Q. 왜요?**
        - 동시 쓰기인지, 덮어써도 되는 쓰기인지 구분하기 위함임
- 모든 복제본의 버전 번호 모음을 버전 벡터라고 함
    - 도디트 버전 벡터가 그 예시임
        - **Q. 이거 머임?**
            - 버전 벡터를 더 정교하게 만든 방식임
            - 개별 쓰기 하나를 정확히 표시하고 싶을 때 사용함
            
            ```lua
            {
              "context": { "A": 3, "B": 2 },
              "dot": ["A", 4]
            }
            
            의미 : 이 값은 A의 3번까지, B의 2번까지 알고 있던 상태에서 A가 새로 만든 4번 변경임
            ```
            
- 버전 벡터는 값을 읽을 때 클라이언트로 보내고, 이후 값이 기록될 때 클라이언트가 데이터베이스로 보내야함
- 버전 벡터는 하나의 복제본을 읽은 다음에 이어 다른 복제본에 쓰는 작업이 안전함을 보장함
    - **Q. 이게 무슨 말인가?**
        - A 복제본에서 읽은 값의 버전 정보를 B 복제본에 쓸 때 함께 보낸다는 것
        - 읽은 복제본과 쓴 복제본이 달라도 인과관계를 잃지 않게 해줌
    - 이렇게 하면 형제가 생성돼도 형제가 올바르게 병합되면 데이터 손실이 없음
