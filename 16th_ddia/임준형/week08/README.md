## 프로세스 중단 (Process Pauses)

- 상황 설정: 파티션마다 리더가 하나씩 있고, 리더만 쓰기를 받는 DB임
    - 근데 어떤 노드가 "내가 지금도 리더 맞나?"를 어떻게 확신함?
- 흔한 방법이 **임차권(lease)** 임. 타임아웃 달린 잠금이라고 보면 됨
    - "너 30초 동안 리더 해, 대신 만료 전에 계속 갱신해" 이런 식
    - 특정 시점엔 딱 하나의 노드만 임차권을 쥘 수 있음
- 근데 이 임차권 코드에 두 가지 문제가 있음

**문제 1 — 동기화된 시계에 의존함**
- 임차권 만료 시각은 임차권 발급해준 다른 장비가 자기 시계로 찍은 값임
- 근데 `System.currentTimeMillis()`는 내 시계라서, 둘이 몇 초만 어긋나도 "아직 유효한데 만료됐다고 착각"하거나 그 반대가 됨
- 8장 전반의 "벽시계 못 믿는다"가 여기서 그대로 터지는 거임

**문제 2 — 단조 시계로 고쳐도 여전히 위험함 (진짜 핵심)**
- 이 코드는 은연중에 "isValid()로 확인한 시점"이랑 "process(request) 실행 시점"이 거의 붙어있다고 가정함
- 보통은 코드가 빛처럼 빨라서 10초 버퍼면 차고 넘침
- 근데 isValid() 통과한 직후에 스레드가 15초 멈춰버리면?
    - 깨어난 시점엔 이미 임차권 만료 + 다른 노드가 리더임
    - **근데 멈췄던 스레드는 자기가 멈췄다는 걸 모름** → 깨어나면 아무 일 없던 듯 다음 줄(process)을 그냥 이어서 실행함
    - "어 나 15초 잤네?"를 인지하는 장치가 없음 → 만료된 임차권으로 쓰기 저지른 뒤에야(다음 루프에서) 깨달음

**스레드가 그렇게 오래 멈춤? → 원인 수두룩함**
- **stop-the-world GC**: 가장 유명한 범인. 돌면 모든 앱 스레드 멈춤, 때로 몇 분씩 감. CMS 같은 동시적 GC도 일부 구간은 결국 STW 필요함
- **VM 서스펜드 / 라이브 마이그레이션**: 클라우드에서 VM 통째로 멈췄다 다른 호스트에서 재개. 게스트 입장에선 시간이 그냥 점프함
- **컨텍스트 스위치 / steal time**: 다른 VM이 CPU 가져감
- **동기 디스크 I/O**: 자바 클래스 lazy 로딩처럼 코드에 안 보이는 것도 포함
- **스왑/페이징 → 스래싱**, **SIGSTOP**(누가 Ctrl+Z 눌러버림) 등등

**왜 분산 시스템에서 특히 골치냐면**
- 사실 이건 단일 장비 멀티스레드 안전성 문제랑 똑같음. 단일 장비도 OS 스케줄러가 임의로 스레드 멈추고 컨텍스트 스위치함
- 근데 거긴 뮤텍스, 세마포어, 원자적 카운터, 블로킹 큐 같은 무기가 있음
- 문제는 이 무기들을 분산 시스템엔 못 가져옴 → 공유 메모리 없고, 못 믿을 네트워크로 메시지만 주고받으니까

> **결론**: 노드는 언제든, 심지어 함수 한복판에서도 한참 멈출 수 있다고 가정해야 함. 그리고 멈췄다 깨어나도 본인은 그걸 모름. (해결책이 뒤의 펜싱 토큰임)

## 응답 시간 보장

- "중단이 문제면 중단을 아예 없애면 되잖아?" → 노력하면 가능함. 그게 **엄격한 실시간 시스템(hard real-time)**임
    - 데드라인 못 지키면 시스템 전체가 고장 나는 환경 — 항공기, 로켓, 에어백 같은 거
    - 센서가 충돌 감지했는데 하필 GC 돌아서 에어백 늦게 터지면 사람 죽음
- 보장하려면 소프트웨어 스택 전 계층의 지원이 필요함
    - **실시간 OS(RTOS)**가 "이 프로세스는 매 주기마다 CPU 시간 보장받음"으로 스케줄링
    - 라이브러리가 최악 실행 시간을 문서화
    - 동적 메모리 할당 제한/금지 (할당 시간 들쭉날쭉 + 실패 가능 + 회수 비용 → 예측 불가능해짐)
    - 엄청난 양의 테스트로 보장 확인
- 이게 다 비싸서 안전이 목숨인 임베디드에서나 씀
- 오해 금지: **"실시간"은 "고성능"이 아님**. 오히려 "제때 응답" 최우선이라 처리량은 더 낮을 수도 있음
- 결론: 일반 서버 데이터 처리엔 비경제적이라, 그냥 중단이랑 시계 불안정 안고 살기로 함

## 가비지 컬렉션의 영향을 제한하기

- 실시간까진 못 가도 GC 충격을 줄이는 실용적 방법 있음
- 근거: 언어 런타임은 객체 할당률 + 남은 메모리를 추적함 → "곧 GC 필요"를 어느 정도 예측 가능함
- **전략 1 (GC를 계획된 휴식으로 다루기)**
    - 런타임이 "나 곧 GC 해야 함" 경고하면 → 그 노드로 새 요청 보내는 거 멈춤 → 처리 중이던 요청만 마저 끝냄 → 한가할 때 GC 돌림
    - GC 중단이 클라이언트한테 안 보여서 **응답 시간 상위 백분위(p99 같은 꼬리 지연)**가 좋아짐
    - 지연에 민감한 금융 거래 시스템이 실제로 이렇게 함
- **전략 2 (수명 짧은 객체만 GC + 주기적 재시작)**
    - 빨리 끝나는 young GC만 돌리고, 무거운 full GC 필요해지기 전에 프로세스를 아예 재시작함
    - 한 번에 노드 하나씩, 재시작 전에 트래픽 다른 노드로 옮기면 무중단 가능(순회식 재시작)
- 완전히 없애진 못해도 받는 충격은 의미 있게 줄임

## 지식, 진실, 그리고 거짓말

- 지금까지 결론: 네트워크도 못 믿고, 시계도 못 믿고, 내 프로세스의 시간 감각도 못 믿음
    - 노드는 받은(혹은 못 받은) 메시지로 추측만 할 수 있음
    - 원격 노드가 응답 안 하면 그게 네트워크 탓인지 노드가 죽은 건지조차 구별 못 함
- 그럼 절망적이냐? ㄴㄴ. 책의 답: **"동작에 관한 가정(=시스템 모델)을 명시하고, 그 가정 만족하는 한 올바르게 동작하도록 설계하면 됨"**
    - 기반이 아주 적은 보장만 줘도 그 위에 신뢰성을 쌓을 수 있음
- 대표 예시 **TCP/IP**
    - IP는 "패킷 유실·중복·순서 바뀜 가능"이라는 형편없는 보장만 줌
    - TCP는 그 위에 "순서 번호 + 재전송"을 얹어서 → 개발자가 물리 네트워크 신경 안 쓰고 "데이터 순서대로 옴"을 믿게 만듦
    - 못 믿을 기반 위에 믿을 만한 추상을 쌓는 것 = 분산 시스템 설계의 본질임

## 진실은 다수결로 결정된다 (정족수)

노드가 자기 판단을 못 믿는 이유를 세 장면으로 보여줌.

1. **비대칭 결함**: 들어오는 메시지는 다 받는데 나가는 메시지만 유실됨
    - 본인은 멀쩡하다고 느끼는데, 응답이 남한테 안 닿으니 다른 노드들은 "쟤 죽었다" 선언함
    - 비유: 끊긴 노드가 묘지로 끌려가면서 "난 안 죽었어!" 비명 지르는데, 아무도 못 들어서 장례 행렬 그냥 진행됨
2. **덜 끔찍한 버전**: 끊긴 노드가 "어 내 메시지 응답 안 오네, 네트워크 문제구나" 스스로 깨달음. 근데 남들이 이미 죽었다 선언했으면 손쓸 도리 없음
3. **긴 GC 노드**: 1분짜리 STW에 빠진 노드. 남들은 죽은 줄 알고 영구차에 실음. 근데 GC 끝나면 아무 일 없던 듯 깨어남
    - 비유: 죽은 줄 알았던 노드가 관 뚜껑 열고 머리 내밀어 행인들과 수다 떨기 시작 → 다들 기겁함. 정작 본인은 1분 지난 줄도, 죽었다고 선언된 줄도 모름

- **교훈**: 노드는 자기 상황에 대한 자기 판단을 못 믿음 → 한 노드 말고 **정족수(quorum)** 투표에 의존함
    - 뭔가 결정하려면 여러 노드한테 최소 몇 표 이상 받아야 함. "쟤 죽었다" 선언도 투표로 함
    - 정족수가 죽었다고 하면, 본인이 멀쩡하다 느껴도 물러나야 함
- 보통 **과반수(절반 초과)**를 씀. 왜?
    - ① 일부 노드 죽어도 시스템 계속 돌아감 (3대 중 1대, 5대 중 2대까지 OK)
    - ② 과반수는 동시에 둘이 존재 못 해서 → 충돌하는 두 결정이 동시에 내려질 수가 없음
    - (정족수/합의 본격적인 건 9장에서 팜)

## 리더와 잠금 — "선택된 자" 문제

- 시스템에 딱 하나만 있어야 하는 것들: 파티션 리더 하나(스플릿 브레인 방지), 잠금은 한 클라이언트만, 사용자명은 한 사람만
- 핵심 통찰: 노드가 스스로 "내가 그 하나야"라고 믿어도, **정족수가 동의한다는 보장은 없음**
    - 예전엔 리더였어도, 그 사이 (네트워크 끊김/GC로) 남들이 "쟤 죽었다" 선언하고 새 리더 뽑았을 수 있음
    - 강등된 줄 모르는 옛 리더가 계속 리더 행세하면 사고남
- **HBase 실제 버그 (그림 8-4)**
    - 파일 하나에 클라이언트 하나만 접근하게 임차권 씀
    - 클라이언트 1이 임차권 얻은 뒤 GC로 너무 오래 멈춰서 임차권 만료됨
    - 클라이언트 2가 같은 파일 임차권 얻어 쓰기 시작
    - 깨어난 클라이언트 1이 "난 아직 유효해" 착각하고 파일에 써버림 → 두 쓰기 충돌 + 파일 오염
    - 이거 1번 절(프로세스 중단)의 임차권 문제랑 정확히 같은 상황임

## 펜싱 토큰 (Fencing Token) — 

- 문제의 뿌리: "노드가 자기 혼자 '난 유효해'라고 판단함"
    - → 그럼 자원(저장소) 쪽에서 강제로 거르면 됨. 그게 **펜싱(fencing)**임
- 아이디어
    - 잠금 서버가 잠금/임차권 줄 때마다 **단조 증가하는 숫자 = 펜싱 토큰**을 같이 줌
    - 클라이언트가 저장소에 쓰기할 때 반드시 자기 토큰을 같이 보냄
- 동작 (그림 8-5)
    - 클라이언트 1: 토큰 **33**으로 임차권 획득 → 긴 GC로 멈춤
    - 클라이언트 2: 토큰 **34**로 임차권 획득 → 토큰 34로 쓰기 (성공)
    - 클라이언트 1: 깨어나서 토큰 33으로 쓰기 시도
    - 저장소: "이미 34 봤는데 33이 왔네" → **거부**
- 핵심: 저장소가 "지금까지 본 최대 토큰"을 기억하고, 그보다 작거나 같은 토큰의 쓰기를 거절함 → 멈췄다 깨어난 옛 리더의 늦은 쓰기가 깔끔하게 막힘
- 구현 팁
    - **주키퍼**를 잠금 서비스로 쓰면 트랜잭션 ID `zxid`나 노드 버전 `cversion`을 펜싱 토큰으로 씀 (단조 증가 보장)
    - 중요한 건 **자원(서버) 쪽이 토큰을 능동적으로 검사**해야 함. 클라이언트가 "나 잠금 있나?" 스스로 확인하는 걸론 부족함
    - 왜 서버 측 검사냐면 → 클라이언트가 항상 얌전할 거라 믿으면 안 됨. 클라이언트 만드는 사람과 서비스 운영하는 사람은 우선순위가 다름 (Halo 4 출시 때 클라이언트들이 서버 DoS 친 일화 있음)

## 비잔틴 결함 (Byzantine Faults)

- 펜싱 토큰은 부주의한 오류(멈췄다 깨어난 착각)는 막음. 근데 노드가 고의로 가짜 펜싱 토큰을 보내면? 못 막음
- 지금까지 책은 노드가 "신뢰성은 없어도 정직하다"고 가정함. 느리거나 응답 못 할 순 있어도, 응답하면 진실을 말한다고
- 노드가 거짓말(안 받은 걸 받았다고 우김)하는 경우 = **비잔틴 결함**, 이 환경에서 합의하는 문제 = **비잔틴 장군 문제**
    - 여러 장군이 전령(=못 믿을 네트워크)으로만 소통하며 동시 공격/후퇴를 합의해야 함
    - 근데 배신자가 섞여서 A한텐 "공격", B한텐 "후퇴" 거짓 보냄. 누가 배신자인지도 모름
    - (비잔티움=콘스탄티노플=이스탄불. 램포트가 특정 국적 비하 안 하려고 고른 이름임)
- 컴퓨터에서 "거짓말"의 형태
    - ① 해킹/악의적 공격
    - ② 안 받은 걸 받았다고 응답하는 소프트웨어 버그 (예: DB 저장 실패했는데 "200 OK")
    - ③ 비트 플립(0↔1) 같은 하드웨어 오염을 정상인 척 보냄
- 일부 노드가 오작동/악의적이어도 시스템이 올바르게 동작하면 **비잔틴 내결함성(BFT)** 가졌다고 함. 필요한 곳:
    - **항공우주**: 우주 방사선이 메모리 비트 뒤집을 수 있어서 잘못된 동작도 견뎌야 함
    - **블록체인/비트코인 P2P**: 서로 신뢰 못 하는 참가자들이 중앙 권한 없이 다수결로 진실 정해야 함

**근데 보통 서버 시스템은 BFT 안 씀 — 왜?**
- 일반 웹 백엔드(VPC 안 Spring Boot + DB)는 한 조직이 모든 노드를 통제하고 신뢰 가능함. 방사선도 문제될 수준 아님. BFT 프로토콜은 엄청 복잡하고 비싸서 비현실적임
- 웹에서 클라이언트(브라우저)는 분명 악의적일 수 있음. 근데 그건 BFT로 안 풀고 → **서버가 절대 권한 갖고 "이 행동 허용되냐" 판단**함 (입력 검증, 살균, 출력 이스케이핑 = SQL 인젝션·XSS 방어)
    - 즉 클라이언트랑 합의하는 게 아니라 서버가 "이상하면 400 거부"하는 구조임. BFT는 중앙 판단자 없는 P2P에서나 필요함
- **결정적으로, 소프트웨어 버그는 BFT로 못 막음**
    - BFT 전제는 "고장이 독립적"이라는 거임 (4대 중 1대 미쳐도 나머지 3대 정상)
    - 근데 똑같은 .jar를 4대에 복붙하면 → 같은 버그가 4대에서 동시에 똑같이 터져서 다수결 무의미해짐
    - 진짜 막으려면 소통 안 하는 팀들이 같은 명세로 Java/Go/C++/Rust 독립 구현(N-version) 해야 하는데 → 우주선 아니면 비현실적임

**약한 형태의 거짓말**
- 완전한 BFT까진 아니어도, 약한 거짓말(하드웨어 결함, 버그, 설정 오류로 인한 유효하지 않은 메시지) 막는 간단한 장치는 가치 있음
    - **체크섬**: 패킷 오염은 TCP/UDP 체크섬으로 대부분 잡힘. 근데 가끔 샘 (우연히 같은 체크섬 나오거나, 체크섬 이후 앱 내부에서 손상) → 애플리케이션 계층(HTTP)에서 받은 데이터로 체크섬 한 번 더 계산해 비교하면 더 안전함
    - **입력 정상성 점검**: 값 범위, 문자열 크기 제한(거대한 입력 DoS 방지)
    - **NTP 다중 서버**: 여러 서버에서 시간 받아 다수 동의 범위 추정, 엉뚱한 시간 보내는 서버를 이상치로 골라 제거

## 시스템 모델과 현실

알고리즘을 제대로 논하려면 "어떤 결함을 가정하는가"를 정형화해야 함. 그게 **시스템 모델**임. 두 축으로 나눔.

**① 타이밍 가정**

| 모델 | 가정 | 현실성 |
|---|---|---|
| 동기식 | 네트워크 지연·중단·시계 오차에 모두 상한 있음 (시계 완벽 동기는 아님) | 비현실적 |
| 부분 동기식 | 대부분 동기식처럼 동작하나 때때로 상한 초과 | 많은 시스템에서 현실적 |
| 비동기식 | 타이밍 가정 전혀 없음, 시계도 없을 수 있음 | 매우 제한적 |

**② 노드 장애 가정**

| 모델 | 가정 |
|---|---|
| 죽으면 중단(crash-stop) | 멈추면 영원히 안 돌아옴 |
| 죽으면 복구(crash-recovery) | 죽어도 시간 흐른 뒤 다시 응답 가능. 메모리는 잃어도 안정된 저장소(디스크) 데이터는 남는다고 가정 |
| 비잔틴 | 속이는 것 포함 무슨 일이든 함 |

> **현실 모델링엔 "crash-recovery + 부분 동기식"이 제일 유용함.** 거의 모든 실용적 분산 시스템이 이 위에서 설계됨

**알고리즘의 정확성이란?**
- 단일 서버는 "1+1=2 나오면 정확"이지만, 분산에선 **"가정한 모든 험한 상황에서도 미리 정한 속성을 항상 지키느냐"**가 기준임
- 펜싱 토큰 알고리즘 속성 예시
    - **유일성**: 두 요청이 같은 토큰 값 안 받음
    - **단조 일련번호**: 요청 x 끝난 뒤 요청 y 시작했으면 t_x < t_y
    - **가용성**: 토큰 요청하고 죽지 않은 노드는 결국 응답 받음
- 시스템 모델이 가정한 모든 상황에서 이 속성들 항상 만족하면 "그 모델에서 정확하다"고 함

**안전성 vs 활동성 — 장애 때 뭘 버리고 뭘 사수할지**

| | 안전성(safety) | 활동성(liveness) |
|---|---|---|
| 정의 | "나쁜 일은 절대 안 일어남" | "좋은 일은 결국 일어남" |
| 특징 | 위반되면 시점 콕 짚을 수 있고 되돌릴 수 없음(이미 손상) | 시점 못 짚어도 미래에 만족될 희망 있으면 됨 |
| 키워드 | — | 정의에 "결국에는(eventually)" 자주 들어감 (최종적 일관성도 활동성!) |
| 예시 | 유일성, 단조 일련번호 | 가용성 |

- 분산 알고리즘 일반 원칙
    - **안전성은 모든 상황에서 항상 보장** (모든 노드 죽고 네트워크 다 끊겨도 잘못된 결과는 안 냄)
    - **활동성은 조건(경고) 붙여도 됨** ("노드 과반 살아있고 네트워크 결국 복구됐을 때만 응답"처럼)
- 실무 번역: **"장애 나면 당장 응답 못 하는 건(활동성 포기) 받아들여도, 데이터 정합성 깨지는 건(안전성 위반) 절대 안 되게 설계함."** 잘못된 값 줄 바엔 차라리 멈추는 게 안전성을 지키는 길임

**시스템 모델을 현실에 대응시키기 — 모델은 결국 단순화일 뿐**
- 증명된 알고리즘이라도 현실 구현이 항상 맞게 도는 건 아님. 가정이 깨질 수 있음
    - crash-recovery는 "디스크 데이터 죽어도 남음" 가정하는데 → 현실은 디스크가 조용히 오염(bit-rot)되거나, 펌웨어 버그로 재부팅 후 디스크 인식 못 하거나, 설정 오류로 데이터 지워질 수 있음
    - 정족수 알고리즘은 "저장했다고 응답한 노드는 그걸 기억함"에 의존하는데, 노드가 기억상실 걸리면 정족수 조건 깨지고 정확성도 무너짐
- 책의 유명한 농담: 이론에선 "그런 일 안 일어남" 가정 가능하지만, 실제 구현엔 그 "불가능한 일" 터졌을 때 처리하는 코드가 들어가야 할 수도 있음. 그게 고작 `printf("Sucks to be you"); exit(666);`로 인간 운영자한테 "망했으니 알아서 치우셈" 넘기는 거더라도 (이게 컴퓨터 과학과 소프트웨어 공학의 차이임)
- 그렇다고 모델이 쓸모없는 건 아님. 정반대임. 현실의 무한한 복잡함에서 다룰 수 있는 결함 집합을 뽑아내 체계적으로 이해·증명하게 해줌
    - 실무는 **투 트랙**: 설계는 단순화된 추상 모델 위에서 논리적으로 증명, 구현은 "이론상 안 일어날" 일까지 대비해 체크섬·모니터링으로 방어
    - 증명은 훌륭한 첫걸음이지만, 이론과 경험적 실험은 둘 다 똑같이 중요함
