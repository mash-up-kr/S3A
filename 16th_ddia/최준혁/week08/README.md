## 프로세스 중단 (Process Pauses)

- 상황: 파티션마다 리더가 하나씩 있고, 리더만 쓰기를 받는 DB
    - 노드가 "내가 아직 리더인가"를 어떻게 아는가?
- 한 가지 방법: 임차권(lease)
    - 타임아웃이 있는 잠금과 비슷
    - 특정 시점에 오직 하나의 노드만 임차권 획득 가능
    - 임차권을 획득한 노드는 만료될 때까지 자신이 리더라고 판단
- 임차권 기반 코드의 문제점
    1. 동기화된 시계에 의존
        - 임차권 만료 시간은 다른 장비에서 설정됐는데 로컬 시계와 비교
        - 시계가 몇 초만 어긋나도 이상하게 동작
    2. 로컬 단조 시계로 바꿔도 문제 남음
        - "시간 확인 시점"과 "요청 처리 시점" 사이가 짧다고 가정
        - 그 사이 프로세스가 통째로 멈추면, 깨어났을 때 이미 임차권 만료 + 다른 노드가 리더
        - 알아챘을 때는 이미 안전하지 않은 일을 저지른 뒤
- 스레드가 오랫동안 멈추는 원인
    - 가비지 컬렉터(GC) - Stop-The-World
    - 가상 환경 - 하이퍼바이저가 가상 장비를 서스펜드 후 재개
    - OS 컨텍스트 스위칭 - 다른 스레드/VM으로 CPU 전환
    - 동기식 디스크 I/O - 느린 디스크 연산 대기
    - 페이지 폴트(스왑) - 메모리 접근만으로 디스크에서 페이지 로딩
- 핵심: 선점된 스레드는 자신이 멈췄다는 사실을 모름
    - 깨어나면 아무 일 없었다는 듯 다음 명령어를 이어서 실행
- 단일 장비 멀티스레딩과의 평행이론
    - 단일 장비: OS 스케줄러가 언제 스레드를 멈추고 컨텍스트 스위치할지 모름 → Mutex/Semaphore로 임계 구역 보호
    - 분산 시스템: 'OS 스케줄러' 역할을 'GC 일시 정지'나 '네트워크 지연'이 대신함
    - 결론: "로컬 시계로 5초 지났으니 아직 유효"라는 타이밍 가정이 무의미
    - 단, 분산 시스템은 멀티스레딩 도구를 그대로 못 씀 (공유 메모리 없음, 신뢰성 없는 네트워크로 메시지만 주고받음)

### 이게 만드는 최악의 장애 - 분산 락

- 시나리오 (Redis 분산 락)
    1. 노드 1이 Redis에서 TTL 10초짜리 락 획득
    2. 노드 1에서 15초짜리 GC(STW) 발생 → 멈춤
    3. 그 사이 Redis 락은 10초 지나 만료
    4. 노드 2가 같은 락 획득 → DB에 쓰기
    5. 15초 뒤 노드 1 GC 종료 → 깨어남
    6. 노드 1은 자신이 멈췄던 걸 모름 → 여전히 락 보유 중이라 착각 → DB에 쓰기
    7. 결과: 노드 2의 데이터를 노드 1이 덮어씀 (Dirty Write) → 정합성 붕괴
- 시사점: 시간의 흐름/순차적 타이밍에 의존해 동시성을 제어하면 반드시 실패
    - 해결책 → 뒤에 나올 펜싱 토큰

## 응답 시간 보장

- 스레드/프로세스는 기약 없이 멈출 수 있음. 노력하면 중단 원인 제거는 가능
- 엄격한 실시간(hard real-time) 시스템
    - 데드라인 미준수가 전체 시스템 장애로 이어지는 시스템
    - 필요 조건
        - 실시간 OS(RTOS) - 명시된 CPU 시간을 보장하도록 스케줄링
        - 라이브러리 함수의 최악 실행 시간 문서화
        - 동적 메모리 할당 제한 또는 금지
- "실시간"과 "고성능"은 다름
    - 실시간은 제때 응답을 최우선 → 오히려 처리량이 낮을 수 있음
    - 대부분의 서버측 데이터 처리 시스템에게 실시간 보장은 비경제적이고 부적절

## 가비지 컬렉션의 영향을 제한하기

- 실시간 스케줄링까지 안 가도 프로세스 중단 영향 완화 가능
- 근거: 언어 런타임은 객체 할당률 + 남은 여유 메모리를 추적 → 언제 GC할지에 대한 유연성 보유
    - JVM은 "지금 트래픽이면 N분 뒤 Full GC 필요"를 어느 정도 예측 가능
- 전략 1: GC를 '계획된 짧은 중단'으로 다룸
    1. 런타임이 곧 긴 GC가 필요하다고 신호
    2. LB/라우팅 계층에 "이 노드는 잠시 트래픽 안 받음" 통보
    3. 진행 중이던 요청만 마저 처리하고 GC 실행
    4. 가벼워지면 다시 트래픽 수신
    - 효과: GC 중단을 클라이언트로부터 감춤 → 응답 시간 상위 백분위(p99) 개선
- 전략 2: 수명 짧은 객체만 GC + 주기적 프로세스 재시작
    - 수명 긴 객체가 Full GC를 유발할 만큼 쌓이기 전에 재시작
    - 순회식 업그레이드처럼 한 번에 노드 하나씩, 재시작 전 트래픽을 다른 노드로 이동
- 실무 포인트
    - 순간 최대 TPS보다 CPU 100%를 안 치고 안정 상태(Steady State) 유지가 생존에 더 중요
    - 통제 안 된 무거운 GC는 이 안정성을 훼손

## 지식, 진실, 그리고 거짓말

- 분산 시스템의 특성
    - 공유 메모리 없음
    - 지연 변동 큰 신뢰할 수 없는 네트워크로 메시지만 전송
    - 부분 장애, 신뢰성 없는 시계, 프로세스 중단에 시달림
- 핵심: 원격 노드가 응답하지 않으면 그 상태를 알 방법이 없음
    - 네트워크 문제와 노드 자체 문제를 확실히 구별 불가
- 그럼에도 신뢰성 있는 시스템을 만드는 방법
    - 동작에 관한 가정을 명시 (= 시스템 모델 정의)
    - 그 가정을 만족시키는 방식으로 시스템 설계
    - 어떤 시스템 모델 안에서 알고리즘이 올바르게 동작하는지 증명 가능
    - 기반 시스템 모델이 적은 보장만 줘도 신뢰성 있는 동작 달성 가능
- 대표 예시: TCP/IP
    - IP(기반 시스템): 패킷 순서 섞기/유실/중복 등 '아주 적은 보장'만 제공
    - TCP(알고리즘): IP의 불완전함을 가정하고 '순서 번호 + 재전송'을 올림
    - 결과: 개발자는 물리 네트워크와 무관하게 데이터가 순서대로 도착한다고 신뢰하며 개발

## 진실은 다수결로 결정된다

- 비대칭 결함 상황: 들어오는 메시지는 다 받지만 나가는 메시지가 유실/지연
    - 노드 자신은 멀쩡하다고 느끼지만, 다른 노드들은 응답 없으니 죽었다고 판단
- 핵심: 노드는 자신의 판단을 반드시 믿을 수 있는 게 아님
    - 한 노드에만 의존 불가 → 그 노드 장애 시 시스템 전체가 멈출 수 있음
- 해결: 정족수(quorum) = 노드들 사이의 투표
    - 결정하려면 최소 개수 이상의 투표 필요
    - 정족수가 어떤 노드를 죽었다고 선언하면, 그 노드가 스스로 살아있다 확신해도 죽은 것으로 간주 → 물러나야 함
    - 과반수 정족수 사용 시 개별 노드 장애에도 시스템 계속 동작

## 리더와 잠금 - "선택된 자" 문제

- 한 노드만 특정 역할이어야 하는 상황
    - 파티션 리더는 단 하나 (split brain 방지)
    - 특정 락은 한 클라이언트만 보유
    - 특정 사용자명은 한 사용자만 등록
- 문제: 노드가 스스로 "선택된 자"라 믿어도 정족수가 동의한다는 보장 없음
    - 이전엔 리더였지만 그 사이 다른 노드들이 죽었다고 선언 + 새 리더 선출 가능
    - 강등된 노드가 계속 리더처럼 행동하면 시스템이 잘못된 일을 저지름
- HBase 사례
    - 저장소 파일을 한 번에 클라이언트 하나만 접근하게 하려고 임차권 사용
    - 임차권 보유 클라이언트가 (GC 등으로) 너무 오래 멈춤 → 임차권 만료
    - 다른 클라이언트가 같은 파일 임차권 획득 후 쓰기
    - 멈췄던 클라이언트가 깨어나 자기도 쓰기 → 쓰기 충돌, 파일 오염

## 펜싱 토큰 (Fencing Token)

- 목적: "선택된 자라고 잘못 믿는 노드"가 시스템을 방해하지 못하게 함
- 아이디어
    - 락/임차권 승인 시 펜싱 토큰(단조 증가 숫자)도 함께 반환
    - 클라이언트가 저장소에 쓰기 요청 보낼 때 현재 펜싱 토큰 포함
- 동작 예시
    - 클라이언트 1: 토큰 33으로 락 획득 → 긴 GC로 멈춤
    - 클라이언트 2: 토큰 34로 락 획득 → 토큰 34로 쓰기 (성공)
    - 클라이언트 1: 깨어나서 토큰 33으로 쓰기 시도
    - 저장소: "이미 34를 봤는데 33이 왔네" → 거부
- 즉, 이미 처리한 토큰보다 작거나 같은 토큰의 쓰기를 거부
- 구현
    - ZooKeeper: 트랜잭션 ID(zxid) 또는 노드 버전(cversion)을 펜싱 토큰으로 사용 (단조 증가 보장)
    - 자원이 펜싱 토큰을 명시적으로 지원 안 해도 우회 가능하나, 보호 안 된 요청을 막을 확인 장치 필요
- 서버 측 토큰 검증
    - 클라이언트가 항상 잘 동작할 거라 가정하는 건 비현명
    - 서버 측에서 토큰 확인 → 의도치 않게 폭력적인(buggy) 클라이언트로부터 서비스 보호

## 비잔틴 결함 (Byzantine Faults)

- 이전 문제들: 노드가 '응답을 안 하는' 문제
- 비잔틴 결함: 노드가 '적극적으로 거짓말하는' 문제
    - 예: 실제로는 받지 않은 메시지를 받았다고 주장
- 비잔틴 장군 문제 (Lamport, 1982)
    - 여러 장군(노드)이 전령(네트워크)으로만 소통하며 적의 성 포위
    - 모두 '동시 공격' 또는 '동시 후퇴'해야 승리
    - 배신자(비잔틴)가 섞여 A에겐 "공격", B에겐 "후퇴" 거짓 전달
    - 충성스러운 장군들이 속지 않고 합의에 도달할 수 있는가의 문제
- 컴퓨터 환경에서 '거짓말'의 형태
    - 해킹/악의적 공격 - 노드 장악 후 위조 데이터 전송
    - 소프트웨어 버그 - DB 저장 실패했는데 "200 OK 저장 성공"이라고 잘못 응답
    - 하드웨어 오류 - 비트 플립(0↔1)으로 오염된 데이터를 정상인 척 전송
- 비잔틴 내결함성(BFT)
    - 일부 노드 오작동/악의적 공격에도 시스템이 올바르게 동작
    - 보통 노드의 2/3 이상이 올바르게 동작하기를 요구
- 일반 웹 백엔드가 BFT를 안 쓰는 이유 (권한 구조의 차이)
    - 웹 애플리케이션(중앙 집중형)
        - 클라이언트와 서버는 동등하지 않음 → 서버가 절대 권한 보유
        - 클라이언트가 거짓말(위조 패킷, SQL 인젝션, XSS)을 해도 합의 불필요
        - 입력 검증(Validation) 후 이상하면 400 Bad Request 반환으로 끝
        - → 입력 확인/살균/출력 이스케이핑이 중요한 이유
    - BFT가 필요한 곳(P2P)
        - 비트코인 등 중앙 통제자 없고 서로 신뢰 안 하는(Trustless) 환경
        - 피어들끼리 다수결로 진실을 정해야 할 때 사용
- 소프트웨어 버그를 BFT로 못 막는 이유
    - BFT 전제: 노드들의 고장이 '독립적'이어야 함 (4개 중 1개가 미쳐도 나머지 3개 정상)
    - 동일한 .jar를 4개 노드에 복붙 배포 시 → 같은 요청에 4대가 똑같은 타이밍에 똑같이 버그 → 다수결 무의미
    - BFT로 버그를 잡으려면: 소통 안 하는 4개 팀이 같은 기획서로 각각 Java/Go/C++/Rust로 독립 구현 (N-Version Programming)
    - 우주선/항공기 제어가 아닌 이상 비현실적
- 실무적 시사점
    - 일반 기업 백엔드(VPC 안의 Spring Boot + DB)는 비잔틴 결함 거의 미고려
    - 내부 서버가 악의적으로 거짓말할 확률이 낮다고 신뢰 (비비잔틴 = Crash-recovery 모델)
    - 비잔틴이라는 극단을 소개함으로써, 우리의 평범한 분산 시스템이 얼마나 많은 '신뢰' 위에서 돌아가는지 깨닫게 함

### 약한 형태의 거짓말

- 완전한 BFT는 아니어도, 약한 거짓말(하드웨어 결함, 설정 오류)로부터 보호하는 단순 메커니즘은 추가할 가치 있음
    - 애플리케이션 수준 프로토콜에서 체크섬으로 패킷 오염 감지
    - 입력 검증 (예: 메모리 대량 할당 DoS 방지 위해 문자열 크기 제한)

## 시스템 모델과 현실

- 알고리즘은 특정 하드웨어/소프트웨어 설정에 너무 의존하지 않게 작성해야 함
- 시스템 모델 = 알고리즘이 가정하는 것을 기술한 추상화 → 발생할 결함의 종류를 정형화
- 타이밍에 대한 가정
    - 동기식 - 지연/중단/시계 오차가 결코 고정 상한치를 안 넘는다고 앎 (비현실적)
    - 부분 동기식 - 대부분 동기식처럼 동작하나 때때로 한계치 초과. 가끔 타이밍 가정이 깨질 수 있음을 고려 (현실적)
    - 비동기식 - 타이밍 가정 전혀 없음, 심지어 시계도 없을 수 있음. 매우 제한적
- 노드 장애에 대한 가정
    - 죽으면 중단(crash-stop) - 멈추면 영원히 안 돌아옴
    - 죽으면 복구(crash-recovery) - 죽어도 알려지지 않은 시간 후 다시 응답 가능. 메모리 상태는 잃지만 안정된 저장소(디스크) 데이터는 남는다고 가정
    - 비잔틴 - 다른 노드를 속이는 것 포함 무슨 일이든 함
- 현실 모델링에 가장 유용: 죽으면 복구하는 결함을 지닌 부분 동기식 모델

## 알고리즘의 정확성

- 단일 서버: 1+1=2면 정확
- 분산 환경: "아무리 험악한 상황(시스템 모델)이 닥쳐도 미리 정한 속성을 무조건 지켜내는가"가 기준
- 펜싱 토큰 생성 알고리즘의 속성 예시
    - 유일성(Uniqueness) - 두 요청이 같은 토큰 값을 반환하지 않음
    - 단조 일련번호(Monotonic) - 요청 x가 끝난 뒤 요청 y가 시작했다면 t_x < t_y
    - 가용성(Availability) - 토큰을 요청하고 죽지 않은 노드는 결국 응답 받음
- 알고리즘은 시스템 모델에서 가정한 모든 상황에서 이 속성들을 항상 만족하면 그 모델에서 '정확'

## 안정성과 활동성

- 위 3개 속성을 성격에 따라 두 그룹으로 분류
- 안정성(Safety): "나쁜 일은 절대 일어나지 않는다"
    - 해당 속성: 유일성, 단조 일련번호
    - 서버가 전부 정전돼도 반드시 지켜져야 함
    - 예: 네트워크 장애로 토큰 서버가 멈춤 → 안정성 안 깨짐 (서버가 멈췄을 뿐 중복 발급은 안 함)
    - 즉, 잘못된 값을 주느니 차라리 멈추는 게 안정성을 지키는 길
- 활동성(Liveness): "좋은 일은 언젠가(결국에는) 일어난다"
    - 해당 속성: 가용성 (최종적 일관성도 포함)
    - 정의에 흔히 "결국에는(eventually)" 포함
    - 예: 장애로 토큰을 당장 못 받으면 활동성은 잠시 깨짐. 하지만 활동성에는 경고가 허용됨 → 네트워크 복구 후 결국 발급 가능하면 만족으로 인정
- 둘을 구별하는 이유: 장애 시 "무엇을 포기하고 무엇을 사수할지" 가이드라인
    - 설계 선언: "장애 시 당장 응답 못 하는 상황(활동성 포기)이 와도, 데이터 정합성 붕괴(안정성 위반)만큼은 절대 없게 설계"
- 분산 알고리즘 일반 원칙
    - 안정성은 모든 상황에서 항상 만족 요구
    - 활동성은 경고 허용 (예: 노드 다수가 죽지 않고 네트워크가 결국 복구됐을 때만 응답)
    - 부분 동기식 모델 정의 자체가 "네트워크 끊김은 한정된 기간만 지속 후 복구"를 요구

## 시스템 모델을 현실 세계에 대응시키기

- 안정성/활동성/시스템 모델은 정확성 분석에 유용하나, 현업 구현 시 현실의 지저분한 사실들이 발목을 잡음
- 시스템 모델은 현실의 단순화된 추상화일 뿐
- 예: "디스크에 저장했으니 안전하겠지"의 함정
    - crash-recovery 모델은 "안정된 저장소 데이터는 죽어도 남는다"고 가정
    - 현실: 디스크 섹터 조용한 오염(bit-rot), 펌웨어 버그로 재부팅 시 디스크 인식 실패, 설정 오류로 데이터 삭제
- 기억상실증이 무너뜨리는 정족수
    - 정족수 알고리즘 대전제: "저장했다고 응답한 노드는 그 사실을 안 까먹음"
    - 하드웨어 오류로 노드 데이터가 날아가면 → 노드 '기억상실' → 정족수 조건 붕괴 → 알고리즘 정확성도 붕괴
- 그럼에도 이론 모델은 유용 (정반대)
    - 현실의 수만 가지 결함을 다 넣으면 모델이 복잡해져 설계/증명 불가
    - 실무는 투 트랙 전략 사용
        1. 설계 단계(이론적 추상화): 예외를 쳐내고 단순화된 모델 위에서 논리적 완벽성을 수학적으로 증명
        2. 구현 단계(방어적 프로그래밍): 이론상 안 일어날 디스크 오염 대비 체크섬 추가, 이상 상태 모니터링
    - 추상 모델은 현실 복잡함에서 '관리 가능한 결함 집합'을 뽑아내 체계적으로 이해/해결하게 해줌

## 한 줄 요약

- 프로세스 중단: GC/스왑 등으로 스레드는 임의로 멈출 수 있고 자신은 모름 → 타이밍 기반 동시성 제어는 실패
- GC 영향 제한: 런타임의 예측 능력으로 노드를 '계획된 휴식'시켜 p99 방어
- 진실은 다수결: 한 노드의 판단을 믿지 말고 정족수(투표)로 결정
- 펜싱 토큰: 단조 증가 토큰 + 서버 측 검증으로 멈췄다 깨어난 옛 리더의 쓰기 차단
- 비잔틴 결함: 노드가 거짓말하는 문제. 중앙 권한이 있는 웹 백엔드는 BFT 불필요
- 시스템 모델: "죽으면 복구하는 부분 동기식 모델"이 현실에 가장 유용
- 안정성 vs 활동성: 장애 시 활동성(응답)은 잠시 포기해도 안정성(정합성)은 무조건 사수
- 모델 vs 현실: 설계는 추상 모델로 증명, 구현은 체크섬/모니터링으로 방어 (투 트랙)
