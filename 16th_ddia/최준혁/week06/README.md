## 쓰기 스큐와 팬텀

지금까지는 같은 객체에 동시에 쓰려고 할 때 생기는 문제(더티 쓰기, 갱신 손실)를 봤다. 

이제 **서로 다른 객체에 쓰지만, 그 결정이 서로 얽혀 있어서 발생하는** 더 미묘한 경쟁 조건을 다룬다.

### 시나리오: 호출 대기 의사

병원 규칙: **최소 한 명의 의사는 항상 호출 대기(on_call) 상태여야 한다.**

현재 앨리스와 밥, 둘 다 호출 대기 중. 두 사람이 거의 동시에 퇴근을 시도한다.

1. 앨리스 트랜잭션: `SELECT count(*) WHERE on_call = true` → **2명**
2. 밥 트랜잭션: `SELECT count(*) WHERE on_call = true` → **2명** (스냅숏 격리니까)
3. 앨리스: "2명이니까 빠져도 되네" → 자기 상태 false로 UPDATE, 커밋
4. 밥: "2명이니까 빠져도 되네" → 자기 상태 false로 UPDATE, 커밋
5. 결과: 호출 대기 의사 **0명**. 비즈니스 규칙 깨짐. 

### 쓰기 스큐를 특징짓기

이걸 **쓰기 스큐(write skew)** 라고 한다. 갱신 손실의 일반화로 볼 수 있다.

- 두 트랜잭션이 **같은 객체들을 읽는다.**
- 그중 **각자 다른 객체를 갱신**한다.
- 만약 둘이 같은 객체를 갱신했다면 더티 쓰기/갱신 손실이 됐을 텐데, 다른 객체라서 잠금에 안 걸린다.

> 💡 **왜 일반적인 잠금이 안 통하는가**
> 
> 
> 행 단위 잠금은 "같은 행을 동시에 수정하려는 경우"만 감시한다. 앨리스는 앨리스 행, 밥은 밥 행을 수정하니까 충돌이 없어 보인다. 하지만 **둘이 읽은 전제 조건("의사 2명")이 서로의 쓰기로 깨져버린다는 것**이 핵심.
> 

### 쓰기 스큐는 자동으로 안 막힌다

- **단일 객체 원자적 연산**: 객체가 여러 개라 무의미.
- **갱신 손실 자동 감지**: PostgreSQL Repeatable Read, Oracle Serializable, MySQL Repeatable Read, SQL Server Snapshot Isolation 모두 **쓰기 스큐는 감지 못 함**.
- **DB 제약 조건**: 쓸 수 있는 경우엔 좋지만, "최소 1명 이상"처럼 여러 행에 걸친 조건은 표현하기 어렵다.
- **직렬성 격리**: 진짜로 막으려면 이게 필요.

차선책은 **명시적 잠금**:

```sql
SELECT * FROM doctors
 WHERE on_call = true AND shift_id = 1234
 FOR UPDATE;
```

`FOR UPDATE`로 조회 결과 행들을 미리 잠근다. 그러면 밥은 앨리스가 끝날 때까지 같은 행들을 못 건드린다.

### 다른 예시들

쓰기 스큐는 많은 예시가 있다.

- **회의실 예약**: "이 시간에 예약이 없으면 새로 잡는다" — 두 사람이 동시에 보면 둘 다 빈 걸로 보고 둘 다 예약.
- **사용자명 중복 방지**: "이 username 없으면 가입" — 동시 가입 시 중복 생성.
- **다중 플레이어 게임**: 두 플레이어가 같은 칸으로 이동.
- **잔고 초과 출금 방지**: 두 출금이 각각은 잔고 내인데 합치면 초과.

## 쓰기 스큐를 유발하는 팬텀

쓰기 스큐의 패턴은 항상 똑같다.

1. **SELECT로 어떤 조건을 만족하는 행을 찾는다.** (또는 없음을 확인)
2. **그 결과에 따라 애플리케이션 코드가 다음 행동을 결정한다.** (if 문)
3. **결정에 따라 DB에 쓰기를 한다.** (INSERT/UPDATE/DELETE)

문제는 1번과 3번 사이에 **다른 트랜잭션의 쓰기가 1번 쿼리의 결과를 바꿔버리는 것**. 만약 쓰기 직후 1번 쿼리를 재실행하면 다른 결과가 나온다.

이 현상을 **팬텀(phantom)** 이라고 한다. "유령처럼 없던 데이터가 갑자기 끼어들거나, 있던 데이터가 사라져서 결과가 달라지는 것."

### 의사 예시 vs 회의실 예시의 결정적 차이

| 시나리오 | 1단계 SELECT 결과 | `FOR UPDATE`로 막을 수 있나? |
| --- | --- | --- |
| 호출 대기 의사 | 행이 **존재함** (앨리스, 밥 행) | 가능. 행을 잠그면 됨 |
| 회의실 예약 | 행이 **없음** (예약 0건) | 불가능. **잠글 대상이 없음** |

> 💡 **"없는 것"에는 자물쇠를 채울 수 없다**
> 
> 
> `SELECT ... FOR UPDATE`는 검색 결과로 나온 행에만 잠금을 건다. 0건이 반환되면 잠글 게 없다.
> 
> 그래서 트랜잭션 A가 "12시 예약 없네 → 예약하자"를 하는 동안, 트랜잭션 B도 똑같이 "12시 예약 없네 → 예약하자"를 해서 이중 예약이 발생한다.
> 
> "INSERT로 새 행을 만드는" 패턴의 쓰기 스큐는 행 단위 잠금으로 못 막는다.
>

## 충돌 구체화

팬텀을 막을 잠금 대상이 없다면, **인위적으로 잠금 대상을 만들어 두자**는 아이디어.

회의실 예약의 경우:

- 회의실 × 시간 슬롯의 **모든 조합에 대해 행을 미리 만들어 둔다.**
- 예약 트랜잭션은 먼저 그 행을 `FOR UPDATE`로 잠근다.
- 그 다음 실제 예약 정보를 다른 테이블에 INSERT 한다.

이걸 **충돌 구체화(materializing conflicts)** 라고 한다. 팬텀을, DB에 실제로 존재하는 행 집합에 대한 잠금 충돌로 변환한 것.

### 단점

- 어떤 잠금 대상을 미리 만들어둘지 알아내기 어렵다.
- 동시성 제어 메커니즘이 애플리케이션 데이터 모델로 새어 나온다(지저분).
- 회의실 × 시간 같이 조합이 많아지면 미리 만드는 비용도 크다.

**다른 대안이 모두 불가능할 때의 최후 수단.** 가능하면 직렬성 격리 수준이 낫다.

## 직렬성

**직렬성(Serializability)**: 트랜잭션들이 동시에 실행되더라도, 최종 결과가 **한 줄로 줄 세워 실행한 것과 동일하도록 보장**하는 가장 강한 격리 수준.

직렬성이 있으면 모든 경쟁 조건(더티 읽기/쓰기, 비반복 읽기, 갱신 손실, 쓰기 스큐, 팬텀)이 막힌다.

그런데 왜 다들 안 쓰냐? **비싸기 때문.** 구현 방식 셋 중 하나를 골라야 하는데, 각자 성능 비용이 다르다.

1. **실제 직렬 실행** — 진짜로 한 줄로 줄 세움
2. **2단계 잠금(2PL)** — 수십 년간 표준이었던 방식
3. **직렬성 스냅숏 격리(SSI)** — 낙관적 동시성 제어

## 실제적인 직렬 실행

가장 단순하다. **트랜잭션을 단일 스레드에서 한 번에 하나씩 실행한다.** 동시성이 없으니 경쟁 조건도 없다.

### 왜 갑자기 이게 현실성이 생겼나?

두 가지 변화 때문.

1. **RAM이 충분히 싸졌다** — OLTP 데이터셋 정도는 통째로 메모리에 올릴 수 있게 됨. 디스크 I/O 대기가 없으면 단일 스레드도 충분히 빠르다.
2. **OLTP는 보통 짧고 적은 쓰기를 한다** — 분석 쿼리는 어차피 읽기 전용이라 직렬 루프 밖에서 스냅숏으로 처리 가능.

Redis, VoltDB가 이 방식의 대표 사례.

### 스토어드 프로시저로 트랜잭션 캡슐화

단일 스레드 직렬 실행의 가장 큰 문제는 **네트워크 대기**.

일반적인 인터랙티브 트랜잭션은:

```
App → DB: SELECT
DB → App: 결과
App: (생각, 계산)
App → DB: UPDATE
DB: 처리
App → DB: COMMIT
```

각 단계 사이에 네트워크 왕복이 끼어 있다. 다중 스레드 DB는 그 사이 다른 트랜잭션을 처리하면 되지만, **단일 스레드는 그 시간 동안 기다려야 한다**. 처리량이 떨어진다.

> 
> 
> 
> 💡 **왜 스토어드 프로시저가 필요한가**
> 
> 일반적인 인터랙티브 트랜잭션은 `SELECT → 결과 수신 → 애플리케이션에서 계산 → UPDATE → COMMIT` 식으로 진행된다. 각 구문 사이마다 애플리케이션과 DB가 네트워크로 왕복한다.
> 
> 다중 스레드 DB는 이 대기 시간 동안 다른 트랜잭션을 처리하면 되지만, 단일 스레드 직렬 실행은 그럴 수 없다. 한 트랜잭션이 다음 구문을 보내올 때까지 DB가 그냥 멈춰 있어야 한다. 처리량이 망가지는 결정적 이유다.
> 
> 해결책은 **트랜잭션 로직 전체를 스토어드 프로시저 형태로 DB에 미리 등록해두는 것**. 애플리케이션은 "이 프로시저 실행" 한 번의 호출로 끝내고, DB는 받자마자 처음부터 끝까지 쭉 실행한 뒤 다음 트랜잭션으로 넘어간다. 네트워크 왕복이 사라지면서 단일 스레드도 충분한 처리량을 낼 수 있게 된다.
> 

VoltDB 같은 시스템은 인터랙티브 다중 구문 트랜잭션을 아예 허용하지 않는다. 스토어드 프로시저로만 받는다.

### 파티셔닝의 한계

단일 스레드는 CPU 코어 하나의 속도가 한계. 그래서 **파티셔닝**으로 확장한다 — 각 파티션에 독립적인 단일 스레드 셰프를 배정.

문제는 **여러 파티션을 건드리는 트랜잭션**.

- 모든 관련 파티션에 걸쳐 잠금을 획득하고, 코디네이션 해야 직렬성이 보장됨.
- VoltDB는 다중 파티션 트랜잭션을 **초당 1000건** 정도밖에 처리 못 한다. 단일 파티션 처리량보다 훨씬 낮고, **장비를 추가해도 안 늘어난다** (코디네이션 오버헤드가 같이 늘어남).

> 💡 **데이터 모델링이 곧 성능**
> 
> 
> 단일 파티션으로 끝나는 트랜잭션이 많은 데이터 구조여야 진짜 직렬 실행이 성능을 발휘한다.
> 
> - 나쁨: 유저 정보는 A 파티션, 주문은 B, 장바구니는 C → 뭐만 해도 다중 파티션.
> - 좋음: `user_id`로 파티셔닝 → 한 유저의 모든 데이터가 한 파티션에 → 대부분의 트랜잭션이 단일 파티션.

| 직렬 실행이 잘 맞는 경우 | 잘 안 맞는 경우 |
| --- | --- |
| 활성 데이터셋이 메모리에 들어감 | 데이터셋이 너무 큼 |
| 트랜잭션이 짧고 빠름 | 장기 분석/배치 쓰기 |
| 쓰기 처리량이 단일 CPU로 충당됨 | 쓰기 처리량이 매우 높음 |
| 여러 파티션 트랜잭션이 적음 | 파티션 횡단 트랜잭션이 잦음 |

## 2단계 잠금 (2PL)

수십 년간 유일한 직렬성 구현 알고리즘이었다. 2단계 커밋(2PC)이랑 이름은 비슷한데 **완전히 다른 것**이니 헷갈리지 말 것.

### 핵심 규칙

기존 행 단위 쓰기 잠금보다 훨씬 빡세다.

- **여러 트랜잭션이 동시에 같은 객체를 읽는 것은 OK** (공유 잠금).
- **누구라도 그 객체에 쓰려고 하면 독점적 접근 필요** (독점 잠금).

이게 곧 다음을 의미한다.

- **읽기가 쓰기를 막는다**: 누가 읽고 있으면 그 객체에 쓰기 못 함.
- **쓰기가 읽기를 막는다**: 누가 쓰고 있으면 그 객체에서 읽기 못 함.

> 💡 **스냅숏 격리와의 결정적 차이**
> 
> 
> 
> |  | 읽기가 쓰기를 막는가 | 쓰기가 읽기를 막는가 |
> | --- | --- | --- |
> | 스냅숏 격리 (MVCC) | ❌ | ❌ |
> | 2PL | ✅ | ✅ |
> 
> 스냅숏 격리: "쓰는 사람은 새 버전 만들고, 읽는 사람은 옛 버전 봐. 서로 안 막음." (구글 문서 버전 기록)
> 
> 2PL: "내가 읽는 동안 아무도 못 고쳐. 내가 고치는 동안 아무도 못 봐." (도서관 단행본)
> 
> 2PL은 일관성을 위해 동시성을 희생한다.
> 

### 구현 디테일

각 객체에 **공유 모드(S-Lock)** 또는 **독점 모드(X-Lock)** 잠금.

1. 읽으려면 → S-Lock 획득. 다른 트랜잭션이 X-Lock 쥐고 있으면 대기.
2. 쓰려면 → X-Lock 획득. 다른 트랜잭션이 S-Lock이든 X-Lock이든 쥐고 있으면 대기.
3. 읽다가 쓰기로 바뀌면 → S-Lock을 X-Lock으로 **업그레이드**. 다른 누군가 S-Lock 쥐고 있으면 대기.
4. **잠금은 트랜잭션 종료 시점에 한 번에 푼다** (이게 "2단계" — 1단계: 획득만, 2단계: 해제만).

MySQL InnoDB, SQL Server는 Serializable 격리 수준을 2PL로 구현한다. DB2는 Repeatable Read를 2PL로 구현.

### 성능: 상당히 나쁨

2PL의 큰 단점은 **성능**. 트랜잭션 처리량과 응답 시간이 완화된 격리 수준 대비 크게 나빠진다.

- 잠금 획득/해제 자체의 오버헤드.
- **읽기와 쓰기가 서로 막아서 대기 시간이 폭발**.
- 경쟁이 심하면 응답 시간이 매우 불안정. 99th percentile이 처참해질 수 있다.

### 교착 상태(Deadlock)

2PL의 또 다른 문제.

- 트랜잭션 A: 객체 X에 X-Lock 쥐고, 객체 Y의 X-Lock 기다림.
- 트랜잭션 B: 객체 Y에 X-Lock 쥐고, 객체 X의 X-Lock 기다림.
- 영원히 못 끝남.

DB가 데드락을 감지하면 한쪽을 강제로 어보트시킨다. 어보트된 트랜잭션은 처음부터 재시도.

> 💡 **데드락의 비용이 단순 재시도가 아니다**
> 
> 
> Spring으로 짠 트랜잭션 흐름을 상상해보자.
> 
> 1. DB에서 데이터 10건 조회 (네트워크 + DB CPU)
> 2. 애플리케이션 메모리에서 비즈니스 로직 계산 (App CPU)
> 3. DB에 업데이트
> 4. 데드락 → 강제 어보트
> 5. 처음부터 재시도
> 
> 1~3에서 쓴 자원이 통째로 날아간다. 데드락이 잦으면 **시스템이 자기 자신과 싸우느라 정작 사용자 요청은 처리를 못 하게 된다.**
>

## 서술 잠금과 색인 범위 잠금

2PL만으로 부족하다. 행 단위 잠금은 **존재하지 않는 행에 대한 팬텀**을 못 막는다 (회의실 예약 케이스).

### 서술 잠금 (Predicate Lock)

이론적인 해답. 잠금을 **객체가 아니라 WHERE 조건(서술)에 건다.**

> 
> 
> 
> 💡 **서술 잠금이 일반 잠금과 다른 점**
> 
> 기존 행 단위 잠금은 이미 존재하는 행에만 걸 수 있다. `SELECT ... FOR UPDATE`로 조회해도 결과가 0건이면 잠글 대상 자체가 없다.
> 
> 서술 잠금은 객체가 아니라 **WHERE 조건 자체에 잠금을 건다.** `room_id = 101 AND time = '12:00'`이라는 조건에 잠금이 걸리면, 그 조건을 만족하는 어떤 데이터도 추가/수정/삭제될 수 없다.
> 
> 행이 0건이어도 조건만으로 잠금이 성립하기 때문에, INSERT로 새 행이 생기는 패턴의 팬텀도 막을 수 있다.
> 

2단계 잠금 + 서술 잠금이면 모든 형태의 쓰기 스큐와 경쟁 조건이 막힌다. 진짜 직렬성 완성.

문제: **너무 느리다.** 새 행을 INSERT 할 때마다 "지금 진행 중인 모든 트랜잭션의 모든 WHERE 조건 중 하나라도 이 새 행과 매칭되는가?"를 계산해야 한다. 비용이 어마어마.

### 색인 범위 잠금 (Next-Key Lock)

서술 잠금의 **현실적인 해결책**. 실무에서 쓰이는 방식.

핵심 아이디어: **서술 조건을 "근사화"해서 인덱스(색인)에 잠금을 건다.**

- 원래 조건: `room_id = 101 AND time = '12:00'`
- 근사: "`room_id` 인덱스 트리에서 101번 노드 근처 영역을 잠근다."

이제 누가 101호의 어떤 시간이든 INSERT 하려고 하면, 그 인덱스 위치를 갱신해야 하는데 거기 잠금이 걸려있어서 막힌다.

> 💡 **오버로킹(Over-locking)이라는 타협**
> 
> 
> 색인 범위 잠금은 정밀하지 않다. 누군가 "101호 12시"를 잠갔는데, 다른 사람이 "101호 오후 3시"를 예약하려 해도 막힐 수 있다 (같은 인덱스 영역).
> 
> 대신, 매번 조건식을 수학적으로 계산할 필요 없이 **인덱스 자료구조에 자물쇠 표식이 있는지만 물리적으로 확인**하면 되니까 엄청나게 빠르다.
> 
> 정밀함을 일부 포기하고 성능을 얻은 타협이다. MySQL InnoDB가 이 방식.
> 

인덱스가 없으면 더 무식하게 테이블 전체를 잠가야 한다. **그래서 인덱스가 동시성에도 영향을 미친다.**

| 잠금 방식 | 정밀도 | 성능 | 실무 사용 |
| --- | --- | --- | --- |
| 서술 잠금 | 매우 높음 | 매우 나쁨 | ❌ 거의 안 씀 |
| 색인 범위 잠금 | 보통 (오버로킹 발생) | 양호 | ✅ MySQL InnoDB 등 |
| 행 단위 잠금만 | 낮음 (팬텀 못 막음) | 빠름 | 일반 격리 수준 |

## 직렬성 스냅숏 격리 (SSI)

상황 정리:

- **2PL**: 직렬성은 보장하지만 성능이 나쁘고 확장이 어렵다.
- **실제 직렬 실행**: 빠르지만 단일 CPU 한계, 다중 파티션에 약하다.
- **완화된 격리 수준**: 빠르지만 경쟁 조건에 뚫린다.

**직렬성과 좋은 성능은 공존할 수 없는가?** 그렇지 않다. 비교적 최근(2008) 등장한 알고리즘인 **직렬성 스냅숏 격리(Serializable Snapshot Isolation, SSI)** 가 답을 제시했다. PostgreSQL 9.1+ 의 Serializable이 이걸로 구현됐다.

### 비관적 vs 낙관적 동시성 제어

- **비관적**: "충돌이 날 거라고 가정하고 미리 막는다." → 2PL, 진짜 직렬 실행. 자물쇠 + 대기.
- **낙관적**: "충돌이 안 날 거라고 가정하고 일단 다 같이 달리게 둔다. 커밋 시점에 진짜로 충돌했는지 검사해서 사고 친 놈만 어보트시킨다." → SSI.

> 💡 **낙관적 동시성 제어의 도박**
> 
> 
> "대부분의 트랜잭션은 서로 안 부딪칠 거야"라는 가정 위에서 동작한다.
> 
> 충돌이 거의 없을 때: 자물쇠 오버헤드 없이 풀스피드. 짱 빠름.
> 충돌이 잦을 때: 어보트와 재시도가 폭증. 비관적 방식보다 더 느려질 수 있음.
> 
> **시스템 부하가 높지 않고, 트랜잭션 간 충돌이 드물 때 빛난다.**
> 

### SSI의 토대: 스냅숏 격리

SSI는 이름 그대로 **스냅숏 격리 위에 올라간 알고리즘**. 모든 읽기는 일관된 스냅숏을 본다. 잠금 없음. **읽기와 쓰기가 서로 막지 않는다** — 이게 2PL 대비 핵심 이점.

그 위에 **"쓰기 작업 사이의 직렬성 충돌을 감지하고 어보트 결정을 내리는 알고리즘"** 을 얹는다.

### 뒤처진 전제에 기반한 결정

SSI가 잡으려는 패턴:

1. 트랜잭션이 어떤 데이터를 읽는다 (= 어떤 **전제**를 확인한다).
2. 그 전제를 바탕으로 결정한다.
3. 그 결정에 따라 쓴다.

문제는 1번과 3번 사이에 **다른 트랜잭션이 그 전제를 깨버릴 수 있다는 것**. (이게 쓰기 스큐의 본질이었음.)

DB가 어떻게 알아내냐? 두 가지를 추적한다.

### (1) 오래된 MVCC 읽기 감지

트랜잭션이 MVCC 가시성 규칙 때문에 **무시한 쓰기들**을 기록해둔다. 커밋 시점에 그 무시된 쓰기 중 이미 커밋된 게 있으면 **현재 트랜잭션을 어보트**.

> 💡 **"무죄 추정의 원칙"**
> 
> 
> 의문: "오래된 데이터 읽었다는 걸 감지한 그 순간 바로 어보트시키면 안 됨? 왜 커밋까지 기다림?"
> 
> 세 가지 이유 때문에 기다린다.
> 
> 1. **읽기 전용 트랜잭션일 수도 있다** — 쓰기 안 하면 쓰기 스큐 위험 자체가 없다. 어보트시킬 이유 없음.
> 2. **그 시점엔 이 트랜잭션이 나중에 뭘 할지 모른다** — 독심술 못 함.
> 3. **원인 제공한 트랜잭션이 어보트될 수도 있다** — 그러면 이쪽이 읽은 게 다시 맞는 값이 됨. 미리 죽이면 헛수고.
> 
> SSI는 최대한 살려두고 **커밋 시점에 진짜로 사고 났는지** 본다.
> 

### (2) 과거의 읽기에 영향을 미치는 쓰기 감지

반대 방향. 트랜잭션이 쓸 때, **그 영향 받는 데이터를 최근에 읽은 다른 트랜잭션이 있는지** 인덱스에서 확인.

> 💡 **자물쇠 대신 "방명록" + "지뢰선"**
> 
> 
> 2PL: 누군가 읽으면 그 인덱스 영역에 **자물쇠**를 채워서 길을 막는다.
> 
> SSI: 길을 막지 않는다. 대신 인덱스에 **"T42가 여기 읽고 갔음" 방명록**을 적어둔다. 트랜잭션은 막힘없이 쌩쌩 달림.
> 
> 누군가 그 인덱스 영역을 수정하려고 오면, **방명록을 보고 "T42, 네가 읽은 데이터 방금 누가 바꿨음" 경고장(트립와이어)을 T42 등에 붙인다.** 여전히 막진 않는다.
> 
> 최종 커밋 시점에 등에 경고장이 붙어 있는지 보고, 붙어 있고 + 원인 제공자가 이미 커밋했으면 → 어보트.
> 

이게 색인 범위 잠금과 비슷한 구조지만 **다른 트랜잭션을 차단하지 않는다는 점**이 결정적 차이.

### 충돌 시나리오 풀어보기

다시 의사 호출 대기. 트랜잭션 42(앨리스 퇴근), 43(밥 퇴근)이 동시 실행.

1. T42, T43 모두 `WHERE on_call=true AND shift_id=1234`로 조회 → 둘 다 "2명" 봄.
    - SSI는 인덱스에 "T42 읽음, T43 읽음" 기록.
2. T42가 자기 행 UPDATE.
    - 인덱스 방명록 보고 "T43 등에 경고장: 네가 읽은 거 T42가 바꿨음."
3. T43이 자기 행 UPDATE.
    - "T42 등에 경고장: 네가 읽은 거 T43이 바꿨음."
4. T42 커밋 시도.
    - T42 등에 경고장 있음. 하지만 T43이 아직 커밋 안 했음 → 경고장의 원인이 아직 "진짜"가 아님 → **T42 커밋 성공**.
5. T43 커밋 시도.
    - T43 등에 경고장 있음. 원인인 T42는 이미 커밋했음 → 경고장이 진짜가 됨 → **T43 어보트**.

T43은 처음부터 재시도. 재시도하면 이번엔 T42의 변경(앨리스 휴식)이 보이니까 "의사 1명뿐이네, 못 빠지겠다"로 결정.

### SSI의 성능

| 항목 | 평가 |
| --- | --- |
| 잠금 대기 |  없음 — 읽기/쓰기 서로 안 막음. 읽기 전용은 잠금 없이 스냅숏만 봄 |
| 추적 비용 | 트랜잭션의 읽기/쓰기를 기록해야 함 — 너무 정밀하면 오버헤드, 너무 거칠면 불필요한 어보트 |
| 단일 CPU 한계 |  없음 — 진짜 직렬 실행과 달리 여러 코어 활용 가능 (FoundationDB는 충돌 감지를 분산) |
| 어보트율 | 충돌이 적은 워크로드에선 매우 낮음. 충돌이 잦으면 처참 |
| 긴 트랜잭션 |  약함 — 오래 걸리는 읽기/쓰기 트랜잭션은 충돌 확률이 높아 어보트되기 쉬움. **트랜잭션을 짧게 유지하는 게 중요** |

읽기 작업이 압도적으로 많은 워크로드에서 특히 매력적이다. 읽기 트랜잭션은 잠금 없이 그냥 흐른다.

## 핵심 정리

- **쓰기 스큐**: 두 트랜잭션이 **같은 객체들을 읽고, 각자 다른 객체를 수정**할 때 발생. 행 단위 잠금으론 못 막는다. **결정의 근거(전제)** 가 깨지는 게 본질.
- **팬텀**: 한 트랜잭션의 쓰기가 다른 트랜잭션의 **검색 결과 집합** 자체를 바꾸는 현상. 쓰기 스큐의 까다로운 케이스를 만든다.
- **존재하지 않는 행에는 `FOR UPDATE`로 잠금을 걸 수 없다.** INSERT로 새 행이 생기는 패턴의 쓰기 스큐는 행 잠금으로 못 막음.
- **충돌 구체화**: 잠금 대상이 없으면 인위적으로 만들어두는 꼼수. 최후의 수단.
- **직렬성**의 세 가지 구현:
    - **실제 직렬 실행**: 단일 스레드. 데이터가 메모리에 들어가고 트랜잭션이 짧을 때 빠름. 다중 파티션 트랜잭션이 잦으면 망함. 스토어드 프로시저 필수.
    - **2PL**: 읽기/쓰기 둘 다 자물쇠. 읽기가 쓰기를 막고 그 반대도 성립. 성능 나쁨. 데드락 잦음.
    - **SSI**: 낙관적. 일단 다 같이 달리고 커밋 시점에 충돌 검사. 충돌이 적을 때 빛난다.
- **스냅숏 격리 vs 2PL의 핵심 차이**: "읽는 쪽이 쓰는 쪽을 막는가, 그 반대도 성립하는가." 스냅숏 격리는 안됨, 2PL은 가능.
- **서술 잠금**은 이론적으론 완벽한 답이지만 계산 비용이 너무 비싸서 실무에선 **색인 범위 잠금**으로 근사한다. 그래서 인덱스 설계가 동시성에도 영향을 준다.
- **SSI의 핵심 트릭**: 자물쇠 대신 인덱스에 "누가 읽고 갔는지" 방명록을 남기고, 누가 수정하면 읽은 사람에게 경고장(트립와이어)을 붙인다. **커밋 시점**에 경고장 + 원인 제공자가 이미 커밋했으면 어보트.
- **SSI는 짧은 트랜잭션에 강하다.** 긴 트랜잭션은 충돌에 노출되는 시간이 길어서 어보트되기 쉽다.
- 같은 격리 수준 이름이라도 DB마다 보장이 다르다. **이름 말고 실제 동작을 봐야 한다.** (오라클 "Serializable"은 사실 스냅숏 격리.)
