### 관계형 모델 vs 문서 모델

### 객체 관계형 불일치 (Impedance Mismatch)

애플리케이션은 객체지향 언어(Java 등)로 개발되는데, 관계형 데이터베이스에 저장하려면 애플리케이션 코드의 모델(Entity)과 데이터베이스 모델 객체(테이블/행/열) 사이에 전환 계층이 필요하다. 이 간극을 **임피던스 불일치**라 부른다.

ORM 프레임워크가 이를 일정 부분 해결해주지만, 두 모델 간의 차이를 완벽하게 좁힐 수는 없다.

### 문서 모델의 장점

JSON 표현은 **지역성(locality)**을 가진다. 모든 관련 정보가 한 곳에 있어서 한 번의 질의로 충분하다. 관계형 모델에서는 같은 데이터를 가져오려면 다중 조인이 필요하다.

이력서처럼 일대다(one-to-many) 관계로 이루어진 데이터는 트리 구조와 같아서 문서 모델이 자연스럽다.

### 다대일과 다대다 관계

- 문서 모델은 일대다 트리 구조를 위해 조인이 필요하지 않지만, **조인에 대한 지원이 보통 약하다.**
- 초기에 문서 모델에 적합하더라도, 데이터 간 연결은 거의 반드시 발생한다. 데이터베이스 자체가 조인을 지원하지 않으면 애플리케이션 레벨에서 다중 질의를 만들어 해결해야 한다.

### 문서 데이터베이스는 역사를 반복하는가?

과거에도 비슷한 문제가 있었다. **계층 모델(Hierarchical Model)**은 일대다 관계에서는 잘 작동했지만, 다대다 관계 표현이 어려웠고 조인을 지원하지 않았다. 문서 데이터베이스도 이와 유사한 특성을 가진다.

이걸 해결하기 위해 두 가지 모델이 등장했다.

### 네트워크 모델 (CODASYL)

- 계층 모델에서 모든 레코드는 정확히 하나의 부모를 갖는 반면, 네트워크 모델에서 레코드는 **다중 부모**를 가질 수 있다.
- 최상위 레코드에서부터 연속된 연결 경로(접근 경로)를 따라가면서 한 번에 하나의 레코드만 보면서 탐색한다.
- 문제: 레코드가 다중 부모를 가지면 애플리케이션 코드가 다양한 관계를 모두 추적해야 한다. 접근 경로를 변경하려면 많은 질의 코드를 재작성해야 한다.

### 관계형 모델의 강점

- 얽히고 설킨 중첩 구조와 복잡한 접근 경로가 없다.
- *질의 최적화기(Query Optimizer)**가 질의의 어느 부분을 어떤 순서로 실행할지, 사용할 색인을 **자동으로** 결정한다.
- 새로운 방식으로 데이터에 질의하려면 새로운 색인을 선언하기만 하면 된다. 애플리케이션에 새 기능을 추가하는 작업이 훨씬 쉽다.

### 문서 모델 vs 관계형 모델 — 결론

- 다대일, 다대다 관계를 표현할 때 둘은 근본적으로 다르지 않다. 관계형 모델에서는 **외래 키**, 문서 모델에서는 **문서 참조**로 관련 항목을 참조한다.
- **문서 모델을 선호하는 이유**: 스키마 유연성, 지역성에 기인한 더 나은 성능, 일대다 관계 트리를 한 번에 적재하는 구조에 적합
- **관계형 모델을 선호하는 이유**: 조인, 다대일, 다대다 관계를 더 잘 지원

> 일반적으로 어떤 데이터 모델이 애플리케이션 코드를 더 간단하게 만드는지 말할 수 없다. 데이터 항목 간에 존재하는 관계 유형에 따라 다르다.
>

### 스키마 유연성: 쓰기 스키마 vs 읽기 스키마

"스키마리스(schemaless)"라는 표현은 오해를 부른다. 정확히는 **스키마 검증 시점이 다를 뿐**이다.

| 구분 | 설명 | 비유 |
| --- | --- | --- |
| **쓰기 스키마 (schema-on-write)** | 저장할 때 스키마를 강제 (MySQL, PostgreSQL) | 정적 타입 언어 (Java, C++) — 컴파일 때 타입 체크 |
| **읽기 스키마 (schema-on-read)** | 저장은 자유, 읽을 때 해석 (MongoDB) | 동적 타입 언어 (Python, JS) — 런타임 때 타입 체크 |

### 실제 차이 — 컬럼을 추가할 때

`name` 필드를 `first_name` + `last_name`으로 쪼개고 싶다고 가정하자.

**MongoDB (읽기 스키마):** DB를 건드릴 필요 없이 코드에서 읽을 때 처리한다.

```jsx
if (user && user.name && !user.first_name) {
    user.first_name = user.name.split(" ")[0];
}
```

**MySQL (쓰기 스키마):** 전체 테이블을 마이그레이션해야 한다.

```sql
ALTER TABLE users ADD COLUMN first_name text;
UPDATE users SET first_name = split_part(name, ' ', 1);
```

### ALTER TABLE의 실체

`ALTER TABLE`이 느리다는 평판은 과장된 면이 있다. PostgreSQL은 밀리초 단위로 처리한다. MySQL은 테이블 전체를 복사하기 때문에 대규모 테이블에서 수 분~수 시간이 걸릴 수 있지만, 이건 MySQL의 구현 문제다.

진짜 문제는 `UPDATE`로 기존 데이터를 전부 바꾸는 작업이 오래 걸린다는 것이다. 그래서 새 컬럼을 NULL 기본값으로만 추가하고, 실제 값은 읽을 때 채우는 방식을 쓰기도 한다. 이건 사실상 관계형 DB에서도 읽기 스키마처럼 동작하는 것이다.

**결론:** 

- 데이터 구조가 자주 바뀌거나 이종(heterogeneous) 데이터라면 읽기 스키마(문서 모델)가 유리하고
- 구조가 고정적이면 쓰기 스키마(관계형)가 유용한 문서화 겸 검증 메커니즘이 된다.

### 저장소 지역성 (Storage Locality)

문서 DB는 문서 하나를 보통 하나의 연속된 문자열(JSON, XML 등)로 저장한다. 애플리케이션이 자주 전체 문서에 접근해야 할 때, 한 번의 디스크 탐색으로 전체 문서를 가져올 수 있으므로 **성능 이점**이 있다.

반면 관계형 모델에서는 여러 테이블에 걸쳐 조인해야 하므로 다중 색인 검색이 필요하다.

단, 지역성의 이점은 **문서의 많은 부분을 한 번에 필요로 하는 경우에만** 적용된다. 문서의 작은 부분만 접근하더라도 전체 문서를 로드해야 하므로, 큰 문서에서는 낭비가 될 수 있다. 갱신 시에도 보통 전체 문서를 다시 쓰게 된다.

### 데이터를 위한 질의 언어: 선언형 vs 명령형

SQL은 **선언형 질의 언어**인 반면, IMS와 코다실은 **명령형 코드**를 사용해 데이터베이스에 질의했다.

- **명령형 언어**: 특정 순서로 특정 연산을 수행하도록 컴퓨터에 지시한다. "첫 번째 레코드로 가라 → 확인하라 → 다음으로 이동하라"
- **선언형 언어**: 목표를 달성하기 위한 **방법이 아니라**, 알고자 하는 데이터의 패턴(결과가 충족해야 하는 조건)과 데이터를 어떻게 변환할지를 지정하기만 하면 된다.

### 선언형 질의 언어의 장점

1. **최적화 여지**: SQL이 기능적으로 더 제한적이라는 사실이 오히려 데이터베이스에게 자동으로 최적화할 수 있는 여지를 더 많이 준다. 질의를 변경하지 않고도 데이터베이스 시스템의 성능을 향상시킬 수 있다.
2. **병렬 실행에 적합**: 명령형 코드는 명령어를 특정 순서로 수행하도록 지정하기 때문에 다중 코어나 다중 장비에서 병렬 처리가 매우 어렵다. 선언형 언어는 결과만 명시하므로 실행 방법은 엔진이 알아서 병렬화할 수 있다.
3. **물리적 독립성**: 데이터베이스가 내부적으로 디스크 공간을 회수하기 위해 레코드를 옮겨도, SQL은 특정 순서를 보장하지 않으므로(`ORDER BY` 없이) 결과에 영향이 없다. 반면 명령형 코드는 물리적 저장 순서에 의존할 수 있어, 데이터베이스가 내부 구조를 마음대로 변경하기 어렵다.

### 맵리듀스 질의

MapReduce는 분산 처리를 위한 프로그래밍 모델이지만, 꽤 저수준이다. map과 reduce 함수는 **순수 함수(pure function)**여야 한다 — 입력 데이터만 사용하고, 외부 접근과 사이드 이펙트가 없어야 한다. 이 제약 덕분에 어떤 순서로든, 어디서든, 실패 시 재실행해도 결과가 보장된다.

- **map (collect)**: 각 문서에 대해 호출되어, 키-값 쌍을 뽑아낸다.
- **reduce (fold/inject)**: map이 뽑아낸 키-값 쌍들을 같은 키끼리 모아서 하나의 결과로 합친다

하지만 JavaScript 함수 2개를 직접 작성해야 하므로 복잡하고, 선언형 언어처럼 DB가 최적화하기 어렵다. 그래서 MongoDB 2.2부터 **Aggregation Pipeline**이라는 선언형 질의 언어가 추가되었다.
같은 결과를 내지만, 각 단계(`$match`, `$group`, `$sum`)가 DB가 이해할 수 있는 선언적 연산자이므로 내부적으로 최적화가 가능하다. SQL의 `WHERE`, `GROUP BY`, `SUM`과 사실상 같은 구조인 셈이다.

```jsx
db.observations.aggregate([
  { $match: { family: "Sharks" } },
  { $group: {
    _id: {
      year:  { $year:  "$observationTimestamp" },
      month: { $month: "$observationTimestamp" }
    },
    totalAnimals: { $sum: "$numAnimals" }
  }}
]);
```

### 그래프형 데이터 모델

다대다 관계가 다양한 데이터 모델을 구별하는 중요한 기능이다. 애플리케이션이 주로 일대다 관계이거나 레코드 간 관계가 없다면 문서 모델이 적합하지만, 데이터 간 연결이 복잡해질수록 그래프로 모델링하는 편이 자연스럽다.

그래프는 **정점(vertex)**과 **간선(edge)**으로 구성된다. 대표적인 예시로 소셜 네트워크, 웹 그래프, 도로·철도 네트워크 등이 있다.

### 속성 그래프 모델 (Property Graph)

각 정점은 고유 식별자, 나가는/들어오는 간선의 집합, 속성(키-값 쌍) 컬렉션을 갖는다. 각 간선도 고유 식별자, 시작/끝 정점, 관계를 설명하는 레이블, 속성 컬렉션을 갖는다.

핵심 유연성:

- 어떤 정점이든 다른 어떤 정점과 간선으로 연결할 수 있다. 스키마로 제한하지 않는다.
- 주어진 정점에서 나가는/들어오는 간선을 효율적으로 순회할 수 있다.
- 서로 다른 종류의 관계에 서로 다른 레이블을 사용하여, 하나의 그래프에 다양한 유형의 정보를 저장할 수 있다.

대표적인 질의 언어: **Cypher** (Neo4j에서 만든 선언적 질의 언어)

### 트리플 저장소와 SPARQL

모든 정보를 **(주어, 서술어, 목적어)** 형태의 세 부분으로 된 구문으로 저장한다. 예: (루시, 나이, 33), (루시, 결혼상대, 알랭)

- 목적어가 원시 데이터 타입의 값이면 → 주어 정점의 속성
- 목적어가 그래프의 다른 정점이면 → 서술어가 간선이 됨

질의 언어: **SPARQL** (RDF 데이터 모델에 기반)

### Datalog

트리플 저장소보다 더 오래된 언어. *서술어(주어, 목적어)* 형태로 데이터를 작성한다. 규칙(rule)을 정의하고 이를 결합·재사용하여 복잡한 질의를 작은 단계로 나눌 수 있다.

## 3장: 저장소와 검색

이 장의 핵심 질문: **"데이터베이스는 데이터를 어떻게 저장하고, 저장한 데이터를 어떻게 다시 찾는가?"**

### 가장 단순한 데이터베이스

```bash
db_set () { echo "$1,$2" >> database; }
db_get () { grep "^$1," database | sed -e "s/^$1,//" | tail -n1; }
```

- `db_set`: 키-값 쌍을 파일 끝에 추가(append). → **쓰기: O(1), 매우 빠름**
- `db_get`: 파일 전체를 스캔해서 마지막 값을 찾음. → **읽기: O(n), 레코드 수에 비례**

여기서 핵심 트레이드오프가 드러난다: **append-only 로그에 쓰기는 빠르지만, 읽기를 효율적으로 만들려면 별도의 자료 구조(인덱스)가 필요하다.**

### 로그(Log)란?

여기서 말하는 로그는 에러 기록이 아니라, **"추가만 가능한(append-only) 데이터 파일"**을 의미한다. 이미 쓴 내용은 절대 건드리지 않고, 새로운 내용이나 수정사항은 무조건 파일의 맨 마지막에 덧붙인다.

**왜 이렇게 비효율적으로 보이는 방식을 쓸까?**

1. **순차 쓰기(Sequential Write)가 랜덤 쓰기(Random Write)보다 훨씬 빠르다.** 
2. **동시성 문제 해결이 쉽다.** 각자 자기 내용을 끝에 붙이기만 하면 되므로 충돌을 피하기 쉽다.
3. **장애 복구(Crash Recovery)가 단순하다.** 제자리(in-place) 덮어쓰기 중 충돌하면 이전 값과 새 값이 뒤섞이지만, append-only면 마지막에 쓰다 만 부분만 잘라내면 된다.

**수정과 삭제는 어떻게?** 수정하고 싶으면 파일 끝에 같은 키로 새 값을 추가한다. 조회 시 가장 최신(마지막) 기록을 반환한다. 삭제는 **툼스톤(tombstone)**이라는 삭제 표시를 추가한다.

### 인덱스의 기본 아이디어

인덱스란 데이터베이스의 기본 데이터(primary data)에서 파생된 **추가적인 자료 구조**다. 질의 내용에는 영향을 주지 않고 오직 질의 성능에만 영향을 준다.

핵심 트레이드오프:

- 인덱스가 있으면 **읽기(질의)가 빨라진다.**
- 하지만 모든 **쓰기가 느려진다.** 데이터를 쓸 때마다 인덱스도 갱신해야 하기 때문이다.

이 때문에 데이터베이스는 모든 것을 자동으로 인덱싱하지 않고, 개발자/DBA가 질의 패턴을 기반으로 적절한 인덱스를 수동으로 선택한다.

### 해시 색인 (Hash Index)

가장 단순한 인덱싱 전략. 키-값 데이터에 대해 **인메모리 해시 맵**을 유지하면서, 각 키가 데이터 파일에서 어떤 **바이트 오프셋**에 위치하는지를 기록한다.

Bitcask(Riak의 기본 저장소 엔진)가 이 방식을 사용한다. 사용 가능한 RAM에 모든 키가 저장된다는 조건을 전제로 고성능 읽기/쓰기를 보장한다. 각 키에 대한 값이 자주 갱신되는 워크로드(동영상 조회수, URL별 방문자 수 등)에서 매우 잘 동작한다.

### 해시 색인의 동작 흐름

`put("user_1", 30)` 호출 시 두 가지가 **동시에** 일어난다:

1. **디스크(로그 파일)**: 파일 맨 끝에 `user_1: 30`을 append. 이 데이터의 시작 바이트 오프셋(예: 500)을 파악.
2. **메모리(해시 맵)**: 즉시 `user_1 → 500`으로 업데이트.

따라서 `get("user_1")` 호출 시 메모리에서 오프셋을 조회하여 디스크의 정확한 위치로 바로 접근할 수 있다.

> 메모리 인덱스의 오프셋 업데이트는 쓰기가 발생할 때마다 즉시 이루어진다. 컴팩션 때까지 기다리지 않는다. 그래야 방금 수정한 데이터를 조회했을 때 최신값을 볼 수 있다.
> 

> **디스크에도 키가 저장된다.** 메모리 해시 맵은 "지름길"이고, 디스크는 "원본(Source of Truth)"이다. 서버가 죽으면 메모리가 증발하므로, 디스크에 (Key, Value) 쌍이 통째로 있어야 재시작 시 해시 맵을 재구성할 수 있다.
> 

### 컴팩션과 세그먼트 병합

append-only 로그를 계속 쓰면 디스크 공간이 부족해진다. 이를 해결하기 위한 두 가지 작업:

**컴팩션(Compaction)** — "중복 데이터 삭제"

- 로그에서 중복된 키를 버리고, 각 키의 **가장 최신 갱신값만 유지**하는 것

**세그먼트 병합(Segment Merging)** — "파일 통합"

- 로그가 특정 크기에 도달하면 현재 세그먼트 파일을 닫고 새 세그먼트 파일에 쓰기 시작한다.
- 여러 세그먼트 파일을 하나로 합쳐서 파일 개수를 줄인다.

실제로는 이 두 작업이 **동시에** 수행된다. 여러 파일을 하나로 합치면서(병합), 중복된 키를 제거하고 최신값만 남긴다(컴팩션). 병합은 백그라운드에서 이루어지며, 완료되면 읽기 요청을 새 세그먼트로 전환하고 오래된 파일을 삭제한다.

### append-only 로그의 장점 (덮어쓰기 대비)

- **순차적 쓰기는 랜덤 쓰기보다 훨씬 빠르다.**
- **동시성과 장애 복구가 단순하다.** 제자리 덮어쓰기는 충돌 시 이전 값과 새 값이 뒤섞일 위험이 있지만, append-only면 기존 데이터를 건드리지 않으므로 더 안전하다.
- **세그먼트 병합으로 파일 조각화(fragmentation) 문제를 피할 수 있다.**

### 해시 색인의 한계

- **키가 전부 메모리에 들어가야 한다.** 키 수가 매우 많으면 문제가 된다. 디스크 기반 해시 맵은 가능하지만 성능이 좋지 않다(랜덤 I/O가 많고, 해시 충돌 처리가 복잡).
- **범위 질의(range query)에 비효율적이다.** `kitty00000` ~ `kitty99999` 같은 범위를 스캔하려면 모든 키를 개별적으로 조회해야 한다.

### SS테이블 (Sorted String Table, SSTable)

해시 색인 세그먼트와 유사하지만, **키-값 쌍이 키 순서대로 정렬되어 저장된다**는 점이 다르다.

### SSTable의 장점 (해시 색인 세그먼트 대비)

1. **세그먼트 병합이 효율적이다.** 여러 세그먼트가 각각 정렬되어 있으므로 **병합 정렬(merge sort)**의 원리를 적용할 수 있다. 같은 키가 여러 세그먼트에 있으면 가장 최신 세그먼트의 값을 취한다.
2. **인메모리 인덱스를 희소(sparse)하게 유지할 수 있다.** 모든 키의 오프셋을 기록할 필요 없이, 일정 간격마다만 기록하면 된다. "handbag"을 찾으려면 "handiwork"과 "handsome" 사이를 스캔하면 된다. 수 킬로바이트 정도만 스캔하므로 매우 빠르다.
3. **디스크 쓰기 전에 레코드들을 블록 단위로 묶어 압축할 수 있다.** 키가 정렬되어 있으면 비슷한 키들이 모여 있으므로 접두사 압축(delta encoding)과 블록 압축의 효율이 극대화된다. 희소 인메모리 색인의 각 항목이 압축된 블록의 시작을 가리키게 하면, 디스크 공간뿐 아니라 **I/O 대역폭 사용도 줄인다** — 디스크에서 메모리로 옮기는 물리적 데이터량이 줄어들기 때문이다.

> **참고: 키는 메모리에만 있는 것이 아니다.** 키 전체는 디스크(SSTable 파일)에 값과 함께 저장된다. 메모리에는 조회를 돕기 위한 일부 키(희소 색인, 랜드마크 역할)만 올라간다. 해시 색인에서도 디스크에 키가 저장되지만, 정렬되어 있지 않으므로 SSTable만큼 효율적으로 압축할 수 없다.
> 

> **"디스크에 쓰기 전에 압축한다"의 의미:** 이것은 "나중에 읽을 때(Read)를 대비해서, 쓸 때(Write) 미리 블록 단위로 압축해두는 것"이다. 읽기 요청 시 특정 범위의 키를 스캔해야 하는데, 해당 블록만 디스크에서 읽어오면 되고 블록이 압축되어 있으므로 물리적 I/O 양이 줄어든다.
> 

### SSTable을 생성하고 유지하는 방법

데이터는 키 순서대로 들어오지 않는데, 어떻게 정렬된 세그먼트를 만드는가? **메모리에서 정렬을 유지하는 것이 훨씬 쉽다.** 레드-블랙 트리나 AVL 트리 같은 균형 이진 탐색 트리를 사용한다.

**저장소 엔진의 동작 방식:**

1. **쓰기 요청** → 인메모리 균형 트리(**멤테이블, memtable**)에 추가
2. 멤테이블이 일정 크기(보통 수 MB)를 넘으면 → **SSTable 파일로 디스크에 기록** (트리가 이미 정렬되어 있으므로 순서대로 쓰면 됨). 새 쓰기는 새 멤테이블에 기록.
3. **읽기 요청** → 멤테이블 → 가장 최신 SSTable → 그 다음 SSTable → ... 순서로 탐색
4. 백그라운드에서 **컴팩션과 병합** 수행

**장애 복구:** 멤테이블은 메모리에만 있으므로 DB가 죽으면 손실된다. 이를 방지하기 위해 디스크에 별도의 **WAL(Write-Ahead Log)** — 정렬되지 않은 append-only 로그 — 을 유지한다. 모든 쓰기는 멤테이블에 추가하기 전에 먼저 WAL에 기록하고, 멤테이블이 SSTable로 플러시되면 해당 WAL은 버린다.

### LSM 트리 (Log-Structured Merge-Tree)

위에서 설명한 인덱싱 구조 — 정렬된 파일 병합과 컴팩션의 원리에 기반한 저장소 엔진 — 를 **LSM 저장소 엔진**이라 한다.

이름에 "로그 구조화(Log-Structured)"가 붙은 이유는 처음에 살펴본 로그(append-only) 방식의 철학을 계승하기 때문이다. 데이터를 디스크에 제자리 수정(in-place update)하지 않고, **새로운 데이터를 로그처럼 계속 덧붙이는 방식을 유지**하면서, 정렬(SSTable)과 병합(Merge) 기술을 더한 것이 LSM 트리다.

### LSM 트리 최적화

- **블룸 필터(Bloom Filter)**:
    - LSM 트리는 데이터를 찾을 때 **메모리(MemTable) → 최신 SSTable → 오래된 SSTable** 순서로 다 뒤져야 하므로 성능이 나쁘다. **블룸 필터**는 디스크를 읽기 전, 메모리에서 아주 적은 용량으로 "이 키가 여기 있을 가능성이 있는가?"를 먼저 판단한다. 따라서 **키가 DB에 존재하지 않음을 알려줘서** 불필요한 디스크 읽기를 피할 수 있다.
    - 작동 원리 : 키를 여러 개의 해시 함수로 돌려 비트 배열(0과 1)에 표시한다.
    - 특징
        - **False Positive (긍정 오류):** "데이터가 있다"고 해서 확인해 보니 실제로는 없을 수 있다.(해시 충돌 때문)
        - **No False Negative (부정 오류 없음):** **"데이터가 없다"고 하면 100% 없다.**
- **컴팩션 전략**: SSTable을 압축·병합하는 순서와 타이밍을 결정한다.
    - **크기 계층 컴팩션(size-tiered)**: 
    비슷한 크기의 SSTable이 일정 개수 이상 모이면, 이들을 합쳐서 **더 큰 하나의 SSTable**로 만드는 방식. 쓰기 작업이 단순해서 쓰기 성능이 매우 좋다. 하지만 같은 키가 여러 파일에 흩어져 있을 확률이 높아 읽기 성능은 떨어질 수 있다.  (HBase, Cassandra 지원)
    - **레벨 컴팩션(leveled)**: 
    데이터를 여러 '레벨(L0, L1, L2...)'로 나누고, 각 레벨마다 **키 범위가 겹치지 않는** 작은 SSTable들로 관리하는 방식. 각 레밸 내에서 키 범위가 명확히 분리되어 있어 읽기 성능이 뛰어나다. 공간 효율성이 좋지만, 파일을 자주 쪼개고 합쳐야 해서 쓰기 부하가 크다.(LevelDB, RocksDB, Cassandra 지원)
