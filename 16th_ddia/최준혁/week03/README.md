## 복제란?

네트워크로 연결된 여러 장비에 **동일한 데이터의 복사본을 유지**하는 것.

5장 전제: 데이터셋이 작아서 각 장비에 전체 복사본이 들어간다고 가정. 

진짜 어려운 건 **변경을 어떻게 전파할 것인가**.

**노드 간 변경 복제 알고리즘 3가지**

1. 단일 리더(Single-leader)
2. 다중 리더(Multi-leader)
3. 리더 없는(Leaderless)

각각 **동기 vs 비동기**, **잘못된 복제본 처리** 같은 트레이드오프가 따라온다. 대부분 DB 설정 옵션으로 노출됨.

---

## 리더 기반 복제 (Single-Leader)

가장 일반적인 해결책. PostgreSQL, MySQL, Oracle Data Guard, SQL Server AlwaysOn 등 대부분의 관계형 DB가 내장.

**구조**

- 복제 서버 중 하나를 **리더(leader)** 로 지정.
- 나머지는 **팔로워(follower)**.
- 클라이언트의 **쓰기는 무조건 리더에게만** 전송. 리더가 로컬 저장소에 먼저 기록.
- 리더는 변경 내용을 **복제 로그(replication log)** 또는 **변경 스트림**으로 팔로워에게 전송.
- 팔로워는 리더가 처리한 것과 **동일한 순서로** 쓰기를 적용해 로컬 복사본 갱신.
- 읽기는 리더 또는 팔로워 어디서든 가능.

### 동기식 vs 비동기식 복제

| 구분 | 동기식 | 비동기식 |
| --- | --- | --- |
| 동작 | 팔로워 응답 기다린 후 클라이언트에 성공 응답 | 팔로워 응답 안 기다림 |
| 일관성 | 리더와 팔로워가 일치 보장 | 팔로워가 뒤처질 수 있음 |
| 가용성 | 동기 팔로워가 죽으면 쓰기 불가 | 팔로워 다 죽어도 리더는 쓰기 처리 가능 |
| 지속성 | 보장 | 리더 장애 시 미복제 쓰기 유실 |

**현실적 선택: 반동기식(semi-synchronous)**

모든 팔로워를 동기식으로 두는 건 비현실적이다. 

보통 **팔로워 하나만 동기식, 나머지는 비동기식**으로 한다. 이러면 최소 2개 노드(리더 + 동기 팔로워)에 데이터가 보장된다.

> **왜 비동기가 데이터를 잃을 수 있는지**
비동기 방식은 "읽씹"이다. 
클라이언트가 "데이터 A 저장!" 요청 → 리더가 자기 디스크에 A 적자마자 즉시 "성공!" 응답 → 그 후에 팔로워들에게 "이거 복제해" 메시지 전송. 
만약 리더가 "성공" 응답 직후, 팔로워에 메시지 보내기 전에 터지면? 클라이언트는 성공 받았는데 시스템 어디에도 그 데이터가 없게 된다. **지속성(Durability) 원칙이 깨지는 지점**.
> 

### 새로운 팔로워 설정

새 팔로워가 리더의 데이터를 정확히 복사하려면:

1. 리더의 **스냅숏**을 가져온다.
2. 스냅숏을 새 팔로워에 복사한다.
3. 팔로워는 리더에 연결해 스냅숏 **이후** 발생한 모든 데이터 변경을 요청 (스냅숏이 리더 복제 로그의 정확한 위치와 연관돼야 함).
4. 팔로워가 미처리 변경분을 다 처리하면 따라잡기 완료.

### 노드 중단 처리

**팔로워 장애: 따라잡기 복구(Catch-up Recovery)**

팔로워가 자기 로그에서 결함 직전 마지막 트랜잭션을 알아낸 뒤, 리더에 그 이후 데이터 변경을 요청하면 됨. 비교적 단순.

**리더 장애: 장애 복구(Failover)**

복잡함. 자동 장애 복구 단계:

1. 리더가 장애인지 판단 (보통 **타임아웃**으로).
2. 새 리더 선택 (보통 **최신 데이터를 가진 복제 서버**).
3. 시스템을 새 리더 사용하도록 재설정.

**장애 복구가 잘못될 수 있는 지점들**

- 새 리더가 옛 리더의 미복제 쓰기를 받지 못한 경우 → 해당 쓰기를 폐기해야 함 → 외부 시스템과 정합성 깨질 위험.
- **스플릿 브레인(split brain)**: 두 노드가 동시에 자신을 리더라고 믿음.
- 리더가 진짜 죽지 않았는데 죽었다고 잘못 판단.

## 복제 로그 구현

### 1. 구문 기반 복제 (Statement-based)

리더가 실행한 SQL 문 자체를 팔로워에게 전송.

**문제**: 비결정적 함수(`NOW()`, `RAND()`)가 있으면 팔로워에서 다른 결과가 나옴. 자동증가 칼럼·트리거도 문제.

→ 거의 안 쓰임. MySQL은 비결정성 감지하면 **로우 기반 복제로 자동 전환**.

### 2. 쓰기 전 로그(WAL) 배송

저장소 엔진이 디스크에 변경을 기록하는 그 로그(WAL)를 그대로 팔로워에 전송.

**장점**: 구현 간단, 정확.

**치명적 단점**: 로그가 **너무 저수준**임. "7번 디스크 블록의 10번째 바이트를 'A'에서 'B'로 바꿔라" 수준의 정보. 저장소 엔진과 강하게 결합됨.

→ 결과적으로 **리더와 팔로워의 DB 소프트웨어 버전이 반드시 일치해야 한다**. 

무중단 롤링 업그레이드가 불가능하다는 뜻이라 운영상 큰 부담.

### 3. 논리적 로그 복제 (Row-based) — 가장 많이 쓰임

복제 로그를 저장소 엔진과 분리. 로우 단위로 변경을 기록한다.

- **삽입(Insert)**: 모든 칼럼의 새 값
- **삭제(Delete)**: 로우를 식별할 정보 (보통 PK)
- **갱신(Update)**: 식별 정보 + 모든 칼럼의 새 값

**장점**

- 저장소 엔진과 분리돼 **버전 불일치 허용** → 무중단 롤링 업그레이드 가능.
- 외부 애플리케이션이 파싱하기 쉬움 → **변경 데이터 캡처(CDC, Change Data Capture)** 의 기반. Kafka, Elasticsearch, BigQuery로 변경 사항 흘려보내는 데 활용.

MySQL의 binlog(row 모드)가 이 방식.

### 4. 트리거 기반 복제

DB의 트리거 기능으로 변경을 별도 테이블에 기록 → 외부 프로세스가 그걸 읽어서 다른 시스템에 복제.

**장점**: 매우 유연. 데이터 서브셋만 복제, 다른 종류의 DB로 복제, 사용자 정의 충돌 해소 가능.

**단점**: 오버헤드 크고, 디버깅 어렵고, 버그 잘 생김.

오라클 GoldenGate가 대표적.

## 복제 지연 문제

비동기 복제에서 팔로워가 리더보다 뒤처지는 게 정상이다. 이게 일시적이면 **최종적 일관성(eventual consistency)** 으로 해결되지만, 그동안 사용자가 마주칠 수 있는 이상 현상들이 있다.

### 1. 쓰기 후 읽기 (Read-after-write)

내가 방금 쓴 글이 안 보이는 현상.

**시나리오**: 사용자가 프로필 사진 변경(리더에 저장) → 즉시 새로고침 → 아직 미복제된 팔로워에서 읽음 → 옛날 사진이 보임.

**해결책**

1. **사용자가 수정한 데이터는 리더에서 읽기**. 단, 대부분이 자기 데이터를 편집하면 리더에 부하 몰림.
2. **마지막 갱신 후 1분 동안만 리더에서 읽기** 같은 시간 기반 라우팅.
3. 클라이언트가 **마지막 쓰기의 타임스탬프 기억** → 그 이상 따라잡은 팔로워에서만 읽기.
4. 데이터센터가 여러 개면 **리더가 있는 데이터센터로 라우팅**.

디바이스 간(폰 + PC) 일관성은 더 까다롭다. 중앙집중식 메타데이터 필요.

### 2. 단조 읽기 (Monotonic Reads)

**시간이 거꾸로 흐르는 현상**을 방지. 사용자가 같은 쿼리를 두 번 했는데 두 번째가 더 옛날 데이터를 반환하면 안 된다.

**해결책**: 각 사용자의 읽기를 **항상 동일한 복제 서버에서 처리** (사용자 ID 해싱 등).

### 3. 일관된 순서로 읽기 (Consistent Prefix Reads)

**인과성 위반** 방지. 질문이 답변보다 늦게 보이면 안 된다.

**해결책**: 인과성 있는 쓰기를 **동일한 파티션에 기록**. 또는 인과 관계를 명시적으로 추적하는 알고리즘 사용.

### 결국엔 트랜잭션이다

> 애플리케이션 개발자가 미묘한 복제 문제를 걱정하지 않고 "올바른 작업 수행"을 위해 항상 데이터베이스를 신뢰할 수 있다면 훨씬 좋다. **이것이 트랜잭션이 있는 이유다.**
> 

단일 노드 트랜잭션은 오래전부터 있었지만, 분산 DB로 가면서 많은 시스템이 트랜잭션을 포기했다. 성능과 가용성 때문. 최근엔 다시 분산 트랜잭션을 지원하는 NewSQL(Spanner, CockroachDB)이 부상 중.

## 다중 리더 복제 (Multi-Leader)

단일 리더의 치명적 단점: **리더에 연결 못 하면 쓰기 불가능**. 이걸 해결하려고 쓰기를 허용하는 노드를 여러 개 두는 게 다중 리더.

### 사용 사례

### (1) 다중 데이터센터 운영

각 데이터센터마다 리더 하나씩 둔다. 데이터센터 내부는 단일 리더처럼 동작, 데이터센터 간은 비동기 복제.

| 측면 | 단일 리더 | 다중 리더 |
| --- | --- | --- |
| 성능 | 모든 쓰기가 리더 DC로 → 지연 큼 | 로컬 DC에서 처리 → 빠름 |
| DC 중단 내성 | 리더 DC 죽으면 장애 복구 필요 | 다른 DC가 독립적으로 동작 |
| 네트워크 문제 내성 | DC 간 동기 복제 → 민감 | 비동기 → 잘 견딤 |

> **왜 다중 리더에서 DC 간 복제가 비동기일 수밖에 없는가**
동기로 하면 다중 리더의 장점이 다 사라진다. 서울 리더에 쓸 때마다 뉴욕·런던 리더의 응답을 기다리면 RTT가 수백 ms씩 박힘. 또 한 DC만 죽어도 전체 쓰기가 멈춤. 그래서 "지구 반대편이 대답할 때까지 기다리지 않고, 일부 서버 죽어도 내 서버는 돌아가게" 하기 위해 비동기를 선택. **대신 충돌이라는 숙제를 짊어진다.**
> 

### (2) 오프라인 작동 클라이언트

캘린더 앱이 비행기 모드에서도 작동하는 이유. **각 디바이스가 작은 리더**다. 인터넷 연결되면 디바이스 간 비동기 복제.

> 아키텍처 관점에서 보면 이건 **데이터센터 간 다중 리더 복제와 동일**하다. 디바이스 = 데이터센터, 디바이스 간 네트워크 = 극히 신뢰할 수 없는 네트워크. 복제 지연이 며칠씩 걸릴 수 있다.
> 

CouchDB가 이 모드 지원.

### (3) 협업 편집

구글 독스, 노션 같은 실시간 협업. 사용자 한 명 한 명이 리더처럼 동작 → 로컬에 즉시 반영 → 비동기로 서버·다른 사용자에 전파. 충돌 해소 필요.

### 쓰기 충돌 다루기

다중 리더의 **가장 큰 문제**.

**동기식 충돌 감지는 의미 없다**

이론상 가능은 하지만, 그러려면 모든 리더가 응답할 때까지 기다려야 함 → 다중 리더의 장점(독립적 쓰기) 다 잃음 → **차라리 단일 리더 쓰는 게 낫다.**

### 충돌 회피 (가장 간단)

**특정 레코드의 모든 쓰기가 동일한 리더를 거치도록** 보장. 충돌이 아예 안 생기게 만든다.

> 많은 서비스에서 "특정 레코드의 주인 = 특정 사용자"이기 때문에, 사용자 단위로 라우팅을 고정하면 결과적으로 그 사용자의 레코드는 항상 같은 리더만 보게 됨. 사용자 입장에선 단일 리더처럼 동작.
> 

방법: 사용자 ID 해싱, 고정 세션. 다만 데이터센터 장애 시 페일오버하면 다시 충돌 가능성 생김.

### 일관된 상태로 수렴

다중 리더는 쓰기 순서가 정해지지 않아 최종 값이 무엇인지 명확하지 않다. 리더 1은 C, 리더 2는 B로 끝나면 안 된다. **모든 복제본이 최종적으로 동일한 값으로 수렴**해야 한다.

**수렴 충돌 해소 방법**

1. **최종 쓰기 승리 (LWW, Last Write Wins)**: 각 쓰기에 고유 ID/타임스탬프 부여하고 가장 큰 거 선택. 대중적이지만 **데이터 유실 위험**.
2. 복제 서버 ID로 우선순위 결정.
3. 값들을 어떻게든 병합 (예: B와 C를 알파벳 순으로 합쳐서 "B/C").
4. 충돌을 명시적 데이터 구조로 보존 → 애플리케이션 코드가 나중에 해소.

### 사용자 정의 충돌 해소 로직

대부분의 다중 리더 도구는 애플리케이션 코드로 해소.

- **쓰기 시점**: 변경 사항 로그에서 충돌 감지하면 핸들러 호출.
- **읽기 시점**: 충돌 쓰기를 모두 저장 → 다음 읽기에서 여러 버전 반환 → 애플리케이션이 해소 → 결과를 다시 DB에 기록 (CouchDB 방식).

### 자동 충돌 해소 연구

- **CRDT (Conflict-free Replicated Data Type)**: 셋, 맵, 정렬 목록, 카운터를 위한 데이터 구조. 동시 편집 + 자동 해소.
- **방향 가능한 영속 데이터 구조**: 깃처럼 명시적 히스토리 추적 + 삼중 병합.
- **운영 변환(OT, Operational Transformation)**: 이더패드, 구글 독스의 협업 편집 알고리즘.

### 다중 리더 복제 토폴로지

쓰기를 한 노드에서 다른 노드로 어떻게 전달할지의 통신 경로.

| 토폴로지 | 구조 | 특징 |
| --- | --- | --- |
| 전체 연결 (All-to-All) | 모든 리더가 다른 모든 리더에게 직접 전송 | 내결함성 좋음 |
| 원형 (Ring, MySQL) | A→B→C→D→A | 한 노드 죽으면 흐름 끊김 |
| 별 (Star) | 중앙 노드가 다른 모든 노드에 전달 | 중앙 노드 = 단일 장애점 |

**무한 복제 루프 방지**: 각 노드에 고유 ID, 복제 로그에 거쳐온 모든 노드 ID 태깅. 이미 처리한 ID 보면 무시.

> 원형/별 모양은 중간에 놓치면 그 뒤는 다 못 받음. 전체 연결은 길이 여러 개라 한두 군데 끊겨도 우회 가능.
> 

**전체 연결의 함정: 인과성 깨짐**

전체 연결이 더 안전해 보이지만, 일부 네트워크 경로가 다른 경로보다 빠르면 메시지가 추월될 수 있음. 

갱신이 삽입보다 먼저 도착할 수도 있음.

**해결책: 버전 벡터(version vector)**

타임스탬프로는 부족하다 — 노드 간 시계가 정확히 동기화됐다고 신뢰할 수 없기 때문(**클록 스큐, clock skew**).

> 책에선 "PostgreSQL BDR은 쓰기의 인과적 순서를 제공하지 않으며 MySQL Tungsten Replicator는 충돌 감지 시도조차 하지 않는다"고 언급. **유명한 도구들도 이걸 완벽히 구현 못함.**
>

## 리더 없는 복제 (Leaderless)

리더 개념을 버리고 **모든 복제 서버가 클라이언트 쓰기를 직접 받는** 방식. Amazon Dynamo가 부활시킨 후 다이나모 스타일이라 부름. Riak, Cassandra, Voldemort.

### 노드 다운 시 쓰기

리더 기반은 장애 복구 필요. 리더 없는 방식은 **장애 복구 자체가 필요 없다**.

- 클라이언트는 읽기를 **여러 노드에 병렬로** 보냄.
- 응답이 다를 수 있음 (어떤 건 최신, 어떤 건 옛날) → **버전 번호로 최신 결정**.

### 읽기 복구와 안티 엔트로피

복제본 동기화를 위한 두 메커니즘.

**읽기 복구 (Read Repair)**

클라이언트가 여러 노드 병렬 읽기 시 옛날 응답 발견하면 → 그 복제본에 새 값 즉시 기록. 사용자가 "수동 수리"하는 느낌.

**안티 엔트로피 (Anti-Entropy)**

백그라운드 프로세스가 복제본 간 차이를 지속적으로 찾아서 동기화. 정기 점검 같은 것. 

특정한 순서로 복사하지 않아 **상당한 지연** 가능.

> **둘이 상호 보완적인 이유**
읽기 복구는 **자주 읽는 데이터(hot data)** 만 고침. 1년에 한 번도 안 읽히는 데이터는 영원히 안 고쳐짐. 그 사이 복제본 한두 개가 죽으면 데이터 자체가 사라질 수 있음. 안티 엔트로피는 **차가운 데이터(cold data)** 까지 샅샅이 검사해서 지속성을 보장. Cassandra의 `nodetool repair`가 이걸 수행.
> 

### 읽기/쓰기 정족수 (Quorum)

`n`개 복제 서버에서 모든 쓰기는 `w`개에 성공해야 확정, 모든 읽기는 최소 `r`개에 질의.

**핵심 공식: `w + r > n`**

이걸 만족하면 읽기 시 최신 값을 얻을 것으로 기대. 쓴 노드 셋과 읽은 노드 셋이 **반드시 1개 이상 겹치기 때문**.

> **3인 1조 비밀 투표 비유**
n=3, w=2, r=2일 때: 쓰기 단계에서 3명 중 2명(A, B)에게 새 데이터 저장 → 읽기 단계에서 아무 2명에게 물어봐도 그중 한 명은 무조건 A나 B라서 최신 정보를 가지고 있음. 더 최신 버전 골라내면 끝.
> 

**일반적 선택**: `n`을 홀수로, `w = r = (n+1)/2`. n/2 노드 장애까지 견딤.

| 설정 (n=3) | 특징 | 트레이드오프 |
| --- | --- | --- |
| w=2, r=2 | 균형형 | 무난 |
| w=3, r=1 | 읽기 최적화 | 1대만 죽어도 쓰기 불가 |
| w=1, r=3 | 쓰기 최적화 | 읽기 느림 |

읽기/쓰기는 항상 n개 모두에 병렬 전송. w와 r은 **얼마나 기다릴지**를 결정할 뿐.

### 정족수 일관성의 한계

`w + r > n`이 **수학적 보장**이지 **절대적 보장**은 아니다. 현실에서 옛날 값 반환되는 에지 케이스들:

| 시나리오 | 무슨 문제 |
| --- | --- |
| 느슨한 정족수(sloppy quorum) | w 노드와 r 노드가 안 겹칠 수 있음 |
| 동시 쓰기 | 어느 게 먼저인지 불명 → LWW + 클록 스큐로 데이터 유실 |
| 쓰기와 읽기 동시 발생 | 일부 복제본만 반영된 상태에서 읽힘 |
| 부분 성공 후 실패 | 성공한 복제본은 롤백 안 됨 → "실패한 쓰기"가 나중에 읽힐 수 있음 |
| 새 값 가진 노드 죽음 | 옛 값 가진 노드가 복구되며 정족수 깨짐 |

> 정족수 설정은 **확률을 조절하는 도구**일 뿐, 절대적 보장이 아니다. "내가 쓴 글은 바로 보여야 한다" 같은 강한 보장은 트랜잭션이나 합의가 필요. (7장, 9장에서 다룸)
> 

### 느슨한 정족수와 암시된 핸드오프

네트워크 장애로 클라이언트가 정족수를 못 채울 때 두 가지 선택:

1. 모든 요청에 에러 반환 (엄격한 정족수)
2. 일단 쓰기 받고, 보통 저장되는 n개 노드는 아니지만 **연결 가능한 다른 노드에 기록** (느슨한 정족수)

후자를 **느슨한 정족수(sloppy quorum)** 라 한다.

**암시된 핸드오프(hinted handoff)**: 임시 보관한 노드는 "이건 원래 A가 받아야 했어"라는 힌트를 붙여뒀다가, A가 복구되면 전달하고 자기는 삭제.

**트레이드오프**: 가용성 ↑, 일관성 ↓. 임시 노드에 저장된 동안은 `w + r > n` 보장이 깨질 수 있음. 옆집에 임시로 맡겨둔 택배를 원래 집에서 찾으려 하면 못 찾는 상황.

### 다중 데이터센터 운영

리더 없는 복제도 다중 데이터센터에 적합. 동시 쓰기 충돌, 네트워크 중단, 지연 시간 급증을 모두 허용하기 때문.

- n개의 복제 서버에는 **모든** 데이터센터 **노드 포함**.
- 쓰기는 모든 데이터센터의 모든 복제본에 전송되지만, 클라이언트는 보통 **로컬 데이터센터 안에서만 정족수 응답**을 기다림.
- 다른 데이터센터로의 쓰기는 비동기.

→ 데이터센터 간 연결 지연·중단에 영향 안 받음. Cassandra가 글로벌 서비스에 인기 있는 이유.

## 동시 쓰기 감지

다이나모 스타일은 여러 클라이언트가 같은 키에 동시 쓰기를 허용 → 엄격한 정족수에서도 충돌 발생. 노드마다 도착 순서가 다를 수 있음.

각 노드가 단순히 덮어쓰면 영구적으로 일관성이 깨진다. **데이터 손실을 피하려면 애플리케이션 개발자가 충돌 처리 방식을 알아야 한다.**

### 최종 쓰기 승리 (LWW)

각 쓰기에 타임스탬프 → 가장 큰 게 승리. 다른 건 버림.

**문제**: 동시에 성공한 여러 쓰기 중 하나만 남아 **지속성이 희생됨**. 클록 스큐로 인한 데이터 유실도 발생.

LWW로 안전하게 쓰는 유일한 방법: **키를 한 번만 쓰고 이후엔 불변 값으로 다루기**.

### "이전 발생" 관계와 동시성

- A가 B의 **이전 발생(happens-before)**: B가 수행될 때 A를 알았거나 의존하거나 기반으로 함.
- **동시(concurrent)**: 어느 작업도 다른 작업을 모름.

판단 기준은 **시간이 아니라 인과율**. "내가 작업 시작할 때 상대 결과를 봤는가?"

### 이전 발생 관계 파악하기 (5-13 그림 단계별 분석)

서버는 **버전 번호로 두 작업이 동시인지 결정**. 값 자체를 해석할 필요는 없어서 값을 데이터 구조로 사용 가능.

**알고리즘**

1. 서버는 모든 키에 버전 번호 유지, 키 기록할 때마다 증가.
2. 클라이언트가 읽으면 서버는 **최신 버전뿐 아니라 덮어쓰지 않은 모든 값** 반환.
3. 클라이언트가 기록할 때 **이전 읽기의 버전 번호 포함**, 이전 읽기에서 받은 모든 값 합쳐야 함.
4. 서버가 특정 버전 번호의 쓰기 받으면, **해당 버전 이하 값 모두 덮어쓸 수 있음**. 더 높은 버전 값은 유지.

**장바구니 5단계 시뮬레이션**

| 단계 | 동작 | 서버 상태 |
| --- | --- | --- |
| 1 | A가 [우유] 추가 (이전 버전 없음) | v1: [우유] |
| 2 | B가 [계란] 추가 (이전 버전 없음, v1 못 봄) | v1: [우유], v2: [계란] (형제) |
| 3 | A가 v1 보고 [밀가루] 추가 ("v1까지 봤어") | v2: [계란], v3: [우유, 밀가루] (v1 우유는 덮어씀) |
| 4 | B가 v2 보고 [햄] 추가 ("v2까지 봤어") | v3: [우유, 밀가루], v4: [계란, 햄] (v2 계란은 덮어씀) |
| 5 | A가 v3, v4 둘 다 보고 [베이컨] 추가 ("v3, v4 다 봤어") | v5: [우유, 밀가루, 계란, 햄, 베이컨] |

**핵심**:

- 클라이언트가 "n번까지 봤어"라고 하면 → 서버는 **n번 이하 모두 덮어씀**.
- 클라이언트가 못 본 버전은 → **형제로 보존**.
- 마지막에 클라이언트가 모든 형제를 합쳐서 쓰면 → 비로소 하나로 수렴.

### 동시에 쓴 값 병합

알고리즘이 데이터 자동 삭제를 막아주지만, **클라이언트가 형제를 직접 병합**해야 함. 이건 다중 리더 충돌 해소와 본질적으로 같은 문제.

**단순 합집합의 함정**: 삭제가 들어가면 망함.

```
원래 장바구니: [우유]
A: [우유] + [계란] = [우유, 계란]
B: [우유] 삭제 = []
병합 시 합집합 → [우유, 계란]  ← 삭제했던 우유가 부활!
```

**해결책: 톰스톤(Tombstone)**

데이터를 진짜 삭제하지 않고 "이 데이터는 vX에서 삭제됨"이라는 묘비를 세움. 병합 시 톰스톤 보면 그 항목 제외.

> 무덤에 비석 세우는 것과 같다고 해서 톰스톤. 카산드라에서 자주 듣게 됨. 톰스톤이 누적되면 디스크 차고 읽기도 느려져서 주기적으로 컴팩션으로 정리.
> 

병합은 복잡하고 오류 잘 남. 이걸 자동화하려는 게 CRDT.

### 버전 벡터

지금까지는 단일 복제본 기준. **다중 복제본 + 리더 없음**이면 단일 버전 번호로는 부족하다.

→ **키당 + 복제본당** 버전 번호를 쓴다. 각 복제본은 자기 쓰기 처리 시 자기 카운터 증가, 다른 복제본 카운터도 추적.

전체 모음을 **버전 벡터(version vector)** 라 함.

```
{A: 2, B: 1, C: 0}  ← "A에서 2번, B에서 1번, C는 0번 쓰기 발생"
```

**규칙**

- 데이터 읽을 때 서버는 버전 벡터를 클라이언트에 전송.
- 다시 쓸 때 클라이언트는 받은 벡터를 함께 전송.
- 서버는 두 벡터 비교:
    - 한 벡터가 다른 벡터를 **모든 차원에서 크거나 같음** → 인과 관계, 덮어쓰기.
    - 한쪽은 크고 한쪽은 작음 → **동시 쓰기**, 형제로 보존.

이 구조 덕분에 한 복제본 읽고 다른 복제본에 쓰는 게 **안전**해진다. 형제가 생겨도 올바르게 병합되면 데이터 유실 없음.

## 정리

- 복제는 **고가용성, 지연 시간 단축, 처리량 확장**을 위한 핵심 기술.
- **단일 리더**: 가장 단순. 동기/비동기/반동기 트레이드오프. 페일오버가 까다로움.
- **복제 로그**: 구문 기반 → WAL → 논리적 로그(가장 유연, CDC 기반) → 트리거 기반(가장 유연하지만 무거움).
- **복제 지연**: 쓰기 후 읽기, 단조 읽기, 일관된 순서로 읽기 — 각각 별도 해결책 필요. 결국 트랜잭션이 답.
- **다중 리더**: 다중 DC, 오프라인 작동, 협업 편집에 유리. 충돌이 핵심 숙제.
- **충돌 해소**: 회피(라우팅 고정) → LWW → 사용자 정의 → CRDT/OT.
- **리더 없는 복제**: 다이나모 스타일. 정족수 `w + r > n`이 핵심.
- **정족수의 한계**: 수학적으로 맞아도 현실 에지 케이스에서 깨짐. 느슨한 정족수는 가용성과 일관성을 더 트레이드.
- **버전 벡터**: 다중 복제본 환경에서 인과성 추적의 핵심 도구. 서버는 이걸로 덮어쓰기 vs 형제 보존을 결정.
- **현실의 백엔드 설계**: 데이터 정합성이 절대적이면 합의 알고리즘(Raft, Paxos), 성능이 우선이면 정족수 + 애플리케이션 레벨 충돌 해소.
