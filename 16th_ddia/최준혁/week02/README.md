## 3장 후반: 저장소 엔진과 분석

### B 트리 (B-Tree)

가장 널리 쓰이는 색인 구조. LSM 트리와는 접근 방식이 완전히 다르다.

**핵심 구조**

- LSM 트리는 가변 크기 세그먼트를 순차 기록하지만, B 트리는 **4KB 고정 크기 페이지** 단위로 쪼개서 읽기/쓰기를 한다.
- 한 페이지에서 하위 페이지를 참조하는 수를 **분기 계수(branching factor)**라고 한다. 보통 수백 개라 수십억 건 데이터도 깊이 3~4 레벨.
- 루트 → 내부 노드 → 리프 노드로 내려가 개별 키를 찾음.
- **루트와 내부 노드는 "몇부터 몇까지는 이쪽 자식으로 가라"는 범위 안내판 역할**만 하고, **실제 값(또는 값을 가리키는 포인터)은 리프 노드에만 존재**한다. 트리 탐색은 루트에서 범위를 따라 내려가다가 리프 노드에 도달하면 거기서 값을 꺼내는 방식.

**쓰기 동작**

- 키 업데이트: 리프 페이지 찾아서 값 바꾼 뒤 다시 씀.
- 새 키 삽입: 공간 있으면 삽입, 없으면 **페이지 분할(split)** → 부모 페이지도 업데이트.

**신뢰할 수 있는 B 트리 만들기**

B 트리 쓰기는 **제자리 갱신(in-place)** 방식이라 위험하다. 페이지 분할 도중 크래시 나면 **고아 페이지**가 생길 수 있다.

해결책은 **쓰기 전 로그(WAL)**. 트리 페이지에 변경 적용 **전에** 모든 변경을 추가 전용 파일에 먼저 기록한다. 크래시 시 로그 재생으로 복구.

> B 트리는 모든 데이터를 **최소 두 번** 기록한다. WAL 한 번, 트리 페이지 한 번. 이게 쓰기 증폭의 원인.

동시성 제어는 **래치(latch)**라는 가벼운 락으로 처리.

> **래치(latch)란?**
> 일반적인 락(lock)은 트랜잭션 단위로 오래 걸려있는 반면, 래치는 자료구조(B 트리 페이지)를 잠깐 보호하기 위한 **가벼운 락**이다. 수십 나노초~밀리초 수준으로 짧게 걸었다 푼다. 두 스레드가 동시에 같은 페이지를 분할하거나 수정하려 할 때 충돌을 막는 용도.

**B 트리 vs LSM 트리**

| 항목 | B 트리 | LSM 트리 |
|---|---|---|
| 쓰기 | 제자리 갱신 | append만 |
| 읽기 | 일반적으로 빠름 | 여러 SSTable 확인 필요 |
| 쓰기 증폭 | WAL + 페이지 재기록 | 컴팩션에서 발생 |
| 쓰기 처리량 | 상대적으로 낮음 | **더 높음** |
| 공간 | 페이지 단편화 | **압축률 좋음** |

LSM은 append-only 순차 쓰기라 처리량이 높지만, 컴팩션이 백그라운드에서 돌면서 상위 백분위 응답시간이 길어질 수 있다.

### 기타 색인 구조: 값을 어디에 저장할 것인가?

키는 찾는 수단, 값을 어디에 두느냐가 또 하나의 선택지.

- **힙 파일(Heap File)**: 색인엔 포인터만, 실제 데이터는 별도 힙에. PostgreSQL 기본 방식.
- **클러스터드 인덱스**: 색인 안에 로우 데이터 자체가 저장됨. MySQL InnoDB PK가 대표적. PK 조회는 빠르지만 보조 인덱스는 double lookup 발생.
- **커버링 인덱스**: 쿼리가 필요로 하는 일부 칼럼을 인덱스에 포함. 테이블 접근 생략 가능하지만 저장 공간·쓰기 비용 증가.

> **MySQL InnoDB는 정말 인덱스에 값이 들어있다**
> 우리가 흔히 쓰는 MySQL(InnoDB 엔진)은 기본키(PK) 기준으로 **클러스터드 인덱스** 구조다. 즉 테이블 자체가 PK로 정렬된 B 트리이고, **리프 노드에 로우 전체가 저장**돼 있다. `SELECT * FROM users WHERE id = 1` 같은 PK 조회는 인덱스 타고 내려가면 바로 값이 나오므로 빠르다.
>
> 반면 보조 인덱스(예: `INDEX(email)`)는 리프 노드에 로우가 아니라 **PK 값**이 저장된다. 그래서 `SELECT * FROM users WHERE email = ?`는:
> 1. email 인덱스 타고 내려가서 PK 값 획득
> 2. 그 PK로 클러스터드 인덱스 다시 타고 내려가서 로우 획득
>
> 이걸 **double lookup**이라고 부른다. PK 조회보다 한 단계 더 거치는 것.

**루씬의 인메모리 색인**

루씬도 용어 사전에 SSTable 같은 구조를 쓰는데, 인메모리 색인이 독특하다.

- **LevelDB**: 1000개 단어마다 하나씩만 메모리에 기록하는 **희소 컬렉션**. 백과사전 옆면의 '가/나/다' 인덱스랑 같은 원리.
- **루씬**: 다뤄야 할 용어가 너무 많아서 희소 컬렉션으로도 부족. **트라이(Trie)**로 접두사를 공유하고, **유한 상태 오토마톤(FSA)**으로 접미사까지 합쳐서 극단적으로 압축한다.

### 트랜잭션 처리 vs 분석 (OLTP vs OLAP)

| 속성 | OLTP | OLAP |
|---|---|---|
| 접근 패턴 | 키로 소량 레코드 조회/수정 | 대량 레코드 스캔 + 집계 |
| 쓰기 | 빈번 (사용자 입력) | 거의 없음 (배치 ETL) |
| 중요한 것 | 낮은 지연시간 | 높은 처리량 |

**분리 이유: 데이터 웨어하우스**

분석가가 OLTP DB에 즉석 질의를 날리면 서비스 성능이 망가진다. 그래서 **ETL**로 주기적으로 별도 DB(데이터 웨어하우스)에 복제한다.

> **ETL (Extract - Transform - Load)**
> OLTP DB의 데이터를 분석용 DB로 옮기는 파이프라인.
> - **Extract(추출)**: 운영 DB에서 데이터 뽑아오기
> - **Transform(변환)**: 분석하기 좋은 형태로 가공 (컬럼 정리, 타입 변환, 조인·집계 등)
> - **Load(적재)**: 데이터 웨어하우스에 넣기
>
> 보통 새벽에 배치로 돌린다. "어제 하루치 데이터를 새벽 2시에 DW로 밀어넣기" 같은 식.

DW 스키마는 보통 **별(Star) 스키마** — 이름대로 팩트 테이블이 가운데, 차원 테이블이 광선처럼 둘러싸는 모양.

> **DW 스키마란?**
> 데이터 웨어하우스(Data Warehouse) 안에 테이블을 어떻게 설계할지에 대한 구조. 운영 DB는 정규화해서 중복을 줄이는 게 원칙이지만, DW는 분석 속도를 위해 **별 스키마**처럼 비정규화된 구조를 많이 쓴다.

- **팩트 테이블**: "누가 언제 어디서 뭘 했다"는 이벤트 로우.
- **차원 테이블**: 이벤트의 속성들. 팩트 테이블이 외래 키로 참조.

차원이 하위차원으로 더 세분화되면 **눈꽃송이(Snowflake) 스키마**.

### 칼럼 지향 저장소

OLAP 질의는 **수백 개 칼럼 중 몇 개만** 선택해서 수억 행을 스캔한다. 로우 지향이면 필요 없는 칼럼까지 다 읽어야 하니까 낭비.

핵심 아이디어: **같은 칼럼 값들을 연속된 블록에 몰아서 저장**. 필요한 칼럼만 읽으면 되니 I/O가 확 준다.

적용 제품: BigQuery, Redshift, ClickHouse, Parquet.

---

### 칼럼 압축

같은 칼럼 값은 카디널리티(고유 값 수)가 낮은 경우가 많아 압축이 잘 먹힌다.

- **비트맵 부호화**: 고유 값마다 비트맵 생성. `country='KR'`이면 해당 로우 위치에 1, 아니면 0. `WHERE country IN ('KR','JP')` 같은 쿼리가 비트맵 OR 연산 한 방.
- **런 렝스 부호화(RLE)**: "A가 1000번 연속" 식으로 요약. 비트맵이 듬성듬성할 때 효율적.

**메모리 대역폭과 벡터화 처리**

칼럼 압축의 진짜 위력은 CPU 주기를 효율적으로 쓰는 것.

1. 로우 지향은 타입이 섞여있어 CPU가 매번 분기·함수 호출 → 주의 분산.
2. 칼럼 저장소는 같은 타입만 모여있어 **타이트 루프**로 빠르게 반복.
3. 압축된 데이터가 **L1 캐시**에 딱 맞게 들어가서 메인 메모리 접근 불필요.
4. 압축 해제 없이 비트 AND/OR로 바로 필터링 가능.

> **타이트 루프(tight loop)란?**
> 루프 내부에 조건 분기, 함수 호출, 복잡한 포인터 접근 같은 것 없이 **단순한 연산만 빽빽하게** 반복하는 루프를 말한다.
>
> ```c
> // 타이트 루프 예시
> for (int i = 0; i < n; i++) {
>     sum += ages[i];  // 메모리 연속 접근 + 덧셈만
> }
> ```
> 이런 형태는 CPU가 분기 예측 실패나 캐시 미스 없이 파이프라인을 꽉 채워서 돌릴 수 있어 매우 빠르다. 칼럼 저장소가 이런 루프를 돌리기에 딱 좋은 구조인 것.

이걸 **벡터화 처리**라고 한다. 한 번의 명령으로 여러 데이터를 동시에 처리.

---

### 칼럼 저장소의 순서 정렬

칼럼별로 저장해도 **전체 로우를 한 번에 정렬**해야 한다. 각 칼럼을 독립적으로 정렬하면 로우 대응이 깨진다.

**정렬이 압축에도 도움**

정렬 칼럼의 카디널리티가 낮으면 정렬 후 같은 값이 길게 반복된다.

```
[전자제품 × 10000개, 의류 × 5000개, 식품 × 3000개, ...]
```

---

### 칼럼 지향 저장소에 쓰기

읽기 최적화 대신 쓰기를 희생했다. **B 트리의 제자리 갱신은 불가능**하다.

> **"B 트리의 제자리 갱신은 불가능"이 무슨 뜻?**
> B 트리는 값을 수정할 때 **그 자리 페이지를 찾아서 바로 덮어쓴다**(in-place update). 이게 가능한 이유는 로우 지향 저장소에선 로우 하나가 한 페이지 안에 뭉쳐 있고, 주변 데이터와 독립적이기 때문이다.
>
> 근데 칼럼 저장소는 그게 안 된다. 이유는 아래 세 가지:

1. **위치가 이름표**: 로우는 칼럼 내 위치로 식별. 중간 삽입하려면 모든 칼럼 파일을 한 칸씩 밀어야 함.
2. **압축이 단단함**: RLE로 `[A가 100개]`라 압축된 덩어리 중간에 `B` 넣으려면 압축 풀고 다시 압축 → 파일 재작성.
3. **모든 칼럼 동기화**: 칼럼 100개면 로우 하나 삽입에 100개 파일 수정.

해결책은 **LSM 방식**. 새 데이터를 인메모리 델타에 모았다가 배치로 머지. 읽기는 인메모리 + 디스크 둘 다 확인.

---

### 집계: 데이터 큐브와 구체화 뷰

- **일반 뷰**: 쿼리의 별칭. 호출할 때마다 원본에서 재계산.
- **구체화 뷰**: 쿼리 결과를 **실제로 디스크에 저장**. 집계 쿼리가 빨라지는 대신 원본 변경 시 갱신 비용 발생.

**데이터 큐브(OLAP Cube)**: 구체화 뷰의 특수 사례. (날짜 × 지역)처럼 두 개 이상의 차원에 대해 미리 집계해둔 격자. 읽기는 매우 빠르지만 특정 축에 고정돼 유연성 떨어짐.

## 4장: 부호화와 발전

4장의 큰 흐름: **"서비스가 계속 바뀌는데 데이터 포맷은 어떻게 호환성을 유지할 것인가?"**

### 부호화와 복호화

- **인메모리 표현** (객체, 구조체) → **바이트열** 전환을 **부호화(encoding)**.
- 반대가 **복호화(decoding)**. 직렬화/역직렬화라고도 함.

### 하위/상위 호환성

- **하위 호환성**: 새 코드가 **옛 데이터**를 읽을 수 있어야 함. (미래 → 과거)
- **상위 호환성**: 옛 코드가 **새 데이터**를 읽을 수 있어야 함. (과거 → 미래, 더 어려움)

롤링 배포 환경에선 구버전·신버전이 공존하므로 둘 다 필수.

---

### 이진 부호화

JSON, XML은 읽기 좋지만 용량 낭비. 필드 이름, 따옴표, 공백까지 전부 텍스트로 보내야 함.

**이진 부호화의 장점**

1. 용량 절약 (숫자 `12345`를 5바이트 → 2~4바이트).
2. 빠른 파싱 (정해진 길이만 읽으면 끝).
3. 스키마 기반 타입 안정성.

**JSON의 숫자 함정**

JSON은 정수/부동소수점 구분 안 함, 정밀도 지정 없음. 2^53 초과 정수는 IEEE 754 부동소수점으로 정확히 표현 불가 → 파싱 시 정밀도 깨짐.

트위터는 이 문제로 API에서 트윗 ID를 **숫자 한 번, 문자열 한 번** 중복 전송한다.

---

### MessagePack — "JSON의 이진 버전"

스키마 없이 이진 부호화. 태그로 용량만 줄인 형태.

`{"id": 1, "age": 25}`를 MessagePack으로:

```
0x82              → 데이터 2개짜리 맵 시작
0xa2 0x69 0x64    → 길이 2짜리 문자열 "id"
0x01              → 숫자 1
0xa3 0x61 0x67 0x65 → 길이 3짜리 문자열 "age"
0x19              → 숫자 25
```

총 10바이트 (JSON 18바이트). 근데 필드 이름은 여전히 포함됨. 여기서 더 나아간 게 Thrift, Protobuf.

---

### Thrift와 Protocol Buffers

스키마를 미리 정의하고, 데이터에는 **필드 이름 대신 태그 번호**만 넣는다.

```protobuf
message UserProfile {
  string user_name = 1;
  int32 age = 2;
  string email = 3;
}
```

네트워크엔 `[1번 태그][Junhyuk][2번 태그][25]` 식으로 흐름. 필드 이름 전송 안 됨.

**Thrift의 두 프로토콜**

| 구분 | BinaryProtocol | CompactProtocol |
|---|---|---|
| 정수 저장 | 고정 길이 (4~8바이트) | 가변 길이 (1~10바이트) |
| 필드 번호 | 2바이트 고정 | 델타 인코딩 |
| 용량 | 보통 | 매우 작음 |

Protobuf는 CompactProtocol과 비슷.

### 필드 태그와 스키마 발전

스키마는 시간 지나면 변한다. 이를 **스키마 발전(Schema Evolution)**이라고 함.

- 필드 이름은 데이터에 안 들어가니 자유롭게 변경 가능.
- **태그 번호는 절대 바꾸면 안 됨.** 바꾸는 순간 기존 데이터 인식 불가.

**상위 호환성 작동 방식**

새 코드가 3번 태그 추가해서 데이터 보냄. 옛 코드는 3번이 뭔지 모름. 근데 데이터 안에 **와이어 타입(wire type)**이 포함돼 있어서 "8바이트짜리 고정 길이구나"를 파악 → 8바이트 건너뛰고 다음 필드로.

즉, 이진 데이터 안에 최소한의 타입 힌트가 있어서 모르는 필드를 안전하게 스킵 가능.

**하위 호환성 주의점**

새로 추가한 필드를 `required`로 지정하면 안 됨. 옛 데이터엔 그 필드가 없으니 읽기 실패.

**repeated 필드의 매력**

Protobuf는 목록 타입 대신 `repeated` 표시자를 쓴다. 같은 태그를 여러 번 나열하면 리스트.

`optional` → `repeated` 변경해도 호환됨:

- 새 코드가 옛 데이터(단일 값) 읽기 → 원소 1개짜리 리스트로 처리
- 옛 코드가 새 데이터(다중 값) 읽기 → 마지막 원소만 취함

---

### Avro

Thrift/Protobuf와 다른 철학. **데이터 안에 태그 번호도 타입 정보도 없다.**

대신 **두 개의 스키마**를 비교해서 해석:

- **쓰기 스키마**: 데이터 부호화할 때 쓴 스키마. 데이터와 함께 전달.
- **읽기 스키마**: 복호화하는 쪽이 기대하는 스키마.

**스키마 해석(schema resolution)** 규칙:

- 쓰기에 있고 읽기에 없는 필드 → 무시
- 읽기에 있고 쓰기에 없는 필드 → **기본값**으로 채움

필드 추가/삭제 시 반드시 기본값을 정의해야 호환성이 유지된다.

**필드 이름 변경 — 하위 호환만 가능**

Avro엔 별칭(alias) 기능이 있어서 "예전 이름은 X였어"를 **새 읽기 스키마**에 적어둘 수 있음. 근데 옛 버전 스키마엔 별칭 정보 없으니 상위 호환은 불가.

- 새 코드 + 옛 데이터 (하위 호환) → 별칭 덕분에 성공 
- 옛 코드 + 새 데이터 (상위 호환) → 별칭 모름 → 실패 

유니온 타입에 원소 추가도 같은 원리.

**Avro가 동적 스키마에 강한 이유**

Thrift/Protobuf는 태그 번호 관리가 귀찮다. DB 테이블을 자동으로 스키마 변환할 때 번호 배정이 꼬일 수 있음.

Avro는 이름 매칭이라 DB 칼럼 이름을 그대로 쓰면 끝. 스키마 바뀌어도 그냥 다시 변환만 하면 됨.

**스키마 발전의 효과**

> 저장소엔 여러 버전이 섞여있어도, 개발자 눈엔 **단일 스키마처럼** 보인다.

물리적으로는 버전 1, 2, 3이 뒤섞여있지만 논리적으로는 하나의 최신 버전. 무중단 배포의 핵심 기술.

### 데이터플로 모드

프로세스 간 데이터 흐름은 세 가지로 나뉨.

### (1) DB를 통한 데이터플로

쓰는 쪽이 기록하고 나중에 읽는 쪽이 조회. 시간 축이 길어서 **몇 년 뒤 코드**가 읽을 수도 있다.

주의: 새 필드 생겼을 때 **옛 코드가 읽고 다시 쓰면서 새 필드를 날려먹는 문제**. 옛 코드는 모르는 필드를 보존하는 방식으로 처리해야 함.

---

### (2) 서비스를 통한 데이터플로: REST와 RPC

**REST**

- HTTP 위에 API. HTTP 메서드(GET/POST/PUT/DELETE)와 상태 코드 그대로 활용.
- 네트워크 프로토콜이라는 사실을 숨기지 않음. 중간 장비(캐시, LB, CDN)가 요청 이해 가능.
- JSON 기반이라 디버깅·브라우저 친화적.

**RPC란?**

"원격 서버의 함수를 내 로컬 함수처럼 호출하자"는 아이디어. `service.getPrice("AAPL")`이라고 쓰면 알아서 네트워크 타고 가서 결과 받아오는 방식.

요즘 gRPC는 HTTP/2 기반이라 연결을 계속 열어두고 스트리밍도 가능 — 그래서 연결이 유지된다는 말이 나온다.

**RPC의 근본적 결함**

네트워크 요청은 로컬 함수 호출과 근본적으로 다르다.

| 구분 | 로컬 호출 | RPC |
|---|---|---|
| 예측 가능성 | 실행 or 프로세스 죽음 | 네트워크 끊김·서버 느림·응답 유실 등 변수 많음 |
| 시간 | 수 나노초 | 수 밀리초~초, 변동 심함 |
| 타임아웃 | 결과 없으면 뻗음 | 요청이 안 간 건지, 처리됐는데 응답만 못 받은 건지 불명 |
| 중복 실행 | 1회 호출 = 1회 실행 | 재시도로 2번 실행될 수 있음 (멱등성) |

RPC 문법이 네트워크 위험성을 숨겨버려서 개발자가 실수하게 만든다. **누수되는 추상화(leaky abstraction)**.

**차세대 RPC 프레임워크**

gRPC, Finagle, Rest.li 등은 원격 호출이 로컬과 다름을 **분명히 드러낸다**.

- **퓨처(Future)/프로미스**: 비동기 결과를 담는 빈 상자. 여러 서비스 병렬 호출·취합이 쉬워짐.
- **스트림**: 단일 요청/응답이 아닌, 시간에 따른 요청/응답 스트림 지원.
- **서비스 찾기(Service Discovery)**: 살아있는 서비스 인스턴스를 자동으로 찾아줌. Eureka, Consul 등.

**REST vs RPC 선택**

- 내부 서비스 간 통신 → gRPC 같은 RPC 유리 (성능, 타입 안정성)
- 공개 API, 브라우저/모바일 → REST가 표준

**배포 순서와 호환성**

보통 **서버를 먼저 갱신**한다. 이때:

- 요청 (옛 클라이언트 → 새 서버) → 서버는 **하위 호환**
- 응답 (새 서버 → 옛 클라이언트) → 클라이언트는 **상위 호환**

---

### (3) 메시지 전달 데이터플로

RPC(동기)와 DB(영구 저장)의 중간. 메시지 브로커(Kafka, RabbitMQ)가 데이터를 잠시 맡아서 전달.

**RPC 대비 장점**

- 수신자가 죽어있어도 메시지 유실 안 됨 (브로커가 버퍼).
- 송신자가 수신자 IP를 몰라도 됨 → 결합도 낮음.
- 한 메시지를 여러 소비자에게 전달 가능 (pub/sub).
- 완전 비동기 — 응답 기대 안 함.

단방향이라 응답 필요하면 별도 채널 사용.

**분산 액터 프레임워크**

액터 모델: 논리를 **액터** 단위로 캡슐화, 액터끼리는 **메시지 전달로만** 통신. 공유 메모리 안 써서 `synchronized`나 락 불필요.

분산 액터 프레임워크(Akka, Erlang/OTP, Orleans)는 여기에 **위치 투명성**을 얹는다. 상대가 같은 서버든 다른 서버든 똑같은 코드로 메시지 전송.

= 메시지 브로커 + 액터 모델 통합. 별도 Kafka 없이도 분산 비동기 통신 가능.

다만 스키마 호환성은 여전히 중요. 순회식 업그레이드 중 신/구 버전 노드가 공존하므로, 여기서도 상위/하위 호환성이 그대로 적용됨.

## 정리

- **B 트리**는 제자리 갱신 + WAL. 쓰기 최소 두 번.
- **클러스터드 vs 힙 파일**은 "인덱스 안에 데이터 있음 vs 포인터만 있음".
- **OLTP는 건당 조회, OLAP는 대량 스캔** — 워크로드가 달라 DB 자체를 분리.
- **칼럼 지향 + 칼럼 압축 + 벡터화 처리**가 결합하면 OLAP 성능 폭발적 상승.
- **정렬된 칼럼 + RLE**는 카디널리티 낮은 칼럼에서 극단적 압축률.
- **구체화 뷰·데이터 큐브**는 집계를 미리 계산해둔 캐시.
- **스키마 발전**은 여러 버전이 섞여있어도 단일 스키마처럼 보이게 해주는 기술.
- **Protobuf/Thrift**는 태그 번호 기반, **Avro**는 이름 기반 매칭.
- **RPC는 네트워크를 숨기려다 실수를 유발** — 현대 RPC는 이를 드러내는 방향.
- **데이터플로(DB/서비스/메시지)** 모두 호환성 설계가 핵심.
