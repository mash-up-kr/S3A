## 파티셔닝이란?

데이터셋이 매우 크거나 질의 처리량이 매우 높다면 **복제만으로는 부족**하고 데이터를 파티션으로 쪼갤 필요가 있다. 이걸 **샤딩(Sharding)** 이라고 한다.

파티션을 나눌 때는 보통 각 데이터 단위가 하나의 파티션에 속하게 된다. 

결과적으로 **각 파티션은 그 자체로 작은 데이터베이스**가 된다.

**파티셔닝의 핵심 이유: 확장성**

- 단일 파티션 질의는 각 노드가 자기 파티션만 독립적으로 처리하면 됨 → 노드 추가하면 처리량 증가.
- 비공유(Shared-Nothing) 클러스터: 각 노드가 CPU, 메모리, 디스크를 독점 소유. 네트워크로만 통신.
- 자원 경합 없음 → "내 데이터는 내 디스크에 있고, 내 CPU가 처리한다."

> **"각 파티션은 작은 데이터베이스"의 의미**
데이터를 파티션으로 나눈다는 건 단순히 파일을 쪼개는 게 아니라 **독립적인 조각으로 분리**한다는 뜻. ID 1~100번이 1번 파티션에 있다면 1번 파티션만 뒤져서 해당 데이터를 완결성 있게 찾을 수 있어야 한다. 분할 정복.
> 

### 파티셔닝과 복제

보통 복제와 파티셔닝을 **함께** 적용. 각 파티션의 복사본을 여러 노드에 저장.

- 각 레코드는 정확히 한 파티션에 속함.
- 그 파티션을 여러 노드에 복제해서 내결함성 보장.
- 한 노드가 어떤 파티션에는 리더, 다른 파티션에는 팔로워가 될 수 있음.

## 키-값 데이터 파티셔닝

파티셔닝의 목적은 **데이터와 질의 부하를 노드 사이에 고르게 분산**시키는 것.

- **쏠림(Skew)**: 파티션 간 데이터/질의가 불균형한 상태.
- **핫스팟(Hotspot)**: 부하가 한 파티션에 몰리는 현상.

**무작위 할당의 함정**: 핫스팟 회피의 가장 단순한 방법은 무작위 할당이지만, 어떤 레코드가 어디 있는지 알 수 없으니 모든 노드에 병렬 질의해야 함. → 비효율.

### 키 범위 기준 파티셔닝

각 파티션에 **연속된 범위의 키**를 할당. (백과사전처럼)

- 경계만 알면 어떤 키가 어느 파티션에 있는지 즉시 파악 가능.
- 키 범위 크기가 동일할 필요는 없음. 데이터가 고르게 분포하지 않을 수 있어서 경계를 데이터에 맞게 조정.
- 각 파티션 내에서 키를 **정렬된 순서로 저장** → 범위 스캔 효율적.

**단점**: 특정 접근 패턴이 핫스팟 유발. 예: 타임스탬프를 키 첫 요소로 쓰면 최신 시간 파티션에 쓰기 부하 몰림. → 첫 요소를 센서 이름 같은 다른 값으로 두고 타임스탬프를 뒤로 빼면 분산됨.

### 키의 해시값 기준 파티셔닝

쏠림과 핫스팟 회피용. **키를 해시 함수에 통과시켜** 그 결과의 범위로 파티션을 나눔.

- 좋은 해시 함수는 비슷한 입력도 균일하게 분산시킴.
- 파티셔닝용 해시는 **암호적으로 강력할 필요 없음**.
- 일관성 해싱(Consistent Hashing): 파티션 경계를 무작위에 가깝게 선택하는 기법.

**치명적 단점**: 정렬 순서가 깨져서 **범위 질의를 효율적으로 못함**. 인접했던 키들이 모든 파티션에 흩어짐.

> **해시 vs 키 범위 트레이드오프**
> 
> - 부하 분산만 보면 해시가 압도적.
> - 하지만 "20세~30세 사용자 조회" 같은 범위 질의는 키 범위가 유리.
> - 결국 읽기 패턴이 **점 조회(point lookup)** 위주인지 **범위 스캔** 위주인지에 따라 선택.

**카산드라의 절충안**: 복합 기본키(Composite Primary Key) 사용.

- 키의 **첫 부분에만 해싱 적용** → 파티션 결정.
- **남은 칼럼은 정렬용 색인** → 같은 파티션 내에서 범위 스캔 가능.

| 구분 | 첫 번째 칼럼 (Partition Key) | 남은 칼럼 (Clustering Column) |
| --- | --- | --- |
| 처리 방식 | 해싱(Hashing) | 정렬(Sorting) 유지 |
| 목적 | 여러 서버에 분산 | 한 서버 내에서 순서 보관 |
| 조회 조건 | 일치(=) 만 가능 | 범위(>, <) 가능 |

→ 시계열 데이터 모델링(예: 한 종목의 시간대별 시세)에 매우 강력. **"어느 서버로 갈지는 해시로 정하고, 그 서버 안에서는 순서대로 쌓아둔다."**

### 쏠린 작업부하와 핫스팟 완화

해싱이 만능은 아니다. **항상 동일한 키를 읽고 쓰면** (예: 인기 연예인 계정) 모든 요청이 동일 파티션으로 쏠림. 동일 ID의 해시값은 동일하므로 해싱이 무의미.

**현재 시스템들은 자동 보정 못함** → 애플리케이션이 직접 완화해야 함.

- 키 시작/끝에 임의의 숫자 2자리 붙이기 → 한 키의 쓰기가 100개 키로 분산됨.
- 단점: 읽을 때 100개 키를 다 모아서 합쳐야 함. 추가 메타데이터 관리 필요.
- → 핫키 소수에만 적용해야 함.

## 파티셔닝과 보조 색인

보조 색인(Secondary Index)은 특정 값이 발생한 항목을 검색하는 수단. 관계형/문서 DB의 핵심, 검색 엔진(엘라스틱서치, 솔라)의 존재 이유.

문제: **보조 색인은 파티션에 깔끔하게 대응되지 않음.**

→ 두 가지 방법: **문서 기반** vs **용어 기반**.

### 문서 기준 보조 색인 파티셔닝 (Local Index)

각 파티션이 **자기 데이터에 대해서만** 색인 유지.

- 1번 서버의 색인: "우리 서버에 있는 빨간색 차는 10번, 20번."
- 2번 서버의 색인: "우리 서버에 있는 빨간색 차는 600번, 700번."

| 동작 | 효율 | 이유 |
| --- | --- | --- |
| 쓰기 | 매우 좋음 | 데이터 저장하는 파티션 한 곳만 색인 업데이트 |
| 읽기 | 나쁨 | 모든 파티션에 물어봐야 함 → **스캐터/개더(Scatter/Gather)** |

스캐터/개더는 응답 시간이 꼬리 지연에 취약. 그래도 몽고DB, 카산드라, 엘라스틱서치가 이 방식을 쓰는 이유는 **쓰기 성능 우선**이기 때문.

### 용어 기준 보조 색인 파티셔닝 (Global Index)

모든 파티션의 데이터를 담당하는 **전역 색인**. 단, 한 노드에만 두면 병목이라 **색인 자체도 파티셔닝**.

- 데이터 파티셔닝: car_id 기준
- 색인 파티셔닝: 색상이나 제조사(용어) 기준 — 예: a~r은 색인 파티션 0, s~z는 1.

> **그림 6-5 시나리오**
모든 파티션의 빨간색 자동차 정보가 색인 파티션 0(a~r)에 모임.
"Red 검색" → 파티션 0번만 보면 끝. 모든 데이터 파티션을 뒤질 필요 없음.
> 

| 동작 | 효율 | 이유 |
| --- | --- | --- |
| 읽기 | 빠름 | 해당 용어 담당 색인 파티션만 방문 |
| 쓰기 | 느림 | 여러 색인 파티션 업데이트 필요. 분산 트랜잭션 문제. |

→ 이상적으로는 색인이 항상 최신이어야 하지만, 분산 트랜잭션 부담 때문에 **실무에선 비동기 업데이트**로 타협.

> **벤더들의 권장 사항과 현실**
"보조 색인 질의가 단일 파티션에서 실행되도록 설계하라"는 게 권장. 하지만 다차원 검색, 데이터 간 관계, 다양한 기획 요구 때문에 항상 가능하진 않음. 핵심 검색은 단일 파티션, 가끔 발생하는 복잡 검색은 성능 저하 감수 또는 전용 검색 엔진(Elasticsearch) 분리.
>

## 파티션 재균형화

시간이 지나면 클러스터에 변화가 생김. 한 노드의 부하를 다른 노드로 옮기는 과정이 **재균형화(Rebalancing)**.

**재균형화의 요구사항**

- 부하가 노드들 사이에 균등하게 분배되어야 함.
- 재균형화 도중에도 읽기/쓰기는 처리되어야 함.
- 데이터 이동량이 최소여야 함 (네트워크/디스크 I/O 최소화).

### 쓰면 안되는 방법: 해시값에 mod N 연산

`hash(key) % N`은 N이 바뀌면 거의 모든 키가 이동해야 함.

- hash(key) = 12345
- N=10: 노드 6 (12345 % 10 = 5, 그러니까 노드 5겠지만 의미는 같음)
- N=11: 노드 3 (12345 % 11)
- N=12: 노드 9

→ 재균형화 비용 높음. **쓰지 말 것.**

### 파티션 개수 고정

파티션을 노드 대수보다 **훨씬 많이** 만들고 각 노드에 여러 파티션 할당.

- 노드 10대 → 파티션 1000개 → 노드당 100개.
- 노드 추가 시 새 노드가 기존 노드에서 파티션 몇 개를 뺏어옴.
- 파티션 개수와 키 할당은 변하지 않음. 단지 **노드-파티션 매핑만 변경**.
- 데이터 전송 중에도 기존 매핑으로 읽기/쓰기 처리 가능.
- 성능 좋은 노드에 파티션을 더 할당해서 부하 비대칭 처리도 가능.

**한계**: 파티션 개수가 처음에 고정되면 이후 변경 어려움.

> **고정 개수의 문제**
처음에 파티션 1000개로 시작 → 데이터가 100GB일 땐 파티션당 100MB로 가벼움.
데이터가 100TB로 늘면 파티션 개수는 그대로 1000개 → 파티션당 100GB로 비대해짐.
> 
> - 파티션이 너무 크면: 재균형화/장애 복구 비용 폭증 (100GB 덩어리 옮기기).
> - 파티션이 너무 작으면: 메타데이터 오버헤드, 스캐터/개더 비용 증가.
> → 데이터셋 크기 변동이 심한 환경에 부적합.

### 동적 파티셔닝

파티션 크기가 임계치를 넘으면 **자동 분할**, 너무 작아지면 **자동 병합**.

- 파티션 개수가 데이터 양에 비례.
- 데이터 양이 적으면 파티션 적음 → 오버헤드 작음.
- 데이터 양이 많아지면 파티션 크기 자동 제한.

**콜드 스타트 문제**: 빈 DB는 파티션 경계 정보가 없어서 시작할 땐 파티션 1개. 데이터가 적을 때 모든 쓰기가 한 노드로 몰림 → 다른 노드 유휴 상태.

> **사전 분할(Pre-splitting)**
HBase, MongoDB는 빈 DB에 초기 파티션 집합을 설정 가능.
"데이터가 없어도 미리 파티션 10개를 만들어 노드 10대에 배포하라."
단, 키 범위 파티셔닝의 경우 키 분포를 **미리 알아야** 효과적.
> 

### 노드 비례 파티셔닝

파티션 개수가 **노드 대수에 비례**. 노드당 파티션 개수를 고정.

- 노드 10대 + 노드당 파티션 10개 = 총 100개.
- 데이터만 늘면 파티션 크기 증가, **노드 추가하면 파티션 분할되어 크기 감소**.
- 데이터 많아지면 노드도 늘어나는 게 일반적이므로 → 파티션 크기 안정적.

새 노드 추가 시: 고정 개수의 파티션을 무작위 선택해 분할 → 절반은 기존 노드, 절반은 새 노드.

→ **카산드라의 vnode가 이 방식**. 일관성 해싱의 원래 정의에 가까움.

| 방식 | 비례 관계 | 특징 |
| --- | --- | --- |
| 동적 | 파티션 개수 ∝ 데이터 양 | 파티션 크기 일정 |
| 고정 개수 | 파티션 크기 ∝ 데이터 양 | 데이터 폭증 시 부담 |
| 노드 비례 | 파티션 개수 ∝ 노드 수 | 노드 늘리면 크기 안정화 |

### 운영: 자동 vs 수동 재균형화

**완전 자동 재균형화는 위험**.

- 자동 장애 감지와 결합되면 연쇄 장애 가능.
- 네트워크 일시 장애로 노드가 1초 응답 없는 걸 죽었다고 판단 → 수백 GB 복사 시작 → 시스템 전체 부하 → 진짜 죽음.
- 카우치베이스, 리약, 볼드모트는 자동 제안 + 관리자 확정 방식.

## 요청 라우팅

클라이언트는 **어느 노드로 접속해야 하는지** 어떻게 알까? 파티션이 재균형화되면 매핑이 바뀌는데.

이건 **서비스 디스커버리(Service Discovery)** 의 일종.

**3가지 접근**

1. **클라이언트가 아무 노드에나 접속** → 해당 노드에 파티션이 있으면 직접 처리, 없으면 올바른 노드로 전달.
2. **라우팅 계층**이 모든 요청을 받아서 적절한 노드로 전달. 라우팅 계층 자체는 처리 안 함.
3. **클라이언트가 파티셔닝 정보를 알고** 직접 올바른 노드로 접속.

핵심 문제: **누가 노드-파티션 매핑 변경 사항을 추적하는가?**

**주키퍼 등의 코디네이션 서비스 활용**

- 각 노드는 주키퍼에 등록.
- 주키퍼가 매핑 정보 관리.
- 라우팅 계층이나 클라이언트가 주키퍼를 구독.
- 매핑 변경 시 주키퍼가 알림.

| 시스템 | 라우팅 방식 |
| --- | --- |
| 링크드인 에스프레소 | 헬릭스(주키퍼 기반) |
| HBase, 솔라클라우드, 카프카 | 주키퍼 |
| 몽고DB | 자체 설정 서버 + 몽고스(mongos) |
| 카산드라, 리약 | 가십 프로토콜 (노드 간 직접 전파) |
| 카우치베이스 | 자동 재균형화 X, 목시(Moxi) 라우팅 계층 |

> **카우치베이스의 단순함**
자동 재균형화를 하지 않아 설계가 단순. 운영자가 원하는 시간(트래픽 적은 새벽 등)에 명시적으로 실행.
클라이언트는 Moxi에게 요청 → Moxi가 vBucket Map(클러스터 지도)을 보고 적절한 노드로 전달.
서버 IP는 거의 안 바뀌니 DNS로 충분. 데이터 위치(파티션 매핑)는 Moxi가 들고 있는 최신 지도 필요.
> 

### 병렬 질의 실행

대규모 병렬 처리(MPP) 관계형 DB는 복잡한 질의를 여러 단계와 파티션으로 분해 → 클러스터 노드들에서 병렬 실행. 데이터셋 대부분을 스캔하는 분석 질의에 특히 유리.

## 정리

- **파티셔닝의 목적**: 데이터/질의 부하를 노드들에 고르게 분산 → 확장성 확보. 비공유 클러스터에서 진가 발휘.
- **키-값 파티셔닝 방식**:
    - 키 범위 → 범위 질의 효율적, 핫스팟 위험.
    - 키 해시 → 부하 분산 우수, 범위 질의 비효율.
    - 카산드라식 복합 키 → 둘의 절충안.
- **핫스팟 완화**: 해싱이 만능 아님. 동일 키 쏠림은 애플리케이션 레벨에서 키 분할로 직접 처리.
- **보조 색인 파티셔닝**:
    - 문서 기준(Local) → 쓰기 빠름, 읽기 느림 (스캐터/개더).
    - 용어 기준(Global) → 읽기 빠름, 쓰기 복잡 (분산 트랜잭션 부담).
- **재균형화 전략**:
    - mod N 절대 금지.
    - 파티션 개수 고정, 동적 파티셔닝, 노드 비례 — 각각 트레이드오프.
- **자동 재균형화는 신중히**. 자동 장애 감지와 결합되면 연쇄 장애 위험.
- **요청 라우팅**: 주키퍼 같은 코디네이션 서비스로 매핑 추적이 일반적. 카우치베이스는 명시적 운영, 카산드라는 가십 프로토콜.
- **현실의 백엔드 설계**: 파티션 키 선택이 시스템 성능의 시작과 끝. 자주 일어나는 핵심 검색은 단일 파티션, 가끔 발생하는 복잡 검색은 별도 검색 엔진(Elasticsearch) 분리 운영.
