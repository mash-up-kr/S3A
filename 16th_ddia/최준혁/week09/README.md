## 일관성 보장 (Consistency Guarantees)

- 분산 DB의 근본적 한계: 동시에 두 노드를 보면 데이터가 다를 가능성이 큼
    - 쓰기 요청이 각 노드에 도착하는 시간이 다르기 때문
    - 복제 방법(단일 리더/다중 리더/리더 없는)과 무관하게 발생 → 물리적으로 노드 간 전송 시간은 0이 될 수 없음
- 최종적 일관성 (Eventual Consistency)
    - "지금은 노드마다 달라도, 쓰기가 멈추고 시간이 충분히 흐르면 결국 모든 복제본이 수렴"
    - 사실상 '수렴(Convergence)'이 더 정확한 이름
    - 매우 약한 보장: 언제 수렴할지 아무것도 말해주지 않음
    - 개발자에게 어려움 → 방금 쓴 값을 바로 읽었는데 옛날 값이 나오거나 실패할 수 있음
- 이번 장의 목표: 더 강한 일관성 모델 탐구 (성능을 대가로 지불)
    - 트랜잭션 격리: 동시 실행 트랜잭션의 경쟁 조건 회피
    - 분산 일관성: 지연/결함 속에서 복제본 상태를 코디네이션

## 선형성 (Linearizability)

- 기본 아이디어: 시스템에 **데이터 복사본이 하나만 있는 것처럼** 보이게 하고, 모든 연산이 **원자적**으로 보이게 함
    - 현실엔 여러 복제본이 있어도 애플리케이션은 신경 쓸 필요 없음
    - = **최신성(Recency) 보장**: 읽은 값이 최신 값이며, 뒤처진 캐시/복제본 값이 아님

>  **"선형성 시맨틱"이 무슨 뜻?**
> '시맨틱'은 시스템이 외부(애플리케이션)에 약속하는 **동작 방식/계약(Contract)**을 의미. "선형성 시맨틱을 제공한다" = "내부가 몇 대로 쪼개져 있든, 너에겐 완벽한 최신성을 보장하는 단일 DB처럼 동작할 것을 약속한다"는 뜻.

- 선형성을 만드는 조건
    - x의 값이 **원자적으로 한 시점에** 바뀌는 순간이 존재해야 함
    - 한 클라이언트의 읽기가 새 값을 반환하면, **이후의 모든 읽기도 새 값**을 반환해야 함 (쓰기가 아직 안 끝났어도)
    - 연산을 모은 선은 항상 **시간순으로 진행**, 결코 뒤로 가지 않음

### 선형성 vs 직렬성 (헷갈리기 쉬움)

| 비교 항목 | 직렬성 (Serializability) | 선형성 (Linearizability) |
|---|---|---|
| 대상 | **트랜잭션** (여러 객체/연산의 묶음) | **개별 객체** (단일 레지스터) |
| 핵심 목표 | 동시성 제어 (격리성) | 최신성 보장 |
| 시간 기준 | 실제 시간 순서 무관 (결과만 정상이면 됨) | 실제 시간(Wall-clock) 기반 순서가 절대적 |
| 방어 문제 | 쓰기 스큐, 갱신 손실 | 오래된 데이터(Stale) 읽기 |

- 선형성은 연산을 트랜잭션으로 묶지 않음 → 충돌 구체화 같은 수단 없이는 쓰기 스큐를 못 막음

## 선형성에 기대기 (언제 필요한가)

- **잠금과 리더 선출**
    - 단일 리더 복제는 리더가 진짜 하나만 존재하도록 보장해야 함 → **스플릿 브레인** 방지
    - 모든 노드가 시작 시 잠금 획득 시도, 성공한 노드가 리더 → 이 잠금은 **선형적**이어야 함
    - 복제 지연으로 "어? 잠금 비었네?" 하고 동시에 둘이 잠금을 얻으면 스플릿 브레인 발생
    - **ZooKeeper, etcd**: 합의 알고리즘(Raft/Paxos)으로 내결함성 있는 선형성 연산 제공
    - Oracle RAC: 공유 디스크 구조 → 디스크 페이지 단위 세분화된 분산 잠금 + 전용 초고속 클러스터 네트워크 필수
- **유일성 제약 조건**
    - 사용자명/이메일 유일성 = 사용자명에 "잠금"을 거는 것과 유사 (원자적 compare-and-set)
    - 모든 노드가 동의하는 하나의 최신 값 필요 → 선형성 요구
- **채널 간 타이밍 의존성**
    - 부가적인 통신 채널(예: 파일 저장소 + 메시지 큐)이 둘 있을 때
    - 메시지 큐가 저장소 복제보다 빠르면 → 크기 변경 모듈이 과거 이미지를 읽거나 아무것도 못 읽는 경쟁 조건

## 선형성 구현과 정족수

- 가장 간단한 해답: 진짜로 데이터 복사본 하나만 사용 (단, 내결함성 0이라 현실성 없음)
- **정족수(Quorum)는 선형적일까?** → 직관적으론 그래 보이지만 아님

>  **그림 9-6 상황 (정족수의 함정)**
> W+R > N이면 교집합이 생기니 최신값을 읽을 것 같지만, 네트워크 지연 변동이 심하면 깨짐.
> - 앨리스가 노드 2,3 읽음 → 노드 3에서 최신값 1을 읽고 성공
> - 그 **이후에** 밥이 노드 1,2 읽음 → 둘 다 아직 쓰기 미수신 → 과거값 0을 읽음
> - **시간이 거꾸로 흐름** = 선형성 위반

- 억지로 선형적으로 만들 순 있음 (성능 대가)
    - 읽기 클라이언트: 결과 반환 전 **동기식 읽기 복구** 수행
    - 쓰기 클라이언트: 쓰기 전 정족수로부터 최신 상태 읽기
- 현실: **Riak**은 성능상 동기식 읽기 복구 안 함, **Cassandra**는 LWW(최종 쓰기 승리) 충돌 해소 때문에 동시 쓰기 시 선형성 잃음
- **결론: 다이나모 스타일(리더 없는) 시스템은 선형성을 제공하지 않는다고 보는 게 안전**

## 선형성의 비용 (CAP 정리)

- 데이터센터 간 네트워크 단절 시나리오
    - **다중 리더**: 각 데이터센터가 독립 동작, 쓰기는 큐에 쌓였다 복구되면 전달 → 계속 동작
    - **단일 리더**: 팔로워 데이터센터 클라이언트는 리더와 연결 끊기면 쓰기 불가 + 선형성 읽기 불가 → 중단
- **CAP 정리의 트레이드오프**
    - 선형성 요구 O → 네트워크 단절 시 일부 복제본은 요청 처리 불가 (대기 or 오류 반환) = **가용성 포기**
    - 선형성 요구 X → 단절돼도 독립적으로 요청 처리 = **가용성 유지하나 비선형적**
    - 결론: 선형성이 필요 없는 애플리케이션이 네트워크 문제에 더 강인함

### 선형성과 네트워크 지연

- 현실에서 진짜 선형적인 시스템은 의외로 드묾
- **멀티코어 CPU 비유**: 각 코어가 자기 캐시/저장 버퍼를 가짐 → 코어 1이 쓴 값을 코어 2가 바로 못 읽음 (단일 컴퓨터 안의 '복제 지연')
    - 단, 여기서 선형성을 포기한 이유는 **내결함성이 아니라 성능** (메인보드엔 '네트워크 단절' 개념이 없음)
    - → CAP로 멀티코어 메모리 모델을 정당화하는 건 말이 안 됨
- **물리적 한계**: 선형성 읽기/쓰기 응답 시간은 네트워크 지연의 불확실성에 비례
    - 가장 느린 노드의 응답에 발목 잡힘
    - 선형성을 제공하는 **더 빠른 알고리즘은 존재하지 않음** (아티야-웰치 정리로 증명된 한계)
    - 완화된 일관성 모델은 훨씬 빠를 수 있음

## 순서화 보장 (Ordering)

- 순서화 ↔ 선형성 ↔ 합의 사이엔 깊은 연결 관계가 있음

### 순서화와 인과성

- 인과성은 이벤트에 순서를 부과 → 원인이 결과보다 먼저
- 시스템이 인과적 순서를 지키면 **인과적으로 일관적(Causally Consistent)**
- **인과적 순서 = 부분 순서 (Partial Order)**, 선형성 = 전체 순서 (Total Order)

>  **Git 브랜치 비유**
> - **선형성(전체 순서)**: 1차선 외길. 어떤 두 연산이든 무조건 누가 먼저인지 줄 세움. 동시적 연산이 없음.
> - **인과성(부분 순서)**: Git 브랜치처럼 타임라인이 갈라짐. 질문글→답변은 순서가 있지만(인과 관계), 서로 무관한 두 작업은 비교 불가(동시적). 나중에 머지될 때 타임라인이 합쳐짐.

- **선형성은 인과성보다 강함**: 선형적이면 인과성도 자동으로 유지됨
- **하지만** 선형성은 성능/가용성에 해로움 → 절충 가능
    - 인과적 일관성은 선형성의 성능 손해 없이 인과성 보존
    - **인과적 일관성 = 네트워크 지연에 느려지지 않고, 장애에도 가용한 일관성 모델 중 가장 강한 것**
    - 많은 경우 선형성이 필요해 보여도 진짜 필요한 건 인과적 일관성

### 일련번호 순서화

- 모든 인과적 의존성을 추적하는 건 비실용적 → 일련번호/타임스탬프로 순서 정함
- 인과성에 일관적인 전체 순서 생성 가능: A가 B의 원인이면 전체 순서에서도 A < B, 동시 연산은 순서 제멋대로
    - 이 전체 순서는 인과 정보를 다 담지만, 필요 이상으로 순서를 부과함
- **단일 리더 복제**: 복제 로그가 인과성에 일관적인 전체 순서를 정의
    - 리더가 연산마다 카운터 증가 → 단조 증가 일련번호 할당
    - 팔로워가 로그 순서대로 적용하면 항상 인과적으로 일관적 (Kafka 파티션이 좋은 예)

### 램포트 타임스탬프 vs 버전 벡터

- **램포트 타임스탬프**: (카운터, 노드 ID) 쌍
    - 모든 노드/클라이언트가 본 최댓값 추적, 더 큰 값 받으면 자기 카운터를 그 값으로 증가
    - 전체 순서화 강제, 인과성에 일관적

| 비교 항목 | 램포트 타임스탬프 | 버전 벡터 |
|---|---|---|
| 핵심 목적 | 무조건 1열로 줄 세우기 | 동시 수정 충돌 감지 |
| 동시성 감지 | **불가능** (동시 연산도 순서가 매겨짐) | **가능** (갈라진 가지를 찾아냄) |
| 공간 복잡도 | O(1) (고정 크기) | O(N) (노드 수에 비례) |
| 활용 예 | 분산 잠금, 유일 ID 발급 | Cassandra/Riak 충돌 해소 |

### 타임스탬프 순서화로는 충분하지 않다

- 일련번호로 사후에 승자를 가리는 건 효과적 (로그 다 모아서 타임스탬프 비교)
- **하지만 실시간으로 당장 결정해야 할 때는 부족**
    - 노드가 사용자명 생성 요청을 받고 즉시 성공/실패를 정해야 함
    - 다른 노드가 동시에 더 낮은 타임스탬프로 같은 사용자명을 처리 중일 수 있음 → 모르고 성공 처리하면 충돌
    - 확신하려면 모든 노드 확인 필요 → 한 노드라도 장애 시 시스템 멈춤
- **핵심 결론**: 전체 순서가 있는 것만으론 부족. **언제 그 순서가 확정되는지**도 알아야 함
    - "내 앞에 끼어들 연산이 더는 없다"고 확신할 수 있어야 성공 선언 가능

## 전체 순서 브로드캐스트 (Total Order Broadcast)

- 단일 리더의 한계: 처리량이 리더 한 대를 넘으면? 리더 장애 시 복구는? → 이 문제가 전체 순서 브로드캐스트(원자적 브로드캐스트)
- **두 가지 안전성 속성**
    1. **신뢰성 있는 전달**: 메시지가 한 노드에 전달되면 모든 노드에 전달 (손실 없음)
    2. **전체 순서가 정해진 전달**: 모든 노드에 같은 순서로 전달
- 노드/네트워크 결함이 있어도 두 속성 보장 (재시도로 결국 복구 시 전달)

>  **왜 타임스탬프가 단일 리더보다 약한가?** (순서 '확정'의 권한 차이)
> - **램포트 타임스탬프**: 창구마다 번호표 기계가 따로. 10번을 줘도 "인터넷 끊긴 다른 창구에서 9번 뽑은 사람이 뒤늦게 올까봐" 즉시 확정 불가.
> - **단일 리더**: 번호표 기계가 딱 1대. 10번을 받으면 "미래에 9번이 나타날 일은 절대 없다"고 즉시 100% 확신 → 바로 성공 처리.

### 전체 순서 브로드캐스트 사용처

- **상태 기계 복제(State Machine Replication)**: 모든 메시지가 쓰기를 나타내고 모든 복제본이 같은 순서로 처리 → 일관성 유지
- 직렬성 트랜잭션 구현, 펜싱 토큰을 제공하는 잠금 서비스, 로그 생성에도 사용
- 중요 측면: **메시지가 전달되는 시점에 순서가 고정됨**

### 선형성 ↔ 전체 순서 브로드캐스트 ↔ 합의 (삼위일체)

- TOB는 **비동기식**: 순서는 보장하나 언제 전달될지는 미보장
- 선형성은 **최신성 보장**: 읽기가 최근 값을 봄
- **TOB로 선형성 쓰기(compare-and-set) 구현**
    1. 점유하려는 사용자명을 로그에 메시지로 추가 (침 발라놓기)
    2. 로그를 읽으며 내 메시지가 돌아오길 대기
    3. 해당 사용자명의 첫 메시지가 **내 것이면 성공(커밋)**, 다른 사람 것이면 **어보트**
    - 단, 이건 선형성 **쓰기**만 보장. **읽기**는 비동기 갱신 저장소라 옛날 값 읽힐 수 있음
- **반대로 선형성 레지스터로 TOB 구현**
    - 원자적 increment-and-get 선형성 정수 가정 → 메시지마다 번호 뽑아 붙이고 순서대로 전달
    - 램포트와 차이: 선형성 레지스터 번호는 **틈 없는 순열** → 4 다음 6을 받으면 5를 기다려야 함을 앎 (이게 TOB와 타임스탬프 순서화의 핵심 차이)
- **심오한 통찰**: 선형성 compare-and-set 레지스터 = 전체 순서 브로드캐스트 = **합의(Consensus)**, 셋은 서로 동등
    - 하나를 풀면 나머지도 풀림
    - 틈 없는 번호를 내결함성 있게 발급하려 고심하다 보면 **필연적으로 합의 알고리즘에 도달**
- → 이번 장의 나머지는 합의 문제를 정면으로 다룸
