### 전체 순서 브로드캐스트 (Total Order Broadcast) 심화 정리

1. 전체 순서 브로드캐스트란?

1-1. 정의

전체 순서 브로드캐스트는 노드 간 메시지 전달에 대한 프로토콜이다. 단순히 "메시지를 보낸다"가 아니라, 어떤 보장 위에서 보내는가를 정의하는 것이다.

핵심 정의: 모든 노드에게 같은 메시지를 같은 순서로 전달하는 프로토콜

분산 시스템에서 여러 노드가 동시에 메시지를 보내면, 네트워크 지연의 차이로 인해 각 노드가 메시지를 다른 순서로 받을 수 있다. 전체 순서 브로드캐스트는 이 혼란을 해결하여 "하나의 공식적인 순서"를 만들어낸다.

1-2. 왜 이름이 "전체 순서" + "브로드캐스트"인가


브로드캐스트(Broadcast): 메시지를 한 노드가 아니라 모든 노드에 전달한다.
전체 순서(Total Order): 모든 메시지 쌍에 대해 "A가 먼저" 또는 "B가 먼저"를 결정할 수 있는 전체 순서를 부여한다. 비교 불가능한 쌍이 없다.

2. 두 가지 보장 속성 — 깊이 이해하기

전체 순서 브로드캐스트는 정확히 두 가지 속성을 보장한다. 이 두 속성이 동시에 만족되어야만 전체 순서 브로드캐스트라고 할 수 있다.

2-1. 신뢰성 있는 전달 (Reliable Delivery)

"메시지가 한 노드에 전달되면, 모든 정상 노드에 전달된다"

이것이 보장되지 않으면 어떤 일이 벌어지는지 보자:

[신뢰성 없는 전달]

리더:  "x = 10으로 변경" 메시지를 복제본 A, B, C에 전송

복제본 A: msg 수신 ✅ → x = 10
복제본 B: msg 수신 ✅ → x = 10
복제본 C: 네트워크 문제로 msg 수신 실패 ❌ → x = 이전 값

→ 복제본 C만 다른 상태
→ C에서 읽는 클라이언트는 옛날 값을 봄
→ 시스템 불일치!

신뢰성 있는 전달은 "하나가 받았으면 모두 받는다"를 보장한다. 네트워크 장애로 일시적으로 전달이 안 되더라도, 복구 후 반드시 전달되어야 한다.

주의할 점: "모든 노드"가 아니라 "모든 정상(non-crashed) 노드"다. 영구적으로 죽은 노드까지 보장하지는 않는다. 하지만 크래시-복구 모델에서 복구된 노드는 빠진 메시지를 따라잡아야(catch up) 한다.

이것이 단순한 UDP 멀티캐스트와 다른 점이다. UDP 멀티캐스트는 best-effort로, 패킷이 유실되면 그냥 잃어버린다. 전체 순서 브로드캐스트는 유실 시 재전송, 확인(acknowledgment) 등의 메커니즘으로 반드시 전달을 보장한다.

2-2. 전체 순서 전달 (Totally Ordered Delivery)

"모든 노드가 메시지를 동일한 순서로 받는다"

[순서가 보장되지 않으면]

세 개의 쓰기 요청이 거의 동시에 발생:
  msg1: x = 10
  msg2: x = 20
  msg3: x = 30

노드 A가 본 순서: msg1 → msg2 → msg3 → 최종 x = 30
노드 B가 본 순서: msg3 → msg1 → msg2 → 최종 x = 20
노드 C가 본 순서: msg2 → msg3 → msg1 → 최종 x = 10

→ 같은 연산을 적용했는데 세 노드의 최종 상태가 다 다름!


[전체 순서가 보장되면]

모든 노드가 본 순서: msg1 → msg2 → msg3

노드 A: x = 10 → 20 → 30 → 최종 x = 30
노드 B: x = 10 → 20 → 30 → 최종 x = 30
노드 C: x = 10 → 20 → 30 → 최종 x = 30

→ 모든 노드가 동일한 최종 상태에 도달 ✅

핵심 통찰: 순서가 "올바른" 순서(실제 발생 시간 기준)일 필요는 없다. msg2가 msg1보다 물리적으로 먼저 발생했더라도, 모든 노드가 msg1 → msg2 순서로 보기로 합의하면 그것이 이 시스템의 진실이 된다. 모든 노드가 같은 순서를 보는 것 자체가 중요하다.

2-3. 두 속성의 결합이 주는 힘

두 속성이 개별적으로는 부족하다:


신뢰성만 있고 순서가 없으면 → 모든 노드가 메시지를 받지만, 순서가 달라서 상태가 갈라짐
순서만 있고 신뢰성이 없으면 → 순서는 같지만 일부 노드가 메시지를 못 받아서 상태가 뒤처짐


두 속성이 합쳐지면: 모든 노드가 같은 메시지를 같은 순서로 처리하므로, 동일한 초기 상태에서 출발하면 동일한 최종 상태에 도달한다. 이것이 바로 상태 머신 복제(State Machine Replication, SMR) 의 원리다.


3. 전체 순서 브로드캐스트가 없으면 생기는 문제들

3-1. 데이터베이스 복제 불일치

리더-팔로워 복제에서 팔로워들이 쓰기를 다른 순서로 적용하면 데이터가 갈라진다:

리더가 두 개의 쓰기를 순서대로 처리:
  write1: 계좌 잔고 100 → 200 (입금)
  write2: 계좌 잔고 200 → 150 (출금)

[순서가 꼬이면]
팔로워 A: write1 → write2 적용 → 잔고 = 150 ✅
팔로워 B: write2 → write1 적용 → 잔고 = 200 ❌

→ 팔로워 B는 출금을 먼저 적용했는데,
  "100에서 50 빼기"가 아니라 "100에서 200으로 설정"하는
  절대값 쓰기였다면 잔고 200 → 100으로 다른 결과가 나옴

3-2. 유일성 제약 위반

동시에 두 사용자가 같은 이름으로 가입:
  요청1: register("celina") from 서울
  요청2: register("celina") from 부산

[전체 순서 없으면]
노드 A: 요청1이 먼저 도착 → "celina"는 서울 유저
노드 B: 요청2가 먼저 도착 → "celina"는 부산 유저
→ 노드마다 다른 결론! 유일성 위반!

[전체 순서 있으면]
모든 노드: 요청1 → 요청2 순서로 확정
→ 요청1 승인, 요청2 거부 → 모든 노드가 동일한 결론

3-3. 분산 잠금 충돌

두 클라이언트가 동시에 잠금 획득 시도:
  lockReq1: 클라이언트 A가 잠금 요청
  lockReq2: 클라이언트 B가 잠금 요청

[전체 순서 없으면]
노드 X: A의 요청이 먼저 → A에게 잠금 부여
노드 Y: B의 요청이 먼저 → B에게 잠금 부여
→ 두 클라이언트 모두 "내가 잠금을 갖고 있다"고 생각 → Split Brain

[전체 순서 있으면]
모든 노드: lockReq1 → lockReq2 순서 확정
→ A만 잠금 획득, B는 대기 → 모든 노드가 동의

3-4. 트랜잭션 직렬화

두 트랜잭션이 동시에 실행:
  T1: if (stock > 0) { stock -= 1; order += 1; }
  T2: if (stock > 0) { stock -= 1; order += 1; }

현재 stock = 1

[순서가 없으면]
두 트랜잭션 모두 stock > 0을 확인하고 실행
→ stock = -1, order = 2 (재고 없는데 두 번 주문!)

[전체 순서로 직렬화하면]
T1 먼저 → stock = 0, order = 1
T2: stock > 0 실패 → 거부ㅃㅁ
→ 정합성 유지
