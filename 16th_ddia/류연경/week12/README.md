# DDIA 11장: 스트림 처리 (Stream Processing)


## 1. 이벤트 스트림이란?

### 1-1. 이벤트의 개념

스트림 처리에서 데이터의 단위는 **이벤트(event)** 다.

```
이벤트 예시:
- 유저가 페이지를 조회함 (timestamp, user_id, page_url)
- 센서가 온도를 측정함 (timestamp, sensor_id, temperature)
- 유저가 상품을 주문함 (timestamp, order_id, items, total)
```

이벤트는 한 번 생성되면 **불변(immutable)** 이다. "유저가 14:30에 페이지를 조회했다"는 사실은 바뀌지 않는다. 이 점이 데이터베이스의 가변(mutable) 레코드와 다른 핵심 차이다.

### 1-2. 이벤트의 생산자와 소비자

```
생산자 (Producer)          이벤트             소비자 (Consumer)
─────────────            ─────            ─────────────
웹 서버            →     클릭 이벤트    →    추천 시스템
IoT 센서           →     측정 이벤트    →    모니터링 대시보드
결제 서비스        →     결제 이벤트    →    회계 시스템, 알림 서비스
```

하나의 이벤트를 **여러 소비자**가 받을 수 있다. 결제 이벤트를 회계 시스템도, 알림 서비스도, 사기 탐지 시스템도 동시에 소비할 수 있다.

## 2. 메시징 시스템 (Messaging Systems)

생산자가 만든 이벤트를 소비자에게 전달하는 방법이다.

### 2-1. 직접 메시징 (Direct Messaging)

생산자가 소비자에게 직접 보내는 방식:

- **UDP 멀티캐스트**: 금융 시장 데이터 피드 등. 빠르지만 메시지 유실 가능
- **TCP 직접 연결**: 생산자가 소비자의 주소를 알고 직접 전송
- **웹훅(Webhook)**: HTTP 요청으로 이벤트를 전달

문제점: 소비자가 죽거나 느려지면 생산자가 대기해야 하거나 메시지가 유실된다.

### 2-2. 메시지 브로커 (Message Broker)

생산자와 소비자 사이에 **중간 서버(브로커)** 를 두는 방식:

```
생산자 → [메시지 브로커] → 소비자

생산자: 브로커에 메시지를 보내고 바로 리턴 (비동기)
브로커: 메시지를 저장하고, 소비자가 가져갈 때까지 보관
소비자: 자기 속도에 맞춰서 메시지를 가져감
```

**브로커의 장점**:

- 소비자가 일시적으로 죽어도 메시지가 유실되지 않음 (버퍼링)
- 생산자와 소비자의 속도 차이를 흡수 (backpressure 해결)
- 생산자와 소비자가 서로를 몰라도 됨 (디커플링)

### 2-3. 메시지 브로커의 두 가지 모델

**① 전통적 메시지 큐 (JMS/AMQP 스타일)**

RabbitMQ, ActiveMQ 등이 해당된다.

- 메시지를 소비자에게 전달하면 브로커에서 **삭제**
- 하나의 메시지는 **하나의 소비자**에게만 전달 (work queue)
- 순서 보장이 어려움 (여러 소비자가 병렬로 가져가면 순서가 꼬임)

```
큐: [msg1, msg2, msg3, msg4]

소비자 A가 msg1 가져감 → 처리 중...
소비자 B가 msg2 가져감 → 빨리 처리 완료
소비자 B가 msg3 가져감 → 빨리 처리 완료

→ msg2, msg3이 msg1보다 먼저 처리됨 → 순서 역전!
```

**② 로그 기반 메시지 브로커 (Log-based)**

Apache Kafka, Amazon Kinesis, Redpanda 등이 해당된다.

- 메시지를 **로그에 추가(append)** 하고 삭제하지 않음
- 소비자가 **오프셋(offset)** 으로 자기 위치를 추적
- 파티션 내에서 **순서 보장**
- 같은 메시지를 **여러 소비자 그룹**이 독립적으로 소비 가능

```
파티션 0: [offset 0] [offset 1] [offset 2] [offset 3] ...
                                     ↑
                              소비자 그룹 A의 현재 위치

                         ↑
                  소비자 그룹 B의 현재 위치

→ A와 B가 같은 데이터를 독립적으로 소비
→ 각자 자기 속도로 읽을 수 있음
→ 메시지가 삭제되지 않으므로 과거 데이터도 재처리 가능
```

### 2-4. 로그 기반 vs 전통적 메시지 큐 비교

| 구분 | 전통적 큐 (RabbitMQ) | 로그 기반 (Kafka) |
|---|---|---|
| 메시지 보관 | 소비 후 삭제 | 보관 (retention 기간까지) |
| 소비 모델 | 하나의 메시지 → 하나의 소비자 | 하나의 메시지 → 여러 소비자 그룹 |
| 순서 보장 | 어려움 | 파티션 내 보장 |
| 재처리 | 불가능 (삭제됨) | 가능 (오프셋 되감기) |
| 적합한 경우 | 각 메시지를 정확히 한 번 처리해야 할 때 | 이벤트 스트림, 여러 소비자가 같은 데이터를 봐야 할 때 |

## 3. 데이터베이스와 스트림

이 파트가 11장에서 가장 통찰력 있는 부분이다. 데이터베이스를 이벤트 스트림의 관점에서 바라본다.

### 3-1. 변경 데이터 캡처 (CDC, Change Data Capture)

데이터베이스에 일어나는 모든 변경을 **이벤트 스트림으로 추출**하는 기법이다.

```
[CDC 없이]

사용자 → 주문 DB에 INSERT
         → 검색 인덱스도 업데이트해야 함
         → 캐시도 무효화해야 함
         → 데이터웨어하우스에도 보내야 함
         → 각 시스템마다 개별적으로 동기화 로직 필요 → 복잡


[CDC 사용]

사용자 → 주문 DB에 INSERT
              │
              ▼
         CDC가 변경을 캡처 → 이벤트 스트림 (Kafka)
              │
              ├── → 검색 인덱스 (Elasticsearch) 업데이트
              ├── → 캐시 (Redis) 무효화
              ├── → 데이터웨어하우스 동기화
              └── → 알림 서비스
```

CDC의 핵심: DB의 변경 로그(WAL, binlog 등)를 읽어서 이벤트로 변환한다. DB가 **유일한 쓰기 원천(single source of truth)** 이 되고, 나머지 시스템은 파생 데이터(derived data)를 유지한다.

대표 도구: Debezium (MySQL/PostgreSQL → Kafka), Maxwell, Canal

### 3-2. 이벤트 소싱 (Event Sourcing)

CDC가 "기존 DB의 변경을 캡처"하는 것이라면, 이벤트 소싱은 **처음부터 이벤트를 1차 데이터로 저장**하는 패턴이다.

```
[일반적인 방식 — 현재 상태만 저장]

장바구니 테이블:
  user_123: [상품A, 상품B, 상품C]   ← 현재 상태만 알 수 있음
  → 상품B를 언제 넣었는지? 삭제했다가 다시 넣은 건지? 모름


[이벤트 소싱 — 모든 이벤트를 저장]

이벤트 로그:
  1. CartItemAdded(user_123, 상품A, 14:00)
  2. CartItemAdded(user_123, 상품B, 14:05)
  3. CartItemRemoved(user_123, 상품B, 14:10)
  4. CartItemAdded(user_123, 상품B, 14:15)  ← 다시 넣음
  5. CartItemAdded(user_123, 상품C, 14:20)

→ 현재 상태: [상품A, 상품B, 상품C] (이벤트를 재생하면 도출)
→ 전체 히스토리를 알 수 있음
→ "왜 이 상태가 됐는지"를 추적 가능
```

### 3-3. 이벤트 소싱 vs CDC 비교

| 구분 | CDC | 이벤트 소싱 |
|---|---|---|
| 이벤트 원천 | 기존 DB의 변경 로그에서 추출 | 애플리케이션이 직접 이벤트를 기록 |
| 이벤트 수준 | 저수준 (row 변경: INSERT/UPDATE/DELETE) | 고수준 (비즈니스 의미: "주문 생성", "배송 시작") |
| 기존 시스템 | 기존 DB를 그대로 두고 추가 | 처음부터 이벤트 중심으로 설계 |

### 3-4. 불변 이벤트의 힘

11장의 핵심 통찰 중 하나: 이벤트를 삭제하지 않고 모두 남겨두면 강력한 이점이 있다.

- **감사(Audit)**: 무슨 일이 일어났는지 전체 히스토리를 추적
- **디버깅**: 버그 발생 시 이벤트를 재생하여 원인 파악
- **새 시스템 추가**: 과거 이벤트를 처음부터 재생하면 새 시스템의 상태를 구축 가능
- **실험**: 같은 이벤트를 다른 로직으로 처리해보기

회계에서 사용하는 **원장(ledger)** 과 같은 개념이다. 잘못된 거래가 있으면 삭제하는 게 아니라 "취소 거래"를 추가한다.

## 4. 스트림 처리 (Processing Streams)

이벤트 스트림을 받아서 실제로 처리하는 방법들이다.

### 4-1. 스트림 처리의 세 가지 유형

**① 스트림 → 다른 시스템으로 내보내기**

Kafka → Elasticsearch (검색 인덱스 구축), Kafka → 데이터웨어하우스 (분석) 등. CDC의 소비자 측에 해당한다.

**② 스트림 → 이벤트를 사용자에게 직접 전달**

알림, 이메일, 실시간 대시보드 등이다.

**③ 스트림 → 스트림 처리 → 새 스트림 출력 (연산자 체이닝)**

입력 스트림을 변환, 필터링, 집계하여 새 스트림을 생성한다. 여러 처리 단계를 파이프라인으로 연결할 수 있다.

### 4-2. 스트림 조인 (Stream Joins)

스트림 처리에서 가장 어려운 부분 중 하나다. 세 가지 유형이 있다.

**① 스트림-스트림 조인 (Stream-Stream Join / Window Join)**

두 개의 흘러가는 스트림을 시간 윈도우 내에서 매칭하는 방식이다.

```
예시: "유저가 검색한 후 5분 안에 클릭한 결과를 찾아라"

검색 이벤트 스트림: user_123이 "노트북"을 검색 (14:00)
클릭 이벤트 스트림: user_123이 상품X를 클릭 (14:03)

→ 두 스트림을 시간 윈도우(5분) 내에서 user_id로 조인
→ "user_123이 '노트북' 검색 후 3분 뒤에 상품X를 클릭함"
```

두 스트림 모두 흘러가고 있으므로, 한쪽의 이벤트를 일정 시간(윈도우) 동안 상태로 보관하면서 다른 쪽과 매칭해야 한다.

**② 스트림-테이블 조인 (Stream-Table Join / Enrichment)**

흘러가는 스트림의 이벤트에 테이블의 정보를 덧붙이는 방식이다.

```
예시: "주문 이벤트에 유저 프로필 정보를 덧붙여라"

주문 스트림: {order_id: 1, user_id: 123, item: "노트북"}
유저 테이블: {user_id: 123, name: "연경", tier: "VIP"}

→ 주문 이벤트가 올 때마다 유저 테이블에서 user_id로 조회
→ 결과: {order_id: 1, user: "연경", tier: "VIP", item: "노트북"}
```

스트림 프로세서가 테이블의 로컬 복사본을 유지하거나, DB에 질의해서 이벤트를 풍부하게(enrich) 만든다.

**③ 테이블-테이블 조인 (Table-Table Join / Materialized View)**

두 테이블의 변경 스트림을 조인하여 물질화 뷰를 실시간으로 유지하는 방식이다.

```
예시: "트위터 타임라인 — 팔로우하는 사람의 트윗을 보여줘"

트윗 테이블 (CDC 스트림): 새 트윗 발생
팔로우 테이블 (CDC 스트림): 팔로우 관계 변경

→ 두 테이블의 변경 스트림을 조인
→ 각 유저의 타임라인(물질화 뷰)을 실시간 업데이트
```

### 4-3. 시간과 윈도우 (Time and Windows)

스트림 처리에서 "시간"은 까다로운 문제다.

**이벤트 시간 vs 처리 시간:**

```
이벤트 시간 (Event Time): 이벤트가 실제로 발생한 시각
  → 모바일 앱에서 14:00에 클릭

처리 시간 (Processing Time): 서버가 이벤트를 받은 시각
  → 네트워크 지연으로 14:05에 도착

차이: 5분 → 이걸 무시하면 집계 결과가 틀림
```

**윈도우 유형:**

**텀블링 윈도우 (Tumbling Window)**: 겹치지 않는 고정 크기 윈도우. "1분마다 이벤트 수를 세라."

```
|──1분──|──1분──|──1분──|
  윈도우1  윈도우2  윈도우3
```

**호핑 윈도우 (Hopping Window)**: 겹치는 고정 크기 윈도우. "1분마다, 최근 2분간의 이벤트 수를 세라."

```
|────2분────|
     |────2분────|
          |────2분────|
```

**슬라이딩 윈도우 (Sliding Window)**: 이벤트 간 시간 차이 기준. "5분 이내에 발생한 이벤트끼리 그룹핑."

**세션 윈도우 (Session Window)**: 활동 간 간격(gap) 기준. "30분 이상 활동이 없으면 새 세션."

### 4-4. 내결함성 (Fault Tolerance)

스트림 처리에서 노드가 죽으면? 배치와 달리 "처음부터 다시"가 어렵다 — 입력이 무한하니까.

**마이크로배칭 (Microbatching):**
- Spark Streaming이 사용하는 방식
- 스트림을 작은 배치(예: 1초 단위)로 잘라서 각각을 배치 처리
- 배치가 실패하면 해당 배치만 재실행

**체크포인팅 (Checkpointing):**
- Apache Flink이 사용하는 방식
- 주기적으로 상태의 스냅샷을 저장
- 실패 시 마지막 체크포인트부터 재개

**멱등성 (Idempotency):**
- 같은 이벤트를 두 번 처리해도 결과가 같도록 설계
- 중복 처리의 영향을 제거

## 5. 배치 처리 vs 스트림 처리

| 구분 | 배치 처리 (10장) | 스트림 처리 (11장) |
|---|---|---|
| 입력 | 유한한 데이터셋 | 끝없는 이벤트 스트림 |
| 처리 시점 | 데이터를 모아서 한꺼번에 | 이벤트 발생 즉시 (또는 거의 즉시) |
| 지연 | 분~시간 | 밀리초~초 |
| 대표 도구 | MapReduce, Spark (배치) | Kafka Streams, Flink, Spark Streaming |
| 상태 | 작업 간 상태 없음 (stateless) | 윈도우, 조인 등을 위해 상태 유지 필요 |
| 내결함성 | 실패 시 전체 재실행 | 체크포인팅, 마이크로배칭 |

## 6. 11장의 핵심 흐름 정리

```
"데이터가 끝없이 흘러올 때 어떻게 처리할 것인가?"
        │
        ├── 이벤트 전달: 메시징 시스템
        │     ├── 전통적 큐: 소비 후 삭제, 하나의 소비자
        │     └── 로그 기반 (Kafka): 보관, 여러 소비자, 순서 보장
        │
        ├── DB와 스트림의 연결
        │     ├── CDC: DB 변경을 이벤트 스트림으로 캡처
        │     ├── 이벤트 소싱: 이벤트를 1차 데이터로 저장
        │     └── 불변 이벤트의 힘: 감사, 디버깅, 재처리
        │
        ├── 스트림 처리
        │     ├── 조인: 스트림-스트림, 스트림-테이블, 테이블-테이블
        │     ├── 시간: 이벤트 시간 vs 처리 시간, 윈도우
        │     └── 내결함성: 마이크로배칭, 체크포인팅, 멱등성
        │
        └── 배치와 스트림의 통합
              ├── Lambda 아키텍처: 배치 + 스트림 병행
              └── Kappa 아키텍처: 스트림만으로 통합
```

> **11장의 핵심 메시지**: 이벤트 스트림은 단순한 메시징을 넘어서, 시스템 간 데이터 통합의 핵심 메커니즘이다. CDC와 이벤트 소싱으로 데이터베이스를 이벤트 관점에서 바라보면, 파생 데이터를 유연하게 구축하고 시스템을 디커플링할 수 있다.
