
## 파티셔닝이란?

데이터가 너무 많아서 하나의 노드(서버)에 다 담을 수 없을 때, 데이터를 **여러 노드에 쪼개서 분산 저장**하는 것.

```
[파티셔닝 전]                  [파티셔닝 후]
┌────────────────┐      ┌─────────┐ ┌─────────┐ ┌─────────┐
│  전체 데이터     │      │ 파티션1  │ │ 파티션2  │ │ 파티션3  │
│  (한 노드)      │  →   │ A ~ H   │ │ I ~ P   │ │ Q ~ Z   │
│  10TB          │      │ 노드1    │ │ 노드2    │ │ 노드3    │
└────────────────┘      └─────────┘ └─────────┘ └─────────┘
```

### 목적

- **확장성(Scalability)**: 단일 노드의 저장/처리 한계를 넘기 위해
- 각 파티션은 그 자체로 작은 데이터베이스처럼 독립적으로 동작
- 질의 처리량(throughput)을 노드 수에 비례해서 늘릴 수 있음

### 파티셔닝과 복제(Replication)의 관계

파티셔닝과 복제는 보통 **함께 사용**된다.

```
노드1: [파티션A 리더]  [파티션B 팔로워]  [파티션C 팔로워]
노드2: [파티션A 팔로워] [파티션B 리더]   [파티션C 팔로워]
노드3: [파티션A 팔로워] [파티션B 팔로워]  [파티션C 리더]
```

- 각 파티션은 여러 노드에 복제본을 가짐
- 한 노드가 여러 파티션의 리더가 될 수 있음
- 파티셔닝 방식과 복제 방식은 독립적인 선택

## 키-값 데이터의 파티셔닝

핵심 목표: 데이터와 질의 부하를 노드 간에 **균등하게** 분배하는 것.

불균등하게 분배되면 **쏠림(skew)** 이 발생하고, 부하가 집중된 파티션을 **핫스팟(hot spot)** 이라 부른다.

### 방법 1: 키 범위 기반 파티셔닝 (Key Range Partitioning)

```
파티션1: a ~ g     (apple, banana, cherry, ...)
파티션2: h ~ n     (ham, ice, jam, ...)
파티션3: o ~ z     (orange, peach, quince, ...)
```

- 키를 **정렬된 순서**로 연속 범위에 따라 분할
- 각 파티션 내에서 키가 정렬되어 있으므로 **범위 검색(range scan)** 이 효율적
- 백과사전의 권별 분류(A~C권, D~F권...)와 같은 원리

#### 장점

- 범위 질의가 빠름: `WHERE timestamp BETWEEN '2024-01-01' AND '2024-01-31'` → 하나의 파티션에서 순차 읽기
- 키가 정렬되어 있어 인접한 데이터를 한 번에 가져올 수 있음

#### 단점: 핫스팟 위험

```
예: 센서 데이터를 타임스탬프로 파티셔닝
파티션1: 2024-01 ~ 2024-04  (과거, 거의 읽기만)
파티션2: 2024-05 ~ 2024-08  (과거, 거의 읽기만)
파티션3: 2024-09 ~ 2024-12  (현재! 모든 쓰기가 여기로 집중)
                              ↑ 핫스팟 발생!
```

- 시계열 데이터에서 특히 문제: 현재 시간대 파티션에 쓰기가 집중됨
- 해결: 키 앞에 센서 이름 등을 붙임 → `sensor01_2024-12`, `sensor02_2024-12` → 쓰기 분산

### 방법 2: 해시 기반 파티셔닝 (Hash Partitioning)

```
hash("apple")  = 0x3A → 파티션1 (0x00 ~ 0x55)
hash("banana") = 0xB2 → 파티션3 (0xAB ~ 0xFF)
hash("cherry") = 0x71 → 파티션2 (0x56 ~ 0xAA)
```

- 키에 해시 함수를 적용하고, 해시값의 범위로 파티션 결정
- 비슷한 키라도 해시값은 완전히 달라지므로 **균등 분배**에 유리
- 좋은 해시 함수는 쏠린 데이터도 균일하게 분산시킴

#### 장점

- 핫스팟 위험이 크게 줄어듦
- 데이터가 자동으로 균등 분배

#### 단점

- 키의 정렬 순서가 사라짐 → **범위 질의 불가능**
- `WHERE price BETWEEN 100 AND 500` 같은 질의는 모든 파티션에 보내야 함

#### 일관된 해싱 (Consistent Hashing)

- 노드 추가/제거 시 재배치되는 데이터량을 최소화하는 기법
- CDN 등에서 사용하지만, 데이터베이스 파티셔닝에서는 잘 안 씀 (혼동 주의)
- 책에서는 "해시 파티셔닝"이라는 용어를 선호

### 절충안: 복합 키 (Compound Key)

키의 첫 부분으로 파티션을 결정하고, 두 번째 부분으로 파티션 내 정렬을 하는 방식.

```
키 설계: (user_id, timestamp)

hash(user_id) → 파티션 결정    (균등 분배)
timestamp     → 파티션 내 정렬  (범위 검색 가능)
```

- Cassandra의 복합 기본키가 이 방식
- `user_id = 'user123'`인 데이터는 하나의 파티션에 모이고, 그 안에서 timestamp 순으로 정렬
- 특정 사용자의 최근 데이터를 범위로 조회 가능

## 쏠림(Skew)과 핫스팟 완화

해시 파티셔닝도 완벽하지 않다. 극단적 경우에는 여전히 핫스팟이 생길 수 있다.

### 예시: 유명인 문제 (Celebrity Problem)

```
SNS에서 팔로워가 1억인 유저가 글을 올리면:
hash("celebrity_user_id") → 파티션7

→ 파티션7에 읽기 요청이 폭주 (핫스팟)
```

- 동일한 키에 대한 읽기/쓰기가 극단적으로 몰리는 경우
- 해시로도 분산 불가능 (같은 키 = 같은 해시값)

### 현재의 해결 방법

- 대부분의 DB는 자동으로 해결해주지 않음
- 애플리케이션 레벨에서 처리해야 함

```
// 핫 키에 랜덤 접두사/접미사 붙이기
key = "celebrity_123" + "_" + random(0, 9)
→ "celebrity_123_0", "celebrity_123_1", ... "celebrity_123_9"
→ 10개 파티션에 분산

// 단점: 읽을 때 10개 키를 모두 읽어서 합쳐야 함
```

- 소수의 핫 키에만 적용해야 함 (모든 키에 적용하면 오버헤드)
- 어떤 키가 핫 키인지 아는 것도 애플리케이션의 몫

## 파티셔닝과 보조 인덱스 (Secondary Index)

키-값 데이터의 파티셔닝은 기본 키(PK)로만 접근하면 간단하다. 문제는 **보조 인덱스**가 있을 때. 보조 인덱스는 기본 키와 다른 기준으로 데이터를 찾기 때문에 파티션 경계와 잘 맞지 않는다.

### 방법 1: 문서 기준 보조 인덱스 (Document-Partitioned / Local Index)

각 파티션이 **자기 파티션의 데이터만** 커버하는 로컬 보조 인덱스를 가짐.

```
파티션0                          파티션1
┌─────────────────────┐   ┌─────────────────────┐
│ 데이터:              │   │ 데이터:              │
│  id=1, color=red     │   │  id=3, color=red     │
│  id=2, color=blue    │   │  id=4, color=blue    │
│                      │   │  id=5, color=red     │
│ 보조 인덱스:          │   │ 보조 인덱스:          │
│  color:red  → [1]    │   │  color:red  → [3,5]  │
│  color:blue → [2]    │   │  color:blue → [4]    │
└─────────────────────┘   └─────────────────────┘
```

#### 쓰기

- 해당 파티션의 인덱스만 갱신하면 됨 → 빠름
- 하나의 파티션 안에서 완결

#### 읽기: 스캐터/개더 (Scatter/Gather)

```
WHERE color = 'red' 질의:
→ 모든 파티션에 질의를 보내야 함 (어디에 red가 있는지 모르니까)
→ 결과를 합침

클라이언트 → 파티션0 (color:red → [1])
          → 파티션1 (color:red → [3,5])
          → 합치기: [1, 3, 5]
```

- **모든 파티션**에 질의를 보내야 하므로 비효율적
- 파티션이 많을수록 지연 시간 증가
- 그럼에도 널리 사용됨: MongoDB, Cassandra, Elasticsearch, Riak, SolrCloud 등

### 방법 2: 용어 기준 보조 인덱스 (Term-Partitioned / Global Index)

보조 인덱스 자체를 파티셔닝. 인덱스의 **용어(term)** 가 어떤 파티션에 속할지 결정.

```
인덱스 파티션0 (a~m 용어)        인덱스 파티션1 (n~z 용어)
┌─────────────────────┐   ┌─────────────────────┐
│ color:blue → [2,4]   │   │ color:red → [1,3,5]  │
│ color:black → [7]    │   │ color:white → [6]    │
└─────────────────────┘   └─────────────────────┘

(데이터 자체는 별도 파티션에 분산되어 있음)
```

#### 읽기

- 원하는 용어가 속한 인덱스 파티션 **하나만** 읽으면 됨 → 효율적
- `WHERE color = 'red'` → 인덱스 파티션1만 조회

#### 쓰기

- 하나의 문서를 쓸 때 여러 인덱스 파티션에 영향을 줄 수 있음
- 예: `{id=8, color=red, make=BMW}` 삽입 → color 인덱스 파티션 + make 인덱스 파티션 모두 갱신
- **분산 트랜잭션**이 필요하거나, 비동기로 갱신 (약간의 지연 발생)

### 두 방식 비교

|관점|문서 기준 (로컬)|용어 기준 (글로벌)|
|---|---|---|
|쓰기|빠름 (로컬 파티션만)|느림 (여러 파티션 갱신)|
|읽기|느림 (스캐터/개더)|빠름 (한 파티션만)|
|인덱스 최신성|즉시 반영|비동기 시 지연 가능|
|구현 복잡도|단순|복잡 (분산 트랜잭션 or 비동기)|
|사용 예|MongoDB, Elasticsearch|Amazon DynamoDB (GSI)|

## 파티션 재균형화 (Rebalancing)

시간이 지나면 재균형화가 필요해진다:

- 질의 처리량 증가 → 노드 추가
- 데이터 크기 증가 → 디스크 추가
- 노드 장애 → 다른 노드가 인수

### 재균형화의 요구사항

1. 재균형화 후 부하가 노드 간에 균등해야 함
2. 재균형화 중에도 읽기/쓰기가 가능해야 함
3. 필요 이상의 데이터를 옮기면 안 됨 (네트워크/디스크 I/O 최소화)

### ❌ 나쁜 방법: hash(key) mod N

```
노드 3개: hash(key) % 3 → 0, 1, 2
노드 4개로 늘리면: hash(key) % 4 → 0, 1, 2, 3

→ 대부분의 키가 다른 노드로 재배치됨!
→ 불필요한 데이터 이동이 대량 발생
```

- N(노드 수)이 바뀌면 거의 모든 키의 파티션이 바뀜
- 절대 쓰면 안 되는 방법

### 전략 1: 고정 개수 파티션 (Fixed Number of Partitions)

```
처음부터 파티션을 넉넉히 만들어 놓음 (예: 1000개)

노드 3개일 때:
노드1: 파티션 1~333
노드2: 파티션 334~666
노드3: 파티션 667~1000

노드 4개로 늘리면: 각 노드에서 파티션 일부를 새 노드에 넘김
노드1: 파티션 1~250
노드2: 파티션 334~583
노드3: 파티션 667~916
노드4: 파티션 251~333, 584~666, 917~1000  ← 기존 노드에서 받아옴
```

- 파티션 **개수는 고정**, 노드 간 파티션의 **배정만 변경**
- 파티션 자체는 쪼개지거나 합쳐지지 않음
- 사용: Riak, Elasticsearch, Couchbase, Voldemort

#### 파티션 개수 선택의 어려움

- 너무 적으면: 노드 추가 시 유연성 부족, 파티션당 데이터가 너무 큼
- 너무 많으면: 관리 오버헤드, 파티션 간 데이터 이동 시 부담
- 처음에 정해야 하고 나중에 바꾸기 어려움

### 전략 2: 동적 파티셔닝 (Dynamic Partitioning)

```
파티션이 임계 크기를 넘으면 → 둘로 분할 (split)
파티션이 너무 작아지면     → 인접 파티션과 병합 (merge)

[파티션A: 10GB]  → 임계치 초과
  ↓
[파티션A1: 5GB] [파티션A2: 5GB]  → 분할
```

- 키 범위 파티셔닝에서 특히 유용 (HBase, RethinkDB)
- 데이터 양에 맞게 파티션 수가 자동 조절
- B 트리의 페이지 분할과 유사한 원리

#### 초기 문제: 사전 분할 (Pre-splitting)

- 처음에 데이터가 적으면 파티션이 1개뿐 → 한 노드에 모든 부하 집중
- 해결: 초기 파티션 경계를 미리 설정 (HBase, MongoDB의 pre-splitting)

### 전략 3: 노드 비례 파티셔닝 (Proportional to Nodes)

```
규칙: 노드당 고정 개수의 파티션 (예: 노드당 256개)

노드 3개: 3 × 256 = 768 파티션
노드 4개: 4 × 256 = 1024 파티션 (기존 파티션을 랜덤 분할)
```

- 새 노드가 추가되면 기존 파티션을 랜덤으로 선택하여 절반을 가져옴
- 파티션 크기가 비교적 안정적으로 유지됨
- 사용: Cassandra, Ketama

### 세 전략 비교

|관점|고정 개수|동적|노드 비례|
|---|---|---|---|
|파티션 수 결정|초기에 고정|데이터 크기에 따라 자동|노드 수에 비례|
|분할/병합|없음|자동 분할/병합|노드 추가 시 분할|
|설정 난이도|초기 파티션 수 선택 필요|자동 (초기 pre-split 권장)|노드당 파티션 수만 설정|
|대표 사용|Elasticsearch, Riak|HBase, MongoDB|Cassandra|

### 자동 재균형화 vs 수동 재균형화

- **완전 자동**: 편리하지만 위험할 수 있음
    - 재균형화 중 네트워크 부하 증가 → 다른 노드가 느려짐 → 장애로 오인 → 연쇄 재균형화 (Cascading)
- **수동 개입**: 관리자가 재균형화 계획을 확인 후 승인
    - 예상치 못한 연쇄 반응 방지
    - 자동 제안 + 수동 승인 = 좋은 절충안

## 요청 라우팅 (Request Routing)

클라이언트가 요청을 보낼 때, **어떤 노드에 어떤 파티션이 있는지** 어떻게 아는가?

이것은 **서비스 디스커버리(Service Discovery)** 의 문제.

### 세 가지 접근법

```
[방법 1] 아무 노드에 요청 → 해당 노드가 올바른 노드로 전달

클라이언트 → 노드1 → "이건 내 파티션 아닌데" → 노드3에 전달 → 응답

[방법 2] 라우팅 계층(파티션 인식 로드밸런서)을 거침

클라이언트 → 라우팅 계층 → "이 키는 노드3" → 노드3 → 응답

[방법 3] 클라이언트가 직접 파티션-노드 매핑을 알고 있음

클라이언트 (파티션 맵 보유) → 노드3 → 응답
```

### 파티션-노드 매핑 정보의 관리

어떤 접근법을 쓰든, **파티션이 어떤 노드에 있는지** 정보를 누군가 관리해야 한다.

#### ZooKeeper를 이용한 방식

```
┌────────────┐
│ ZooKeeper  │  ← 파티션-노드 매핑의 진실의 원천 (Source of Truth)
│            │  ← 파티션 변경 시 여기에 반영
└─────┬──────┘
      │ 변경 통보 (watch)
      ↓
┌────────────┐     ┌────────┐
│ 라우팅 계층  │     │클라이언트│
│ or 각 노드  │     │ (구독)  │
└────────────┘     └────────┘
```

- ZooKeeper가 파티션 할당의 권위적 매핑을 유지
- 파티션이 재균형화되면 ZooKeeper에 반영
- 라우팅 계층(또는 노드, 클라이언트)이 ZooKeeper를 구독(watch)하여 변경 감지
- 사용: HBase, SolrCloud, Kafka (ZooKeeper 의존)

#### 가십 프로토콜 (Gossip Protocol)

- 노드끼리 상태 정보를 주고받아 파티션 매핑을 공유
- 별도의 외부 코디네이터 불필요
- 사용: Cassandra, Riak (방법 1 + 가십)

#### DNS 수준 라우팅

- 가장 단순: IP 주소로 노드를 찾음
- 세밀한 파티션 라우팅은 불가

### 대표 DB별 라우팅 방식

|데이터베이스|라우팅 방식|
|---|---|
|HBase|ZooKeeper + 라우팅 계층|
|Kafka|ZooKeeper (→ KRaft로 전환 중)|
|Cassandra|가십 프로토콜 (아무 노드에 요청)|
|MongoDB|Config Server + mongos 라우터|
|Elasticsearch|아무 노드에 요청 → 내부 전달|
|CockroachDB|가십 + 각 노드가 라우팅|
