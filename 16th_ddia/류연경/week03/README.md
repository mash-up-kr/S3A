### 복제
- 같은 데이터의 복사본을 네트워크로 연결된 여러 머신에 유지하는 것 
##### 왜 복제를 하는가?
1. 지연 시간 감소 - 지리적으로 가깝게 유지 
2. 가용성
3. 읽기 처리량 
##### 복제를 위한 알고리즘
1. 리더 기반 복제 (Single-Leader)
2. 다중 리더 복제 (Multi-Leader)
3. 리더 없는 복제 (Leaderless)

##### 생각해야할 포인트
- 데이터가 변경 될때 복제가 발생한다면? 변경된 내용을 모든 복제본에 어떻게, 언제, 어떤 순서로 반영할 것인가?

#### 1. 리더 기반 복제 (Leader-Based Replication)
##### 1-1 구조 
- 쓰기는 리더에게만 보낸다
- 읽기는 리더, 아무 팔로워에서 읽을 수 있다
- PostgreSQL, MySQL, MSSQL Always On AG, Oracle Data Guard, MongoDB, Kafka, RabbitMQ
##### 1-2 동기식 vs 비동기식
- 리더가 팔로워에게 복제 로그를 보냈을 때, 팔로워의 완료 응답을 기다리느냐 마느냐의 차이 
- 동기식: 팔로워에 항상 최신 데이터 보장,  but 팔로워 하나 때문에 전체 시스템 쓰기 블로킹됨 
- 비동기식: 빠르다. 리더가 갑자기 죽으면 아직 팔로워에게 전달되지 않는 쓰기 유실됨, 내구성의 보장 약해짐
- 반동기식 복제: 팔로워 중 딱 하나만 동기식, 나머지는 비동기, 동기 팔로워가 죽으면 비동기를 동기로 승격
##### 1-3 새 팔로워 세팅
```
1단계: 리더의 스냅샷을 뜬다 (DB를 안 멈추고 뜰 수 있다 — 백업용 기능이 있음)
2단계: 스냅샷을 새 팔로워에 복사한다
3단계: 팔로워가 리더에 연결해서 "이 스냅샷 이후에 뭐가 바뀌었어?"라고 요청한다
4단계: 밀린 변경 로그를 모두 따라잡으면 → 정상 운영에 합류!
```
- 스냅샷이 복제 로그의 정확한 위치와 연결되어야 한다 
- PostgreSQL: **LSN (Log Sequence Number)**
- MySQL: **binlog coordinates**
- MSSQL: **LSN in transaction log**
	**LSN** 
	트랜잭션 로그에 기록된 각 레코드의 고유한 순서 번호, 모든 데이터의 변경은 반드시 트랜잭션 로그에 먼저 기록 
	```
	LSN = VLF번호 : 로그블록번호 : 로그레코드번호
	예:  00000025 :  000001a0  :  0001
	```
	- VLF번호 (Virtual Log File Number): 트랜잭션 로그 파일은 내부적으로 여러 개의 가상로그 파일로 나뉨, 몇번재 VLF인지
	- 로그 블록 번호: 해당 VLF안에서의 오프셋 위치 
	- 로그 레코드 번호: 해당 블록 안에서 몇번째 레코드 
	=>  모든 INSERT, UPDATE, DELETE는 물론이고 트랜잭션의 BEGIN과 COMMIT까지 전부 LSN이 붙어서 기록
	
	- LSN이 사용되는 핵심 장면들
	  1. 크래시 복구 (Crash Recovery)
	  2. 백업과 복원
	  3. 특정 시점 복원
	  4. Log Shipping과 미러링: 리더에서 트랜잭션 로그 백업을 주기적으로 떠서 팔로워에 복원하는 방식

##### 1-4 노드 장애 처리 
- 팔로워 장애: 따라잡기 복구 (Catch-up Recovery): 팔로워는 자기가 마지막으로 처리한 로그의 위치를 로컬 디스크에 기록해둔다. 팔로워가 다운됐다가 다시 살아나면, 그 위치부터 리더에게 로그를 다시 요청
- 리더 장애: 페일오버 (Failover)
	- 페일오버에서 발생할 수 있는 문제들
	 1. 데이터 유실
	 2. 스플릿 브레인 (Split Brain): 두 리더 동시 조냊 
	 3. 외부 시스템과의 불일치

##### 1-5 복제 로그 구현 방식
- 리더가 팔로워에게 "뭐가 바뀌었는지"를 전달하는 방법

1. 구문 기반 복제 (Statement-Based)
2. WAL(Write-Ahead Log) 전송: DB가 디스크에 데이터를 쓰기 전에 먼저 기록하는 로그(WAL)를 그대로 팔로워에 보낸다.
	-  **치명적 단점**: WAL은 **스토리지 엔진의 내부 구조**에 종속적이다. DB 버전이 다르면 디스크 포맷이 달라서 WAL을 해석할 수 없다. → **무중단 업그레이드가 불가능하다.** 팔로워를 먼저 새 버전으로 올린 뒤 페일오버하는 식의 롤링 업그레이드를 못 한다.
	- PostgreSQL, Oracle
3. 논리적(행 기반) 로그 복제: 어떤 테이블의 어떤 행이 어떻게 바뀌었는지 기록 
	- 스토리지 엔진과 독립적 → 리더와 팔로워의 DB 버전이 달라도 됨 → 무중단 업그레이드 가능
	- 이기종 DB 간 복제 가능 (MSSQL → MySQL)
	- 외부 시스템(데이터 웨어하우스, Elasticsearch, 캐시 등)으로 스트리밍 가능 → **CDC(Change Data Capture)**
	- MySQL의 binlog(ROW format), MSSQL의 CDC
		**CDC**
		테이블에서 발생한 데이터 변경(INSERT, UPDATE, DELETE)을 자동으로 캡처해서 별도의 변경 테이블에 기록해주는 SQL Server 내장 기능
4. 트리거 기반 복제: 애플리케이션 레벨에서 DB 트리거로 변경을 감지하고 별도 테이블에 기록한 뒤, 외부 프로세스가 이를 읽어서 다른 DB에 적용
	- Oracle Databus, PostgreSQL Bucardo
	- 트리거 기반과 CDC는 뭐가 다른가?
	- **트리거 기반**: DML이 실행될 때 **같은 트랜잭션 안에서** 트리거 코드가 실행. 원래 쿼리 성능에 직접 영향
	- CDC: 트랜잭션 로그를 **비동기적으로 읽어서** 변경 내용을 추출. 원래 쿼리에 영향을 거의 안 줌. SQL Server Agent 작업이 백그라운드에서 로그를 스캔하는 방식
		

#### 2. 복제 지연 문제 
- 왜 복제 지연이 생기는가?
	리더 기반 복제에서 읽기 요청을 팔로워로 분산시키면 읽기 처리량을 늘릴 수 있다. 하지만 이게 잘 동작하려면 **비동기 복제**를 써야 한다. (동기식으로 하면 팔로워 하나가 느릴 때 전체가 느려지니까)
	
	비동기 복제를 쓰면 리더와 팔로워 사이에 **시간차(replication lag)** 가 생긴다. 평소에는 1초 이내지만, 부하가 걸리면 수 초에서 수 분까지 벌어질 수 있다.
	
	이론적으로는 "최종적 일관성(eventual consistency)"이라고 한다. 팔로워가 언젠가는 따라잡는다는 뜻인데, "언젠가"가 얼마나 걸릴지 보장이 없다.
	
	이 시간차 동안 세 가지 이상한 현상이 발생할 수 있다.

##### 2-1. 자기 쓰기 읽기 (Read-Your-Own-Writes)
- 자기가 방금 쓴 데이터를 자기가 못 읽는 것
- 해결: Read-After-Write Consistency (= Read-Your-Writes Consistency)

##### 2-2. 단조 읽기 (Monotonic Reads)
- **시간이 뒤로 가는 것처럼** 보인다. 새로고침할 때마다 데이터가 나타났다 사라졌다 한다.
- 해결: Monotonic Reads

##### 2-3. 일관된 순서 읽기 (Consistent Prefix Reads)
- 주로 **파티셔닝(샤딩)** 된 DB에서 발생
- 인과 관계(causality)가 뒤집히는 것
- 해결: Consistent Prefix Reads


#### 3. 다중 리더 복제 
- **여러 노드가 각각 쓰기를 받을 수 있게** 하는 것이 다중 리더 복제
- 핵심 문제: 쓰기 충돌
- 충돌 해소 방법들
	- LWW (Last Write Wins)
	- **노드 우선순위**
	- **값 병합 (Merge)**
	- 사용자에게 맡기기 (Conflict Resolution on Read)
	- 자동 충돌 해소 (CRDT, Operational Transformation 등)


#### 4. 리더 없는 복제
- 리더/팔로워 구분 없이, **모든 노드가 쓰기와 읽기를 모두 받을 수 있다.**
- 노드가 다운됐을 때 쓰기
- 리더 기반에서 팔로워가 다운되면 리더가 돌아올 때까지 로그를 모아뒀다가 보내주면 된다.
- 페일오버 발생 안함
- 뒤처진 노드를 따라잡게 하는 방법
	- 읽기 복구 (Read Repair)
	- 안티엔트로피 프로세스 (Anti-Entropy Process)
- 정족수 (Quorum)


---
####  5. 전체 요약: 복제 구조 비교

##### 단일 리더 (Single Leader)

- **쓰기**: 리더 하나만
- **충돌**: 없음
- **일관성**: 강함 (동기식이면)
- **장애 대응**: 페일오버 필요 (복잡)
- **적합한 상황**: 대부분의 워크로드. 가장 이해하기 쉽고 안전함

##### 다중 리더 (Multi-Leader)

- **쓰기**: 여러 리더
- **충돌**: 있음 → 해소 필요
- **일관성**: 약함
- **장애 대응**: DC 단위 내결함
- **적합한 상황**: 다중 데이터센터, 오프라인 동작, 공동 편집

##### 리더 없는 (Leaderless)

- **쓰기**: 아무 노드
- **충돌**: 있음 → 버전 관리로 해소
- **일관성**: 정족수에 의존 (보장은 아님)
- **장애 대응**: 가장 높은 내결함성
- **적합한 상황**: 극단적 가용성이 필요한 경우
