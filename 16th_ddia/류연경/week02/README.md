##  B 트리
- 대부분의 관계형 데이터베이스에서 표준 색인 구현으로 사용
- 키로 정렬된 키-값을 유지하기 때문에 키-값 검색과 범위 질의에 효과적
- 고정 크기 블록이나 페이지로 구성 (가변 크기 세그먼트x)
- 루트에서 시작하여 찾고자 하는 페이지까지 도달한다. 


### 쓰기 동작 (업데이트 & 삽입)

- **값 수정**: 해당 키가 포함된 리프 페이지를 찾아서 값을 덮어쓰고 디스크에 기록
- **새 키 삽입**: 해당 범위의 리프 페이지를 찾아서 추가
- **페이지 분할(split)**: 리프 페이지에 공간이 부족하면 두 개의 반쪽 페이지로 분할 → 상위 페이지도 갱신

### 신뢰성: WAL (Write-Ahead Log)

- B 트리의 변경 사항을 실제 페이지에 적용하기 **전에** 먼저 로그에 기록
- 충돌 복구 시 이 로그를 통해 B 트리를 일관된 상태로 복원
- 동시성 제어: **래치(latch)** = 가벼운 잠금으로 페이지 보호

### B 트리 최적화 기법

- **페이지 분할 대신 덮어쓰기**: 일부 DB는 copy-on-write 방식 사용 (예: LMDB)
- **키 축약**: 내부 노드에 전체 키 대신 경계 정보만 저장 → 분기 계수(branching factor) 증가 → 트리 깊이 감소
- **리프 페이지의 디스크 배치**: 리프 노드를 순차적으로 배치하려 시도하지만, 트리가 커지면 유지가 어려움
- **형제 페이지 포인터**: 리프 노드에 좌우 형제 참조 추가 → 순서대로 스캔 시 상위로 돌아갈 필요 없음

## 클러스터드 인덱스 vs 비클러스터드 인덱스

### 색인에서 값을 저장하는 방식

색인의 값(value)에는 두 가지를 저장할 수 있다:

1. **실제 로우 데이터** → 클러스터드 인덱스
2. **힙 파일 내 위치에 대한 참조(포인터)** → 비클러스터드 인덱스

### 힙 파일 (Heap File)

- 실제 데이터가 저장되는 공간
- 비클러스터드 인덱스는 이 힙 파일의 위치를 참조
- 장점: 여러 보조 인덱스가 있을 때 데이터 중복을 피할 수 있음
- 값 갱신 시 새 값이 기존 공간에 들어가면 힙 위치 유지, 그렇지 않으면 새 위치로 이동 → 모든 인덱스 갱신 필요

### 클러스터드 인덱스 (Clustered Index)

- 색인 안에 **행 데이터를 직접 저장**
- 데이터의 물리적 저장 순서 = 인덱스 순서
- 테이블당 **하나만** 존재 가능
- 예시: MySQL InnoDB의 **기본 키(PK)** 가 클러스터드 인덱스
- 장점: 힙 파일로의 추가 이동(hop) 불필요 → 읽기 성능 향상
- 단점: 쓰기 오버헤드 증가, 저장 공간 증가

### 비클러스터드 인덱스 (Non-Clustered Index)

- 색인에 **힙 파일 위치(포인터)** 만 저장
- 실제 데이터 접근 시 힙 파일로 추가 이동 필요
- 테이블당 **여러 개** 생성 가능

### 커버링 인덱스 (Covering Index)

- 클러스터드와 비클러스터드의 **절충안**
- 인덱스에 일부 칼럼의 값을 포함시킴
- 해당 칼럼만 조회하는 쿼리는 인덱스만으로 응답 가능 → **인덱스만으로 커버**
- 읽기 속도 향상, 쓰기 오버헤드 증가

## OLTP vs OLAP

|속성|OLTP|OLAP|
|---|---|---|
|주 사용자|최종 사용자/애플리케이션|내부 분석가, 의사결정 지원|
|읽기 패턴|키 기준 소량 레코드 조회|대량 레코드에 대한 집계|
|쓰기 패턴|사용자 입력 기반의 랜덤 쓰기|벌크 임포트(ETL) 또는 이벤트 스트림|
|데이터 크기|GB ~ TB|TB ~ PB|
|병목|디스크 탐색(seek)|디스크 대역폭(bandwidth)|
|인덱스 설계|트랜잭션 최적화|칼럼 지향 저장|

### OLTP (Online Transaction Processing)

- 사용자 대면 요청을 처리하는 **트랜잭션** 중심
- 소량의 레코드를 키 기반으로 읽고 쓰기
- 디스크 탐색(seek)이 병목
- B 트리 인덱스가 핵심적인 역할

### OLAP (Online Analytic Processing)

- 비즈니스 인텔리전스(BI) 질의 처리
- 대규모 데이터에 대한 스캔과 집계
- 별도의 **데이터 웨어하우스**로 분리하는 것이 일반적
- 칼럼 지향 저장소가 유리

### 데이터 웨어하우스

- OLTP 시스템에서 ETL(Extract-Transform-Load)로 데이터를 가져옴
- 분석 질의에 최적화된 스키마 사용
- 대표적인 스키마 패턴:
    - **별 모양 스키마(star schema)**
    - **눈꽃 모양 스키마(snowflake schema)** 

### 칼럼 지향 저장소 (Column-Oriented Storage)

- 모든 값을 하나의 로우에 함께 저장하지 않는 대신 각 칼럼별로 모든 값을 함께 저장
- 같은 칼럼의 값은 서로 비슷하거나 반복되는 경우가 많아서 압축 효율이 높다. 

## 구체화 뷰 (Materialized View)

### 일반 뷰 vs 구체화 뷰  

|구분|일반 뷰 (View)|구체화 뷰 (Materialized View)|
|---|---|---|
|저장|쿼리 정의만 저장 (가상 테이블)|**쿼리 결과를 디스크에 실제 저장**|
|조회 시|매번 원본 테이블에서 재계산|미리 계산된 결과를 바로 반환|
|갱신|항상 최신 데이터 반영|원본 변경 시 갱신 필요 (비용 발생)|
|성능|복잡한 쿼리는 느림|읽기 빠름, 쓰기 부담 증가|


## 부호화(인코딩)와 복호화(디코딩)

데이터는 두 가지 형태로 존재한다:

1. **인메모리 표현**: 객체, 구조체, 리스트, 해시맵 등 → CPU가 효율적으로 접근
2. **바이트 열**: 파일에 쓰거나 네트워크로 전송할 때 → 자기 완결적(self-contained) 바이트 시퀀스

이 둘을 변환하는 과정:

- **부호화(인코딩)** = 직렬화(serialization) = 마샬링: 인메모리 → 바이트 열
- **복호화(디코딩)** = 역직렬화 = 언마샬링: 바이트 열 → 인메모리

### 스키마 발전과 호환성

시스템은 한 번에 전부 업그레이드되지 않는다. 신/구 버전이 공존하므로 호환성이 중요하다.

- **하위 호환성(backward compatibility)**: 새 코드가 예전 데이터를 읽을 수 있음
- **상위 호환성(forward compatibility)**: 예전 코드가 새 데이터를 읽을 수 있음

## 데이터플로 모드

데이터가 한 프로세스에서 다른 프로세스로 흐르는 방식은 크게 세 가지다.

### 1. 데이터베이스를 통한 데이터플로

```
[프로세스 A] → 부호화 → DB 저장 → 복호화 → [프로세스 B]
                         ↑
              "미래의 자신에게 보내는 메시지"
```

- DB에 쓰는 행위 = 부호화, DB에서 읽는 행위 = 복호화
- 쓴 프로세스와 읽는 프로세스가 다를 수 있음 (또는 미래의 같은 프로세스)
- **핵심 문제**: 새 버전 코드가 쓴 데이터를 구 버전 코드가 읽을 수 있어야 함 (상위 호환성)
#### 주의: 데이터가 코드보다 오래 산다

- 코드는 배포 시 교체되지만, 5년 전에 쓴 DB 데이터는 그대로 남아있음
- 스키마 마이그레이션이 가능하지만 비용이 큼 → 대부분 null 허용 칼럼 추가 방식으로 발전
- 전체 데이터를 새 스키마로 재작성하는 일은 드묾

#### 아카이빙(덤프)

- DB 스냅샷을 주기적으로 백업/내보내기
- 이 시점에 최신 스키마로 일관되게 부호화할 수 있는 기회

### 2. 서비스를 통한 데이터플로 (REST & RPC)

```
[클라이언트] ←→ [서버/서비스]
                    ↕
              [다른 서비스]
```

- 서버가 API를 공개하면, 클라이언트가 요청을 보내고 응답을 받음
- 서버가 다른 서비스의 클라이언트가 될 수도 있음 → **마이크로서비스 아키텍처** (서비스 지향 아키텍처, SOA)


#### 마이크로서비스의 핵심 목표

- 각 서비스를 **독립적으로 배포**할 수 있게 함
- 팀별로 서비스를 소유하고, 내부 구현을 자유롭게 변경
- 이를 위해 서비스 간 API 호환성이 필수적 (신/구 버전 공존)

#### API 버전 관리

- REST: URL 경로(`/v1/orders`), 헤더, 쿼리 파라미터 등으로 버전 구분
- RPC: 스키마 기반 호환성 (Protocol Buffers의 필드 번호 등)


### 3. 메시지 전달을 통한 데이터플로

```
[송신자] → 메시지 브로커(큐) → [수신자]
```

- 직접 호출(RPC)과 DB 사이의 **중간 형태**
- 대표적 구현: RabbitMQ, Apache Kafka, Amazon SQS

#### RPC와의 차이점

|관점|RPC / REST|메시지 전달|
|---|---|---|
|통신 방식|동기 (요청 → 응답 대기)|**비동기** (보내고 잊음)|
|수신자 상태|즉시 가용해야 함|일시적으로 불가용해도 괜찮음|
|수신자 수|보통 하나|여러 수신자에게 전달 가능|
|결합도|송신자가 수신자 주소를 알아야 함|송신자가 수신자를 몰라도 됨|
|응답|즉시 응답 기대|보통 단방향 (응답은 별도 채널)|

#### 메시지 브로커의 역할

- **버퍼 역할**: 수신자가 느리거나 불가용할 때 메시지를 보관
- **재전달**: 수신자가 처리에 실패하면 재전달 가능
- **송수신자 분리**: 송신자는 토픽/큐에 발행하고, 누가 소비하는지 모름
- **팬아웃**: 하나의 메시지를 여러 수신자에게 전달

## REST vs RPC

### REST (Representational State Transfer)

- HTTP를 기반으로 한 설계 철학
- 리소스(resource)를 URL로 식별하고, HTTP 메서드로 조작
- **장점**: 광범위한 도구 지원 (브라우저, curl, Postman 등), 디버깅 용이, 캐싱 활용 가능
- OpenAPI(Swagger)로 문서화와 코드 생성 가능

### RPC (Remote Procedure Call)

- 원격 서비스 호출을 **로컬 함수 호출처럼** 보이게 하려는 접근
- 클라이언트가 마치 같은 프로세스 안의 함수를 부르듯 호출
- 원격 서버에 있는 함수를 마치 내 코드 안의 함수처럼 호출하는 것

```
// RPC 스타일 (로컬 함수처럼 보임)
OrderService.createOrder(customerId, items);

// 실제로는 네트워크를 넘어 원격 서버에서 실행됨
```

- 대표적 구현: gRPC (Protocol Buffers), Apache Thrift, Apache Avro RPC
- → RPC가 로컬 호출인 척 하는 것은 **위치 투명성의 허상**. 네트워크의 특성을 숨기면 오히려 혼란을 초래한다.
### RPC의 근본적 문제점

로컬 함수 호출과 네트워크 요청은 본질적으로 다르다:

|구분|로컬 함수 호출|네트워크 요청 (RPC)|
|---|---|---|
|실패 모드|예측 가능|타임아웃, 네트워크 단절, 원격 서버 장애|
|결과|반환 or 예외|응답 없음(요청이 간 건지도 모름)|
|재시도|불필요|필요하지만 멱등성 보장 필요|
|지연 시간|매우 짧고 예측 가능|가변적, 수 ms ~ 수 s|
|참조 전달|포인터/참조 가능|모든 데이터를 바이트로 부호화해야 함|
