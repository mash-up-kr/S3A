# DDIA 9장 전반: 일관성과 합의 (Consistency and Consensus)



## 1. 일관성 모델 (Consistency Models)

### 1-1. 왜 일관성 모델이 필요한가

단일 DB에서는 쓰고 읽으면 방금 쓴 값이 나온다. 하지만 분산 시스템에서는 **복제(replication)** 가 있다. 데이터가 여러 노드에 복사되는데, 복제에는 시간이 걸린다. 그 사이에 읽으면 무엇이 나와야 하는가? 이 질문에 대한 답의 스펙트럼이 일관성 모델이다.

### 1-2. 선형성 (Linearizability) — 가장 강한 일관성

**한 줄 정의**: 시스템이 복제본이 하나뿐인 것처럼 행동하고, 모든 연산이 원자적(atomic)으로 보이는 것.

"내가 쓰기를 했으면, 그 직후부터 **누구든 어디서든** 새 값을 읽어야 한다"는 보장이다.

```
시간 →

클라이언트 A:  ──write(x=1)──────────────────
클라이언트 B:           ──read(x)──→ 반드시 1
클라이언트 C:                    ──read(x)──→ 반드시 1
```

더 정확히 말하면, 쓰기 연산이 시작된 시점과 완료된 시점 사이 어딘가에 **원자적으로 효력이 발생하는 시점**이 있고, 그 이후의 모든 읽기는 새 값을 반환해야 한다.

**선형성의 핵심 규칙**: 한 번이라도 새 값을 읽은 클라이언트가 있으면, 그 이후 모든 클라이언트의 읽기는 새 값을 반환해야 한다. 시간이 되돌아가는 것처럼 보이면 안 된다.

**선형성이 필요한 대표적 상황:**

- **잠금과 리더 선출**: 두 노드가 동시에 "내가 리더"라고 보면 안 된다. 잠금의 상태는 모든 노드에서 동일하게 보여야 한다.
- **유일성 제약**: 같은 사용자명으로 두 명이 가입하면 안 되는 경우. "이 이름이 사용 중인지"를 모든 노드가 동일하게 봐야 한다.
- **은행 잔고**: 잔고가 0원인데 두 ATM에서 동시에 출금하면 안 된다.

**선형성의 비용**: 성능과 가용성을 희생해야 한다. 모든 노드가 합의해야 하므로 느리고, 네트워크 파티션 시 응답을 못 할 수도 있다.

### 1-3. 인과적 일관성 (Causal Consistency) — 현실적인 타협점

선형성은 비용이 크다. "꼭 필요한 순서만 보장하자"는 것이 인과적 일관성이다.

**핵심 아이디어**: 인과 관계가 있는 이벤트는 순서를 보장하지만, 인과 관계가 없는(concurrent) 이벤트는 순서를 강제하지 않는다.

```
예시:
A가 질문: "오늘 저녁 뭐 먹지?"    ← 인과 관계 있음 (순서 보장)
B가 답변: "치킨 어때?"

C가 전혀 다른 글: "오늘 날씨 좋다"  ← A, B와 인과 관계 없음 (순서 자유)
```

모든 노드에서 "질문 → 답변" 순서로 보여야 하지만, C의 글은 어떤 순서로 보여도 상관없다.

### 1-4. 선형성 vs 인과적 일관성 비교

| 구분 | 선형성 (Linearizability) | 인과적 일관성 (Causal Consistency) |
|---|---|---|
| 순서 | 전체 순서 (Total Order) | 부분 순서 (Partial Order) |
| 보장 수준 | 모든 이벤트에 대해 단일 순서 | 인과 관계 있는 이벤트만 순서 보장 |
| 성능 | 느림 (모든 노드 합의 필요) | 빠름 (네트워크 지연에 영향 적음) |
| 가용성 | 파티션 시 가용성 포기 | 파티션 시에도 동작 가능 |
| 관계 | 선형성 ⊃ 인과성 | 인과성 ⊄ 선형성 |

**핵심 통찰**: 선형성을 보장하면 인과적 일관성은 자동으로 보장된다. 하지만 많은 경우 실제로 필요한 것은 선형성이 아니라 인과적 일관성이다. 인과적 일관성은 **가장 강한 일관성 모델 중 성능을 희생하지 않는 것**으로 평가받는다.

## 2. CAP 정리 (CAP Theorem)

### 2-1. 세 가지 속성

분산 시스템에서 다음 세 가지를 **동시에 모두 만족시킬 수 없다**는 정리:

- **C (Consistency)**: 선형성(Linearizability). 모든 노드가 동일한 최신 데이터를 본다.
- **A (Availability)**: 모든 요청이 에러가 아닌 의미 있는 응답을 받는다.
- **P (Partition Tolerance)**: 네트워크 파티션이 발생해도 시스템이 동작한다.

### 2-2. 파티션 시 선택: CP vs AP

네트워크 파티션은 선택이 아니라 현실이다 (8장). **P는 사실상 필수**이므로, 파티션이 발생했을 때 C와 A 중 선택해야 한다.

```
[네트워크 파티션 발생]

파티션 1: 노드 A, B        파티션 2: 노드 C, D, E
       ← 통신 불가 →

유저가 노드 A에 x=1을 씀.
```

- **CP (일관성 선택)**: 소수 파티션의 노드는 응답을 거부한다. 일관성은 지키지만 가용성 포기.
- **AP (가용성 선택)**: 모든 노드가 응답은 하지만, 데이터가 일치하지 않을 수 있다. 가용성은 지키지만 일관성 포기.

### 2-3. CAP 정리의 한계

CAP는 유명하지만 실무에서 직접 적용하기엔 한계가 있다:

- **이분법이 너무 단순**: 현실은 "일관성 vs 가용성"의 연속적 스펙트럼이다.
- **파티션이 없을 때**: 파티션이 없으면 C와 A 둘 다 가질 수 있다. CAP는 파티션 발생 시에만 트레이드오프를 강제한다.
- **지연(latency)을 다루지 않음**: 실무에서는 일관성 vs 가용성보다 **일관성 vs 지연**이 더 중요한 트레이드오프인 경우가 많다.

> "CP냐 AP냐"로 시스템을 분류하는 것보다, **구체적으로 어떤 일관성 보장을 제공하는지**를 따지는 것이 더 유용하다.

## 3. 순서 보장 (Ordering Guarantees)

### 3-1. 순서가 왜 중요한가

순서는 인과성과 직결된다:

- "질문 → 답변" 순서가 보장되어야 대화가 성립
- "계좌 입금 → 출금" 순서가 보장되어야 잔고가 음수가 되지 않음
- "잠금 획득 → 잠금 해제" 순서가 보장되어야 동시 접근 방지

### 3-2. 전체 순서 vs 부분 순서

| 구분 | 전체 순서 (Total Order) | 부분 순서 (Partial Order) |
|---|---|---|
| 정의 | 모든 이벤트를 하나의 타임라인에 줄 세울 수 있다 | 인과 관계가 있는 이벤트만 순서가 정해진다 |
| 비교 가능성 | 어떤 두 이벤트든 "A가 먼저" 또는 "B가 먼저" 결정 가능 | 일부 이벤트는 비교 불가능(concurrent) |
| 비유 | 자연수: 3과 5 중 뭐가 큰지 항상 비교 가능 | 집합 포함 관계: {a,b}와 {b,c}는 비교 불가능 |
| 대응하는 일관성 | 선형성 | 인과적 일관성 |

## 4. Lamport 타임스탬프 (Lamport Timestamp)

### 4-1. 동작 원리

물리적 시계를 믿을 수 없으므로, 논리적 카운터로 순서를 정하는 방법이다. 각 노드가 `(카운터, 노드ID)` 쌍을 유지한다.

**규칙:**

1. 각 노드는 자기 카운터를 유지하고, 연산할 때마다 카운터를 1 증가
2. 요청을 보낼 때 자기 카운터 값을 포함
3. 요청을 받으면 `max(내 카운터, 받은 카운터) + 1`로 업데이트
4. 카운터가 같으면 **노드 ID**로 타이브레이크

```
노드 A (ID=1)          노드 B (ID=2)
카운터=0                카운터=0
  │                      │
  ├─ op1: (1, A)         │
  │                      ├─ op2: (1, B)
  │                      │
  ├───── 메시지 전송 (카운터=1) ─────→
  │                      │
  │                      ├─ 받음: max(1,1)+1=2
  │                      ├─ op3: (2, B)
```

**보장**: A→B 인과관계가 있으면 Lamport(A) < Lamport(B)가 반드시 성립한다.

### 4-2. Lamport 타임스탬프의 한계

**전체 순서를 만들어주지만, 실시간으로 판단할 수 없다.**

```
노드 A: "celina" 가입 요청 → Lamport(3, A)
노드 B: "celina" 가입 요청 → Lamport(3, B)
```

나중에 모든 요청을 모아보면 (3, A) < (3, B)이므로 A가 먼저라고 결정할 수 있다. 하지만 **각 노드가 요청을 받은 그 순간에는** 다른 노드에서 같은 사용자명 요청이 동시에 처리되고 있는지 알 수 없다.

유일성 같은 제약을 실시간으로 판단하려면 "모든 노드의 상태를 지금 이 순간 아는 것"이 필요한데, Lamport 타임스탬프는 **사후적(after the fact)** 으로만 순서를 결정할 수 있다.

> Lamport 타임스탬프만으로는 부족하다 → **전체 순서 브로드캐스트**가 필요하다.

## 5. 전체 순서 브로드캐스트 (Total Order Broadcast)

### 5-1. 정의

**모든 노드에게 같은 메시지를 같은 순서로 전달**하는 프로토콜이다. 두 가지 속성을 보장해야 한다:

- **신뢰성 있는 전달 (Reliable delivery)**: 메시지가 한 노드에 전달되면, 모든 노드에 전달된다. 누락 없음.
- **전체 순서 전달 (Totally ordered delivery)**: 모든 노드가 동일한 순서로 메시지를 받는다.

### 5-2. 왜 필요한가 — Lamport의 한계 극복

Lamport 타임스탬프는 사후적으로만 순서를 매길 수 있었지만, 전체 순서 브로드캐스트는 **실시간으로** 모든 노드가 같은 순서를 보게 해준다.

```
[사용자명 유일성 보장 - 전체 순서 브로드캐스트 사용]

1. "celina 등록" 요청이 들어오면 → 전체 순서 브로드캐스트로 전송
2. 모든 노드가 같은 순서로 메시지를 받음
3. 각 노드: "이 사용자명이 처음 등장하는 메시지인가?" 확인
4. 처음이면 등록 성공, 아니면 거부
5. 모든 노드가 같은 순서로 받았으므로 → 모든 노드가 같은 결론에 도달
```

### 5-3. 전체 순서 브로드캐스트와 선형성의 관계

둘은 **동치(equivalent)** 관계다. 하나를 구현하면 다른 하나도 구현할 수 있다.

**전체 순서 브로드캐스트 → 선형성 구현:**
- 쓰기: 전체 순서 브로드캐스트로 메시지를 보내면 모든 노드가 같은 순서로 적용
- 읽기: 읽기도 전체 순서 브로드캐스트에 넣거나, 마지막으로 받은 메시지 기준으로 읽기

**선형성 → 전체 순서 브로드캐스트 구현:**
- 선형적인 카운터가 있으면 (compare-and-set 등), 각 메시지에 순번을 매겨서 전체 순서를 만들 수 있음

### 5-4. 전체 순서 브로드캐스트의 활용

- **데이터베이스 복제**: 모든 복제본이 같은 순서로 쓰기를 적용 → 상태 머신 복제(State Machine Replication). 모든 복제본이 동일한 상태를 유지한다.
- **직렬화 가능 트랜잭션**: 트랜잭션을 하나의 순서로 나열하면 직렬 실행과 동일한 효과
- **로그 기반 메시지 브로커**: Kafka의 파티션 내 메시지 순서 보장이 바로 이것
- **펜싱 토큰 발급**: 잠금 서비스에서 단조 증가 토큰을 발급하는 것도 전체 순서의 일종

## 9장 전반 핵심 흐름 정리

```
"이벤트 순서를 어떻게 보장할 것인가?"
        │
        ├── 강한 보장: 선형성 (전체 순서)
        │     └── 비용 높음: 성능↓, 가용성↓
        │
        ├── 현실적 타협: 인과적 일관성 (부분 순서)
        │     └── 성능은 좋지만, 유일성 같은 것은 보장 불가
        │
        ├── CAP 정리: 파티션 시 일관성 vs 가용성 택1
        │     └── 이분법이 단순하므로 구체적 일관성 보장을 따지는 것이 유용
        │
        ├── Lamport 타임스탬프로 순서 매기기
        │     └── 한계: 사후적으로만 가능, 실시간 판단 불가
        │
        └── 전체 순서 브로드캐스트로 해결
              ├── 모든 노드가 같은 순서로 메시지를 받음
              ├── 선형성과 동치
              └── 합의(consensus)와 연결 → 9장 후반으로
```

> **9장 전반의 핵심 메시지**: 분산 시스템에서 순서를 보장하는 것은 곧 합의(consensus) 문제다. 전체 순서 브로드캐스트는 합의와 동치이며, 이것이 9장 후반의 합의 알고리즘(Raft, Paxos 등)으로 연결된다.
