# DDIA 9장 후반: 합의와 코디네이션 (Consensus and Coordination)

## 이 파트의 큰 그림

9장 전반에서 "이벤트 순서를 보장하려면 전체 순서 브로드캐스트가 필요하고, 이것은 합의(consensus)와 동치"라는 결론에 도달했다. 9장 후반은 바로 그 **합의를 어떻게 달성하는가**, 그리고 **합의로 무엇을 할 수 있는가**를 다룬다.

---

## 1. 합의 (Consensus)

### 1-1. 합의란 무엇인가

**여러 노드가 하나의 값에 동의하는 것**이다.

단순해 보이지만 분산 환경에서는 매우 어렵다. 노드가 죽을 수 있고, 네트워크가 끊길 수 있고, 메시지가 지연될 수 있기 때문이다.

**합의 알고리즘이 만족해야 하는 속성:**

- **일치(Uniform Agreement)**: 모든 노드가 같은 값을 결정한다. 두 노드가 서로 다른 값을 결정하는 일이 없어야 한다.
- **무결성(Integrity)**: 어떤 노드도 두 번 결정하지 않는다. 한 번 결정하면 번복 없음.
- **유효성(Validity)**: 결정된 값은 어떤 노드가 실제로 제안한 값이어야 한다. 아무도 제안하지 않은 값이 결정되면 안 된다.
- **종료(Termination)**: 죽지 않은 모든 노드는 언젠가 반드시 값을 결정한다. 영원히 대기하는 일이 없어야 한다.

처음 세 속성(일치, 무결성, 유효성)은 **안전성(Safety)** — "나쁜 일이 절대 일어나지 않는다."
종료 속성은 **활성(Liveness)** — "좋은 일이 언젠가는 일어난다."

### 1-2. 합의가 어려운 이유

**FLP 불가능성 정리(FLP impossibility)**: 비동기식 모델에서 노드가 단 하나라도 크래시할 수 있으면, 합의를 보장하는 알고리즘은 존재하지 않는다.

하지만 이것은 **비동기식 모델** 한정이다. 현실의 분산 시스템은 부분 동기식 모델이고, 타임아웃과 장애 감지기를 사용할 수 있으므로 실용적인 합의 알고리즘이 가능하다.

### 1-3. 합의 알고리즘의 핵심 아이디어 — 리더 기반 합의

대부분의 실용적 합의 알고리즘(Raft, Paxos, Zab, Viewstamped Replication)은 **리더 기반**으로 동작한다.

**기본 흐름:**

```
[에포크(epoch/term) 기반 리더 선출]

1. 리더 선출: 에포크 번호(단조 증가)를 부여하며 리더를 선출
2. 제안: 리더가 값을 제안
3. 투표: 팔로워들이 제안에 투표 (정족수 이상 찬성 필요)
4. 결정: 정족수 이상 찬성하면 값이 결정됨
```

**에포크(epoch)의 역할:**

에포크는 Raft에서는 term, Paxos에서는 ballot number, Zab에서는 epoch라 부른다.

- 매 리더 교체 시 에포크 번호가 증가한다
- 두 리더가 충돌하면, **에포크 번호가 큰 쪽이 이긴다**
- 이전 에포크의 리더가 보낸 메시지는 무시된다
- 8장에서 배운 **펜싱 토큰**과 같은 원리다

**왜 리더 기반인가?**

리더가 없으면 모든 노드가 동시에 제안하고 투표해야 하는데, 이러면 충돌이 너무 많이 발생한다. 리더가 제안을 독점하면 충돌을 줄일 수 있다. 물론 리더가 죽으면 새 리더를 선출해야 하는데, 이 리더 선출 자체가 합의 문제이기도 하다.

### 1-4. 합의의 두 가지 핵심 활용: 리더 선출과 원자적 커밋

**① 리더 선출 (Leader Election)**

- 분산 시스템에서 "누가 리더인가"를 모든 노드가 합의해야 한다
- 두 노드가 동시에 자신이 리더라고 생각하면 → **Split Brain** → 데이터 불일치
- 합의 알고리즘으로 리더를 선출하면, 한 시점에 정확히 한 명의 리더만 존재하도록 보장할 수 있다

**② 원자적 커밋 (Atomic Commit)**

- 여러 노드에 걸친 트랜잭션에서, 모든 노드가 커밋하거나 모든 노드가 어보트해야 한다
- 일부만 커밋되면 데이터 불일치 발생
- "커밋할 것인가, 어보트할 것인가"를 합의해야 한다

## 2. 2단계 커밋 (2PC, Two-Phase Commit)

### 2-1. 왜 필요한가

단일 노드에서는 트랜잭션의 커밋/어보트가 간단하다. 하지만 **여러 노드에 걸친 분산 트랜잭션**에서는 문제가 된다:

```
문제 상황:
- 노드 A: 재고 차감 트랜잭션
- 노드 B: 결제 트랜잭션

노드 A는 커밋했는데 노드 B가 어보트하면?
→ 재고는 줄었는데 결제는 안 됨 → 데이터 불일치
```

모든 노드가 **원자적으로** 커밋하거나 어보트해야 한다. 이것을 보장하는 프로토콜이 2PC다.

### 2-2. 동작 원리

2PC에는 **코디네이터(coordinator)** 라는 특별한 역할이 존재한다.

```
[Phase 1: 준비 (Prepare)]

코디네이터 → 모든 참여자: "커밋할 수 있어?"
참여자 A → 코디네이터: "Yes" (준비 완료)
참여자 B → 코디네이터: "Yes" (준비 완료)

※ 참여자가 "Yes"를 보내면 → 이후 반드시 커밋할 수 있어야 한다는 약속
※ 참여자가 "No"를 보내면 → 즉시 어보트

[Phase 2: 커밋/어보트 (Commit/Abort)]

모든 참여자가 "Yes" → 코디네이터: "커밋!" → 모든 참여자가 커밋
하나라도 "No"      → 코디네이터: "어보트!" → 모든 참여자가 어보트
```

**핵심 지점 — 되돌릴 수 없는 순간:**

- 참여자가 "Yes" 투표를 보내면 → **반드시 코디네이터의 최종 결정을 따라야** 한다. 혼자 마음대로 어보트할 수 없다.
- 코디네이터가 "커밋" 결정을 디스크에 기록하면 → **되돌릴 수 없다**. 이 결정은 최종적이다.

### 2-3. 2PC의 치명적 문제 — 코디네이터 장애

```
1. 코디네이터: "커밋할 수 있어?" (Phase 1)
2. 모든 참여자: "Yes"
3. 코디네이터가 크래시 ← 여기서 문제!
4. 참여자들: "커밋해야 해? 어보트해야 해? 모르겠어..."
```

참여자들은 "Yes"를 보냈으므로 혼자 어보트할 수 없고, 코디네이터의 결정을 모르므로 커밋할 수도 없다. **무한 대기(blocking)** 에 빠진다.

이것이 2PC의 가장 큰 문제다. 2PC는 **블로킹 프로토콜(blocking protocol)** 이다. 코디네이터가 죽으면 참여자들이 멈춰버린다.

### 2-4. 2PC vs 합의 알고리즘 비교

| 구분 | 2PC | 합의 알고리즘 (Raft 등) |
|---|---|---|
| 코디네이터 | 단일 코디네이터 (단일 장애 지점) | 리더가 죽으면 새 리더 선출 |
| 블로킹 | 코디네이터 장애 시 블로킹 | 논블로킹 (과반수만 살아있으면 진행) |
| 투표 규칙 | **모든** 참여자가 Yes해야 커밋 | **과반수**가 동의하면 결정 |
| 장애 복구 | 코디네이터 복구까지 대기 | 자동 장애 복구 (리더 재선출) |

> **2PC는 합의 알고리즘이 아니다.** 2PC는 코디네이터 장애 시 진행이 불가능하므로 종료(Termination) 속성을 보장하지 못한다.

### 2-5. 3PC (Three-Phase Commit)

2PC의 블로킹 문제를 해결하기 위해 제안된 프로토콜이다. Phase 1과 Phase 2 사이에 "pre-commit" 단계를 추가한다. 하지만 **네트워크 지연에 상한이 있는 동기식 모델**을 가정하기 때문에 현실에서는 잘 쓰이지 않는다. 실무에서는 2PC 대신 합의 알고리즘(Raft 등)을 사용하는 추세다.

## 3. 합의로 해결 가능한 문제들

합의, 전체 순서 브로드캐스트, 선형성은 모두 **동치(equivalent)** 다. 하나를 풀면 나머지도 풀 수 있다. 이를 통해 해결할 수 있는 대표적 문제들을 살펴보자.

### 3-1. 선형성 레지스터 (Linearizable Register)

"읽기는 항상 가장 최근에 쓴 값을 반환한다"를 보장하는 저장소다.

- **compare-and-set (CAS)**: "현재 값이 X이면 Y로 바꿔라"를 원자적으로 실행
- 합의 알고리즘을 통해 구현 가능하다
- 전체 순서 브로드캐스트로 모든 쓰기를 전역 순서로 나열하면, 모든 노드가 같은 "최신 값"을 볼 수 있다

### 3-2. 잠금과 임차권 (Locks and Leases)

분산 잠금은 본질적으로 합의 문제다:

```
"이 잠금의 소유자는 누구인가?" → 모든 노드가 합의해야 한다

잠금 획득 = compare-and-set(lock, null, "내 ID")
잠금 해제 = compare-and-set(lock, "내 ID", null)
```

- 두 클라이언트가 동시에 잠금을 획득하면 안 된다
- 잠금의 상태에 대해 모든 노드가 동의해야 한다
- **임차권(lease)** 은 만료 시간이 있는 잠금이다. 8장에서 배운 리더 리스가 바로 이것.
- 펜싱 토큰과 결합하면, GC pause 이후의 안전성까지 보장 가능하다

### 3-3. 유일성 제약 조건 (Uniqueness Constraints)

"같은 사용자명으로 두 명이 가입할 수 없다"는 제약:

```
유일성 보장 = compare-and-set(username, null, "user_id")
→ 현재 값이 null(미사용)이면 등록, 아니면 거부
```

- 9장 전반에서 Lamport 타임스탬프로는 실시간 판단이 불가능하다고 배웠다
- 전체 순서 브로드캐스트를 사용하면 해결 가능하다: 모든 등록 요청을 하나의 순서로 나열하고, 먼저 도착한 요청만 승인
- 결국 합의가 필요하다

### 3-4. 이 문제들의 공통점

| 문제 | 본질 |
|---|---|
| 선형성 레지스터 | "최신 값이 무엇인가"에 합의 |
| 잠금/임차권 | "소유자가 누구인가"에 합의 |
| 유일성 제약 | "이 값이 사용 중인가"에 합의 |
| 리더 선출 | "리더가 누구인가"에 합의 |
| 원자적 커밋 | "커밋인가, 어보트인가"에 합의 |

모두 결국 **"여러 노드가 하나의 사실에 동의해야 한다"** 는 같은 문제다. 그래서 합의, 선형성, 전체 순서 브로드캐스트가 동치인 것이다.

## 4. 코디네이션 서비스 (ZooKeeper, etcd)

### 4-1. 왜 직접 합의를 구현하지 않는가

합의 알고리즘(Raft, Paxos 등)을 직접 구현하는 것은 매우 어렵고 위험하다. 에지 케이스가 많고, 잘못 구현하면 데이터 유실로 이어진다. 그래서 **이미 검증된 코디네이션 서비스**를 사용한다.

대표적으로 **ZooKeeper**와 **etcd**가 있다.

### 4-2. ZooKeeper가 제공하는 것

ZooKeeper는 합의 알고리즘(Zab)을 내부적으로 사용하여, 애플리케이션이 쓸 수 있는 다음과 같은 기능을 제공한다:

**① 선형성 읽기/쓰기**

- 소량의 데이터(설정 값, 메타데이터)를 선형적으로 저장
- 대용량 데이터 저장소가 아니라, **코디네이션 정보** 전용

**② 연산의 전체 순서 보장**

- 모든 쓰기 연산에 zxid(단조 증가)를 부여
- 전체 순서 브로드캐스트를 구현

**③ 장애 감지 (Failure Detection)**

- 클라이언트와 ZooKeeper 사이에 **세션**을 유지 (주기적 heartbeat)
- 클라이언트가 heartbeat를 보내지 않으면 → 세션 만료 → 해당 클라이언트의 임시 노드(ephemeral node) 삭제
- 다른 클라이언트는 이 삭제를 **감시(watch)** 하여 장애를 감지

**④ 변경 감지 (Watch/Notification)**

- 특정 키의 변경을 구독할 수 있다
- 값이 바뀌면 ZooKeeper가 클라이언트에게 알림
- 리더 선출에서 "현재 리더가 죽었나?"를 감지하는 데 사용

### 4-3. ZooKeeper로 구현하는 대표적 패턴들

**① 리더 선출**

```
1. 각 노드가 ZooKeeper에 임시 노드(ephemeral node)를 생성 시도
2. 첫 번째로 성공한 노드가 리더
3. 나머지 노드는 해당 임시 노드에 watch를 설정
4. 리더가 죽으면 → 세션 만료 → 임시 노드 삭제 → watch 알림 발생
5. 나머지 노드들이 다시 리더 선출 시도
```

**② 분산 잠금**

```
1. 잠금 획득: ZooKeeper에 임시 순차 노드(ephemeral sequential node) 생성
2. 가장 작은 번호의 노드가 잠금 획득
3. 잠금 소유자가 죽으면 → 임시 노드 삭제 → 다음 번호 노드가 잠금 획득
```

**③ 작업 할당 (Work Assignment)**

```
1. 작업자 노드가 ZooKeeper에 임시 노드로 자신을 등록
2. 파티셔너가 ZooKeeper의 작업자 목록을 보고 작업을 분배
3. 작업자가 죽으면 → 임시 노드 삭제 → 파티셔너가 감지하여 재분배
```

### 4-4. ZooKeeper가 아닌 것 — 주의할 점

ZooKeeper는 **범용 데이터베이스가 아니다**:

- 대용량 데이터를 저장하는 곳이 아니다 (설정, 메타데이터 전용)
- 모든 쓰기가 합의를 거치므로 **처리량이 제한적**이다
- 애플리케이션 데이터는 일반 DB에, **코디네이션 데이터만** ZooKeeper에 저장해야 한다

적합한 데이터 예시: "현재 리더는 누구인가", "이 파티션의 담당 노드는 어디인가", "서비스의 주소 목록(서비스 디스커버리)"

### 4-5. etcd와 ZooKeeper 비교

| 구분 | ZooKeeper | etcd |
|---|---|---|
| 합의 알고리즘 | Zab | Raft |
| API 스타일 | 트리 구조 (znodes) | 키-값 저장소 |
| 주요 사용처 | Hadoop, Kafka, HBase 등 | Kubernetes |
| 언어 | Java | Go |
| 공통점 | 선형성 보장, 장애 감지, 변경 감시, 리더 선출 지원 |

둘 다 본질적으로 같은 역할을 한다 — **합의 알고리즘 위에 구축된 코디네이션 서비스**.

---

## 5. 합의의 한계

합의가 만능은 아니다. 알아야 할 한계들:

### 5-1. 성능 비용

- 모든 결정에 **정족수 투표**가 필요하므로 네트워크 왕복이 추가된다
- 쓰기 처리량이 제한적이다. 합의를 거치는 만큼 단일 노드보다 느리다.

### 5-2. 노드 수 요구

- 과반수가 살아있어야 진행 가능하다. 3노드 시스템은 1노드 장애까지, 5노드는 2노드 장애까지 허용.
- 노드를 추가/제거하는 **멤버십 변경(membership change)** 이 복잡하다.

### 5-3. 네트워크 불안정 시 성능 저하

- 리더 선출에 타임아웃을 사용하므로, 네트워크가 불안정하면 **리더가 빈번하게 교체**된다
- 리더 교체 중에는 쓰기가 불가능 → 일시적 불가용
- 극단적인 경우, 네트워크 불안정으로 리더 선출이 반복되며 진행이 안 되는 상황(livelock)이 발생할 수 있다

## 9장 후반 핵심 흐름 정리

```
"합의를 어떻게 달성하는가?"
        │
        ├── 합의 알고리즘 (Raft, Paxos, Zab)
        │     ├── 리더 기반: 에포크 + 정족수 투표
        │     ├── 안전성: 항상 보장
        │     └── 활성: 과반수 생존 시 보장
        │
        ├── 2PC (비교 대상)
        │     ├── 코디네이터 단일 장애 지점
        │     ├── 블로킹 프로토콜 → 합의 알고리즘이 아니다
        │     └── 코디네이터 장애 시 무한 대기
        │
        ├── 합의로 해결 가능한 문제들
        │     ├── 선형성 레지스터 (CAS)
        │     ├── 잠금/임차권
        │     ├── 유일성 제약
        │     └── 리더 선출, 원자적 커밋
        │     → 모두 "여러 노드가 하나의 사실에 동의"하는 문제
        │
        └── 코디네이션 서비스 (ZooKeeper, etcd)
              ├── 합의 알고리즘을 내부에 캡슐화
              ├── 선형성 읽기/쓰기, 장애 감지, 변경 감시 제공
              ├── 리더 선출, 분산 잠금, 작업 할당 패턴
              └── 코디네이션 데이터 전용 (범용 DB가 아님)
```
