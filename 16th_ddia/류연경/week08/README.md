# DDIA 8장: 분산 시스템의 골칫거리 (The Trouble with Distributed Systems)

## 1. 신뢰할 수 없는 네트워크 (Unreliable Networks)

### 1-1. Shared-Nothing 아키텍처와 비동기 네트워크

대부분의 분산 시스템은 **shared-nothing** 아키텍처를 사용한다. 각 노드(서버)가 자기만의 CPU, 메모리, 디스크를 갖고 있고, **네트워크가 유일한 통신 수단**이다.

이 네트워크는 **비동기 패킷 네트워크(asynchronous packet network)** 로, 패킷이 언제 도착할지, 도착할지조차 보장이 없다.

### 1-2. 응답이 없을 때 — 원인을 알 수 없다

노드 A가 노드 B에게 요청을 보냈는데 응답이 없으면, 가능한 원인은 최소 5가지다:

1. **요청이 네트워크에서 유실** — 케이블 손상, 스위치 고장 등
2. **요청이 상대방의 큐에서 대기 중** — 노드 B가 과부하
3. **상대방 노드가 죽음** — 프로세스 크래시, 하드웨어 고장
4. **응답이 유실** — 돌아오는 경로에서 패킷 손실
5. **응답이 지연 중** — 네트워크 혼잡, 스위치 큐잉

보내는 쪽에서는 이 5가지를 구분할 방법이 없다. 유일한 수단은 일정 시간 기다려보는 **타임아웃(timeout)** 이다.

### 1-3. 네트워크 장애는 실제로 자주 발생한다

네트워크 파티션(network partition)은 이론이 아니라 **현실에서 일어나는 일**이다.

- 중간 규모 데이터센터에서 매달 약 12건의 네트워크 장애 발생 (2011년 연구)
- AWS, Azure 등 클라우드에서도 네트워크 파티션은 정기적으로 보고된다
- 소프트웨어가 반드시 이를 처리할 수 있도록 설계해야 한다

### 1-4. 타임아웃 딜레마

| 타임아웃 설정 | 장점 | 단점 |
|---|---|---|
| **길게 (예: 30초)** | 일시적 지연을 장애로 오판하지 않음 | 진짜 죽은 노드를 오래 기다려서 사용자 경험 저하 |
| **짧게 (예: 1초)** | 장애를 빨리 감지 | 일시적 지연을 장애로 오판 → 불필요한 장애 조치 |

짧은 타임아웃이 위험한 이유: 노드 B가 살아있는데 "죽었다"고 판단해서 노드 C에게 일을 넘기면, 작업이 중복 실행되고, C에 부하가 몰리면서 C도 느려지고, C까지 죽은 것으로 판단되는 **연쇄 장애(cascading failure)** 가 발생할 수 있다.

### 1-5. 네트워크 지연에 상한이 없는 이유

인터넷은 **패킷 교환(packet switching)** 방식으로 회선을 공유한다. 데이터센터 내에서도 여러 지점에서 큐잉(queueing)이 발생한다:

- 네트워크 스위치의 큐
- 상대방 OS의 수신 버퍼
- TCP 재전송 타이머
- VM 환경의 가상 네트워크 큐

이 모든 큐잉 때문에 지연에 **상한이 없다(unbounded delay)**. 평소에는 1ms인데 갑자기 수 초가 될 수도 있다. 완벽한 타임아웃 값은 존재하지 않으며, 실험적으로 결정하거나 **Phi Accrual Failure Detector** 같은 적응형 메커니즘을 써야 한다.

## 2. 신뢰할 수 없는 시계 (Unreliable Clocks)

### 2-1. 시계가 분산 시스템에서 중요한 이유

- **이벤트 순서 판단**: 요청 A가 먼저인가, B가 먼저인가?
- **타임아웃 측정**: 5초 지났으니 이 노드는 죽은 것으로 판단
- **캐시 만료**: 이 데이터는 오래됐으니 갱신 필요
- **로그 분석**: 장애가 어떤 순서로 발생했는지

각 노드가 자기만의 수정 시계(quartz clock)를 갖고 있어서 미세하게 다르다. 이를 **clock drift**라 한다.

### 2-2. 두 종류의 시계

**① Time-of-day clock (벽시계)**

- "지금 몇 시 몇 분"을 알려주는 시계
- `System.currentTimeMillis()`, `datetime.now()` 등
- 1970년 1월 1일(epoch) 기준 경과 밀리초
- NTP(Network Time Protocol)로 동기화
- **문제점**: NTP 동기화 시 시계가 뒤로 갈 수 있음 → 경과 시간 측정에 부적합

**② Monotonic clock (단조 시계)**

- "방금부터 지금까지 얼마나 지났는지"를 알려주는 스톱워치
- `System.nanoTime()`, `clock_gettime(CLOCK_MONOTONIC)` 등
- **절대 뒤로 가지 않음** — 항상 앞으로만
- **노드 간 비교는 의미 없음** — 각 노드의 기준점이 다르기 때문

> 정리: 경과 시간(duration)은 단조 시계, 특정 시점(point in time)은 벽시계를 사용해야 한다. 그런데 벽시계는 신뢰할 수 없다.

### 2-3. NTP 동기화의 한계

NTP로 시계를 맞추더라도 여러 가지 문제가 존재한다:

- **네트워크 지연**: NTP 서버의 응답이 오는 데 걸리는 시간이 가변적. 인터넷 경유 시 수십 ms 오차, 같은 데이터센터에서도 수 ms 오차
- **Quartz 시계 드리프트**: 온도에 따라 시계가 빨라지거나 느려짐. 동기화 사이에 drift가 누적
- **방화벽이 NTP를 차단**: 생각보다 흔한 설정 실수로, 시계가 점점 틀어짐
- **리프 초(Leap second)**: 1초가 추가/삭제될 때 처리 못하는 시스템 존재
- **VM 환경**: VM이 suspend/resume 되면 시계가 갑자기 점프

### 2-4. 시계 오차가 만드는 실제 문제: LWW의 함정

**Last Write Wins (LWW)** — 같은 데이터에 두 번 쓰면 타임스탬프가 큰 쪽이 이기는 전략. Cassandra, Riak 등에서 사용된다.

```
노드 A의 시계: 12:00:00.100  →  x = 1을 씀
노드 B의 시계: 12:00:00.050  →  x = 2를 씀 (실제로는 A보다 나중에 씀)
```

실제로는 B의 쓰기가 A보다 나중이지만, B의 시계가 느려서 타임스탬프가 더 작다. LWW 규칙에 따르면 A가 이겨서 `x = 1`이 남고, **B의 쓰기가 에러 메시지 없이 조용히 사라진다.**

### 2-5. 논리적 시계 (Logical Clock)

이벤트 순서를 정확히 판단하려면 물리적 시계 대신 **논리적 시계**를 사용해야 한다:

- **Lamport 타임스탬프**: 각 노드가 카운터를 유지. 메시지를 보낼 때 카운터를 포함하고, 받는 쪽은 `max(내 카운터, 받은 카운터) + 1`로 업데이트. 실제 시각은 모르지만 **인과 관계(causality)** 를 보존한다.
- **벡터 시계(Vector Clock)**: 더 세밀하게 "동시 발생(concurrent)"까지 구분 가능.

논리적 시계는 실제 시각이 아니라 이벤트의 **상대적 순서**를 추적하는 카운터 기반 방식이다.

## 3. 프로세스 중단 (Process Pauses)

### 3-1. 리더 리스(Lease) 시나리오

분산 시스템에서 리더를 정할 때 **리스(lease)** 를 사용한다. "10초 동안 리더 권한을 부여하고, 만료 전에 갱신해야 유지된다"는 방식이다.

**위험한 시나리오:**

```
1. 노드 A가 리스 획득 (만료: 12:00:10)
2. 노드 A가 현재 시간 확인: 12:00:05 → "아직 5초 남았으니 내가 리더"
3. → 이 순간, GC가 15초간 실행 (Stop-the-world GC)
4. GC 종료. 현재 시각 12:00:20이지만 A는 "5초 남았다"고 생각
5. 노드 A가 리더인 줄 알고 데이터를 씀
6. 하지만 이미 노드 B가 새 리더가 되어 쓰기를 받고 있었음
7. → 두 노드가 동시에 쓰기 → 데이터 충돌 (Split Brain)
```

### 3-2. 프로세스가 예고 없이 멈추는 이유들

- **GC (Garbage Collection)**: Java, Go 등에서 수 초~수십 초까지 전체 정지 가능 (특히 full GC)
- **VM suspend/resume**: 클라우드에서 라이브 마이그레이션 시 VM을 일시 중지했다가 다른 호스트에서 재개
- **OS swap (paging)**: 메모리가 부족하면 디스크로 스왑 → 프로세스가 수 초간 멈춤
- **SIGSTOP/SIGCONT**: Unix에서 프로세스를 일시 중지시키는 시그널
- **디스크 I/O**: 동기 I/O가 예상치 못하게 블로킹

핵심 교훈: 코드에서 `if (lease.isValid())`를 체크해도 소용없다. 체크한 **직후**에 GC가 시작될 수 있기 때문이다. 체크와 실행 사이의 어떤 가정도 안전하지 않다.

## 4. 지식, 진실, 그리고 거짓말 (Knowledge, Truth, and Lies)

### 4-1. 진실은 다수결로 정한다

분산 시스템에서 한 노드가 혼자서 판단할 수 있는 것은 거의 없다. 따라서 **정족수(quorum)**, 즉 다수결로 진실을 결정한다:

- "이 노드는 죽었다" → 과반수가 동의하면 죽은 것으로 간주
- "이 노드가 리더다" → 과반수가 인정해야 리더

노드 C가 "나 살아있어!"라고 외쳐도 과반수가 "죽었다"고 하면 죽은 것으로 간주된다. 네트워크 파티션으로 C가 격리된 것일 수 있기 때문에 이 판단이 합리적이다.

### 4-2. 펜싱 토큰 (Fencing Token) — 프로세스 중단 문제의 해결책

잠금(lock)을 사용할 때 프로세스 중단 문제를 해결하는 기법이다:

```
1. 노드 A가 잠금 획득 → 펜싱 토큰 #33 발급
2. 노드 A가 GC로 멈춤
3. 잠금 만료
4. 노드 B가 잠금 획득 → 펜싱 토큰 #34 발급
5. 노드 B가 토큰 #34와 함께 스토리지에 쓰기 → 성공
6. 노드 A가 GC에서 깨어나 토큰 #33과 함께 쓰기 시도
7. 스토리지: "이미 #34를 처리했는데 #33이 왔네? → 거부!"
```

스토리지가 **단조 증가하는 토큰 번호를 추적**해서, 오래된 토큰의 쓰기를 거부한다. ZooKeeper의 zxid나 Raft의 term이 이 역할을 한다.

### 4-3. 비잔틴 결함 (Byzantine Faults)

지금까지는 노드가 "정직하지만 고장날 수 있다"고 가정했다. 하지만 노드가 **악의적으로 거짓 메시지를 보내면?** 이것을 **비잔틴 결함**이라 한다.

이름의 유래는 **비잔틴 장군 문제**: 비잔틴 제국의 장군들이 전령으로 소통하는데, 일부 장군이 배신자라서 거짓 메시지를 보내는 상황에서 충성스러운 장군들이 합의에 도달할 수 있는가의 문제다.

**비잔틴 내결함성이 필요한 경우:**
- 블록체인 — 참여자를 신뢰할 수 없으므로
- 항공우주 시스템 — 우주 방사선으로 메모리 비트가 뒤집힐 수 있으므로

**일반 데이터센터에서는 불필요**하다. 서버를 직접 통제하기 때문이다. 대신 약한 보호 수단으로 충분하다:
- 네트워크 패킷 손상 → 체크섬
- 악의적 입력 → 입력 검증
- NTP 서버 오류 → 여러 NTP 소스 사용

## 5. 시스템 모델 (System Model)

분산 알고리즘의 정확성을 논하려면 **시스템 모델**, 즉 "어떤 가정 위에서 동작하는가"를 정의해야 한다.

### 5-1. 타이밍 모델

| 모델 | 설명 | 현실성 |
|---|---|---|
| **동기식 (Synchronous)** | 네트워크 지연, 프로세스 중단, 시계 오차에 상한이 존재 | 비현실적 |
| **부분 동기식 (Partially Synchronous)** | 대부분은 상한 안에 완료되지만, 때때로 상한을 넘음 | **가장 현실적** |
| **비동기식 (Asynchronous)** | 시간에 대해 아무 가정도 하지 않음. 시계조차 없음 | 알고리즘 설계가 매우 어려움 |

### 5-2. 노드 장애 모델

| 모델 | 설명 | 현실성 |
|---|---|---|
| **크래시-정지 (Crash-Stop)** | 한 번 죽으면 영원히 죽음 | 단순하지만 제한적 |
| **크래시-복구 (Crash-Recovery)** | 죽었다가 살아남. 디스크 데이터는 보존 | **가장 현실적** |
| **비잔틴 (Byzantine)** | 노드가 거짓말 포함 아무 행동이나 할 수 있음 | 특수 환경에서만 필요 |

> 대부분의 실용적 분산 시스템은 **부분 동기 + 크래시-복구** 모델을 기반으로 설계된다.

### 5-3. 안전성과 활성 (Safety and Liveness)

알고리즘의 정확성을 검증할 때 두 가지 속성을 구분한다:

- **안전성 (Safety)**: "나쁜 일이 절대 일어나지 않는다" — 예: 두 노드가 동시에 리더가 되지 않는다. 위반되면 되돌릴 수 없기 때문에 **항상 보장**해야 한다.
- **활성 (Liveness)**: "좋은 일이 언젠가는 일어난다" — 예: 요청을 보내면 언젠가는 응답을 받는다. **특정 조건(예: 과반수 노드 생존)에서만 보장**하면 된다.

## 정리: 8장의 세 가지 불확실성

| 문제 영역 | 핵심 | 대응 |
|---|---|---|
| **네트워크** | 패킷 유실, 지연, 순서 뒤바뀜. 상한 없는 지연. | 타임아웃, 재시도, 적응형 장애 감지 |
| **시계** | 노드 간 시계 오차(drift). NTP 동기화도 불완전. | 논리적 시계 사용, 물리적 타임스탬프에 의존하지 않기 |
| **프로세스** | GC, VM 일시 중지 등으로 예고 없이 멈춤. | 펜싱 토큰, 정족수 기반 합의 |

> **분산 시스템에서는 네트워크, 시계, 프로세스 모두 신뢰할 수 없다. 이 불확실성을 인정하고 설계하는 것이 출발점이다.**
