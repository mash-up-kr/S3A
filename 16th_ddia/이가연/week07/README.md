# 8장 분산 시스템의 골칫거리

## 결함과 부분장애

단일 컴퓨터에서 동작을 안하는건 대부분 코드 문제, 단일 컴퓨터는 항상 **결정적이다**.

분산 컴퓨터에선?

- 서버 간 네트워크 이슈
- 어떤 서버는 잘 동작하는데 어떤 서버는 고장 (부분 장애)
- 부분 장애는 **비결정적**이라 다루기 어렵고, 뭐가 성공이고 실팬지 알기도 어려움

### 클라우드 컴퓨팅과 슈퍼컴퓨팅

대규모 컴퓨팅 시스템 구축 방법은 고성능 컴퓨팅과 클라우드 컴퓨팅이 있다.

슈퍼컴퓨터는 장애로 계산이 중단되면 체크포인트부터 다시 계산하기 때문에 단일 컴퓨터에 가깝다.

다만 대부분의 사용자 대상 어플리케이션은 클라우드 컴퓨팅으로 제공한다.

## 신뢰성 없는 네트워크

클라우드 기반 분산 서버를 이용하는 데는 다음과 같은 이유가 있다.

- 특별한 하드웨어가 필요하지 않아서 저렴하다.
- 상품화된 클라우드 서비스는 편리하다.
- 지리적으로 분산된 여러 데이터센터에 배치함으로써 신뢰성이 높다.

이러한 분산 서버는 네트워크가 유일한 통신 수단이고, 이 네트워크는 도착을 보장하지 않는다.

- 요청 손실, 노드 장애, 지연 등..

이 문제는 대표적으로 타임아웃으로 제어한다.

#### 현실의 네트워크 결함

아직 신뢰성 있는 네트워크를 만드는 완전한 방법은 없으므로 네트워크 결함이 발생할 수 있다는 사실은 인지하고 소프트웨어를 설계해야 한다. 

그렇다고 반드시 네트워크 결함을 견뎌내도록 처리할 필요는 없다. 네트워크가 믿을만하다면 문제가 있을 때 그냥 사용자에게 오류 메세지를 보여줘도 좋다. 다만 소프트웨어만큼은 네트워크 문제에 어떻게 반응하는지 알고 스스로 복구할 수 있도록 보장해야 한다.

#### 결함 감지

시스템은 결함이 있는 노드를 자동으로 감지할 수 있어야 한다.

- 로드밸런서는 죽은 노드로 요청을 그만 보내야 한다.
- 분산 DB에서 리더가 죽으면 팔로워가 리더로 승격돼야 한다.

## 타임아웃과 기약 없는 지연

타임아웃은 결함을 감지하는 확실한 수단이지만 수치를 정하는 것은 애매하다.

타임아웃이 길면 노드가 죽었다고 판단하는 시간이 길어지고, 짧으면 일시적인 지연도 노드가 죽었다고 판단해버릴 수 있다.

노드가 죽지 않았는데 죽었다고 판단하면 해당 노드로 갈 부하가 다른 노드로 가기 때문에 불필요한 추가 부하로 이어진다. **지연과 결함 감지가 애매하다.**

#### 네트워크 혼잡과 큐 대기

네트워크 패킷 지연의 변동성은 큐 대기 때문인 경우가 많다.

- 라우터/스위치/로드밸런서 큐
- 네트워크 카드 송수신 큐
- 커널 소켓 버퍼 큐
- TCP 재전송/혼잡 제어 큐

지연 시간에 민감한 애플리케이션은 TCP 대신 UDP를 사용한다. UDP는 TCP와 달리 흐름 제어를 하지 않고 손실된 패킷을 재전송하지 않으므로 지연을 줄일 수 있다.

#### 동기 네트워크 VS 비동기 네트워크

동기식 네트워크는 **회선 교환망**이다. 여기선 일정 대역폭이 계속 보장되기 때문에 큐에 패킷이 쌓일 일이 없고, 지연 시간이 예측 가능하다.

반면 비동기식 네트워크는 **패킷 교환망**이다. 각 패킷이 네트워크 장비를 거치며 그때그때 전달되고, 같은 링크에 여러 사용자의 패킷이 몰리면 큐에 쌓이기 때문에 패킷마다 지연 시간이 달라지게 된다. 인터넷은 보통 패킷 교환망을 쓰기 때문에 지연 시간이 일정하지 않다.

#### **네트워크 지연을 예측 가능하게 만들 수는 없을까?**

다른 말로, 왜 데이터센터 네트워크와 인터넷은 패킷 교환망을 쓸까?

순간적으로 몰리는 트래픽에 최적화됐기 때문이다.

전화망 같은 회선 교환망은 통신 시작 전에 **경로 전체에 대역폭을 예약**한다. 이 예약된 대역폭은 다른 사용자가 쓰지 못한다. 순간적으로 몰리는 데이터 전송에 회선을 쓰면 네트워크 용량을 낭비하고 전송이 불필요하게 느려진다. 인터넷처럼 일정하게 흐르지 않는 트래픽(bursty traffic)은 미리 예약하게 되면 사용자가 트래픽을 보내지 않는 순간에도 대역폭이 낭비된다.

## 신뢰성 없는 시계

어플리케이션은 다음과 같은 질문에 대답하기 위해 다양한 방식으로 시계에 의존한다.

1. 이 요청이 타임아웃됐나?
2. 이 서비스의 99분위 응답 시간은 어떻게 되나?
3. 이 서비스는 지난 5분 동안 평균 초당 몇 개의 질의를 처리했나?
4. 사용자가 우리 사이트에서 시간을 얼마나 보냈나?
5. 이 기사가 언제 게시됐나?
6. 며칠 몇 시에 미리 알림 이메일을 보내야 하나?
7. 이 캐시 항목은 언제 만료되나?
8. 로그 파일에 남은 이 오류 메시지의 타임스탬프는 무엇인가?

분산 시스템은 통신이 즉각적이지 않으므로 시간을 다루기 까다롭다.

보통 NTP(Network Time Protocol) 메커니즘으로 서버 시간이 틀어지지 않도록 외부 시간 서버와 OS 시계를 조정(동기화)한다.

### 단조 시계 vs 일 기준 시계

#### 일 기준 시계(벽 시계)

현재 날짜/시각 표현 (“현재 2026-05-25 19:30인가?”, LocalDateTime.now() 등)

일 기준 시계는 **뒤로 갈 수 있다**. OS 시간이 보정될 수 있기 때문이다.

#### 단조 시계

어떤 기준점 이후 시간이 얼마 흘렀는지 재는 시계(타이머?, System.nanoTime())

타임아웃은 보낸 시각으로부터 재야 하기 때문에 단조 시계를 써야 한다. 받은 서버에서 일 기준 시계로 계산하면 틀어질 수 있음

#### 분산 시스템에서 시간을 믿어도 되는가?

분산 시스템에서는 여러 서버가 함께 동작하고, 각 서버마다 시계가 조금씩 다를 수 있다.

이는 LWW(Last Write Wins) 문제에서 분명하게 드러난다. 마지막으로 작성한 서버가 이기는 구조라면, 시계는 중요하다. 실제로는 B 서버의 쓰기가 나중에 발생했는데, B 서버 시계가 느려서 timestamp가 작게 찍힌다면 시스템은 B의 값을 버릴 수 있다. 

특히 카산드라, 리악 같은 **리더 없는 데이터베이스**나 **다중 리더 복제**에서는 이런 문제가 중요하다. 동시 쓰기 충돌 문제를 LWW로 해결하기 때문이다.

구글 Spanner와 TrueTime은 정확한 시간이 아니라 신뢰 구간을 준다. (현재 시간은 아마도 10:00:00.000 ~ 10:00:00.007 사이입니다.) 신뢰 구간이 겹치지 않는다면 순서를 확실히 알 수 있지만, 겹친다면 알 수 없다. 따라서 Spanner는 필요 시 잠깐 기다려서 구간이 절대 안 겹치게 만든다.

#### 전역 스냅숏용 동기화된 시계

가장 흔한 스냅샷 격리 구현은 단조 증가하는 트랜잭션 ID가 필요하다. 스냅샷보다 나중에 쓰기가 실행됐다면 그 내용은 스냅샷 트랜잭션에게 보이지 않으며, 단일 노드 DB에서는 문제 없다. 그러나 이가 분산됐을 시, 전역 단조 증가 트랜잭션 ID를 생성하기 어렵다. 트랜잭션이 작고 빠르면 전역적으로 하나의 순서를 만들기 어렵기 때문이다. (실제 시간 순서 보장 X)

전역 카운터를 하나 둔다 해도 트랜잭션 ID 발급 서버를 거쳐야 하는 네트워크 지연 문제와 강결합, 단일 장애점 문제가 생긴다.

동기화된 일 기준 시계 타임스탬프를 쓴다 해도 시계 정확도는 불확실하다. Spanner는 신뢰성 있는 시간 구간을 제공해주므로 의심의 여지 없이 사용할 수 있다.

#### 리더와 lease(임차권) 문제

분산 DB에서 서버 A가 리더일 때, 서버 A는 어떻게 자신이 아직 리더라는 사실을 알 수 있을까? 

이는 임차권이다.

```sql
서버 A는 30초 동안 리더 자격을 빌린다.
30초가 지나기 전에 갱신해야 한다.
갱신 못 하면 리더 자격을 잃는다.

lease 만료 시간이 가까워짐
→ lease 갱신
→ 계속 리더 유지

서버 A 장애
→ lease 갱신 중단
→ lease 만료
→ 서버 B가 새 리더가 됨
```

이 임차권 갱신 코드가 이런 식으로 구성돼있다 해보자.

```java
while (true) {
    request = getIncomingRequest();

    // lease가 10초 미만으로 남았으면 갱신
    if (lease.expiryTimeMillis - System.currentTimeMillis() < 10_000) {
        lease = lease.renew();
    }

    if (lease.isValid()) {
        process(request);
    }
}
```

이는 일기준 시계에 의존하기 때문에 NTP 보정에 의해 lease 만료 판단이 틀어질 수 있다.

또한 process가 즉시 실행된다면 문제 없지만, 지연이 생긴다면(GC pause 등) 다른 서버가 리더가 될 수도 있다. 이는 Fencing Toekn으로 해결할 수 있다 (뒤에서 설명)
