## 쓰기 스큐와 팬텀

트랜잭션이 각각 자신의 행을 업데이트했는데 비즈니스 규칙이 무너진 것.

업데이트는 행 단위인데 검사한 조건은 테이블 단위이기 때문

이렇게 어떤 트랜잭션에서 실행한 쓰기가 다른 트랜잭션의 검색 질의 결과를 바꾸는 것을 팬텀이라고 한다.

보통 테이블 전체를 대상으로 하는 질의에서 발생함 (COUNT, AVG 등)

- 쓰기 스큐 예시
    
    **예시: 의사 당직 시스템**
    
    규칙:
    
    > 병원에는 항상 최소 1명의 의사가 당직 중이어야 한다.
    > 
    
    현재 상태:
    
    | doctor | on_call |
    | --- | --- |
    | A | true |
    | B | true |
    
    ### 트랜잭션 1
    
    ```sql
    SELECT COUNT(*)
    FROM doctors
    WHERE on_call=true;
    -- 결과: 2명
    
    UPDATE doctors
    SET on_call=false
    WHERE doctor='A';
    ```
    
    ### 트랜잭션 2
    
    ```sql
    SELECT COUNT(*)
    FROM doctors
    WHERE on_call=true;
    -- 결과: 2명
    
    UPDATE doctors
    SET on_call=false
    WHERE doctor='B';
    ```
    
    각 트랜잭션 입장에서는 이렇게 판단해.
    
    > 현재 당직자가 2명이니까 나는 빠져도 최소 1명은 남겠지.
    > 
    
    그런데 둘 다 커밋되면 최종 상태는 이렇게 돼.
    
    | doctor | on_call |
    | --- | --- |
    | A | false |
    | B | false |
    
    즉, **당직자가 0명**이 됨.
    
    규칙이 깨졌지. → 두 트랜잭션이 **같은 row를 수정하지 않았기 때문**이야.
    
    트랜잭션 1은 A를 수정하고, 트랜잭션 2는 B를 수정해.
    
    그래서 DB 입장에서는 단순히 보면:
    
    ```
    T1: A row 수정
    T2: B row 수정
    ```
    
    이라서 write-write conflict가 아님.
    
    하지만 실제 비즈니스 규칙은 개별 row가 아니라:
    
    ```
    WHERE on_call=true
    ```
    
    조건 전체에 걸려 있어.
    
    즉, **읽은 조건 범위는 겹치는데, 수정한 row는 다르기 때문에 생기는 문제**야.
    

팬텀리드는 격리 수준으로는 SERIALIZABLE에서만 방지가 가능하다.

그렇다면 어떻게 방지할 수 있을까?

### 충돌 구체화

- 대상 row를 미리 만들어두고 lock을 건다 (특정 범위의 모든 조합에 대해 미리 row를 만들어서 락)
    - 동시성 제어 메커니즘이 애플리케이션 데이터모델로 새어 나오는 것은 보기 좋지 않음, 다른 대안이 불가능할 때 최후의 수단으로 고려

### 직렬성

격리수준으로 동시성을 제어하는 것은 데이터베이스마다 구현에 일관성이 없다. (애플리케이션 코드만으로 예측이 어렵다.)

→ 모든 DB는 직렬성 격리를 사용하면 막을 수 있긴 하다.

- 모든 트랜잭션을 순차적으로 실행하기 → 성능상 사실 불가능
- 트랜잭션을 스토어드 프로시저 안에 캡슐화하기
    - 하나의 기능을 구성하는 여러 로직을 하나의 트랜잭션으로 표현하고 원자적으로 커밋하기
    - ex. 항공권 예약에서의 경로선택, 요금선택, 가용 좌석 탐색 등..
- 파티셔닝
    - 직렬 실행은 원래 트랜잭션을 한 번에 하나씩 순서대로 실행해서 동시성 문제를 없애는 방식이다. 그런데 전체 DB를 하나의 스레드로 직렬 실행하면 너무 느리다. 그래서 데이터를 파티션으로 나누고, **각 파티션 안에서만 직렬 실행**하면 병렬성과 직렬성을 둘 다 어느 정도 얻을 수 있다.

→ 요약 : 모든 트랜잭션이 작고 빨라야 하며, 활성화된 데이터셋이 메모리에 적재될 수 있는 경우로 사용이 제한됨, 단일 스레드 트랜잭션에서 디스크에 접근한다면 시스템이 매우 느려짐. 쓰기 처리량도 단일 CPU 코어에서 처리할 수 있을 정도로 낮아야 하며 여러 파티션에 걸친 트랜잭션은 권장되지 않음

→ 쓰기 어렵다!

## 2단계 락(2PL 잠금)

트랜잭션 A가 객체를 읽고 B가 쓰기를 원한다면 A의 커밋 혹은 롤백을 기다려야 한다.

반대로 트랜잭션 A가 객체를 썼고 B가 읽기를 원한다면 B는 또 A의 커밋 혹은 롤백을 기다려야 한다.

스냅샷 격리와는 다르다. (스냅샷은 읽기를 원하면 최신 버전을 읽기 때문)

### 2단계 락구현/성능

배타락 혹은 공유락으로 구현됨. 데드락 발생 위험 높음

성능에 취약하고 동시성이 줄어든다.

## 서술 락

조건에 부합하는 모든 객체에 잠금을 획득하는 것

```sql
SELECT * FROM bookings
   WHERE room_id = 123 AND
      end_time > '2018-01-01 12:00' AND
      start_time < '2018-01-01 13:00'
```

- 일반 row 락은 조건에 해당하는 row가 없다면 락이 걸리지 않고, 팬텀 리드와 쓰기 스큐 문제가 발생함
- 서술 락은 조건에 해당하는 데이터가 **지금은 없더라도 앞으로 이 조건을 만족하는 row도 삽입을 막는 락**
- 조건에 부합하는 잠금을 확인하는 데 시간이 걸린다.
- 따라서 2PL 잠금은 대부분 인덱스, 넥스트 키 락으로 구현한다.

## 인덱스 범위 락

서술락처럼 조건 자체를 직접 잠그는 대신 **그 조건을 찾을 때 사용하는 인덱스의 특정 범위에 락**을 거는 방식

서술락은 비싸서 인덱스가 사용되는 조건절이라면 인덱스 범위 락이 더 효율적이다.

예를 들어 위 쿼리에서 room_id에 인덱스가 걸려있다면 인덱스 범위 락은 room_id = 123인 인덱스 범위 전체를 잠근다.

이는 서술락보다는 범위가 더 크게 잡히지만 구현 비용이 싸다는 장점이 있다.

대표적으로 MySQL의 **갭 락과 넥스트 키 락**이 인덱스 범위를 잠근다.

## 직렬성 스냅샷 격리, SSI(Serializable Snapshot Isolation)

스냅샷의 성능 장점은 유지하면서 결과는 직렬 실행처럼 보장하려는 격리 수준 구현 방식

2PL은 비관적 동시성 제어라면 SSI는 낙관적 동시성 제어 방식이다.

스냅샷의 단점 → 쓰기 스큐 문제

SSI는 이런 패턴을 감지. 핵심 아이디어는

**트랜잭션들이 서로의 변경을 못 보고 판단한 결과, 직렬 순서로 설명할 수 없는 상황이 생기면 그중 하나를 abort시킨다.**

낙관적 락이랑 비슷함. 읽기-쓰기 의존성을 추적한다. 경쟁이 심하면 abort 비율이 높아지므로 오히려 성능이 떨어짐.

PostgreSQL에서는 `SERIALIZABLE` 격리 수준이 내부적으로 SSI를 사용한다.

### MySQL이 팬텀 리드를 방지하는 방법

MySQL의 기본 격리수준 : `REPEATABLE_READ`

`REPEATABLE_READ` 에서는 항상 스냅샷을 읽는다. → 팬텀 리드 발생 가능

MySQL은 범위에 락을 거는 갭락/넥스트 키 락을 사용한다. (정확히는 인덱스가 걸려있으면 해당하는 인덱스 범위에, 안 걸려있으면 PK로)

- 갭 락 : 레코드와 레코드 사이의 빈 공간을 잠그는 락
- 넥스트 키 락: 범위와 범위에 해당하는 레코드를 잠그는 락

```sql
-- Transaction 1 (REPEATABLE READ 격리 수준)
START TRANSACTION;

SELECT * FROM Employees WHERE Age BETWEEN 20 AND 30 FOR UPDATE;
-- → 이 쿼리는 (20, 30) 범위에 Next key Lock을 건다

-- Transaction 2 (동시에 실행)
START TRANSACTION;

INSERT INTO Employees (Name, Gender, Age) VALUES ('DOHYUN', 'Male', 25);
-- Gap Lock 때문에 이 삽입은 대기(block) 또는 실패

-- Transaction 1이 COMMIT 해야 Transaction 2가 진행 가능
```

- 인덱스가 걸려있지 않으면 갭 락/넥스트 키 락은 걸리지 않는가?
- 갭락/넥스트키락/레코드락을 결정하는 기준은?
    - 조건 절의 컬럼이 유니크 인덱스면 범위가 아니라 레코드 단일로 결정 나므로 레코드 락
    - 범위에 해당하는 레코드와 범위를 모두 잠가야 하면 넥스트 키 락
    - 범위나 조건에 해당하는 row가 하나도 없을 때는 갭 락
