# 3장 후반

## B트리

노드 → 논리적 단위, 페이지 → 물리적 단위, 1:1 대응

키를 찾을 때는 루프 노드(페이지)부터 시작, 최종적으로는 리프 노드에 도달한다. 

- 값 갱신 : 키를 포함한 리프 노드를 찾아 값을 바꾼 뒤 페이지를 다시 디스크로 기록
- 키 추가 : 키가 들어갈 범위의 페이지를 찾아 추가

 → 페이지에 자리가 없다면? : 인덱스 스플릿

### 신뢰할 수 있는 B 트리 만들기

모든 비트리의 쓰기 동작은 새로운 데이터를 디스크 페이지에 덮어쓴다.

여러 페이지를 덮어쓰기 하다 중간에 고장난다면? 고아 페이지가 생길 수도 있다.

데이터베이스가 장애 상황에서 스스로 복구하려면 **리두 로그(WAL,** Write-Ahead Logging라고도 함) 가 필요하다. 리두 로그는 트랜잭션 내에서 변경된 모든 내용을 기록해놓는 파일이다.

동시성 제어는 **래치**라는 가벼운 잠금으로 보호한다.

### B 트리 최적화

- 일부 DB는 덮어쓰기와 WAL 대신 새로운 레코드로 버전 관리를 한다 (PostgreSQL 등)
- 비트리 분기 계수를 낮추기 위해 키를 축약해 쓸 수 있다.
- 비트리는 대부분 리프 노드가 디스크 상에 연속된 순서로 나타나게끔 트리를 배치
- 리프 노드는 연결 리스트로 노드 간 참조 관계를 가진다. 이를 통해 상위 페이지로 역행하지 않아도 키를 순서대로 스캔할 수 있다.

### B 트리와 LSM 트리 비교

- LSM 트리는 데이터가 여러 파일에 분산되어 있어 읽기에 불리하고, B 트리는 정렬된 트리를 타고 내려가는 방식으로 읽기에 유리하다.
- B 트리는 WAL와 트리 수정, 데이터 수정이 총 2번 일어난다. 쓰기 부하에 불리하다. (쓰기 증폭) 디스크에 순차적으로 flush 하는 SSTable이 쓰기 부하에 더 유리하다.
- B 트리의 파편화 문제를 LSM 트리는 갖지 않는다 (압축률)
- LSM 트리의 컴팩션 과정을 진행 중인 읽기와 쓰기의 성능에 영향을 줄 수 있다.
    - 컴팩션 과정은 디스크 쓰기 대역폭을 공유하기 때문에 설정을 주의 하지 않으면 쓰기 속도를 따라가지 못할 수도 있다.
- LSM 트리는 다른 세그먼트에 같은 키의 다중 복사본이 존재할 수 있어 트랜잭션 격리 등에서 불리할 수 있다.

### 힙 파일 VS 클러스터드 인덱스

**힙 파일**

특정 순서 없이 데이터 저장, 인덱스의 노드 트리가 **힙 파일 주소**를 가진다. 레코드 업데이트 시, 레코드 값을 덮어쓸 수 있다면 참조를 변경하지 않을 수 있다. 

```sql
name = "Kim" → "Lee"
```

다음과 같이 길이도 비슷하고 공간도 충분하면 데이터 덮어쓰기가 가능하므로 인덱스의 힙 파일 주소 참조는 변경되지 않는다.

```sql
name = "Kim" → "Christopher"
```

다만 이런 경우는 새로운 레코드를 생성해서 바꿔야 하므로 인덱스도 업데이트된다.

힙 파일 주소를 참조하는 인덱스는 Oracle, PostgreSQL이 대표적이다. 

- PostgreSQL 방식
    
    PostgreSQL의 경우는 덮어쓰기를 안하고 업데이트 시 항상 새 레코드를 생성하기 때문에 인덱스 참조가 바뀌지만, **HOT** (Heap Only Tuple) 라는 최적화를 통해, 인덱스 컬럼이 변경되지 않는 업데이트 작업이라면 힙 내부에서 레코드를 생성하고 기존 레코드와 체인 연결을 통해 인덱스 참조를 변경하지 않을 수 있다. 인덱스는 여전히 기존 레코드 주소를 가리키기 때문이다.
    
    그렇다면 HOT에서 체인으로 연결된 기존 레코드들은 계속 유지되나? → 아니다. VACUUM이라고 트랜잭션에 참여하지 않은 죽은 레코드들을 정리하는 프로세스가 레코드를 정리한다. 하지만 인덱스가 직접 참조하고 있는 레코드는(루트 레코드)는 정리하지 않는다! 인덱스 참조를 직접 수정하는 건 비용이 너무 비싸기 때문. VACUUM FULL을 하면 그때는 갱신된다.
    
    ```sql
    인덱스 → ROOT → (정리됨) → (정리됨) → 최신 튜플
    ```
    

**클러스터드 인덱스**

InnoDB의 경우는 테이블의 PK가 항상 클러스터드 인덱스이고, 세컨더리 인덱스의 리프 노드는 PK를 참조한다. 테이블 데이터는 PK를 기반으로 정렬된다.

### 인메모리 데이터베이스

데이터 손실을 허용하는 맴캐시드와 다르게 인메모리 데이터베이스는 **지속성**을 목표로 한다.

또한 성능상 이점 뿐만 아니라, 인메모리 데이터베이스는 디스크에서는 제공하기 어려운 자료구조 (우선순위 큐, set) 를 제공한다.

최근 연구에 따르면 인메모리 데이터베이스는 가용 메모리보다 더 큰 데이터셋을 지원하게끔 확장할 수 있다 (**안티 캐싱**) 이 방식은 메모리에서 가장 최근 사용하지 않은 데이터를 디스크로 내보내고 다시 접근할 때 메모리에 적재하는 방식으로 동작한다. (가상 메모리처럼)

## 트랜잭션 처리나 분석

온라인 트랜잭션 : 사용자 요청을 실시간으로 처리한느 짧고 빠른 트랜잭션

온라인 분석 처리 : 경영진이 더 나은 의사결정을 하기 위해 분석하는 쿼리

| 특성 | 트랜잭션 처리 시스템 (OLTP) | 분석 시스템 (OLAP) |
| --- | --- | --- |
| 주요 읽기 패턴 | 질의당 적은 record, 키 기준 fetch | 많은 record에 대한 집계 |
| 주요 쓰기 패턴 | 임의 접근, 사용자 입력을 낮은 지연 시간으로 처리 | 대규모 불러오기 (bulk import, ETL), 이벤트 스트림 |
| 주요 사용처 | 웹 앱, 사용자, 소비자 | 의사 결정을 위한 내부 분석가 |
| 데이터 표현 | 데이터의 최신 상태 (현재 시점, 실시간) | 시간이 지나며 발생된 이력 (과거) |
| 데이터셋 크기 | 기가바이트 ~ 테라바이트 | 테라바이트 ~ 페타바이트 |
| 최적화 방법 | 트랜잭션 효율화, 무결성 극대화 | 조회 속도, 정보의 가치, 편의성 |
| 데이터의 특성 | 트랜잭션 중심 | 정보 중심 |

데이터웨어하우스 : 운영 시스템(OLTP)에서 나온 데이터를 모아서 분석(OLAP)에 최적화된 형태로 저장하는 시스템, 대량 읽기 중심, 과거 데이터 중심, 정규화보다 분석 친화 구조

분석 접근 패턴에 맞게 최적화할 수 있다는 장점이 있다.

1. 서비스 DB (OLTP)
2. 데이터 수집 (ETL / ELT)
3. 데이터웨어하우스 저장
4. 분석 / BI 도구에서 조회

### 분석용 스키마

많은 데이터 웨어하우스는 별 모양 스키마(차원 모델링)를 사용한다.

특정 시각에 발생한 이벤트(제품 구매) 사실 테이블의 다른 컬럼은 차원 테이블(dimension table)이라 부르는 다른 테이블을 가리키는 외래 키 참조다

눈꽃송이 모양 스키마는 별 모양 스키마의 변형으로, 차원이 하위 차원으로 더 세분화된다.

### 컬럼 지향 저장소

일반적인 웨어하우스 질의는 쿼리 한 번에 4-5개 컬럼만 접근한다 (SELECT * 를 안 쓴다는 뜻)

```sql
SELECT
     dim_date.weekday,
     dim_product.category,
     SUM(fact_sales.quantity) AS quantity_sold
FROM fact_sales
     JOIN dim_date       ON fact_sales.date_key = dim_date.date_key
     JOIN dim_product  ON fact_sales.product_sk = dim_product.product_sk
WHERE
     dim_data.year = 2013
AND
     dim product.category IN ('fresh fruit', 'Candy')
GROUP BY
     dim_date.weekday, dim_product.category;
```

사람들이 요일에 따라 신선 과일을 사고싶어하는지 사탕을 더 사고싶어 하는지 분석하는 쿼리

OLTP 데이터베이스의 경우 **로우 지향 방식**으로 테이블에서 한 로우의 모든 값은 서로 인접하게 저장된다.

하지만 로우 지향 방식은 여전히 디스크로부터 모든 로우를 메모리로 적재한 다음 구문을 해석해 필터링하는 과정을 거쳐야 한다. → 시간이 오래 걸린다.

<img width="841" height="634" alt="image" src="https://github.com/user-attachments/assets/756e4ed9-4825-46b8-8c91-ef4537ff8db0" />


컬럼 지향 저장소는 모든 값을 하나의 로우에 함께 저장하지 않는 대신 각 컬럼 별로 모든 값을 함께 저장한다. 각 컬럼을 개별 파일에 저장하면 필요한 컬럼만 읽고 구문 분석하면 된다.

### 컬럼 압축

데이터를 압축하면 디스크 처리량을 줄일 수 있으며 컬럼 저장소는 대개 압축에 적합하다. 대표적으로 비트맵 부호화가 있다.

비트맵 부호화는 컬럼 값들을 비트(0/1)로 표현해서 빠르게 필터링·집계하는 기법이다.

```sql
WHERE gender = 'M' AND age = 30
```

비트 연산:

- gender=M → `1 0 1 0`
- age=30 → `0 1 1 0`

AND 연산 → `0 0 1 0`

→ 3번째 row만 해당

비트 연산이므로 CPU 친화적이고 압축된 비트만 읽으므로 I/O를 절약할 수 있다. **카디널리티가 낮을 때** 쓰기 좋다. 벡터화 처리에도 좋다.

벡터화 처리란 row를 n개씩 배치 처리하는 것이다. 같은 컬럼 데이터가 연속되므로 CPU cache hit가 증가하고, 함수 호출 오버헤드도 감소한다.

### 컬럼 저장소의 순서 정렬

로우가 저장되는 순서가 중요하진 않지만 각 컬럼이 독립적으로 정렬되는건 안된다. 한 컬럼의 k번째 항목이 다른 컬럼 k번째 항목과 같은 로우에 속해야 한다. 그래야 그룹화, 필터링도 용이하고 컬럼 압축에도 도움이된다.

### 다양한 순서 정렬

데이터 복원력을 위해 데이터를 여러 장비에 복제하고, 서로 다른 방식으로 정렬해 저장하면 질의 특성에 맞는 버전을 선택해 사용할 수 있다.

로우 지향 저장에서 세컨더리 인덱스를 갖는 것과 같은 맥락이다.

### 컬럼 지향 저장소에 쓰기

컬럼 지향 저장소, 압축, 정렬은 모두 읽기에 더 빠르고, 쓰기를 위한 해결책은 LSM 트리 구조가 적절하다.

### 집계: 데이터 큐브와 구체화 뷰

데이터 웨어하우스 질의는 집계 함수를 많이 쓰는데, 매번 원시 데이터를 처리하는 것은 낭비다. 따라서 미리 집계된 데이터를 구체화 뷰라고 한다. 데이터 큐브(OLAP 큐브)가 대표적이다.

# 4장 REST와 GRPC

REST와 SOAP 모두 웹 서비스 통신 프로토콜이지만, REST가 더 가벼운 느낌이다.

- **REST(Representational State Transfer)**: 웹의 구조를 그대로 활용하는 **가벼운 스타일**
    - 자원을 URL로 표현
    - HTTP 메서드로 행위 표현
    - HTTP를 그대로 활용하고, JSON으로 요청/응답을 주고 받으며, Stateless하다.
- **SOAP(Simple Object Access Protocol)**: 엄격한 규격을 가진 **프로토콜**
    - XML 사용
    - 무겁고, 느리고, 표준화가 엄격해서 난이도가 있다.

### RPC(Remote Procedure Call)

다른 서버에 있는 함수를 로컬 함수처럼 호출하는 방식(패러다임)

REST가 자원 위주라면 RPC는 함수 호출(행위) 위주다.

```sql
POST /getUser
POST /createUser
POST /deleteUser
```

MSA에서 GRPC가 각광 받는 이유 → 서비스 간 내부 통신 최적화

**성능**

- gRPC: **Protocol Buffers(바이너리)**
- REST: **JSON(텍스트)**

→ 따라서

- 데이터 크기 ↓
- 직렬화/역직렬화 비용 ↓
- 네트워크 전송 속도 ↑

특히 서비스 간 호출이 많을수록 차이 크게 난다.

또한 GRPC는 내부적으로 HTTP/2를 쓰는데, 이는 하나의 TCP 연결로 여러 요청을 병렬 처리할 수 있는 멀티플렉싱을 제공하므로 성능 상 이점이 있다.

**규약**

또 하나는 규약이다. REST는 런타임에서 오류가 발생하지만, GRPC는 강한 규약으로 컴파일 단계에서 오류가 잡힌다.

예를 들어, REST에선 A 서버가 B 서버를 호출할 때 B 서버 API의 스펙이 변경되어도 이를 모르므로, 런타임에서 호출할 때야 404 에러나 파싱 에러로 알 수 있다. → 저도 맨날 같은 사고 쳐요 마치 기억력 테스트 같음

하지만 GRPC는 `.proto`에 API를 정의해 클라이언트와 서버가 이를 공유해 코드를 자동 생성하므로 컴파일 단계에서 잡힌다. 이걸 stub이라 한다. proto 파일은 보통 repo를 파서 라이브러리로 따로 관리하고, 서버와 클라이언트가 이걸 import 해 사용한다.

stub 단계

1. `.proto` 작성
2. 컴파일러 실행 (`protoc`)
3. 언어별 코드 생성 (Java, Go, Python 등)
