## 트랜잭션 개념

### ACID 

- 원자성 : 모두 실행되거나 모두 실행 안되거나
- 일관성 : 트랜잭션 커밋 후 데이터베이스의 규칙이 깨지지 않아야 한다.
- 격리성 : 동시에 실행되는 트랜잭션은 서로 격리되어 방해할 수 없다.
- 지속성 : 트랜잭션이 커밋됐다면 DB에 장애가 생기더라도 데이턴느 손실되지 않는다.

### 단일 객체 연산과 다중 객체 연산

원자성과 격리성은 단일 객체를 변경하는 경우에도 적용된다.

예를 들어 한 객체를 쓰다가 네트워크 이슈로 끊겼을 때, 다른 트랜잭션이 해당 객체를 읽어선 안된다. 롤백돼야 한단 소리?

정합성이 유지돼야 하는 다중 객체에서도 원자성과 격리성은 보장되어야 한다.

# 완화된 격리 수준

SERIALIZABLE과 READ UNCOMMITTED도 있지만 거의 안 쓰이므로 READ COMMITTED와 REPEATABLE READ만 알아두면 된다.

SERIALIZABLE > REPEATABLE READ > READ COMMITTED > READ UNCOMMITTED

## READ COMMITTED

대부분의 RDBMS가 제공하는 기본 격리 수준

읽을 때도 커밋된 데이터만 보고, 쓸 때도 커밋된 데이터만 덮어쓰게 해서 더티 읽기/쓰기를 방지한다.

- 더티 리드: 다른 트랜잭션이 커밋하지 않은 데이터를 읽는 것
    - 언두로그를 읽는다
- 더티 쓰기 : 동시에 쓰기르 할 때, T1이 아직 커밋하지 않은 값을 T2가 덮어씀
    - 커밋이 될 때까지 락을 걸어놓는다

## 스냅샷 격리와 REPEATABLE READ

READ COMMITTED의 문제는 SELECT 를 날릴 때마다 최신 커밋을 읽어와서 반복적으로 SELECT를 날리면 다른 트랜잭션의 커밋 결과에 따라 다른 값이 나올 수 있다는 점이다. (새로고침 등)

REPEATABLE READ에서 트랜잭션이 시작되면 다른 트랜잭션의 커밋 여부와 무관하게 처음 읽은 스냅샷을 끝까지 읽는다. (트랜잭션 번호로 언두로그에서 관리)

이건 일반적인 SELECT의 경우인거고, 락을 거는 DML은 최신 커밋 스냅샷을 읽는다.

### **스냅샷 격리 구현**

MVCC : DB가 객체의 여러 버전을 함께 유지하는 방법

READ COMMITTED는 쿼리마다 독립된 스냅샷을 사용하지만 REPEATABLE READ는 트랜잭션 동안 동일한 스냅샷을 사용한다.

아까 트랜잭션 번호로 관리한다 했는데, 정확히는 **트랜잭션 ID가 더 큰(즉 현재 트랜잭션이 시작한 후에 시작한) 트랜잭션이 쓴 데이터는 그 트랜잭션의 커밋 여부에 관계 없이 모두 무시**된다.

## 인덱스와 스냅샷 격리

그러면 언두로그와 테이블에 데이터의 여러 버전이 공존할 때, 인덱스는 어떻게 될까? 

만약 name이라는 인덱스가 있고 name = A라는 버전(테이블)과 name = B라는 버전(언두로그)가 있을 때 where name = ‘B’라는 조건절이 인덱스를 탄다면?

MySQL 기준으로,

인덱스도 동시에 여러 버전을 관리하되, 예전 엔트리는 (언두 로그에 있는 데이터)는 deleted-marked 상태로 관리하고 사용이 될 때 **언두 로그를 통해 재구성** 된다. 이는 언두로그가 삭제될 때 같이 삭제된다.

```java
name이 B에서 A로 바뀌면, 보조 인덱스 idx_name은 기존 엔트리 하나를 그냥 B → A로 덮어쓰지 않는다.
old secondary index record는 delete-marked
new secondary index record는 insert
delete-marked record는 나중에 purge
```

```java
1. idx_name에서 'B' 엔트리를 찾는다.
2. 그 엔트리가 delete-marked 상태이거나, 보조 인덱스 페이지가 더 새로운 트랜잭션에 의해 갱신된 상태라면,
   보조 인덱스만 보고 결과를 확정하지 않는다.
3. 보조 인덱스 엔트리에 들어 있는 PK id=1로 클러스터드 인덱스 row를 찾는다.
4. 클러스터드 인덱스의 현재 row는 name='A'다.
5. row의 DB_TRX_ID와 T1의 Read View를 비교한다.
6. 현재 row 버전이 T1에게 보이면 안 되는 버전이면 undo log를 따라가서 과거 버전을 재구성한다.
7. 재구성한 과거 버전이 name='B'이고 T1의 스냅샷에서 보이는 버전이면 결과로 반환한다.
```

## Lost Update 방지

만약 두 트랜잭션이 작업을 동시에 하면 두번째 쓰기 작업이 첫 번째 변경을 포함하지 않으므로 변경 중 하나는 손실될 수 있음. 이는 읽기와 쓰기 사이의 갭 때문

- 원자적 쓰기 연산 : 단일 업데이트 쿼리로 읽기와 쓰기의 갭을 없앤다.
- 명시적 잠금 : 읽을 때 락을 걸어서 다른 트랜잭션의 쓰기 막기
    
    ```sql
    BEGIN TRANSACTION;
    
    SELECT * FROM figures
      WHERE name = 'robot' AND game_id = 222
      FOR UPDATE; (1)
    
    -- 이동이 유효한지 확인한 후
    -- 이전의 SELECT에서 반환된 것의 위치를 갱신한다.
    UPDATE figures SET position = '4' WHERE id = 1234;
    
    COMMIT;
    ```
    
- 갱신 손실 자동 감지(낙관적 락)
    - 값을 마지막으로 읽은 후로 변경되지 않았을 때만 갱신을 허용함으로써 갱신 손실을 회피하는 것
    - 여러 트랜잭션의 병렬 실행을 허용하고 트랜잭션 관리자가 갱신 손실을 발견하면 트랜잭션을 abort 시키고, 재시도하도록 강제
    
    ```sql
    UPDATE wiki_pages SET content = 'new content'
      WHERE id = 1234 AND content = 'old content';
    ```
