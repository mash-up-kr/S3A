# 6장 파티셔닝

데이터가 매우 많다면 데이터를 쪼갤 필요가 있다 → 샤딩, 파티셔닝

파티셔닝은 테이블을 나누는 것, 샤딩은 파티션을 각 노드에 분배하는 것

테이블은 논리적 단위, 물리적으로는 여러 개의 테이블로 분리해서 관리

<img width="1400" height="516" alt="image" src="https://github.com/user-attachments/assets/564f3e18-4445-417d-aa46-394e9425ed1d" />


파티셔닝을 하는 이유

- 나눠서 저장함으로서 질의 부하 분산
- 불필요한 데이터 간단 삭제 (과거 로그 등)

## 파티셔닝과 복제

<img width="518" height="276" alt="image" src="https://github.com/user-attachments/assets/4713e6e1-a8aa-4221-b9f5-60e8684db4a2" />


보통 복제와 파티셔닝을 함께 사용해 각 파티션의 복사본을 노드에 저장 (파티션 단위로 복제)

즉, **하나의 데이터베이스 전체를 통째로 복제하는 게 아니라, 각 파티션마다 복제본을 둔다**고 보면 된다.

각 파티션의 리더는 하나의 노드에 할당되고 팔로워들은 다른 노드에 할당된다. 노드는 어떤 파티션에게는 리더이면서 다른 파티션에게는 팔로워가 될 수 있다.

## 키-값 데이터 파티셔닝

- 쏠림(skewed) : 파티셔닝이 고르게 이뤄지지 않아 다른 파티션보다 데이터 또는 질의가 많은 파티션
- 핫스팟 : 불균형하게 부하가 높은 파티션

→ 둘의 차이는 쏠림은 분포가 뷸균등한 것, 핫스팟은 특정 파티션으로 부하가 몰리는 것

### 키 범위 기준 파티셔닝

파티션에 연속된 범위의 키를 할당.

```sql
CREATE TABLE employees (
    id INT NOT NULL,
    first_name VARCHAR(30),
    last_name VARCHAR(30),
    reg_date DATE NOT NULL DEFAULT '1970-01-01',
    ....
) PARTITION BY RANGE( YEAR(hired) ) (
  PARTITION p0 VALUES LESS THAN (2000),
  PARTITION p1 VALUES LESS THAN (2010),
  PARTITION p2 VALUES LESS THAN (2020),
  PARTITION p3 VALUES LESS THAN MAXVALUE
);
```

**장점**

- 키를 정렬된 순서로 저장해 범위 스캔에 유리

**단점**

- 특정 접근 패턴이 핫스팟 유발 가능 (ex. 최근 1년 데이터 등)
    - 이 경우 나머지 파티션은 유휴 상태로 남는다.

### 키 해시값 기준 파티셔닝

<img width="636" height="217" alt="image" src="https://github.com/user-attachments/assets/555fd562-c592-4834-8a25-8cccb49f0de7" />


해시 함수가 강력할 필요는 없다..? → 보안이 목적이 아니라 여러 파티션에 고르게 분산시키는게 목적이기 때문. 단 분산 품질은 좋아야 한다.

장점

- 데이터 균등 분산

단점

- 범위 질의에는 약함
- 정렬 순서가 유지되지 않음

### 복합 키

여러 컬럼을 함께 사용해서 **파티션을 결정**하는 방식

```
partition_key = (order_date, bucket)
hash(order_date, bucket) → 파티션 결정
```

이 경우 `order_date`와 `bucket`을 조합한 값이 파티션을 결정

```
2026-05-04 + bucket 0 → Partition A
2026-05-04 + bucket 1 → Partition B
2026-05-04 + bucket 2 → Partition C
```

이 방식은 특정 날짜에 주문이 너무 몰릴 때 유용하다.

```
오늘 주문 전체 → 하나의 파티션
```

이렇게 몰리는 걸 막고,

```
오늘 주문 + bucket 0
오늘 주문 + bucket 1
오늘 주문 + bucket 2
오늘 주문 + bucket 3
```

이렇게 나눌 수 있어.

즉, **복합 파티션 키는 데이터 분산을 개선하기 위한 목적**이 강하다.

## 쏠림과 핫스팟 완화

키 해싱은 핫스팟을 줄이는 데 도움이 되지만, 완벽히 제거는 불가능. **특정 키가 인기** 있는 경우에는 결국 요청이 특정 파티션으로 몰린다. (SNS의 인플루언서 ID 등!)

보완책

- 각 키의 시작이나 끝에 랜덤 수(10진수 2개 정도?)를 붙이면 작업이 100개의 다른 키로 각 파티션에 분산된다. (Salt / 랜덤 suffix 붙이기)
    - 단 이 방식은 읽기 시 쪼갠 키만큼 데이터를 읽어서 조합해야 하므로 추가 공수가 든다.
    - 이 기법은 요청이 몰리는 소수의 키에만 적용

## 파티셔닝과 보조 인덱스

### 문서 기준 보조 인덱스 파티셔닝 (로컬 인덱스)

기본키로 파티션에 접근할 때는 쉽지만, 보조 인덱스로 접근할 때는 애매하다.

파티셔닝이 기본키 기준으로 돼있기 때문이다. 

예를 들어 

```java
SELECT * FROM users WHERE email = 'a@example.com';
```

이렇게 조회한다면 email = 'a@example.com'인 사용자가 어느 파티션에 있는지는 email만 보고는 알 수 없다. 파티셔닝은 userId(기본키) 기준으로 결정돼있기 때문이다.

<img width="539" height="263" alt="image" src="https://github.com/user-attachments/assets/c37ff2fd-2696-4cfc-acbb-b58242bf6b9a" />


각 파티션에 속한 문서만 담당하며 완전히 독립적으로 동작하는 인덱스를 **로컬 인덱스**라고 한다. 로컬 인덱스는 담당하는 데이터 파티션에 같이 저장된다.

만약 email 인덱스로 데이터를 조회하고 싶다면 모든 파티션에 질의를 보내고 결과를 합쳐야 한다. 이를 `scatter-gather query` 라고 한다.

쓰기는 단순하다.

```java
user_id = 10 데이터가 Partition 2에 저장됨
→ Partition 2의 보조색인만 갱신하면 됨
```

하지만 읽기가 비싸진다.

```java
city = Seoul 검색
→ 모든 파티션에 scatter
→ 각 파티션에서 결과 조회
→ gather해서 합침
```

MySQL의 경우는 파티션 테이블에서 인덱스는 전부 로컬 인덱스에 해당하고, 글로벌 인덱스는 지원하지 않는다. 

따라서 조건절에 일치하는 레코드를 정렬된 순서대로 읽으면서 우선순위 큐에 임시로 저장하고, 다시 필요한 순서대로 데이터를 가져간다. 즉, 단일 서버에서 ORDER BY는 인덱스 스캔하면 정렬된 순서로 조회되는 것과 달리 **내부적으로 큐 처리**가 한 번 더 필요하다.

### 용어 기준 보조 인덱스 파티셔닝 (글로벌 인덱스)

<img width="556" height="252" alt="image" src="https://github.com/user-attachments/assets/a3140a13-e242-4ff8-8673-63ed41fa9aa0" />


로컬 인덱스와 다르게 모든 파티션의 데이터를 담당하는 인덱스를 **글로벌 인덱스** 라고 한다.

특정 파티션 하나의 데이터만 보는 게 아니라 **모든 파티션의 데이터를 대상**으로 만든 인덱스이다. 로컬 인덱스와 다르게 인덱스 전용 파티션에 저장된다.

찾고자 하는 용어에 따라 파티션이 결정되므로 용어 기준 파티셔닝이라 한다.

```
Global city index

Seoul → user_id 1, user_id 2, user_id 3
Busan → user_id 4, user_id 6
Daegu → user_id 5
```

이제 `city = Seoul`을 찾을 때 모든 데이터 파티션을 뒤질 필요가 없다.

```
city = Seoul
→ Global city index 조회
→ user_id 1, 2, 3 획득 (파티션 키)
→ 각 user_id가 있는 데이터 파티션 조회
```

글로벌 인덱스는 한 노드에 두면 해당 노드가 병목이 될 수 있기 때문에 보통 파티셔닝한다.

```sql
1. city = Seoul 값을 해시해서 글로벌 인덱스 파티션 찾기
2. 해당 인덱스 파티션에서 user_id 목록 찾기
3. user_id로 실제 데이터 파티션 조회
```

**장점**

- 읽기 효율 상승 (scatter-gather 불필요)

**단점**

- 단일 문서를 쓸 때 해당 색인이 여러 파티션에 영향을 미치며 따라서 쓰기가 느리다.
    - 해당 쓰기에 영향받는 모든 파티션에 걸친 분산 트랜잭션이 필요하다.
    - 그러나 분산 트랜잭션은 모든 DB에서 제공되진 않는다. 현실적으로는 글로벌 인덱스는 대개 비동기로 갱신된다.

→ 결론 : 글로벌 인덱스는 쓰기 부하 때문에 보통 로컬 인덱스를 많이 쓴다.

## 파티션 재균형화

데이터베이스가 스케일 업한다면? (새로운 노드가 추가된다면?)

노드의 부하를 분산해야 한다.

- 부하 균등
- 순단 X
- 데이터 이동 최소화

### 재균형화 전략

- 해시 값에 mod N(노드 개수) 연산
    - N이 바뀌면 대부분 키가 이동해야 함 → 비용 상승
- 정적 파티셔닝(파티션 개수 고정)
    - 적절한 크기를 구한다면 괜찮지만, 그게 어려움
- 동적 파티셔닝
    - 임계치 미만 시 동적 합병, 초과 시 동적 분할
- 노드 비례 파티셔닝
    - 완전 자동 재균형화와 완전 수동 재균형화가 있다.

### 요청 라우팅

파티션이 재균형화되면 노드에 할당되는 파티션이 바뀐다. 즉, 데이터 위치가 바뀐다. 이러면 파티션 할당 정보를 실시간으로 파악하고 있어야 하는 요구사항이 생긴다.

이건 단순 분산 DB 뿐만 아니라 고가용성을 지향하는 소프트웨어에서는 모두 갖는 문제다.

그렇다면 라우팅 결정을 내리는 구성요소가 노드에 할당된 파티션의 변경사항을 어떻게 아느냐

- 주키퍼
- 구성 요소 없이 개별 노드가 파티션 할당 정보 소유
- 라우팅 계층이 파티션 할당 정보 소유
