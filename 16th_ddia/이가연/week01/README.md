# 3장 저장소와 검색

## 데이터베이스를 강력하게 만드는 데이터 구조

### 키-값 저장소

색인의 문제점은 결국 트레이드 오프 : 잘못 선택하면 읽기 속도에 향상을 주지 못하며 모든 색인은 쓰기 속도를 떨어트린다.

### 해시 색인

해시 색인은 모든 키를 메모리에 유지해야 하기 때문에 충분한 메모리가 필요하지만, 값은 디스크에 저장되므로 메모리보다 큰 데이터를 다룰 수 있다. 데이터 파일 일부가 캐시에 존재하면 디스크 접근 없이 빠르게 읽을 수 있다. 동일한 키에 대한 값이 자주 갱신되는 환경에 적합하며, Bitcask와 같은 저장 엔진에서 사용된다.

**컴팩션과 세그먼트 병합**

<img width="842" height="447" alt="image" src="https://github.com/user-attachments/assets/604e8424-5166-4732-85bc-c1274d221829" />


데이터를 계속 append만 하면 디스크 공간이 증가하기 때문에 로그를 일정 크기의 세그먼트로 나누어 관리한다. 컴팩션은 동일한 키에 대한 중복 데이터를 제거하고 최신 값만 남기는 과정이며, 이 과정에서 여러 세그먼트를 병합할 수 있다. 세그먼트는 불변 구조이기 때문에 컴팩션과 병합은 백그라운드에서 수행되며, 기존 읽기와 쓰기 작업을 방해하지 않는다.

삭제 연산은 실제 데이터를 즉시 제거하는 대신 톰스톤이라는 삭제 마커를 기록하는 방식으로 처리된다. 이후 컴팩션 과정에서 해당 키의 이전 값과 함께 제거된다.

데이터베이스가 재시작되면 인메모리 해시 맵이 사라지기 때문에 로그를 재생하거나 스냅샷을 통해 복구한다. 또한 로그 기록 중 장애가 발생할 수 있으므로 체크섬을 이용해 손상된 데이터를 감지하고 무시한다.

append-only 로그 구조는 순차 쓰기를 사용하기 때문에 랜덤 쓰기보다 디스크 성능이 좋다. 파일이 불변이기 때문에 동시성 제어가 단순하며, 쓰기는 단일 스레드로 처리하고 읽기는 여러 스레드에서 동시에 수행할 수 있다.

**한계**

- 모든 키를 메모리에 저장해야 하기 때문에 데이터 규모가 커질수록 메모리 사용량이 문제가 됨.
- 키의 정렬을 보장하지 않기 때문에 특정 범위를 조회하는 range query에 비효율적
- 디스크 기반 해시 구조는 랜덤 I/O로 인해 성능이 떨어진다.

## SS 테이블과 LSM 트리

SS테이블은 키-값 데이터를 키 기준으로 정렬하여 저장하는 구조로, 각 키는 하나의 세그먼트 내에서 한 번만 등장한다. 

이러한 정렬 구조 덕분에 해시 색인과 달리 범위 기반 탐색이 가능하며, 병합 과정에서도 효율적인 처리가 가능하다.

**SS테이블의 특징과 장점**

<img width="797" height="533" alt="image" src="https://github.com/user-attachments/assets/9d93fef3-2ce0-4ebe-a95c-251407b42e98" />


SS테이블은 **정렬**된 구조를 기반으로 하기 때문에 세그먼트 병합이 매우 효율적으로 수행된다. 병합 과정은 병합 정렬과 유사하게 각 세그먼트를 순차적으로 읽으면서 가장 작은 키를 선택해 새로운 파일에 기록하는 방식으로 진행된다. 

여러 세그먼트에 동일한 키가 존재하는 경우에는 **가장 최신 세그먼트의 값을 유지**하고 이전 값은 제거된다.

<img width="793" height="404" alt="image" src="https://github.com/user-attachments/assets/f5bd6528-52b7-4cdd-bf01-5e18d199f7b1" />


또한 모든 키를 메모리에 유지할 필요가 없으며, 일부 키만 인덱스로 유지해도 된다. 특정 키를 찾을 때는 정렬된 순서를 활용해 탐색 범위를 좁힐 수 있다. 읽기 시에는 필요한 범위의 데이터를 스캔하게 되며, 이때 레코드를 블록 단위로 묶어 압축하면 디스크 공간 절약과 함께 I/O 횟수를 줄일 수 있다.

**SS테이블 생성과 유지**

SS테이블은 디스크에서 직접 정렬 상태를 유지하기보다는 **메모리에서 먼저 정렬한 뒤 디스크로 flush**하는 방식으로 생성된다. 쓰기 요청은 먼저 메모리 내의 균형 트리 구조에 저장되며, 이를 memtable이라고 한다. memtable은 레드-블랙 트리나 AVL 트리와 같은 자료구조를 사용해 항상 정렬 상태를 유지한다.

memtable의 크기가 일정 임계값을 넘으면 이를 디스크에 정렬된 SS테이블 파일로 기록하고, 해당 파일은 새로운 세그먼트로 추가된다. 이후 새로운 쓰기 요청은 다시 비어있는 memtable에 기록된다. 읽기 요청이 들어오면 먼저 memtable을 조회하고, 없을 경우 최신 세그먼트부터 순차적으로 확인한다.

- 세그먼트 파일은 주기적으로 병합과 컴팩션을 수행하며, 이는 백그라운드에서 처리된다.
- 덮어쓰기나 삭제된 데이터는 병합 과정에서 제거된다.

**장애 복구 방식**

memtable은 메모리에만 존재하기 때문에 장애 발생 시 데이터가 유실될 수 있다. 이를 방지하기 위해 append-only 로그를 디스크에 함께 기록한다. 이 로그는 순서대로 기록되며 정렬되어 있지 않아도 되며, 장애 발생 시 해당 로그를 재생하여 memtable을 복구한다. memtable이 SS테이블로 flush되면 해당 로그는 더 이상 필요 없어져 삭제할 수 있다.

**LSM 트리 구조**

LSM 트리는 SS테이블과 memtable 구조를 기반으로 한 저장소 엔진으로, 정렬된 파일을 지속적으로 생성하고 병합하는 방식으로 동작한다. 쓰기는 메모리에 먼저 기록된 후 순차적으로 디스크에 저장되며, 여러 SS테이블은 백그라운드에서 점진적으로 병합된다. 이러한 구조는 디스크 쓰기를 순차적으로 만들기 때문에 높은 쓰기 처리량을 제공한다.

또한 데이터셋이 메모리보다 훨씬 크더라도 효율적으로 동작할 수 있다는 특징이 있다.

**성능 최적화 기법**

- 존재하지 않는 키를 찾을 때 불필요한 디스크 탐색을 줄이기 위해 블룸 필터를 사용한다.
  - 블룸 필터는 특정 키가 데이터에 없음을 빠르게 판단할 수 있는 메모리 효율적 자료구조
- SS테이블은 블록 단위 압축을 통해 디스크 사용량과 I/O를 줄인다.
- 컴팩션과 병합 전략에 따라 성능 특성이 달라진다.
- 크기 계층 컴팩션은 작은 SS테이블을 큰 SS테이블에 순차적으로 병합하는 방식으로 단순하지만 쓰기 증폭이 발생할 수 있다.
- 레벨 컴팩션은 데이터를 여러 레벨로 나누어 관리하며 점진적으로 병합하여 디스크 공간 사용을 줄이고 읽기 성능을 개선한다.

**정리**

LSM 트리는 정렬된 SS테이블을 기반으로 데이터를 순차적으로 기록하고, 백그라운드에서 병합과 컴팩션을 수행하는 구조이다. 이 방식은 높은 쓰기 성능과 대용량 데이터 처리에 강점을 가지지만, 읽기 시 여러 세그먼트를 확인해야 한다는 특징을 가진다.
