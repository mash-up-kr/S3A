### 리더와 팔로우

<img width="887" height="332" alt="image" src="https://github.com/user-attachments/assets/11b9c83f-58c7-431e-b0a1-ec652c44e1d0" />


모든 복제 서버에 데이터가 있는 걸 보장하기 위해 **리더 기반 복제**

- **리더**(마스터, 프라이머리)
    - 쓰기, 읽기 담당
    - 쓰기 후 팔로워에게 복제
- **팔로워**(슬레이브, 2차)
    - 리더가 보낸 데이터 변경 로그, 스트림 등을 받아 복제본 갱신
    - 읽기만 담당

→ RDBMS, NoSQL, MQ 등 다양하게 마스터 슬레이브 구조를 사용

### 동기식 VS 비동기식 복제

<img width="913" height="402" alt="image" src="https://github.com/user-attachments/assets/e3aee568-d26d-4e0c-9c97-c358aef647eb" />


**동기식 복제**

- 팔로워 1의 경우 동기식 복제다. 리더가 팔로워의 응답을 받은 뒤 사용자에게 응답을 보낸다.
- 리더와 팔로워 간의 강력한 일관성을 보장하나 팔로워에 장애가 발생하면 리더에게도 전파된다.

**비동기식 복제**

- 팔로워 2의 경우 비동기식 복제다. 동기와 반대로 일관성을 떨어지지만 사용자 응답 지연시간이 적고 고가용성을 제공한다.
- 완전한 비동기식 복제는 리더 쓰기 장애 시 복제되지 않은 데이터 유실 문제와 쓰기 지속성 보장 불가 이슈로 **많은 팔로워**가 존재하거나, 노드가 **지리적으로 분산**된 경우 사용한다.

**반동기식 복제**

- 모든 팔로워가 동기를 사용할 순 없다. 고가용성이 너무 떨어짐
- 사진과 같이 팔로워 하나는 동기식, 하나는 비동기식으로 구성하는 걸 반동기식 복제라 한다.
- 하나의 팔로워라도 ACK를 보내면(동기 복제가 성공하면) 나머지는 비동기로 복제
- 동기식 팔로워가 장애 시 다른 비동기 팔로워가 동기식으로 동작한다.

→ 결론 : 동기식/비동기식만을 쓸 순 없고, 대부분 반동기식 복제를 사용한다.

### 새로운 팔로워 설정

1. 데이터베이스를 잠그지 않고 리더의 데이터베이스 스냅샷을 가져온다.
2. 스냅샷을 새로운 팔로워 노드에 복사한다.
3. 이후 팔로워는 리더에 연결해 스냅샷 이후 발생한 모든 데이터 변경 내역을 요청한다.
4. 요청한 데이터 변경 미처리분(backlog)을 모두 처리하면 팔로워가 리더를 따라잡았다고 말한다.
5. 이제 팔로워는 리더의 데이터 변경을 처리할 수 있다.

### 노드 중단 처리

노드는 장애나 유지보수 등의 이유로 중단될 수 있지만 시스템은 중단 되어선 안된다.

- 팔로워 장애
    - **따라잡기 복구**
    - 리더의 데이터 변경 로그를 로컬 디스크에 저장해놨다가 복구, 로그의 마지막 트랜잭션 이후 데이터 변경 내역은 리더에게 요청해 적용하면 따라잡기 완료
- 리더 장애
    - **장애 복구**
    - 팔로워 중 하나를 새로운 리더로 승격
    - 리더 팔로워 구조에서 **합의** 방식은 노드 수는 최소 **3개**여야 한다. 투표는 항상 과반수여야 하는데, 2개의 경우에는 리더가 죽어도 팔로워가 혼자라 과반수 투표가 불가능하다. (Kafka, ES 등)
    - 또는 제어 노드가 리더를 임명하기도 한다. (MySQL 등 DB)
    - 만약 이전 리더가 복구되면 새로운 리더를 인식하고 자신은 팔로워가 된다.

리더 장애에서 자동 장애 복구의 위험

- 만약 비동기식 복제를 사용하고 있었다면 새로 승격된 리더 노드의 데이터 최신성은 보장하지 못한다.
- **스플릿 브레인**이 발생하는 경우 해당 쓰기는 폐기하는데, 신뢰도가 떨어지게 된다.
    - 두 노드가 자신이 리더라고 인식하는 상황
    - 동시에 쓰기 요청을 처리하기 때문에 쓰기 충돌이 발생 → 데이터 오염
- 죽었다고 판단할 타임아웃 값 정하기 모호
    - 일시적 부하 급증, 네트워크 문제 등으로 일정할 수 없음

### 복제 로그 구현

- **구문(쿼리) 복제**
    - 비결정적 요인(NOW(), RAND(), 쿼리 순서) 등에 의해 복제 깨질 수 있음
    - 비결정적 함수 호출을 고정 값으로 반환하도록 대체해서 보완할 수 있지만, 대부분 쓰지 않음, MySQL의 경우 비결정적 요인 있으면 로우 기반 복제 방식으로 변경
- **쓰기 전 로그(리두로그,WAL)**
- **로우 기반(논리적) 복제**
    - “어떤 row의 값이 어떻게 바뀌었는지”를 그대로 기록해서 전송
    - 논리적 로그와 저장소 엔진 내부를 분리할 수 있다는 장점이 있다. InnoDB든, 다른 엔진이든 스토리지 구조와 엔진에 상관 없이 복제 가능
    - 비결정적 요인에 영향 받지 않고, 결과를 복제하므로 복잡한 쿼리도 안전
    - CDC(변경 데이터 캡처)에서 로우 기반 복제 사용
    - 하지만 로그 크기가 커진다는 단점
- **트리거 기반 복제**

### 복제 지연 문제(Replication Lag)

<img width="1101" height="430" alt="image" src="https://github.com/user-attachments/assets/4855b0ec-bd2a-499d-a4b4-82a64da6cd96" />


읽기 확장 아키텍처 : 하나의 리더와 여러 팔로워

- 확장성, 지연 시간 단축 장점
- 비동기식 복제 방식에서만 동작한다.
- 따라서 데이터 일관성을 강하게 보장하진 못한다.
- 하지만 최종적 일관성에 의해 결국 팔로워는 리더를 따라잡게 된다.

**복제 지연**

→ 아직 팔로워에게 복제되기 전에 팔로워에서 조회해서 최신 데이터를 읽지 못하는 이슈!

해결 방법

- 사용자가 수정한 내용은 항상 리더에서 읽는다. 그 외는 팔로워에서 읽는다. 하지만 앱 내 대부분의 내용이 사용자 편집이 가능하다면 무의미하다.
- 팔로워 복제 검증
    - 마지막 갱신 시각, 클라이언트의 타임스탬프 등으로 팔로워가 마지막 복제가 완료됐는지 확인

**멀티 디바이스 간 쓰기 후 읽기 일관성**

- 웹에서 방금 입력한 정보가 앱에서도 조회가 되어야 한다.
- 멀티 디바이스의 경우 클라이언트의 타임스탬프를 알 수 없다. → 메타데이터를 중앙집중형으로 관리
- 복제 서버가 여러 데이터센터에 분산된 경우 → 사용자의 요청은 항상 동일한 데이터센터로 라우팅

### 단조 읽기

<img width="1099" height="541" alt="image" src="https://github.com/user-attachments/assets/8ed5b477-ff5a-417f-bcfb-2aa8d41327d6" />


사용자가 최신 복제 팔로워에서 먼저 읽고 그 다음 아직 복제가 안된 팔로워에서 읽으면 시간 역전 현상이 발생한다.

사용자는 동일한 읽기 요청을 여러 번 보냈는데 복제 여부에 따라 값이 다를 수 있다. 모든 팔로워의 복제 상태가 동일하지 않기 때문에 발생하는 문제다.

**단조 읽기**

위와 같은 현상을 발생하지 않도록 보장, 강한 일관성보단 덜하고, 최종적 일관성보단 강한 보장이다.

**사용자는 새로운 데이터를 읽은 후 예전 데이터를 읽지 않는다.**

사용자의 읽기는 항상 동일한 복제 서버에서 수행되도록 한다. 이는 사용자 ID 해시값으로, 복제 서버 고장 시 다른 팔로워로 재라우팅할 필요가 있다.

### 일관된 순서 읽기

인과성 위반 대화

<img width="916" height="621" alt="image" src="https://github.com/user-attachments/assets/cfa54e2e-6f55-4767-ac88-a7cab0b9b5f6" />


순서가 중요한 데이터에서 순서대로 복제되지 않아서 발생하는 현상

일관된 순서로 읽기는 순서가 중요한 쓰기 작업에서 다른 사용자도 해당 순서대로 데이터를 읽는 것을 보장

인과성 위반은 파티셔닝/샤딩에서 발생한다.

분산 데이터베이스에서 파티션은 서로 독립적으로 동작해 쓰기의 전역 순서가 없기 때문

인과성이 있는 쓰기의 경우 동일한 파티션에 기록되도록 할 수 있다. 하지만 더 좋은 알고리즘이 있다.

### 다중 리더 복제

쓰기 처리를 하는 각 노드는 모든 데이터 변경을 모든 노드에 전달하며, 여기서 각 리더는 동시에 다른 리더의 팔로워 역할도 한다.

모든 쓰기는 해당 리더를 거쳐야 하고, 리더 연결이 불가능한 경우 쓰기 불가능한 단점을 보완한다.

**성능**

- 단일 리더 : 쓰기는 인터넷을 통해 리더가 있는 데이터 센터로 이동해야해서 쓰기 지연 발생
- 다중 리더 : 쓰기는 로컬 데이터센터 처리 후 비동기 방식으로 다른 데이터센터에 복제

**데이터센터 중단 내성**

- 단일 리더 : 리더가 있는 데이터센터가 고장 나면 다른 데이터센터의 팔로워를 리더로 승진
- 다중 리더 : 각 데이터센터 독립적으로 동작, 고장난 데이터센터가 온라인으로 돌아왔을때 복제

**네트워크 문제 내성**

- 단일 리더 : 데이터 센터 내 쓰기는 동기식이기에 데이터 센터 내 연결 문제에 민감
- 다중 리더 : 비동기 복제를 사용해 네트워크 문제에 보다 잘 견딤, 일시적 네트워크 중단에도 쓰기 처리는 진행되기 때문

**사용 사례 - 오프라인 작업을 하는 클라이언트**

- 인터넷 연결이 끊어진 동안 애플리케이션 동작해야 하는 경우
- 협업 편집 : 동시에 여러 사람이 문서를 편집할 수 있는 애플리케이션 ex) 이더패드, 구글 독스 등

### 쓰기 충돌 다루기

<img width="931" height="554" alt="image" src="https://github.com/user-attachments/assets/33342976-4096-4caa-b6d3-cf06a619da65" />


다중 리더 구조에서 발생하는 쓰기 충돌 처리는 어렵기 때문에 **회피**가 권장된다.

특정 레코드의 모든 쓰기는 동일한 리더에서 처리하면 된다. → 특정 사용자의 요청을 동일한 데이터센터로 항상 라우팅

그럼에도 해결 방법

- 쓰기에 고유 ID 부여해 높은 것만 선택(LWW) → 데이터 유실 가능성, 그러나 대중적
- 버전 기반 관리 (낙관적 락처럼 실패 시 재시도)
- 모든 충돌 기록 보존, 나중에 사용자에게 알려 충돌 해결 코드 작성
